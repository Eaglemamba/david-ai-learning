I CLAUDE.md：最被低估的功能
如果你只記住這篇文章的一件事，就記這個：寫好你的 CLAUDE.md。CLAUDE.md 是 Claude Code 在每次對話開始時自動讀取的指令檔案。它就像你給新同事寫的 onboarding doc — 你希望他知道什麼，你就寫什麼。
"If you only remember one thing from this article, remember this: write a good CLAUDE.md. CLAUDE.md is the instruction file that Claude Code automatically reads at the start of every conversation."
很多人不寫 CLAUDE.md，或者隨便寫兩行。結果就是每次對話都要從頭解釋專案結構、編碼規範、技術棧選擇。這就像每天早上都要重新給同事介紹一遍公司。
"Many people don't write a CLAUDE.md, or just write a couple of lines. The result is having to explain the project structure, coding standards, and tech stack from scratch every conversation."
一個好的 CLAUDE.md 有幾個關鍵原則：
第一，寫 Claude 從程式碼裡讀不出來的東西。專案的「為什麼」比「是什麼」更重要。你不需要解釋 React 怎麼用，但你需要告訴它「我們選 Tailwind 是因為團隊統一了這個規範」。
"Write things Claude can't read from the code. The 'why' of a project matters more than the 'what'."
第二，控制在 200 行以內。官方文件明確提到，CLAUDE.md 太長會導致 Claude 忽略規則。用 markdown 標題和列表，保持可掃描性。
"Keep it under 200 lines. Official docs explicitly mention that too-long CLAUDE.md causes Claude to ignore rules."
第三，不要放頻繁變化的內容。詳細的 API 文件、逐檔案描述這些東西不適合放在 CLAUDE.md 裡。放連結就好。
"Don't put frequently changing content. Detailed API docs and per-file descriptions don't belong in CLAUDE.md. Just put links."
💡 比喻說明
CLAUDE.md = 新同事的 Onboarding Doc。你不會每天重新介紹公司，同理也不應該每次對話都重新解釋專案。一次寫好，每次對話自動生效。
📚 CLAUDE.md 四級層級
層級位置用途
企業策略管理員設定全組織強制規則
全域~/.claude/CLAUDE.md個人通用偏好
專案專案根目錄 CLAUDE.md專案規範（Git 追蹤）
功能子目錄 CLAUDE.md特定模組規則
🛠️ 全域 CLAUDE.md 推薦四行
- 我是一名全端工程師，不需要過度解釋基礎概念
- 回覆盡量簡潔，不要加無關的客套話
- 程式碼改動後不要總結你做了什麼，我會看 diff
- 優先用簡單方案，不要過度工程

作者表示：這四行，能省掉你幾百次重複糾正的時間。

💻 進階：.claude/rules/ 目錄

專案變大後，可將規則拆分到 .claude/rules/ 目錄下，並用 YAML frontmatter 限定到特定檔案路徑：

---
paths: ["src/api/**/*.ts", "src/routes/**/*.ts"]
---
API 路由必須包含輸入驗證和錯誤處理。

這樣規則只在 Claude 存取匹配檔案時才載入，節省上下文空間。

II 提示詞的品質決定產出的品質
好的指令應該包含三個要素：目標、約束、上下文。
"Good instructions should contain three elements: goal, constraints, context."
差的指令：「給使用者列表加個搜尋功能」
好的指令：「在 @src/pages/UserList.tsx 的表格上方加一個搜尋框。搜尋走後端 API /api/users?search=xxx，不要前端過濾。用 debounce 300ms，樣式和現有的 Filter 元件保持一致。」
"The difference? Good instructions reduce ambiguity. Claude Code doesn't need to guess frontend vs backend search, debounce time, or style conventions."
用 @ 引用具體檔案 — 比起說「那個認證相關的程式碼」，直接寫 @src/auth/login.ts 精確得多。你甚至可以指定行號範圍：@src/auth/login.ts#5-30。
"Use @ to reference specific files. You can even specify line number ranges."
給驗證標準 — 不要只說「幫我修這個 bug」，而是給 Claude 一個可以自我驗證的標準：測試用例、截圖、預期輸出。這樣它能自己檢查工作品質，減少來回修改。
"Give verification criteria — test cases, screenshots, expected output — so Claude can self-check work quality."
明確「不要做」的事 — Claude Code 有時候會過度熱心。你讓它修一個 bug，它順手把周圍的程式碼也重構了。明確說「只改這個函式，不要動其他程式碼」。
"Claude Code sometimes gets over-enthusiastic — you ask it to fix a bug, it refactors surrounding code too."
模糊指令也有用武之地 — 探索性任務用模糊指令反而更好：「這個檔案有什麼可以改進的地方？」「幫我理解一下這個模組的架構」。但到了實作階段，切回精確模式。
"Vague instructions have their place — for exploratory tasks. But switch back to precise mode during implementation."
⚠️ 重點提示

好指令三要素：

1. 目標：做什麼（加搜尋框）
2. 約束：怎麼做（後端 API、debounce 300ms）
3. 上下文：在哪裡做（@具體檔案路徑）

多一分鐘寫清楚指令，省十分鐘來回修改。

🛠️ 四個官方推薦技巧

@ 引用：@src/auth/login.ts#5-30 精確定位程式碼

貼截圖：Ctrl+V 直接貼圖，UI 除錯效果是文字描述的十倍

驗證標準：「修完後跑 @tests/auth.test.ts 確認通過」

負面約束：「只改這個函式，不要動其他程式碼」

⚖️ 精確 vs 模糊指令使用時機
模式適用場景
精確指令實作功能、修 bug、程式碼變更
模糊指令探索架構、Code Review、找改進點
III Plan Mode：先想清楚，再動手
Claude Code 有一個 Plan Mode，這是作者認為最被低估的工作流。在 Plan Mode 下，Claude Code 只能讀取檔案、搜尋程式碼，不能做任何修改。它會探索程式碼庫，分析你的需求，然後給出一個詳細的實作計畫。
"Claude Code has a Plan Mode — the most underrated workflow. In Plan Mode, Claude Code can only read files and search code, it cannot make any changes."
進入方式有三種：啟動時用 claude --permission-mode plan；對話中按兩次 Shift+Tab；或在提示詞裡直接說「先不要改程式碼，幫我做個計畫」。
"Three ways to enter: start with --permission-mode plan; press Shift+Tab twice in conversation; or say 'don't change code yet, make a plan'."
關鍵操作：按 Ctrl+G 可以在你的編輯器裡打開並編輯計畫。你可以讓 Claude 生成初始計畫，然後在編輯器中刪掉不同意的部分、補充自己的想法，再切回正常模式讓 Claude 按修改後的計畫執行。
"Press Ctrl+G to open and edit the plan in your editor. Delete parts you disagree with, add your own ideas, then switch back to normal mode."
改一個計畫只需要一句話，改已經寫好的程式碼要花十倍的時間。
"Changing a plan takes one sentence; changing already-written code takes ten times longer."
什麼時候不需要 Plan Mode？改一個變數名、修一個明顯的 typo、加一行日誌。判斷標準很簡單：如果這個任務的實作方式只有一種，直接做。如果有多種選擇，先規劃。
"When not to use Plan Mode? Renaming a variable, fixing an obvious typo. If there's only one way to implement, just do it. Multiple choices? Plan first."
⚠️ 重點提示

Plan Mode 決策標準：

實作方式只有一種 → 直接做
有多種選擇 → 先規劃

複雜任務的時間分配應該是：30% 規劃 + 70% 實作，而非 0% 規劃 + 100% 重做。

🛠️ 推薦的複雜任務工作流

Step 1：進入 Plan Mode

Step 2：讓 Claude 閱讀相關程式碼

Step 3：請求計畫（如「制定 OAuth2 遷移方案」）

Step 4：Ctrl+G 審查和編輯計畫

Step 5：切回正常模式

Step 6：「按照計畫實作，寫完跑測試」

Step 7：讓 Claude 對照需求自查

IV 子 Agent：Claude Code 的分身術
子 Agent (Sub Agent) 是 Claude Code 的一個強大機制 — 它可以啟動獨立的 AI 進程來並行處理任務，每個子 Agent 有自己的上下文視窗，不會汙染你的主對話。
"Sub Agents can launch independent AI processes for parallel tasks, each with its own context window, without polluting your main conversation."
子 Agent 最大的價值：隔離上下文。當你讓 Claude 跑測試、分析日誌、搜尋大量檔案時，這些操作會產生海量輸出，塞滿你的上下文視窗。用子 Agent 來做這些事，輸出留在子 Agent 的上下文裡，只有摘要返回給你。
"The biggest value: context isolation. Testing, log analysis, and file searches produce massive output. Sub Agents keep that in their own context, returning only summaries."
按 Ctrl+B 可以把子 Agent 放到背景執行。你可以繼續和 Claude 聊其他事，等背景任務完成後會自動通知你。
"Press Ctrl+B to put a Sub Agent in the background. Continue chatting while it runs, and get notified when done."
你還可以在 .claude/agents/ 目錄下建立自訂 Agent，例如一個專門做 Code Review 的 Agent，然後用 @"code-reviewer (agent)" 隨時呼叫。
"Create custom agents in .claude/agents/ directory, like a dedicated code reviewer, callable with @'code-reviewer (agent)'."
📚 內建子 Agent 類型
類型能力用途
Explore唯讀，快速搜尋探索、找檔案、找定義
Plan唯讀，研究分析Plan Mode 下的程式碼分析
General完整能力複雜的多步驟任務

• 跑測試套件（通常需要幾分鐘）
• 大範圍程式碼搜尋

適合前台呼叫：

• 「用子 Agent 跑全量測試，列出失敗用例」
• 「搜尋所有使用 deprecated API 的檔案」

V 上下文管理：最容易被忽視的關鍵
Claude Code 的上下文視窗雖然大（最多 1M token），但它不是無限的，而且上下文管理直接決定了輸出品質。
"Claude Code's context window is large (up to 1M tokens), but not infinite, and context management directly determines output quality."
上下文裡裝了什麼：你的對話歷史、Claude 讀取的所有檔案內容、命令執行的輸出、CLAUDE.md 檔案（每次都載入）、Memory 檔案（前 200 行）、載入的 Skill 和 MCP 工具定義。
"What's in context: conversation history, all read file contents, command outputs, CLAUDE.md (loaded every time), Memory (first 200 lines), loaded Skill and MCP tool definitions."
作者給出五個實用策略：/clear 一件事做完就清；/compact 手動壓縮上下文（可加焦點如「專注於 API 改動和測試結果」）；/context 查看上下文使用情況；用子 Agent 隔離噪音；在 CLAUDE.md 裡寫壓縮保留指令。
"Five practical strategies: /clear when done; /compact to compress (with focus); /context to inspect usage; Sub Agents for isolation; compression retention instructions in CLAUDE.md."
⚠️ 五個上下文管理策略

/clear：一件事做完就清，不要混用對話

/compact [焦點]：壓縮時指定保留重點

/context：找出哪些檔案/MCP 吃掉了 token

子 Agent：隔離測試/日誌等大量輸出

CLAUDE.md 保留指令：壓縮後自動保留關鍵資訊

⚖️ Memory vs CLAUDE.md
面向MemoryCLAUDE.md
適合存個人偏好、歷史決策專案規範、編碼標準
不適合存程式碼細節、Git 歷史頻繁變化的 API 文件
持久性跨專案、跨對話僅限該專案
管理/memory直接編輯檔案
💻 CLAUDE.md 壓縮保留指令
# 壓縮指令
壓縮上下文時，始終保留：
- 已修改檔案的完整列表
- 測試命令和結果
- 關鍵的架構決策
VI 權限管理：安全和效率的平衡
Claude Code 提供了五種權限模式，用 Shift+Tab 在對話中循環切換：Plan（唯讀）、Auto Edit（自動編輯）、Full Auto（完全自動）等。
"Claude Code provides five permission modes, cycled with Shift+Tab: Plan (read-only), Auto Edit, Full Auto, etc."
在 .claude/settings.json 裡，你可以精細控制每種工具的權限。規則優先級：deny → ask → allow（deny 最優先）。這意味著你可以大膽地 allow 常用命令，同時用 deny 保護敏感檔案。
"Rule priority: deny → ask → allow (deny has highest priority). You can boldly allow common commands while using deny to protect sensitive files."
Settings 也有多級層級：組織策略（最高優先級）→ CLI 參數 → .claude/settings.local.json（本地，不提交到 Git）→ .claude/settings.json（專案級，團隊共享）→ ~/.claude/settings.json（全域）→ 預設值。
"Settings have multiple levels: organization policy → CLI params → settings.local.json → settings.json → global → defaults."
💻 權限設定範例
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git commit *)",
      "Edit(src/**/*.ts)",
      "Read(*.md)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Edit(.env)"
    ]
  }
}
🛠️ Settings 分層建議

.claude/settings.json：團隊共享規則（提交到 Git）

~/.claude/settings.json：個人偏好（全域）

.claude/settings.local.json：敏感配置（加入 .gitignore）

VII Hooks：讓規則變成鐵律
CLAUDE.md 裡的指令是「建議」— Claude 大部分時候會遵守，但偶爾會忘。Hooks 是「鐵律」— 無論如何都會執行。
"CLAUDE.md instructions are 'suggestions' — Claude mostly follows but occasionally forgets. Hooks are 'iron rules' — executed no matter what."
Hooks 是在特定生命週期事件上自動觸發的 shell 命令，配置在 settings.json 裡。關鍵事件類型包括：PreToolUse（工具執行前）、PostToolUse（工具執行後）、SessionStart（會話開始）等。
"Hooks are shell commands triggered on lifecycle events: PreToolUse, PostToolUse, SessionStart, etc."
Hook 命令返回 exit code 0 表示允許，exit code 2 表示阻止。這意味著你可以寫任意複雜的判斷邏輯。Hook 有四種類型：command（shell 命令）、http（POST 請求）、prompt（LLM 判斷）、agent（子 Agent 驗證）。
"Exit code 0 = allow, exit code 2 = block. Four hook types: command, http, prompt, agent."
📚 CLAUDE.md vs Hooks
面向CLAUDE.mdHooks
性質建議（偶爾忘記）鐵律（必定執行）
執行方式LLM 解讀Shell 命令自動觸發
適合編碼風格、偏好格式化、保護檔案
💻 自動格式化 Hook
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
      }]
    }]
  }
}

每次編輯後自動跑 Prettier，確保格式一致。

🛠️ 壓縮後重新注入關鍵資訊
"SessionStart": [{
  "matcher": "compact",
  "hooks": [{
    "type": "command",
    "command": "echo '用 Bun 不用 npm。提交前跑 bun test。'"
  }]
}]

解決常見痛點：對話壓縮後重要指令丟失。

VIII Git 工作流：用好 Worktree
Git Worktree 會在 .claude/worktrees/ 下建立一個獨立的 Git 分支副本。Claude 在裡面怎麼折騰都不影響你的主分支。退出時：沒有改動就自動清理，有改動就提示你保留或刪除。
"Git Worktree creates an independent branch copy. Claude can do anything inside without affecting your main branch."
這對於探索性任務特別有用。不確定某個重構方案能不能行？開個 worktree 試試，不行就扔掉，零風險。
"Especially useful for exploratory tasks. Not sure if a refactoring approach works? Try in a worktree — zero risk."
作者列出幾個 Git 鐵律：永遠不要讓 Claude Code 自動 push；頻繁 commit（用 /rewind 回退）；警惕 git reset --hard、git push --force、rm -rf 等破壞性操作。
"Git iron rules: never let Claude auto-push; commit frequently; beware destructive operations like git reset --hard."
💻 Worktree 使用方式
# 指定名稱的隔離工作空間
claude --worktree feature-auth
claude --worktree bugfix-123

# 自動產生名稱
claude --worktree

# 從 PR 恢復上下文
claude --from-pr 123
⚠️ Git 四條鐵律

1. 永遠不要讓 Claude 自動 push（commit 可以，push 要你確認）

2. 頻繁 commit — 保留 /rewind 回退能力

3. 警惕 --hard、--force、rm -rf

4. 可以在權限規則裡直接 deny 這些危險命令

IX 快捷鍵和命令速查
作者列出每天都在用的快捷鍵和斜線命令。記住這些快捷鍵可以大幅提升操作效率。
特別值得一提的是 /init 命令 — 它會自動分析你的程式碼庫，幫你生成 CLAUDE.md 的初稿。雖然通常需要手動調整，但比從零開始快得多。
"The /init command automatically analyzes your codebase and generates a CLAUDE.md draft. Usually needs manual adjustment, but much faster than starting from scratch."

Ctrl+G在編輯器中編輯計畫
Ctrl+B背景執行子 Agent
Ctrl+V貼上圖片
Option+T切換 Extended Thinking
Escape中斷目前操作

X Extended Thinking：讓 Claude 想深一點
對於複雜問題，可以開啟 Extended Thinking 模式，讓 Claude 在回答前花更多時間推理。用 Option+T（Mac）或 Alt+T 切換，也可用 /effort 命令設定級別。
"For complex problems, enable Extended Thinking to let Claude spend more time reasoning before answering."
適用場景：複雜的架構決策、棘手的 debug、多步驟重構方案、權衡多個方案利弊。不需要的場景：簡單的程式碼修改、格式化、重命名。
"Use for: complex architecture decisions, tricky debugging, multi-step refactoring. Skip for: simple code changes, formatting, renaming."
小技巧：在提示詞裡寫 ultrathink 可以強制觸發最高推理深度，不需要手動調整 effort 設定。
"Tip: writing 'ultrathink' in your prompt forces maximum reasoning depth without manually adjusting effort settings."

XI MCP Server：連接外部世界
MCP (Model Context Protocol) 讓 Claude Code 能和外部工具通訊 — GitHub、資料庫、Slack、Notion 等。配置在 .mcp.json 裡。
"MCP lets Claude Code communicate with external tools — GitHub, databases, Slack, Notion. Configured in .mcp.json."
注意上下文開銷：每個 MCP server 會額外消耗 100-500+ token 的工具定義。用 /mcp 命令可以查看開銷，停用不用的 server。作者的經驗：不要貪多，常用的 2-3 個就夠了。裝太多反而影響程式碼分析品質。
"Each MCP server costs 100-500+ tokens for tool definitions. Don't be greedy — 2-3 commonly used servers is enough. Too many hurts code analysis quality."

XII IDE 整合與程式碼審查
VS Code 整合：安裝 Claude Code 擴充功能後可在 VS Code 內直接使用。好用功能包括並排 diff 檢視、Option+K 快速插入 @引用、多標籤對話。
"VS Code integration: side-by-side diff view, Option+K for quick @references, multi-tab conversations."
程式碼審查原則：信任但要驗證。Claude Code 寫的程式碼品質總體不錯，但不能盲目信任。常見問題包括：過度工程（簡單函式搞出抽象層）、幻覺 API（用不存在的 API）、改動範圍膨脹（你改一個函式它重構整個檔案）。
"Trust but verify. Common issues: over-engineering, hallucinated APIs, scope creep in modifications."
Writer/Reviewer 模式：官方推薦的進階模式 — 用兩個獨立的會話分別扮演「寫程式碼」和「審查程式碼」的角色。兩個會話有獨立的上下文，Reviewer 不受 Writer 的思路影響，能發現 Writer 的盲點。
"Writer/Reviewer mode: two independent sessions playing 'write code' and 'review code' roles, catching each other's blind spots."
⚠️ 三大常見問題與對策

過度工程 → CLAUDE.md 寫「優先用簡單方案」

幻覺 API → 跑測試驗證，指令含驗證標準

範圍膨脹 → 明確說「只改 X，不要動其他程式碼」

🛠️ Writer/Reviewer 工作流

會話 A（Writer）：「實作 API 限流中間件」

會話 B（Reviewer）：「審查 @src/middleware/rateLimiter.ts，重點看邊界情況、競態條件」

會話 A：「修復審查反饋：[貼上 B 的輸出]」

XIV 不要用 Claude Code 做的事
1. 不要讓它做你完全不了解的事。如果你對 Kubernetes 一無所知，不要讓 Claude Code 幫你寫部署配置然後直接用。你無法審查你不懂的東西。
"Don't let it do things you don't understand at all. You can't review what you don't understand."
2. 不要在沒有版本控制的環境下用。沒有 Git 就沒有回退的能力。Claude Code 的改動可能覆蓋你的檔案，沒有版本控制就是裸奔。
"Don't use without version control. No Git = no rollback ability. Without version control, you're flying blind."
3. 不要一口氣扔一個巨大的任務。「把整個專案從 JavaScript 遷移到 TypeScript」— 這種指令等於讓 Claude Code 自由發揮。結果不可控。拆成小步驟，每一步確認後再做下一步。
"Don't throw a massive task all at once. Break into small steps, confirm each before proceeding."
4. 不要指望一次就完美。迭代是正常的。用 /rewind 回退，用精確的反饋描述「哪裡不對」。
"Don't expect perfection on the first try. Iteration is normal."
⚠️ 核心哲學

「它是一個極其能幹的協作者，
但不是自動駕駛。
方向盤始終在你手裡。」

工具的價值，永遠取決於使用它的人。

💡 比喻說明

Claude Code 就像一台超強力的電動工具：它能在幾秒鐘完成你手動幾小時的工作，但如果你不知道切割方向，它也能在幾秒鐘摧毀你的作品。安全裝置（Git、權限、Plan Mode）不是限制，而是讓你敢放手用的前提。

Key Takeaways
CLAUDE.md 是投資報酬率最高的動作
一次寫好 200 行以內的 CLAUDE.md（含專案規範、編碼慣例、架構決策），每次對話都自動生效。搭配 .claude/rules/ 做路徑級別的規則管理，上下文空間用在刀口上。
Plan Mode + 驗證標準 = 品質保證
複雜任務走「探索 → 規劃 → 實作 → 驗證」四步流。Ctrl+G 在編輯器中修改計畫比修改已寫好的程式碼快十倍。指令含測試用例讓 Claude 自我驗證。
上下文管理是隱形品質槓桿
/clear、/compact、子 Agent 三件套保持主對話上下文乾淨。MCP server 精簡到 2-3 個，避免工具定義吃掉分析空間。Memory 存偏好，CLAUDE.md 存規範。
安全防線要層層設置
權限規則（deny 優先）+ Hooks（鐵律，不會忘）+ Git Worktree（隔離實驗）+ 頻繁 commit（保留回退能力）。永遠不要讓 Claude 自動 push，永遠不要在沒有 Git 的環境下使用。
Practice Questions
Q1 概念理解 CLAUDE.md 指令和 Hooks 的本質差異是什麼？在什麼情況下你應該用 Hooks 而非 CLAUDE.md？

CLAUDE.md 的指令是「建議」性質 — 由 LLM 解讀執行，大部分時候會遵守但偶爾會忘記或誤解。Hooks 是「鐵律」— 是 shell 命令在特定生命週期事件上自動觸發，不經過 LLM 判斷，100% 會執行。

應該用 Hooks 的場景：需要 100% 保證的規則（如每次編輯後跑 formatter、阻止修改 .env 檔案、壓縮後重新注入關鍵指令）。如果「偶爾忘記」的代價很高（如生產配置被改），就用 Hook 而非 CLAUDE.md。

Q2 產業應用 在 GMP 環境的 CDMO 工廠中，如果要用 Claude Code 輔助開發批次記錄系統或偏差調查工具，你會如何設計權限規則和 CLAUDE.md？

權限規則：deny 優先保護所有驗證相關檔案（如 CSV 格式、21 CFR Part 11 審計追蹤模組）、deny rm -rf 和 git push --force。allow 限定在測試目錄和開發分支操作。所有涉及資料完整性的程式碼修改必須 ask。

CLAUDE.md：明確寫入「此專案遵循 GMP 軟體驗證要求（GAMP 5 Category 4/5）」「所有資料庫操作必須有審計追蹤」「不得繞過或修改驗證相關邏輯」「變更必須對應變更控制單號」。這是典型的 Copilot Mode — 人類審查不可繞過。

Q3 批判思考 作者說「不要讓 Claude Code 做你完全不了解的事」。但如果你正在學習一項新技術（如 Kubernetes），Claude Code 能否加速你的學習過程？這個建議的邊界在哪裡？

作者的建議核心是「你無法審查你不懂的東西」— 這在生產部署場景完全正確。但在學習場景中，Claude Code 反而是很好的學習加速器：你可以用 Plan Mode 讓它解釋架構、用模糊指令探索「這個 K8s 配置的各個欄位是什麼意思」。

邊界在於「輸出的用途」：學習用（sandbox 環境）可以大膽嘗試；生產用（真實部署）必須你能審查。這也呼應了 Copilot vs Agent 的框架 — 學習場景可以偏 Agent（讓 Claude 主導探索），生產場景必須嚴格 Copilot（人類審查每一步）。

✔ 核心洞察
Claude Code 的價值天花板不在模型能力，而在使用者的「上下文工程」能力 — CLAUDE.md 品質、指令精確度、上下文管理策略三者共同決定產出品質。這與我在文件生成 pipeline 中學到的「Context Engineering over Prompt Engineering」原則完全一致：精心策劃進入 context window 的內容，比打磨單條 prompt 更重要。Hooks 作為「鐵律」vs CLAUDE.md 作為「建議」的區分，對應到 GMP 環境就是 SOP 硬規定 vs 操作指引的差別。
