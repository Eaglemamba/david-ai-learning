Part 1：AI Agent 搜尋的隱性問題
每個搭建過 AI Agent 的人都遇過同一個問題：你給 Agent 搜尋工具，它去搜了，然後把整個網頁的 Headers、Footers、廣告、Cookie 橫幅全部塞進 Context Window 一起推理。
"Headers, footers, navigation menus, cookie banners, ads. All of it crammed into the context window."
你的 Agent 正在花 Token 讀垃圾。更糟的是，這些噪音實際上會降低回應品質——訊號被垃圾埋沒了。
"Your agent is now spending tokens reading junk. And worse, all that noise actually degrades the quality of the response."
這件事一直在發生，只是你看不到它。每一次網路搜尋，你的 Agent 就在默默地、低效率地處理大量無關內容。
"This is what your AI agents have been doing every single time they search the web. You just didn't see it."
💡 比喻說明
想像你要一個助理去圖書館找資料。舊方法是：把整本百科全書（包括版權頁、目錄、廣告頁）都影印帶回來，然後你們兩個再一起翻找。Dynamic Filtering 就像助理在圖書館就直接把相關頁面標記好、把廢頁丟掉，帶回來的已經是整理好的摘要。
⚠️ 重點提示
這個問題不是你的 Prompt 寫得不好。這是模型接收原始 HTML 時的架構性低效——以前沒有優雅的解法，你只能接受。現在 Anthropic 在模型層面解決了它。
Part 2：Dynamic Filtering 是什麼
Dynamic Filtering 的核心：Claude 在推理搜尋結果之前，會先自動撰寫並執行 Python 程式碼來清理資料。它決定什麼相關、什麼不相關，然後在更小的 Context Window 上給出更乾淨的答案。
"Claude writes and executes Python to filter the results first. It strips out the noise, keeps only what's relevant, and then reasons over the clean data."
這個模型即時生成自己的預處理程式碼，是一種新的能力模式——不是你寫 Prompt 告訴它怎麼做，而是它自主決定如何清理輸入。
"The model is writing its own preprocessing code on the fly. It decides what's relevant, throws away what isn't..."
最重要的是：這不需要你做任何事。沒有 Prompt 技巧，沒有自定義程式碼。它在模型層面自動發生。
"No prompt engineering trick. No custom code you had to build. It just happens at the model level now."
📚 核心概念
Dynamic（動態）的意思是：過濾邏輯不是預先寫死的規則，而是 Claude 根據每次查詢的具體內容即時生成的。同樣是搜尋「藥廠 GMP 合規要求」和「Python 語法教學」，它寫出的過濾程式碼會完全不同。
🛠️ 實務應用
對你的 Make.com 或 n8n workflow 來說，這個升級是零配置的——只要你把模型版本從 claude-sonnet-4-5 切換到 claude-sonnet-4-6，Dynamic Filtering 就自動啟用。不需要改任何 Prompt 或架構。

💰 成本複利效應
24% Token 降低會複利累積。假設你的 Agent 每月執行 10,000 次網路搜尋任務，每次平均消耗 5,000 tokens：

舊成本：10,000 × 5,000 tokens = 5,000 萬 tokens
新成本：10,000 × 3,800 tokens = 3,800 萬 tokens
每月節省：1,200 萬 tokens——同等任務，效果更好，成本更低。

✏️ 練習題
Q1 概念理解 Dynamic Filtering 與傳統 Prompt Engineering（如「只回傳相關內容，忽略廣告」）的根本差異是什麼？

Prompt Engineering 是在推理時告訴模型「請忽略無關內容」——但無關內容已經佔用了 Context Window 的 Token 空間，成本已發生。

Dynamic Filtering 是在推理之前讓模型自動撰寫並執行 Python 程式碼，從資料來源就清除噪音。進入推理流程的 Context 本身就更小、更乾淨。這是「預處理」vs「推理時過濾」的根本區別——前者節省 Token，後者不節省。

Q2 產業應用 在 CDMO 的 feasibility assessment 場景中，Dynamic Filtering + 1M Context 會帶來哪些具體的工作流改變？

目前痛點：當 Agent 去搜集客戶藥品資訊（FDA 文件、競品資料、市場報告）時，大量噪音導致 context 爆炸，需要人工手動篩選或分段處理。

改變後：一個 feasibility assessment Agent 可以在單次請求中：(1) 搜尋並自動過濾多個 FDA 數據庫，(2) 將清洗後的資料連同 SOP、設備規格（1M Token）一起推理，(3) 輸出結構化評估報告。這將原本需要 2-3 天的初步評估壓縮到數小時。

注意：輸出仍需要 QA/RA 人員審核——這屬於「Copilot Mode」而非「Agent Mode」，符合 GMP 合規要求。

Q3 批判思考 文章作者說「如果你還在用 Sonnet 4.5，現在就該切換了」。這個建議對所有 Agent 都成立嗎？

不完全成立。以下情況要謹慎：

1. 已驗證的生產 workflow：模型升級需要重新測試。在受 GMP 或 21 CFR Part 11 監管的環境中，模型版本切換屬於重大系統變更，需要驗證文件。不能因為「24% 便宜」就直接切換生產環境。

2. 非搜尋型 Agent：如果你的 Agent 主要做程式生成、文本分類或資料分析，Dynamic Filtering 的優勢不顯著——切換的 ROI 需要重新評估。

3. API 版本鎖定：部分第三方平台可能需要時間更新至 4.6。先確認你的工具鏈支援再行動。
