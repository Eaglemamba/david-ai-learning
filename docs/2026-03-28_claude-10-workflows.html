Executive Summary

一位內容創作者追蹤兩週的 Claude Pro 使用數據，整理出 10 個可重複使用的 Prompt Workflow（提示工作流），涵蓋研究、內容製作、程式碼審查、數據分析等場景。核心概念是「Workflow = 可重複的 Prompt + 固定結構」，將 AI 從一次性問答工具轉化為系統化的生產力引擎。作者宣稱每週節省約 13 小時，但需注意此數據為個人自報、未經第三方驗證，且部分場景的「省時」計算可能過度樂觀（適用 2-3x 折扣原則）。

文章的真正價值不在於具體數字，而在於每個 Workflow 的 Prompt 結構設計模式——特別是「反向指令」（告訴 AI 不要做什麼）和「品質錨定語句」（如 "Be harsh"、"Don't speculate"）的使用技巧，這些可直接套用到任何 AI 工作流設計中。

Content

WF1: Research

作者描述研究任務的傳統痛點：開 15 個分頁、在 Google Doc 中複製貼上引用、注意力分散到 Twitter，兩小時後得到一份「還需要整理」的雜亂文件。這個 Workflow 的核心是將開放式研究轉化為結構化輸出。
"open 15 tabs, skim docs, copy-paste quotes into a Google Doc, try to make sense of it... come back 2 hours later with a messy doc that still needed organizing"
關鍵設計元素：Prompt 末尾加入「If data is insufficient for any claim, say so. Don't speculate. Don't pad.」——這是一個「品質錨定語句」。沒有這條指令，Claude 會自信地編造數字。
"The line that makes this work is 'If data is insufficient, say so.' Without it Claude confidently invents numbers."
比喻說明
品質錨定語句就像 GMP 中的偏差調查 SOP 裡的「不得在無數據支持的情況下做出結論」條款——它不是教 AI 怎麼做，而是設定一條不可跨越的底線，防止 AI 的「幻覺」行為。
Prompt Template
Research [TOPIC].

Structure:
1. Executive summary (3 sentences max)
2. Key findings (top 5, ranked by impact)
3. What's missing: gaps in available info
4. Sources with URLs

If data is insufficient for any claim, say so.
Don't speculate. Don't pad.
Prompt 設計模式
固定輸出結構：4 個編號區塊，強制結構化
反向指令：Don't speculate / Don't pad
品質錨定：承認不足 > 編造數據
省時估計：2hr → 15min（作者宣稱）

WF2: Content Research (多來源整合)

內容創作最耗時的不是寫作本身，而是處理資訊的前置作業——閱讀 5-10 個來源（GitHub README、文件、Twitter 討論串、部落格、更新日誌），手動閱讀、標記重點、交叉比對、萃取關鍵資訊，每篇要花超過一小時。
"reading 5-10 sources: GitHub READMEs, docs, Twitter threads, blog posts, changelog entries"
Prompt 設計的亮點是受眾定義（builders, not consumers）和差異化視角（contradicts common narrative / angle nobody else is covering）。這不只是摘要工具，而是內容策略分析工具。
"If two sources disagree, show me both sides. Don't summarize fluff. Only signal."
Prompt Template
Here are my sources on [TOPIC]:
[PASTE ALL RAW MATERIAL]

Extract:
1. The 5 facts that matter most for my audience 
   (builders, not consumers)
2. Anything that contradicts the common narrative
3. Specific numbers: stars, users, funding, benchmarks
4. One angle nobody else is covering

If two sources disagree, show me both sides.
Don't summarize fluff. Only signal.
實務應用
此模式可直接套用到 CDMO 的 Tech Transfer 文件審查：將多份來源文件（藥品規格書、製程開發報告、穩定性數據）貼入 Claude，要求萃取「最重要的 5 個製程參數差異」和「任何來源之間的矛盾」。

WF3: GitHub Repo Analysis (大量篩選)

作者需要從 1,000+ 個 GitHub repo 中挑選 40 個值得介紹的。手動操作每個 repo 需 3-5 分鐘，總計 50+ 小時。解決方案是批次處理 + AI 預篩選 + 人類驗證的三層架構。
"I still opened every 'Yes' repo myself to verify, but instead of reviewing 1,000 repos I only needed to check around 80."
最有效的篩選條件是「skip anything with no commits in 30+ days」——半數熱門 repo 實際上已經被棄置。這是一個 Heuristic Filter（啟發式篩選器），用簡單規則排除大量噪音。
"Half the hyped repos on trending lists are already abandoned."
比喻說明
這就像 CDMO 供應商資格審查（Vendor Qualification）的初篩階段——先用硬性條件（GMP 認證、產能、地理位置）排除 80% 不合格的供應商，再對剩下的 20% 做深度評估。AI 做初篩，人類做終審。
Prompt Template
Here's a list of GitHub repos with descriptions:
[PASTE BATCH]

For each repo, evaluate:
1. What it actually does (1 sentence, no marketing)
2. Traction signals: stars, recent commits, 
   contributor count
3. Category: agent framework / dev tool / MCP / 
   infrastructure / other
4. Worth featuring? Yes/No with one reason

Skip anything that's a wrapper, tutorial repo, 
or has no commits in 30+ days.
Sort "Yes" picks by most interesting first.
O-Ring 觀察
注意 AI 的角色是預篩選（1,000 → 80），人類仍然做最終判斷（80 → 40）。這是 Copilot Mode 的教科書案例——AI 處理不可逆決策之前的可逆工作。

WF4: Data Analysis (無需試算表)

上傳 CSV 檔案，讓 Claude 直接做趨勢分析、異常偵測、欄位相關性分析。輸出是表格 + 兩段純文字摘要，避免了試算表工具的學習曲線。
"export CSV, open Google Sheets, spend 40 minutes building formulas, realize one was wrong, start over"
Prompt 的巧妙之處在於「If the dataset is too small for a conclusion, say so」——再次使用品質錨定，防止 AI 從不足的數據中硬擠出結論。
Prompt Template
Analyze this data. I need:
1. Top 3 trends over time
2. Anything unusual or unexpected
3. Correlations between [COL A] and [COL B]

Table first, then a 2-paragraph summary explaining
what this means in plain English.
If the dataset is too small for a conclusion, say so.

WF5: Competitor Analysis

競爭者分析的關鍵設計是在 Prompt 中加入「關於我」的上下文。沒有這個上下文，AI 只能產出通用的 SWOT 分析。有了它，AI 能針對你的定位給出差異化建議。
"The key is giving Claude context about YOU, not just the competitor. Otherwise you get a generic SWOT analysis that could apply to anyone."
反向指令「Don't say 'they have a strong brand.' Tell me WHY and what specifically makes it work」——強制 AI 從表面描述進入因果分析。
Prompt Template
I'm analyzing [COMPETITOR/ACCOUNT].

Based on what you know + the data I'm providing:
1. Top 3 things they're doing well (be specific)
2. Gaps or weaknesses in their approach
3. What I can learn from them
4. How my positioning is different

About me: [YOUR CONTEXT]

Don't say "they have a strong brand." Tell me WHY 
and what specifically makes it work.
實務應用
CDMO 競標場景：將競爭對手的公開資訊（產能、技術平台、客戶案例）輸入 Claude，加上自己的「About me」上下文（SA25 isolator 技術、QS-21 經驗），要求分析差異化定位。

WF6: Code Review

作者坦承他大量使用 Vibe Coding（AI 寫大部分程式碼），因此需要在部署前驗證安全性。Prompt 的亮點是嚴重等級分類（Critical/High/Medium/Low）和「Be harsh」品質錨定。
"Without it Claude defaults to polite feedback like 'the code is well-structured, perhaps consider...' With it you get the mean code reviewer energy you actually need."
這揭示了 LLM 的一個重要行為特徵：預設禮貌偏差（Default Politeness Bias）。AI 天生傾向給正面回饋，除非你明確要求嚴格。在需要批判性分析的場景中，必須用指令覆蓋這個預設行為。
比喻說明
就像 GMP 稽核員的角色切換——如果你請同事「看看這份批次紀錄」，他會禮貌地說「看起來不錯」。但如果你說「假裝你是 FDA 稽核員，找出所有不合規的地方」，同樣的人會變得極其嚴格。「Be harsh」就是把 AI 從同事模式切換到稽核員模式。
Prompt Template
Review this code for:
- Security issues (exposed keys, injection, XSS)
- Logic errors and edge cases
- Performance problems
- Anything that would make a senior dev uncomfortable

For each issue: severity (Critical/High/Medium/Low),
exact location, why it matters, corrected code.

Be harsh. "Looks good overall" is not helpful.

[PASTE CODE]
重點提示
注意限制：Claude 的 Code Review 不等於專業安全稽核。它能抓到常見模式（硬編碼 API key、基本 XSS），但無法做動態分析、滲透測試、或複雜的商業邏輯漏洞檢測。適合做「第一道防線」，不能替代專業安全審查。

WF7: Long Content ↔ Short Content

經營多個社群媒體帳號（X、Telegram、Instagram）最痛苦的是為每個平台寫不同版本。此 Workflow 是一對多的內容轉換引擎：一篇長文 → tweet hook、TG 貼文、quote-tweet caption、獨立推文。
"One prompt, 10 minutes of editing. Used to be 2 hours of work."
關鍵設計：「Each piece must work independently. Someone who never read the article should still get value.」——這防止 AI 產出需要原文上下文才能理解的碎片。
作者也提到反向操作：將一週的短文貼文全部輸入，請 Claude 找出連結主題，起草長文大綱。短 → 長同樣有價值。
"I'll take 5-6 short TG posts from the week... and ask Claude to find the connecting thread and draft a long-form article outline."
Prompt Template
Here's my article: [PASTE OR UPLOAD]

Create:
1. A 2-sentence hook for X (include a specific 
   number or claim from the article)
2. A 4-paragraph TG post with the key insight
3. A provocative quote-tweet caption (1 sentence)
4. 3 standalone insights that work as separate 
   tweets throughout the week

Each piece must work independently. Someone who 
never read the article should still get value.

WF8: Email Drafting

寫 email 的痛點是語氣拿捏的認知負荷：太正式像機器人，太隨意不專業。作者的解法是在 Prompt 中明確定義「不要像什麼」。
"Too formal and you sound like a robot. Too casual and you sound unprofessional."
「Does not sound like: a cold pitch template, corporate speak, or something ChatGPT would write. No 'I hope this email finds you well.'」——這是反向指令的經典應用。透過排除不想要的風格，比正面描述想要的風格更有效。
Prompt Template
Draft an email.
To: [NAME + how I know them]
Goal: [WHAT I WANT THEM TO DO]
Tone: professional but sounds like a real person
Max: 5 sentences
Context: [THE SITUATION]

Does not sound like: a cold pitch template, 
corporate speak, or something ChatGPT would write.
No "I hope this email finds you well."
實務應用
在 CDMO 情境中，這個模板可用於客戶溝通的初稿——特別是需要拿捏語氣的情況（如通知客戶偏差調查結果、請求延長時程）。但最終版本必須經過人類審閱，確保技術準確性和法規合規性。

WF9: Morning Briefing

用 Claude 取代每天早上 45 分鐘的社群媒體滑動。核心價值是訊號與噪音分離——社群媒體把新聞混雜著戲劇性內容、meme、釣魚式參與誘餌和 20 分鐘的兔子洞。
"Claude gives you signal without noise."
作者誠實承認限制：準確率約 80%，特別容易漏掉最近一小時的突發新聞。剩下 20% 靠群組聊天和通知自然補上。這個「80% 自動化 + 20% 人類補完」的比例很有參考價值。
"Is it perfect? No. It misses things sometimes, especially breaking news from the last hour. But it catches maybe 80% of what matters."
Prompt Template
3-minute briefing:
1. Top 3 AI news from last 24 hours (one sentence each)
2. Crypto: major moves, liquidations, new narratives
3. Anything I should know before posting content today

Be specific: names, numbers, links.
Skip anything that isn't genuinely important.
3 real updates > 10 filler items.
重點提示
限制注意：Claude 的 web search 有延遲和涵蓋範圍限制，不適合作為即時新聞來源。此 Workflow 更適合作為「背景知識更新」而非「即時警報系統」。在 CDMO 場景中，法規更新追蹤需要更可靠的來源（如 PDA/ISPE Newsletter）。

WF10: Weekly Review (最高 ROI)

每週日晚上將整週的筆記、書籤、半成形的想法全部倒入 Claude。作者認為這是AI 從工具轉變為思考夥伴的轉折點——它能看到你自己因為太靠近而錯過的想法之間的連結。
"This is where Claude stops being a tool and starts being a thinking partner. It sees connections across your own ideas that you miss because you're too close to your own work."
品質錨定：「Be honest. If an idea is weak, say so. Don't tell me everything is great.」——再次覆蓋 AI 的預設禮貌偏差。
Prompt Template
Here are my notes and ideas from this week:
[PASTE EVERYTHING]

Help me:
1. Find patterns: what topics am I gravitating toward?
2. Which 3 ideas have the most content potential?
3. What am I ignoring that I shouldn't be?
4. Content plan for next week: 3 TG posts + 1 article

Be honest. If an idea is weak, say so.
Don't tell me everything is great.
實務應用
這個模式可套用到個人學習系統：每週日將本週讀的文章、做的筆記、產出的教育文件全部列出，請 Claude 找出「我正在忽略哪個學習維度」和「下週最應該深入的 2 個主題」。

Key Takeaways

Workflow = System, Not Prompt
Workflow 的定義是「可重複的 Prompt + 固定結構 + 可預測的輸出」。一次性提問是工具使用，Workflow 是系統設計。真正的價值在於從 ad-hoc 使用轉向系統化生產力。
反向指令 > 正向指令
告訴 AI「不要做什麼」（Don't speculate / Don't pad / No corporate speak）比描述「要做什麼」更有效。這覆蓋了 LLM 的預設禮貌偏差和填充行為，是 Prompt Engineering 的高槓桿技巧。
Context About You 是差異化關鍵
在 Prompt 中加入「關於我」的上下文（受眾定義、角色定位、專業背景），將 AI 輸出從通用建議轉化為個人化洞察。這正是 Claude Projects 的 System Prompt 和 Memory 功能設計的核心理念。
AI 做篩選，人類做判斷
最有效的 Workflow 遵循「AI 預處理 + 人類終審」模式（1,000 repos → 80 候選 → 40 精選）。這是 Copilot Mode 的本質——AI 加速可逆工作，人類保留不可逆決策。

Practice

Q1 概念理解 作者在多個 Workflow 中反覆使用的「品質錨定語句」（如 "If data is insufficient, say so"）解決了 LLM 的什麼行為特徵？這與 GMP 中的哪個概念最接近？

品質錨定語句解決的是 LLM 的幻覺傾向（Hallucination）和預設禮貌偏差（Default Politeness Bias）。LLM 傾向在資訊不足時仍然給出看似自信的回答，而非承認不確定性。品質錨定語句強制模型在證據不足時停下來。

這與 GMP 中的數據完整性原則（ALCOA+ 中的 Attributable 和 Complete）最為接近——你不能在沒有數據支持的情況下簽署批次放行。SOP 中的「不得在無數據支持的情況下做出結論」就是品質錨定語句的 GMP 版本。

Q2 產業應用 將 WF3（GitHub Repo 大量篩選）的「批次處理 + AI 預篩選 + 人類驗證」架構套用到 CDMO 的供應商資格審查流程中。你會如何設計 AI 的初篩條件？人類終審應該關注什麼？

AI 初篩條件（對應 "skip anything with no commits in 30+ days"）：GMP 認證狀態（無 GMP = 自動排除）、產能是否符合需求範圍、地理位置是否在可接受的供應鏈半徑內、近 2 年是否有 FDA Warning Letter 或 Import Alert。

人類終審（對應「仍然親自打開每個 Yes repo 驗證」）：技術能力的深度評估（不只是「有無」aseptic 經驗，而是具體的產品類型和批量規模）、供應商的客戶參考查核、合約條款和商業條件的判斷、文化適配度和溝通品質。

O-Ring 原則：AI 處理結構化資訊的篩選（可逆，錯過了可以重跑），人類處理需要判斷力的評估（不可逆，選錯供應商影響巨大）。

Q3 批判思考 作者宣稱這些 Workflow 每週省 13 小時。你認為這個數字的主要偏誤來源是什麼？如果你要設計一個更嚴謹的「AI 省時效益追蹤」方法，你會怎麼做？

主要偏誤：(1) 自報偏差——「Before」時間很可能被高估（人傾向記住最慢的那次，而非平均值）；(2) 品質等價假設——省時計算假設 AI 輸出品質等於人類輸出，但多數場景需要額外編輯時間未被計入；(3) 頻率樂觀——假設每週都用滿全部 10 個 Workflow，實際上可能只用 5-6 個。

更嚴謹的追蹤方法：(1) 用計時工具（如 Toggl）而非估算記錄 Before/After；(2) 將「AI 產出 + 人類編輯」的總時間計為 After，而非只計 AI 生成時間；(3) 追蹤 2-3 個月取平均值，避免新鮮感效應（Novelty Effect）；(4) 加入品質評估——省時但品質下降 30% 不算真正的效率提升。

My Bottom Line
Core Insight
這篇文章的核心價值不在 10 個具體 Prompt（這些可以自己寫），而在於兩個 Prompt Engineering 模式的反覆驗證：(1) 反向指令比正向指令更有效控制 LLM 行為，(2) 品質錨定語句是防止 AI 幻覺的最簡單方法。這兩個模式直接適用到我的 Claude Code 自動化管線中——每個 repo 的 prompt file 都應該包含至少一條品質錨定語句和一條反向指令。

本週審查我的 repo-registry.conf 中所有 prompt files，檢查是否都包含品質錨定語句（如「如果資訊不足以支持結論，明確說明」）。特別是 daily digest 和 COO tracking 的 prompt——這兩個最容易產生「看起來有道理但實際是幻覺」的輸出。同時，將 WF10（Weekly Review）的模式加入我的週日學習復盤流程中。
Gap to Fill
