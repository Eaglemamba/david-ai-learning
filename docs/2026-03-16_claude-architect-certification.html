Part 0: Exam Overview & Learning Path
這篇文章的前提很直白：Claude Certified Architect 認證考試僅開放給 Claude Partner，一般開發者無法參加。但作者認為這不重要——重要的是掌握考試涵蓋的知識，因為這些正是建構 production-grade AI 應用所需的技能。
"You don't need the certificate to build production-grade applications. You just need the knowledge."
考試涵蓋六大 scenario-based 情境，每題有一個正確答案和三個高度合理的干擾選項。及格分數 720/1000。考試偏好確定性解決方案（programmatic enforcement）勝過概率性方案（prompt-based guidance），尤其在高風險場景。
"The exam consistently rewards deterministic solutions over probabilistic ones when stakes are high, proportionate fixes, and root cause tracing."
⚖️ 六大考試情境
1. Customer Support Resolution Agent (Agent SDK + MCP + escalation)
2. Code Generation with Claude Code (CLAUDE.md + plan mode + slash commands)
3. Multi-Agent Research System (coordinator-subagent orchestration)
4. Developer Productivity Tools (built-in tools + MCP servers)
5. Claude Code for CI/CD (non-interactive pipelines + structured output)
6. Structured Data Extraction (JSON schemas + tool_use + validation loops)
⚠️ 重點提示
考試設計哲學：錯誤選項聽起來像好的工程實踐，正確答案需要精確知道哪個技術對應哪個問題。

學習策略：作者為每個 Domain 提供了一個完整的 study prompt，可以直接貼進 Claude 開始結構化學習。這是「用 AI 學 AI」的自舉學習法。

Domain 1: Agentic Architecture & Orchestration 27%

1.1 Agentic Loops — stop_reason 是唯一的真理
Agentic loop 的生命週期很簡單：送 request 給 Claude → 檢查 stop_reason → 如果是 "tool_use" 就執行工具、把結果附回對話歷史、再送回 Claude → 如果是 "end_turn" 就結束。工具結果必須 append 到對話歷史，模型才能在下一輪推理新資訊。
"Tool results must be appended to conversation history so the model can reason about new information on the next iteration."
考試測試三個你必須一眼辨認的反模式：
❌ 反模式 1：解析自然語言判斷迴圈結束（例如檢查助手是否說了「I'm done」）。自然語言模糊不可靠，stop_reason 才是正解。
❌ 反模式 2：任意迭代次數上限作為主要停止機制（例如「跑 10 次就停」）。要嘛截斷有用工作，要嘛跑不必要的迭代。
❌ 反模式 3：檢查助手回覆是否包含 text 作為完成指標。模型可以同時回傳 text 和 tool_use blocks。
📚 核心概念
Model-driven vs Pre-configured：
考試偏好 model-driven decision-making（Claude 根據 context 決定呼叫哪個 tool），而非預設的 decision tree。

但例外：當涉及財務或安全關鍵邏輯時，必須用 programmatic enforcement（hooks + prerequisite gates），不能只靠 prompt。
⚠️ 考試陷阱
開發者的 agent 有時過早終止，因為他們檢查 response.content[0].type == "text" 來判斷完成。Bug：模型可能在 text block 旁邊同時回傳 tool_use block。必須檢查 stop_reason，而非 content type。
1.2-1.3 Multi-Agent Orchestration — 子代理隔離原則
最大的誤解：人們假設子代理（subagents）與 coordinator 共享記憶。他們不共享。子代理有獨立的 context。每一筆資訊都必須在 prompt 中明確傳遞。
"Subagents operate with isolated context. Every piece of information must be passed explicitly in the prompt."
Hub-and-spoke 架構中，coordinator 負責：任務分解、選擇子代理、傳遞 context、彙總結果、錯誤處理。所有通訊都經過 coordinator，子代理之間永不直接通訊。
Context 傳遞的關鍵：使用結構化資料格式分離 content 和 metadata（source URLs、document names、page numbers），以保留跨 agent 的 attribution。
Narrow Decomposition Failure：考試中有題目是 coordinator 把「AI 對創意產業的影響」只分解成視覺藝術子主題，遺漏了音樂、寫作、電影。根本原因是 coordinator 的分解邏輯，不是下游 agent。考試期望你追溯失敗到根源。
💻 程式碼/API
Task tool：coordinator 用來生成子代理的機制。coordinator 的 allowedTools 必須包含 "Task"。

fork_session：從共享分析基線建立獨立分支，用於探索不同方法（如比較兩種測試策略）。

Parallel spawning：coordinator 在單一 response 中發出多個 Task tool calls，平行生成子代理，比逐步調用更快。
💡 CDMO 類比
Multi-agent orchestration 就像 CDMO 的 tech transfer：coordinator 是 Project Manager，subagents 是各部門（QA、QC、Production、RA）。PM 不會假設 QC 知道 Production 的 batch record 細節——每次 handoff 都要有明確的 tech transfer protocol，附帶完整文件。子代理記憶隔離 = 部門間沒有共享文件系統。
1.4-1.5 Workflow Enforcement & Hooks — 確定性 vs 概率性
考試的決策規則很明確：當後果涉及財務、安全、合規時，必須使用 programmatic enforcement（hooks / prerequisite gates）。低風險場景（格式偏好、風格指南）用 prompt-based guidance 即可。
"When consequences are financial, security-related, or compliance-related: use programmatic enforcement."
✅ 正確模式：PostToolUse hook 在工具執行後、模型處理結果前攔截，標準化異質資料格式（Unix timestamp → ISO 8601）。Tool call interception hook 在執行前攔截，阻擋超過 $500 的退款並轉向人工。
❌ 反模式：用 enhanced system prompt 或 few-shot examples 來確保退款前驗證帳戶所有權。8% 的失敗率 = 真實的財務損失。
🛠️ 實務應用
Decision Framework：
Hooks = 確定性保證（100% 遵守）
Prompts = 概率性引導（大多數時候有效）

判斷標準：如果單次失敗就會導致公司損失金錢或面臨法律風險，用 hooks。這與 GMP 環境中「critical process parameter 必須有 hard interlock，不能只靠 SOP」的邏輯完全一致。
1.6-1.7 Task Decomposition & Session Management
兩種分解模式：Fixed sequential pipeline（預定步驟，適合結構化任務如 code review）vs Dynamic adaptive decomposition（根據發現生成子任務，適合開放式調查）。
Attention Dilution 問題：單次處理 14 個檔案會導致深度不一致，某些檔案的明顯 bug 被遺漏。解法：多 pass 架構——per-file local analysis + separate cross-file integration pass。
"Processing too many files in a single pass produces inconsistent depth."
Session 管理三選項：--resume（繼續 session）、fork_session（分支探索）、fresh start with summary injection（context 過時時重新開始）。當程式碼已修改，resume 會帶著過時的 tool results，造成矛盾建議。
🔨 Domain 1 建構練習
建構一個 coordinator agent + 2 個 subagents（web search + document analysis），實作：proper context passing with structured metadata、programmatic prerequisite gate、PostToolUse normalisation hook。用 multi-concern request 測試。

進階：加入 3-4 個 MCP tools、proper stop_reason handling、tool call interception hook 阻擋 policy violations。

Domain 2: Tool Design & MCP Integration 18%

2.1 Tool Interface Design — 描述決定一切
Tool descriptions 是 Claude 選擇工具的主要機制。不是輔助功能，就是「那個」機制。如果描述模糊或重疊，選擇就會不可靠。
"Tool descriptions are the primary mechanism Claude uses for tool selection."
考試的 Q2 情境：get_customer 和 lookup_order 的描述幾乎一樣，導致持續 misrouting。正確修正不是 few-shot examples、不是 routing classifier、不是 tool consolidation——而是改善 descriptions。好的描述包含：工具用途、輸入格式、適用查詢範例、邊界情況、與相似工具的明確區分。
Tool 拆分原則：把 analyze_document 這種通用工具拆成 extract_data_points、summarize_content、verify_claim_against_source，每個有明確的 input/output contract。
⚠️ 重點提示
Tool overload 問題：給 agent 18 個 tools 會降低選擇可靠度。最佳做法：每個 agent 4-5 個 tools，scope 到其角色。Synthesis agent 不該有 web search tools，web search agent 不該有 document analysis tools。
💻 tool_choice 三模式
"auto"：模型決定是否呼叫工具。預設值。
"any"：必須呼叫工具，自選哪個。用於保證結構化輸出。
{"type":"tool","name":"..."}：必須呼叫指定工具。用於強制第一步。
2.2-2.3 Structured Errors & Tool Distribution
四種錯誤類型：Transient（暫時性，可重試）、Validation（輸入無效，修正後重試）、Business（政策違規，不可重試，需替代流程）、Permission（權限不足，需 escalation）。
關鍵區分：Access failure vs Valid empty result。前者是工具無法連線（timeout、auth failure），可考慮重試。後者是工具成功查詢但沒有匹配——不該重試，這就是答案。搞混這兩者會破壞 recovery logic。
"Confusing these two breaks recovery logic. The exam tests this."
Scoped cross-role tools：對高頻簡單操作，給 agent 一個受限工具直接使用，避免每次都要 round-trip 回 coordinator。例如 synthesis agent 有 scoped verify_fact 做簡單查詢，複雜查證才走 coordinator。考試 Q9 測的就是這個模式。
🛠️ MCP Server 整合
Scoping 層級：
Project-level: .mcp.json（版控、團隊共享）
User-level: ~/.claude.json（個人、不共享）

環境變數：${GITHUB_TOKEN} 語法，credentials 不進版控。

Build vs Use 決策：標準整合（Jira, GitHub, Slack）用 community MCP servers。只在團隊特有流程時才 build custom server。
💡 CDMO 類比
Tool description 就像 SOP 的「範圍 (Scope)」章節。如果兩份 SOP 的範圍描述幾乎一樣，operator 就不知道何時用哪份。同理，API 層級的 error category 就像 deviation classification（Critical / Major / Minor），每個類別有不同的 CAPA 流程。
🔨 Domain 2 建構練習
建立 3 個 MCP tools（其中 2 個故意描述模糊），寫 4 種 error category 的 responses，在 .mcp.json 中用環境變數配置。先體驗 misrouting，再修正描述，感受差異。

Domain 3: Claude Code Configuration & Workflows 20%

3.1-3.2 CLAUDE.md Hierarchy & Custom Commands
CLAUDE.md 有三個層級，層級決定誰能看到指令：
User-level ~/.claude/CLAUDE.md：只有你自己。不受版控、不共享。新團隊成員 clone repo 不會得到這些指令。
Project-level .claude/CLAUDE.md：所有人。版控、共享。團隊標準放這裡。
Directory-level（子目錄 CLAUDE.md）：在該目錄工作時才適用。
❌ 考試最愛的陷阱：新團隊成員沒收到指令。根本原因：指令在 user-level config 而非 project-level。
Commands vs Skills 的區分：
.claude/commands/ = project-scoped，版控共享
~/.claude/commands/ = 個人，不共享
.claude/skills/ + SKILL.md = on-demand invocation，有 frontmatter 配置

Skills 是任務特定的 workflow（需要時調用），CLAUDE.md 是始終載入的通用標準。不要把任務流程放在 CLAUDE.md，不要把通用標準放在 skills。
💻 Skill Frontmatter
context: fork — 在隔離子代理中執行，verbose 輸出不影響主對話

allowed-tools — 限制 skill 可用的工具，防止破壞性操作

argument-hint — 無參數調用時提示開發者輸入必要參數
📚 Path-Specific Rules (3.3)
.claude/rules/ 目錄下的檔案可帶 YAML frontmatter glob pattern：

paths: ["**/*.test.tsx"]

Glob pattern 匹配整個 codebase 的檔案，不像 directory-level CLAUDE.md 只限單一目錄。50+ 個目錄的 test 規範用一條 rule 搞定。且 path-scoped rules 只在編輯匹配檔案時載入，節省 token。
3.4-3.6 Plan Mode, Iterative Refinement & CI/CD
Plan mode 決策框架：
Plan mode：monolith 重構、多檔案 migration、架構決策
Direct execution：單檔 bug fix、一個 validation check、scope 明確的修改
混合模式最常見：plan mode 做調查設計，direct execution 做實作。
Iterative Refinement 技術層級：
1. Concrete input/output examples（2-3 個 before/after 範例）打敗 prose descriptions
2. Test-driven iteration：先寫測試，分享 failure 引導改善
3. Interview pattern：讓 Claude 先問問題再實作
CI/CD 整合重點：
✅ -p flag：非互動模式（print mode）。沒有這個 flag，CI job 會掛起等待互動輸入。考試 Q10 就考這個。
同一 session 產生的程式碼，由獨立 instance 來 review 效果更好——同 session review 會保留推理 context，較不容易質疑自己的決策。
"The same Claude session that generated code is LESS effective at reviewing its own changes."
⚠️ CI/CD 實務要點
--output-format json + --json-schema：產生機器可解析的結構化輸出

Incremental review：新 commit 後重跑 review 時，帶入先前 review findings，指示只報告新的或未解決的問題。避免重複 comments 侵蝕開發者信任。

CLAUDE.md for CI：記錄 testing standards、valuable test criteria、available fixtures，否則 test generation 只會產出低價值 boilerplate。
🔨 Domain 3 建構練習
建立專案：CLAUDE.md hierarchy（project + directory level）、.claude/rules/ 含 glob patterns（test files + API files）、custom skill 用 context: fork、CI script 用 -p flag + JSON output。測試 plan mode 做 multi-file refactor，direct execution 做 single bug fix。

Domain 4: Prompt Engineering & Structured Output 20%

4.1-4.2 Explicit Criteria & Few-Shot Prompting
整個 Domain 4 的核心兩個字：be explicit。「Be conservative」不會改善精確度。「Only report high-confidence findings」不會降低 false positives。
❌ 無效指令："Be conservative." / "Only report high-confidence findings."
✅ 有效指令："Flag comments only when claimed behaviour contradicts actual code behaviour. Report bugs and security vulnerabilities. Skip minor style preferences and local patterns."
False positive trust 問題：某一類別的高 false positive rate 會摧毀所有類別的信任。解法：暫時停用高 false positive 的類別，邊改善 prompt 邊恢復。
Few-shot examples 是最高槓桿技術：2-4 個針對 ambiguous case 的範例，每個展示推理過程（為什麼選擇 A 而非 B）。這教的是泛化能力，不只是 pattern matching。
"Few-shot examples are the highest-leverage technique tested."
📚 Severity Calibration
不要用 prose 描述 severity。用實際程式碼範例展示 "critical" vs "minor" 的差異。具體的程式碼比抽象的文字說明有效 10 倍。
💡 CDMO 類比
這和 GMP 偏差調查的道理一模一樣。「嚴格調查」是空話。有效的 SOP 會寫：「當 particulate count > 10 particles/mL at >= 25um，classify as Major deviation，啟動 CAPA。當 particulate count 介於 warning limit 到 action limit 之間，記錄為 Minor deviation，trend monitoring。」明確的閾值 + 具體範例 = 可執行。
4.3-4.6 Structured Output, Batch API & Multi-Instance Review
tool_use + JSON schema 消除語法錯誤，但不防止語義錯誤（line items 加總不等於 stated total）、field placement 錯誤、fabrication。Schema 設計原則：source 可能沒有的資訊用 nullable fields（防止 fabrication）、模糊情況用 "unclear" enum、可擴展分類用 "other" + detail string。
Validation-Retry Loop：把原始文件 + 失敗的 extraction + 具體 validation error 一起送回。對 format mismatch 有效，對 source 中根本不存在的資訊無效。考試會同時出現這兩種場景，你必須辨別哪個可修復。
Message Batches API：50% 成本節省、最長 24 小時處理、無 latency SLA、不支援 multi-turn tool calling。
✅ Synchronous API：blocking workflows（pre-merge checks，開發者在等）
✅ Batch API：latency-tolerant workflows（overnight reports、weekly audits、nightly test generation）
⚠️ Multi-Instance Review
同一 session 的 self-review 保留推理 context，較不會質疑自己的決策。獨立 instance（無先前 context）能抓到更多 subtle issues。

Multi-pass 架構：per-file local analysis + separate cross-file integration pass，避免 attention dilution。

Confidence-based routing：模型自報 confidence，低信心項目轉人工 review。用 labelled validation set 校準閾值。
🔨 Domain 4 建構練習
建立 extraction tool（JSON schema 含 required, optional, nullable fields, enums with 'other'）。實作 validation-retry loop。用 Batches API 批次處理 10 份文件。加 few-shot examples 處理不同文件格式，比較 before/after extraction quality。

Domain 5: Context Management & Reliability 15%

5.1-5.2 Context Preservation & Escalation
Progressive Summarisation 陷阱：壓縮對話歷史會把「客戶要求退款 $247.83，訂單 #8891，3月3日」變成「客戶想退款」。數字、日期、百分比、客戶期望全消失。
✅ 解法：把交易事實提取到 persistent "case facts" block，每次 prompt 都帶入，永不壓縮。
"Lost in the middle" 效應：模型可靠處理長輸入的開頭和結尾，中間的 findings 可能被遺漏。解法：關鍵摘要放開頭，全文用明確 section headers。
三個有效的 escalation triggers：
1. 客戶明確要求人工（立即執行，不要先嘗試解決）
2. 政策例外或空白（請求超出文件化政策）
3. 無法取得實質進展
❌ 兩個不可靠的 triggers：Sentiment-based escalation（挫折感 ≠ 案件複雜度）、Self-reported confidence scores（模型在困難案件上常過度自信，簡單案件上反而不確定）。
🛠️ 實務應用
Tool result trimming：Order lookup 回傳 40+ 欄位，你只需要 5 個。在 append 到 context 前先裁剪，防止 token 預算被不相關資料耗盡。

Upstream agent 最佳化：修改 agent 回傳結構化資料（key facts + citations + relevance scores）而非 verbose content + reasoning chains。
💡 CDMO 類比
Persistent case facts block = Batch Record 的 Master Data Section。不管 batch 進行到哪一步，Product Code、Lot Number、Expiry Date 這些 critical identifiers 永遠保留在 batch record 的固定位置，不會因為中間步驟的註解而被「壓縮」掉。
5.3-5.6 Error Propagation, Exploration & Provenance
Error propagation 反模式：
❌ Silent suppression：回傳空結果標記為成功。阻止任何 recovery。
❌ Workflow termination：單一失敗就殺掉整個 pipeline。丟棄 partial results。
✅ 正確做法：結構化 error context（failure type、attempted query、partial results、alternative approaches）。子代理處理局部 transient failures，只 propagate 無法解決的。
Context degradation 與 Codebase exploration：長 session 後，模型開始引用「典型模式」而非先前發現的具體 class。緩解策略：Scratchpad files（寫 findings 到檔案）、subagent delegation、summary injection、/compact。
Information Provenance：每個 finding 應有 claim + source URL + document name + excerpt + publication date。衝突來源不要任意選一個——標註兩個值和 source attribution，讓 consumer 決定。不同日期解釋不同數字（不是矛盾）。
⚠️ Human Review & Confidence
Aggregate metrics 陷阱：97% overall accuracy 可能隱藏某文件類型 40% 的 error rate。永遠按文件類型 AND 欄位分別驗證。

Stratified random sampling：對高信心 extraction 也抽樣驗證，偵測 novel error patterns。

Field-level confidence calibration：模型輸出每欄位信心值，用 labelled validation set 校準閾值，低信心轉人工 review。
🔨 Domain 5 建構練習
建構 coordinator + 2 subagents。實作 persistent case facts block。模擬 timeout 並驗證 structured error propagation。用衝突來源測試，確認 synthesis 保留 attribution。

Key Takeaways

🤖 Deterministic > Probabilistic
高風險場景（財務、安全、合規）必須用 programmatic enforcement（hooks / gates），不能只靠 prompt。這是整份考試最一致的設計哲學，也是 production AI 系統與 demo AI 系統的根本分界線。
🔧 Tool Description 就是一切
Claude 選擇工具的唯一機制是 tool description。描述模糊 = misrouting。修正 description 是第一優先步驟，在 routing classifier 或 few-shot examples 之前。每個 agent 限 4-5 tools。
⚙ Configuration 必須 Team-Visible
CLAUDE.md 的三層 hierarchy 是考試高頻考點。指令放 user-level = 新成員看不到。Path-specific rules 用 glob pattern 跨目錄生效，比 directory-level CLAUDE.md 更精確且省 token。
🧠 Context 是有限資源
Progressive summarisation 殺死交易資料。Tool results 要 trim 再 append。長 session 的 context degradation 需要 scratchpad files 或 /compact 緩解。Independent review instance 比 self-review 更有效。

Practice Questions

Q1 概念理解 一個 customer support agent 在 8% 的案例中未驗證帳戶所有權就處理退款。最佳修正方案是什麼？

正確答案：Programmatic prerequisite gate（hooks），在退款工具執行前強制要求帳戶驗證完成。

為什麼不是其他選項：

Enhanced system prompt — 概率性解決方案，8% 失敗率意味著真實財務損失，不可接受。

Few-shot examples — 高 token overhead，而且根本原因不是模型不理解（它理解），是偶爾跳過步驟。

Routing classifier — 過度工程化，問題不在路由而在流程控制。

考試原則：財務/安全後果 = 確定性 enforcement，不接受概率性方案。

Q2 產業應用 你在設計 CDMO 的 deviation triage agent。如何確保 Critical deviation 一定經過 QA Head review，而不只是「偶爾」經過？

套用 Domain 1 (Task 1.4) 的 enforcement spectrum：

GMP 環境中，deviation classification 的後果是合規性的（regulatory consequence），屬於 Domain 1 定義的「必須 programmatic enforcement」場景。

實作：Tool call interception hook 在 deviation 工具回傳 classification = "Critical" 時，攔截後續 routing，強制加入 QA Head review step。不能只在 system prompt 寫「Critical deviations 需 QA Head 簽核」——這是概率性指令，在 GMP 環境中風險不可接受。

這完美對應 O-Ring 理論：regulatory judgment nodes 不可被 AI 自主跳過。

Q3 批判思考 作者說「你不需要認證就能建構 production-grade 應用」。這個觀點有什麼限制？認證的價值除了知識本身還有什麼？

作者觀點的限制：

1. Signal value：認證在勞動市場中是一個 credible signal，告訴雇主你通過了標準化評估。自學相同知識但缺乏 signal 時，需要用 portfolio / build projects 來替代。

2. Structured assessment gap：自學者缺乏「你是否真正掌握了」的外部驗證機制。考試的 scenario-based MCQ 會暴露你以為理解但實際上混淆的概念（如 access failure vs valid empty result）。

3. Partner network access：認證通常附帶 partner ecosystem 的 networking 價值，這不是知識可以替代的。

反面觀點：在 AI 領域變化極快的今天，能實際 build 的人比能考試的人更有市場價值。認證的知識有效期可能只有 6-12 個月。
