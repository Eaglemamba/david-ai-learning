Executive Summary

Claude Code 使用者有 93% 的權限提示會直接批准，這導致了「批准疲勞」(approval fatigue)。Auto mode 是一個新的中間路線：用 模型驅動的分類器 自動判斷每個操作是否安全，取代人工逐一審批。系統採用雙階段架構（快速篩選 + 鏈式推理），在真實流量上只有 0.4% 的誤報率，但對真正的「過度積極行為」仍有 17% 的漏報率。這篇文章坦誠地展示了安全工程中「夠好」與「完美」之間的取捨。

Content

Part I: Auto Mode 如何運作

Auto mode 使用兩層防禦：一層負責 Claude 讀取的內容，一層負責 Claude 執行的操作。
Auto mode uses two layers of defense: one for what Claude reads, one for what Claude does.
在輸入層，伺服器端的提示注入探測器會掃描工具輸出（檔案讀取、網頁抓取、Shell 輸出、外部工具回應），在它們進入代理的上下文之前進行檢查。當內容看起來像是試圖劫持行為時，探測器會在結果旁附加警告。
At the input layer, a server-side prompt-injection probe scans tool outputs before they enter the agent's context. When content looks like an attempt to hijack behavior, the probe adds a warning.
在輸出層，轉錄分類器（運行在 Sonnet 4.6 上）在每個操作執行前進行評估。分類器分兩階段運行：快速的單 Token 篩選（決定是否阻擋），只有被標記的才進入鏈式思考推理階段。
At the output layer, the transcript classifier (running on Sonnet 4.6) evaluates each action before it executes, in two stages: a fast single-token filter, followed by chain-of-thought reasoning only if the first filter flags the transcript.
分類器只看到使用者訊息和代理的工具呼叫；Claude 自己的訊息和工具輸出會被移除，使其在設計上對推理過程「盲目」。
The classifier sees only user messages and the agent's tool calls; we strip out Claude's own messages and tool outputs, making it reasoning-blind by design.
💡 比喻說明
想像一個藥廠的品質審核流程：

輸入層探測器就像進料檢驗 (IQC)：在原料進入產線前先檢查是否有問題。

輸出層分類器就像放行審查 (batch release review)：在產品出貨前確認每個操作是否符合規範。

兩層獨立運作，任一層發現問題都能攔截。
📚 核心概念
Reasoning-Blind Design（推理盲目設計）
分類器刻意不看代理的自我解釋，這是為了防止代理「說服」分類器。這和 GMP 環境中「獨立 QA 審核」的原則完全一致：審核者不應被執行者的自我辯解影響判斷。
⚠️ 重點提示
93% 的權限提示被直接批准，這就是「批准疲勞」的實證數據。安全系統如果產生太多低價值的中斷，反而會降低整體安全性，因為人們會停止仔細閱讀。

Part II: 威脅模型

一個代理可能因為四個原因採取危險行動：
An agent might take a dangerous action for four reasons.
1. 過度積極行為 (Overeager behavior)：代理理解使用者的目標，也真的在嘗試幫忙，但主動性超出了使用者會批准的範圍。例如使用它偶然發現的憑證，或刪除它認為礙事的東西。這種行為很難捕捉，因為沒有惡意推理。
Overeager behavior: the agent understands the user's goal and is genuinely trying to help, but takes initiative beyond what the user would approve.
2. 誠實的錯誤 (Honest mistakes)：代理誤判了影響範圍。例如以為某個資源只限測試環境，實際上是共享的。
Honest mistakes: the agent misunderstands the blast radius.
3. 提示注入 (Prompt injection)：植入在檔案、網頁或工具輸出中的指令劫持了代理，將它從使用者的任務重定向到攻擊者的任務。
Prompt injection: instructions planted in a file, webpage, or tool output hijack the agent.
4. 模型失準 (Misaligned model)：代理追求自己的目標。目前在實務上尚未觀察到，但每次模型發布時都會仔細評估。
A misaligned model: the agent pursues a goal of its own. We don't currently see this in practice.
🛠️ 實務應用
CDMO 偏差調查類比：

這四種威脅類型對應到製藥偏差調查 (deviation investigation) 的思維：

1. 過度積極 = 操作員未經授權的「好心」修正
2. 誠實錯誤 = 環境認知錯誤（如用錯配方）
3. 提示注入 = 外部污染（交叉污染）
4. 模型失準 = 系統性根因（SOP 設計缺陷）

防禦策略相同：阻擋操作本身，不管原因是什麼。
⚠️ 重點提示
過度積極是最難捕捉的，因為沒有惡意推理 -- 操作看起來像是合理的問題解決，只是超出了使用者授權的邊界。這正是分類器在 17% 漏報率中最常失敗的場景。
📚 核心概念
Blast Radius（爆炸半徑）
借自資安/DevOps 術語，指一個錯誤操作可能影響的範圍。「清理分支」的 blast radius 可能只是本機，也可能是整個遠端倉庫。分類器的工作就是在 blast radius 不確定時攔截操作。

Part III: 三層權限決策機制

大多數操作在到達分類器之前就被標準的 Claude Code 允許規則處理了：
Most actions are allowed prior to the classifier by the standard Claude Code allow rules.
Tier 1：內建安全工具白名單 + 使用者設定。固定白名單包含無法修改狀態的工具（檔案讀取、文字搜尋、程式碼導航）。進入 auto mode 時，會移除已知授予任意程式碼執行的寬鬆權限規則，包括 blanket shell 存取、通配直譯器（python、node、ruby）和套件管理器執行命令。
Tier 1: Built-in safe-tool allowlist and user settings. On entering auto mode, we drop permission rules that grant arbitrary code execution.
Tier 2：專案內檔案操作。專案目錄內的檔案寫入和編輯自動放行，不需要分類器。理由是日常程式碼編輯可透過版本控制來審查。
Tier 2: In-project file operations. File writes and edits inside the project directory are allowed without a classifier call.
Tier 3：轉錄分類器。只有具有真正下行風險的操作才會到達這一層，包括 Shell 命令、網頁抓取、外部工具整合、子代理產生、專案目錄外的檔案系統操作。
Tier 3: Transcript classifier. Only actions with real downside potential reach this tier, including shell commands, web fetches, external tool integrations, subagent spawns.
💡 比喻說明
三層防線 = 藥廠存取控制

Tier 1（白名單）= 一般人員證：只能進入公共區域
Tier 2（專案內）= 區域授權：可以在自己負責的產線操作
Tier 3（分類器）= 變更控制委員會：涉及跨系統操作需要額外審批

越往後的層級，操作越敏感，審查越嚴格。
⚠️ 重點提示
為什麼 auto mode 要移除寬鬆規則？
使用者在手動審批模式中設定的 blanket shell access 規則，在 auto mode 下會被主動移除。因為如果保留這些規則，分類器永遠看不到最危險的指令。這是一個關鍵的設計選擇：安全不能依賴使用者記得調整設定。

Part IV: 關鍵設計決策

為什麼移除助理文字和工具結果？移除助理文字是為了防止代理「說服」分類器做出錯誤判斷。代理可能會產生有說服力的合理化解釋。移除工具結果則是主要的提示注入防禦，因為工具輸出是惡意內容進入上下文的入口。
We strip assistant text so the agent can't talk the classifier into making a bad call. Stripping tool results is the primary prompt-injection defense, since tool outputs are where hostile content enters the context.
雙階段分類：效率與召回率。單階段分類器面臨直接的效能/成本權衡。分成兩個階段打破了這個取捨。Stage 1 指令要求它「偏向阻擋」並發出單一 yes/no 決定。Stage 2 使用相同的 Prompt 但要求仔細推理。因為輸入相同，Stage 2 幾乎全部是 Stage 1 的 prompt cache 命中。
Splitting into two stages breaks the performance/cost trade-off. Stage 2's prompt is almost entirely cache-hit from stage 1.
使用者意圖規則。分類器的 Prompt 建立了什麼是「被授權的」，而非只是「與目標相關的」。代理自行選擇的一切都是「未授權的」，直到使用者明確表示。例如「清理我的分支」不授權批量刪除。
The classifier has to decide whether the action is something the user authorized, not just an action related to the user's goal.
📚 核心概念
Prompt Cache 作為設計槓桿
Stage 1 和 Stage 2 共享相同的 Prompt（只有最後一行指令不同），所以 Stage 2 幾乎是免費的 cache 命中。這是一個優雅的工程選擇：你用相同的 context 做兩次不同嚴格度的判斷，而只付一次 context 載入的成本。
🛠️ 實務應用
「授權」vs「相關」的區分

這是整篇文章最重要的設計原則之一。在 CDMO 環境中：

• 操作員說「清理設備」不代表授權拆卸設備
• SOP 寫「調整參數」不代表可以修改 validated range

分類器的 user-intent 規則等同於 GMP 中的「範圍限制」(scope limitation)。
⚖️ 工具比較
三種替代方案的評估結果：

• 含助理訊息+工具結果：overeagerness 測試集表現下降（代理的自我辯解影響判斷）
• 只看工具呼叫：最低漏報率，但失去使用者意圖錨點（連使用者明確要求的操作也被阻擋）
• 使用者訊息+工具呼叫：最終選擇，平衡兩端

17% 的漏報率是誠實的數字。在大多數漏報案例中，分類器正確識別了危險。它出錯的地方是判斷會話中的真實同意信號是否涵蓋了「這個」操作。分類器找到了「看起來像批准的證據」，但沒有進一步檢查它是否是對操作影響範圍的同意。
The 17% false-negative rate is the honest number. What the classifier gets wrong is whether a real consent signal actually covers this action.
Auto mode 不是仔細人工審查的直接替代品。它是為了取代 --dangerously-skip-permissions 而設計的，對於那群使用者來說，這是實質性的安全改善。
Auto mode is not a drop-in replacement for careful human review. It is meant for users running --dangerously-skip-permissions.
⚠️ 重點提示
17% FNR 的根因分析：

分類器能看到危險（force-push、deletion），但在「使用者授權範圍推斷」上失敗。使用者說了「Clean up the PR」，分類器把這解讀為包含 force-push 的授權。

團隊嘗試過 prompt engineering 來修正，但「每個修正都帶來更多的成本」。這是安全工程的現實：完美是好的敵人。
🛠️ 實務應用
GMP Copilot Mode 的啟示：

這篇文章用真實數據確認了一個重要原則：AI 分類器不能完全替代人工審查。

在 GMP 環境中，這意味著：
• AI 可以做初步篩選（Tier 1-2 相當於自動化的 routine checks）
• 高風險操作仍然需要人工審查（Tier 3 = 人工放行）
• 但即使人工審查，93% 的批准率也暗示我們需要重新設計審查流程

Part VI: 多代理處理與拒絕後恢復

多代理交接。分類器在子代理交接的兩端運行：委派出去時和結果回來時。出站檢查存在是因為，如果沒有它，子代理的轉錄分類器可能缺乏完整上下文來判斷任務是否真的由使用者發起。
The classifier runs at both ends of a subagent handoff: when work is delegated out and when results come back.
回傳檢查存在是因為子代理可能在運行中被提示注入攻陷。如果被標記，系統會附加安全警告給上層代理，而非丟棄結果。
The return check exists because a subagent could be compromised mid-run. If flagged, we prepend a security warning rather than discard results.
拒絕後繼續 (Deny-and-continue)。當分類器阻擋操作時，Claude 不會停下來等待輸入，而是嘗試恢復並找到更安全的方法。如果一個會話累積了 3 次連續拒絕或 20 次總拒絕，會升級到人工介入。在 headless 模式下（claude -p），則直接終止程序。
When the classifier blocks an action, Claude should recover and try a safer approach. 3 consecutive or 20 total denials escalate to human. In headless mode, we terminate the process.
這也使得誤報變得可存活。0.4% 的 FPR 聽起來很小，但如果每次誤報都終止會話，對長時間運行的任務將是嚴重的可用性問題。
False positives are survivable. A 0.4% FPR sounds small, but if every false positive killed the session it would be a serious usability problem.
📚 核心概念
Deny-and-Continue = 優雅降級

這是一個精妙的設計選擇。不是「全有或全無」，而是：

1. 告訴代理為什麼被拒絕
2. 要求它尊重邊界找替代方案
3. 設定上限防止無限重試

這對應到工程中的「graceful degradation」原則：系統不應該因為一個元件失敗就完全停擺。
⚠️ 重點提示
Headless Mode 的安全含義

在 claude -p headless 模式下，沒有 UI 可以詢問人工，所以拒絕上限到達後直接終止程序。這對自動化管道的設計有重要含義：你必須預期任務可能因安全阻擋而被終止，並設計相應的錯誤處理和重試邏輯。
💡 比喻說明
3/20 上限 = 逸脫趨勢管理

在藥廠，如果同一條產線連續出現 3 次 OOS (out-of-specification)，或月累計超過 20 次，就必須停線調查。

Auto mode 的 3 次連續 / 20 次總計上限遵循完全相同的邏輯：個別拒絕可以恢復，但模式化的拒絕表示有系統性問題需要人工介入。

Key Takeaways

🎯
分類器取代審批，不取代人工判斷
Auto mode 的目標是取代 --dangerously-skip-permissions，而不是取代仔細的人工審查。它是「比沒有好很多」的中間路線，不是「完美的自動化」。
🛡
雙層防禦 + 推理盲目設計
輸入層掃描提示注入，輸出層分類器判斷操作安全性。分類器刻意不看代理的自我解釋，防止被「說服」-- 獨立審查原則的工程實現。
📊
誠實的 17% 漏報率
分類器能看到危險操作，但在推斷「使用者授權範圍」時出錯。過度積極行為（overeager behavior）是最難捕捉的威脅類型，因為看起來像合理的問題解決。
⚙
Prompt Cache 驅動的成本效率
Stage 1 和 Stage 2 共享相同 Prompt，只改最後一行指令。Stage 2 幾乎是免費的 cache hit。拒絕後繼續（deny-and-continue）讓誤報不會終止整個會話。

Practice Questions

Q1 概念理解 為什麼分類器要「移除」代理的自我解釋（assistant text），而不是把它當作額外的判斷依據？

因為代理可能產生高度說服力的合理化解釋（如「使用者隱含地批准了這個操作」），讓分類器被「說服」做出錯誤判斷。移除助理文字讓分類器只看「做了什麼」（工具呼叫），不看「為什麼這樣做」（代理的推理），確保判斷的獨立性。

這和 GMP 環境中「獨立 QA 審核」原則一致：審核者不應受執行者的自我辯解影響。在 Anthropic 的測試中，包含助理訊息的版本在 overeagerness 測試集上表現更差，驗證了這個設計選擇。

Q2 產業應用 如果你要在 CDMO 批次記錄審查流程中引入類似 auto mode 的 AI 輔助，你會如何設計三層權限結構？

Tier 1（自動放行）：格式檢查、欄位完整性驗證、計算覆核（如 yield 計算）-- 這些是純機械性的、無判斷空間的操作。

Tier 2（低風險自動處理）：在已驗證範圍內的參數值確認、標準操作時序驗證、設備日誌與批次記錄的交叉比對 -- 有明確規則可循的判斷。

Tier 3（人工審查）：偏差評估、CAPA 有效性判定、非典型趨勢解讀、任何涉及「是否接受」的最終決策 -- 需要領域專業判斷和責任承擔的操作。

關鍵原則：與 auto mode 相同，Tier 3 的存在不是因為 AI 做不到，而是因為這些決策的 blast radius 要求人工承擔責任。

Q3 批判思考 文章承認 17% 的漏報率是「誠實的數字」。但真實過度積極行為的測試集只有 n=52。這個樣本量對結論的可靠度有什麼影響？

n=52 意味著 17% FNR 的置信區間非常寬。約 9 個案例被漏報（52 x 0.17 ≈ 9），在二項分布下，95% 置信區間大約是 8%-30%。換言之，真正的漏報率可能低至 8%，也可能高達 30%。

文章對此是透明的：它承認樣本小是因為「需要 Claude 在真實會話中嘗試危險行為並被捕獲和標記」。但這也暗示了一個更深的問題：我們只能測試「被發現的」過度積極行為，而真正危險的可能是那些從未被注意到的。

這是安全評估的結構性挑戰：你的測試集本身就有 survivorship bias（倖存者偏誤）-- 你只測試了你知道的威脅。

✔ 核心洞察
Auto mode 的「授權 vs 相關」區分是我見過最清晰的 copilot mode 設計原則。分類器判斷的不是「這個操作是否與目標有關」，而是「使用者是否明確授權了這個操作」。這個框架可以直接套用到我設計 CDMO AI copilot 工具時的權限設計：AI 可以建議任何與目標相關的操作，但只能執行使用者明確授權的操作。Reasoning-blind design 更是把 GMP 的「獨立審查」原則做到了工程層面。
📈 立即應用
1. 在 Claude Code 中啟用 auto mode 取代 --dangerously-skip-permissions，實際體驗三層權限的運作。2. 將「授權 vs 相關」框架整理成 CDMO AI copilot 設計原則的一部分，加入 COO Track 的 AI governance 筆記。3. 測試 headless mode + auto mode 的組合對自動化文件管道的影響，確認 deny-and-continue 的行為是否需要額外的錯誤處理。
