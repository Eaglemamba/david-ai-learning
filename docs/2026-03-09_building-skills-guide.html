Part I
什麼是 Skill？核心設計原則
Skill 是一組打包成簡單資料夾的指令集，教 Claude 如何處理特定任務或工作流程。它是客製化 Claude 最強大的方式之一。不需要每次對話都重新解釋你的偏好、流程和領域專業知識，Skill 讓你教一次就能持續受益。
A skill is a set of instructions - packaged as a simple folder - that teaches Claude how to handle specific tasks or workflows. Skills are one of the most powerful ways to customize Claude for your specific needs.
📚 三層漸進式揭露 (Progressive Disclosure)

第一層 (YAML frontmatter)：永遠載入 Claude 的系統提示中。提供剛好足夠的資訊，讓 Claude 知道何時該使用這個 Skill。

第二層 (SKILL.md body)：當 Claude 認為 Skill 與當前任務相關時才載入。包含完整指令與指導。

第三層 (Linked files)：Skill 目錄中的附加檔案，Claude 僅在需要時才瀏覽和發現。

This progressive disclosure minimizes token usage while maintaining specialized expertise.

可組合性 (Composability)：Claude 可以同時載入多個 Skill。你的 Skill 應該能與其他 Skill 良好協作，不要假設它是唯一可用的能力。
Claude can load multiple skills simultaneously. Your skill should work well alongside others, not assume it's the only capability available.
可攜性 (Portability)：Skill 在 Claude.ai、Claude Code 和 API 上的運作方式完全相同。建立一次，所有介面都能使用。
Skills work identically across Claude.ai, Claude Code, and API. Create a skill once and it works across all surfaces without modification.
💡 廚房比喻：MCP vs Skill

MCP = 專業廚房：提供工具、食材和設備的存取權限。它處理的是「Claude 能做什麼」。

Skill = 食譜：一步步的指示，教你如何用這些工具創造有價值的東西。它處理的是「Claude 應該怎麼做」。

兩者結合，讓使用者能完成複雜任務，而不需要自己摸索每一步。

⚖️ MCP vs Skills 功能對照
MCP (Connectivity)Skills (Knowledge)
連接 Claude 到你的服務教 Claude 如何有效使用你的服務
提供即時資料存取與工具呼叫捕捉工作流程與最佳實踐
Claude 能做什麼Claude 應該怎麼做
⚠️ 沒有 Skill 的痛點

使用者連接 MCP 後不知道下一步該做什麼；每次對話都從零開始；結果不一致因為每個人的提示方式不同；使用者將問題歸咎於 MCP connector，但真正的問題是缺乏工作流程指導。

Part II
規劃與 Use Case 設計
在寫任何程式碼之前，先確定 2-3 個具體的 Use Case。好的 Use Case 定義包含：觸發條件、步驟流程、和預期結果。
Before writing any code, identify 2-3 concrete use cases your skill should enable.
📚 三大 Skill Use Case 類別

Category 1: Document & Asset Creation -- 建立一致、高品質的輸出物（文件、簡報、應用、設計、程式碼）。使用內嵌的風格指南和模板結構。

Category 2: Workflow Automation -- 多步驟流程，受益於一致的方法論。包含驗證閘門、迭代精煉循環。

Category 3: MCP Enhancement -- 為 MCP 伺服器提供的工具存取添加工作流程指導。協調多個 MCP 呼叫、嵌入領域專業知識。

定義成功標準。量化指標（目標）：Skill 在 90% 的相關查詢中觸發、在 X 次工具呼叫內完成工作流程、每個工作流程 0 次失敗的 API 呼叫。質性指標：使用者不需要提示 Claude 下一步、工作流程不需要使用者修正就能完成、跨 Session 結果一致。
These are aspirational targets - rough benchmarks rather than precise thresholds. Aim for rigor but accept that there will be an element of vibes-based assessment.
🛠️ Use Case 定義範例
Use Case: Project Sprint Planning
Trigger: User says "help me plan this sprint"
Steps:
  1. Fetch current project status (via MCP)
  2. Analyze team velocity and capacity
  3. Suggest task prioritization
  4. Create tasks with labels/estimates
Result: Fully planned sprint with tasks created
⚠️ 規劃時問自己的關鍵問題

使用者想完成什麼？這需要什麼多步驟工作流程？需要哪些工具（內建或 MCP）？應該嵌入什麼領域知識或最佳實踐？

💡 我的 Master Prompt = Category 1 Skill

對照三大類別，我的 Master Prompt v1.5.0 就是一個典型的 Category 1: Document & Asset Creation Skill -- 內嵌風格指南（Design System）、模板結構（HTML template）、完成前品質檢查（layout_balance_check）。目前它活在 Project Instructions 裡，但理論上可以重構為標準 SKILL.md 格式。

Part III
YAML Frontmatter 與有效指令撰寫
YAML frontmatter 是 Claude 決定是否載入你的 Skill 的依據。把這個寫好是最關鍵的。
The YAML frontmatter is how Claude decides whether to load your skill. Get this right.
💻 最小必要格式
---
name: your-skill-name
description: What it does. Use when user asks to [specific phrases].
---

name（必要）：kebab-case、無空格、無大寫、應與資料夾名稱一致。description（必要）：必須同時包含「做什麼」和「何時使用」。上限 1024 字元。不可含 XML 標籤。

Description 的結構公式：[做什麼] + [何時使用] + [關鍵能力]。Description 是漸進式揭露的第一層 -- 它出現在 Claude 的系統提示中，提供剛好足夠的資訊讓 Claude 知道何時該使用這個 Skill。
This metadata provides just enough information for Claude to know when each skill should be used without loading all of it into context.
⚠️ 安全限制

Frontmatter 中禁止使用 XML 角括號（< >）-- 因為 frontmatter 出現在 Claude 的系統提示中，惡意內容可能注入指令。Skill 名稱中不可包含 "claude" 或 "anthropic"（保留名稱）。

🛠️ Good vs Bad Description 對照

Analyzes Figma design files and generates developer handoff docs. Use when user uploads .fig files, asks for "design specs" or "design-to-code handoff".

Helps with projects.

End-to-end customer onboarding for PayFlow. Use when user says "onboard new customer" or "create PayFlow account".

Creates sophisticated multi-page documentation systems.

📚 指令撰寫最佳實踐

要具體且可執行。寫 Run python scripts/validate.py --input {filename}，不要寫「驗證資料後再繼續」。

包含錯誤處理。列出常見錯誤訊息、原因和解決方案。

使用漸進式揭露。SKILL.md 專注於核心指令，詳細文件移至 references/。

關鍵驗證用程式碼。程式碼是確定性的，語言解讀不是。對關鍵驗證，考慮用腳本而非語言指令。

💻 Critical 命名規則
# 檔名必須完全一致（大小寫敏感）
SKILL.md   ✅
SKILL.MD   ❌
skill.md   ❌

# 資料夾命名：kebab-case
notion-project-setup   ✅
Notion Project Setup   ❌
notion_project_setup   ❌

# 不可包含 README.md
# 所有文件放在 SKILL.md 或 references/
Part IV
測試與迭代
Skill 可以在不同的嚴謹度等級進行測試：手動測試（Claude.ai 直接執行）、腳本測試（Claude Code 自動化）、程式化測試（Skills API 系統性評估）。選擇與你的品質需求和 Skill 能見度相匹配的方法。
Choose the approach that matches your quality requirements and the visibility of your skill.
⚠️ Pro Tip：先聚焦單一任務

最有效的 Skill 建構者會先在單一困難任務上迭代，直到 Claude 成功為止，然後才把成功的方法提取成 Skill。這利用了 Claude 的 in-context learning，比廣泛測試更快獲得信號。有了可運作的基礎之後，再擴展到多個測試案例以覆蓋範圍。

We've found that the most effective skill creators iterate on a single challenging task until Claude succeeds, then extract the winning approach into a skill.

📚 三層測試方法

1. Triggering 測試：確保 Skill 在正確時機載入。測試顯而易見的任務（應觸發）、改述過的請求（應觸發）、以及不相關的主題（不應觸發）。

2. Functional 測試：驗證 Skill 產出正確的結果。有效輸出已生成、API 呼叫成功、錯誤處理有效、邊界案例已覆蓋。

3. Performance 比較：證明 Skill 改善了基準結果。比較有無 Skill 的來回訊息數、失敗 API 呼叫數、消耗的 Token 數。

Skill 是活的文件。根據回饋持續迭代：觸發不足（Under-triggering）→ 在 description 中添加更多關鍵字和觸發詞；過度觸發（Over-triggering）→ 添加否定觸發條件，更具體化；執行問題 → 改善指令，添加錯誤處理。
Skills are living documents. Plan to iterate based on feedback.
💻 Triggering 測試範例
Should trigger:
- "Help me set up a new ProjectHub workspace"
- "I need to create a project in ProjectHub"
- "Initialize a ProjectHub project for Q4"

Should NOT trigger:
- "What's the weather in San Francisco?"
- "Help me write Python code"
- "Create a spreadsheet"

💡 Debug 小技巧

問 Claude：「When would you use the [skill name] skill?」Claude 會引用 description 回答。根據缺少的部分來調整你的 description。

如果 Skill 觸發太頻繁，在 description 中加入否定觸發：Do NOT use for simple data exploration (use data-viz skill instead).

📚 skill-creator 工具

Claude.ai 內建的 skill-creator skill 可以從自然語言描述生成 Skill、審查並提供改善建議、建議測試案例。使用方式：「Help me build a skill using skill-creator」。如果你有 MCP 伺服器和 2-3 個工作流程，通常 15-30 分鐘就能完成。

Part V
分發與共享
Skill 讓你的 MCP 整合更完整。當使用者比較不同的 connector 時，有 Skill 的方案能更快創造價值，在純 MCP 替代方案中脫穎而出。
Skills make your MCP integration more complete. As users compare connectors, those with skills offer a faster path to value.
📚 當前分發模式（2026 年初）

個人使用者：下載 Skill 資料夾 → 壓縮成 .zip → 上傳至 Claude.ai Settings > Capabilities > Skills，或放在 Claude Code skills 目錄。

組織層級：管理員可以全工作區部署 Skill（2025/12/18 推出），自動更新、集中管理。

API 使用：/v1/skills endpoint 用於列出和管理 Skill；透過 container.skills 參數添加到 Messages API 請求；支援 Claude Agent SDK。

開放標準。Anthropic 將 Agent Skills 發布為開放標準。如同 MCP，他們相信 Skill 應該可以跨工具平台移植 -- 同一個 Skill 無論使用 Claude 或其他 AI 平台都能運作。
Like MCP, we believe skills should be portable across tools and platforms.
🛠️ 推薦的分發策略

Step 1: 在 GitHub 託管公開 repo，清晰的 README（注意：這是 repo-level README，不是 Skill 資料夾內的）、範例用法與截圖。

Step 2: 在 MCP 文件中加入連結，說明為什麼 Skill + MCP 一起使用更有價值。

Step 3: 建立安裝指南，降低使用門檻。

Part VI
五大 Skill Pattern
這些 Pattern 來自早期採用者和內部團隊建立的 Skill。它們代表常見的有效方法，不是規定性的模板。開始前，先釐清你的方法論：Problem-first（使用者描述成果，Skill 處理工具）還是 Tool-first（使用者已有工具存取，Skill 提供專業知識）。
These patterns emerged from skills created by early adopters and internal teams. They represent common approaches we've seen work well, not prescriptive templates.
📚 Pattern 1: Sequential Workflow Orchestration

適用場景：使用者需要按特定順序執行多步驟流程。

關鍵技巧：明確的步驟排序、步驟間的相依性、每階段驗證、失敗時的回滾指令。

範例：客戶入職流程 → Step 1: Create Account → Step 2: Setup Payment (wait for verification) → Step 3: Create Subscription (uses customer_id from Step 1) → Step 4: Send Welcome Email

📚 Pattern 2: Multi-MCP Coordination

適用場景：工作流程跨越多個服務。

關鍵技巧：清晰的階段分離、跨 MCP 的資料傳遞、進入下一階段前的驗證、集中式錯誤處理。

範例：Design-to-Dev Handoff → Phase 1: Figma MCP (export) → Phase 2: Drive MCP (store assets) → Phase 3: Linear MCP (create tasks) → Phase 4: Slack MCP (notify team)

📚 Pattern 3: Iterative Refinement

適用場景：輸出品質透過迭代改善。

關鍵技巧：明確的品質標準、迭代改善循環、驗證腳本、知道何時停止迭代。

範例：報告生成 → Initial Draft → Quality Check (run validation script) → Refinement Loop (repeat until threshold) → Finalization

💡 Home Depot 比喻

你可能帶著問題走進去（「我要修廚房櫃子」→ 員工指引你到正確工具）。或者你可能挑了一個新電鑽，問怎麼用在你的具體工作上。Skill 也一樣運作：Problem-first = 描述成果，Skill 安排正確的 MCP 呼叫；Tool-first = 已有工具存取，Skill 教最佳工作流程。

📚 Pattern 4: Context-Aware Tool Selection

適用場景：相同成果，但根據情境使用不同工具。

關鍵技巧：清晰的決策標準、備用選項、向使用者說明選擇原因。

範例：Smart File Storage → 大檔案(>10MB) → Cloud Storage MCP；協作文件 → Notion MCP；程式碼 → GitHub MCP；暫存 → Local Storage

📚 Pattern 5: Domain-Specific Intelligence

適用場景：Skill 添加超越工具存取的專業領域知識。

關鍵技巧：領域專業嵌入邏輯中、行動前的合規檢查、全面的文件記錄、清晰的治理。

範例：Payment Processing with Compliance → 先檢查制裁名單、管轄權許可、風險等級 → 通過才處理交易 → 記錄所有合規檢查的稽核軌跡。

CDMO 對映：這與我們的偏差調查工作流程非常類似 -- 先評估影響範圍，再決定 CAPA 等級。

Part VII
疑難排解
⚠️ Skill 無法上傳

"Could not find SKILL.md" → 檔名大小寫不正確。必須是 SKILL.md。

"Invalid frontmatter" → YAML 格式問題。最常見：缺少 --- 分隔符、引號未關閉。

"Invalid skill name" → 名稱含空格或大寫。使用 kebab-case。

⚠️ Skill 不觸發 / 過度觸發

不觸發：修改 description。是否太通用？是否包含使用者會實際說的觸發詞？是否提到相關檔案類型？

過度觸發：添加否定觸發（Do NOT use for...）、更具體化、釐清範圍（Use specifically for online payment workflows, not for general financial queries）。

⚠️ 指令未被遵循

指令太冗長：保持精簡、使用項目符號和編號清單、詳細參考移至 references/。

指令被埋沒：關鍵指令放在最前面，使用 ## Critical 標題，必要時重複重點。

語言模糊：不要寫「確保適當驗證」，要寫「CRITICAL: 在呼叫 create_project 之前，驗證 Project name 非空、至少指派一位團隊成員、開始日期不在過去」。

💻 MCP 連線問題排查清單
Checklist:
1. 確認 MCP server 已連接
   Claude.ai: Settings > Extensions
   應顯示 "Connected"

2. 檢查認證
   - API key 有效且未過期
   - 權限/scopes 正確
   - OAuth token 已刷新

3. 獨立測試 MCP（不經過 Skill）
   "Use [Service] MCP to fetch my projects"
   # 如果這也失敗，問題在 MCP 不在 Skill

4. 驗證 tool 名稱
   # Tool 名稱大小寫敏感
   # 對照 MCP server 文件
🛠️ Large Context 問題

症狀：Skill 似乎變慢或回應品質下降。

原因：Skill 內容太大、同時啟用太多 Skill、所有內容一次載入而非漸進式揭露。

解決方案：

1. 優化 SKILL.md 大小 -- 詳細文件移至 references/，保持 SKILL.md 在 5,000 字以內。

2. 減少同時啟用的 Skill -- 評估是否超過 20-50 個 Skill。考慮相關能力的 Skill「套件包」。

💡 對抗模型「偷懶」

添加明確的鼓勵語句：「Take your time to do this thoroughly. Quality is more important than speed. Do not skip validation steps.」

重要：將這些鼓勵語加在 user prompt 中比加在 SKILL.md 中更有效。

Key Takeaways
Skill = 可攜式知識封裝
Skill 不是程式碼，是指令集。一個資料夾、一個 SKILL.md、跨 Claude.ai / Claude Code / API 完全一致運作。三層漸進式揭露設計最小化 Token 使用，同時保持專業能力。
Description 決定一切
YAML frontmatter 的 description 是 Claude 決定何時載入 Skill 的唯一依據。公式：[做什麼] + [何時使用] + [關鍵能力]。包含觸發詞、檔案類型、否定條件。這 1024 字元的欄位比 Skill 內容本身更重要。
五大 Pattern 覆蓋 90% 場景
Sequential Workflow（按序執行）、Multi-MCP Coordination（跨服務協調）、Iterative Refinement（品質迭代）、Context-Aware Selection（情境選擇）、Domain Intelligence（領域專業嵌入）。大多數 Skill 是其中一到兩種的組合。
開放標準的戰略意義
Anthropic 將 Skill 定位為開放標準，如同 MCP。這表示 Skill 不是 Claude 的專屬功能，而是 AI Agent 生態系的基礎設施。MCP builder 如果同時提供 Skill，在 connector 比較中具有差異化優勢。
Practice Questions
Q1 概念理解 Skill 的三層漸進式揭露（Progressive Disclosure）中，每一層分別在什麼時候被載入？為什麼這個設計對 Token 效率很重要？

第一層（YAML frontmatter）永遠載入 Claude 的系統提示中 -- 這是「門牌」，告訴 Claude 這個 Skill 存在和何時該用它。第二層（SKILL.md body）只在 Claude 判斷 Skill 與當前任務相關時才載入 -- 這是「完整指令手冊」。第三層（Linked files）只在執行中需要時才瀏覽 -- 這是「參考文件庫」。

Token 效率的關鍵在於：如果所有 Skill 的完整內容都在系統提示中，context window 會迅速被佔滿。透過分層，系統提示中只需要放幾百字元的 description（最多 1024），而不是數千字的完整指令。以我們的情境為例，我的 Master Prompt 就有超過 1000 行 -- 如果同時啟用 10 個這樣的 Skill，context window 就直接被塞滿了。

Q2 產業應用 以 CDMO 的偏差調查流程為例，如果要建構一個 Skill，你會選擇哪個 Pattern？如何設計 Use Case 定義？

偏差調查最適合 Pattern 5: Domain-Specific Intelligence 與 Pattern 1: Sequential Workflow 的組合。

Pattern 5 的適用原因：偏差調查需要嵌入 GMP 法規知識（PIC/S、Annex 1）、根據偏差類型判斷影響範圍、評估 CAPA 等級 -- 這些都是超越工具存取的領域專業知識。

Pattern 1 的適用原因：調查有嚴格的順序要求 -- 先記錄偏差事實、再評估根本原因、然後決定 CAPA 行動、最後追蹤效果確認。

Use Case 定義範例：Trigger: "triage this deviation" or "assess deviation impact" → Steps: 1. 收集偏差基本資訊 2. 對照歷史偏差資料庫判斷是否重複發生 3. 評估對產品品質的影響（套用 Decision Tree） 4. 建議 CAPA 等級和初步調查方向 → Result: 結構化的偏差初步評估報告。

Q3 批判思考 這份指南聲稱 Skill 是「開放標準」，但目前實際上只在 Claude 生態系中運作。這個定位策略有什麼潛在風險和機會？

機會：如果其他 AI 平台也採用 Skill 格式，Anthropic 就成為 AI Agent 基礎設施的標準制定者 -- 類似 MCP 在工具連接層的位置。對於 MCP builder 來說，寫一次 Skill 能跨平台使用，降低了投入成本。對企業使用者來說，不會被鎖定在單一 AI 供應商。

風險：「開放標準」聲明目前更像是願景而非現實。指南本身承認 "some skills are designed to take full advantage of a specific platform's capabilities" -- 意味著實務上很多 Skill 仍然有平台依賴性。如果競爭者（OpenAI、Google）推出不同的 Skill/Plugin 標準，可能出現標準碎片化。

對我的啟示：目前投入建構 Skill 是安全的，因為最壞情況下它們仍在 Claude 生態系內有價值。但在設計時應避免過度依賴 Claude 獨有的功能（如特定的 tool calling 語法），以保持未來的可攜性。

🎯 我的行動結論
✔ 核心洞察
Skill 將我在 Claude.ai Projects 中累積的 Master Prompt 和工作流程知識，從「每個 Project 各自維護」提升為「可攜式、可分享、可版本控制」的標準化格式。最關鍵的設計洞察是 description field 的絕對重要性 -- 這不只是文件，它是 Claude 的決策引擎。三層漸進式揭露的架構設計，完美解釋了為什麼 Claude.ai Projects 和 Claude Code 在 Skill 載入行為上有差異（Projects = 全量注入，Code = 選擇性載入）。
📈 立即應用
本週重新審視現有的 Amaran Presentation Skill 和 CDMO Feasibility Assessment Skill 的 description field，確認它們符合 [做什麼 + 何時使用 + 關鍵能力] 的公式。特別檢查是否包含足夠的觸發詞和否定條件。接著，嘗試用 skill-creator 工具從自然語言描述生成一個簡單的新 Skill（例如 CDMO Deviation Triage Skill），測試 15-30 分鐘建構一個可用 Skill 的說法是否成立。
⚠ 需要補充
指南偏向 MCP builder 視角，對「純 standalone Skill」（如我的 Master Prompt 轉換場景）的具體轉換步驟著墨不夠。Performance 測試的量化指標（如 90% 觸發率）在實務中如何系統性測量，需要等 Anthropic 後續提供的測量工具。此外，Skill 與 Claude.ai Project Instructions 的功能重疊如何取捨，指南沒有明確建議 -- 這需要自己實驗。
