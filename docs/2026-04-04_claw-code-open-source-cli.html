Part I
ProjectContext — 專案環境探索引擎

Claw Code 啟動時做的第一件事就是探索專案環境。ProjectContext 結構體負責收集四類關鍵資訊：工作目錄路徑、當前日期、Git 狀態（status + diff）、以及指令檔案（instruction files）。

On startup, Claw Code discovers project context: working directory, date, git state, and instruction files — all fed into the system prompt.

指令檔案的探索邏輯特別精巧：從當前目錄向上遍歷所有祖先目錄，尋找 CLAW.md、CLAW.local.md 和 .claw/instructions.md。找到後以內容雜湊去重，避免 symlink 或重複掛載導致同一份指令被載入兩次。

discover_instruction_files() traverses ancestor directories for CLAW.md, CLAW.local.md, and .claw/instructions.md, deduplicating by stable content hash.

每個指令檔案有字元預算：單檔 4,000 字元、總計 12,000 字元。超過就截斷。這是一個務實的設計 — 避免用戶放了一本百科全書進 CLAW.md 就把上下文窗口塞爆。

Character budgets: 4,000 per file, 12,000 total — preventing context window overflow from oversized instruction files.

📚 核心概念 — Instruction File 層級

三種指令檔案，三種用途：

1. CLAW.md — 團隊共享的專案規則，應提交到 Git（等同 Claude Code 的 CLAUDE.md）

2. CLAW.local.md — 個人偏好，不提交到 Git（等同 CLAUDE.local.md）

3. .claw/instructions.md — 目錄層級的指令，適合 monorepo 中不同子專案有不同規則

💡 類比 — 對照 Claude Code

Claw Code 的 CLAW.md 就是 Claude Code 的 CLAUDE.md。檔名不同，機制幾乎一模一樣。兩者都支援：祖先目錄遍歷、local 覆蓋、內容去重。最大差異是 Claw Code 明確設了字元上限（4K/12K），Claude Code 的截斷策略則未公開。

struct ProjectContext {
    working_directory: PathBuf,
    date: String,
    git_status: Option<String>,
    git_diff: Option<String>,
    instruction_files: Vec<ContextFile>,
}

struct ContextFile {
    path: PathBuf,
    content: String,
}
Part II
ConfigLoader — 多層配置合併系統

配置系統是 Claw Code 最精密的模組之一。ConfigLoader 按優先順序從五個位置載入配置：User legacy .claw.json、User ~/.claw/settings.json、Project .claw.json、Project .claw/settings.json、Local .claw/settings.local.json。

ConfigLoader discovers and loads config files from 5 locations in precedence order, with later files overriding earlier ones via deep merge.

合併策略是深度合併（deep merge），不是簡單覆蓋。如果 User config 設了 MCP server A，Project config 設了 MCP server B，最終結果是兩個都有。只有同一個 key 的值才會被後者覆蓋。這讓團隊配置和個人偏好能無衝突共存。

Deep merge preserves nested objects — team and personal configs coexist without conflict.

每個配置來源都標記了 ConfigSource 枚舉：User、Project、Local。這不只是元資料 — 權限系統會根據來源決定信任等級。例如，Project config 不能覆蓋 User 的安全設定。

Each config entry is tagged with its source (User/Project/Local), enabling trust-level decisions in the permission system.

⚠️ 配置載入順序（後者覆蓋前者）
#位置來源
1~/.claw.jsonUser (legacy)
2~/.claw/settings.jsonUser
3.claw.json (project root)Project
4.claw/settings.jsonProject
5.claw/settings.local.jsonLocal
🛠️ 實務應用 — RuntimeFeatureConfig

合併後的配置會被轉換為 RuntimeFeatureConfig，聚合了：

• Hooks — pre/post tool-use 攔截器

• Plugins — 插件狀態與目錄

• MCP Servers — 命名的 MCP 服務器集合

• OAuth — 認證配置

• Permissions — 權限模式

• Sandbox — 沙箱隔離設定

Part III
SystemPromptBuilder — 動態提示詞組裝器

SystemPromptBuilder 是整個系統的大腦 — 它把 ProjectContext、RuntimeConfig、和多個靜態段落組裝成最終的系統提示詞。採用經典的 Builder Pattern，依序累加：輸出風格、OS 資訊、專案上下文、運行時配置、附加段落。

SystemPromptBuilder uses the Builder Pattern to accumulate output style, OS info, project context, runtime config, and append sections into the final system prompt.

系統提示詞有明確的靜態/動態邊界，用 SYSTEM_PROMPT_DYNAMIC_BOUNDARY 常數標記。靜態部分（工具使用規則、行為準則）每次都一樣；動態部分（Git 狀態、指令檔案、日期）每次啟動都重新生成。這個設計允許快取靜態部分，減少 token 計算成本。

A boundary marker separates static (cacheable) and dynamic (regenerated per session) prompt sections — enabling token cost optimization.

最終輸出有兩種格式：build() 返回段落向量（用於需要結構化處理的場景），render() 返回合併後的純文字（用於直接傳給 API）。

Two output formats: build() returns a vector of sections, render() joins them into a single string for the API.

💻 Builder Pattern 實作
struct SystemPromptBuilder {
    output_style: Option<String>,
    os_info: Option<String>,
    project_context: Option<ProjectContext>,
    runtime_config: Option<RuntimeConfig>,
    append_sections: Vec<String>,
}

impl SystemPromptBuilder {
    fn new() -> Self { ... }
    fn with_project_context(mut self, ctx: ProjectContext) -> Self { ... }
    fn with_runtime_config(mut self, cfg: RuntimeConfig) -> Self { ... }
    fn build(self) -> Vec<String> { ... }
    fn render(self) -> String { ... }
}
📚 靜態段落清單

Claw Code 的系統提示詞包含這些靜態段落：

• get_simple_intro_section() — 自我介紹與角色定位

• get_simple_system_section() — 系統能力描述

• get_simple_doing_tasks_section() — 任務執行準則

• get_actions_section() — 工具使用規範

• render_config_section() — 動態配置渲染

常數 FRONTIER_MODEL_NAME 設為 "Opus 4.6" — 表示此版本針對最新模型最佳化。

💡 類比 — 樂高積木式提示詞

把系統提示詞想像成樂高：靜態段落是預製模組（每盒都有），動態段落是你自己加的零件（CLAW.md 內容、Git diff）。Builder Pattern 讓你可以選擇性組裝 — 不需要 Git 的場景就跳過，不需要 OS 資訊的場景也能省略。

Part IV
MCP 多傳輸協議 — 工具擴展的六條通道

Model Context Protocol (MCP) 是 AI Agent 連接外部工具的標準協議。Claw Code 支援六種傳輸模式，這是目前開源實作中覆蓋最全面的之一。每種模式對應不同的部署場景。

Claw Code supports 6 MCP transport modes — one of the most comprehensive open-source implementations available.

Stdio 是最常見的本地模式：啟動一個子程序，透過 stdin/stdout 通訊。配置包含 command、args、環境變數。SSE 和 HTTP 用於遠端服務器。WebSocket 用於需要雙向即時通訊的場景。SDK 是按名稱引用已註冊的服務器。ManagedProxy 則由平台代管。

Stdio for local processes, SSE/HTTP for remote servers, WebSocket for bidirectional real-time, SDK for named references, ManagedProxy for platform-managed services.

每個 MCP Server 配置都包裹在 ScopedMcpServerConfig 中，標記了來源範圍（scope）。這意味著 Project 配置裡定義的 MCP server 和 User 配置裡定義的有不同的信任等級 — 避免惡意 repo 透過 .claw.json 注入危險工具。

Each MCP server is scoped to its config source — preventing malicious repos from injecting dangerous tools via project config.

🔌 MCP 六種傳輸模式對照表
傳輸模式場景配置重點
Stdio本地 CLI 工具command, args, env
SSE遠端串流服務url, headers
HTTP遠端 REST APIurl, headers, OAuth
WebSocket即時雙向通訊url, headers
SDK內建服務引用server name
ManagedProxy平台代管url, id
💻 MCP 配置範例（Stdio）
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}
⚠️ OAuth 支援

HTTP 和 SSE 模式支援完整的 OAuth 配置（OAuthConfig），包含 authorization URI、token URI、client ID、scopes。MCP 專用的 McpOAuthConfig 還能設定自動刷新。這讓 Claw Code 可以安全連接需要認證的企業級工具。

Part V
權限系統與 Hooks — 安全的三層防線

Claw Code 的權限模型分三級：ReadOnly（只能讀取檔案和執行查詢）、WorkspaceWrite（可以修改工作區內的檔案）、DangerFullAccess（完全存取，包括網路和系統操作）。

Three permission levels: ReadOnly (queries only), WorkspaceWrite (modify workspace files), DangerFullAccess (full system access).

這不只是 UI 選項 — 權限等級直接影響哪些工具可以被呼叫。在 ReadOnly 模式下，即使模型要求執行 rm -rf /，runtime 層就會攔截。這是硬約束（hard constraint），不是靠 prompt 告訴模型「不要做壞事」的軟約束。

Permission levels enforce hard constraints at the runtime layer — not soft prompt-based suggestions.

Hooks 系統提供了額外的攔截層。RuntimeHookConfig 支援 pre-tool-use 和 post-tool-use 兩種掛鉤。Pre-hook 可以在工具執行前檢查參數、記錄日誌、甚至阻止執行。Post-hook 可以在執行後驗證結果、觸發通知。

Pre/post tool-use hooks provide an additional interception layer — inspect, log, block, or validate tool calls.

📚 三級權限模型

Level 1: ReadOnly
適合瀏覽程式碼、問問題。不能修改任何檔案，不能執行有副作用的命令。最安全。

Level 2: WorkspaceWrite
日常開發的最佳選擇。可以讀寫工作區檔案、執行測試、建置專案。不能存取工作區外的系統資源。

Level 3: DangerFullAccess
完全信任模式。可以安裝套件、修改系統配置、存取網路。名稱裡的 "Danger" 就是提醒。

🛠️ Hooks 的實際用途

Pre-tool-use Hook 範例：

• 在每次 Bash 執行前記錄命令到審計日誌

• 阻止包含敏感路徑（如 ~/.ssh/）的檔案讀取

• 自動在 git commit 前執行 linter

Post-tool-use Hook 範例：

• 工具執行失敗時發送 Slack 通知

• 統計每個工具的呼叫頻率用於效能分析

💡 類比 — 對照 Harness Engineering

如果把 Claw Code 映射到 Harness Engineering 的三層模型：權限系統 = 圍欄（硬邊界）、Hooks = 韁繩（可調節的控制力）、CLAW.md = 對馬說的話（prompt 層指引）。三者結合才構成完整的 harness。

Part VI
設計模式與 Rust 語言選擇

Claw Code 選擇 Rust 而非 TypeScript（Claude Code 的語言）有幾個工程理由：零成本抽象帶來的啟動速度、記憶體安全保證（不會因為 null pointer 掛掉）、以及強大的枚舉（enum）系統讓狀態機建模極為自然。

Rust was chosen for startup speed, memory safety guarantees, and expressive enum types for state machine modeling.

幾個值得注意的設計模式：Builder Pattern（SystemPromptBuilder）、Strategy Pattern（MCP 傳輸模式的枚舉分派）、Chain of Responsibility（配置載入的優先順序鏈）。每個模式都不是為了「設計模式」而用，而是因為 Rust 的型別系統讓這些模式幾乎是自然浮現的。

Builder, Strategy, and Chain of Responsibility patterns emerge naturally from Rust's type system rather than being imposed.

錯誤處理也很 Rust 風格：PromptBuildError 和 ConfigError 都是枚舉，包裹了 IO 和解析兩種錯誤。實作了 Display 和 Error trait，讓錯誤可以自動向上傳播。沒有 try-catch，只有 Result 和 ? 運算子 — 每個可能失敗的操作都必須被明確處理。

Error handling via enum variants (IO/Parse) with Display and Error trait implementations — no exceptions, only explicit Result types.

開源學習價值
作為 Claude Code 的開源替代品，Claw Code 讓你看到「黑盒子裡面是什麼」。對於想建構自己的 AI Agent CLI 的團隊來說，這是最好的參考實作。
Q1 架構設計 為什麼 SystemPromptBuilder 要區分靜態和動態段落？直接每次全部重新生成不是更簡單嗎？

效能考量：靜態段落（行為準則、工具規範）每次會話都一樣，可以利用 API 的 prompt caching 機制。Anthropic 的 API 會對 system prompt 的前綴進行快取 — 如果靜態部分放在前面且不變，就能大幅減少首次回應的 token 處理時間和成本。

除錯考量：用 SYSTEM_PROMPT_DYNAMIC_BOUNDARY 標記邊界，讓開發者可以快速定位「這個 bug 是靜態規則的問題還是動態上下文的問題」。

測試考量：靜態段落可以用 snapshot test 驗證（內容固定，diff 就是 bug）；動態段落需要 property-based test（格式正確就行，內容每次不同）。

Q2 安全實務 MCP Server 的 ScopedConfig 如何防止「惡意 repo 注入」攻擊？

攻擊場景：惡意 repo 在 .claw.json 裡配置了一個 MCP server，指向攻擊者控制的端點。開發者 clone 這個 repo 後啟動 Claw Code，攻擊者的 MCP server 就能被 Agent 呼叫。

防禦機制：每個 MCP server 配置都標記了 ConfigSource（User/Project/Local）。Project 來源的 MCP server 在高敏感操作（如存取檔案系統、網路請求）時會被額外審核或限制。只有 User 來源的配置才被視為完全信任。

對照 Claude Code：Claude Code 也有類似機制 — project-level 的 MCP server 首次使用時會提示用戶確認。Claw Code 把這個模式做成了型別系統的一部分，編譯期就能確保 scope 資訊不會丟失。

Q3 批判思考 Claw Code 是 Claude Code 的「完美複製品」嗎？它有什麼本質性的限制？

不是。Claw Code 重現了架構和配置系統，但 Claude Code 的核心競爭力在於 Anthropic 對自家模型行為的深度調校 — 例如 tool use 的微調、回應格式的最佳化、以及與 Anthropic 內部基礎設施的整合。這些不是靠讀原始碼就能複製的。

生態差距：Claude Code 背後有 Anthropic 的持續迭代和社群支援。Claw Code 作為開源專案，更新頻率和穩定性取決於社群貢獻。目前文件不足、issue 追蹤不活躍，是早期開源專案的常見問題。

但學習價值極高：正因為是開源的，你可以看到每一行程式碼的設計決策。這就像讀 Linux 原始碼來理解作業系統 — 你不會因此造出 Windows，但你會成為更好的系統工程師。
