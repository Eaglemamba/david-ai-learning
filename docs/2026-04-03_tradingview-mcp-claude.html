Executive Summary

這篇文章介紹一個開源研究專案：透過 Chrome DevTools Protocol (CDP) 將 Claude Code 連接到 TradingView Desktop 應用程式。TradingView 沒有公開 API，但因為桌面版基於 Electron（Chromium 核心），開啟 debug port 後就能透過 CDP 讀取圖表上所有資料。

專案提供 78 個 MCP 工具和完整 CLI (tv 指令)，核心應用場景是 Pine Script AI 輔助開發迴圈：描述需求 → Claude 寫腳本 → 自動注入編譯 → 讀取錯誤修正 → 重新編譯。所有資料都在本地端處理，不會離開你的電腦。

來源偏差提醒：作者是該工具的開發者，文章本質上是專案介紹文。技術描述可信度高（可從 GitHub 原始碼驗證），但「改變工作流程」等主觀評價需保留判斷。此外，使用此工具可能涉及 TradingView 服務條款（作者也明確提到這點），需自行評估風險。

Part I：運作原理 — 三層架構

Part I
How It Works
整個系統由三個部分組成，全部在你的本地機器上運行。Claude Code 與 Node.js MCP 伺服器溝通，該伺服器透過 Chrome DevTools Protocol 在 port 9222 連接到你正在執行的 TradingView Desktop 應用程式。
Claude Code talks to a Node.js MCP server. That server connects to your running TradingView Desktop app over Chrome DevTools Protocol on port 9222.
CDP 是 Google 內建於每個 Chromium 應用程式的標準除錯介面。預設關閉，一個啟動旗標就能開啟它。沒有任何資料離開你的電腦。
CDP is a standard Google debugging interface built into every Chromium app. Off by default. One flag turns it on. Nothing touches TradingView's servers. No data leaves your machine.
整套工具也包裝成 tv CLI，JSON 輸出，支援管道串接，可在任何終端機或 shell 腳本中使用。
The whole tool set also ships as a tv CLI, JSON output, pipe-friendly, works from any terminal or shell script.
這會存取 TradingView 未公開的內部 API。TradingView 可以在任何更新中不經通知就更改這些內部介面。如果你希望保持穩定，請釘選你的 TradingView Desktop 版本。
This accesses undocumented internal TradingView APIs through the Electron debug interface. TradingView can change those internals in any update without notice. If you use this and want it to stay stable, pin your TradingView Desktop version.
📚 核心概念：Chrome DevTools Protocol (CDP)

CDP 不是什麼神秘技術 — 它就是你在瀏覽器按 F12 打開開發者工具時使用的同一個協議。每個基於 Chromium 的應用程式（VS Code、Slack、Discord）都內建這個介面。差別只是有沒有在啟動時開啟 debug port。

這意味著理論上，任何 Electron 應用程式都可以用類似方法被 LLM 存取 — 不只是 TradingView。這個模式本身比單一工具更有價值。

💡 比喻說明：旁聽者模式

想像 TradingView 是一個正在進行的即時簡報。CDP 就像讓你坐在旁邊，不干擾簡報進行，但可以看到螢幕上所有投影片、數據和標註。你沒有修改任何東西，只是「讀取」已經顯示的資訊。

⚠️ 風險提醒：未公開 API 的脆弱性

使用未公開內部 API 的最大風險是隨時可能失效。TradingView 的任何版本更新都可能改變內部資料結構，導致所有 78 個工具需要逐一修復。這就是為什麼作者建議「釘選版本」。

此外，這是否違反 TradingView 服務條款是灰色地帶。作者坦承需要使用者自行評估。

Part II：Claude 能看到什麼

Part II
What Claude Can Actually See
除了明顯的商品代碼、時間框架和指標數值之外，Claude 能讀取 Pine Script 指標在圖表上繪製的實際繪圖物件。line.new()、label.new()、table.new()、box.new() — 你的指標在螢幕上畫的任何東西，Claude 都能以結構化資料讀回。
Beyond the obvious stuff like symbol, timeframe, and indicator values, Claude can read the actual drawing objects that Pine Script indicators put on your chart. The line.new() calls, label.new(), table.new(), box.new().
這對受保護的指標也有效。你只需傳入以指標名稱為目標的篩選器，就能取得每個價格水平、每個文字標註、完整表格內容、每個繪製的區域。
This works on protected indicators too. You just pass a filter targeting the indicator by name and you get back every price level, every text annotation, full table contents, every drawn zone.
完整可用資料清單：目前商品和時間框架、所有指標名稱和 ID、即時 OHLC 和成交量、最多 500 根 K 線或精簡摘要、所有資料視窗數值、所有類型的 Pine 繪圖、委買委賣簿深度、策略測試器結果和交易清單、商品後設資料，以及任何圖表區域的截圖。
The full picture: current symbol and timeframe, all indicator names and IDs, real-time OHLC and volume, up to 500 price bars or a compact summary, all data window values, Pine drawings of every type, order book depth, strategy tester results and trade list, symbol metadata, and screenshots of any chart region.
值得明確說明：所有這些資料在 TradingView 開啟的那一刻就已經顯示在你的螢幕上。這個工具只是讓 LLM 能夠讀取它。
All of this data is already on your screen the moment TradingView is open. The app is displaying it. This tool just makes it readable to an LLM.
🛠️ 實務應用：可讀取的資料類型
  • 價格資料：即時 OHLCV、最多 500 根 K 線
  • 指標資料：名稱、ID、所有計算值
  • Pine 繪圖：線、標籤、表格、方框等所有物件
  • 策略測試：回測結果、交易清單、績效指標
  • 委買委賣簿：深度資料
  • 截圖：任何圖表區域的視覺擷取
  • 後設資料：商品資訊、時間框架等
⚠️ 重點提示：「受保護指標」的意涵

作者提到可以讀取「受保護指標」的繪圖資料。這意味著即使指標的原始碼被鎖定，只要它在圖表上產生視覺輸出（線條、標籤、表格），CDP 就能讀取這些渲染後的數值。

這是合理的 — 因為資料已經被渲染到螢幕上。但這也是 TradingView 可能在服務條款中限制的行為之一。

⚖️ CDMO 類比：設備資料讀取

這個模式類似製藥設備的 OPC-UA 或 SCADA 整合：設備（TradingView）已經在運行並產生資料，你只是建立一個讀取通道讓另一個系統（Claude）能消費這些資料。差別在於 GMP 環境的資料整合需要驗證（CSV/21 CFR Part 11），而這個工具是未驗證的研究級工具。

Part III：技術上的可能性

Part III
What's Technically Possible
1. 帶有 AI 迴圈的 Pine Script 開發。你描述你想要的東西，Claude 寫腳本、注入 Pine 編輯器、編譯、讀取錯誤、修復、重新編譯。這個迴圈持續運行直到腳本乾淨。整個過程中它都有你當前圖表和上面所有指標的完整上下文。
Pine Script development with AI in the loop. You describe what you want, Claude writes the script, injects it into the Pine editor, compiles it, reads the errors back, fixes them, and recompiles. That loop runs until the script is clean.
2. 你的圖表作為本地資料來源。串流指令透過 CDP 在 localhost 上輪詢你正在執行的 TradingView Desktop，輸出 JSONL — 每行一個 JSON 物件。你可以把它管道到 Python、Node、shell 腳本、jq 篩選器，任何你想在本地建構的東西。
Your chart as a local data source. The streaming commands poll your running TradingView Desktop via CDP on localhost and output JSONL, one JSON object per line. You can pipe that into Python, Node, shell scripts, jq filters, whatever you want to build locally.
3. 腳本化重複性圖表任務。每個工具都可以作為 CLI 命令使用，具有一致的 JSON 輸出。任何你手動重複做的事情都是腳本化的候選對象。多商品畫面、截圖執行、早晨版面設定、圖表狀態快照、批次指標配置。
Scripting repetitive chart tasks. Every tool is available as a CLI command with consistent JSON output. Anything you do manually on repeat is a candidate for a script. Multi-symbol screens, screenshot runs, morning layout setup, chart state snapshots, batch indicator configuration.
4. 指標驗證。寫腳本、編譯、讀回數值、檢查數學是否符合你的預期。循環各種輸入參數並在每次讀取輸出。在多個商品和時間框架上執行相同指標。讀取每次編譯後的策略測試器指標。
Indicator validation. Write a script, compile it, read the values back, check that the math matches your expectations. Loop through input parameters and read the output each time.
5. 帶有結構化日誌的回放。逐步走過歷史 K 線，在每一步記錄指標數值到 JSONL，記錄你的交易決策，在進場和出場時擷取截圖。一份正式的練習記錄。
Replay with structured logging. Step through historical bars, log the indicator values at each step to JSONL, record your trade decisions, capture screenshots at entries and exits.
📚 核心概念：AI-in-the-Loop 開發迴圈

Pine Script 開發迴圈是這個工具最有價值的模式。傳統流程是：在文字編輯器寫 → 複製到 Pine 編輯器 → 手動編譯 → 看錯誤 → 回去改 → 再複製。每次迭代需要人工介入多個步驟。

有了 MCP 工具後：描述需求 → Claude 產生程式碼 → 自動注入 + 編譯 → 自動讀取錯誤 → 自動修正 → 重新編譯。人類只在最初描述需求和最後驗證結果時介入。

這是典型的 Copilot Mode 而非 Agent Mode — 人類保持最終控制權，AI 處理重複性的編譯除錯迴圈。

🛠️ 實務應用：五大應用場景
  • Pine Script AI 開發：寫 → 編譯 → 除錯全自動迴圈
  • 即時資料串流：JSONL 管道到任何本地腳本
  • 圖表任務自動化：版面設定、截圖、批次配置
  • 指標驗證：跨商品/時間框架的參數測試
  • 回放記錄：結構化交易練習日誌
⚖️ CDMO 類比：文件迴圈自動化

Pine Script 的 AI 開發迴圈模式與我們的 GMP 文件工作流程類比：

  • Pine Script 寫 → 編譯 → 除錯迴圈 ≈ SOP 草擬 → 審查 → 修正迴圈
  • 指標驗證（跨參數測試）≈ 設備確效（IQ/OQ/PQ 多條件測試）
  • 結構化回放日誌 ≈ 批次紀錄的結構化數據擷取

關鍵差異：交易場景容許 Agent 自主迭代直到成功；GMP 環境的每一步修改都需要人工審核和文件記錄。

Part IV：CLI 工具集

Part IV
The CLI
每個 MCP 工具同時也是一個終端機指令。整套工具包裝為 tv CLI，JSON 輸出可管道到任何地方。
Every MCP tool is also a terminal command. The whole thing ships as a tv CLI with JSON output that pipes anywhere.
# 讀取圖表狀態
tv status              # 連線狀態
tv quote               # 即時報價
tv ohlcv --summary     # OHLCV 精簡摘要
tv data lines          # Pine 繪圖物件
tv data values         # 資料視窗數值

# 導航
tv symbol AAPL         # 切換商品
tv timeframe D         # 切換時間框架
tv pane layout 2x2     # 設定多窗格版面
tv pane symbol 1 ES1!  # 指定窗格商品

# Pine Script 操作
tv pine set < script.pine   # 注入腳本
tv pine compile              # 編譯
tv pine analyze              # 離線靜態分析
tv pine check                # 伺服器端編譯檢查
tv pine save                 # 儲存

# 串流
tv stream quote | jq '.close'               # 串流收盤價
tv stream bars                               # 串流 K 線
tv stream lines --filter "NY Levels"         # 篩選指標線條
tv stream tables --filter Profiler           # 篩選表格資料
tv stream all                                # 串流所有資料

# Pine CI 單行指令
cat script.pine | tv pine analyze && cat script.pine | tv pine check
⚠️ 重點提示：tv stream all 的研究價值

作者特別標出 tv stream all 是「從研究角度最有趣的指令」。設定多窗格版面後，執行這個指令就能同時串流所有商品的 JSONL 資料到下游任何處理程序。這本質上是把 TradingView 變成一個本地端的即時資料源 — 無需付費 API，無需第三方資料供應商。

注意：這不是「免費獲取資料」— 你仍然需要 TradingView 的付費訂閱。這只是改變了資料的消費方式（從視覺 → 程式化）。

⚖️ CLI 設計模式：Unix 哲學

這個 CLI 的設計遵循經典 Unix 哲學：每個指令做一件事、JSON 輸出、支援管道組合。這與我們在 Claude Code 自動化管線中使用的模式完全一致 — claude -p 的 headless mode 也是用管道串接輸入輸出。

關鍵學習：好的 CLI 工具設計 = 可組合性（composability）。每個指令的輸出都能成為另一個指令的輸入，這比單一的 monolithic 工具更有彈性。

Key Takeaways

🔗 CDP 是通用的 Electron 後門
任何基於 Electron 的桌面應用程式都能透過 Chrome DevTools Protocol 被 LLM 存取。這個模式不僅限於 TradingView — 它是一個可複製的架構。開啟 debug port，建立 MCP 伺服器，LLM 就能讀取應用程式狀態。
🔄 AI-in-the-Loop 開發迴圈
Pine Script 的「寫 → 編譯 → 讀錯誤 → 修正 → 重編譯」自動迴圈是 Copilot Mode 的教科書案例。人類負責描述需求和驗證結果，AI 處理中間的重複性迭代。這個模式可以遷移到任何有「撰寫 → 驗證 → 修正」迴圈的工作流程。
📊 圖表資料程式化
將視覺資料轉換為結構化 JSONL 串流，讓下游腳本可以消費。這不是新的資料來源，而是改變資料的消費介面 — 從「人眼看圖表」到「程式讀 JSON」。Unix 管道哲學的資料工程應用。
⚠️ 未公開 API 的代價
78 個工具建構在未公開內部 API 上，任何版本更新都可能導致全部失效。這是「快速獲得能力」與「長期維護成本」的經典取捨。生產環境使用需釘選版本並準備好隨時修復的心理準備。

Practice Questions

Q1 概念理解 為什麼 TradingView 的 Electron 架構讓 CDP 整合成為可能？如果 TradingView 改用原生桌面應用程式（非 Electron），這個方案還能運作嗎？

Electron 是基於 Chromium 的框架，因此繼承了 Chrome DevTools Protocol。CDP 是 Chromium 的內建除錯介面，可以透過 WebSocket 連線讀取和操控瀏覽器/應用程式的狀態。

如果 TradingView 改用原生應用程式（如 Qt、Swift），CDP 就不可用了。你需要找到該平台的等效除錯介面（如 Windows 的 UI Automation API、macOS 的 Accessibility API），但這些介面通常提供的資料粒度遠低於 CDP，且更難以取得應用程式內部的結構化數據。Electron 的「一切都是網頁」本質是這個方案成立的前提。

Q2 產業應用 這篇文章中的「CDP 存取 Electron 應用程式」模式，能否應用到 CDMO 環境中使用的 Electron-based 軟體（如某些 LIMS 或 ELN 的桌面版本）？有哪些 GMP 合規性障礙？

技術上可行 — 如果 LIMS/ELN 的桌面版本基於 Electron，CDP 存取的原理完全相同。但 GMP 環境的障礙遠超技術層面：

合規障礙：(1) 21 CFR Part 11 要求系統驗證 — 未公開 API 的讀取通道無法通過 CSV（電腦系統驗證）；(2) 資料完整性 (ALCOA+) — 透過 debug port 讀取的資料缺乏審計追蹤；(3) 變更控制 — 開啟 debug port 需要經過完整的變更管理流程。

結論：在 GMP 環境中，這種「黑客式」整合方式與合規要求根本矛盾。正確做法是要求供應商提供經過驗證的 API 介面。但概念驗證階段可以用 CDP 快速驗證「AI 讀取系統資料」的可行性，再推動正式 API 開發。

Q3 批判思考 作者聲稱「所有資料都已經顯示在你的螢幕上，這個工具只是讓它可被 LLM 讀取」。這個論點是否足以證明繞過未公開 API 的合理性？

這個論點有技術上的事實基礎但在法律/倫理層面不完整。

技術事實：確實，CDP 讀取的資料就是已經渲染到螢幕上的資料。沒有存取「隱藏的」伺服器端資料。從技術角度，這和截圖然後 OCR 沒有本質區別。

不完整之處：(1) 服務條款通常區分「人類使用」和「自動化存取」— 同樣的資料，自動化讀取可能違反 ToS 但手動查看不會；(2) 「受保護指標」的繪圖數據讀取可能違反指標作者的預期 — 他們鎖定原始碼是假設輸出不會被程式化擷取；(3) 批量化、高頻率的資料擷取可能對 TradingView 的服務產生負載影響。

這是典型的「技術上可以做到」≠「應該這樣做」的問題。作者的坦誠態度值得肯定，但使用者需要自行評估風險而非依賴這個論點作為免責盾牌。

🎯 我的行動結論
✔ 核心洞察
CDP + MCP 是一個可複製的架構模式：任何 Electron 桌面應用程式都可以透過 debug port 被 LLM 存取。這比 TradingView 本身更重要 — 它意味著 MCP 生態系的整合對象不限於有公開 API 的服務。對於我的自動化管線，這提供了一個應對「無 API 可用」場景的備選路徑（如 Browser Use 的 CDP 模式處理 JS-rendered 網站，概念相同）。
📈 立即應用
不直接安裝此工具（我不做交易），但記錄 CDP + Electron MCP 整合模式到技術筆記。下次遇到「這個桌面軟體沒有 API」的場景時，先檢查它是否基於 Electron — 如果是，CDP 就是潛在的整合路徑。同時，這篇文章的 CLI 設計（JSON 輸出、管道友善、單一 binary）是我設計 Claude Code Skill CLI 介面的參考範本。
⚠️ 需要補充
文章缺少穩定性數據（78 個工具中有多少在 TradingView 更新後需要修復？頻率？）。也沒有討論 CDP 連線的安全性考量（port 9222 暴露是否有本地端攻擊面？）。未來如果要在 CDMO 環境探索類似模式，需要研究 CDP 連線的安全強化（TLS、認證）和審計追蹤能力。
