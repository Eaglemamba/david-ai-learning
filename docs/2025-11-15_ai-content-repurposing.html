痛點：手動內容再利用的困境

作者每週花費 5 小時 手動將電子報重新格式化成社群貼文，感覺像是在做機械式的複製貼製工作。

「諷刺的是：我是一個寫 AI 自動化主題的作家，卻在做機器人的工作。」

每週的痛苦流程：

  • 週一上午：開始撰寫週四電子報的草稿
  • 週一下午：嘗試預先 Repurpose (再利用)，將最強的段落轉成 LinkedIn 草稿、重新格式化、加換行、檢查字數、調整語氣
  • 週二：製作 Substack Notes — 10 個精簡到約 200 字元的短洞見。規劃 Twitter thread 大綱
  • 週三：定稿電子報，編輯、加例子、精煉開頭結尾

但有些週，作者完全跳過這個流程。寫完 2,000-3,000 字的長篇內容後，已經精疲力盡，無法再花 5 小時手動調整各平台格式。

核心挑戰：每個平台都有不同語言

Content Repurposing (內容再利用) 不只是「到處複製貼上同樣的東西」。

各平台特性：

  • LinkedIn：需要 Thought Leadership (思想領袖) 風格，搭配策略性換行和宣言式開場
  • Substack Notes：需要 200 字元的洞見，創造「mic-drop（一句話打趴全場）」的效果
  • Twitter Threads：需要對話式的流暢感，加上 Curiosity Gaps (好奇心缺口)

每個平台都有自己的語言、節奏和受眾期待。

解決方案：AI 自動化系統

「如果我能自動化處理語音備忘錄，為什麼不能自動化內容再利用？」

作者需要的系統必須：

  • ✅ 理解每個平台的特定適應規則
  • ✅ 保留作者的對話風格和框架思維
  • ✅ 自動運行，不需要每週手動請求 ChatGPT
  • ✅ 產出 80% 完成度的內容，只需 10-20% 最後潤飾

系統運作方式：

把電子報 URL 加到 Google Sheet，將狀態從 "Live" 改成 "Repurpose"。就這樣。

自動化產出：

  • 5 篇 LinkedIn 貼文（各種風格：步驟、教訓、範例、數據、錯誤）
  • 10 則 Substack Notes（200 字元洞見，粗體格式，精彩結尾）
  • 3 條 Twitter threads（對話式開場，好奇心驅動流程）

原本需要 5 小時的工作，現在只需 30 分鐘 最後潤飾。

建置步驟概覽

所需工具：

  • Google Sheet（用於 Webhook 整合和 URL 管理）
  • OpenAI API key（約 $5 額度）
  • Firecrawl API key（免費處理 500 個 URL）
  • Make.com 帳號（免費方案即可）

六大建置步驟：

  1. 建立 Google Sheet webhook 整合：在 Make.com 新增「Google Sheets — Watch Changes」模組，監控試算表的編輯變更
  2. 新增 Google Sheets 模組管理 URL：加入「Google Sheets — Search Rows」模組，讀取 webhook 整合的試算表資料
  3. 設定 "Repurpose" 篩選器：設定篩選器只監聽狀態變更為 "Repurpose" 的列
  4. 使用 Firecrawl 抓取電子報內容：輸入要抓取的 URL，Firecrawl 會擷取網頁內容
  5. 新增 Router 建立社群頻道分流：三個 Router 分別處理 Substack、LinkedIn、Twitter。每個頻道有自己的 System Prompt
  6. OpenAI 模組產生各平台內容：設定平台專屬的 prompt，輸出到 Google Docs

Google Sheet 欄位設計

欄位 功能說明
URL 貼上已發布的電子報 URL
Status 當狀態變更為 "Repurpose" 時觸發自動化
Channel 選擇要再利用到哪些社群頻道
Outputs 生成的 Substack Notes、LinkedIn 貼文、Twitter threads 草稿

Router 路由邏輯

篩選器來自前一個 Google Sheets 模組，邏輯如下：

  • 如果 Column C (Channel) 包含 "Substack"，送到 Substack router
  • 如果 Column C (Channel) 包含 "LinkedIn"，送到 LinkedIn router
  • 如果 Column C (Channel) 包含 "Twitter"，送到 Twitter router
  • 如果 Column C (Channel) 包含 "All"，送到所有 router

💡 教學解析 Tutorial Commentary

關鍵術語 Key Terms

Content Repurposing (內容再利用)：將一份原始內容轉換成適合不同平台、格式或受眾的多種版本。不是複製貼上，而是「轉譯」。

Webhook (網路鉤子)：一種讓應用程式之間即時通訊的機制。當某個事件發生（如試算表更新），自動通知另一個系統。

Make.com (原 Integromat)：視覺化的自動化平台，讓你用「拖拉」方式串接不同的 App 和 API。

Firecrawl：網頁抓取（Web Scraping）工具，可以擷取網頁內容並轉成結構化資料。

Router (路由器)：在自動化流程中，根據條件將資料分流到不同的處理路徑。

比喻說明：內容再利用就像「翻譯」

想像你寫了一篇中文文章，要分享給英文、日文、西班牙文的讀者。你不會直接複製中文給他們，而是：

  • 翻譯成他們的語言
  • 調整文化背景的例子
  • 配合當地讀者的閱讀習慣

社群平台就像不同的「語言區」。LinkedIn 是商務正式區、Twitter 是快節奏對話區、Substack Notes 是精煉金句區。同一個好點子，要用不同的「語法」來表達。

重點提示：為什麼「複製貼上」行不通？

演算法懲罰：各平台的演算法都會偵測「低品質」內容。如果你的貼文格式不符合平台特性，互動率會很低，演算法就會降低曝光。

受眾期待不同：LinkedIn 用戶期待專業洞見；Twitter 用戶期待快速、有趣的觀點；Substack 用戶期待深度思考。

格式限制：Twitter 有字數限制、LinkedIn 有最佳貼文長度、Substack Notes 要短而精。

比喻說明：自動化流程就像「生產線」

想像一座巧克力工廠：

  • 原料進場（Google Sheet）：你把電子報 URL 放入「原料區」
  • 品管觸發（Webhook + Filter）：品管員（篩選器）看到「可以加工」的標籤，就啟動生產線
  • 萃取原料（Firecrawl）：機器人把原始巧克力豆（網頁內容）擷取出來
  • 分流加工（Router）：根據要做什麼產品（LinkedIn/Twitter/Substack），送到不同的生產線
  • AI 塑形（OpenAI）：每條生產線有專屬的模具（prompt），把原料塑造成成品
  • 成品出貨（Google Docs）：各種口味的巧克力（貼文）送到包裝區等你驗收

效率計算：ROI 投資報酬率

舊方法：

時間成本 = 5 小時/週 × 52 週 = 260 小時/年

新方法：

時間成本 = 0.5 小時/週 × 52 週 = 26 小時/年

節省：234 小時/年（效率提升 10 倍）

金錢成本：

OpenAI API ≈ $5/月 = $60/年

如果你的時薪是 $50，節省 234 小時 = 節省 $11,700

ROI = ($11,700 - $60) / $60 = 19,400%

關鍵術語：平台特性深入解析

LinkedIn Thought Leadership：LinkedIn 推崇「思想領袖」型內容。特點是：宣言式開頭、策略性換行（製造視覺停頓）、帶有教育意義的觀點分享。

Curiosity Gaps (好奇心缺口)：Twitter thread 的秘訣。每一則推文結尾留下「懸念」，讓讀者想知道「然後呢？」而繼續往下讀。

Mic-drop Moments：Substack Notes 的精髓。一句話就讓讀者「哇」出聲的效果，通常是濃縮的洞見或反直覺的觀點。

重點提示：80% 完成度的哲學

這個系統的目標不是 100% 自動化，而是 80% 自動化 + 20% 人工潤飾。為什麼？

  • 保留人味：AI 產出的內容需要你最後的「靈魂注入」
  • 品質把關：你可以修正 AI 可能的錯誤或不準確
  • 彈性調整：根據時事或特殊情況做微調

這不是懶人自動發文機，而是「AI 助手 + 人類編輯」的協作模式。

比喻說明：Webhook 就像「門鈴」

傳統方式：你每隔 5 分鐘跑去門口看有沒有訪客（Polling 輪詢），超累。

Webhook 方式：你裝了門鈴。有訪客來按門鈴時，你才需要去開門。系統只在「有事情發生」時才通知你，省下大量無效的檢查。

在這個自動化中，Google Sheet 的變更就是「按門鈴」，Make.com 聽到門鈴就啟動流程。

實務應用：如何選擇適合的內容再利用？

最適合再利用的內容類型：

  • Evergreen Content（常青內容）：不受時間影響的知識、技巧
  • Framework（框架）：可重複使用的思考模型
  • 案例研究：具體的成功/失敗故事
  • 數據洞見：有數字支撐的觀點

不太適合的：

  • 時效性新聞（會過時）
  • 純個人日記（缺乏普遍價值）
  • 需要大量上下文的深度分析

練習思考 Practice Questions

問題 1：如果你有一篇 3000 字的技術教學文章，要轉成 LinkedIn 貼文，你會保留什麼、刪除什麼？為什麼？

提示：想想 LinkedIn 用戶的閱讀情境（可能在通勤、午休快速瀏覽），他們最想看到什麼？

問題 2：為什麼作者選擇讓系統產出「5 種不同風格」的 LinkedIn 貼文，而不是只產出 1 種最好的？

提示：想想 A/B Testing 的概念，以及內容多樣性對演算法的影響。

問題 3：如果你要把這個系統應用到自己的工作流程，你會從哪個步驟開始？為什麼？

提示：考慮你目前最大的痛點是什麼——是內容創作、格式轉換、還是發布排程？

Voice Preservation Framework (保留聲音框架)

這是整個系統的靈魂。AI 產出的內容最大的問題就是「聽起來像 AI」，失去個人特色。

如何讓 AI 保留你的聲音？

  • 餵食範例：在 prompt 中提供你過去寫得最好的貼文當範例
  • 定義風格規則：明確告訴 AI 你的語氣（幽默/嚴肅）、用詞習慣、句式偏好
  • 禁止清單：列出你絕對不會用的詞彙或表達方式
  • 最後人工審核：把 AI 當助手而非替身，最後一定要過目
