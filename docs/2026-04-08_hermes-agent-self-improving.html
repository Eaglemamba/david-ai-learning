Hermes Agent 是 Nous Research 開發的開源自我進化 AI Agent，核心賣點是內建學習閉環：透過持久記憶 (SQLite + FTS5) 和自動 Skill 生成 (Markdown Skill Files) 實現跨對話的能力累積。支援 200+ 模型（OpenRouter）、CLI/Telegram/Discord 多平台部署、MIT 授權完全開源。本文整理其架構概念、安裝流程、與 OpenClaw 的比較，並從 CDMO 營運視角評估其模式的可借鏡之處。

Part I: Hermes Agent 核心架構

Part I
自我進化 Agent 的四大支柱
Hermes Agent 最核心的設計理念是學習閉環 (Closed Learning Loop)：每次任務完成後，Agent 不是「歸零」重來，而是將經驗沉澱為可復用的資產。這與傳統 AI 工具「用完即棄」的模式根本不同。
"Core innovation lies in its built-in closed learning loop."
1. 持久多層記憶 (Persistent Multi-Layer Memory)：使用 SQLite + FTS5 全文搜索儲存對話歷史、使用者偏好、行為模式。LLM 自動摘要長期記憶，避免 Context Window 溢出。跨會話永久保存，不會「健忘」。
"Uses SQLite + FTS5 full-text search + LLM auto-summarization, permanently remembers your preferences, style and history across sessions."
2. 自動技能進化 (Auto Skill Evolution)：任務完成後自動生成 Markdown Skill 文件，下次遇到類似任務直接調用。更進一步，Agent 會自我迭代優化已有的 Skill。
"After task completion, automatically generates Markdown Skill files; will self-iterate to optimize existing Skills."
3. 自主執行力 (Autonomous Execution)：支援終端命令、瀏覽器操作、檔案處理、程式碼生成、Web 搜尋等工具鏈。可在 CLI 或 Telegram/Discord 等平台運行。
4. 模型靈活性 (Model Flexibility)：透過 OpenRouter 支援 200+ 模型，也支援 OpenAI、Anthropic、本地 Ollama，切換成本趨近於零。MIT 開源授權。
Analogy: Chef vs Recipe Book

傳統 AI Agent 像是每天到廚房都要重新看食譜的廚師 -- 昨天做過的菜今天完全不記得。Hermes Agent 像是一位會寫自己的食譜書的廚師：做完一道菜就記下來，下次不用從頭開始，還會根據經驗改良配方。

但注意：這位廚師的「食譜品質」取決於他用的食材（底層 LLM 能力）和廚房設備（工具鏈）。自動記錄不代表自動變好。

Unverified Claim Flag

原文宣稱：「記憶和能力指數級提升」

實際評估：指數級成長需要量化基準。Skill 文件累積是線性的（每完成一個任務 +1），記憶檢索效率取決於 FTS5 索引品質。「指數級」更可能是修辭誇張，而非技術描述。建議打折理解為「隨使用時間有明顯累積改善」。

CDMO Mapping: Deviation Investigation Memory

Hermes 的記憶 + Skill 自動生成模式，概念上類似你規劃中的 /triage-deviation 工作流：每次偏差調查完成後，將調查模式（root cause pattern、CAPA template）沉澱為可復用的 Skill。差異在於 GMP 環境需要 Copilot Mode（人工 100% 審核），不能讓 Agent 自主迭代 Skill 而不經驗證。

Pattern Recognition: Auto-Skill vs Manual-Skill

這裡的核心設計取捨值得關注：

  • 自動生成 Skill（Hermes 模式）：低摩擦、快速累積，但品質不可控 -- 垃圾任務也會生成垃圾 Skill
  • 手動管理 Skill（OpenClaw/Claude Code 模式）：品質可控、有人工審核，但累積速度慢

你的 Claude Code Skills 採用的是後者。Hermes 的自動模式在非 GMP 場景可能有效，但在需要驗證的環境（你的 CDMO 場景）中，自動生成未經審核的 SOP 是不可接受的。

Part IV: 架構模式的可借鏡之處

Part IV
從 Hermes 學到的設計思考
即使 Hermes 不適合直接採用，其架構中有幾個值得反思的設計模式：
1. 記憶的三層架構：短期（Context Window）→ 中期（Session Summary）→ 長期（SQLite 持久化 + FTS5 檢索）。這比單純的「把所有東西塞進 Context」更有結構感。你的 index.md routing target 設計已經朝這個方向走了。
2. Skill 的生命週期管理：Create → Use → Evaluate → Iterate。Hermes 自動化了整個流程，但缺乏品質關卡。理想模式可能是：自動 Draft + 人工 Review + 版本控制。
3. 遷移工具的設計思維：hermes claw migrate 支援 dry-run，這是良好的 DevOps 實踐。任何從系統 A 到系統 B 的遷移，都應該有預覽模式。
Key Insight: Auto-Draft + Human-Review Hybrid

結合 Hermes 和你現有的 Claude Code Skills 模式，最佳實踐可能是：

  • Step 1：任務完成後，AI 自動生成 Skill Draft（草稿）
  • Step 2：Knowledge Linting Level 1 (結構檢查) 自動通過
  • Step 3：Knowledge Linting Level 2 (LLM 語意檢查) 標記需人工審核的部分
  • Step 4：人工 Review + Commit + 版本控制

這保留了自動化的速度優勢，同時維持了品質關卡 -- 特別是在 GMP 環境中，這個 human-in-the-loop 是不可省略的。

GMP Lens: Copilot vs Agent Mode

Hermes 的自動 Skill 迭代本質上是 Agent Mode（自主運作，無人審核）。在 GMP 製藥環境中，這等同於讓 AI 自行修改 SOP 而不經過變更控制 -- 任何稽核員看到這種做法都會開 Critical Observation。

你的原則仍然正確：AI 在 GMP 環境中只能以 Copilot Mode 運作。Hermes 的模式在研究/個人項目中合理，但在受管制環境中需要加上審批層。

Key Takeaways

1. Self-Improving Loop is the Core Innovation
Hermes 的 Task → Skill → Memory → Recall 閉環是與傳統 Agent 的關鍵區隔。概念有效，但「指數級成長」的宣稱需打折理解。
2. Auto-Skill vs Manual-Skill is a Design Tradeoff
自動生成快但品質不可控，手動管理慢但可靠。最佳路徑：Auto-Draft + Human-Review 混合模式。
3. Hermes is Explore, Not Adopt
對已有 Claude Projects + Claude Code + GitHub 工具鏈的使用者而言，Hermes 的獨特價值被現有工具部分覆蓋。屬於 explore/demo 類別。
4. Source Quality Matters
X/Twitter 推文的促銷性語言和單方面比較，再次提醒：Source Discount 是知識管理的必要步驟。

Practice Questions

Q1 概念理解 Hermes Agent 的「自我進化」具體包含哪兩個機制？它們分別解決什麼問題？

兩個核心機制：(1) 持久多層記憶（SQLite + FTS5）-- 解決傳統 Agent 跨對話「健忘」的問題，讓 Agent 記住使用者偏好、歷史互動。(2) 自動 Skill 生成（Markdown Skill Files）-- 解決重複任務需要每次從頭設定的問題，任務完成後自動沉澱為可復用的能力模組。

關鍵區分：記憶解決的是「認識你」，Skill 解決的是「會做事」。兩者結合形成學習閉環。

Q2 產業應用 如果將 Hermes 的自動 Skill 生成模式應用到 CDMO 的偏差調查流程，需要加上什麼限制條件才能符合 GMP 要求？

至少需要以下限制：

(1) Human-in-the-Loop 審批：自動生成的 Skill（調查模板、CAPA 範本）必須經過 QA 審核後才能正式啟用，不得自動上線。

(2) 變更控制整合：每個 Skill 的建立、修改、廢除都要走變更控制流程（Change Control），有版本號和審核簽名。

(3) 驗證記錄：Skill 的使用結果必須可追溯（ALCOA+ 原則），AI 生成的建議和人工修改的差異要有 Audit Trail。

(4) 定期回顧：Skill 不能無限累積 -- 需要定期 Review 淘汰過時或不適用的模板，類似 SOP 的定期覆核。

Q3 批判思考 原文宣稱「很多人從 OpenClaw 轉投 Hermes」-- 這個說法有哪些可信度問題？你會如何驗證？

可信度問題包括：

(1) 樣本偏差：原文引用「社區共識」但未提供數據來源。X 上的討論可能只代表少數活躍用戶的意見。

(2) 比較偏向性：文章明顯站在 Hermes 一方，OpenClaw 的優勢（成熟生態、社區 Skill 庫）被輕描淡寫。

(3) 遷移工具的存在：hermes claw migrate 的存在恰恰說明 Hermes 是挑戰者，需要降低轉換門檻 -- 這通常是後來者的策略。

驗證方法：(a) 比較兩者的 GitHub Stars / Issues 活躍度趨勢；(b) 搜尋獨立第三方評測，而非只看官方或推廣者說法；(c) 在 Reddit、HN 等社區搜尋真實用戶反饋；(d) 實際安裝兩者做同一任務的 A/B 測試。

My Bottom Line
Core Insight
Self-Improving Agent 的核心價值不在於「自動化」本身，而在於經驗的結構化沉澱。Hermes 用 SQLite + Markdown Skill 實現了一種粗糙但有效的知識累積機制。我的 Claude Code Skills + Knowledge Linting 系統其實走的是同一條路，只是在品質控制上更嚴格。關鍵洞察：Auto-Draft + Human-Review 的混合模式可能是下一步優化方向。
Immediate Action
本週不需要安裝 Hermes（Removal Test 結果：explore 而非 adopt）。但可以在 Claude Code Skills 的工作流中加入一個實驗：讓 Claude Code 在完成 artifact 文件後，自動生成一份 Skill Draft 提案，放在 /drafts/ 資料夾待人工 Review。測試「Auto-Draft」概念是否能加速 Skill 建立。
