Skills, Explained: Why Folders of Markdown Files Are So Effective
Gabriel Chua / OpenAI DevRel | 2026-02-26
★★★★☆ 4.0/5 ↑ 實務應用, 學習曲線 ↓ 技術深度, 時效性
這篇文章由 OpenAI 開發者關係團隊的 Gabriel Chua 撰寫，解釋了 AI 領域中「Skill」的概念 — 本質上就是將反覆使用的 Prompt 和相關資源打包成可重複使用的資料夾結構。文章提出 Skill 相對於傳統 Prompt 的三大差異：Progressive Disclosure（漸進式揭露）、Structure（結構化）、和 Access to Additional Resources（資源整合）。概念清晰、比喻恰當，但技術深度偏淺，更適合作為入門概覽。對於已有 Claude Projects Skill 建置經驗的讀者，本文的最大價值在於提供一個系統化的思維框架來理解自己正在做的事。

Part I：Skill 的本質 — 讓 Prompt 可重複使用
PART I
從「大致能用」到「穩定產出」
大多數使用 AI 工具的人，都曾經有過一個「大致能用」的 Prompt。可能是一份檢查清單、一個工作流程、或是一組規則。你調整了幾次直到輸出看起來對了，然後存起來以便下次使用。
"Most people who use AI tools have, at some point, ended up with a prompt that 'mostly works.'"
隨著時間推移，這個 Prompt 慢慢改變。你在某次出錯後加了一條規則，刪掉一行看似不再需要的指令。有人把它複製到不同專案並稍作修改。最終，同一個流程有三四個版本在流傳，沒人確定哪個版本才能產出你真正想要的結果。
"Eventually, there are three or four versions of the same process floating around, and no one is quite sure which one produces the output you actually want."
Skill 本質上就是將這個習慣正式化的方式。核心來說，Skill 就是寫在 Markdown 檔案裡的 Prompt 集合，有時附帶額外的參考檔案或腳本。
"A skill is essentially a way of formalizing that habit. At its core, a skill is just a collection of prompts written down in a markdown file."
核心概念
Skill = Prompt + 資源 + 結構

傳統 Prompt 是「一次性指令」，Skill 是「可重複使用的工作流程包」。差異不在於技術能力，而在於組織方式：把指令放進資料夾、搭配範例和腳本，讓 Agent 可以當作可重複的流程來執行。
比喻說明
想像你是一個主廚。Prompt 就像口頭告訴助手「做一道番茄義大利麵」— 每次說法可能不同，結果也不穩定。Skill 則是把食譜寫成標準作業程序（SOP），放在固定位置，任何助手都能照著做出一致的成品。
重點提示
作者特別強調：Skill「不是根本上不同於 Prompting — 它就是 Prompting，只是做成可重複使用的形式」。這個定位很重要，因為它降低了學習門檻：你不需要學新技術，只需要更好地組織你已經在做的事。
Part II：三大差異 — Progressive Disclosure
PART II
漸進式揭露：只載入需要的指令
第一個概念差異是漸進式揭露（Progressive Disclosure）。你通常不會一開始就把整個工作流程載入模型的上下文。每個 Skill 都有一段簡短描述，說明它的用途。Agent 可以看這段描述來判斷該 Skill 是否與當前任務相關，只有確定相關時才會打開完整指令。
"You are usually not loading the entire workflow into the model's context from the beginning. Only when it determines that the skill applies does it inspect the full instructions."
實際運作起來，這就像是一本可用工作流程的目錄。Agent 知道有哪些工具可用，但只在需要時才打開詳細指令。
"In practice, this behaves like a glossary of available workflows."
這有點像去圖書館。你先從目錄看某個主題有哪些書。找到相關的書後，你不會讀完整本 — 你翻到索引或詞彙表，找到你需要的確切章節或頁面。詳細指令在真正需要之前都保持關閉。
"It's a bit like going to a library. You start with the catalogue to see what books exist on a topic."
比喻說明
圖書館比喻：目錄（Skill 描述）→ 找到對的書（判斷相關性）→ 翻到對的章節（載入完整指令）。這個設計的核心好處是節省 Context Window，讓模型不需要一次處理所有工作流程的全部指令。
實務應用
你的 Master Prompt 系統已經在實踐這個概念。

Claude Projects 的系統 Prompt 就是 Skill 描述層，而你的 v1.4.3 Master Prompt XML 就是完整指令層。當你開新對話時，Claude 先讀描述決定要怎麼處理，然後才根據需要展開特定區塊的指令。差異在於：你目前是「全部載入」模式，而 Skill 架構支援「按需載入」。
核心概念
Context Window 效率：Progressive Disclosure 的實際效益是減少每次對話消耗的 Token 數。如果你有 10 個 Skill 但只需要 1 個，傳統做法會載入全部 10 個的指令；Skill 架構只載入 1 個的完整指令 + 其他 9 個的簡短描述。
Part III-A：Structure — 結構即語義
PART III-A
資料夾結構本身就傳達用途
第二個差異是結構（Structure）。現代模型非常擅長遍歷檔案系統。某個東西存在於特定資料夾這件事本身就傳達了它應該如何被使用的資訊。就像你組織自己桌面的方式 — 按專案或用途分類 — Skills 目錄的結構就為模型提供了上下文。
"Modern models are quite good at traversing file systems. The fact that something lives in a particular folder already conveys information about how it should be used."
這也引入了模組化（Modularity）。Skill 成為一個打包好的工作流程單元，可以重複使用、共享、版本控制、或與其他 Skill 組合。你不再需要把同一個 Prompt 的微調版本複製貼上到每個專案，而是把工作流程當作模組來管理，在專案和團隊成員之間分發。
"A skill becomes a packaged unit of workflow that can be reused, shared, versioned, or composed with other skills."
核心概念
資料夾結構 = 隱含的 Prompt

當模型看到 /skills/docx/SKILL.md，光是路徑就告訴它「這是用來處理 Word 文件的工作流程」。這和人類組織桌面的邏輯相同 — 你不需要額外說明，位置本身就是說明。
重點提示
模組化的最大好處是「更新一次，全局生效」。 當流程改變時，你只需要更新一個 Skill 檔案，而不是要求每個人各自調整自己的 Prompt。這對團隊協作特別重要 — 消除了「哪個版本才是對的？」的問題。
Part III-B：Resource Access — 不只有指令，還有工具
PART III-B
Skill 可以攜帶執行所需的一切資源
第三個差異是存取額外資源的能力。傳統 Prompt 通常局限於你記得貼進去的內容。而 Skill 可以包含附錄、範例、腳本、評分標準，甚至是模型在執行任務時可以使用的可執行工具。包括呼叫 MCP Server（模型上下文協定伺服器）、執行 CLI 腳本、或引用內部指南。
"A skill can include appendices, examples, scripts, evaluation rubrics, or even executable tooling. This might include calling MCP servers, running a CLI script, or referencing internal guidelines."
從這個角度看，Skill 不只是告訴模型該做什麼，還給它執行任務所需的支援材料。工作流程不僅建立在指令之上，更建立在上下文和工具之上。
"The workflow becomes grounded not only in instructions, but in context and tools."
實務應用
Claude Projects 的 Skill 結構映射：

你目前使用的 Claude 環境就有完整的 Skill 實作：/mnt/skills/public/docx/、/mnt/skills/public/pptx/ 等。每個資料夾是一個獨立模組，包含 SKILL.md（主指令）和輔助腳本。這正是作者描述的三層架構的完整實現。
比喻說明
傳統 Prompt 像是口頭交代任務：「幫我做個報告」。Skill 則像是交付一個任務包：裡面有指令書、參考範例、品質標準、甚至可以直接使用的工具。接到任務包的人（或 Agent）不需要再到處找資源，一切所需都在包裡。
Part IV：團隊應用 — 隱性知識的顯性化
PART IV
從個人習慣到組織資產
組織內部被視為「專業」的很多東西，不是一組事實，而是一系列習慣和檢查步驟。知道部署前要驗證什麼、什麼假設需要測試、什麼樣的輸出才算合格。
"Much of what counts as expertise inside an organization is not a set of facts but a sequence of habits and checks."
通常這些知識是非正式傳遞的，或是埋在過時的文件裡。透過 Skill，這些程序性知識可以打包成 Agent 可以直接遵循的東西。每個人最終都在執行相同的步驟，新成員不必從頭逆向工程整個流程。
"Normally this knowledge is passed around informally or buried in outdated documentation. With skills, that procedural knowledge can be packaged."
Skill 並沒有擴展模型能做的事，但它們讓你更容易以可預測的方式執行可重複的任務。如果你發現自己重複同一個流程超過幾次，這通常就是可以轉化為 Skill 的信號。
"Skills do not expand what the model is capable of doing, but they make it easier to perform repeatable tasks in a predictable way."
實務應用
CDMO 情境直接對應：

在製藥工廠裡，「隱性知識」的例子比比皆是：偏差調查時該先查什麼、Tech Transfer 的關鍵檢查點、批次記錄審查的常見陷阱。這些目前大多靠資深人員口耳相傳。如果能打包成 Skill，新進人員或 AI 助手就能直接參照標準流程，而不是每次都要從頭摸索。
比喻說明
Tacit → Explicit 的轉化：這就像把一位資深工程師腦中的「直覺」寫成 SOP。直覺本身無法複製，但如果你能把「當你看到 X 現象時，先檢查 Y，然後確認 Z」這樣的決策路徑寫下來，它就變成可複製的組織資產。
重點提示
作者的結論非常精準：Skill 引入的不是新能力，而是把隱性流程變成顯性的、可分享的、可一致執行的東西。這和 GMP 環境中「如果沒寫下來，就等於沒做」的精神完全一致。
重點整理
Skill = 結構化的 Prompt 重用
Skill 不是新技術，而是把你已經在做的 Prompting 工作組織成可重複使用的模組。核心元素：skill.md 主指令 + 附帶資源 + 資料夾結構。
Progressive Disclosure 節省 Token
Agent 先讀簡短描述判斷相關性，只在確認需要時才載入完整指令。這減少了 Context Window 的浪費，類似圖書館先查目錄再翻書的邏輯。
資料夾結構本身就是語義
模型擅長遍歷檔案系統。東西放在什麼資料夾、怎麼命名，本身就傳達了用途資訊。模組化設計讓 Skill 可以版本控制、共享、和組合使用。
隱性知識的顯性化工具
團隊的真正專業往往是「習慣和檢查步驟」而非事實。Skill 讓你把這些程序性知識打包，消除「哪個版本是對的」問題，新成員也能直接上手。
練習題
Q1 概念理解 Progressive Disclosure 對 Context Window 管理的具體影響是什麼？如果沒有這個機制，會產生什麼問題？

Progressive Disclosure 讓 Agent 只在確認某個 Skill 與當前任務相關時，才載入該 Skill 的完整指令。如果沒有這個機制，所有 Skill 的完整指令都會被一次性載入 Context Window，導致：(1) Token 消耗大幅增加，推高 API 成本；(2) 不相關的指令可能干擾模型對當前任務的理解；(3) 在 Context Window 有限的情況下，真正需要的指令可能因為窗口滿了而被截斷。這就像開會時把所有部門的 SOP 都印出來擺在桌上 — 大部分是噪音，反而讓你找不到真正需要的那一份。

Q2 產業應用 在 CDMO 的 GMP 環境中，哪些「隱性知識」最適合轉化為 AI Skill？請舉出三個具體例子並說明理由。

(1) 偏差調查的初步分類邏輯：資深 QA 在看到偏差描述時，會根據經驗快速判斷是 Critical/Major/Minor，以及應該先檢查哪些系統（環境監控、設備校驗、人員訓練記錄）。這個決策路徑可以寫成 Skill，讓 AI 輔助做初步分類建議。

(2) 批次記錄審查的常見問題清單：每個 CDMO 都有自己的「經驗法則」— 哪些欄位最容易填錯、哪些計算最容易出誤差、哪些簽名最容易漏。這些可以打包成審查 Skill。

(3) Tech Transfer 的文件對照檢查：接收新客戶的製程時，需要比對原廠文件和內部 SOP 的差異。這個對照流程有固定的檢查維度（設備規格、環境條件、關鍵製程參數），適合結構化為 Skill。

這三個例子的共通點：都是「重複性高、有固定模式、但需要經驗才能做好」的工作 — 正是 Skill 最能發揮價值的場景。

Q3 批判思考 作者認為 Skill「不擴展模型的能力」，你同意嗎？在什麼情境下，Skill 可能實際上「擴展」了模型能做到的事？

作者的說法在技術層面是正確的 — Skill 沒有改變模型的底層能力。但在實務層面，這個說法值得挑戰。當一個 Skill 整合了 MCP Server、CLI 腳本、和外部 API 時，它實際上讓模型「能做到」之前做不到的事（例如：直接操作資料庫、生成特定格式的文件、或呼叫外部服務）。此外，一個寫得很好的 Skill 可以透過精確的 Prompt 結構，讓模型在特定任務上的表現品質遠超過泛用 Prompt — 這是否算「擴展能力」取決於你如何定義「能力」。如果把「能力」定義為「在實際工作場景中能穩定產出合格結果」，那 Skill 確實擴展了模型的能力邊界。

我的行動結論
核心洞察
我的 Master Prompt 系統本質上就是一個大型 Skill，但目前是「全量載入」模式。這篇文章讓我意識到，Progressive Disclosure 不只是效率問題，更是可擴展性問題 — 當 Skill 數量從 1 個成長到 10 個、50 個時，沒有分層機制就會撞到 Context Window 的天花板。
立即應用
本週可以做的事：盤點目前所有 Claude Projects 中反覆使用的 Prompt 模式，列出哪些已經是「Skill 等級」的工作流程（如 Master Prompt、文件自動化），哪些還停留在「大致能用」的階段。為後者建立 skill.md 的初步結構。
需要補充
文章偏重概念解釋，缺乏實作指南。特別缺少：(1) skill.md 的最佳寫作模板；(2) 如何設計 Skill 描述讓 Agent 的分類判斷更準確；(3) GMP 環境中使用 Skill + MCP 的合規考量（Audit Trail、Data Integrity）。後續需要參考 OpenAI Codex 的實際 Skill 範例來補充。
