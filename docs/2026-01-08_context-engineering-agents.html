Context Engineering for AI Agents

AI Agent 的上下文工程：從 Prompt 到 Context 的範式轉移

Anthropic Engineering Blog | 2025年9月29日發布
Agent LLM Prompt Engineering Tool Design
Executive Summary 快速摘要 閱讀時間：15-20 分鐘
Core Concept 核心概念 Context Engineering（上下文工程）是 Prompt Engineering 的自然演進。重點從「如何寫好指令」轉變為「如何策展最佳的 Token 集合」。核心原則：用最小、最高訊號的 Token 集合來最大化達成預期結果的機率。
Practical Value 實務價值 解決 AI Agent 在長時間任務中的「Context Rot（上下文腐敗）」問題。提供三大技術：Compaction（壓縮）、Structured Note-Taking（結構化筆記）、Sub-Agent Architectures（多代理架構），可達 54% 的效能提升。
Key Points 關鍵要點 Context 是有限資源，具有遞減邊際效益。LLM 有「注意力預算」限制。Just-in-Time（即時）檢索優於預先載入所有資料。工具設計應最小化功能重疊，避免模糊的決策點。
Action Items 行動建議 1) 審視現有 System Prompt，找到「正確高度」的指令層級；2) 評估工具集是否有功能重疊或模糊性；3) 考慮從 Pre-computed RAG 轉向 Hybrid Retrieval 策略；4) 對長時間任務導入 Compaction 機制。

Part 1 of 4 第一部分：核心概念
Original Content 原文

Context engineering refers to the set of strategies for curating and maintaining the optimal set of tokens (information) during LLM inference, including all the other information that may land there outside of the prompts.

Prompt Engineering: Writing instructions for single tasks. Key focus: "What to say"

Context Engineering: Managing all tokens (instructions + tools + data + history) across multi-turn interactions. Key shift: "What information to include"

The engineering problem at hand is optimizing the utility of those tokens against the inherent constraints of LLMs in order to consistently achieve a desired outcome.

Commentary 解說
Concept 核心概念

Context（上下文）= 每次 LLM 推論時包含的所有 Token 集合

Engineering（工程）= 在 LLM 固有限制下，優化這些 Token 的效用

這不只是「寫好提示詞」，而是「策展最佳資訊組合」的系統性方法。

Analogy 比喻說明

想像你是一位製藥廠的 QC 主管，要訓練新進分析員執行 HPLC 檢測：

Prompt Engineering = 寫一份 SOP，告訴他「如何操作這台儀器」

Context Engineering = 設計整個訓練系統：SOP + 過往異常案例 + 規格標準 + 即時查詢管道 + 筆記機制，確保他在任何情況都能做出正確判斷

Key Point 重點提示

範式轉移的關鍵：從「單次互動」到「多輪對話」

當 Agent 在迴圈中執行任務時，每一輪都會產生更多「可能相關」的資料。Context Engineering 就是不斷策展這些資訊的藝術與科學。

Part 2 of 4 第二部分：Context Rot 問題
Core Problem: Context Rot

More tokens = worse recall/accuracy

Context is finite with diminishing returns.

LLMs have "attention budget" that depletes.

Guiding Principle:

Smallest set of high-signal tokens for desired outcome

Studies on needle-in-a-haystack-style benchmarking have uncovered the concept of context rot: as the number of tokens in the context window increases, the model's ability to accurately recall information from that context decreases.

Commentary 解說
Concept Context Rot 上下文腐敗

定義：隨著 Context Window 中的 Token 數量增加，模型準確回憶資訊的能力下降。

技術原因：Transformer 架構讓每個 Token 都要「注意」其他所有 Token，產生 n 平方的配對關係。Context 越長，注意力越分散。

Analogy 比喻說明

就像人類的工作記憶容量有限：

你能同時記住 7 正負 2 個項目。如果會議中有人一次丟給你 50 個待辦事項，你會開始遺漏重要的細節。

LLM 的「注意力預算」也是如此，每新增一個 Token 都會消耗這個預算。

Practical 實務應用：製藥 QC

在設計 AI 輔助的 Stability Testing 系統時：

不要把所有歷史數據都塞進 Context。

而是只提供「當前批次相關」的規格、方法、趨勢摘要。

需要歷史數據時，讓 Agent 用工具「即時查詢」。

Part 3 of 4 第三部分：有效 Context 的組成
Effective Context Components

System Prompts:

  • Simple, direct language at right altitude
  • Use XML tags or Markdown sections
  • Start minimal, add based on failures
  • No hardcoded logic or vague guidance

Tools:

  • Token-efficient, self-contained, minimal overlap
  • Descriptive parameters, clear purpose
  • No bloated tool sets or ambiguous choices

Examples:

  • Curate diverse, canonical examples
  • No edge case laundry lists
Commentary 解說
Key Right Altitude 正確高度

太低（過度具體）：硬編碼複雜的 if-else 邏輯，脆弱且難維護

太高（過度模糊）：「請做好工作」這種指令，假設 AI 知道你的上下文

剛好：具體到能引導行為，但保留彈性讓模型運用判斷力

Compare 工具設計原則

最常見的失敗模式：工具集膨脹，功能重疊，決策點模糊。

檢驗標準：如果一個人類工程師無法明確說出「在這種情況下該用哪個工具」，那 AI Agent 也不可能做得更好。

Practical CDMO 應用案例

設計 Tech Transfer 輔助 Agent 的工具時：

好的設計：search_batch_records(batch_id) 與 get_process_parameters(product_code) 功能明確分離

壞的設計：get_data(type, id) 這種萬用工具，Agent 不知道何時該用什麼參數

Part 4 of 4 第四部分：檢索策略與長期任務
Retrieval Strategies

Just-in-Time (JIT):

  • Maintain identifiers (paths, links)
  • Load data dynamically with tools
  • Progressive discovery
  • Trade-off: Slower but efficient

Hybrid (Recommended):

  • Critical data upfront
  • JIT exploration as needed
  • Example: Claude Code

Long-Horizon Techniques:

Task TypeTechnique
Back-and-forthCompaction
Iterative devNote-taking
Research/analysisSub-agents
GeneralHybrid retrieval
Commentary 解說
Concept 三大長期任務技術

1. Compaction（壓縮）：對話接近 Context 上限時，摘要內容後重啟新視窗。保留關鍵決策、未解決問題、實作細節。

2. Structured Note-Taking（結構化筆記）：Agent 定期將筆記寫入 Context 外的持久化記憶體（如 NOTES.md），之後再拉回。

3. Sub-Agent Architectures（子代理架構）：主 Agent 協調高層計畫，專門的子 Agent 處理深度技術工作，只回傳濃縮摘要（1000-2000 tokens）。

Analogy 比喻說明

人類認知模式：我們不會記住整個資料庫，而是建立檔案系統、書籤、收件匣等外部索引系統，按需檢索。

Claude Code 的做法：維護輕量級識別符（檔案路徑、查詢、連結），用 glob 和 grep 等工具在執行時動態載入資料。

Practical 決策框架

對話式任務（大量來回） → 使用 Compaction

迭代開發（有明確里程碑） → 使用 Note-Taking

複雜研究（平行探索有價值） → 使用 Multi-Agent

一般任務 → Hybrid Retrieval（部分預載 + 按需探索）

Key Takeaways 重點整理

Context = 有限資源

Token 越多不代表越好。LLM 有「注意力預算」，遞減邊際效益是真實存在的限制。

最小高訊號集合

核心原則：找到最小的高訊號 Token 集合來最大化預期結果的機率。質量優於數量。

Just-in-Time 檢索

從「預先載入一切」轉向「維護識別符 + 按需載入」。像人類一樣用索引系統而非記憶全部。

做最簡單有效的事

隨著模型能力提升，最佳建議仍是「do the simplest thing that works」。別過度工程化。

Practice Questions 練習題

Q1概念理解一位工程師在設計 AI Agent 時，把所有可能相關的歷史數據、規格文件、案例庫都預先載入 Context Window。根據 Context Engineering 的原則，這種做法有什麼問題？應該如何改進？

問題分析：

1. Context Rot 風險：Token 數量增加會降低模型準確回憶資訊的能力，更多資料反而讓 Agent 表現更差。

2. 注意力預算耗盡：LLM 的注意力機制有限，每個 Token 都會消耗預算，可能導致關鍵資訊被忽略。

3. 違反核心原則：Context Engineering 的指導原則是「最小高訊號 Token 集合」，而非「越多越好」。

改進方案：

1. 改用 Hybrid Retrieval 策略：只預載最關鍵的資料（如當前任務的規格），其他用工具按需查詢。

2. 維護輕量級識別符（檔案路徑、ID），讓 Agent 在需要時才載入完整內容。

3. 對長時間任務，導入 Compaction 機制定期壓縮歷史對話。

Q2實務應用（CDMO 場景）你正在為 Aseptic Fill/Finish 產線設計一個 AI Agent，協助操作員進行偏差調查（Deviation Investigation）。根據本文的 Context Engineering 原則，你會如何設計這個 Agent 的 Tool Set？請考慮「正確高度」和「避免模糊決策點」的原則。

Tool Set 設計建議：

1. 功能明確分離的工具：

  • search_deviation_history(product_code, deviation_type) - 查詢同產品/同類型的歷史偏差
  • get_batch_record(batch_id) - 取得特定批次的生產記錄
  • get_em_data(room_id, date_range) - 取得環境監測數據
  • check_equipment_status(equipment_id) - 查詢設備校驗/維護狀態
  • get_sop(sop_number) - 取得相關 SOP 內容

2. 避免的設計：

  • 不要用 get_data(type, params) 這種萬用工具
  • 不要把「查詢」和「寫入」功能合併在同一工具
  • 每個工具的用途要清晰到人類工程師能明確判斷何時使用

3. System Prompt 高度：

提供偏差調查的流程框架（如 5 Why、魚骨圖結構），但不要硬編碼「如果 EM 超標就一定是什麼原因」這種邏輯。讓模型根據查詢到的資料做判斷。

Q3批判思考文章提到「do the simplest thing that works」是最佳建議。但在高度監管的製藥環境中（需要 21 CFR Part 11 合規、完整審計軌跡），這個原則是否仍然適用？如何平衡「簡單」與「合規」？

原則仍然適用，但需要重新定義「works」：

在製藥環境中，「works」的定義必須包含合規要求。一個不符合 GMP 的簡單方案根本不算「work」。

平衡策略：

1. 分層設計：

  • AI Agent 本身遵循「最簡單有效」原則
  • 但外層包裹合規層（Audit Trail、電子簽章、權限控制）
  • 這些合規功能是「必要的複雜度」，不是「過度工程化」

2. Context Engineering 的合規應用：

  • Compaction 時必須保留關鍵的審計資訊（who, what, when）
  • Note-Taking 的持久化儲存需要符合 Data Integrity 要求（ALCOA+）
  • Tool 設計需要包含適當的輸入驗證和錯誤處理

3. 關鍵洞察：

「簡單」指的是「不要過度複雜化 AI 邏輯」，而非「省略必要的控制」。在製藥 AI 應用中，最簡單有效的方案是讓 AI 專注於輔助決策，最終驗證和批准仍由受過訓練的人員執行。
