Executive Summary

Andrej Karpathy 於 2026 年 3 月釋出 autoresearch，一個僅 630 行 Python + 40 行 Markdown 的極簡框架，讓 AI Agent 在單顆 GPU 上自主進行機器學習實驗。核心理念：人類寫研究策略（program.md），Agent 執行局部搜索（train.py）。5 天內超過 32K GitHub stars，Shopify CEO Tobi Lutke 用同一模式一夜之間讓 0.8B 模型超越手動調校的 1.6B 模型（+19%），更進一步將 Liquid 模板引擎效能提升 53%。本文深度拆解 repo 架構、program.md 設計哲學，並連結到 CDMO 環境中的 Feedback Loop 應用。

Part I
Karpathy 是誰？為什麼他做的東西值得認真看
Andrej Karpathy 是 OpenAI 早期成員之一，曾在 Tesla 主導自駕 AI 團隊，後來做了大量對開發者影響深遠的 AI 教學與開源專案：nanoGPT、micrograd、nanochat 等。他不只是研究者，更是少數能同時橋接研究與工程落地的人。
Former co-founder of OpenAI, former Director of AI at Tesla. Creator of nanoGPT, micrograd, nanochat — projects that bridged research and engineering practice.
所以這次他釋出 autoresearch，不只是又一個 AI demo，而是一個深刻理解模型訓練、研究流程和工程落地的人，親自示範 AI 參與研究本身的最小可行架構。repo 發布 5 天內達到 32.6K stars，成為 2026 年成長最快的 GitHub 專案之一。
The repo hit 32,600+ GitHub stars in five days, becoming one of the fastest-growing repositories of 2026. Not just a demo — a minimal viable architecture for AI participating in research itself.
📚 核心概念
為什麼 Karpathy 的背景很重要？
autoresearch 不是一個理論概念，而是一個已經被創建者自己驗證過的系統。Karpathy 在自己手動優化過的 nanochat 上跑 autoresearch，Agent 仍然找到了他遺漏的改進——包括 QK-Norm 缺少 scalar multiplier 這個已經存在很久的 bug。這代表 Agent 不是靠運氣，而是靠耐心和覆蓋率取勝。
💡 比喻說明
想像你是一位資深藥師，每天親手檢查 100 批次的批次紀錄。你很厲害，但注意力是有限的。autoresearch 的邏輯是：如果我給一個不會累的「助手」一套明確的檢查規則、一個固定的評估標準，讓它 24 小時不停檢查，它不需要比你聰明——它只需要比你有耐心。
Part II
三檔案極簡架構：為什麼「小」是設計目的
整個 repo 最重要的只有三個檔案：prepare.py（固定常數、資料準備、tokenizer、evaluation）、train.py（模型架構、optimizer、訓練迴圈——Agent 唯一可改的檔案）、program.md（給 Agent 的研究指令——Human 唯一需改的檔案）。
The repo is deliberately kept small and only really has three files that matter: prepare.py (fixed), train.py (agent edits), program.md (human edits).
這個設計不是因為 Karpathy 偷懶，而是刻意為之。整個 codebase 只有約 630 行 Python，剛好能塞進一個現代 LLM 的 Context Window。這意味著 Agent 可以一次讀完全部程式碼，在腦中保持「完整理解」，而不需要 RAG 或複雜的工具鏈。
The ~630-line constraint ensures that the entire codebase fits within the context window of modern LLMs, minimizing errors and allowing the agent to maintain a holistic understanding.
Single file to modify：Agent 只能改 train.py。這讓改動範圍可控、diff 可 review。對比過去那些讓 Agent 在整個 codebase 自由修改的嘗試，這個約束反而是讓 Agent 能力最大化的關鍵。
The agent only touches train.py. This keeps the scope manageable and diffs reviewable.
💻 專案結構
⚠️ 重點提示
Context Window Fit 設計原則
整份 train.py（~630 行）刻意控制在 LLM 一次能讀完的大小。這不是巧合。如果 codebase 太大，Agent 就需要 RAG、需要 retrieval、需要分段理解——每一層都會引入錯誤和延遲。autoresearch 的極簡正是為了讓 Agent 的推理品質最大化。這跟 Context Engineering 的核心原則完全一致：精準和相關性比大量資訊更重要。
🛠️ CDMO 映射
這個「只准改一個檔案」的約束，跟 GMP 環境的變更控制邏輯高度相似。你不會讓一個新進人員同時改 SOP、設備參數和原料規格——你會把可變範圍限縮到最小，其他東西都鎖住。autoresearch 對 Agent 做的事完全一樣。
prepare.py # 固定：資料準備 + runtime 工具（不可改） train.py # Agent 修改：模型、optimizer、訓練 loop program.md # Human 修改：Agent 的研究指令 pyproject.toml # 依賴管理
Part III
program.md 導讀：寫給 Agent 的研究協定
program.md 是整個 repo 真正的核心。它不是程式碼，而是一份寫給 Agent 的純文字研究協定（research protocol）。The New Stack 的分析指出，這份 Markdown 同時承載了三種溝通：指令（Agent 應搜索什麼）、約束（什麼不能改）、停止條件（什麼時候結束並報告）。
program.md simultaneously carries three registers of communication: instructions (what the agent should search for), constraints (what must not change), and stopping criteria (when the loop should wrap up and report).
Setup 階段：Agent 先建立實驗 branch（例如 autoresearch/mar5），讀取三個核心檔案以建立完整理解，確認資料目錄存在，初始化 results.tsv，然後等人類確認後開始。
Create the branch from current master. Read the in-scope files for full context. Verify data exists. Initialize results.tsv with just the header row. Confirm and go.
Experiment Loop：每次實驗是一個固定流程——(1) 修改 train.py (2) 執行 uv run train.py > run.log 2>&1（重點：把輸出重導到 log，不要灌進 Agent 的 context）(3) 用 grep 讀取 val_bpb 結果 (4) 記錄到 TSV (5) 改善就 git commit，沒改善就 git reset。
Tune train.py with an experimental idea. Run the experiment with output redirected. Read out the results via grep. If val_bpb improved, keep the commit. If equal or worse, git reset back.
NEVER STOP 指令：program.md 包含一條全大寫的核心指令——永不停止。一旦迴圈開始，Agent 不應暫停詢問人類任何事，必須持續跑到被手動停止。這不是對話，是一個無限研究迴圈。值得注意的是，Codex 無法遵守這個指令（會自行停止），而 Claude 能持續執行 12+ 小時、118 次實驗。
NEVER STOP. Once the loop begins, the agent must not pause to ask the human anything. It runs until manually killed. This is not a dialogue. It's an infinite research loop.
💻 program.md 核心流程
📚 核心概念
program.md = Agent Skill
Karpathy 自己說 program.md 本質上就是一個超輕量的「skill」。這跟 Anthropic 的 Agent Skills 架構異曲同工——都是用結構化文字定義 Agent 的行為邊界、工作流程和品質標準。差別在於 autoresearch 的 skill 只需要一份 40 行 Markdown，而 Anthropic 的 SKILL.md 有更完整的 YAML frontmatter 和生命週期管理。
💡 比喻說明
把 Agent 的 log 重導，不灌進 context
這個設計細節非常值得學習。想像你請一位助手做實驗，如果他每次實驗都把所有原始數據唸給你聽，你的注意力很快就會被耗盡。正確做法是：實驗結果存檔，只報告關鍵指標。> run.log 2>&1 就是在保護 Agent 的 context window 不被無關輸出污染。
# 1. Setup git checkout -b autoresearch/<tag> # 讀取 README.md, prepare.py, train.py # 確認 ~/.cache/autoresearch/ 資料存在 # 初始化 results.tsv # 2. Experiment Loop while true: # 修改 train.py（架構/超參數/optimizer） uv run train.py > run.log 2>&1 grep "^val_bpb:\|^peak_vram_mb:" run.log # 記錄到 results.tsv if val_bpb improved: git commit # keep else: git reset # discard
Part IV
四個設計決策：為什麼 Agent 在這裡突然變得有用
autoresearch 做了四件關鍵的事，讓原本「什麼都不太行」的 Agent 突然變得非常高效：
1. 只准改一個檔案：避免 Agent 在整個 codebase 亂跑。搜索空間被壓縮到一個 630 行的文件裡。

2. 固定 5 分鐘時間預算：不管 Agent 改了什麼（模型大小、batch size、架構），每次實驗都跑一樣久。這讓每次實驗可以直接比較，也意味著每小時約 12 次實驗，一整夜約 100 次。

3. 單一標量 Metric：val_bpb（validation bits per byte），越低越好，且不受 vocabulary size 影響。Agent 不需要做任何主觀判斷——數字低就是好的。

4. 失敗即回退：沒有改善就 git reset，模型永遠只會往好的方向走。這是一個 ratchet mechanism（棘輪機制）——只進不退。
Single file to modify. Fixed 5-minute time budget. One scalar metric (val_bpb). Git reset on regression — the model never goes backwards.
這四個約束把原本開放式的研究問題，轉化成 Agent 最擅長處理的東西：高頻迭代 + 明確回饋 + 可回退的局部優化。
The open-ended research problem becomes what agents handle best: high-frequency iteration + clear feedback + reversible local optimization.

⚠️ 重點提示
Git as Memory
autoresearch 用 Git 作為 Agent 的記憶系統。成功的改動是 commits，失敗的改動被 reset。Agent 可以用 git log 看過去什麼有效，用 results.tsv 看已經試過什麼。沒有向量資料庫、沒有 embeddings——就是 commits 和一個 TSV 檔。這種極簡的「記憶」設計值得借鏡。
Part V
實證結果：數字會說話
Karpathy 自己的 nanochat 測試：在已經手動優化過的 GPT-2 訓練碼上跑 autoresearch，兩天內 Agent 執行約 700 次實驗，找到約 20 個真正的改進。堆疊起來，Time-to-GPT-2 從 2.02 小時降到 1.80 小時（快 11%）。更關鍵的是，這些在小模型（depth-12）發現的改進，全部都能轉移到大模型（depth-24）。
Over two days, ~700 experiments, ~20 real improvements. Time-to-GPT-2 dropped from 2.02 to 1.80 hours (11% faster). All improvements transferred from depth-12 to depth-24 models.
Shopify CEO Tobi Lutke 的實驗：第一輪——用 autoresearch 模式優化 QMD（本地搜尋工具）的 query-expansion model。8 小時 37 次實驗後，0.8B 模型超越了手動調校的 1.6B 模型（+19% validation score）。小模型打敗了兩倍大的模型。
After 37 experiments in 8 hours, a 0.8B model scored 19% higher than the previous 1.6B hand-tuned model.
Tobi 第二輪——Liquid 模板引擎：把 autoresearch 模式套到 Shopify 用了 20 年的 Liquid Ruby 程式庫上。120 次自動化實驗、93 個 commits 後：parse+render 時間降低 53%，記憶體分配減少 61%。974 個單元測試全數通過，零回歸。這是一個被數百名貢獻者優化了 20 年的 codebase，Agent 在兩天內找到了人類漏掉的微優化。
53% faster combined parse+render time, 61% fewer object allocations on a 20-year-old codebase. Zero test regressions — all 974 unit tests pass.

🛠️ 實務應用
Liquid 案例的泛化意義
Liquid 的例子特別重要，因為它證明 autoresearch 模式不只適用於 ML 訓練。只要你的優化目標有：(1) 明確的 benchmark script (2) 完整的測試套件可確認無回歸 (3) 可量化的改善指標——Agent 就能在裡面跑 autoresearch loop。Tobi 自己總結：建一個 auto/ 資料夾，放一份 program.md 和一個 benchmark script，就能開始。
💡 批判視角
打個折扣：Tobi 自己承認 53% 的改善數字「可能有些 overfit」。Karpathy 的 700 次實驗中只有約 3% 是真正的改進。這些初期結果令人興奮，但都是小規模、非受控的。真正的問題是：在更複雜的系統上，這個模式是否同樣成立？答案目前未知。適用 2-3x 折扣原則。
Part VI
真正的範式轉移：從「親手做」到「設計做的系統」
很多人看 autoresearch 第一眼覺得：「AI 幫自己改 training code」。但真正重要的不是這個，而是背後的 interaction model 轉變：人類不再直接改所有 code，而是開始設計 Agent 的研究規則。
The real shift: humans no longer directly modify all code. Instead, they design the agent's research rules.
Karpathy 的原話：「你不是在編輯 Python 檔案。你是在 programming program.md——那些提供 context 給 AI agents、設定你的自主研究組織的 Markdown 檔案。」這代表高價值工作正在從「親手做」轉向「設計那個會自己做的系統」。
"You're not touching any of the Python files like you normally would as a researcher. Instead, you are programming the program.md Markdown files that provide context to the AI agents and set up your autonomous research org."
這個轉變適用範圍遠超 AI 研究。行銷、內容、銷售、營運、產品實驗——本質上都是一樣的模式：把任務縮小、把評估標準明確化、把流程標準化、讓 Agent 在裡面反覆迭代。
The pattern applies beyond AI research: marketing, content, sales, operations, product experiments. Narrow the task, clarify the metric, standardize the process, let the agent iterate.
📚 核心概念
O-Ring 理論的完美例證
autoresearch 精確示範了 O-Ring Bottleneck Theory：Agent 接管了可自動化的高頻迭代（局部搜索），而人類退到了不可替代的判斷節點——定義研究方向、設計評估指標、決定什麼「問題」值得問。真正的瓶頸不是「跑實驗」，而是「設計值得跑的實驗框架」。
🛠️ CDMO 映射
營運主管的新角色模型
這個轉變對 COO-track 的人特別有意義。未來的營運領導者不只是「最會做 SOP 的人」，而是「最會設計 Agent 能有效執行的 SOP 結構的人」。你的 program.md 等價於一份精確定義了目標指標、可改範圍、回退條件的 SOP——只不過執行者從人變成了 Agent。
Part VII
限制與 Goodhart's Law：Agent 會 Game 你的 Metric
autoresearch 不是萬能的，它有非常明確的邊界。目前優化的 val_bpb 是一個 proxy metric——而 proxy 被優化到極致時，不一定等於你真正想要的結果。
The metric you optimize is not necessarily the outcome you truly want. Goodhart's Law applies with particular force when the agent runs 100 experiments per night.
val_bpb 變好，不代表下游任務一定更好。loss 變好，不代表 sample quality 一定更好。局部最優，不一定等於真正的研究突破。Agent 做的是 bounded empirical research（有邊界的經驗性研究），不是會自己提出全新科學問題、重寫 evaluation paradigm 的 autonomous scientist。
It does bounded empirical research — not autonomous science that poses new questions or rewrites evaluation paradigms.
此外還有幾個實務限制：(1) 需要 NVIDIA GPU（有 Mac/Windows fork 但非官方）(2) 結果不可跨平台比較（因為 5 分鐘時間預算在不同硬體上跑出不同量的 training）(3) Agent 跑多了之後會「沒靈感」，開始做隨機改動 (4) 改進是堆疊的，很難拆解個別貢獻。
Requires NVIDIA GPU. Results not comparable across platforms. Agent runs out of ideas and resorts to random changes. Improvements are stacked, making individual contributions unclear.
⚠️ 重點提示
Goodhart's Law in Agentic Loops
「當一個指標變成目標時，它就不再是好的指標。」在 autoresearch 裡，Agent 以 100 次/夜的速度優化 val_bpb，Goodhart 效應的風險被放大。任何需要委員會才能判斷的 metric，或者優化 proxy 而非實際結果的 metric，都會被自主迴圈無情地「exploit」。這是設計 Agent 研究閉環時最需要警惕的事。
🛠️ CDMO 映射
Metric 選擇 = Deviation Triage 的核心
GMP 環境中的偏差分類也面臨同樣的問題：如果你用「關閉速度」作為唯一 KPI，Agent（或人）就會學會快速關閉偏差而不是深入調查根因。autoresearch 的啟示是：在設計任何 Agent 閉環之前，先確認你的 metric 真正代表你想要的結果。
⚖️ 適用 vs 不適用場景
適合 autoresearch 模式：訓練細節優化、hyperparameter 搜尋、效能 benchmark 改善、小型架構調整、有明確 pass/fail 的程式碼優化

不適合 autoresearch 模式：需要主觀判斷的品質評估、多目標權衡（速度 vs 準確度 vs 成本）、需要跨系統互動的複雜工作流、沒有穩定 feedback signal 的創意工作
Key Takeaways
重點整理
🔎
Feedback Loop > Model Power
Agent 的威力不在於它多像人，而在於你能不能把工作切成一個它可以持續獲得回饋的閉環。autoresearch 的核心創新不是模型本身，而是那個 5 分鐘 ratchet loop 的設計。
📝
program.md = 新型態的「程式設計」
未來有價值的不只是寫程式碼，而是寫「Agent 的研究協定」。program.md 同時承載指令、約束和停止條件——這是一種用自然語言做系統設計的新技能。
🔧
極簡設計是 Agent 成功的前提
630 行程式碼、1 個可改檔案、1 個 metric、5 分鐘時間預算。每一個約束都是在幫 Agent 聚焦。最可怕的不是 Agent 不夠聰明，而是搜索空間太大、feedback 太模糊。
⚠
Goodhart 警告：Metric 選擇是最高風險決策
Agent 會以 100 次/夜的速度優化你給的 metric。如果 metric 是 proxy 而非真正目標，你會得到一個完美優化了錯誤東西的結果。設計閉環的第一步是確認 metric 代表真正的品質。
Practice
練習題
Q1 概念理解 autoresearch 的 program.md 同時承載哪三種溝通功能？為什麼 YAML 或 JSON 無法取代 Markdown 扮演這個角色？

program.md 承載三種溝通：指令（Agent 應搜索什麼方向）、約束（什麼不能改）、停止條件（何時結束並報告）。

YAML 擅長編碼結構但缺乏敘事能力，JSON 是可執行格式但不具備策略表達力，Python 可執行但對非工程師不友善。Markdown 的獨特優勢在於它是人類可讀的敘事格式，能同時表達「為什麼」和「怎麼做」。LLM 原生理解 Markdown——它是目前最接近「人類意圖」的結構化表達方式。

Q2 產業應用 在你的 CDMO 環境中，如果要設計一個類似 autoresearch 的 Agent 閉環來優化偏差調查效率，你的 prepare.py / train.py / program.md 分別對應什麼？val_bpb 等價於什麼 metric？

一個可能的映射：prepare.py（不可改的固定環境）= 法規要求（21 CFR Part 211）、SOP 模板結構、偏差分類標準。train.py（Agent 可改的部分）= 偏差調查報告的撰寫模板、根因分析邏輯、CAPA 建議格式。program.md（Human 設計的研究指令）= 偏差分類規則、品質標準、回退條件。

val_bpb 的等價物：這是最困難的部分。理想 metric 應該是「偏差調查品質評分」——但這需要先建立一個可自動化評估的 rubric。可能的 proxy：調查結論被 QA 退回的比率（越低越好）、從偏差開案到根因確認的天數、CAPA 有效性追蹤分數。但任何 proxy 都有 Goodhart 風險——如果 Agent 學會用模板化語言快速通過 QA 審查而非真正深入調查，反而會降低品質。

Q3 批判思考 原文作者說「誰能把一件事切成 Agent 能穩定跑的閉環，誰就比較有機會先拿到效率紅利」。這個觀點有什麼盲點？在什麼情況下，建立閉環反而會造成負面效果？

這個觀點的最大盲點是假設所有有價值的工作都可以被切成閉環。但很多高價值工作的本質是「發現新問題」而非「優化已知問題」——而 autoresearch 模式只能處理後者。

建立閉環反而有害的情況：(1) Metric 錯誤——Agent 以每夜 100 次的速度往錯誤方向奔跑，比人類犯錯快得多 (2) 過早收斂——閉環找到局部最優後停滯，而真正的突破需要打破現有框架 (3) 機會成本——投入大量時間設計 Agent 閉環，但問題本身可能不值得這樣的投資 (4) 合規風險——在 GMP 環境中，自動化迭代若未經適當的變更控制和人類審查，可能違反法規要求。autoresearch 模式的價值是在 Copilot 層級，不是在 Autonomous Agent 層級。
