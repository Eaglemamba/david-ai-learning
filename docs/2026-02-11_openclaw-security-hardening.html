Executive Summary

★★★★☆ 4.2/5 ↑ 實務應用, 技術深度 ↓ 工具生態, 學習曲線

這篇文章是 OpenClaw（前身 ClawdBot）安全強化的完整實作指南，由 AI 安全專家 Fernando Lucktemberg 撰寫。文章的核心前提是「警告無法阻止採用」（Harm Reduction，危害減少策略）：既然人們已經在使用 OpenClaw，與其說「不要用」，不如教他們如何安全地用。

指南採用三層漸進式防護架構：Tier 1（基本隔離，3-4 小時）、Tier 2（主動監控，+2 小時）、Tier 3（深度防禦，+2-3 小時），涵蓋 32 個步驟，從 VPS 隔離、Tailscale VPN、防火牆、認證加密、到 Docker/Podman 容器化、LiteLLM 代理、Squid 網路出口過濾等。每個步驟都附有可複製的指令和驗證程序。

文章同時提供手動部署和 Ansible Playbook 自動化部署兩條路徑（手動 7-9 小時 vs 自動 15-30 分鐘），並坦承即使 Tier 3 也無法防禦 Prompt Injection 和供應鏈攻擊這兩個根本性弱點。

Content

Part I: 為什麼需要這篇指南

Wyndo 在前一篇文章介紹了 OpenClaw 的爆紅現象和基本安全建議，但他坦承沒有展示如何真正做到端對端的安全強化。他自己每天都在使用 OpenClaw 的三個 Agent（Morty、Goggins、Pepper Potts），其中 Pepper Potts 已經改變了他的工作方式——早上委派任務，Agent 自主運行，晚上回顧成果。

"Pepper Potts especially has changed how I work. I delegate tasks to it in the morning. It runs its own workflow while I run mine. I review its work at the end of the day and move projects forward."

正因為 OpenClaw 如此強大——它能存取你的 API Key、應用程式、檔案——安全問題才如此重要。預設情況下，這些敏感資訊以明文儲存在伺服器上。

"OpenClaw is powerful precisely because it has access to a lot. Your API keys. Your apps. Your files. And by default, some of that sits in plain text on whatever server you're running it on."

💡 比喻說明

想像你請了一位超級能幹的助理，但給了他你家的所有鑰匙、銀行密碼、和公司門禁卡。這位助理確實能幫你做很多事，但如果他被騙了（Prompt Injection），或者有人偷了他的通行證（Credential Leak），後果就不只是「助理出錯」，而是「你的整個數位生活被攤開」。這篇指南就是在教你：既然決定請這位助理，至少要分配不同的鑰匙給不同的任務，而且絕對不能給他銀行密碼。

📚 Harm Reduction 危害減少策略

Fernando 用了一個精準的比喻：「我知道這像在教人更安全地玩電鋸雜耍，而正確答案是『不要玩電鋸雜耍』。但他們已經在玩了。至少現在他們可能會戴手套。」

這反映了一個重要的安全現實：理想安全建議 vs 實際採用行為之間的落差。純粹的「不要用」建議在技術社群中往往無效。

Part II: NEVER 清單與可接受連線

⛔ 絕對不可連接 OpenClaw 的服務

無論安全做得多好，以下帳號永遠不應連接：主要電子郵件、銀行/金融服務、密碼管理器（1Password/Bitwarden）、工作帳號（Slack、Google Workspace）、不可替代歷史的社群媒體、加密貨幣錢包、政府/醫療入口、主要 GitHub 帳號。

判斷規則很簡單：連接到 OpenClaw 的每個帳號，都應該是你失去了也不會有重大影響的帳號。

"The rule: Every account connected to OpenClaw should be one you could lose without significant impact."

🛠️ 可接受的連線（僅限專用 Burner 帳號）
  • 專為 OpenClaw 通知建立的 Gmail
  • Telegram Bot 帳號（非個人 Telegram）
  • 專用於 AI 排程的行事曆
  • RSS 和新聞聚合器
  • 僅用於測試倉庫的 GitHub 帳號
  • 任何你能在一小時內重建的低風險服務
⚠️ 不可修復的限制

Prompt Injection：惡意電子郵件或文件可以操控 OpenClaw 執行指令。研究者 Matvey Kukuy 用一封精心設計的郵件在五分鐘內攻破了一個實例。

Supply Chain Attack：ClawdHub（技能註冊表）聲明「所有下載的程式碼都被視為受信任的程式碼——目前沒有審核流程。」

Tier 1  Part III: 基本防護 — 最低可行安全（Steps 1-14）

Tier 1 是每個人的必修。沒有「跳過基礎直接學進階」的選項。Tier 1 建立了所有後續步驟依賴的基礎隔離和設定。如果你跳過 Tier 1 的步驟，你運行的不是「強化過的 OpenClaw」，而是「有額外功能的暴露 OpenClaw」。

"Everyone must complete Tier 1. There is no 'I'll skip the basics and jump to advanced' option."

1設置隔離 VPS

物理隔離確保即使被攻破也不會蔓延到你的主機。建立專用的非 root 使用者、啟用 SSH 金鑰認證、關閉 root 登入和密碼認證。推薦 Hetzner CX23（月費 $3.49）。

"Physical isolation ensures compromise doesn't spread to your primary computer."

2設定防火牆

預設拒絕所有入站流量。絕對不要開放 port 18789 給公網。只允許 SSH（port 22）。

"Deny all incoming by default. Never open port 18789 publicly."

3安裝 Tailscale

安全、加密的遠端存取，不需要暴露任何埠。所有存取都透過 Tailscale VPN 的 100.x.x.x 地址。

"Secure, encrypted access without exposing ports to the internet."

7設定 Gateway（關鍵安全設定）

三個絕對規則：host 永遠是 127.0.0.1（不是 0.0.0.0）、dangerouslyDisableDeviceAuth 永遠是 false、mDNS 永遠關閉。trustedProxies 必須包含 Tailscale 的 100.64.0.0/10 範圍。

"host: '127.0.0.1': Only accept connections from localhost. 0.0.0.0 would expose to all interfaces including public internet."

⚠️ Step 9: 認證加密

預設的明文儲存讓認證資訊容易成為竊密軟體的目標。提供兩個加密方案：age（輕量級）或 pass（GPG-based）。核心是啟動時解密、使用後清除明文。

Tier 2  Part IV: 標準防護 — 主動監控（Steps 15-19）

Tier 2 是大多數業餘使用者的甜蜜點——在不需要成為安全工程師的前提下，提供堅實的安全防護。核心轉變是從 Denylist（黑名單）思維切換到 Allowlist（白名單）思維。

"This is the sweet spot for most hobbyists."

15工具白名單

黑名單失敗是因為攻擊者總能找到替代方案。白名單是窮舉的——如果沒有列出，就不能執行。只允許 ls、cat、grep 等安全的唯讀命令。

"Denylists fail because attackers find alternatives. Allowlists are exhaustive."

16MCP Server 安全設定

每個 MCP Server 都是以你 Agent 權限運行的程式碼。三條鐵律：絕不啟用 enableAllProjectMcpServers: true、永遠版本鎖定（不用 "latest"）、永遠關閉 autoUpdate。禁用高風險 Server：filesystem、shell、ssh、docker。

"Each MCP server is code running with your agent's permissions. Never use 'enableAllProjectMcpServers: true'. Always version pin. Always set 'autoUpdate: false'. Only enable servers you explicitly need."

17最小化 OAuth 範圍

最小權限原則：Gmail 用 readonly（不是 full）、GitHub 用 public_repo（不是 repo）。每月在 Google 和 GitHub 的權限頁面審查已連接的 OAuth 工作階段。

"Principle of least privilege. Only grant necessary permissions. Review connected OAuth sessions monthly."

Tier 3  Part V: 深度防禦 — 容器沙箱架構（Steps 20-32）

Tier 3 的核心是容器級隔離（Container-level Isolation）。Step 20 是整篇指南最重要的技術步驟：部署 OpenClaw + LiteLLM 在 Docker/Podman 沙箱中。

"Container-level isolation with LiteLLM as gateway. OpenClaw never talks directly to model providers."

20Docker/Podman 沙箱部署

架構的精妙之處在於網路隔離設計：OpenClaw 容器「完全沒有外部網路存取」，只能透過內部網路和 LiteLLM 通訊。LiteLLM 容器是唯一有外部 API 存取權的容器，負責注入真正的 API Key、速率限制和成本控制。OpenClaw 永遠看不到真正的 Anthropic API Key。

"OpenClaw container has no direct internet access. LiteLLM container is the only one with external API access. OpenClaw never sees real API keys."

Podman (rootless) vs Docker：如果容器逃逸（Container Escape），Podman 下攻擊者只是一個非特權使用者；Docker 下攻擊者直接是 root。推薦使用 Podman（rootless）。

"Podman (rootless) runs without a daemon and without root privileges. If an attacker escapes the container, they land as an unprivileged user."

29Squid Proxy 網域白名單

即使 OpenClaw 被攻破，它也只能連接到明確允許的網域。Squid Proxy 容器介於 OpenClaw 和網際網路之間，執行預設拒絕（deny-by-default）的出口過濾。白名單只包含必要的服務（github.com、npmjs.org 等）。

"Network egress filtering prevents data exfiltration. Even if OpenClaw is compromised, it can only reach explicitly allowed domains."

30細粒度 Exec Approvals

超越基本工具白名單，實現每個 Agent 獨立的命令政策。dev Agent 只能讀取檔案、files Agent 每個命令都需要手動核准、research Agent 可以 curl 但受 Squid 過濾。

"Goes beyond basic tool allowlisting with per-agent, per-command policies and approval workflows."

📚 LiteLLM Credential Brokering

LiteLLM 作為認證代理的設計模式：OpenClaw 只知道 LiteLLM 的 Master Key（內部金鑰），LiteLLM 負責注入真正的 Anthropic API Key 後轉發請求。即使 OpenClaw 被攻破，真正的 API Key 不在其容器中，無法被竊取。這就是Credential Brokering（認證代理）模式。

Part VI: Tier 3 網路架構 + 事件回應

Tier 3 的完整架構有四層防禦：(1) Loopback 綁定 + Tailscale/SSH Tunnel（Gateway 永不暴露）、(2) 容器網路隔離（內部網路無外部存取）、(3) Squid 網域白名單（出口過濾）、(4) LiteLLM 認證代理（API Key 隔離 + 速率限制）。

"Defense layers: Loopback binding + Tailscale/SSH tunnel - Gateway never exposed to public internet. Network segmentation - Internal network isolation. Squid domain allowlist - Egress filtering. LiteLLM credential brokering - API key isolation, rate limiting."

紅旗訊號（立即關閉）：Port 18789 突然在 0.0.0.0、dangerouslyDisableDeviceAuth 變成 true、Session 日誌中出現你沒發出的指令、安裝了你不認識的 MCP Server、Squid 日誌顯示被封鎖的網域突然被允許。

"Red Flags (Shut Down Immediately): Port 18789 suddenly on 0.0.0.0. dangerouslyDisableDeviceAuth: true appeared. Commands you didn't issue in session logs. New MCP servers you didn't install. Squid logs showing blocked domains suddenly allowed."

緊急處置三步驟：(1) 停止所有服務 + 封鎖所有網路流量、(2) 保存證據（Session 日誌、Squid 日誌、連線狀態）、(3) 假設所有認證已被洩露，在不同裝置上撤銷所有 API Key 和 OAuth 工作階段、重設密碼。

"Immediate Containment: Stop all services. Block all network traffic. Preserve evidence. Credential Rotation (Assume All Compromised): Revoke API keys. Revoke OAuth. Reset passwords (from different device)."

Key Takeaways

Allowlist > Denylist

安全的核心思維轉換：從「封鎖已知危險」到「只允許已知安全」。這個原則貫穿命令、MCP Server、OAuth 範圍、網域出口的每一層。

隔離是基礎，不是保險

VPS 隔離 + 容器隔離 + 網路隔離是三層同心圓。每一層都限制爆破半徑（Blast Radius），但沒有任何一層能完全消除風險。

Credential Brokering 是關鍵模式

LiteLLM 作為認證代理的設計，讓 OpenClaw 永遠不接觸真正的 API Key。這是企業級 API Gateway 模式的簡化版。

坦承不可修復的弱點

文章最有價值的部分是它坦承 Prompt Injection 和供應鏈攻擊「不可修復」。這種誠實比假安全感更有用。

Practice Questions

Q1 概念理解 為什麼文章強調 Allowlist 比 Denylist 更安全？在什麼情況下 Allowlist 反而會造成問題？

Allowlist 更安全的原因：Denylist 只封鎖已知的危險命令（如 rm -rf），但攻擊者可以用 wget、nc、python -c 等替代方案繞過。Allowlist 則相反——只有明確列出的命令才能執行，未列出的全部被拒絕。這就是「預設拒絕」（deny-by-default）原則。

Allowlist 的潛在問題：(1) 維護成本高——每次 Agent 需要新功能都要手動更新白名單；(2) 過度限制可能降低 Agent 的實用性，導致使用者為了方便而放寬設定；(3) 在快速迭代的開發環境中，嚴格的白名單可能成為瓶頸。關鍵是找到安全和實用性的平衡。

Q2 產業應用 如果你要在 CDMO 環境中部署類似 OpenClaw 的 AI Agent，Tier 3 的哪些安全原則可以直接對應到 GMP 的哪些法規要求？

幾個直接對應：

(1) Credential Brokering / 認證代理 → 21 CFR Part 11 的電子簽章和存取控制：API Key 隔離的概念直接對應到系統間認證的分離原則，確保每個系統只有最小必要的存取權限。

(2) Audit Log / 監控腳本 → ALCOA+ 的 Attributable 和 Contemporaneous：Step 18 的自動化安全檢查腳本等同於 GMP 的 Audit Trail，需要即時記錄、不可篡改、可追溯。

(3) Allowlist 思維 → Validated Systems 的 Change Control：GMP 環境中，任何系統變更都需要經過審核——這就是白名單的本質。不在 Change Control 清單上的修改不應該發生。

(4) Incident Response Plan（Step 28） → Deviation Investigation + CAPA：書面程序、定期測試、根本原因分析——CDMO 的偏差調查流程和這篇指南的事件回應計畫在結構上幾乎一致。

Q3 批判思考 文章的「Harm Reduction」策略是否在無意中正當化了使用有根本安全缺陷的工具？這個框架的邊界在哪裡？

這是文章最值得深思的張力。支持 Harm Reduction 的論點：安全社群的「不要用」建議在實際上已經失敗（OpenClaw 已經爆紅），提供具體的安全指南比道德勸說更能減少實際傷害。這和公共衛生領域的危害減少策略（如針具交換計畫）有相似的邏輯。

反對的論點：(1) 詳細的安全指南可能製造一種「做了這些就安全了」的虛假安全感，但文章自己承認 Prompt Injection 不可修復；(2) 指南的存在本身可能鼓勵更多人嘗試（「既然有安全指南，那應該沒問題」），形成反效果；(3) Google VP 等專家的直接建議是「不要用」，這篇指南是否在挑戰比自己更有資格的判斷？

邊界判斷：Harm Reduction 的邊界應該在「風險可以被合理量化且使用者能理解剩餘風險」的地方。文章做得好的是坦承 NEVER 清單和不可修復的限制；做得不夠的是可能低估了「大多數使用者會跳過步驟」的現實。
