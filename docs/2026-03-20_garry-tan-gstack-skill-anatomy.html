Executive Summary

Y Combinator 現任 CEO Garry Tan 開源了自己的 Claude Code 工具集 gstack，其中 plan-ceo-review 這個約 600 行的 Skill 引發廣泛討論。它的核心設計不是給 AI 領域知識，而是給 AI 審查姿態、領域無關的元問題、和 不可跳過的結構化流程。結果是：一個為工程審查設計的 Prompt，直接能用在商業計劃評估上——因為它約束的是 AI 的行為模式，不是內容。

這篇文章從三個層面拆解這個 Skill：(1) 它的結構設計 (2) 為什麼跨領域有效 (3) 對我們自己建構 Skill 的啟發。

Analysis 分析
💡 比喻說明

想像你訓練一個新的 QA 審計員。你可以選擇兩種方式：

方式 A：給他一本 500 頁的產品規格手冊，告訴他「照著這個查」。他只能查這一個產品。

方式 B：教他「永遠先質疑前提」「每個步驟都問 failure mode 是什麼」「不要放過任何模糊地帶」。他可以查任何產品。

plan-ceo-review 採用的是方式 B。

🛠️ CDMO 對照

這跟 GMP 環境裡的 deviation investigation 邏輯一樣：好的調查員不是背下所有 SOP，而是掌握「根本原因分析」的思維框架（5 Why、魚骨圖、Is/Is Not），然後把這個框架套用到任何偏差類型上。

框架 > 知識。這是 plan-ceo-review 證明的同一件事。

Part II: Layer 1 — 姿態定義（Posture, Not Content）

Content 內容
不是審查什麼，是以什麼態度審查
Skill 的開頭就定義了審查姿態——不是告訴 AI 要看什麼指標，而是告訴 AI 用什麼態度面對這個計劃。原文寫道：「你不是來給這個計劃蓋橡皮章的。你是來讓它變得卓越的。」
"You are not here to rubber-stamp this plan. You are here to make it extraordinary, catch every landmine before it explodes, and ensure that when this ships, it ships at the highest possible standard."
然後提供三種模式讓使用者選擇，每種模式定義了不同的姿態強度：
Then it provides three modes for the user to select, each defining a different posture intensity:
模式姿態核心問題
SCOPE EXPANSION
範圍擴張		建造大教堂什麼能用 2x 努力帶來 10x 提升？
HOLD SCOPE
保持範圍		嚴謹審查者怎麼讓它防彈？
SCOPE REDUCTION
範圍縮減		外科醫生最小可行版本是什麼？
關鍵規則：一旦使用者選擇了模式，AI 必須全力投入，不能悄悄偏移到另一種模式。這個約束非常重要——它防止了 AI 的常見問題：在該嚴厲的時候變得討好，或在該收斂的時候繼續發散。
"Critical rule: Once the user selects a mode, COMMIT to it. Do not silently drift toward a different mode."
Analysis 分析
📚 核心概念：Posture Engineering

大多數 Prompt 設計者專注於「輸入什麼資訊」和「輸出什麼格式」。Garry Tan 做的事更底層：他定義了 AI 的審查姿態（posture）。

Posture 包含三個維度：

1. 權力關係：你有權說「全部推翻」（不是僕人，是對等審查者）

2. 承諾強度：選定模式後不可偏移（防止 AI 的 people-pleasing 傾向）

3. 標準錨定：以最高標準上線（設定了品質下限）

💡 比喻說明

這就像在 GMP 環境裡，你不會告訴 QA 人員「去檢查批次記錄的第 37 頁」。你會告訴他們：「你的角色是 independent quality oversight。你的職責是確保每一個 critical process parameter 都在規格範圍內。你有權 reject 整個批次。」

角色定義決定了檢查的深度和嚴謹度——比任何 checklist 都重要。

🛠️ Skill 設計原則萃取

原則：先定義姿態，再定義任務。

在你的 Skill 開頭，不要急著寫「請分析以下數據」。先回答：AI 在這個任務中的角色是什麼？它的權力邊界在哪裡？它應該多嚴格？

Part III: Layer 2 — 領域無關的元問題（Meta-Questions）

Content 內容
Step 0 的核彈級質問
在進入 10 個審查環節之前，Skill 先執行 Step 0：核彈級範圍挑戰。這裡的問題完全不涉及任何特定領域：
Before entering the 10 review stages, the Skill executes Step 0: Nuclear Scope Challenge. These questions are entirely domain-agnostic:
0A. 前提挑戰：這是對的問題嗎？如果什麼都不做會怎樣？這是真實的痛點，還是假想的痛點？
"Is this the right problem to solve? What would happen if we did nothing? Real pain point or hypothetical one?"
0B. 現有資源利用：已經有什麼東西部分解決了這個問題？是不是在重建已有的東西？
"What existing code already partially or fully solves each sub-problem? Is this plan rebuilding anything that already exists?"
0C. 理想狀態映射：描述 12 個月後的理想最終狀態。這個計劃是在靠近那個狀態，還是在遠離它？
"Describe the ideal end state of this system 12 months from now. Does this plan move toward that state or away from it?"
0E. 時間線審問：提前想到實施過程——第 1 小時打地基需要知道什麼？第 2-3 小時核心邏輯會碰到什麼模糊地帶？第 4-5 小時整合時什麼會讓人意外？
"Think ahead to implementation: What decisions will need to be made during implementation that should be resolved NOW in the plan?"
Analysis 分析
📚 核心概念：Meta-Questions 為什麼有效

這些問題不涉及工程、商業、製造或任何特定領域。它們問的是更底層的事情：

1. 前提有效性：你確定你在解決對的事情嗎？

2. 資源重複性：你是不是在重造輪子？

3. 方向一致性：你往對的方向走嗎？

4. 實施可行性：你想清楚怎麼做了嗎？

這些是元認知問題——關於思考本身的思考。它們在任何領域都適用，因為它們質問的不是具體方案，而是做出方案的思考過程。

🛠️ CDMO 遷移範例

把這些元問題套到 CDMO 的 deviation investigation：

0A → Deviation Triage：這個偏差的根本原因真的是我們認為的嗎？還是只是表面症狀？

0B → 現有 CAPA：之前有沒有類似的偏差已經產生了 CAPA？我們是不是在重複調查？

0C → 理想狀態：如果這條產線的品質系統完美運作，12 個月後偏差率應該是多少？

0E → 實施審問：這個 CAPA 在執行時，第一週會碰到什麼阻力？需要哪些部門配合？

Analysis 分析
📚 核心概念：結構即約束

這 10 個環節的核心價值不是它們的內容——而是它們的結構。

「不可跳過」是關鍵設計。Skill 明確要求：如果某個環節沒有問題，要說明白然後繼續，但不能跳過。這確保了 AI 對每個面向都進行了系統性思考，而不是只挑容易的部分回答。

這就是 Checklist Manifesto 的核心論點：清單的價值不在於提醒你不知道的事，而在於確保你不會跳過你知道但會遺忘的事。

⚠️ 重點：自動遷移的機制

Claude 能自動遷移概念，是因為 LLM 的底層是語義理解，不是關鍵字匹配。當 Skill 說「單點故障」而輸入是商業計劃時，Claude 不會去找程式碼中的 SPOF——它理解「單點故障」的語義是「如果這一個環節失敗，整個系統崩潰」，然後在商業計劃的語境中找到對應的概念（例如只依賴單一客戶、單一供應商）。

這個遷移不是 Prompt 寫的，是模型自己做的。但 Prompt 給了一個足夠完整的檢查結構，讓模型沿著這個結構逐步遷移，不會遺漏。

🛠️ Skill 設計原則萃取

原則：用工程語境寫，讓模型自己遷移。

不需要為每個領域寫一套 Skill。用最嚴謹的領域（通常是工程）作為模板，定義完整的檢查結構，然後信任 LLM 的語義遷移能力。但前提是：結構必須完整且不可跳過。

Part V: 設計心法與批判分析

Content 內容
九條首要指令與提問規則
Skill 定義了 9 條 Prime Directives（首要指令），這些不是功能規格——而是行為約束。精選幾條最具啟發性的：
The Skill defines 9 Prime Directives — not feature specs, but behavioral constraints. Selected highlights:
#1 零靜默失敗：每一種失敗模式都必須可見——對系統、對團隊、對用戶。翻譯到任何領域：不允許問題被隱藏。
"Zero silent failures. Every failure mode must be visible — to the system, to the team, to the user."
#7 所有延後的工作都必須寫下來：「模糊的意圖等於謊言。」這是對 AI 和人類都適用的約束——如果你說「以後再處理」，你必須精確記錄什麼時候、處理什麼。
"Everything deferred must be written down. Vague intentions are lies."
#8 為 6 個月後優化：「如果這個計劃解決了今天的問題但製造了下個季度的噩夢，明確說出來。」這迫使 AI 從短期修復思維跳到長期系統思維。
"Optimize for the 6-month future, not just today. If this plan solves today's problem but creates next quarter's nightmare, say so."
#9 你有權說「全部推翻」：「我寧可現在就聽到。」這打破了 AI 的預設討好行為，賦予它 pushback 的權限。
"You have permission to say 'scrap it and do this instead.' I'd rather hear it now."
提問規則也值得注意：一個問題 = 一次提問。永遠不要把多個問題合併。並且假設使用者已經 20 分鐘沒看視窗了——每次提問都要重新定位上下文。
"One question = one AskUserQuestion. Never combine. Assume the user hasn't looked at this window in 20 minutes."
Analysis 分析
📚 核心概念：行為約束 vs 功能規格

回顧整個 Skill 的設計，我們可以歸納出一個核心洞察：

頂級 Skill 定義的是行為模式（behavioral constraints），不是功能規格（feature specs）。

功能規格告訴 AI「做什麼」——分析這個數據、生成這個報告、回答這個問題。行為約束告訴 AI「怎麼做」——以什麼態度、什麼深度、什麼順序、什麼標準。

功能規格是領域綁定的。行為約束是領域無關的。這就是為什麼一個工程 Skill 能用在商業計劃上。

⚠️ 批判分析：限制與注意事項

1. 倖存者偏差：我們看到的是一個成功案例。不代表所有「姿態定義型」Skill 都能跨領域遷移。遷移成功的前提是目標領域有類似的結構性問題（計劃 → 審查 → 改進）。

2. 模型依賴：自動遷移能力高度依賴模型的語義理解水平。Claude 能做到，不代表所有 LLM 都能。用較弱的模型可能只會照字面意思執行工程檢查。

3. 深度 vs 廣度權衡：10 個環節 × 每個環節多個子項 = 檢查非常全面但耗時。對於快速迭代的場景，這個 Skill 可能太重了。

4. GMP 適用性警告：在受監管環境中，AI 的「自動遷移」是一個風險。CDMO 偏差調查需要的是 validated process，不是 AI 自行解讀。如果要在 GMP 環境使用類似 Skill，必須加上 human-in-the-loop 的強制檢查點。

🛠️ 你的 Skill 設計清單

基於 plan-ceo-review 的設計模式，建構你自己的 Skill 時應包含：

1. 姿態宣言（2-3 句，定義角色和權力邊界）

2. 模式選擇（2-3 個強度級別，使用者選定後鎖定）

3. Step 0 元問題（3-5 個領域無關的前提挑戰）

4. 結構化環節（N 個檢查環節，每個不可跳過）

5. 提問規則（一問一答、上下文重新定位）

6. 必需輸出（明確列出最終交付物清單）

Key Takeaways

1. 姿態 > 知識
頂級 Skill 不給 AI 領域知識——它給 AI 審查姿態。定義角色、權力邊界、嚴格程度，讓 AI 用自己已有的知識在這個姿態下執行。這是跨領域泛用的根本原因。
2. 元問題打破前提
Step 0 的元問題（這是對的問題嗎？什麼都不做會怎樣？12 個月後的理想狀態？）是領域無關的認知工具。它們質問的不是方案，而是產生方案的思考過程。
3. 結構 = 不可跳過的紀律
10 個環節的價值在於「不可跳過」的約束。AI 傾向於跳過難的部分、專注容易的部分。強制結構消除了這個傾向，確保系統性思考覆蓋每個面向。
4. 信任語義遷移，但設護欄
LLM 能自動把「單點故障」從工程遷移到商業或製造語境。但在受監管環境（GMP），這種自動遷移需要人類驗證。Copilot 模式 + 強制檢查點是安全使用的前提。

Practice Questions

Q1 概念理解 plan-ceo-review 能跨領域生效的三個設計層次分別是什麼？為什麼「姿態定義」比「領域知識」更具泛用性？

三個設計層次：(1) 姿態定義——告訴 AI 以什麼態度審查，而非審查什麼內容；(2) 領域無關的元問題——質問前提、方向、資源重複性等底層認知問題；(3) 結構化流程——10 個不可跳過的檢查環節。

姿態定義比領域知識更具泛用性，因為它約束的是 AI 的行為模式（嚴格程度、權力邊界、承諾強度），而不是特定領域的檢查項目。行為模式在工程、商業、製造等任何需要「計劃 → 審查 → 改進」結構的領域都適用。

Q2 產業應用 如果你要為 CDMO 的 deviation investigation 設計一個類似的 Skill，Step 0 的元問題應該包含哪些？它與 plan-ceo-review 的 Step 0 有什麼共通點和差異？

共通的元問題：(a) 這個偏差的根本原因真的是我們認為的嗎？（= 前提挑戰）(b) 之前有沒有類似偏差已經產生了 CAPA？（= 現有資源利用）(c) 理想狀態下，12 個月後這條產線的偏差率應該是多少？（= 理想狀態映射）

GMP 特有的差異：(d) 這個偏差是否涉及 patient safety impact？（法規強制的優先級分類——plan-ceo-review 沒有等效概念）(e) 調查時限是否符合法規要求（如 30 天完成初步調查）？（= 外部約束，工程審查沒有法規 deadline）(f) 是否需要通報主管機關？（= 法規 escalation，商業計劃沒有對應概念）

核心差異：GMP 環境有外部監管約束（法規時限、patient safety、通報義務），這些在商業或工程語境中不存在。Skill 設計必須把這些作為 hard interlock，不能依賴 AI 自動遷移。

Q3 批判思考 原文作者認為「Claude 自動遷移工程術語到商業語境」是 plan-ceo-review 成功的關鍵。這個說法有什麼潛在風險？在什麼情況下，自動遷移可能產生錯誤的結論？

潛在風險：

1. 語義漂移：「回滾計劃」在工程中是精確的技術操作（git revert、database rollback），但遷移到商業語境變成了「退出策略」——這是一個更模糊、更複雜的概念。AI 可能把工程的精確性誤套到商業的模糊性上，給出看似精確但實際過度簡化的建議。

2. 結構性錯配：不是所有領域的問題都有「架構 → 錯誤處理 → 安全 → 測試 → 部署」的結構。例如人際關係問題、創意項目、藝術創作——強制套用 10 環節可能會產生不相關的分析。

3. 信心過度：自動遷移讓輸出看起來很專業（因為結構完整），但使用者可能因此高估建議的品質。一分鐘內的商業建議，即使結構化，也需要領域專家驗證。

4. GMP 環境的特殊風險：在受監管環境中，AI 的「自動遷移」可能產生 technically sounds right but regulatorily wrong 的建議。例如把「混沌測試」遷移為「故意引入偏差來測試系統韌性」——這在 GMP 環境中是絕對禁止的。

🎯 我的行動結論
✔ 核心洞察
頂級 Skill 的設計核心是「行為約束」而非「領域知識」——定義 AI 的姿態（態度、權力、嚴格度）、用領域無關的元問題挑戰前提、用不可跳過的結構確保全面性。這三層架構可以直接套用到我正在規劃的 /triage-deviation Skill 設計上。特別是「模式選擇後鎖定」這個機制，能解決 AI 在 deviation investigation 時常見的嚴格度不一致問題。
📈 立即應用
這週先把 plan-ceo-review 的三層架構（姿態 → 元問題 → 結構化環節）作為模板，起草 /triage-deviation Skill 的 Step 0 部分。具體做法：(1) 寫 2-3 句姿態宣言（「你是 independent QA investigator，你有權 escalate to Critical」）；(2) 設計 3 個 deviation triage 的元問題；(3) 定義 3 個嚴格度模式（Minor / Major / Critical）。不需要完成整個 Skill，先驗證 Step 0 的效果。
⚠ 需要補充
原文分析基於一個案例（商業計劃），樣本量為 1。需要自己測試 plan-ceo-review 在 CDMO 語境下的遷移品質，特別是：(a) 10 個審查環節中哪些能自動遷移、哪些需要 CDMO-specific rewrite；(b) 自動遷移在 GMP 環境的合規風險——哪些環節必須加上 human-in-the-loop interlock。另外，gstack 的其他 9 個工具也值得逐一評估，看有沒有可借鏡的設計模式。
