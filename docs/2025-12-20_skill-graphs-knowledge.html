Skill Graphs：從單一技能檔到可遍歷知識網路

當 AI Agent 需要的不只是一個指令，而是整個領域的理解

Executive Summary

這篇文章提出一個關鍵洞察：單一技能檔案（skill file）無法承載真正的領域深度。當 AI Agent 需要理解的不只是「如何做一件事」，而是「如何在一個領域中思考」時，你需要的是一張可遍歷的知識圖譜 — Skill Graph。

Heinrich 的核心論點是：用 Markdown 檔案 + Wikilinks + YAML frontmatter 就能建構出讓 Agent 自主導航的結構化知識網路。這不是理論 — 他的 arscontexta 專案已經用 249 個互相連結的檔案實現了這個架構。文章從「為什麼需要」講到「怎麼建構」，並提供了具體的 index 和 node 範例。

PART I 問題：為什麼單一 Skill File 不夠？

人們低估了結構化知識的力量。它能催生全新類型的應用。現在人們寫的技能檔案只捕捉了某個面向 — 一個用來做摘要的 skill，一個用來做 code review 的 skill。一個檔案，一個能力。

"people underestimate the power of structured knowledge. it enables entirely new kinds of applications. right now people write skills that capture one aspect of something. a skill for summarizing, a skill for code review and so on. one file with one capability"

這對簡單任務沒問題，但真正的深度需要另一種東西。想像一個治療師技能，需要提供認知行為模式、依附理論、主動聆聽技巧、情緒調節框架等相關資訊 — 單一技能檔案無法容納這一切。

"thats fine for simple tasks but real depth requires something else. imagine a therapy skill that provides relevant information about cognitive behavioral patterns, attachment theory, active listening techniques, emotional regulation frameworks... a single skill file cant hold that"

💡 比喻說明

把 Skill File 想成一張食譜卡。它能教你做一道菜，但不能教你理解烹飪。要真正「懂」烹飪，你需要理解食材科學、刀工技法、調味原理、熱傳導方式 — 這些知識彼此關聯，形成一張網，不是一張卡片。

Skill Graph 就是那張網。每個節點是一個獨立的完整概念，但透過連結，它們組成了一個可以被 Agent 遍歷的知識體系。

📚 核心概念

Skill File（技能檔案）：單一 Markdown 檔案，包含一個能力的指令。適合獨立、明確的任務。

Skill Graph（技能圖譜）：多個 Skill File 透過 Wikilinks 互相連結形成的知識網路。適合需要領域理解的複雜任務。

PART II 解法：Skill Graph 的三個原語（Primitives）

你已經擁有所需的一切工具。Wikilinks 被編織在散文中，所以它們攜帶的是語意，不只是參照。YAML frontmatter 附帶描述，讓 Agent 可以掃描而不需要讀完整個檔案。MOCs（Maps of Content，內容地圖）將相關技能的集群組織成可導航的子主題。

"you already have everything you need. wikilinks that read as prose in sentences, so they carry meaning not just references. yaml frontmatter with descriptions so the agent can scan without reading full files. MOCs (maps of content) that organize clusters of related skills into navigable sub-topics"

Skill Graph 在圖本身內部遞迴地應用相同的 skill discovery 模式。每個節點都有 YAML 描述讓 Agent 能掃描，每個 wikilink 因為嵌入在散文中而攜帶語意 — Agent 跟隨相關路徑，跳過不重要的。

"a skill graph applies the same skill discovery pattern recursively inside the graph itself. every node has a yaml description the agent can scan without reading the whole file. every wiki link carries meaning because its woven into prose so the agent follows relevant paths and skips what doesnt matter"

⚠️ 重點提示

Progressive Disclosure（漸進式揭露）是 Skill Graph 的核心設計原則：

Index → Descriptions → Links → Sections → Full Content

大多數決策在讀取任何完整檔案之前就已經做出了。這意味著 Agent 的 token 消耗被大幅降低，只在真正需要時才載入完整內容。

⚖️ 三個原語比較

Wikilinks：語意連結。寫在句子裡，Agent 根據上下文判斷是否跟隨。不是列表式的超連結。

YAML Frontmatter：快速掃描層。Agent 讀描述就能決定是否需要深入，不需讀全文。

MOCs：子圖譜導航。當圖譜變大時，MOCs 將相關節點組織成主題群組，避免迷路。

PART III 架構：Index 進入點與 Node 結構

Skill Graph 不需要存在你的 .claude/skills/ 資料夾裡。關鍵是一個 index 檔案，告訴 Agent 什麼存在、如何遍歷。Index 不是查找表，它是一個引導注意力的入口 — Agent 讀完後理解全局，然後跟隨對當前對話重要的連結。

"a skill graph doesnt need to live in your .claude/skills/ folder. the key is an index file that tells the agent what exists and how to traverse it... the index isnt a lookup table its an entry point that points attention. the agent reads it, understands the landscape and follows the links that matter for the current conversation"

每個被連結的檔案都是一個獨立的方法論聲明（methodology claim），也就是一個 skill。看看 wikilinks 如何在散文中告訴 Agent 何時以及為什麼要跟隨它們。

"each linked file is a standalone methodology claim (= skill)... see how the wikilinks inside the prose tell the agent when and why to follow them"

PART IV 應用場景與領域想像

想想看：交易 Skill Graph — 風險管理、市場心理學、部位管理、技術分析，每個部分連結到相關概念，讓上下文在它們之間流動。法律 Skill Graph — 合約模式、合規要求、管轄權細節、判例鏈，全部從一個入口可遍歷。公司 Skill Graph — 組織架構、產品知識、流程、入職上下文、文化、競爭格局。

"a trading skill graph: risk management, market psychology, position sizing, technical analysis, each piece linked to related concepts so context flows between them. a legal skill graph... a company skill graph: org structure, product knowledge, processes, onboarding context, culture, competitive landscape"

這些都無法放進一個檔案，但全部都能作為 Graph 運作。這就是 Skill 和 Skill Graph 的根本差異 — 前者是指令，後者是理解。

"none of these fit in one file but all of them work as graphs"

🛠️ 實務應用：藥廠 CDMO Skill Graph

把這個概念套用到 CDMO 情境：

GMP Skill Graph 可以包含：偏差調查流程 → CAPA 管理 → 根因分析方法 → FDA Warning Letter 案例 → PIC/S 指南對照 → 批次記錄審查 → Change Control 流程

每個節點是獨立的 SOP 級知識，但透過 wikilinks 連結後，Agent 可以在處理一個偏差調查時，自動拉取相關的 CAPA 模板、歷史案例、和法規要求 — 不需要你手動指定。

💡 比喻說明

單一 Skill File 像是給新員工一份 SOP。Skill Graph 像是讓新員工跟著資深前輩待三個月 — 他不只學到步驟，還學到為什麼這樣做、什麼時候該變通、哪些歷史教訓很重要。

PART V 從 Skills 到 Skill Graphs 的演化

Skills 是 Context Engineering（上下文工程） — 本質上是在正確的時機注入策展過的知識。Skill Graphs 是下一步：不是一次注入，而是 Agent 自主導航一個知識結構，精確提取當前情境所需的內容。

"skills are context engineering, basically curated knowledge injected where it matters. skill graphs are the next step. instead of one injection the agent navigates a knowledge structure, pulling in exactly what the current situation requires"

這是一個遵循指令的 Agent 與一個理解領域的 Agent 之間的差異。

"this is the difference between an agent that follows instructions and an agent that understands a domain"

⚠️ 重點提示

演化路徑：

Prompt → Skill File → Skill Graph

每一步都是 Context Engineering 的深化。Prompt 是即時指令，Skill File 是持久指令，Skill Graph 是可導航的知識體系。Agent 從「做你說的」進化到「理解你的領域」。

📚 Context Engineering 的定位

Heinrich 用「context engineering」定義 skills 的本質，這呼應了 AI 社群近期的共識：模型能力的上限由它獲得的上下文決定。

Skill Graph 是 context engineering 的最高形式 — 不是你決定給 Agent 什麼上下文，而是 Agent 根據情境自己決定需要什麼上下文。

Key Takeaways

深度需要結構
單一 Skill File 適合單一任務，但領域理解需要互相連結的知識節點 — Skill Graph 用圖結構解決了這個問題。
三個原語就夠了
Wikilinks（語意連結）+ YAML Frontmatter（快速掃描）+ MOCs（主題導航）— 全部用 Markdown 就能實現，無需外部框架。
Progressive Disclosure 省 Token
Agent 透過漸進式揭露（Index → Description → Links → Content）做出大部分決策，只在必要時才讀完整檔案。
從遵循指令到理解領域
Skills 是 Context Engineering，Skill Graphs 是其進化 — Agent 不再被動接收上下文，而是主動導航知識結構。

Practice Questions

Q1 概念理解 Wikilinks 在 Skill Graph 中為什麼要「嵌入散文」而不是用列表式連結？這對 Agent 的遍歷行為有什麼影響？

嵌入散文的 Wikilinks 攜帶語意上下文（semantic context）。當 Agent 讀到「[[the system is the argument]] — philosophy with proof of work」時，它不只知道有一個叫 "the system is the argument" 的節點存在，還理解這個節點的角色是「以實踐證明的哲學」。

這讓 Agent 能做出更精確的遍歷決策 — 它根據當前任務的需求，判斷是否值得跟隨這條連結。列表式連結只提供「存在」的資訊，散文式連結提供「何時有用」的資訊。這是 Progressive Disclosure 的第三層（Links），在這一層就能過濾掉大量不相關的路徑。

Q2 產業應用 如果要為你的 CDMO 建構一個 Skill Graph，你會如何設計 Index 的「Topic MOCs」分組？哪些節點之間的 Wikilinks 最關鍵？

一個 CDMO Skill Graph 的 Topic MOCs 可能包括：

[[aseptic-manufacturing]] — Vanrx SA25 操作、環境監測、無菌製程驗證

[[quality-systems]] — 偏差管理、CAPA、Change Control、批次記錄審查

[[regulatory-compliance]] — 21 CFR Part 11、PIC/S 指南、FDA 查廠準備

[[tech-transfer]] — 客戶對接、可行性評估、放大策略

[[adjuvant-processing]] — QS-21 處理、佐劑配方、穩定性考量

最關鍵的 Wikilinks 是跨 MOC 的連結。例如：在 [[deviation-investigation]] 節點中連結到 [[capa-management]] 和 [[fda-warning-letters]]，這讓 Agent 在處理偏差時自動獲取 CAPA 模板和歷史 FDA 案例作為參考。

Q3 批判思考 Heinrich 聲稱 Skill Graph 讓 Agent 從「遵循指令」進化到「理解領域」。你認為這個說法有什麼限制？在 GMP 環境中，Agent「理解」是否等同於可信任？

這個說法的核心限制在於「理解」這個詞的含義。Skill Graph 讓 Agent 能遍歷和關聯知識節點，但這更接近「結構化檢索」而非真正的「理解」。Agent 跟隨 wikilinks 做出的判斷仍然受限於 LLM 的推理能力和上下文窗口。

在 GMP 環境中，這個區別至關重要。即使 Agent 能從 Skill Graph 拉取正確的 SOP、法規參考和歷史案例，它的「判斷」仍然需要人類審查（copilot mode）。尤其是偏差調查這類需要根因推理的任務，Agent 可能會過度依賴圖譜中已有的模式，而忽略 novel failure modes。

Skill Graph 的真正價值不在於取代人類判斷，而在於讓 Agent 能更快、更完整地準備好人類決策所需的上下文。這和 David 的「copilot vs agent mode」框架一致 — Skill Graph 增強的是 copilot 的上下文品質，不是 agent 的自主決策權。
