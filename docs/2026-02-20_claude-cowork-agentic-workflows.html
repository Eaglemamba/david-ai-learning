★★★★☆ 3.8/5 ↑ Practical Value, Tool Ecosystem ↓ Technical Depth, Learning Curve

這篇文章是 Anthropic 新產品 Claude Cowork 的完整實戰指南。Cowork 本質上是「不需要終端機的 Claude Code」——它把 Claude Code 的 Agentic 架構（自主多步驟執行、子代理人、檔案系統存取）包裝成任何人都能使用的圖形介面。作者以長期 Claude Code 使用者的視角，誠實比較了 Claude Website、Claude Code、Claude Cowork 三者的定位差異，並提供 7 個可直接複製的進階工作流程，從競品分析、內容策略到產品驗證，每個都附有完整的 Prompt 和資料夾結構。

Part I — 三大產品定位比較

Claude Website vs Cowork vs Code: When to Use What
作者身為長期 Claude 使用者，將 Anthropic 的三大產品定位為清楚的階梯：Claude Website 是對話夥伴——你上傳檔案、來回對話、手動複製輸出。每個動作都需要你的下一個提示。它是協作式的，不是自主式的。
"The limitation: You're always in the loop. Every action requires your next prompt. It's collaborative, not autonomous."
Claude Code 是 Agentic AI 的始祖——住在你的終端機裡，擁有完整的檔案系統存取權，能自主執行多步驟計畫。但它的限制也很明確：終端機門檻過濾掉了 90% 的潛在使用者。
"The power is complete, but the interface filters out 90% of potential users."
Claude Cowork 就是「不需要終端機的 Claude Code」——相同的 Agentic 執行引擎，但包裝成 GUI。它不是降級版；是相同的自主多步驟執行能力，只是變得更容易使用。
"Claude Cowork is what I'd call 'Claude Code without the terminal'—it keeps the same agentic execution model, but wraps it in a GUI that anyone can use."

💡 比喻說明

把它想成開車：Claude Website 是副駕座上的導航員——告訴你怎麼走，但方向盤在你手上。Claude Code 是自動駕駛系統——你設定目的地它就開，但你得會讀儀表板（終端機）。Claude Cowork 是同一套自動駕駛，但換成了觸控螢幕介面——相同引擎，不同操作方式。

⚠️ 重點提示

作者很誠實地指出：如果你已經在用 Claude Code，Cowork 只是一個便利層（convenience layer），不會根本改變你的工作流程。真正的轉型對象是從 Claude Website 過來的使用者——他們是從「對話式 AI」升級到「Agentic AI」。

Part II — Cowork 五大核心功能

What Cowork Actually Adds: Accessibility Features
功能 1：claude.md 指令檔——你可以在工作資料夾中建立一個 claude.md 檔案，寫入你的品牌聲音、格式標準、合成方法論、輸出結構。Claude 會在該專案的每個任務中遵循這些指令。
"Drop in your custom rules—brand voice, formatting standards, synthesis methodology, output structure—and Claude operates by them every time."
最近的重大更新：現在可以設定全域 claude.md，一組指令套用到你整個 Cowork 工作區的所有任務。不再需要為每個專案重複設定。
"You can now set a global claude.md that applies across your entire Cowork workspace."
功能 2：可見 Todo List——給 Claude 一個複雜任務，它不會直接開始執行，而是先展示計畫。一個可見的待辦清單列出每個步驟，你可以在開始前檢視，每個步驟完成後會被勾選。
"This is what transparent autonomous execution looks like. Instead of watching a loading spinner and hoping for the best, you see the reasoning, the sequence, the progress."
功能 3：Claude in Chrome 瀏覽器整合——Cowork 直接整合了 Anthropic 的瀏覽器自動化工具。當你要求 Claude 瀏覽網頁時，它會自動觸發 Claude in Chrome 來控制你的瀏覽器。但作者警告：速度慢，每個動作都跑「截圖 → 判斷 → 下一步」循環。
"Every action runs on a screenshot → decision → next action loop. For quick tasks, manual is faster."
📚 核心概念：claude.md 的本質

claude.md 本質上就是一種持久化的 System Prompt——但它不是存在對話中，而是存在檔案系統裡。這意味著：

  • 它跨越多次任務存在，不會因為新對話而消失
  • 它可以被版本控制（Git）
  • 它可以被團隊共享

這跟我們在 Claude Projects 裡使用 Project Knowledge 的概念很像，但 Cowork 版本是檔案級別的，更接近 Claude Code 的 CLAUDE.md 模式。

🛠️ CDMO 應用思考

想像把 claude.md 應用到 CDMO 偏差調查流程：

  • 全域 claude.md：公司級 GMP 寫作規範、ALCOA+ 原則、文件格式標準
  • 專案級 claude.md：特定產品的製程參數、設備規格（如 SA25 Vanrx）、客戶特殊要求

這種分層指令結構非常符合製藥業的「SOP 層級」思維。

功能 4：平行子代理人（Sub-agents）——這是 Cowork 開始感覺不像單一 AI，而像一個團隊的地方。當你給 Cowork 一個複雜的研究或執行任務，它不會依序處理，而是啟動多個子代理人同時處理獨立的工作流。
"Instead of researching one company at a time, it launches 8 parallel agents, each analyzing a different company, then synthesizes the findings into one report."
作者提醒：子代理人目前在 Opus 4.6 上效果最佳，而且會消耗大量 Token。
功能 5：檔案輸出——這補上了 Claude Website 一直缺的最後一環。Cowork 直接輸出到你的本地檔案系統——已完成、已格式化、可直接使用的交付物：Excel（含公式）、PowerPoint（含版面）、Word、HTML Dashboard、CSV。
"You stop producing drafts and start receiving deliverables."
💡 比喻說明：子代理人 = 派遣團隊

想像你是專案經理，需要調查 8 家供應商的能力。傳統做法是你一家一家查，序列處理（Sequential）。子代理人模式就像你同時派出 8 個分析師，每人負責一家，最後你收集所有報告做綜合評估。

在 CDMO 情境：這就像同時對 8 個偏差案例進行根因分析的初步文獻調查，而不是一個一個來。

💰 Token 消耗警告

作者反覆強調 Token 消耗是 Cowork 的主要限制。作者提到的 Newsletter DNA Extraction 工作流程可能在 Pro 方案下用完整個 session 的額度。建議：

  • Max 方案 + Opus 4.6 是進階工作流程的唯一推薦
  • Sonnet 4.5 可處理部分工作，但在生成 HTML Dashboard 或 PowerPoint 時會失敗
  • 新推出的 Sonnet 4.6 有顯著改善

Part III — 7 個進階 Agentic 工作流程

7 Advanced Workflows to Unlock Cowork's Full Potential
1. 登陸頁競爭審計 (Landing Page Competitive Audit)
模型：Opus 4.6

將 Cowork 指向你的服務頁面和 5-10 個競品。Claude 像潛在客戶一樣閱讀每個網站——分析品牌定位、方案結構、主視覺文案、社會認同、轉換策略。然後編譯成一份完整的 Word 文件報告。
"Claude reads each site the way a potential customer would—analyzing brand positioning, offer structure, hero copy, social proof, and conversion strategy."
底層機制：Cowork 使用 Claude in Chrome 導航到每個 URL，像真人一樣捲動頁面、閱讀文案、擷取截圖。它不是在抓取原始 HTML，而是以潛在客戶的閱讀方式來看頁面，因此能捕捉情感語調、視覺層次和信任訊號。
2. 互動式績效故事板 (Interactive Performance Storyboard)
模型：Opus 4.6

餵入三個檔案：文章績效數據、訂閱成長歷史、讀者問卷。Claude 將三者整合分析（不是分開分析），建構一個 HTML Dashboard。核心亮點是「主題飽和度地圖」——將每個主題放在頻率 vs 表現的四象限中。
"The Topic Saturation Map plots every theme you've covered on two axes: how often vs. how well it performs. Four quadrants: Sweet Spot, Hidden Gems, Overplayed, Untapped."
🛠️ 工作流程 1 的關鍵設計

Prompt 中有一個很聰明的工具回退策略（Tool Fallback Chain）：

  • 優先用 Firecrawl MCP 抓取完整頁面 + 品牌資料
  • 失敗則用 WebFetch 內建工具
  • 再失敗用 Claude in Chrome 的 get_page_text

這種分級回退設計是穩健 Prompt 工程的好範例——預先考慮失敗路徑，而不是假設一切順利。

📚 主題飽和度地圖框架

四象限矩陣（頻率 × 表現）：

  • Sweet Spot（高頻 + 高表現）：持續產出
  • Hidden Gems（低頻 + 高表現）：加倍投入
  • Overplayed（高頻 + 低表現）：減少或重新定位
  • Untapped（低頻 + 低表現 but 有需求）：測試新角度

這個框架也適用於評估 CDMO 的技術能力組合——哪些服務是 Sweet Spot（高需求高能力），哪些是 Hidden Gems（有能力但未充分推廣）。

3. 個人每日作業系統 (Daily Operating System)
模型：Sonnet 4.6

這改變你每天開始工作的方式。資料夾預載了所有 Claude 需要理解你工作的東西：CLAUDE.md（作業系統指令）、profile.md（你是誰）、projects.md（正在做什麼）、schedule.md（週期行程）。首次使用需跑一次「入職」——Claude 訪問你，填充所有檔案。之後每天只需輸入：「Let's start our day.」
"Claude reads your files, knows what day it is, asks you three adaptive questions, then does something most AI can't: it looks at your todo list and tells you what it can handle autonomously."
4. AI 內容飛輪 (Content Flywheel)
模型：Sonnet 4.6

大多數內容創作者每週從零開始，相同的盲點反覆出現。這個工作流程把你的整個內容歷史變成活的智慧系統。關鍵在 memory.md——一開始是空的，每次執行後 Claude 會更新它，記錄哪些主題已被覆蓋、哪些表現好、哪些角度要避免。六個月後，你有一個比你自己更了解你內容歷史的系統。
"Every run builds on the previous one. Six months in, you have a system that knows your content history better than you do."
💡 Daily OS 的設計哲學

這個工作流程的核心設計值得注意：

Onboarding → Daily Loop → Wrap-up

一次性入職 → 每日三個適應性問題 → 自動執行 → 結束時更新狀態。這其實是一個狀態機（State Machine）模式——claude.md 是程式碼，markdown 檔案是資料庫，Cowork 是執行引擎。

這種「資料夾即應用」的模式對非技術人員很有啟發——你不需要寫程式，只需要結構化你的檔案。

⚠️ memory.md 的飛輪效應

Content Flywheel 的真正價值不是一次性的內容建議，而是累積性記憶。每次執行都會更新 memory.md，形成遞增的知識資產。

這跟 Claude Projects 的 Memory 功能異曲同工，但它是你可以手動檢視和編輯的檔案——更透明，更可控。

5. SEO 內容策略引擎
模型：Opus 4.6

同時審計 10+ 篇已發布文章的 SEO。兩個 MCP 做重活：Firecrawl 抓取你的文章結構，Brave Search 搜尋每個主題的前 10 名排名結果進行比較。Cowork 用子代理人平行處理。
6. Newsletter DNA 萃取
模型：Opus 4.6

最消耗 Token 的工作流程。Claude 以外人的視角閱讀你的電子報——截圖首頁、閱讀表現最佳的文章、分析你的聲音模式、反覆主題、表現好壞的內容。輸出品牌 DNA 報告（Word 文件）。
"The reason this works better than any manual analysis is that you're too close to your own work. You can't see your own patterns the way a first-time reader sees them."
7. 產品想法驗證框架
模型：Opus 4.6

在寫一行程式碼之前跑這個。描述你的產品概念，指向 3-5 個競品，Cowork 平行研究整個競爭格局。最關鍵的是第五階段：驗證判決——它會給你誠實的 Go/No-Go 評估，如果想法需要轉向，它會直說。通過後產出三個檔案：市場研究報告、完整 PRD、Claude Code 啟動提示。
"Gives you an honest go/no-go verdict before writing anything. If the idea needs pivoting, it says so."
💻 MCP 配置要點

三個主要 MCP 及用途：

  • Firecrawl MCP：網頁抓取（500 URLs/月免費）
  • Brave Search MCP：搜尋引擎（2,000 queries/月免費）
  • Perplexity MCP：綜合研究（需 API key）

配置在 claude_desktop_config.json 中，格式為 JSON，每個 MCP 需要替換自己的 API key。

但作者也提到：這些 MCP 不是必要的——你仍可依賴 Claude 內建的 web search 和 fetch 功能。

🛠️ 產品驗證的分層工具策略

工作流程 7 的 Prompt 設計包含一個雙層工具策略：

  • Tier 1：Perplexity（綜合研究）+ Firecrawl（一手資料抓取）
  • Tier 2：WebSearch + WebFetch（無 MCP 時的回退）

核心原則：如果兩個工具都能回答同一個問題，優先用能給綜合答案的（Perplexity > WebSearch），只有需要一手資料時才去抓取原始來源。

Key Takeaways

01
Agentic 不是新功能，是新介面
Cowork 的所有能力（多步驟執行、子代理人、檔案輸出）都已存在於 Claude Code。真正的創新是無障礙化——移除終端機門檻，讓 Agentic AI 對所有人開放。產品策略的核心是：相同引擎，不同入口。
02
claude.md = 持久化 System Prompt
全域 + 專案級的 claude.md 結構實現了「設定一次，永久遵循」的指令持久化。這是從每次重新解釋上下文到一次性配置的根本性轉變，特別適合有標準化作業需求的產業。
03
角色從「執行者」變成「定義者」
文章最深層的洞察：你不再是做工作並用 AI 幫忙的人，而是定義成果並讓 AI 執行的人。這是時間關係的根本改變——從「AI 加速我的工作」到「AI 做工作，我做判斷」。
04
Token 經濟學決定可行性
作者反覆強調 Token 消耗是 Cowork 進階工作流程的核心限制。Max 方案 + Opus 4.6 是進階使用的事實標準。這提醒我們：Agentic AI 的瓶頸不再是能力，而是成本。

Practice Questions

Q1 概念理解 文章說 Cowork 是「Claude Code without the terminal」。但作者也承認自己仍然更常回到 Code。如果底層引擎相同，為什麼介面差異會導致使用偏好的分歧？這對 AI 產品設計有什麼啟示？

這揭示了 AI 產品設計中一個常被忽略的維度：介面不僅影響使用門檻，還影響執行效率和認知模型。

作者偏好 Code 的原因是「速度更快」——CLI 的操作延遲更低、回饋更即時。這不是功能差異，而是互動頻寬差異。對於已經內化了終端操作的使用者，GUI 反而增加了認知負擔（需要點擊、等待畫面載入、視覺搜索按鈕位置）。

啟示：AI 產品不該追求「一個介面服務所有人」，而是「相同引擎、多個入口點」——讓不同技術舒適度的使用者都能以最低認知負擔操作。Anthropic 的三層策略（Website → Cowork → Code）正是這種思維的體現。

Q2 產業應用 假設你要在 CDMO 環境中實施類似「Daily Operating System」的工作流程來管理每日的生產偏差追蹤。考慮到 GMP 合規要求，你會如何調整 claude.md 的設計？哪些步驟必須保留人工審核？

在 GMP 環境中，Daily OS 的核心設計需要重大調適：

claude.md 設計原則：

全域 claude.md 應包含：ALCOA+ 資料完整性原則、21 CFR Part 11 電子記錄要求、公司 SOP 文件編號規範、禁止自動修改已審核文件的硬性規則。

專案級 claude.md 應包含：特定產品線的製程參數限值、設備（如 SA25 Vanrx）的特殊操作注意事項、客戶審計的特殊要求。

必須保留人工審核的步驟（Copilot Mode）：

偏差分類（Critical / Major / Minor）的最終判定、根因分析的因果結論、CAPA 行動的核准、任何會被提交給法規機構的文件。

可以讓 AI 自主執行的步驟（Agent Mode）：

偏差歷史資料的整理和趨勢分析、SOP 相關條款的查找和引用、初步的風險評估矩陣草稿、每日偏差摘要報告的編制。

核心原則：AI 做彙整，人做判斷——這正是 O-ring 瓶頸理論的應用。

Q3 批判思考 作者提供的 7 個工作流程都是內容創作 / 行銷導向的。如果 Cowork 的子代理人和檔案輸出能力如此強大，為什麼沒有看到更多結構化資料處理（如財務分析、供應鏈管理）的案例？這是工具限制還是作者偏差？

這是一個重要的觀察。答案可能兩者皆是：

作者偏差面：Wyndo 是 Newsletter 創作者，他的工作流程自然圍繞內容生產。他確實在「What Else People Are Building」部分提到了財務（信用卡帳單審計）和人資應用，但都只是簡短描述，沒有完整 Prompt。

工具限制面：結構化資料處理（財務模型、供應鏈優化）通常需要：(1) 連接到即時資料庫（Cowork 只能處理上傳的靜態檔案）、(2) 精確的數值計算（LLM 的數學能力有已知限制）、(3) 審計追蹤（Cowork 的執行過程不符合 SOX 或 GMP 的記錄要求）。

啟示：Cowork 當前最適合的「甜蜜點」是——輸入是非結構化資料（文字、網頁、文件），輸出是結構化交付物（報告、簡報、Dashboard）。對於「結構化輸入 → 結構化輸出」的場景（如 ERP 資料 → 財務報表），傳統工具或專用 AI 方案仍然更可靠。對我們要用 2-3x discount 原則來評估——作者展示的是最佳情境，實際應用需要打折。

🎯 我的行動結論
✔ 核心洞察
Cowork 代表的不只是一個新產品，而是 Anthropic 對 Agentic AI 大眾化的策略路線圖。從 Code → Cowork 的進化路徑證實了一件事：AI 的能力瓶頸已經解決，現在的瓶頸是介面和可及性。對我而言，這強化了「Pharma x AI x Leadership」定位的邏輯——在 CDMO 環境中，多數同事不會碰終端機，但他們可以使用 Cowork 式的 GUI 來執行結構化的偏差調查或技術轉移文件彙整。我的角色就是設計那個 claude.md 和資料夾結構。
📈 立即應用
本週可以做兩件事：(1) 模仿 Daily OS 的「資料夾即應用」模式，為 Substack 內容創作建一個 Content Flywheel 資料夾——包含 profile.md（寫作聲音規範）、archive/（過去文章）、memory.md（累積記憶），用在 Claude Projects 中實驗。(2) 研究 Cowork 是否已在台灣地區可用，以及 Max 方案的 Token 額度是否足以支撐進階工作流程的實際成本。
⚠ 需要補充
文章完全沒有觸及合規環境（GMP、金融法規、醫療法規）下的 Agentic AI 應用限制——這對 CDMO 情境至關重要。也沒有討論資料安全問題（上傳機密文件到 Cowork 的風險）。下一步需要研究 Anthropic 的企業方案（Gemini Enterprise 已有 Data Residency），以及 21 CFR Part 11 對 AI 輔助文件的合規要求。文章的 7 個工作流程全部是內容/行銷導向，需要自行設計製藥業的對應版本。建議 2 個月後重讀，屆時 Cowork 應該會有更穩定的版本和更多產業應用案例。
