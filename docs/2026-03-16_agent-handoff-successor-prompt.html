Executive Summary

使用 Claude Code、Codex 或 Gemini CLI 等 AI coding agent 時，對話越長，模型的幻覺 (Hallucination) 和錯誤率會明顯上升。周尔复提出「繼任者 Prompt」(Successor Prompt) 技巧：在結束過長對話前，讓 AI 產出一份結構化的「交接文件」給下一個新會話中的 AI，確保工作脈絡不丟失。v2.0 版本提供了完整的七段式交接模板，涵蓋任務目標、進展、上下文、發現、未完成事項、建議路徑和風險提醒。同時延伸出「Prompt 自舉」(Prompt Bootstrap) 的 meta 技巧 — 先寫粗稿 Prompt，再讓 AI 自己打磨。

Part I — 問題：長上下文退化 (Long Context Degradation)
使用 Codex / Claude Code / Gemini CLI 的一個技巧：如果對話太長，AI 的幻覺越來越多，問題越來越難以解決，不妨開一個新的會話。
A tip for Codex / Claude Code / Gemini CLI: if the conversation gets too long and AI hallucinations increase, making problems harder to solve, consider starting a new session.
這不是模型本身變笨了，而是 Context Window 中累積的噪音（失敗嘗試、重複指令、矛盾信息）稀釋了有效上下文的「信噪比」(Signal-to-Noise Ratio)。
The model hasn't gotten dumber — accumulated noise in the Context Window (failed attempts, repeated instructions, contradictory information) dilutes the signal-to-noise ratio of effective context.
💡 比喻說明
想像你在一間白板上寫滿字的會議室裡討論。當白板上 70% 的內容是已經被否定的方案和過時的筆記時，你不斷回頭參照這些內容只會越來越混亂。最有效的做法是：把結論記下來，擦掉白板，從乾淨的狀態重新開始。繼任者 Prompt 就是那份「結論筆記」。
📚 核心概念：Context Window 的 Attention 機制

LLM 的 Transformer 架構使用 Attention 機制處理上下文。當上下文長度增加，模型需要在更多 token 之間分配注意力權重。關鍵信息被「淹沒」在無關 token 中，導致：

1. Lost in the Middle 效應 — 上下文中間的信息容易被忽略

2. 矛盾累積 — 早期的錯誤嘗試和後來的修正共存，模型難以判斷哪個版本有效

3. 指令稀釋 — 原始意圖被後續的反覆討論掩蓋

⚠️ 重點提示：何時該觸發交接？

信號一：Agent 開始重複之前已嘗試且失敗的方案

信號二：你需要反覆澄清同一個需求或約束

信號三：修一個 bug 卻引入兩個新 bug（回歸率上升）

信號四：Agent 開始「道歉 + 重試」循環而沒有實質進展

Part II — 繼任者 Prompt v1.0：核心概念
在退出前讓它做一下工作交接。Prompt 可以這樣說：「本次對話的上下文已經太長了，我打算關掉並重新開一個新的會話。你有什麼想對你的繼任者說的，以便它能更好的理解你當前的工作並順利繼續？」
"The context of this conversation has become too long. I plan to close it and start a new session. What would you like to say to your successor so it can better understand your current work and continue smoothly?"
這個版本是作者 9 個月前（2025 年 6 月）發布的，一直用到現在。核心洞察是把 AI Agent 擬人化為一位「即將離職的員工」，讓它主動整理工作狀態，而非由使用者手動總結。
This version was published 9 months ago (June 2025) and has been in use since. The core insight is personifying the AI Agent as an "outgoing employee," having it proactively organize work status rather than requiring the user to manually summarize.
📚 核心概念：為什麼讓 AI 自己做交接比你自己寫更好？

信息不對稱 — Agent 知道它實際嘗試過什麼、在哪裡卡住、哪些路徑已被排除。這些隱性知識你未必完全掌握。

格式優勢 — Agent 輸出的交接文件天然是 LLM 友好的格式，直接貼給新 Agent 就能高效解析。

認知減負 — 讓 AI 做「知識萃取」，你只需要審核而非從頭整理。

⚠️ 重點提示：v1.0 的局限性

v1.0 是一個開放式 Prompt — 沒有指定交接格式，AI 的輸出品質取決於它當下的「判斷」。可能的問題：

遺漏關鍵信息 — 沒有明確要求列出「風險」和「已排除路徑」，新 Agent 可能重蹈覆轍

粒度不一致 — 有時太概括（「我們討論了架構設計」），有時太冗長

缺乏行動指引 — 告訴繼任者「發生了什麼」但不告訴它「下一步該做什麼」

🛠️ 實務應用：CDMO 偏差調查的交接場景

設想你用 Claude Code 進行一個複雜的 CDMO 偏差調查數據分析，對話已超過 50 輪。你已經排除了設備校準問題、確認了環境監測數據異常、正在分析批次紀錄的時間線。

如果直接開新會話，這些「排除清單」和「正在追蹤的線索」全部消失。繼任者 Prompt 讓 Agent 把調查進展完整記錄下來，新 Agent 可以直接從「分析批次紀錄時間線」這個節點繼續。

Part III — 繼任者 Prompt v2.0：結構化七段交接模板
v2.0 是透過「Prompt 自舉」打磨後的版本 — 從開放式問題升級為結構化的七段式交接模板。每一段都有明確的功能定位，確保交接文件的完整性和可操作性。
v2.0 was refined through "Prompt Bootstrap" — upgraded from an open-ended question to a structured seven-section handoff template. Each section has a clear functional purpose, ensuring completeness and actionability of the handoff document.
📚 核心概念：七段式交接結構解析
段落 功能 為什麼重要
1. 當前任務目標問題、預期產出、完成標準讓新 Agent 知道「終點在哪裡」，避免偏離方向
2. 當前進展已完成的分析、確認、修改避免重複勞動，從斷點而非起點繼續
3. 關鍵上下文背景、需求、約束、決策、假設重建「為什麼這樣做」的隱性知識
4. 關鍵發現結論、規律、異常、根因判斷最高密度的可操作信息
5. 未完成事項待處理內容 + 優先級排序明確的工作清單，不用猜「接下來做什麼」
6. 建議接手路徑優先查看什麼、先驗證什麼、下一步降低新 Agent 的冷啟動時間
7. 風險與注意事項易誤判、已排除路徑、不建議方向防止重蹈覆轍 — 這是 v1.0 最缺的部分
⚠️ 重點提示：v2.0 的關鍵設計原則

寫給 Agent，不是寫給人 — 原文強調「這是交接文檔，不是面向用戶的總結」。語氣應該是工程化的、具體的、引用文件路徑和類名的，不是客套的敘述。

具體名稱優先 — 要求引用文件路徑、類名、模組名、接口名、命令、術語。這確保新 Agent 拿到的是可定位的 pointer，而非模糊的描述。

輸出到文件 — 指定 ./{yymmdd}-handoff.md，讓交接文件持久化在專案目錄中，而非只在對話中消失。

💻 v2.0 完整 Prompt 模板（可直接複製使用）
當前上下文已經過長，我將開啟一個新的會話繼續這項工作。

請你在 ./{yymmdd}-handoff.md 輸出一份寫給「下一位接手的 AI Agent」的交接摘要。
假設它無法訪問當前完整上下文，只能看到你現在這份交接內容，
因此你的目標是讓它盡快理解現狀，並能直接繼續推進工作。

請按以下結構整理：

## 1. 當前任務目標
說明當前要解決的問題、預期產出和完成標準。

## 2. 當前進展
說明目前已經完成了哪些分析、確認、修改、排查、討論或產出。

## 3. 關鍵上下文
包括但不限於：重要背景信息、用戶的明確要求、已知約束、
已做出的關鍵決定、重要假設。

## 4. 關鍵發現
列出目前最重要的結論、規律、異常點、根因判斷、設計判斷
或值得注意的信息。

## 5. 未完成事項
列出仍需要繼續處理的內容，並按優先級排序。

## 6. 建議接手路徑
告訴下一位 Agent：應優先查看哪些文件、模組、數據、日誌、
命令、頁面或線索；應先驗證什麼；推薦的下一步動作是什麼。

## 7. 風險與注意事項
說明哪些點容易誤判、重複勞動或跑偏，
哪些方向已經驗證過且不建議繼續。

要求：
- 這是交接文檔，不是面向用戶的總結
- 信息盡量具體，避免空話和客套話
- 優先保留對繼續工作最有價值的可執行信息
- 盡量引用具體名稱（文件路徑、類名、模組名、接口名、命令、術語）
- 最後補一段「下一位 Agent 的第一步建議」
Part IV — Prompt 自舉：讓 AI 打磨自己的 Prompt
作者提到一項他稱之為「Prompt 自舉」(Prompt Bootstrap) 的技能：先寫一段 Prompt 作為原始需求，然後交給 AI 打磨，以實現「左腳踩右腳上天」的效果。
The author mentions a technique he calls "Prompt Bootstrap": write a rough prompt as the initial requirement, then hand it to AI for polishing, achieving a "bootstrapping" effect.
「我知道這聽起來好像沒什麼，但產生的效果驚人。我過去的很多所謂『自創』的 Prompt，在 AI 打磨後都變得驚人的好。」
"I know this sounds trivial, but the results are astounding. Many of my so-called 'original' prompts became remarkably better after AI polishing."
💡 比喻說明
Prompt 自舉就像找一位資深編輯來潤色你的初稿。你有想法和方向（domain knowledge），AI 有語言結構和完整性的專業（meta-prompting）。兩者結合的產出遠大於各自單獨的能力。v1.0 到 v2.0 的跳躍正是這個技巧的示範。
🛠️ 實務應用：Prompt 自舉的操作流程

Step 1 — 用人腦寫出核心意圖（你最了解「要解決什麼問題」）

Step 2 — 交給 AI：「請打磨這個 Prompt，讓它更結構化、更完整、更不容易產生模糊的輸出」

Step 3 — 審核 AI 的版本，刪掉過度工程的部分，保留有價值的結構改進

Step 4 — 實際使用，根據產出品質迭代（v1 → v2 → ...）

⚠️ 重點提示：自舉的邊界

AI 擅長：結構化、完整性檢查、邊界情況補充、格式優化

AI 不擅長：定義問題本身、判斷什麼信息對你的特定情境最重要、設定優先級

原則：Wetware before Software — 先用人腦思考核心意圖，再用 AI 打磨表達。這和直接讓 AI 從零生成 Prompt 是根本不同的。

Key Takeaways
🚨 長上下文 ≠ 更聰明
對話越長，累積的噪音越多，信噪比下降。辨識退化信號（重複嘗試、回歸 bug、道歉循環），果斷「斷點重啟」而非繼續對抗。
📝 交接文件要寫給 Agent，不是寫給人
繼任者 Prompt 的產出對象是下一個 AI Agent。語氣要工程化、具體引用文件路徑和類名，不要客套。目標是最小化冷啟動時間。
⚠️ 第七段「風險與注意事項」是關鍵差異
v2.0 最大的改進是明確要求列出「已排除路徑」和「不建議方向」。這防止新 Agent 重蹈覆轍，是交接品質的決定性因素。
⚡ Prompt 自舉 = 人腦意圖 + AI 結構化
先用人腦定義「解決什麼問題」，再讓 AI 打磨結構和完整性。這不是偷懶，是認知分工 — 各自做擅長的事。
Practice Questions
Q1 概念理解 為什麼繼任者 Prompt 要求輸出到文件（.md）而非直接在對話中回覆？

持久化 — 對話關閉後內容消失，但文件保留在專案目錄中，可以被版本控制追蹤。

可引用 — 新會話可以直接讀取文件（cat ./250316-handoff.md），比手動複製貼上更可靠。

可累積 — 多次交接的 handoff 文件形成一個決策歷史軌跡，即使是第三位、第四位 Agent 也能回溯完整脈絡。

Q2 產業應用 如果你正在用 Claude Code 開發一個 CDMO 偏差調查的自動化腳本，哪些信息在交接文件中是最關鍵的？

在 GMP 環境中，交接文件的「第 3 段：關鍵上下文」和「第 7 段：風險」尤其重要：

關鍵上下文 — 需要明確記錄：正在處理哪個偏差編號、涉及哪些批次號、已排除的根因（如設備校準已確認合格）、數據源的完整路徑（環境監測 CSV、批次紀錄 PDF）。

風險 — 必須記錄：「不要重新查詢 X 設備的校準紀錄，已確認合格」、「批次紀錄第 47 頁的時間戳異常是已知問題，不是根因」。在 GMP 調查中，重複排查已排除的路徑不只浪費時間，還可能在調查報告中引入混淆。

Q3 批判思考 繼任者 Prompt 有什麼局限性？它無法解決什麼問題？

局限一：信息壓縮有損 — 交接文件是 Agent 對自身工作的「理解」，不是完整上下文的無損備份。如果 Agent 的理解本身有偏差（特別是在上下文已退化的末期），交接文件可能傳遞錯誤的結論。

局限二：無法傳遞「手感」 — 某些調試過程中形成的直覺（「這個 API 在併發時行為古怪」）很難結構化地寫出來。

局限三：不能替代好的架構 — 如果根本問題是任務過於複雜、缺乏分解，交接只是把混亂轉移到下一個會話。正解是先把任務分解為獨立的子任務，每個子任務在一個會話內完成。

改進方向：可以在交接前先要求 Agent 做一次「自我審計」— 評估自己的結論哪些是高置信度的，哪些是猜測性的。

🎯 我的行動結論
✔ 核心洞察
「讓 Agent 做交接，而非讓人做總結」是一個高 ROI 的 workflow 改進。Agent 掌握的隱性上下文（嘗試過的路徑、排除的方案、具體的文件位置）比使用者記得的更完整。v2.0 的結構化模板把這個過程從「看 Agent 心情」變成了可重複的標準流程，特別是「第七段：風險與注意事項」直接解決了我在 Claude Code 長對話中最常遇到的痛點 — 新會話重複走已排除的死路。
📈 立即應用
把 v2.0 Prompt 模板存入 Claude Code 的 CLAUDE.md（或建立為 Slash Command），在任何超過 30 輪的 coding 會話結束前自動觸發。同時可以擴展到教學文件生成 — 當一份複雜文件的生成過程超過 token 限制時，用交接 Prompt 讓當前 Agent 把「已完成段落 + 待完成結構 + CSS 設計決策」記錄下來。
⚠ 需要補充
文章沒有討論「何時不需要交接」— 如果任務本身就該在單一會話內完成，問題出在任務分解而非上下文長度。另外缺乏對不同 Agent 工具（Claude Code vs Codex vs Gemini CLI）的交接格式差異分析。交接文件的品質也受到「Agent 已退化」的影響 — 如果 Agent 已經嚴重幻覺，它的交接摘要可能也不準確。建議在上下文退化初期（信號出現時）就觸發交接，而非等到末期。
