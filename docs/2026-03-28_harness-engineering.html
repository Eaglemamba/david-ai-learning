Executive Summary

2026 年 AI 工程領域的最新關鍵詞：Harness Engineering（韁繩工程）。這不是另一個行銷新詞 — 它描述的是一個真實且正在發生的範式轉移：工程師的核心工作從「寫程式碼」轉向「設計 AI Agent 運行的環境」。本文以花叔的微博長文為核心，交叉驗證 OpenAI Codex 團隊的百萬行程式碼實驗、Anthropic 三 Agent 架構、LangChain 的 Terminal Bench 突破（同模型僅換 harness，排名從 Top 30 跳到 Top 5），以及 Mitchell Hashimoto 的實踐哲學。核心洞察：模型可能不再是瓶頸，你給它搭的環境才是。

Part I
又一個新詞？Harness Engineering 的定義與定位

Harness 這個詞的英文原意是馬具、韁繩。不是馬本身，是套在馬身上讓它能拉車、能被引導的那整套東西。沒有它，馬就是一匹亂跑的野馬，力氣再大也白搭。

"Harness" — not the horse itself, but everything you put on the horse so it can pull a cart and be guided.

三層關係很好懂：Prompt 管你問什麼，Context 管你給模型看什麼，Harness 管整個東西怎麼運轉。Context 是 Harness 的一部分，Harness 還多管了約束、反饋和品質檢查。

Prompt governs what you ask; Context governs what you show the model; Harness governs how the whole thing runs.

這個詞的結晶時刻來自 2026 年 2 月，Mitchell Hashimoto — HashiCorp 共同創辦人、Terraform 創造者 — 發了一篇部落格文章，首次給這個實踐命名。

"It is the idea that anytime you find an agent makes a mistake, you take the time to engineer a solution such that the agent never makes that mistake again."

💡 比喻說明

馬具比喻的精準之處：

Prompt = 對馬的口令
Context = 馬看到的路
Harness = 韁繩 + 馬鞍 + 圍欄 + 道路本身

沒有 Harness，你可以對馬大喊（好 prompt）、給它地圖（好 context），但十匹馬同時跑起來時，沒有韁繩和圍欄，就是混亂。

⚖️ 工具比較
層級管什麼範例
Prompt單次指令「修復這個 bug」
Context模型看到的資訊CLAUDE.md、RAG、知識庫
Harness整套運行系統Hooks、Skills、CI/CD 檢查、權限模型
⚠️ 重點提示

HumanLayer 的定位：Harness Engineering 是 Context Engineering 的子集 — 專注於利用 harness 配置點（系統提示詞、工具/MCP、context、sub-agents）來管理 coding agent 的 context window。

Part II
命名的價值 — 不是發明，是意識覺醒

AI 圈不是發明了 Harness Engineering，是終於意識到自己需要學幾十年前就有的工程紀律。航天工程師 60 年前就在做類似的事。NASA 讓飛船自動執行任務，圍繞自動化系統設計的約束、反饋循環、冗餘檢查、異常處理，和今天說的 harness 沒有本質區別。

AI didn't invent harness engineering — it finally realized it needs engineering discipline that's existed for decades.

但命名還是有價值的。當一群人各自在做類似的事，沒有共同的詞來說，經驗就傳不開。就像 Vibe Coding，你可以笑它，但它確實讓一種做法變成了可以討論的東西。

When a group of people are all doing similar things without a shared word, experience can't spread.

Harness Engineering 也一樣。價值不在於發明了什麼，在於讓一群人意識到：自己工作的重心變了。

The value isn't what it invented, but making people realize: the center of gravity of their work has shifted.

📚 核心概念

命名時間線：

2020：EleutherAI 的 Language Model Evaluation Harness（測試框架意義）

2025/11：Anthropic 稱 Claude Agent SDK 為「general-purpose agent harness」

2026/01：Aakash Gupta 宣告「2025 是 Agents，2026 是 Agent Harnesses」

2026/02：Mitchell Hashimoto 正式命名 Harness Engineering

2026/02：OpenAI 發表 Codex harness 實戰報告

2026/02：Martin Fowler 發表分析長文

🏭 CDMO 類比

命名 = SOP 編號系統

GMP 環境中，每個作業流程都有 SOP 編號。不是因為流程本身很新 — 操作員早就在做了。是因為一旦命名、編號、標準化，就能：跨站點傳承、用於偏差調查引用、建立訓練課程。Harness Engineering 現在走的是同一條路。

Part III
LangChain 實證 — 同模型、換韁繩、排名飆升

最硬的一個數據：LangChain 的 coding agent 在 Terminal Bench 2.0 上，成績從 52.8% 漲到了 66.5%，排名從 Top 30 跳到 Top 5。重點是：模型完全沒換。他們只改了三樣東西：系統提示詞、工具配置、中間件鉤子。

"We used a simple recipe to iteratively improve deepagents-cli 13.7 points from 52.8 to 66.5 on Terminal Bench 2.0."

同一個模型（GPT-5.2-Codex），換了套韁繩，成績天差地別。模型可能已經不是瓶頸了。瓶頸是你給它搭了個什麼樣的環境。

Same model. Different harness. Dramatically better results.

最常見的失敗模式幾乎是搞笑的：Agent 寫完程式碼，重新讀一遍自己的程式碼，覺得看起來不錯，就停了。自我驗證迴圈（build-verify-fix loop）是最關鍵的改善。

Agents would write a solution, re-read their own code, decide it looked fine, and stop.

他們還發現了「推理三明治」策略 — 規劃和驗證用高推理預算，執行用中等推理預算。全程開最高反而因為超時而表現更差（53.9% vs 63.6%）。

"A reasoning sandwich — high reasoning for planning and verification, with medium for execution — yielded the best results."

⚠️ 重點提示

LangChain 的三層 Harness 改善：

1. 系統提示詞：強調 build-verify-fix loop

2. Context 注入：LocalContextMiddleware 預載環境資訊（目錄結構、工具可用性）

3. 中間件：LoopDetectionMiddleware（防無限重試）+ PreCompletionChecklistMiddleware（強制驗證）

📚 核心概念

Agent 也需要「新人 Onboarding」

Agent 花大量時間摸索工作環境 — 目錄結構、可用工具、Python 版本。LangChain 的解法：開機時直接注入環境 context，減少探索錯誤。這和你給新進 QA 人員一份標準化的環境設定文件是同一個道理。

🏭 CDMO 類比

Build-Verify-Fix = IPC + Hold/Review Point

GMP 批次記錄中的 In-Process Check (IPC) + Hold Point 就是 harness 的驗證迴圈。操作員不能「覺得看起來不錯就放行」— 必須有獨立的 QA 驗證。Agent 自我審查和人自我審查一樣不可靠。

Part IV
OpenAI Codex — 百萬行程式碼的實驗與質疑

OpenAI 自己的實驗：Codex 團隊 3 個工程師，後來擴到 7 個，5 個月，用 Codex 搞出了一個100 萬行程式碼的 beta 產品。零行人工手寫。約 1500 個 PR 合併，每個工程師每天 3.5 個 PR，估算速度是傳統方式的 10 倍。

"We intentionally chose this constraint so we would build what was necessary to increase engineering velocity by orders of magnitude."

但花叔提出了一個沒人聊的問題：速度快了 10 倍，不代表產出好了 10 倍。每人每天 3.5 個 PR，誰在做代碼審查？六個月後需要改需求的時候，這 100 萬行好改嗎？

Speed 10x doesn't mean output quality 10x. Who's doing code review at 3.5 PRs per person per day?

AI 寫的程式碼和人寫的有個關鍵區別：人寫程式碼時會無意識地留下結構線索，方便將來的自己理解。AI 不會。它只解決眼前的任務，不考慮六個月後維護這段程式碼的人會不會罵娘。

Humans leave structural clues unconsciously. AI only solves the immediate task.

⚠️ 重點提示 — 花叔的質疑（需打折）

OpenAI 團隊的 Harness 三層設計（Martin Fowler 整理）：

1. 上下文工程：持續更新的程式碼庫知識庫 + 即時系統狀態。給模型地圖，不是 1000 頁說明書。

2. 架構約束：Linter、結構測試、編譯檢查 — 硬規則，不遵守就不讓過。

3. 垃圾回收：專職 Agent 週期性清理文檔矛盾和架構違規。一個專職找碴的 AI。

🛠️ 實務應用

「AI Slop」清理機制：

OpenAI 團隊最初每週五花 20% 時間手動清理「AI slop」。後來將清理規則編碼為「golden principles」直接寫入 repo，並建立自動化清理流程。這是 Harness 的核心精神：把人工查核變成系統約束。

💡 2-3x 折扣提醒

OpenAI 對自己產品的成功案例有明顯利益衝突。「100 萬行零手寫」的說法需要打折：(1) 程式碼品質未經第三方審計 (2) 長期維護成本未知 (3)「每天 3.5 PR」的可持續性存疑。Martin Fowler 本人也坦言：「with respect to the author and the team, OpenAI do have a vested interest in us believing in AI-maintainable code.」

Part V
Anthropic 三 Agent 架構 — GAN 啟發的品質保證

Anthropic 走了另一條路。他們搞了個三 Agent 架構：規劃者（Planner）負責把簡單指令擴展成詳細的產品規格，生成者（Generator）按迭代一次做一個功能，評估者（Evaluator）跑端到端測試。

Planner → Generator → Evaluator: three agents, inspired by GANs (Generative Adversarial Networks).

靈感來自生成對抗網路（GAN）。訓練一個專門的評估者讓它一直挑刺，比讓生成者自己檢查自己管用得多。誰都不擅長批評自己，AI 也一樣。

"When asked to evaluate work they've produced, agents tend to respond by confidently praising the work — even when, to a human observer, the quality is obviously mediocre."

他們還發現 Claude Sonnet 4.5 有「上下文焦慮」。不是人焦慮，是模型焦慮。上下文太多，表現反而變差。壓縮不夠，必須定期清空重來。Harness 不是越大越好。

"Context Anxiety" — when agents sense they're approaching context limits, they wrap up work prematurely.

最重要的發現：隨著模型進步，harness 應該變簡單而不是更複雜。從 Claude Sonnet 4.5 升級到 Opus 4.6 後，他們移除了 sprint 分解機制，因為新模型可以連續兩小時保持連貫工作。

"The space of interesting harness combinations doesn't shrink as models improve. Instead, it moves."

📚 核心概念

GAN 啟發的 Generator-Evaluator 分離

Generator 寫程式碼前，先和 Evaluator 簽訂「sprint contract」— 約定完成標準。Evaluator 不看靜態程式碼，而是用 Playwright 瀏覽器自動化工具，像人類 QA 一樣實際操作應用。這解決了「自己查自己」的根本問題。

🏭 CDMO 類比

Generator-Evaluator = 生產 + 獨立 QC

GMP 的核心原則之一：生產人員不能同時擔任品質放行人員。Anthropic 的三 Agent 架構完美映射這個原則。Generator = 生產操作員，Evaluator = QC 獨立檢驗。Sprint Contract = 批次記錄的接受標準。

⚠️ Tool Decay 原則

Harness 隨模型進化而簡化

每個 harness 組件都編碼了一個假設：「模型自己做不好什麼」。當模型變強，這些假設必須重新測試。Anthropic 自己實踐了這一點 — Opus 4.6 推出時，他們移除了 sprint 分解，改為利用模型內建的 AI 功能。這就是 Tool Decay 原則的最佳案例。

Part VI
個人實踐 — Harness 是長出來的，不是設計出來的

花叔去年 8 月搭了 Claude Code 自動化寫作工作流。他的 CLAUDE.md 從一個簡單的規則文件，變成了一個路由器：判斷當前任務屬於哪個工作區，然後指向對應的規則。寫公眾號時不會被 iOS 開發的規則干擾。

CLAUDE.md evolved from a simple rules file into a router — directing to the right workspace rules.

Hooks 是另一層。在 agent 執行關鍵操作前後注入腳本。編輯檔案之前自動跑 linting，生成程式碼之後自動做型別檢查。這不是 prompt 裡寫的「請注意代碼規範」，是物理上攔住它，不合格就不讓過。建議和約束，完全兩回事。

Hooks physically block the agent — unlike a prompt "suggestion", constraints are non-negotiable.

Skills 解決了模組化的問題。每個 skill 是獨立的能力包。平時不佔 context，需要時才調用。路由器、hooks、skills、知識庫，加在一起就是一個 harness。沒人告訴他這叫什麼，它自己長出來的。

Skills solve modularity — independent capability packages, loaded only when needed.

和 Mitchell Hashimoto 的做法一樣：每次 agent 犯錯，就工程化一個方案，讓它再也犯不了同樣的錯。他的 AGENTS.md 裡每一行都對應著 agent 過去犯過的一次錯。檔案是活的，一直在長。

"Anytime you find an agent makes a mistake, you take the time to engineer a solution such that the agent never makes that mistake again."

🛠️ 實務應用 — 三步開始

1. 給地圖不給說明書
CLAUDE.md 應該像地圖：專案結構、檔案關係、關鍵約束。不要把每步都寫死。AI 需要方向感，不需要僵化步驟。

2. 每次犯錯加一條規則
空檔案開始，agent 犯一個錯就加一條。三個月後那個檔案就是你的 harness。高度客製，因為全是你場景裡真實出過的問題。

3. 讓 AI 查 AI
Anthropic 的 Evaluator 思路。最簡單的做法：寫完後開一個新對話，把結果貼進去：「找出所有問題」。

🏭 CDMO 類比

活的 CLAUDE.md = CAPA 驅動的 SOP 更新

「每次犯錯加一條規則」就是 CAPA（矯正與預防措施）的精神。GMP 環境中，每個偏差調查的產出就是 SOP 更新或新增控制點。Hashimoto 的 AGENTS.md 和你的 CLAUDE.md 本質上就是一份持續被 CAPA 驅動的 SOP。

Part VII
經驗工程 — 下一代能設計好 Harness 嗎？

Martin Fowler 在文章裡說：如果太早把人類從「in the loop」移到「on the loop」，將來可能沒人真正懂得怎麼回事，也就沒人能設計好的 harness。這句話值得多讀兩遍。

"If we move humans too early from 'in the loop' to 'on the loop', no one may truly understand what's happening."

現在設計 harness 的人，都是寫過很多年程式碼的老手。Mitchell Hashimoto 能給 Ghostty 寫好 harness，因為他理解終端模擬器的每個細節。但下一代呢？新手工程師從第一天起就不寫程式碼，只寫 CLAUDE.md，他能設計出好的 harness 嗎？

Current harness designers are veteran coders. Can newcomers who never write code design good harnesses?

花叔自己是個有意思的樣本。他從來沒手寫過程式碼，所有產品都是 AI 寫的。小貓補光燈上了 AppStore 付費榜 Top 1，累計用戶超百萬。但他的判斷力不來自寫程式碼的經驗，來自和 AI 反覆較勁的上千小時經驗。

His judgment comes not from coding experience, but from thousands of hours wrestling with AI.

問題可能不是「寫程式碼的經驗」能不能被替代。而是：不管你積累的是什麼經驗，足夠多的經驗本身就是設計 harness 的前提。沒有捷徑，換了個賽道而已。也許 Martin Fowler 擔心的不是「沒人寫程式碼了」，而是「沒人願意花夠多時間踩夠多坑了」。

Maybe the concern isn't "no one codes anymore" but "no one is willing to spend enough time making enough mistakes."

⚠️ 重點提示

In the Loop vs On the Loop

In the Loop：人在迴圈中，每步都參與決策

On the Loop：人在迴圈旁，監督但不直接介入

GMP 的「Copilot Mode」本質上就是「in the loop」— 人類做最終放行決策。Harness Engineering 的「on the loop」模式在法規環境中目前不適用。

🏭 CDMO 類比 — O-Ring 理論連結

經驗工程 = 你的 O-Ring 價值

Harness Engineering 加速了所有可自動化的環節，但設計 harness 本身需要深度領域經驗。這正是 O-Ring Bottleneck Theory：AI 越強，設計「韁繩」的人的判斷力越不可替代。花叔用上千小時和 AI 較勁積累的直覺，和你用十年 GMP 經驗積累的偏差調查直覺，本質相同。

💡 比喻說明

花叔的自我樣本分析

他知道 CLAUDE.md 該怎麼寫，但讓他教別人「為什麼這麼寫」，他會卡住。很多決定是直覺做的，直覺來自踩坑，踩坑來自大量重複，大量重複來自時間。這和老程式設計師說的「你寫幾萬行程式碼自然就懂了」其實是一回事 — 只是賽道不同。

Key Takeaways

🎯
模型是商品，Harness 是護城河
LangChain 同模型從 Top 30 跳到 Top 5。效能差異不在模型，在環境設計。投資 harness 的回報率遠高於等待下一代模型。
🔄
Harness 隨模型進化而簡化
Anthropic 從 Sonnet 4.5 升級到 Opus 4.6 時移除了 sprint 分解。每個 harness 組件都編碼一個假設 — 模型變強時要重新測試。
🛡️
約束 > 建議
Prompt 裡寫「請注意品質」是建議。Hooks 裡設定 linter 檢查是約束。一個是口頭叮嚀，一個是物理攔截。效果天差地別。
🧠
設計 Harness 需要足夠的踩坑經驗
不管你積累的是程式碼經驗還是 AI 協作經驗，足夠多的經驗本身就是設計 harness 的前提。沒有捷徑，賽道不同而已。

Practice Questions

Q1 概念理解 Context Engineering 和 Harness Engineering 的邊界在哪裡？你的 Master Prompt 屬於哪一層？

Context Engineering 管「給模型看什麼資訊」，Harness Engineering 則包含 context + 約束 + 反饋迴圈 + 品質檢查。

Master Prompt 本身是 context（告訴模型文件該怎麼生成），但它搭配的 present_files 流程、meta tag 驗證、Dashboard 索引重建等機制，整體構成了一個 harness。單獨的 prompt 是 context，prompt + 檢查機制 + 自動化流程 = harness。

HumanLayer 的定義更精確：Harness Engineering 是 Context Engineering 的子集，專注於 harness 配置點管理。但在實踐中，兩者的邊界是模糊的 — 重要的是整套系統一起運作。

Q2 產業應用 如果你要為 CDMO 的偏差調查流程設計一個 AI Harness，Generator-Evaluator 分離原則會怎麼應用？

Generator Agent：根據偏差描述、批次記錄數據、歷史 CAPA 資料庫，生成初步根因分析報告和 CAPA 建議。

Evaluator Agent：獨立檢查 Generator 的產出 — 確認根因分析是否覆蓋了 Ishikawa 圖的六大類別（人機料法環測）、CAPA 是否具體可執行、是否引用了正確的 SOP 條款、是否遺漏了相關的歷史偏差。

Harness 約束層：硬規則 — 必須在輸出中包含法規引用（21 CFR 211 條款）、必須有效力等級分類、生成的報告必須通過 ALCOA+ 數據完整性檢查（可歸因、可追溯）。

關鍵：最終放行仍然是人類 QA/RA — 這就是 Copilot Mode 下的「in the loop」。

Q3 批判思考 花叔說「沒人願意花夠多時間踩夠多坑了」是真正的風險。但 Harness 的可攜性和標準化不正是在解決這個問題嗎？

這是本文最值得深思的張力。兩面都有道理：

支持標準化：Martin Fowler 想像了一個未來 — 團隊從標準化的 harness 模板開始（類似今天的 service template / golden path），然後根據自己的場景微調。如果 harness 可以像 SOP 一樣被繼承，新人確實不需要從零踩坑。

支持踩坑：但 Hashimoto 的 AGENTS.md 裡「每一行對應一次過去的錯誤」恰恰說明 — harness 的精華在於高度場景化的失敗經驗。通用模板能提供起點，但真正的護城河來自你的場景裡踩過的坑。

CDMO 視角：GMP 的 SOP 就是標準化 harness。新進人員不需要重新發明偏差調查流程。但資深 QA 的判斷力 — 知道哪些偏差需要提升等級、哪些根因分析走錯方向 — 這種直覺無法被 SOP 編碼。標準化解決 80% 的問題，最後 20% 仍然需要經驗。
