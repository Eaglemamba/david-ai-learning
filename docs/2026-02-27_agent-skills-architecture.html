Executive Summary

Agent Skills 是 Anthropic 為 Claude 設計的模組化能力擴展機制。每個 Skill 是一個檔案目錄，包含指令（SKILL.md）、可執行腳本和參考資源，Claude 在需要時自動觸發並載入。核心架構採用三層漸進式揭露（Progressive Disclosure）：第 1 層元資料（~100 tokens，始終載入）、第 2 層指令（<5k tokens，觸發時載入）、第 3 層資源（按需載入，不消耗上下文）。這解決了傳統 Prompt Engineering 的核心痛點——將大量指令塞入系統提示會消耗上下文窗口。Skills 可跨 Claude API、Claude.ai、Claude Code 和 Agent SDK 使用，但不會跨平台自動同步。

Part I: 為什麼需要 Agent Skills？

從 Prompt 到 Skill — 解決重複指令與上下文消耗的痛點

Skills 是可重複使用的、基於檔案系統的資源，為 Claude 提供特定領域的專業知識：工作流程、上下文和最佳實踐，將通用代理轉變為專家。

Skills are reusable, file-system-based resources that give Claude domain-specific expertise: workflows, context, and best practices that transform a general agent into a specialist.

與提示詞（對話層級的一次性任務指令）不同，Skills 按需載入，消除了在多個對話中重複提供相同指導的需要。

Unlike prompts (one-time task instructions at the conversation level), Skills load on demand, eliminating the need to repeat the same guidance across multiple conversations.

💡 比喻說明

想像你是一個新進員工。Prompt 就像同事每次口頭告訴你「這個任務要這樣做」；Skill 則是那份完整的 SOP 手冊放在書架上——你知道它在哪，需要時翻開對應章節就好，不需要每次都請人重複講解。

📚 三大核心優勢

專業化 Claude：為特定領域任務量身定制能力（如 PDF 處理、試算表操作）。

減少重複：建立一次，自動使用——不需要每次對話都重新載入指令。

組合能力：多個 Skills 可以結合使用，建構複雜的多步驟工作流程。

Part II: 三層漸進式揭露架構

Progressive Disclosure — 只載入需要的，節省每一個 Token

Skills 利用 Claude 的 VM 環境來提供超越提示詞所能實現的能力。Claude 在具有檔案系統存取權限的虛擬機器中運作，允許 Skills 以目錄形式存在。

Skills leverage Claude's VM environment to provide capabilities beyond what prompts alone can achieve. Claude operates in a virtual machine with file system access, allowing Skills to exist as directories.

這種基於檔案系統的架構實現了漸進式揭露：Claude 根據需要分階段載入資訊，而不是預先消耗上下文。

This file-system-based architecture enables progressive disclosure: Claude loads information in stages as needed, rather than consuming context upfront.

第 1 層：元資料（始終載入）——Skill 的 YAML 前置資料提供發現資訊。Claude 在啟動時載入此元資料並將其包含在系統提示中。每個 Skill 約 100 tokens，你可以安裝很多 Skills 而不會產生上下文損耗。

Tier 1: Metadata (always loaded) — The Skill's YAML front matter provides discovery information. Claude loads this metadata at startup in the system prompt. ~100 tokens per Skill, so you can install many Skills without context penalty.

第 2 層：指令（觸發時載入）——當你的請求與某個 Skill 的描述匹配時，Claude 透過 bash 從檔案系統讀取 SKILL.md。只有在此時，指令才會進入上下文窗口，少於 5k tokens。

Tier 2: Instructions (loaded on trigger) — When your request matches a Skill's description, Claude reads SKILL.md via bash. Only then do instructions enter the context window, under 5k tokens.

第 3 層：資源與程式碼（按需載入）——Skills 可以捆綁額外的 markdown 檔案、可執行腳本和參考資料。Claude 只在被引用時才存取這些檔案。腳本透過 bash 執行，腳本程式碼本身永遠不會進入上下文——只有輸出結果。

Tier 3: Resources & code (loaded on demand) — Skills can bundle additional markdown files, executable scripts, and reference materials. Claude only accesses these files when referenced. Scripts execute via bash — the script code itself never enters context, only the output.

⚠️ 重點提示

為什麼「漸進式揭露」這麼重要？

Context Window 是 LLM 最寶貴的資源。如果你有 10 個 Skills，每個指令 5k tokens，全部預載就是 50k tokens——還沒開始對話就消耗了大量上下文。

漸進式揭露確保：只有「被觸發的 Skill」才佔用上下文，其餘 Skills 只佔用 ~100 tokens 的元資料。

層級載入時機Token 成本
第 1 層：元資料始終（啟動時）~100 tokens/Skill
第 2 層：指令Skill 被觸發時<5k tokens
第 3 層：資源按需無上限（腳本不入上下文）
💡 比喻說明

三層架構就像一間圖書館：

第 1 層 = 圖書館的目錄卡片（你知道有哪些書，但不佔書桌空間）

第 2 層 = 把需要的書從書架上拿到書桌（佔用閱讀空間）

第 3 層 = 書中的附錄、參考資料和光碟——只在你翻到那頁時才取出

Part III: Skill 結構與撰寫要求

SKILL.md 檔案格式、目錄組織與欄位限制

每個 Skill 都需要一個包含 YAML 前置資料（Front Matter）的 SKILL.md 檔案。必填欄位只有兩個：name 和 description。

Every Skill requires a SKILL.md file with YAML front matter. Only two fields are required: name and description.

name 最多 64 個字元，只能使用小寫字母、數字和連字號，不能包含「anthropic」或「claude」等保留字。description 最多 1024 個字元，不能為空——這段描述決定了 Claude 何時觸發這個 Skill。

name: max 64 chars, lowercase alphanumeric and hyphens only, no reserved words like "anthropic" or "claude". description: max 1024 chars, cannot be empty — this description determines when Claude triggers the Skill.

Skill 目錄可以包含多種檔案：額外的 markdown 檔案（如 FORMS.md）作為專門指導、可執行腳本（如 fill_form.py）、以及參考資源如 API 文件和範本。

A Skill directory can contain multiple file types: additional markdown files (like FORMS.md) as specialized guidance, executable scripts (like fill_form.py), and reference resources like API docs and templates.

Part IV: 跨平台部署與限制

API / Claude.ai / Claude Code / Agent SDK 的 Skill 可用性差異

Skills 可在 Claude 的各代理產品中使用，但自訂 Skills 不會跨平台同步。上傳到一個平台的 Skills 不會自動在其他平台上可用——你需要為每個平台分別管理和上傳。

Skills work across Claude's agent products, but custom Skills don't sync across platforms. Skills uploaded to one platform aren't automatically available on others — you need to manage and upload separately for each platform.

共享範圍也不同：Claude.ai 的自訂 Skills 僅限個人使用者（每個團隊成員必須分別上傳）；Claude API 是工作區範圍（所有成員可存取）；Claude Code 則是個人或專案級，也可透過 Plugins 共享。

Sharing scope also varies: Claude.ai custom Skills are per-user only (each team member uploads separately); Claude API is workspace-scoped (all members can access); Claude Code is personal or project-level, shareable via Plugins.

執行環境約束方面：Claude API 的 Skills 無法存取網路，也無法在執行時安裝套件。Claude.ai 的網路存取取決於使用者/管理員設定。Claude Code 則擁有與本地電腦相同的完整存取權限。

Execution environment constraints: Claude API Skills have no network access and cannot install packages at runtime. Claude.ai network access depends on user/admin settings. Claude Code has full access like any local program.

⚠️ 重點提示

API 無網路 = 腳本不能呼叫外部 API。如果你的 Skill 需要網路（如呼叫第三方 API），目前只有 Claude Code 環境支援。API 環境中的 Skills 必須使用預安裝的套件，無法 pip install。

Part V: 安全考量與最佳實踐

像安裝軟體一樣對待 Skills — 信任來源、審查內容

Anthropic 強烈建議僅使用來自可信來源的 Skills：你自己建立的，或從 Anthropic 獲取的。Skills 透過指令和程式碼為 Claude 提供新能力，惡意 Skill 可以指導 Claude 以不符合聲明目的的方式呼叫工具或執行程式碼。

Anthropic strongly recommends only using Skills from trusted sources: ones you create yourself or obtain from Anthropic. Malicious Skills can direct Claude to call tools or execute code in ways inconsistent with the Skill's stated purpose.

主要風險包括：工具濫用（惡意 Skills 可以以有害方式呼叫 bash 命令）、資料暴露（具有敏感資料存取權限的 Skills 可能被設計為向外部洩露資訊）、以及外部依賴風險（從外部 URL 獲取資料的 Skills 可能被入侵）。

Key risks include: tool abuse (malicious Skills calling bash commands harmfully), data exposure (Skills designed to leak sensitive data externally), and external dependency risks (Skills fetching data from external URLs may be compromised).

🛠️ 安全檢查清單

1. 徹底審查：檢查 Skill 中所有捆綁檔案——SKILL.md、腳本、圖片和其他資源。注意意外的網路呼叫或異常檔案存取模式。

2. 視同安裝軟體：你不會隨便在電腦上安裝來路不明的 .exe，對 Skills 也應該保持同樣的警覺。

3. 特別注意外部 URL：從外部獲取資料的 Skills 風險最高——即使 Skill 本身可信，外部內容可能隨時間改變。

4. 生產環境加倍小心：將 Skills 整合到具有敏感資料存取權限的系統時，格外謹慎審查。

💡 比喻說明

Skill 安全風險就像給新來的實習生一份「工作手冊」——如果手冊是內部資深同事撰寫的，你可以放心讓實習生照做。但如果手冊來路不明，裡面可能寫著「把所有客戶資料寄到這個外部信箱」，而實習生會照單全收。

Key Takeaways

1. Skills = 檔案系統上的可重用專業知識模組
不同於 Prompt（一次性指令），Skills 以目錄形式存在於 VM 中，包含指令、腳本和資源。建立一次，Claude 在需要時自動觸發——解決了跨對話重複載入指令的痛點。
2. 三層漸進式揭露是核心設計哲學
元資料（~100 tokens）始終載入 → 指令（<5k tokens）觸發時載入 → 資源按需從不預載。安裝 20 個 Skills 也只消耗 ~2k tokens 元資料，而非 100k tokens 完整指令。
3. description 是觸發的關鍵
Claude 根據 YAML 中的 description 決定何時觸發 Skill。寫不好 = 不觸發或誤觸發。這是 Skill 設計中最需要反覆迭代的部分。
4. 跨平台不同步，安全視同安裝軟體
自訂 Skills 需在每個平台分別上傳/管理。API 無網路，Claude Code 有完整權限。只使用可信來源的 Skills——惡意 Skill 可以指導 Claude 執行有害操作。

Practice

Q1 概念理解 為什麼 Anthropic 選擇「漸進式揭露」而非「全部預載」的架構？如果 Context Window 無限大，這個設計還有意義嗎？

核心原因是 Context Window 效率。即使 Claude 支援 200k tokens 的上下文，預載所有 Skill 指令仍會佔用寶貴的對話空間，降低回應品質（注意力被分散到不相關指令上）。即使 Context Window 無限大，漸進式揭露仍有價值：(1) 減少「注意力稀釋」——LLM 在較少的相關上下文中表現更好；(2) 降低成本——按 token 計費的 API 模式下，每多載入 5k tokens 都是錢；(3) 降低延遲——更少的輸入 tokens 意味著更快的首字回應。

Q2 產業應用 在 CDMO 製藥環境中，你會如何設計一個偏差調查（Deviation Investigation）的自訂 Skill？三層內容分別放什麼？

第 1 層（元資料）：name: deviation-investigation，description 涵蓋觸發關鍵字如「偏差」「deviation」「CAPA」「根因分析」「root cause」。

第 2 層（SKILL.md）：標準偏差調查 SOP 流程——分類（Critical/Major/Minor）、根因分析方法（5 Whys、魚骨圖）、CAPA 撰寫指引、21 CFR Part 211 相關法規要求。控制在 5k tokens 以內。

第 3 層（Resources）：過去偏差案例模板、常見根因資料庫、CAPA 範例、SA25 Vanrx 設備特定的偏差分類表、驗證腳本。只在 Claude 需要具體範例或執行驗證時才載入。

關鍵考量：GMP 環境中只能作為 Copilot 模式——所有輸出需 QA 人員審核簽核。

Q3 批判思考 「description 驅動觸發」機制有什麼潛在弱點？在什麼情況下這個設計可能失敗？

1. 語義模糊的觸發衝突：多個 Skills 的 description 語義重疊時（如「data analysis」同時匹配 Excel Skill 和自訂 Data Pipeline Skill），Claude 可能觸發錯誤的 Skill 或同時觸發多個。

2. 隱式需求無法觸發：使用者可能說「幫我整理這些數字」而非「create a spreadsheet」，Excel Skill 不被觸發。description 需覆蓋自然語言表達，但 1024 字元上限限制了覆蓋範圍。

3. 缺乏否定觸發：目前沒有機制讓使用者明確說「不要用某個 Skill」。誤觸發時使用者沒有直接控制手段。
