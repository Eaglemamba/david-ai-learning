Part I
🎣 Hooks 條件式事件監控系統

新 Hook 事件一覽

從 v2.1.77 到 v2.1.96，Claude Code 的 Hook 系統新增了五個關鍵事件類型， 讓外部系統得以精確響應 AI 代理的每一個動作節點。最重要的是 v2.1.85 引入的 條件式 if 欄位，允許使用者透過 Permission Rule 語法篩選 Hook 的觸發條件，大幅減少不必要的 Hook 調用。

"Hooks can now be conditionally triggered using the same permission rule syntax, reducing noise and enabling precise workflow automation."

StopFailure（v2.1.78）在代理因錯誤停止時觸發， 可用於自動發送告警或執行清理腳本。 CwdChanged 與 FileChanged（v2.1.83） 則實現了工作目錄與檔案的即時監控——類似 GMP 環境中的環境監測系統（Environmental Monitoring System）。

"CwdChanged and FileChanged hooks mirror real-time environmental monitoring — any deviation from expected state can trigger automated corrective action."

TaskCreated（v2.1.84）讓每個新任務的建立都可以被外部系統追蹤， 而 PermissionDenied（v2.1.89）則在自動模式下分類器拒絕某個操作時觸發， 可用於記錄稽核軌跡（Audit Trail）。

"PermissionDenied hook creates an automatic audit trail — every denied action is recorded, supporting compliance review workflows."

Hook 事件版本時序

v2.1.78 StopFailure — 代理因失敗停止時觸發 新事件
v2.1.83 CwdChanged、FileChanged — 工作目錄與檔案變更 新事件
v2.1.84 TaskCreated — 任務建立時觸發 新事件
v2.1.84 WorktreeCreate 擴充支援 HTTP 類型 Hook 擴充
v2.1.85 條件式 if 欄位，使用 Permission Rule 語法 重大更新
v2.1.89 PermissionDenied — 自動模式拒絕時觸發 新事件
💻 條件式 Hook 設定範例
{
  "hooks": {
    "FileChanged": [{
      "if": "path:src/**",
      "command": "scripts/lint-on-change.sh"
    }],
    "PermissionDenied": [{
      "command": "scripts/audit-log.sh"
    }]
  }
}
Part II
🧩 Plugin 生態系統全面成熟

從腳本到完整應用的躍升

這個階段的 Plugin 系統發生了質的轉變。v2.1.78 引入的 ${CLAUDE_PLUGIN_DATA} 變數讓 Plugin 首次擁有 持久化狀態管理能力——Plugin 不再是無狀態的腳本，而是可以記住跨工作階段資料的「微應用」。

"${CLAUDE_PLUGIN_DATA} transforms plugins from stateless scripts into stateful micro-applications with persistent memory across sessions."

v2.1.80 新增 effort frontmatter，讓每個 Skill 可以宣告自己 所需的 effort 等級，避免高精度任務被低 effort 設定影響品質。v2.1.82 的 keep-coding-instructions frontmatter 則允許 Plugin 在輸出時保留程式碼撰寫指引， 確保生成內容符合專案規範。

"Per-skill effort declarations prevent quality degradation — a validation skill can demand high effort even when global settings are relaxed."

v2.1.91 允許 Plugin 在 bin/ 目錄下放置可執行檔並直接呼叫，使 Plugin 得以 封裝任意 CLI 工具。v2.1.90 的離線快取確保 Marketplace Plugin 即使在網路中斷時仍可運作——對受控環境（如藥廠內網）尤為重要。

"Offline marketplace cache is critical for air-gapped or restricted-network environments — plugins remain functional without internet access."

Plugin 能力演進里程碑

🌱 v2.1.78 — 狀態管理誕生
${CLAUDE_PLUGIN_DATA} 變數 + Agent frontmatter 支援，Plugin 可儲存跨 Session 資料並定義 Agent 行為
⚙️ v2.1.80–82 — 精細化控制
effort frontmatter（技能級 effort 宣告）、keep-coding-instructions（輸出規範保留）、 source: 'settings'（設定檔驅動 Marketplace）
🏛️ v2.1.84 — 企業管控
Channel Plugin 白名單（managed settings）、組織政策封鎖安裝（v2.1.85），IT 部門可精確控管哪些 Plugin 可用
📦 v2.1.90–94 — 生態完整
離線 Marketplace 快取（v2.1.90）、bin/ 可執行檔支援（v2.1.91）、 frontmatter name 作為穩定呼叫識別符（v2.1.94）
Part III
🏢 企業受管設定強化

三層管理架構完善

v2.1.83 引入的 managed-settings.d/ drop-in 目錄 是企業部署的重大突破。IT 管理員可以將不同政策拆分為獨立檔案放入此目錄， Claude Code 啟動時會自動合併所有設定——如同 Linux 的 /etc/nginx/conf.d/ 機制。 這讓不同部門的政策可以各自維護，不再需要手動合併單一大型設定檔。

"managed-settings.d/ enables policy-as-code: each department maintains its own settings fragment, merged at startup without conflicts."

v2.1.83 同步推出的 sandbox.failIfUnavailable 設定確保沙箱在無法啟用時 拒絕執行而非靜默降級，對安全合規（如 21 CFR Part 11 的存取控制要求）至關重要。

"sandbox.failIfUnavailable enforces a fail-safe posture — in regulated environments, silent degradation is unacceptable."

v2.1.92 加入的 forceRemoteSettingsRefresh 政策 允許管理員強制所有終端即時同步最新政策，無需使用者手動重啟——這對需要緊急下發安全政策的場景（如發現安全漏洞後的快速修補）特別重要。

"forceRemoteSettingsRefresh enables zero-touch emergency policy updates — critical for rapid security patch distribution across all endpoints."

企業管理能力對照

🏭 GMP 類比：變更管控
managed-settings.d/ ≈ SOP 分冊管理：每個部門各自維護自己的 SOP 章節， 品保部門（QA）負責整合並確保無衝突。修改某部門的規定只需更新對應分冊，不影響其他部門。
⚠️ v2.1.83 — 安全強制設定
sandbox.failIfUnavailable: true
沙箱不可用時拒絕執行，避免在非隔離環境下意外執行危險操作。 建議所有生產環境與受管設備啟用。
📡 v2.1.92 — 即時政策同步
forceRemoteSettingsRefresh 政策 + 互動式 Bedrock 設定精靈， 企業端佈署與政策更新的摩擦力大幅降低。/release-notes 升級為互動式版本選擇器， 讓管理員快速查看特定版本的變更內容。
Part IV
☁️ Bedrock 平台整合與穩定化

Mantle 整合與授權修正歷程

Bedrock/Mantle 整合是這個階段最具戲劇性的演進——從功能引入到回歸（Regression）， 再到緊急修正，歷經三個版本。v2.1.82 首次引入 Mantle 支援 （CLAUDE_CODE_USE_MANTLE=1），同時修正了 Bedrock Bearer Token 的授權問題。

"The Bedrock integration journey illustrates the challenge of platform abstraction — v2.1.94 introduced Mantle natively but also regressed auth, fixed by v2.1.96."

v2.1.94 將 Mantle 支援提升為正式功能，並同步將 預設 effort 等級從 Medium 提升至 High， 適用於 API Key、Bedrock、Vertex 及 Enterprise 用戶。然而此版本同時引入了 Bedrock 請求 403 錯誤的回歸問題，由 v2.1.96 緊急修正。

"Default effort elevation to High signals confidence in cost efficiency — the quality improvement outweighs the token cost for production use cases."

v2.1.92 推出的互動式 Bedrock 設定精靈 大幅降低了第三方驗證的設定門檻，不再需要手動設定複雜的環境變數組合。 v2.1.94 也修正了 Sonnet 3.5 v2 在 Bedrock 上的呼叫問題（改用 us. inference profile ID）。

Bedrock 整合版本時序

v2.1.82 首次 Mantle 支援 CLAUDE_CODE_USE_MANTLE=1 + Bearer Token 修正 Bedrock
v2.1.82 預設 effort 改為 High（API/平台用戶）行為變更
v2.1.92 互動式 Bedrock 設定精靈（第三方驗證）新功能
v2.1.94 Mantle 升為正式功能 + Sonnet 3.5 v2 inference profile 修正 重大更新
v2.1.94 預設 effort → High 擴至 Enterprise 用戶 行為變更
v2.1.96 修正 AWS_BEARER_TOKEN_BEDROCK 403 授權失敗（v2.1.94 回歸）緊急修正
Part V
🔌 MCP 協議標準化進化

從專屬實作到 RFC 標準

v2.1.85 的 MCP OAuth 遵循 RFC 9728 Protected Resource Metadata 探索機制，標誌著 MCP 認證從私有實作轉向開放標準。 這意味著任何支援 RFC 9728 的 OAuth 伺服器都可以直接為 MCP 提供身份驗證， 無需針對 Claude Code 進行特殊適配。

"RFC 9728 compliance transforms MCP OAuth from a proprietary handshake into a standard protocol — any compliant OAuth server works out of the box."

v2.1.85 同步為 headersHelper 腳本新增 MCP 伺服器名稱與 URL 環境變數， 讓一個 headersHelper 腳本可以根據不同的 MCP Server 動態產生對應的請求頭， 實現真正的多 Server 統一認證管理。

v2.1.91 最受矚目的更新是 MCP 工具結果持久化覆寫： 透過 metadata 標注，單次工具呼叫的結果可保留至 500K token， 突破了原本的上下文限制。這對需要長時間保存大型分析結果的場景（如基因組數據分析、 大型文件審查）具有重要意義。

"500K MCP result persistence enables long-horizon tasks — a genomic analysis result stays in context across the entire workflow without re-fetching."

MCP 進化關鍵里程碑

📜 v2.1.85 — RFC 9728 標準化
MCP OAuth 遵循 RFC 9728 Protected Resource Metadata 探索機制。 headersHelper 腳本可透過 MCP_SERVER_NAME、MCP_SERVER_URL 環境變數動態識別呼叫來源。
💾 v2.1.91 — 500K 結果持久化
MCP 工具結果可透過 metadata 標注覆寫持久化上限至 500,000 token。 Shell 執行開關（skills/custom commands 各自獨立控制）同步推出。
🔄 前後對比
v2.1.49–76 時期：MCP 以私有 OAuth、單一 headersHelper 為主

v2.1.77–96 時期：RFC 9728 標準 OAuth + 多 Server 動態 Header + 500K 持久化

⏸️ v2.1.89 — defer 決策暫停機制
在 Headless 模式（無人值守）下，Claude Code 遇到需要人工確認的工具呼叫時， 可以選擇「defer」決策——暫停工作階段並等待人工恢復，而非直接失敗或跳過。 這對 CI/CD 流水線中的半自動化任務特別重要。
🚫 v2.1.89 — 無閃爍渲染 CLAUDE_CODE_NO_FLICKER=1
設定環境變數 CLAUDE_CODE_NO_FLICKER=1 啟用無閃爍渲染模式， 消除長時間輸出時終端機畫面的視覺抖動問題，改善長工作階段的使用舒適度。
📖 v2.1.78 — 逐行串流輸出
v2.1.78 實作逐行串流回應（line-by-line streaming），讓使用者可以更即時地看到 Claude 的輸出，而非等待整段完成後才顯示。（注意：Windows 因渲染問題暫時停用此功能）
🎓 v2.1.90 — /powerup 互動課程
全新 /powerup 指令提供互動式功能學習，內含動畫演示各項 Claude Code 進階功能。設計上特別適合新用戶快速上手，以及舊用戶探索不熟悉的功能。
🔤 v2.1.94 — CJK 多位元組文字修正
修正 stream-json 在 UTF-8 序列跨 chunk 邊界時中文/日文/韓文字元損毀的問題。 此修正對繁體中文、日文等 CJK 用戶具有直接意義——之前在長輸出中可能出現亂碼。
🖥️ v2.1.84 — PowerShell 工具（Preview）
Windows 平台推出 PowerShell 工具作為 opt-in 預覽功能。Windows 用戶可以讓 Claude 直接執行 PowerShell 腳本，不再局限於 Bash 模擬環境。這是 Claude Code Windows 原生化支援的重要里程碑。
⚡ v2.1.86 — Windows 設定磁碟寫入優化
修正了 Windows 上不必要的設定檔磁碟寫入問題（每次操作都會寫入設定，即使設定 未改變），顯著改善 Windows 平台的整體效能，特別是在 HDD 或網路磁碟機上。
⚠️ v2.1.78 — Windows 串流暫時停用
逐行串流輸出功能在 Windows 上因渲染問題被停用。Windows 用戶仍使用整段輸出模式， 待渲染問題解決後將重新啟用。
✅ v2.1.84 — 多螢幕 computer-use 修正

❓ 深度理解問題集
Q1 概念理解 managed-settings.d/ drop-in 目錄的設計邏輯為何？與單一設定檔相比有哪些優勢？

managed-settings.d/ 遵循 Unix 的「drop-in configuration」慣例（如 /etc/nginx/conf.d/、systemd 的 override 機制）。每個設定片段是獨立的 JSON/YAML 檔案，Claude Code 啟動時自動合併所有片段。

主要優勢有三：第一，模組化管理——安全政策、模型設定、Plugin 白名單可以分別由不同團隊維護，修改某片段不影響其他；第二，版本控制友好——每個設定片段可以獨立進入 Git 倉庫，有清晰的變更歷史；第三，緊急修補簡單——發生安全事件時只需替換或刪除特定片段，不需要手動編輯大型設定檔並承擔語法錯誤風險。

Q2 實務應用 MCP 工具結果 500K 持久化在藥廠研究場景中有哪些具體應用？

在生命科學研究中，MCP 工具結果 500K 持久化有幾個關鍵應用場景：

1. 大型臨床試驗數據分析：一次 MCP 查詢可能返回數千筆患者記錄或試驗結果，500K 持久化允許這些結果在整個分析工作流程中保持可用，不需要重複查詢資料庫。

2. 文件審查工作流：藥品申請文件（如 NDA/BLA）通常包含大量技術文件，MCP 工具一次讀取後可持久保留在上下文中，支援後續的交叉引用和一致性檢查。

3. 序列分析（Sequence Analysis）：基因序列或蛋白質結構資料量大，持久化避免了反覆呼叫計算密集型的分析 API，顯著提升工作流效率。

Q3 問題診斷 使用 AWS Bedrock 且升級了 v2.1.94 的團隊，如何快速確認是否受到 403 授權回歸問題影響？

受影響的條件：使用 AWS_BEARER_TOKEN_BEDROCK 環境變數設定 Bearer Token 驗證，或使用 CLAUDE_CODE_SKIP_BEDROCK_AUTH=1 跳過預設授權流程的環境。

快速確認方法：執行 claude --version 確認版本號；若為 v2.1.94 或 v2.1.95，嘗試執行任何 Bedrock API 請求，若回傳 403 "Authorization header is missing" 錯誤，即為受影響。

解決方案：直接升級至 v2.1.96（claude update），此版本已修正回歸問題。若無法立即升級，可臨時回退至 v2.1.93 或更早的穩定版本。

Q4 架構設計 條件式 Hook（if 欄位）如何與 v2.1.76 引入的 MCP Elicitation 協同工作，設計更智能的工作流？

MCP Elicitation（v2.1.76）允許 MCP 伺服器在任務執行中途請求使用者輸入；條件式 Hook（v2.1.85）允許根據上下文自動觸發外部動作。兩者結合可以設計出「事件驅動 + 人工確認」的混合工作流。

實例設計：在藥品批次放行流程中，MCP Server 執行品質檢驗計算（QC calculation）後，若發現偏差（OOS — Out of Specification），可透過 Elicitation 請求品管人員確認；同時 FileChanged Hook 設定條件 if: "path:**/qc-results/**"，自動觸發通知腳本向品保主管發送告警。整個流程做到：自動計算 → 自動監控 → 人工確認關鍵決策 → 自動通知。

Q5 效能優化 Opus 4.6 輸出 Token 從 32K 提升至 64K 對實際工作流有何影響？何時應使用高 Token 上限？

64K Token 輸出上限的實際意義：以繁體中文計算，1 Token ≈ 1–1.5 個中文字，64K Token 約相當於 4–6 萬字的純文字輸出，或等同於一份完整的藥品申請技術文件（CTD Module 2/3 摘要部分）的長度。

適合高 Token 輸出的場景：一次性生成完整的 SOP 文件、長篇技術評估報告、大型程式碼重構（單次輸出完整重寫版本）、複雜法規文件的逐條分析回應。

需要注意的是：高輸出 Token 不代表更好的品質，對話輸入上下文（Context）仍有限制。建議將高 Token 輸出搭配清晰的結構化提示使用，並在輸出完整文件後再進行細部調整，而非迭代式生成。
