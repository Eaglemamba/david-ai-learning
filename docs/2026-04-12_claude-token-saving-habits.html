Part I：Token 消耗的核心機制
Foundation
Claude 計算的是 token。一個 token 大約等於一個英文單字。你送出一則訊息，Claude 會從頭重讀你整段對話。每一則先前的訊息、每一段先前的回答，全部。
"You send one message, and Claude re-reads your entire conversation from the top. Every previous message. Every previous answer. All of it."
所以第 1 則訊息成本很低，但第 30 則訊息？Claude 要先重讀 29 輪先前的交流，然後才開始思考你的新問題。這就是你的額度消失的原因——對話越長，每則訊息越貴。
"So message 1 costs very little. But message 30? Claude is re-reading 29 previous exchanges before it even starts thinking about your new question."
💡 比喻說明

想像你每次問同事問題時，他必須先從頭重讀你們所有的 email 往來。第一封信很快，但到了第 30 封，他光是「重讀歷史」就花了 95% 的時間，實際回答只佔 5%。這就是 LLM 的 context window 運作方式——它沒有「記憶」，每次都是全量重讀。

💰 成本分析

文中引用的數據（需獨立驗證）：

20 則訊息對話 ≈ 105,000 tokens
30 則訊息對話 ≈ 232,000 tokens
一位開發者追蹤發現 98.5% 的 token 花在重讀歷史，僅 1.5% 用於實際產出

Part II：冷門但高效的技巧 (Habits 1-9)
Advanced Habits
1 上傳前先轉檔。單頁 PDF 約花費 1,500-3,000 tokens，截圖更貴（1000x1000 圖片約 1,300 tokens）。上傳前先提取文字，用純文字或 Markdown 格式貼上相關段落。裁切截圖到只剩需要的部分，可從 1,300 降到 100 tokens 以下。
"Before uploading, extract the text. Copy-paste the relevant sections into a plain text or markdown file. Crop screenshots tight to only the part that matters."
2 在 Chat 規劃，最後才建檔。Anthropic 確認檔案建立（試算表、文件、簡報）比一般聊天訊息消耗更多額度。先在 Chat 規劃結構、確認架構，最後才到 Cowork 建立檔案。在便宜的產品做思考，在貴的產品做建構。
"You do the thinking in the cheap product. You do the building in the expensive one."
3 說「問我問題」而非寫長 prompt。500 字的 prompt 每次 Claude 重讀對話就花 500 tokens。但如果你寫 15 字的 prompt，讓 AskUserQuestion 工具幫你收集資訊，點選選項幾乎不花成本。作者的萬用句型：「I want to [任務] to [成功標準]. Read my folder. Ask me questions using AskUserQuestion before you start.」
"Clicking options costs almost nothing. Typing paragraphs of instructions costs a lot."
4 用語音轉文字（Wispr Flow）給出更豐富的回答。打字時你會寫出懶惰的 prompt（「改好一點」），Claude 猜錯，你就不斷發更多訊息修正。用說的，你自然會一次給出更多上下文。更少訊息 = 更少 context 重讀 = 省 token。
"When you speak, you naturally give more context in one-shot."
5 別要求重做整份文件。當報告第 3 節有問題，不要說「重做報告」，說「只重做第 3 節，其他保持不變以省 token」。每次全部重做等於重新產出整份文件的 output token。同時加上「No commentary. No explanations. Just the output.」來避免 Claude 的客套語消耗 token。
"Every sentence of 'Happy to help! Here's what I did...' is tokens you're paying for."
6 批量合併任務到一則訊息。三個分開的 prompt = 三次完整 context 重讀。一個包含三個任務的 prompt = 一次重讀。不要分三次發：「摘要這篇文章」→「列出重點」→「建議標題」，合併成一則：「摘要這篇文章、列出重點、並建議標題。」
"Three separate prompts = three full context reloads. One prompt with three tasks = one reload."
7 每次用相同的 prompt 結構。Anthropic 確認常用的相似 prompt 會被部分快取（partial caching）。維持穩定的 prompt 模板庫，只替換變數部分。
"Keep a stable prompt library and swap only the variable part."
8 編輯訊息，而非發送追加修正。在 Chat 中你可以點擊「編輯」修改原始訊息並重新生成。舊的交流會被取代而非堆疊。每次你發「不是，我是說...」或「其實，把 X 改成 Y」，就是在增加對話歷史。編輯按鈕完全避免這個問題。
"The old exchange gets replaced. Not stacked."
9 根據任務選對產品。快速問答 → Chat + Haiku。寫報告 → Cowork + Opus。做圖表 → Code + Sonnet。每個產品的 token 成本不同，Chat 最輕，Cowork 最重。
"Every product has different token costs per interaction. Chat is the lightest. Cowork is the heaviest."
⚠️ 重點提示

「問我問題」策略的底層邏輯：這是 Pull vs Push 的差異。Push 模式是你一次塞入大量指令（高 input token），Claude 可能還猜錯方向需要多輪修正。Pull 模式是 Claude 主動提問，你用短回答或點選確認（低 token），最終得到更精準的結果。

🛠️ 實務應用

CDMO 情境對應：

技巧 1（轉檔）直接適用於上傳 SOP、Batch Record、Deviation Report 等 PDF 到 Claude 時。這些文件通常 20-50 頁，如果直接上傳 PDF，每次新對話都會燒掉 30,000-150,000 tokens。

更好的做法：提取相關章節為純文字，放在 Claude Project 中供多次引用。

📚 核心概念

編輯 vs 追加的 Token 差異：

假設你在第 10 則訊息發現方向錯誤：

追加修正：第 11 則 = 重讀 10 則 + 修正指令 + 新回覆
編輯第 10 則：重讀 9 則 + 修正後的指令 + 新回覆

如果你在第 5 則就發現問題，編輯可省下重讀第 6-10 則的累積成本。越早編輯，省越多。

⚖️ 工具比較

Claude 產品 Token 成本排序：

Chat（Haiku）< Chat（Sonnet）< Chat（Opus）< Code < Cowork

選擇原則：任務複雜度決定產品，不要用 Cowork + Opus 做可以在 Chat + Haiku 完成的事。

注意：確切的 token 計費機制 Anthropic 未完全公開，此排序基於作者觀察。

Part III：基本但重要的技巧 (Habits 10-17)
Essential Habits
10 個人資料檔控制在 2,000 字以內。Cowork 在每個任務前都會讀取你的資料夾。如果你的 about-me 檔案有 22,000 字，那就是每次開工前先燒掉幾千 tokens。作者建議在每次 Cowork 結束時，用 prompt 產出 session-notes.md 紀錄關鍵決策，下次開工先讀這份筆記。
"You carry the context forward without re-explaining everything from scratch."
11 重啟對話而非發追加修正。當 Cowork 搞錯方向，你的本能是發「不是，我是說...」。但每則追加都疊在完整的對話歷史上。可以用「從這裡重啟對話」回到較早的訊息。越往上重啟，省越多 token。
"The higher up you restart, the more tokens you save."
12 每 15-20 則訊息，摘要後開新對話。長對話是 token 焚化爐。讓 Claude 摘要所有重點，複製摘要，開新對話，貼上作為第一則訊息。
"Ask Claude to summarize everything important, copy that summary, open a new session, paste it as your first message."
13 簡單任務用 Sonnet 或 Haiku，深度工作才用 Opus。文法檢查、腦力激盪、格式調整——Sonnet 完全能處理。Opus + Extended Thinking 是重型機械，不要用重型機械搬椅子。如果任務 Claude 30 秒內就能回答，大概不需要 Opus。
"Don't use heavy machinery to move a chair."
14 不要把整個資料夾丟進 Cowork。每個 Cowork 讀取的檔案都是 token 支出。如果檔案太大，Cowork 會開始「粗略摘要」而非仔細閱讀。如果這個任務不需要某個檔案，就不要讓它讀。不需要任何檔案的任務（如用 connector 寫 email），選擇零資料夾。
"Zero folders = zero local file context = tokens saved before you even type."
15 話題改變時開新對話。你先問 LinkedIn 貼文，接著問客戶提案，然後問食譜——在同一個對話裡。Claude 在想你的晚餐時，還在重讀 LinkedIn 貼文和客戶提案的對話。那些舊訊息是死重量，純粹浪費 token。
"Those old messages are dead weight. Tokens burned on context that does nothing for the current question."
16 關閉不用的功能。Web Search、Connectors、Explore 模式都會在每次回覆時增加 token。寫你自己的內容時？關掉搜尋和工具。做簡單文法檢查？關掉 Extended Thinking。作者的預設：全部關閉，依任務開啟。
"My default: everything off. I turn features on per task, not per account."
17 用 Projects 管理重複使用的檔案。如果你把同一份 PDF 上傳到五個不同的對話，Claude 每次都要重新 tokenize 那份文件。用 Projects 取代——上傳一次，之後都引用快取版本。付費方案的 Projects 還使用 RAG，只檢索相關片段而非載入整份文件。
"Every new conversation inside that project references it without burning tokens again."
🛠️ 實務應用

Session Notes 策略在 CDMO 的應用：

這與 GMP 的 Batch Record 邏輯類似——每次操作結束時記錄關鍵決策和結果，下次操作時先 review 紀錄而非從頭來過。

對 Claude Code 自動化流程也適用：每次 pipeline 執行結束產出 session summary，下次運行時作為 context 注入。

⚠️ 重點提示

Project RAG 的重要性：

技巧 17 是這篇文章中最有結構性影響的建議。Projects 不只省 token，更改變了知識管理的方式：從「每次對話都重新上傳」轉變為「建立持久的知識庫」。

這正是你目前的 GitHub-hosted Learning Dashboard 已經在做的事——但在 Claude.ai 層面，確保常用參考文件（如 PDA Technical Reports、ISPE 指南）放在 Project 而非每次上傳。

💡 比喻說明

功能開關 = 電器開關：

每個開啟的功能就像打開一盞燈——即使你沒在用它，它也在消耗電力（token）。Web Search 開著但你在寫內部文件？那就像打開浴室的燈去照客廳。養成「全關，按需開」的習慣。

📚 核心概念

Prompt Caching（提示快取）：

技巧 7 和 17 都涉及 Anthropic 的 prompt caching 機制。當你重複使用相同的系統提示或 Project 內容時，這些 token 可能被快取，不需要完整重新處理。這是 API 層面的功能，在 claude.ai 消費者端的確切行為 Anthropic 並未完全揭露。

Part IV：進階設定與周邊策略 (Habits 18-23)
Optimization
18 關閉 Memory，改用 User Preferences。沒有存儲 context 的新對話會浪費 3-5 則訊息在設定上（「我是行銷人、我寫輕鬆風格...」）。在 Settings > General > Personal Preferences 設定一次。也設定 Styles（在模型選擇器的 + 中）。一次設定，永久節省。
"One setup, permanent savings."
19 用排程任務處理重複工作。如果你每週都跑同樣的報告或研究任務，不要手動在不斷增長的 Cowork 對話中做。使用 /schedule 功能。
"Use the /schedule plugin."
20 給 Claude Code 明確的範圍。Code 傾向「往外探索」——讀檔案、掃目錄、跑檢查。如果你不告訴它確切需要什麼，它會探索視線內的一切，到處浪費 token。要具體：「從這個 CSV 建立一個長條圖，顯示 2025 年月營收，存為 chart.png。」
"Be specific. Don't leave room for Claude to explore."
21 用 CLAUDE.md 設定永久 context。Code 在每個任務前都會讀取 CLAUDE.md（如果存在）。把你的常用指令放在那裡。但 Anthropic 也警告，臃腫的 CLAUDE.md 會讓 Claude 忽略你的實際指令。保持精簡。偶爾才用的 workflow 搬到 Skills 中——Skills 按需載入，CLAUDE.md 每次都載入。
"Skills load on demand. CLAUDE.md loads every single time."
22 分散工作到全天。Claude 使用滾動式 5 小時窗口計算額度限制。如果你在一個早上就燒光所有額度，當天大部分容量就浪費了。分成 2-3 個時段：早上、下午、晚上。
"Split into 2-3 sessions: morning, afternoon, evening."
23 不要把 Claude 用在它不擅長的事。Claude 不能生成圖片——如果你花 5 則訊息嘗試描述視覺效果，那是浪費在永遠不會成功的任務上的 token。圖像生成用 Gemini 或 ChatGPT，即時搜尋用 Grok。
"That's 5 messages of tokens wasted on a task Claude was never going to solve."
🛠️ 實務應用

CLAUDE.md vs Skills = CDMO 的 SOP vs Work Instructions：

CLAUDE.md 就像 Site Master File——每次都讀，必須精簡到只包含真正「全局適用」的規則。Skills 就像 Work Instructions——特定程序需要時才調用。

你目前的 SKILL.md 架構（v1.5.x）已經體現了這個原則。關鍵是定期審計 CLAUDE.md，移除已被模型能力取代的規則（Tool Decay 原則）。

⚠️ 重點提示

技巧 23 的延伸——O-Ring Bottleneck 應用：

選擇正確的工具做正確的事，正是 O-Ring 理論的實踐。Claude 的價值在語言推理和結構化分析，不在圖像生成。把 Claude 用在它是 bottleneck 解決者的任務上，而非勉強讓它處理它本質上無法勝任的任務。

📚 核心概念

滾動窗口 vs 固定額度：

Claude 的額度不是「每天固定量」，而是 5 小時滾動窗口。這意味著你在 T 時刻使用的 token，到 T+5 小時就「釋放」了。策略上的意義是：避免在單一時段集中使用，而是分散到全天不同時段。

注意：確切的窗口長度和計算方式可能因方案等級而異。

Key Takeaways
📊
Token 累計效應是核心問題
Claude 每次回覆前會重讀整段對話歷史。對話長度呈線性增長，但 token 消耗呈二次方增長。所有節省策略都圍繞「縮短有效對話長度」這一核心。
🔧
Input 精簡是最高 ROI 策略
轉檔（PDF→文字）、裁切截圖、用 Pull 式提問取代 Push 式長 prompt——這些在 input 端的優化影響每一次後續互動的成本。
⚖️
產品 / 模型匹配節省最多
Chat + Haiku 處理簡單任務、Opus 只用於深度工作、Projects 避免重複上傳——工具選擇錯誤是最大的隱性浪費。
🔄
對話生命週期管理
編輯而非追加修正、15-20 則後摘要換新、話題切換時開新對話、分散使用到全天不同時段——把對話視為「有壽命的資源」而非「無限擴充的筆記本」。
Practice Questions
Q1 概念理解 為什麼第 30 則訊息比第 1 則訊息貴得多？這背後反映了 LLM 架構的什麼限制？

因為 LLM 沒有持久記憶（stateless），每次生成回覆時必須將整段對話歷史放入 context window 重新處理。第 30 則訊息需要重讀前 29 輪的所有 input + output token，導致每則新訊息的邊際成本隨對話長度線性增長，而累計總成本呈二次方增長。

這反映了 Transformer 架構的根本設計——attention 機制需要同時處理整個 context window 中的所有 token，無法像人類一樣「只記住重點」。這也是為什麼 Anthropic 和其他廠商在 prompt caching、RAG、context window 管理上持續投資。

Q2 產業應用 在 CDMO 環境中，如果你要用 Claude 分析一份 50 頁的 Deviation Report，如何應用文中的技巧最大化 token 效率？

結合多個技巧的最佳實踐：

1. 轉檔（技巧 1）：從 50 頁 PDF 提取相關章節（如 Root Cause Analysis、CAPA 計畫），轉為純文字或 Markdown。50 頁 PDF 可能花 75,000-150,000 tokens；提取關鍵 5 頁的文字可能只需 5,000 tokens。

2. 放入 Project（技巧 17）：如果這份 Deviation Report 會在多次對話中引用，放在 Project 中讓 RAG 按需檢索，而非每次上傳。

3. 批量提問（技巧 6）：不要分三次問「根本原因是什麼？」「CAPA 是否充分？」「時效性如何？」——合併為一則訊息。

4. 選對模型（技巧 13）：初步摘要用 Sonnet，深度分析和 GMP 合規判斷用 Opus。

Q3 批判思考 文中聲稱「Anthropic 確認常用的相似 prompt 會被部分快取」（技巧 7）。你認為這個說法的可信度如何？它對你的 prompt 設計策略有什麼影響？

可信度評估：作者未提供 Anthropic 文件連結，且以 influencer 身分聲稱「Anthropic confirmed」需要打折扣。Anthropic API 確實有 prompt caching 功能（有官方文件），但 claude.ai 消費者端的快取行為並未完全公開。這可能是作者將 API 層面的功能推論到消費者端。

即便如此的實用建議：無論快取是否存在，維持穩定的 prompt 模板庫本身就是好的 prompt engineering 實踐——它確保一致性、可重現性，也方便迭代優化。你目前的 SKILL.md 系統已經體現了這個原則。

需要補充：查閱 Anthropic 官方關於 prompt caching 在 claude.ai（非 API）的確切行為說明。
