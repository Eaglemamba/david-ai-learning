Executive Summary

Boris Tane 在九個月的密集使用後，發展出一套以「計畫先行」為核心的 Claude Code 工作流。他的方法與多數開發者的「打提示、修錯誤、重複」迴圈截然不同：永遠不讓 AI 寫程式碼，直到你審核並批准了書面計畫。整個流程分為四個階段 — Research（深度閱讀）、Plan（撰寫計畫）、Annotate（人類標註修正）、Implement（機械式執行）。最關鍵的創新是「標註循環」(Annotation Cycle)，開發者在 Markdown 計畫文件中直接寫下修正意見，讓 AI 反覆修改計畫直到完全正確，然後才進入執行。這套方法將「思考」與「打字」分離，確保人類始終掌握架構決策，而 AI 處理機械式的程式碼撰寫。

Part I

Research Phase — 深度閱讀是一切的起點

Boris 的每一個重要任務都從「深度閱讀指令」開始。他要求 Claude 在做任何事之前，徹底理解程式碼庫的相關部分，並且永遠要求將發現寫成持久的 Markdown 文件，而不只是聊天中的口頭總結。

Every meaningful task starts with a deep-read directive. I ask Claude to thoroughly understand the relevant part of the codebase before doing anything else. And I always require the findings to be written into a persistent markdown file, never just a verbal summary in the chat.

注意用語：「deeply」、「in great details」、「intricacies」。沒有這些詞，Claude 會略讀。它會讀一個檔案、看到函式簽名層級的功能就繼續往下。你需要發出訊號告訴它：表面層級的閱讀是不被接受的。

Notice the language: "deeply", "in great details", "intricacies". Without these words, Claude will skim. It'll read a file, see what a function does at the signature level, and move on. You need to signal that surface-level reading is not acceptable.

這是 AI 輔助寫程式中最昂貴的失敗模式，而且不是語法錯誤或邏輯錯誤。是那些在隔離中能運作，但會破壞周圍系統的實作。例如一個忽略現有快取層的函式、一個不考慮 ORM 慣例的資料遷移、一個複製了已存在邏輯的 API 端點。Research 階段可以防止這一切。

This is the most expensive failure mode with AI-assisted coding, and it's not wrong syntax or bad logic. It's implementations that work in isolation but break the surrounding system. The research phase prevents all of this.

💻 Research Prompt 範例
read this folder in depth, understand
how it works deeply, what it does and
all its specificities. when that's done,
write a detailed report of your learnings
and findings in research.md
go through the task scheduling flow,
understand it deeply and look for
potential bugs... keep researching
until you find all the bugs, don't
stop until all the bugs are found.
💡 比喻說明

這就像派一個新進工程師去理解系統之前，要求他先寫一份「系統理解報告」。你不是要考他，而是在他動手改東西之前，先確認他真的懂。如果報告有錯，你在他寫任何程式碼之前就能糾正。這比事後修復便宜 10 倍。

⚠️ 重點提示

Written artifact（research.md）是審查介面。你可以閱讀它、驗證 Claude 是否真正理解系統、在任何規劃發生之前糾正誤解。如果研究是錯的，計畫就會是錯的，實作也會是錯的。Garbage in, garbage out。

Part II

Planning Phase — 用 Markdown 取代內建 Plan Mode

審查完 research 之後，Boris 要求 Claude 在獨立的 Markdown 文件中撰寫詳細的實作計畫。生成的計畫包含方法的詳細說明、展示實際變更的程式碼片段、將被修改的檔案路徑，以及考量與權衡。

Once I've reviewed the research, I ask for a detailed implementation plan in a separate markdown file. The generated plan always includes a detailed explanation of the approach, code snippets showing the actual changes, file paths that will be modified, and considerations and trade-offs.

Boris 明確表示不使用 Claude Code 的內建 plan mode，因為那不夠好。自己的 Markdown 文件給了他完全的控制權：可以在編輯器中編輯、加上行內筆記，而且它作為專案中的真實產物而持續存在。

I use my own .md plan files rather than Claude Code's built-in plan mode. The built-in plan mode sucks. My markdown file gives me full control. I can edit it in my editor, add inline notes, and it persists as a real artifact in the project.

一個他經常使用的技巧：對於範圍明確的功能，如果他在開源專案中看過好的實作，他會將該程式碼作為參考一起分享。Claude 在有具體參考實作可依循時，表現會顯著優於從零設計。

For well-contained features where I've seen a good implementation in an open source repo, I'll share that code as a reference alongside the plan request. Claude works dramatically better when it has a concrete reference implementation to work from rather than designing from scratch.

💻 Planning Prompt 範例
I want to build a new feature <name>
that extends the system to perform
<business outcome>. write a detailed
plan.md document outlining how to
implement this. include code snippets
🛠 實務應用：Reference Implementation 技巧

如果你想加入 sortable IDs，找一個做得好的開源專案，貼上它的 ID 生成碼，然後說：「this is how they do sortable IDs, write a plan.md explaining how we can adopt a similar approach.」這個技巧的核心邏輯是：給 AI 一個「已被證明可行的」錨點，比讓它從白紙開始設計好得多。

the list endpoint should support
cursor-based pagination instead of
offset. write a detailed plan.md for
how to achieve this. read source files
before suggesting changes, base the
plan on the actual codebase
Part III

Annotation Cycle — 工作流中最具創新性的部分

這是 Boris 工作流中最具特色的部分，也是他貢獻最大價值的地方。Claude 寫完計畫後，Boris 在編輯器中打開它，直接在文件中加入行內筆記。這些筆記修正假設、拒絕方法、添加約束條件，或提供 Claude 沒有的領域知識。

This is the most distinctive part of my workflow, and the part where I add the most value. After Claude writes the plan, I open it in my editor and add inline notes directly into the document. These notes correct assumptions, reject approaches, add constraints, or provide domain knowledge that Claude doesn't have.

筆記的長度差異極大。有時只有兩個字：在 Claude 標記為 optional 的參數旁邊寫「not optional」。有時則是一整段文字，解釋業務約束或貼上他期望的資料格式。

The notes vary wildly in length. Sometimes a note is two words: "not optional" next to a parameter Claude marked as optional. Other times it's a paragraph explaining a business constraint or pasting a code snippet showing the data shape I expect.

然後他把 Claude 送回文件：「I added a few notes to the document, address all the notes and update the document accordingly. don't implement yet。」這個循環重複 1 到 6 次。明確的「don't implement yet」防護至關重要。沒有它，Claude 一認為計畫夠好就會跳去寫程式碼。

Then I send Claude back to the document. This cycle repeats 1 to 6 times. The explicit "don't implement yet" guard is essential. Without it, Claude will jump to code the moment it thinks the plan is good enough.

💻 Annotation 筆記實例
📚 核心概念：Shared Mutable State

Markdown 文件充當了人類和 AI 之間的共享可變狀態。Boris 可以按自己的節奏思考、精確標註問題所在位置，然後重新參與而不失去上下文。這與試圖透過聊天訊息引導實作根本不同。計畫是一份結構化、完整的規格書，你可以整體性地審查。聊天對話是你需要滾動瀏覽才能重建決策的東西。

💡 比喻說明

想像你是建築師，AI 是繪圖員。你不會站在繪圖員身後口述每一條線（聊天模式）。你會讓他畫一版藍圖，然後你拿紅筆在藍圖上標註修改（Annotation Cycle），他根據標註改完，你再看一次，直到滿意才讓施工隊進場。

// 領域知識注入
"use drizzle:generate for migrations,
not raw SQL"

// 修正錯誤假設
"no — this should be a PATCH, not PUT"

// 拒絕提議的方法
"remove this section entirely,
we don't need caching here"

// 重新導向整個區段
"this is wrong, the visibility field
needs to be on the list itself, not
on individual items. restructure the
schema section accordingly"
Part IV

Implementation Phase — 讓執行變得無聊

當計畫準備好後，Boris 發出一個標準化的實作指令，他幾乎在每個實作階段都重複使用。到他說「implement it all」的時候，每個決策都已經做完並驗證過了。實作變成機械式的，而不是創造性的。這是刻意為之的 — 創造性工作發生在標註循環中。

By the time I say "implement it all," every decision has been made and validated. The implementation becomes mechanical, not creative. This is deliberate. The creative work happened in the annotation cycles.

Boris 同時解釋了他的標準 prompt 中每個片段的用意：「implement it all」意味著執行計畫中的所有內容，不要挑選；「mark it as completed in the plan document」意味著計畫是進度的唯一來源；「do not stop」意味著不要在流程中途暫停請求確認；「continuously run typecheck」意味著及早發現問題。

This single prompt encodes everything that matters: do everything in the plan, don't cherry-pick; the plan is the source of truth for progress; don't pause for confirmation mid-flow; catch problems early, not at the end.

💻 標準 Implementation Prompt
⚠️ 重點提示：Todo List 作為進度追蹤器

在進入實作之前，Boris 總是要求一個細粒度的任務分解：「add a detailed todo list to the plan, with all the phases and individual tasks necessary」。Claude 在執行時標記已完成項目，所以 Boris 可以隨時一眼看到進度，在長達數小時的工作階段中特別有價值。

implement it all. when you're done with
a task or phase, mark it as completed
in the plan document. do not stop until
all tasks and phases are completed.
do not add unnecessary comments or
jsdocs, do not use any or unknown types.
continuously run typecheck to make sure
you're not introducing new issues.
Part V

Feedback & Staying in the Driver's Seat

一旦 Claude 開始執行計畫，Boris 的角色從架構師轉變為監督者。他的 prompt 變得極為精簡。計畫階段的筆記可能是一整段文字，但實作階段的修正通常只有一句話。

Once Claude is executing the plan, my role shifts from architect to supervisor. My prompts become dramatically shorter. Where a planning note might be a paragraph, an implementation correction is often a single sentence.

前端工作是最迭代的部分。Boris 在瀏覽器中測試，然後快速射出修正：「wider」、「still cropped」、「there's a 2px gap」。對於視覺問題，他有時會附上截圖 — 一張錯位表格的截圖比描述問題更快傳達問題。

Frontend work is the most iterative part. I test in the browser and fire off rapid corrections. For visual issues, I sometimes attach screenshots. A screenshot of a misaligned table communicates the problem faster than describing it.

Boris 也經常引用現有程式碼：「this table should look exactly like the users table」。這比從零描述設計精確得多。成熟程式碼庫中的大多數功能都是現有模式的變體。當方向出錯時，他不嘗試修補，而是回滾然後縮小範圍。

I also reference existing code constantly. When something goes in a wrong direction, I don't try to patch it. I revert and re-scope by discarding the git changes.

🛠 實務應用：四種主動干預模式

Cherry-picking：「for the first one, just use Promise.all; for the third, extract into a separate function; ignore the fourth and fifth」— 逐項決策。

Trimming scope：「remove the download feature from the plan, I don't want to implement this now」— 防止範圍蔓延。

Protecting interfaces：「the signatures of these three functions should not change, the caller should adapt」— 硬性約束。

Overriding choices：「use this library's built-in method instead of writing a custom one」— 快速技術覆寫。

📚 核心概念：Revert > Patch

Boris 的哲學是：當 AI 走錯方向時，回滾 + 縮小範圍幾乎總是比嘗試漸進修復一個糟糕的方法更好。「I reverted everything. Now all I want is to make the list view more minimal — nothing else.」這與「沉沒成本謬誤」相反 — 果斷放棄錯誤路線比堅持修補更有效率。

Part VI

Single Long Sessions — 挑戰傳統認知

Boris 在單一長工作階段中完成 research、planning 和 implementation，而不是將它們拆分到不同的工作階段。他表示沒有看到大家所說的「context window 超過 50% 後效能下降」的問題。到他說「implement it all」的時候，Claude 已經花了整個工作階段在建立理解：在 research 期間閱讀檔案、在標註循環中精煉心智模型、吸收他的領域知識修正。

I run research, planning, and implementation in a single long session. I am not seeing the performance degradation everyone talks about after 50% context window. By the time I say "implement it all," Claude has spent the entire session building understanding.

當 context window 填滿時，Claude 的自動壓縮機制維持了足夠的上下文繼續運作。而計畫文件 — 那個持久化的產物 — 在壓縮後依然完整保存。Boris 可以在任何時間點指向它。這解釋了為什麼持久化的 .md 文件如此關鍵：它不受聊天記憶衰減的影響。

When the context window fills up, Claude's auto-compaction maintains enough context to keep going. And the plan document, the persistent artifact, survives compaction in full fidelity.

Practice Questions

Q1 概念理解 Boris 為什麼堅持使用自己的 Markdown 文件而非 Claude Code 的內建 plan mode？這反映了什麼更深層的原則？

表面原因是內建 plan mode 功能不足 — 無法在編輯器中自由編輯、加入行內筆記。但更深層的原則是「人類必須擁有對產物的完全控制權」。Markdown 文件是 Boris 可以在自己的時間和空間中思考的媒介，不受聊天界面的時間壓力。它作為專案中的真實產物而持續存在，不會隨著對話消失。

這也反映了一個重要的 AI 協作原則：最有價值的產物應該存在於你的工作環境中，而不是被鎖在 AI 工具的介面裡。這確保即使你換工具或工具失效，你的思考成果仍然保留。

Q2 產業應用 如果你要在 CDMO 的偏差調查 (Deviation Investigation) 流程中應用 Boris 的 Annotation Cycle 方法，你會如何設計這個流程？

在 GMP 環境中，Boris 的 Annotation Cycle 天然適合偏差調查：

Research Phase：讓 AI 深度閱讀相關 SOP、批次紀錄、設備維護歷史，產出 research.md 式的「背景分析報告」。人類審查 AI 是否正確理解了製程和規範。

Plan Phase：AI 提出調查計畫（根本原因假設、需要檢查的數據、建議的 CAPA）。人類在 plan.md 中標註：「這個假設不對，我們的 SA25 isolator 的壓差監控邏輯和你描述的不同」或「加上環境監測數據的交叉比對」。

Critical Guard：「don't implement yet」在 GMP 中變成「don't finalize yet」— 在 QA 審核批准之前，任何結論都是草稿。這正是 copilot mode 的精髓：100% 人類審核。

核心價值：AI 加速資料收集和假設生成，但人類的 O-ring 判斷（製程經驗、法規知識、客戶關係考量）透過標註循環注入。

Q3 批判思考 Boris 聲稱在 context window 超過 50% 後沒有看到效能下降。這個觀察有什麼可能的限制或替代解釋？

幾個值得質疑的點：

倖存者偏差：Boris 的工作流本身就把最重要的資訊持久化到 .md 文件中，所以即使 Claude 的內部記憶衰減了，計畫文件作為「外部記憶體」彌補了差距。他可能沒有察覺到衰減，因為他的方法恰好抵消了衰減。

任務結構差異：到 implementation 階段，大多數決策已經做完了。Claude 只需要讀計畫然後執行機械式任務，這對 context window 的依賴度本就較低。與需要在長對話中持續做複雜推理的場景不同。

Auto-compaction 的選擇性：Claude Code 的壓縮機制可能優先保留了 .md 文件的內容（因為它被頻繁引用），而丟棄了早期的聊天歷史。這不代表沒有衰減，只是衰減發生在不重要的部分。

啟示：Boris 的經驗可能證明的不是「context window 沒有衰減」，而是「如果你把關鍵資訊外部化到持久文件中，衰減就不再重要」— 這其實是更有價值的洞察。

🎯 我的行動結論
✔ 核心洞察
Boris 的 Annotation Cycle 本質上就是我一直在實踐的 copilot mode 的最佳範例 — 但他把它結構化成一個可重複的流程。「Markdown 文件作為共享可變狀態」這個概念，與我在 CDMO 偏差調查中讓 AI 起草、人類標註修正、AI 再修改的模式完全吻合。差別在於他把這個循環的媒介從「聊天對話」升級成了「持久化文件」，這大幅降低了上下文損失的風險。
📈 立即應用
在下一次使用 Claude Projects 處理複雜分析任務時，嘗試 Boris 的三步法：(1) 要求 Claude 先寫 research.md 理解背景，我審核後再進行；(2) 要求寫 plan.md 列出分析框架，我在文件中直接標註修改意見而非在聊天中來回討論；(3) 確認計畫無誤後才說「execute」。特別是在為 COO 層級的戰略報告做準備時，這個流程可以確保 AI 的分析方向從一開始就對齊我的意圖。
⚠ 需要補充
Boris 的工作流高度針對軟體開發場景，需要調適到知識工作和文件分析的情境。他的「reference implementation」技巧在編程中很自然，但在報告撰寫或策略分析中，對應的做法是什麼？可能是「給 AI 看一份過去的好報告作為格式和深度的參考」。另外，他的整個流程假設使用者有能力判斷計畫的正確性 — 這在我的專業領域內沒問題，但在學習全新領域時，Annotation Cycle 的價值會降低，因為你可能無法辨識 AI 的錯誤。
