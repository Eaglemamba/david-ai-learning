1
數據的三層結構 The Stages of Data

原文精華 Original Text

想像這個場景：兩位高管用兩個不同的儀表板看「同一份」數據。通常他們會安排三天後與資料團隊開會，結果還是搞不清楚為什麼兩個讀取 Salesforce 或 HubSpot 的儀表板，對同一件事給出不同數字。

"Think of this scenario: Two executives using 2 different dashboards to see the 'same' data... They have explained to you, but it is still not clear why 2 Dashboards reading from Salesforce or Hubspot give different numbers for the same thing."

數據從源頭到儀表板會經過三個階段：Source（來源）是數據產生和存放的地方；Transformation（轉換）是有人決定指標定義並套用商業邏輯的地方；Dashboard（儀表板）是你看到數字的地方。

"Source → Where data is generated and lives. Transformation → Where someone decides what metrics mean and applies business logic to them. Dashboard → Where you see the number."

沿途可能發生很多事造成混淆：時間差異（HubSpot 即時更新，資料庫凌晨 2 點才刷新）、定義漂移（A 團隊的「活躍用戶」≠ B 團隊的「活躍用戶」）、隱藏篩選（儀表板 A 排除超過 $10K 的退款，儀表板 B 包含）。

"Timing Mismatches - e.g. HubSpot updates are on the spot. Database refreshes at 2 AM. Definition Drift - e.g. One team's 'active user' ≠ another team's 'active user.' Hidden Filters - e.g. Dashboard A excludes refunds over $10K. Dashboard B includes them."

💡 比喻說明

想像你在追蹤一批原料從供應商到成品的過程。供應商說「出貨 100kg」，倉庫說「收到 98kg」，生產報表說「投入 95kg」。差異可能來自：運輸損耗（時間差）、不同的計量單位或四捨五入規則（定義差異）、或某些品項被自動排除在報表外（隱藏篩選）。

在 CDMO 環境中，這就像 MES、ERP、QMS 三個系統對「批次完成」的定義可能不同。

⚠️ 重點提示

數據差異三大根源：

1. Timing（時間）：不同系統的更新頻率不同

2. Definition（定義）：同一個詞在不同團隊有不同意思

3. Filter（篩選）：報表可能有你不知道的預設條件

這三個問題在任何組織都會發生，不是資料團隊「做錯了」。

2
Claude Code 到底能做什麼 What Claude Code Actually Does

原文精華 Original Text

Claude Code 的獨特之處：直接連接商業工具（HubSpot、Google Sheets、資料庫）、讀取實際數據而非假設、展示工作過程。

"What makes it different: Connects directly to business tools (HubSpot, Google Sheets, Databases). Reads actual data, not hypotheticals. Shows its work."

當你問「為什麼這些 pipeline 數字不一致？」它會連接 HubSpot、讀取你的資料庫表格、比較邏輯，然後回答：「HubSpot 包含 On Hold 的交易，你的資料倉儲在 90 天後會排除它們。」

"When you ask: 'Why don't these pipeline numbers match?' it will connect to HubSpot, read your database tables, compare the logic and you will do something like 'HubSpot includes On Hold deals. Your warehouse excludes them after 90 days.'"

它成為一個你可以無限客製化的工作流程機器，不會像一般 Claude 那樣受限於共用功能、需要你引導更多它該做什麼，也缺乏 Claude Code 能達到的上下文能力。

"It becomes a workflow machine that you can customize as much as you can, without getting limited by plain Claude, where you have many shared features but requires you to guide much more of what it should be doing, and its also lacking the context power Claude Code can achieve."

📚 核心概念

Claude Code vs 一般 Claude 的關鍵差異：

一般 Claude（claude.ai）是在瀏覽器中與 AI 對話，你需要複製貼上數據給它。Claude Code 是在你的電腦終端機（Terminal）執行，可以直接讀取本地檔案、連接外部服務、執行腳本。

簡單說：一般 Claude 是「你告訴它什麼，它就知道什麼」；Claude Code 是「它可以自己去拿需要的資訊」。

⚖️ 工具比較

Claude Code vs Cowork（剛發布的預覽版）：

作者提到 Anthropic 剛推出 Cowork，號稱是「給非技術人員的 Claude Code」。目前還在 preview 階段，功能可能持續更新。這意味著本文的 Claude Code 學習可能之後能轉移到 Cowork，降低技術門檻。

我的建議：先了解 Claude Code 的概念和能力，等 Cowork 更成熟時會更容易上手。

3
四大核心功能 Claude Code Features

原文精華 Original Text

所有這些功能都存在於 Markdown (.md) 檔案中，這是所有 AI 系統用來呈現文件的格式。

"All these features live under Markdown (.md) files, which is what all AI systems use to present documentation."

Slash Commands（斜線指令）：使用者觸發的捷徑，讓你不用重複寫工作流程。例如：/audit_dashboard 觸發儀表板比較工作流程。

"Slash Commands: User-triggered shortcuts that save you from writing repeatable workflows. Example: /audit_dashboard - triggers dashboard comparison workflow."

Sub Agents（子代理）：平行執行的 Claude Agent，有專門的焦點和獨立的上下文。例如：@data-reconciliation 平行比較 HubSpot vs Warehouse。

"Sub Agents: Parallel Claude Agents with specialized focus and isolated context. Example: @data-reconciliation - compares HubSpot vs Warehouse in parallel"

Skills（技能）：Claude 在相關時自動載入的知識包。Claude Code 不會告訴你它在用某個 skill，但如果它認為需要就會遵循。

"Skills: Auto-invoked knowledge packages Claude loads when relevant. Claude Code won't tell you that it is using the skill per se, but it will follow it if it thinks it needs to."

MCP（Model Context Protocol）：我們提供給 Claude Code 連接上下文的工具。每個 MCP 不同，連接方式需要根據使用案例和認證設定。通常你會被導向一個畫面，點擊確認身份即可，不需要技術知識。

"MCP: Tools we provide Claude Code to connect with context. Each MCP is different and the way to connect them is required in the use case and authentication setup so it can recognize you."

🛠️ 實務應用

四個功能的記憶口訣：「指-代-技-連」

• 指令（Slash Commands）：你主動觸發的快捷鍵

• 代理（Sub Agents）：專門處理特定任務的分身

• 技能（Skills）：背景自動套用的知識包

• 連接（MCP）：與外部工具的橋樑

💻 程式碼/API

MCP 的概念：

MCP（Model Context Protocol）是 Anthropic 設計的標準化協議，讓 AI 可以連接各種外部服務。你可以把它想像成「AI 專用的 USB 插槽」—— 只要外部服務支援 MCP，Claude 就能直接讀寫數據。

目前已有 HubSpot、Google Drive、Confluence、Slack 等常用工具的 MCP。Plugin 市集 aitmpl 有更多預建的 skills 和 agents 可以安裝。

⚠️ 重點提示

GDPR/資安提醒：

如果你處理的是極機密資訊，請先聯繫資安團隊確認 Anthropic 是否已被核准為第三方工具，或找方法透過 API 在公司內部伺服器執行 Claude Code。

在 CDMO 環境中，這點特別重要——21 CFR Part 11 和 ALCOA+ 對資料完整性有嚴格要求。

實戰案例 Hands-On Examples

原文精華 Original Text

案例 1：Pipeline 差異
情況：HubSpot 顯示 $1M，CFO 用資料庫的儀表板顯示 $810K。
你的做法：「比較 HubSpot pipeline 和我們的 warehouse deals 表格。為什麼有差異？」
洞察：現在你可以說「讓我們對齊 On Hold 是否算作活躍 pipeline。」

"Example #1: Pipeline Discrepancies. Your Situation: HubSpot shows $1M. CFO's dashboard using the database shows $810K. What You Do: 'Compare the HubSpot pipeline with our warehouse deals table. Why is there a difference?'"

案例 2：Google Sheet vs 資料倉儲
情況：行銷總監的 Google Sheet 與資料團隊的儀表板不符。
你的做法：「讀取 Google Sheet 'Q1 Campaign Performance'，與 warehouse 表格 marketing.campaigns 比較。為什麼 CPL 數字不同？」
洞察：你的 sheet 更即時，warehouse 更乾淨。它們服務不同目的。

"Example #2: Google Sheet vs Data Warehouse. Your Situation: Marketing Director's Google Sheet doesn't match the data team's dashboard. Insight: Your sheet is fresher. The warehouse is cleaner. They serve different purposes."

案例 3：需求文件
情況：你要求「客戶流失儀表板」，資料團隊做出來的是錯的。重做還是錯。
你的做法：「我需要一個客戶流失儀表板。幫我根據我們的 warehouse 表格寫需求，讓資料團隊一次就做對。」
洞察：你用 AI 拆解你想要的東西，減少很多會議。

"Example #3: Requirements Without the Back-and-Forth. You asked for a 'customer churn dashboard.' The data team built something. Wrong. They rebuilt. Still wrong. Insight: You used AI to break down what you wanted and reduced lots of meetings."

案例 4：會議紀錄轉行動
情況：你在會議中討論了流失儀表板，在 Confluence 記了筆記。需要建立正式需求和給資料團隊的 ticket。
你的做法：「讀取我的 Confluence 頁面 'Q1 Data Requests - Churn Discussion'」然後觸發 /requirements_from_notes
洞察：把鬆散的會議筆記轉成可行動的工作，不用手動重寫或在三個工具間切換。

"Example #4: From Meeting Notes to Action. Transforms loose meeting notes into actionable work without manually rewriting everything or switching between three different tools."

🛠️ CDMO 應用場景

這四個案例對應到 CDMO 的場景：

1. Pipeline 差異 → MES vs ERP 的批次產量差異追蹤

2. Sheet vs Warehouse → Excel 追蹤表 vs Veeva QMS 的偏差數量

3. 需求文件 → 撰寫新製程的 User Requirements Specification

4. 會議轉行動 → 將客戶 KOM 會議紀錄轉成 Project Charter

💡 比喻說明

從「等待解答」到「自己找答案」：

以前：你發現數據有問題 → 寄信給 IT → IT 排隊處理 → 3 天後開會 → 解釋你聽不懂 → 再約一次會議

現在：你發現數據有問題 → 用自然語言問 Claude Code → 5 分鐘得到「HubSpot 包含 On Hold，Warehouse 排除」的答案 → 直接跟團隊說「讓我們對齊定義」

這是從「被動等待」到「主動追蹤」的轉變。

5
重點整理 Key Takeaways

名稱有誤導性

Claude Code 中的「Code」讓人以為要會寫程式，但實際上你是在「指揮」(directing) AI，而非「編碼」(coding)。這是認知上最大的障礙。

閘門正在打開

作者用一個週末建立整個網站，沒有程式背景。非技術人員正在用 AI 建立真正的工具——網站、數據系統、客製化工作流程。

四大功能記住「指代技連」

Slash Commands（指令）、Sub Agents（代理）、Skills（技能）、MCP（連接）。你可以混搭使用，讓 Claude Code 自己判斷該用什麼。

5 分鐘 vs 5 天

數據差異的追蹤從「等 5 天開會還是聽不懂」變成「5 分鐘得到具體原因，直接討論該對齊什麼」。這是效率的質變。

練習題 Practice Questions
Q1 概念理解 為什麼兩個儀表板顯示的「同一個指標」會有不同數字？請用文章提到的三層架構解釋。

答案：數據從 Source（來源）經過 Transformation（轉換）到 Dashboard（儀表板）的過程中，可能因為三個原因產生差異：

  • Timing（時間差）：不同系統的更新頻率不同，例如 HubSpot 即時更新，但資料倉儲可能凌晨 2 點才刷新
  • Definition（定義漂移）：同一個詞在不同團隊有不同定義，例如 A 團隊的「活躍用戶」可能指 30 天內登入，B 團隊可能指有交易紀錄
  • Filter（隱藏篩選）：儀表板可能有預設條件你不知道，例如自動排除超過某金額的退款
Q2 產業應用 在 CDMO 環境中，如果你發現 MES 顯示的批次產量和 ERP 的數字不一致，你會如何用 Claude Code 的概念來追蹤原因？

答案框架：

1. 辨識三層架構：MES 是 Source（生產現場數據），ERP 是經過 Transformation 的數據，報表是 Dashboard

2. 可能的差異原因：

  • Timing：MES 即時記錄，ERP 可能有批次傳輸延遲
  • Definition：MES 可能記錄「理論產量」，ERP 可能記錄「入庫產量」（扣除 QC 取樣）
  • Filter：ERP 可能自動排除「Hold」狀態的批次

3. Claude Code 可以做的：如果有 MCP 連接 MES 和 ERP，可以直接問「比較 Batch 12345 在 MES 和 ERP 的產量數據，找出差異點」

注意：在 GMP 環境中，需要先確認資安合規才能讓 AI 接觸生產數據。

Q3 批判思考 文章聲稱「非技術人員可以用 Claude Code 建立網站和數據系統」，你認為這個說法有什麼限制或風險？

答案框架：

合理的限制：

  • 初始設定仍需要一些技術操作（開 Terminal、安裝、設定 MCP 認證）
  • 複雜的商業邏輯可能需要技術人員協助驗證
  • 生產環境部署通常還是需要專業支援

潛在風險：

  • 資安考量：AI 直接連接敏感數據，需要適當的存取控制
  • 錯誤放大：如果 AI 的邏輯有誤，非技術人員可能無法發現
  • 過度依賴：可能讓人忽略學習基礎概念的必要性

平衡觀點：這篇文章的價值在於「降低入門門檻」，而非「取代專業」。最佳應用場景是「快速驗證想法」和「減少來回溝通」，而非「建立生產級系統」。

🎯 我的行動結論
✔ 核心洞察
Claude Code 的價值不在「寫程式」，而在「縮短從問題到答案的路徑」。在 CDMO 環境中，這意味著我可以更快追蹤 MES/ERP/QMS 之間的數據差異原因，而不是每次都要等 IT 排隊處理。這符合我「AI 增強人類判斷而非取代」的理念——AI 負責追蹤數據路徑，我負責決定該對齊什麼定義。
📈 立即應用
這週：安裝 Claude Code 並完成基本設定（Terminal + 訂閱）。找一個實際的數據差異問題（例如上週會議提到的產量報表不一致）測試看看能追蹤到什麼。先用低敏感度的數據測試，確認資安可行性後再考慮擴大應用。
⚠ 需要補充
文章沒有深入討論：(1) 在高度監管環境（如製藥業）如何處理 21 CFR Part 11 合規；(2) MCP 的實際設定細節和常見錯誤；(3) 與其他工具（如 n8n、Make.com）的比較。建議 3 個月後當 Cowork 更成熟時重新評估，可能有更適合非技術人員的入口點。
