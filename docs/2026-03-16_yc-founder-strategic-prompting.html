Executive Summary

一位 YC 創辦人展示了四層戰略提問法，將 Claude 從「更快的 Google」變成「沒有自尊心的戰略思考夥伴」。核心方法：先餵入大量一手資料（競爭對手頁面、財報電話會議、客戶評論），再用層層遞進的提問挖掘市場的隱性假設、脆弱共識和攻擊面。

批判提醒 這是匿名社群貼文，無法驗證具體案例。「3小時 vs 3個月」的說法屬於行銷誇大——真正的市場研究需要客戶訪談、數據驗證和迭代測試，LLM 無法替代。但提問框架本身有實用價值。

Part I: Context Loading — 上下文裝載的藝術
Part I
先餵資料，不要先問問題
這位創辦人的第一步不是叫 Claude 去「研究市場」。他餵入了 8 個競爭對手的 landing page、3 份財報電話會議逐字稿、12 則客戶評論，和一個 Reddit 抱怨串。
First: he didn't ask Claude to "research the market." He fed it 8 competitor landing pages, 3 earnings call transcripts, 12 customer reviews, and a Reddit thread of complaints.
這是 Context Engineering（上下文工程）的核心原則：你給 LLM 的資料品質，決定了它思考的品質。大多數人給 Claude 一個模糊問題，然後對模糊答案感到失望。這位創辦人反過來——他先把自己會看的資料全部灌進去，讓 Claude 站在和他一樣的資訊起點。
Most people treat Claude like a faster Google. These founders are using it like a thinking partner who has read everything and has no ego about being wrong.
Context Loading 原則
一手資料優先：不是叫 AI 去搜尋（AI 會給你平均觀點），而是餵入你自己策展的原始資料。競爭對手怎麼向客戶說話（landing page）、管理層對投資人說什麼（earnings call）、客戶真正的痛點是什麼（reviews + Reddit）——這三者的交叉比對，才能產出有價值的洞察。
比喻：偵探的案件卷宗
想像你是一個偵探。你不會跟助手說「去查一下這個案子」——你會把所有證物、證詞、監視器畫面攤在桌上，然後問助手：「你看到什麼我沒看到的？」Context Loading 就是把案件卷宗攤開的動作。
CDMO 場景遷移
評估新客戶的可行性時，同樣可以把一手資料先載入：客戶的 CMC 文件摘要、FDA/EMA 審查歷史、同類產品的市場規模資料、類似案例的 deviation 紀錄。讓 AI 在和你相同的資訊基礎上分析，而不是依賴它的通用知識。
批判觀點：Context Window 的現實限制
8 個 landing page + 3 份 earnings call + 12 則 reviews + Reddit 串 = 大量文字。即使 Claude 有 200K context window，真正的 attention quality 在超長上下文中會下降。實務上建議：每次聚焦 2-3 份關鍵文件，分批處理，而非一次灌入所有資料。
Part II: Unspoken Insight — 挖掘隱性洞察
Part II
問「沒人說出口的事」
載入資料後，他問了一個問題：「這個市場中每個成功玩家都理解、但他們的客戶從未說出口的是什麼？」
"What does every successful player in this market understand that their customers never say out loud?"
不是「總結這些資料」。不是「分析競爭對手」。而是隱性洞察（Unspoken Insight）——那種需要創辦人花兩年客戶訪談才能搞清楚的東西。
Not "summarize these." Not "analyze the competition." The unspoken insight. The thing that takes founders 2 years of customer calls to figure out.
為什麼這個問題有效
這個 prompt 的精妙之處在於它要求 交叉推論（cross-reference inference）：模型需要同時分析「公司怎麼賣」和「客戶怎麼買」之間的落差，然後推論出雙方都知道但沒人明說的共識。這不是摘要，是 合成（synthesis）。
比喻：飯桌上沒人提的事
每個家庭都有「大家都知道但飯桌上不會提的事」。市場也一樣——所有成功公司都理解客戶真正在買什麼（通常不是他們宣稱的功能），但這個洞察不會出現在任何公開文件裡。你要問的是那個「飯桌下的共識」。
CDMO 版本的隱性洞察提問
載入客戶的 RFP、你的報價歷史、以及贏/輸案分析後，可以問：「在我們贏得的案子和失去的案子之間，客戶的決策標準中有什麼是他們在 RFP 中從未明確寫出、但實際上決定了結果的因素？」
批判觀點：LLM 的推論 vs 真正的客戶洞察
LLM 的「隱性洞察」本質上是統計模式匹配，不是真正的市場理解。它能從文本中識別反覆出現的模式和未被明說的共同主題，但它無法替代：面對面觀察客戶行為、感受情緒張力、理解組織政治。把它當作假說生成器（hypothesis generator），不是真理機器。
Part III: Assumption Attack — 解構市場假設
Part III
找出整個產業建立在哪些假設上
接著他追問：「現在告訴我這個市場建立在哪 3 個假設上，以及要讓每個假設被推翻，什麼條件必須成立？」
"Now show me the 3 assumptions this entire market is built on, and what would have to be true for each one to be wrong."
15 分鐘內他就得到了整個產業的攻擊面（attack surface）——盲點、脆弱的共識、沒人在討論的切入點。
In 15 minutes he had the attack surface of an entire industry. The blind spots. The fragile consensus. The opening nobody was talking about.
Inversion Thinking（反向思考）
這是 Charlie Munger 最推崇的思維工具之一：不問「如何成功」，而問「什麼會導致失敗」。用在市場分析上，就是不問「這個市場為什麼好」，而問「這個市場建立在哪些可能被推翻的假設上」。LLM 特別適合做這種結構化的反向推論——它不會因為情感依附而忽略不舒服的可能性。
Prompt Template"Based on the documents I provided, identify the 3 foundational assumptions this market is built on. For each assumption, describe: (1) what it is, (2) what evidence supports it, (3) what would have to be true for it to be wrong, and (4) early signals that it might already be weakening."
CDMO 場景：技術平台假設解構
在評估是否投資新技術平台（例如從 vial 轉向 pre-filled syringe）時，可以載入市場報告、客戶需求趨勢、法規變化，然後問：「我們投資 PFS 線的決策建立在哪 3 個假設上？要讓每個假設被推翻，什麼必須成立？」這能提前暴露：客戶需求是否真的在轉移、法規環境是否支持、我們的成本結構是否能競爭。
批判觀點：「15 分鐘得到整個產業的攻擊面」
這裡需要打上2-3x 折扣。LLM 能快速生成「看起來像深度分析」的結構化輸出，但它的假設解構受限於：(1) 你餵入的資料範圍、(2) 訓練數據中的既有觀點、(3) 無法接觸非公開資訊。真正的產業攻擊面往往藏在非公開的供應鏈動態、監管內部討論、和客戶私下的策略轉向中。LLM 產出是起點，不是結論。
提問品質比較
低效問法：「分析這個市場的風險」→ 得到通用的 SWOT 分析

高效問法：「這個市場建立在哪些假設上？什麼能推翻它們？」→ 得到結構化的脆弱點分析

差異：前者要求描述，後者要求反向推論。LLM 在結構化推論上的表現遠優於開放式描述。
Part IV: Stress Test — 投資人級壓力測試
Part IV
用最強反論測試每個弱點
然後他做了一件我從沒見過的事。他問：「寫出一個世界級投資人會問的 5 個問題來摧毀這個商業構想，然後只用這些文件中的證據來回答每個問題。」
"Write 5 questions a world-class investor would ask to destroy this business idea, then answer each one using only the evidence in these documents."
他接下來 2 小時都在壓力測試每個假設。每個薄弱的回答都觸發追問：「這個論點的最強版本是什麼？它在哪裡仍然會崩潰？」
"What's the strongest version of this argument and where does it still break?"
到第 3 小時，他有了一份策略簡報，感覺像是出自一個在該領域深耕十年的人之手。
By hour 3, he had a strategy deck that felt like it came from someone who'd spent a decade in the space.
Steelman 技術（最強反論法）
「最強版本是什麼？它在哪裡仍然崩潰？」——這叫 Steelmanning，是 Strawmanning（稻草人攻擊）的反面。先把對手的論點建到最強，然後再找它的破綻。這是最高品質的批判思維，也是 LLM 特別擅長的：它能無情緒地把每個論點推到極限。
Follow-up Prompt ChainStep 1: "Write 5 questions a world-class investor would ask to destroy this idea. Answer each using only evidence from these documents." Step 2 (for each weak answer): "What's the strongest version of this argument and where does it still break?" Step 3: "Given all the above, what's the one thing that would make you bet against this business?"
比喻：模擬法庭
這就像律師在出庭前做的「模擬法庭」（moot court）——讓同事扮演對方律師，用最尖銳的問題攻擊你的論點。不是為了打擊信心，而是為了在真正的戰場之前先修補所有弱點。Claude 就是那個可以隨時召喚的對方律師。
CDMO 場景：客戶提案壓力測試
在向客戶提交可行性評估報告前，載入報告內容，然後問：「如果你是這個客戶的 CMC VP，你會問哪 5 個問題來質疑我們的能力？只用報告中的資訊回答。」每個薄弱回答就是需要補強的地方——可能是製程經驗不足、批次規模的 scale-up 風險、或法規路徑的不確定性。
批判觀點：「感覺像深耕十年」的危險
這是最需要警惕的地方。LLM 產出的策略簡報會有完美的結構、流暢的論述、看起來深思熟慮的結論——但它缺乏：(1) 被市場打臉的經驗、(2) 對特定客戶關係的理解、(3) 直覺判斷（pattern recognition from lived experience）。一份「感覺像十年經驗」的報告，在真正有十年經驗的人面前，可能一問就露餡。O-Ring 原則提醒我們：AI 加速了分析，但不能替代判斷。
Key Takeaways
1. Context Loading > Question Quality
提問的品質取決於上下文的品質。先餵入策展過的一手資料（landing page、earnings call、reviews），再問高層次問題。不要讓 LLM 自己去搜尋——你策展的資料就是你的競爭優勢。
2. 四層遞進提問法
隱性洞察 → 假設解構 → 投資人壓力測試 → 最強反論。每一層都建立在前一層的輸出上，形成一個完整的戰略分析迴圈。層層遞進才能穿透表面。
3. Steelman, Don't Strawman
追問「最強版本是什麼？它在哪裡仍然崩潰？」這個 pattern 是最高投資回報的提問技術。它迫使 LLM 超越第一層分析，找到更深的脆弱點。
4. 假說生成器，不是真理機器
LLM 產出的「戰略分析」是假說，不是結論。它加速了從 0 到 0.7 的過程，但從 0.7 到 1.0 仍然需要：客戶訪談、市場驗證、專家判斷。工具沒有改變，改變的是提問。
Practice Questions
Q1 概念理解 為什麼這位創辦人選擇餵入「客戶評論 + Reddit 抱怨」而非請 Claude 自己搜尋市場報告？這反映了 Context Engineering 的哪個核心原則？

因為市場報告是二手整理過的平均觀點，而客戶評論和 Reddit 抱怨是未經過濾的一手聲音。Context Engineering 的核心原則是：上下文的精確性和相關性比數量更重要。你策展的資料決定了 LLM 的思考品質——餵入平均資料，得到平均答案；餵入尖銳的一手資料，才能產出尖銳的洞察。

另外，讓 LLM 自己搜尋會引入它的通用知識偏差（generic knowledge bias），而你手動選擇的資料代表你的判斷和視角——這才是人的不可替代價值。

Q2 產業應用 將四層提問法套用到以下 CDMO 情境：你正在評估是否接受一家 Biotech 新客戶的 lyophilized product 專案。請設計每一層的具體提問。

Layer 1 Context Loading：載入客戶的 CMC section 摘要、product profile、同類產品的 FDA 審查歷史、你的設備能力表、過去類似 lyo 專案的 deviation 紀錄。

Layer 2 Unspoken Insight：「根據這些資料，這個客戶在選擇 CDMO 時真正在意但不會直接說出來的是什麼？」（可能是：他們更在意時程而非價格，或者他們的 CMC 團隊經驗不足需要更多 hand-holding。）

Layer 3 Assumption Attack：「我們接受這個案子的決策建立在哪 3 個假設上？什麼會讓每個假設失效？」（假設：lyo cycle 可以順利 scale-up、客戶的 API 品質穩定、法規路徑不會有意外。）

Layer 4 Stress Test：「如果你是我們的 Quality VP，你會問哪 5 個問題來反對接這個案子？用這些文件回答。」

Q3 批判思考 這篇貼文聲稱「3 小時完成 3 個月的研究」。用 O-Ring 理論分析：在這個流程中，哪些環節是 AI 真正能加速的？哪些環節是 AI 無法替代的「不可替代的人類判斷節點」？

AI 能加速的環節：

- 大量文本的交叉比對和模式識別（從 8 個 landing page 中找共同主題）

- 結構化假設生成（列出 3 個市場假設及反駁條件）

- 多角度壓力測試（快速生成投資人可能問的尖銳問題）

- 最強反論的建構（Steelmanning）

AI 無法替代的 O-Ring 節點：

- 資料策展判斷：決定餵入哪些資料本身就是最關鍵的策略判斷——選錯資料，整個分析方向就錯了

- 洞察驗證：LLM 的「隱性洞察」是統計模式匹配，需要和真實客戶對話才能驗證

- 政治/關係判斷：市場動態中有大量非公開的人際關係和權力動態，LLM 完全看不到

- 直覺校準：有經驗的創辦人會對某些分析結果「感覺不對」，這種直覺來自過去被市場打臉的經驗

結論：AI 把「3 個月」壓縮成「3 小時」的部分是資訊處理和假說生成。但真正花 3 個月的核心活動——客戶訪談、市場驗證、策略迭代——AI 一天都省不了。

我的行動結論
核心洞察
提問的品質決定了 AI 輸出的品質，而提問的品質取決於你先給它什麼上下文。這篇貼文最有價值的不是「四層提問法」本身，而是「先策展一手資料，再問高層次問題」的工作流程。這和我一直在實踐的 Context Engineering 原則完全一致——精確的上下文比更多的上下文更重要。
立即應用
本週在下一次客戶可行性評估中，先把客戶 RFP、設備能力表、同類案例 deviation 紀錄載入同一個 Project，然後依序問：(1) 隱性決策因素、(2) 我們接案假設的脆弱點、(3) Quality VP 的 5 個反對問題。測試這個流程是否比我現有的評估方法產出更好的風險預判。
需要補充
這篇貼文是匿名社群內容，缺乏可驗證的案例。「3小時 = 3個月」的說法需要打 2-3x 折扣。此外，文章完全沒討論 Context Window 的實際限制（一次灌入這麼多文件的 attention quality 下降問題）。需要自己測試不同資料量的效果差異。6 個月後重讀時，重點看我是否真的在 CDMO 場景中實踐了這個框架，以及實際效果如何。
