1
配置文件全貌：四層體系
Claude Code 的配置文件有四層結構，各自的作用範圍不同。理解這個層次關係，才能把正確的內容放在正確的位置。
CC has four configuration layers, each with a different scope: global, project, local, and module rules.
全域層 ~/.claude/CLAUDE.md 跨所有專案生效，適合放你的通用偏好。專案層放在專案根目錄，是最核心的配置。本地層僅在你的電腦上生效，不會提交到 Git。模組層是 .claude/rules/ 裡的獨立檔案，用來拆分關注點。
Global layer (~/.claude/CLAUDE.md) applies across all projects. Project layer sits at root. Local layer is machine-specific (not committed). Module rules split concerns into separate files.
📚 核心概念：優先順序

四層是疊加式的，不是覆蓋式。CC 啟動時會把四層內容合併注入 context window。如果有衝突，越具體的層級優先（Module > Local > Project > Global）。

💡 比喻說明

想像你在公司工作：全域是公司員工手冊（對所有人適用）、專案是部門 SOP（只對這個團隊）、本地是你桌上的便利貼（只有你看得到）、模組是每個職能各自的 checklist。

🛠️ 實務應用：CDMO 映射

CDMO 情境中，Global 相當於 GMP 通則（全廠適用），Project 相當於產品特定的批次紀錄（Batch Record），Local 是你個人的執行筆記，Module 則是各工站的 SOP 集合。這種分層治理邏輯完全一致。

2
CLAUDE.md：專案的入職手冊
CLAUDE.md 是最核心的配置文件，放在專案根目錄。CC 每次啟動都會讀它。它要回答一個簡單的問題：這個專案是什麼，你在這裡應該怎麼幹活。
CLAUDE.md is the core config file at project root. CC reads it on every startup. It answers: what is this project, and how should you work here.
建議內容包括：專案概述與整體結構、子目錄各自的職責、工作流約定（什麼要先確認、什麼可以直接做）、程式碼/內容規範、Git 規則和隱私紅線、Skills 管理方式。
Recommended contents: project overview and structure, subdirectory responsibilities, workflow conventions, code/content standards, Git rules and privacy boundaries, Skills management.
一個關鍵限制：控制在 200 行以內。超過之後 CC 對指令的遵守率會明顯下降。內容太多就用 .claude/rules/ 來拆分。
Key limit: keep it under 200 lines. Beyond that, CC's instruction adherence drops noticeably. Overflow goes into .claude/rules/.
⚠️ 重點提示：/init 快速起步

如果不知道該寫什麼，CC 有快捷命令 /init。它會自動掃描專案檔案，生成基礎版 CLAUDE.md。不完美，但至少有個起點可以修改。

🛠️ 實務應用：200 行限制的策略

200 行的限制本質上是 context window 的注意力分配問題。就像你不會把整本 SOP 塞進一頁——重點太多等於沒有重點。CLAUDE.md 只放「全域必須知道」的資訊，細節交給 rules 模組。

💻 CLAUDE.md 基本骨架
# Project: My-App

## Overview
This is a Next.js SaaS application...

## Structure
/src/app     - Pages and routes
/src/lib     - Shared utilities
/src/styles  - Global CSS

## Workflow Rules
- Always run tests before committing
- Ask before modifying .env files
- Use conventional commit messages

## Git Rules
- Never use `git add -A`
- Never commit API keys or secrets
3
.claude/rules/：模組化關注點拆分
CLAUDE.md 負責大方向，但有些東西塞進去會讓它太長——比如個人畫像、Agent 人格設定、決策原則。官方的解法是 .claude/rules/ 目錄。裡面放 .md 檔案，沒有 paths 前綴的會自動全域載入。
CLAUDE.md handles the big picture, but some content makes it too long. The official solution is .claude/rules/ directory. Files without path prefixes auto-load globally alongside CLAUDE.md.
作者建議的拆分方式：
identity.md — Agent 是誰：語氣、態度、底線
principles.md — 決策原則：模糊時怎麼判斷
user.md — 使用者畫像：我是誰、在做什麼、偏好
regression.md — 回歸紀錄：踩過的坑、修正的規則
Recommended split: identity.md (agent persona), principles.md (decision heuristics), user.md (user profile), regression.md (past mistakes and corrections).
💻 目錄結構
.claude/rules/
├── identity.md      <- Agent persona & tone
├── principles.md    <- Decision heuristics
├── user.md          <- Who I am & preferences
└── regression.md    <- Lessons from past errors
⚖️ 工具比較：四個檔案的分工

identity.md：讓輸出風格穩定。定義語氣、關係（助手/同事/朋友）、主動程度。沒有這個檔案，CC 就是通用客氣的助手。

principles.md：處理邊界情況。要快還是要穩？改一個檔案還是三個？不確定時是猜還是問？寫清楚這些，行為就可預期。

user.md：校準回應層級。你的背景、領域、工作風格。CC 就不會在你問進階問題時從零科普。

regression.md：形成閉環。每次 CC 犯錯，記一條進去。定期把教訓回饋到 principles.md。但記得設定觸發條件，否則 CC 可能根本不會去讀這個檔案。

4
Context over Instruction：讓規則效果翻倍
大部分人寫規則只寫「做什麼」，不寫「為什麼」。比如「不要用 git add -A」——CC 會照做，但不理解原因，換個類似場景可能就不遵守了。
Most people write rules with only "what to do", not "why." CC follows them literally but without understanding, compliance breaks in analogous situations.
如果你寫「不要用 git add -A——API keys 和 .env 檔案會被靜默掃進去」，效果完全不同。CC 理解了原因之後，在所有相關場景都會自動規避，不只是你寫的那一條。
If you write "Don't use git add -A — API keys and .env files get silently staged," the effect is transformative. CC generalizes the principle across related scenarios.
社群裡的說法是：Context over Instruction（為什麼比怎麼做更重要）。不需要每條規則都加 Why，只給那些原因不夠直覺的加就行。方法很簡單：規則後面用破折號接上原因，一句話搞定。
The community calls this "Context over Instruction." Not every rule needs a why — just the non-obvious ones. Simply append a dash and the reason.
💡 比喻說明：SOP 寫法的既視感

這跟寫 GMP SOP 完全一樣的邏輯。好的 SOP 不只說「做 X」，還會加上 Rationale：「做 X，因為不做會導致交叉污染」。操作員理解了原因，遇到 SOP 沒覆蓋的邊界情況也能做出正確判斷。CC 的行為模式跟受過訓練的操作員一模一樣。

💻 範例對照
# Bad: instruction only
- Never use git add -A
- Always create a branch

# Good: instruction + context
- Never use git add -A
  — .env and API keys get staged silently
- Always create a branch
  — main is protected; direct push will fail
⚠️ 重點提示：這是 Prompt Engineering 的通用原則

Context over Instruction 不只適用於 Claude Code 配置——它是所有 system prompt 設計的核心原則。與其列出 100 條「不要做 X」，不如解釋 5 個「因為 Y 所以避免 Z」。LLM 的泛化能力強，給出原因比給出指令更有效率。

5
配置完成後的效果與批判分析
配好這套文件後，CC 每次 session 的起點不再是零。它知道專案結構、你是誰、應該用什麼語氣、哪些操作要先問你、上次踩過什麼坑。作者形容這是「質變級別」的體感差距。
After configuration, CC starts every session with full context — project structure, user identity, tone, approval gates, and past lessons. The author describes this as a "qualitative leap."
但需要注意的是：這篇文章是入門級介紹，缺乏進階用法。例如 paths 前綴的條件式載入（只在特定目錄觸發的規則）、多人協作時的 local vs project 衝突處理、配置文件的版本控制策略等都沒有涵蓋。
However, this is an introductory guide lacking advanced topics: conditional path-based loading, multi-user local vs project conflicts, and config versioning strategies are all omitted.
⚠️ 批判分析：2-3x Discount

「質變」宣稱需要打折：作者沒有提供 A/B 比較數據或具體案例。配置文件確實有用，但「質變」可能更準確地描述為「減少每次 session 的冷啟動成本」。實際效果取決於你的專案複雜度和 session 頻率。

200 行限制的來源不明：這是社群經驗值還是官方建議？具體的遵守率下降曲線是什麼？沒有引用來源。

regression.md 的觸發問題：作者自己也承認「記得設定觸發條件」，這暗示預設情況下 CC 不一定會主動讀取所有 rules 檔案，降低了這套體系的「自動化」程度。

Practice Questions
Q1 概念理解 CLAUDE.md 和 .claude/rules/ 的分工邏輯是什麼？為什麼不把所有內容都放在 CLAUDE.md 裡？

CLAUDE.md 負責專案級的「大方向」：結構、工作流、基本規範。rules/ 負責跨專案或關注點獨立的內容：人格設定、決策原則、使用者畫像。分開的原因有二：(1) 200 行限制——超出後遵守率下降；(2) 關注點分離——人格設定不應該跟 Git 規則混在一起，各自獨立更新更方便。

Q2 產業應用 如果你要為 CDMO 的自動化文件生成管線建立 Claude Code 配置，CLAUDE.md 和 rules/ 各放什麼？

CLAUDE.md：專案概述（這是一個教育文件生成 pipeline）、目錄結構（docs/、templates/、scripts/）、工作流規則（先讀 SKILL.md、兩階段建檔策略、必須 verify）、Git 規則（不提交含個人資訊的文件）。

rules/identity.md：AI Expert 角色設定、繁體中文為主、溫暖 Socratic 語氣。rules/user.md：CDMO 背景、COO 目標、雙語需求。rules/principles.md：完整性優先於速度、meta tag 一致性必須確保、不確定時問而不猜。rules/regression.md：過去遇到的截斷問題、雙欄不平衡問題等修正紀錄。

Q3 批判思考 「Context over Instruction」原則有沒有適用邊界？什麼情況下純指令反而更好？

Context over Instruction 的前提是 LLM 能正確理解因果關係並泛化。但有些情況純指令更安全：(1) 安全關鍵操作——「永遠不要刪除 production 資料庫」不需要解釋為什麼，LLM 理解了「為什麼」反而可能在某些邊界情況自行判斷可以刪；(2) 精確格式要求——「commit message 用 conventional commits 格式」給格式範例比解釋為什麼更有效；(3) token 預算有限時——加上 why 會增加配置文件長度，可能觸碰 200 行限制。最佳策略是：安全紅線用純指令（不留解讀空間），非直覺的偏好用 context。
