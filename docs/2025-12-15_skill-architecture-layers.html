Skill Architecture: The 5 Layers Beyond SKILL.md

從單一指令檔到行銷編排系統 — 架構才是真正的護城河
來源：The Boring Marketer @boringmarketer | 2025 | X (Twitter)
★★★★☆ 4.1/5 ↑ 實務應用、工具生態 ↓ 技術深度（無程式碼細節）
大多數人以為 AI skill 就是一份寫得好的 SKILL.md 指令檔——清晰的指示、評分標準、決策樹。這是基線，不是終點。 本文作者建立了 11 個行銷 skill、共 32,000 行，發現真正的效益不在 skill 本身，而在架構層。他分享了五個讓系統從「有用」升級為「越用越聰明」的關鍵設計：持久記憶、評分式 context 載入、schema 契約、學習回饋迴圈、共用協定層。 核心洞察：skill 是起點，圍繞它的架構才是讓它真正 perform 的關鍵。

核心五層架構 Core Architecture
Layer 1 持久記憶 Persistent Memory
問題核心：每個 session 都從零開始。你每次都要重新解釋你的品牌、受眾、市場定位。skill 完全不記得昨天發生了什麼。
"the problem: every session starts at zero. you re-explain your brand, your audience, your positioning every single time. the skill has no idea what happened yesterday."
解法：建立一個共用品牌目錄，每個 skill 都從中讀取、也寫入。檔案有所有權規則：profile 類檔案只有一個 skill 可以覆寫（需附 diff 確認）；append 類檔案只增不減，永遠成長。
"a shared brand directory that every skill reads from and writes to. profile files have one owner. only that skill can overwrite (with a diff and confirmation). append files never truncate. they only grow."
結果：session 1 建立的 context，在 session 20 時仍然有效。系統記得你是誰、做過什麼。
"session 1 builds context that session 20 uses. the skill remembers who you are and what you've done."
📚 目錄結構設計
所有者型（Owner-based）— 只能被特定 skill 覆寫：
voice-profile.md ← /brand-voice 擁有
positioning.md ← /positioning 擁有
audience.md ← /audience-research 擁有
keyword-plan.md ← /keyword-research 擁有

追加型（Append-only）— 所有 skill 都可寫入：
assets.md ← 所有 skill 追加
learnings.md ← 所有 skill 追加
💡 製藥業比喻
這就像 CDMO 的批次記錄系統：每個批次都可讀取既有的製程驗證數據，但只有製程擁有者（Process Owner）可以更新 Master Batch Record。偏差調查結果則是 append-only，永遠保留、不得刪除。記憶設計與 GMP 的 ALCOA+ 原則高度吻合。
Layer 2 評分式 Context 載入 Scored Context Loading
反直覺發現：給 AI 更多 context 不等於更好的輸出。測試顯示：同時把關鍵字計劃、競品分析、創意素材庫、品牌聲音都丟給 email skill，它寫出的是模糊的 email——試圖顧全所有東西，但注意力是有限的。
"give the email skill your keyword plan, competitor analysis, creative kit, and voice profile all at once — it writes muddy emails. tries to honor everything. attention is finite."
解法：一個context 評分矩陣，定義每個 skill 接收什麼、刻意屏蔽什麼。同時加入TTL 新鮮度機制：context 越舊，傳遞的方式越保守。
"a context matrix that scores what each skill receives vs what's withheld. each skill gets what sharpens it. everything else is withheld on purpose."
⚠️ Context TTL 新鮮度規則
< 7 天 → 直接傳入，原文不變
7–30 天 → 標記年齡警告後傳入
30–90 天 → 只傳摘要版本
> 90 天 → 不載入，視為過期資料
💡 比喻：手術室的資訊管控
外科醫生執刀時，護士不會把整本病歷丟在台上——只給當前手術最相關的那頁。精準的 context 控制讓 AI 的注意力聚焦，就像術前簡報只說此次手術的關鍵點，而非患者的童年病史。
🛠️ CDMO 應用場景
在技術移轉可行性評估中，給 AI 「所有客戶資料」反而產生泛化建議。正確做法：只載入與該劑型相關的設備規格和近期驗證結果，屏蔽無關的其他品項數據。
Layer 3 Schema 契約 Schema Contracts Between Skills
問題：Skills 在孤島中運作。關鍵字研究 skill 找到了主題，然後你開一個新對話，對內容撰寫 skill 重新解釋一切。資料無法在 skills 之間流動。
"skills work in isolation. the keyword researcher finds your topics. then you open a new conversation and re-explain everything to the content writer. data doesn't flow between skills."
解法：有型別的介面（typed interfaces）。Skills 不只輸出文字，而是輸出結構化產物（structured artifacts）供其他 skills 消費。共用 6 個 JSON Schema 契約——關鍵字研究 skill 輸出 keyword plan，SEO 內容 skill 直接讀取該 plan 作為輸入，無需重新解釋。
"typed interfaces. skills don't just output text. they output structured artifacts that other skills consume. 6 JSON Schema contracts. /keyword-research outputs a keyword plan. /seo-content reads that plan as input. no re-explaining."
結果：Skills 變成管線中的節點。一個的輸出成為下一個的輸入，資料全程保持結構化。
"skills become nodes in a pipeline. the output of one becomes the input of the next. the data stays structured the whole way."
📚 核心概念：Typed Interface
這是軟體工程中「強型別」的概念移植到 AI workflow。傳統 AI 互動輸出的是自由文字（untyped），任何下游系統都需要人工解讀。Schema 契約讓輸出變成可被機器直接消費的格式，實現真正的自動化管線。
💡 比喻：藥廠的 CoA 文件
Certificate of Analysis（CoA）就是一種 schema 契約：原料商的 CoA 有固定欄位格式，QC 實驗室直接讀取不需電話確認。AI skill 的 JSON Schema 就是 CoA 的數位對等物——確保上下游之間的資料交換不依賴人工轉譯。
⚠️ 重點提示
Claude Projects 目前不支援原生的 skill-to-skill 資料傳遞。作者使用的是共用檔案目錄作為中介層，而非 API 呼叫。在 Claude 環境中實作時，Project Knowledge 可扮演類似角色。
Layer 4 學習迴圈 Learning Loops
問題：Skill 在第 90 天的輸出品質和第 1 天一樣。沒有改善機制，每個 session 都是孤立的。
"the skill produces the same quality output on day 90 as day 1. no mechanism for improvement. every session is isolated."
解法：每次重大交付後，系統詢問一個簡單問題：「這個表現如何？」答案日期戳記並按 skill 標記後記錄。下次任何 skill 執行時，它會讀取這些學習記錄——email skill 停止寫疑問式主旨行，文案 skill 拋棄了過度修飾的風格。
"after every major deliverable the system asks a simple question. how did this perform? answers get logged. date-stamped. skill-tagged. next time any skill runs it reads those learnings."
結果：Session 1 教導 Session 5。錯誤只犯一次，成功永遠重複。系統越用越聰明。
"session 1 teaches session 5. mistakes are made once. wins repeat forever. the system gets smarter the more you use it."
🛠️ 學習回饋問卷（三選一）
執行完主要交付物後，系統詢問：

這次輸出的表現如何？
a) 直接使用（shipped as-is）
b) 小幅修改（minor edits）
c) 大幅重寫（rewrote significantly）

答案記錄到 learnings.md，附上日期與 skill 標籤。
💡 比喻：CAPA 系統
這和 GMP 的 CAPA（糾正與預防措施）精神完全相同：偏差被記錄，根因被分析，預防措施被執行，確保同樣的問題不再發生。學習迴圈就是 AI workflow 的 CAPA 系統。
Layer 5 共用協定層 Shared Protocol Layer
問題：11 個 skills 各自實作自己的記憶邏輯、新鮮度規則、回饋收集機制。處處不一致。
"11 skills each implementing their own memory logic, their own freshness rules, their own feedback collection. inconsistency everywhere."
解法：一份所有 skills 都遵循的協定文件。定義如何讀取品牌檔案、如何寫入、context 過期時的處理方式、如何收集回饋、檔案遺失時如何降級運作。沒有執行期強制——skills 就是遵循規範，就像 HTTP 協定一樣。
"one protocol document that every skill follows. it defines how to read brand files, how to write them, when context is too stale to use, how to collect feedback, how to degrade when files are missing. no runtime enforces it. the skills just follow the spec. like HTTP."
結果：一份文件，11 個 skills，系統級的一致性。
"one document. 11 skills. system-wide coherence."
📚 「像 HTTP 一樣」的深意
HTTP 是網際網路的共用語言——瀏覽器和伺服器各自不同，但都遵循同一套規範。沒有中央執行機構，卻能達到全球一致性。作者的 Protocol Layer 採用同樣思路：透過明確的書面規範，而非程式碼約束，確保所有 skills 行為一致。
🛠️ 協定文件應涵蓋的事項
1. 如何讀取品牌記憶檔案
2. 如何安全地寫入/追加
3. Context 新鮮度判斷規則（TTL）
4. 如何觸發學習回饋收集
5. 檔案遺失時的降級行為（graceful degradation）
💡 比喻：GMP SOPs
製藥工廠的 SOP 系統就是協定層：不同部門（QA、生產、QC）遵循同一套文件規範，確保行為可預測、可稽核。Protocol Layer 是 AI 工作流的 SOP——讓系統在沒有人工監督的情況下仍能一致運作。
系統成熟度進程 What This Adds Up To
系統在第 1 天無需任何設定即可運作。但隨著時間累積，每一層架構都在疊加品質：
Context 狀態系統輸出水準對應架構層

"day 1: it works. day 7: it works better. day 14: it works like it knows you."
重點整理 Key Takeaways
🗂️
記憶設計的所有權原則
品牌記憶不是自由的公共財。Profile 檔案需要明確的所有者，只有主責 skill 可以覆寫。這防止了多個 skills 互相覆蓋造成的「記憶腐化」。
🎯
Context 精準化：少即是多
給 AI 的 context 需要刻意篩選，而非傾倒所有資訊。注意力是有限資源，過多 context 導致輸出模糊。TTL 機制確保只有新鮮的資訊被載入。
🔗
Skills 互聯需要 Schema 契約
讓 AI workflow 從點到線，關鍵是定義結構化的輸出格式。JSON Schema 讓上游的輸出直接成為下游的輸入，消除人工轉譯的瓶頸。
📈
架構才是護城河，不是 Prompt
競爭對手可以複製你的 SKILL.md，但無法複製你累積的 learnings.md 和 positioning.md。架構設計創造了隨時間增長的複利效益——這才是真正的差異化。
練習題 Practice Questions
Q1 概念理解 為什麼「append-only」的 learnings.md 比「可覆寫」的設計更好？這個設計決策的邏輯是什麼？

核心邏輯：可覆寫的學習記錄會丟失歷史資訊。如果允許 skill 刪除舊記錄，當系統判斷「這個教訓已過時」時，可能錯誤刪除仍然有效的學習成果。

Append-only 的三個優點：1) 完整的歷史追溯性（可審計） 2) 系統永遠有更多數據可學習 3) 防止因誤判而丟失寶貴洞察。這與 GMP 的稽核軌跡原則完全一致——記錄只能新增，不能刪除或修改。

潛在缺點：長期累積後，learnings.md 可能變得過大，需要定期做 summarization 或分層歸檔，這是作者未在文章中討論的維護挑戰。

Q2 產業應用 假設你要為 CDMO 的技術移轉評估設計一套 Skill Architecture，哪兩個 Layer 的投資報酬率最高？為什麼？

優先推薦：Layer 2（評分式 Context 載入）+ Layer 4（學習迴圈）

Layer 2 的理由：技術移轉評估涉及大量文件（設備規格、法規要求、客戶 CMC 資料、既有驗證報告）。直覺上會想把所有資料都丟給 AI，但這正是導致「泛化建議」的根因。為每種劑型設計精準的 context 矩陣，只載入與當前評估相關的子集，可以立即提升輸出的針對性。

Layer 4 的理由：每次可行性評估完成後，都會有客戶反饋（接受/修改/拒絕）。這些反饋目前可能只存在於 email 往返中。系統化地捕捉這些學習，讓第 10 個客戶的評估直接受益於前 9 個的經驗，這是目前最容易實作、回報最快的架構投資。

Q3 批判思考 作者說「No runtime enforces it — the skills just follow the spec, like HTTP」。這個設計有什麼潛在風險？在高度受監管的環境中，這個假設能成立嗎？

核心風險：HTTP 之所以能在無中央執行的情況下運作，是因為違反協定的後果是立即可見的技術失敗（頁面無法載入）。但 AI skill 違反協定的後果可能是隱性的品質下降，難以被即時偵測。

在製藥受監管環境中的問題：GMP 要求每個控制措施都必須有驗證的執行機制（21 CFR Part 11 要求系統控制而非依賴人員自律）。「依賴 skills 自律遵循協定」在 FDA 審查中很可能被歸類為「程序控制」而非「系統控制」，符合性風險更高。

建議補強：對於監管關鍵的 AI workflow，應在協定層之上加入自動化合規檢查（Compliance Check Layer），定期驗證各 skill 的輸出是否符合協定規範，並留有可審計的記錄。

🎯 我的行動結論
✔ 核心洞察
我在 Claude Projects 中建立的 Master Prompt 和 Project Knowledge，本質上已是 Layer 1（持久記憶）的初階版本。但缺失的是 Layer 2 的「評分式載入」——我把所有 Project Knowledge 都同時載入，而非根據任務類型精準選擇。這解釋了為什麼有些輸出仍然「太泛化」。真正的升級點不是寫更好的 Prompt，而是設計更精準的 Context 管控機制。
📈 立即應用
本週在現有的可行性評估 Claude Project 中實作 Layer 4 的最簡版本：在每次評估完成後，手動在 Project Knowledge 的 learnings.md 中追加一條記錄（客戶類型 / 劑型 / 評估結果 / 主要修改點）。不需要自動化，先建立習慣和數據，三個月後再評估是否值得自動化。
⚠ 需要補充
文章完全基於行銷場景，且假設使用者有自架檔案系統的能力。在 Claude.ai 的 Projects 環境中，skills-to-skills 的 schema 契約（Layer 3）目前難以實作，需要人工橋接。此外，文章未討論監管合規環境中，「自律遵循協定」是否足夠——這是 CDMO 應用時最需要補充設計的部分。6 個月後重讀，評估 Claude Projects 是否新增了 skill-to-skill 資料傳遞的原生支援。
