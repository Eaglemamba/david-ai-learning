Executive Summary

這篇指南是少數專門為非工程師撰寫的 Cursor 使用教學，涵蓋四個核心用例：用 Mermaid 圖視覺化程式碼架構、用內建 Browser 測試功能、用新推出的 Plugin Marketplace 建立產品分析 Pipeline、以及用 Claude Code + Figma MCP 雙向推送原型設計。文章強調 Cursor 已從「工程師 IDE」轉型為產品團隊的多功能工作站，特別是 Plugin 生態系統讓 PM 可以一鍵串接 Amplitude、Linear、Notion 等工具。內容偏實操導向，概念深度適中，適合想快速上手 Cursor 的產品人員。

Part I — Cursor 定位與介面基礎

Cursor 本質上是一個以 AI 為核心的程式碼編輯器，但它正快速演化為產品團隊的多功能工作站。Stripe 已將 Cursor 部署給超過 3,000 名工程師，而越來越多的 PM 和設計師也開始將它作為日常工具。

"Cursor is used by some of the world's leading tech companies, including Stripe, who recently rolled it out to over 3,000 engineers."

對非工程師來說，Cursor 有兩種主要使用方式：從零建構東西（例如原型）或與現有程式碼互動（例如理解架構、生成文件）。不需要寫程式就能獲得巨大價值。

"...you can build something entirely from scratch... or you can use it to interact with your existing code base..."
⚠️ 重點提示：Working Directory
Working Directory（工作目錄）是 Cursor 的「視野範圍」。開太高層的資料夾 → Agent 雜訊太多；開太低層 → 失去專案全貌上下文。正確的做法是開啟專案根目錄。
💡 比喻說明
Working Directory 就像你走進一間辦公室——如果你站在大樓門口（太高層），你看到的是所有公司的招牌但什麼都不清楚；如果你只進了一間會議室（太低層），你不知道隔壁在做什麼。正確的做法是走進你自己團隊的樓層。
📚 核心概念：四種模式
模式用途PM 適用場景
Agent自主執行任務、寫/改程式碼建構原型、Plugin Pipeline
Ask問答、不修改檔案理解架構、詢問技術細節
Plan規劃方案再確認需求拆解、技術方案評估
Edit精準編輯特定程式碼微調 UI 文字/樣式
⚖️ Web App vs Desktop App
Web 版定位為「啟動與監控」工具——適合遠端委派 Agent 任務；Desktop 版才是真正的開發環境。兩者互補：在外面用 Web 啟動任務，回到桌面用 Desktop 審查和完善。
Part II — 視覺化程式碼架構

第一個實用場景是讓 Cursor 生成程式碼架構圖。只要輸入一句話，Cursor 就會用 Mermaid 圖表語法產出整個應用的架構概覽。

"Create a technical diagram showing the architecture of this app"

你還可以進一步聚焦到特定頁面或功能模組，例如只看首頁的建構邏輯，然後請 Cursor 逐一解釋每個元件的作用。

"...we can ask Cursor to dig deeper and focus specifically on a particular part of our code base..."

這對兩種場景特別有價值：剛上任需要理解技術棧，或新功能上線後想了解背後的技術實作。

🛠️ 實務應用：CDMO 場景
想像你的團隊剛導入新的 MES（Manufacturing Execution System, 製造執行系統）或 Batch Record 電子化系統。即使你看不懂程式碼，你可以把系統的程式碼資料夾拖進 Cursor，請它用 Mermaid 畫出資料流架構圖——從原料秤量、充填參數到批次放行的完整資料路徑。這比等工程師開會解釋快得多。
💡 比喻說明
把 Cursor 的架構視覺化功能想像成 Google Maps 的衛星視圖：你不需要親自走過每條街道，就能看到整個城市的佈局、主要道路和關鍵地標。先看大圖，再 zoom in 到你關心的區域。
Part III — Browser 內建測試與 Design Bar

Cursor 新推出的 Browser 功能在 IDE 內嵌了一個完整的瀏覽器，讓 AI Agent 可以直接控制瀏覽器進行導航、點擊、滾動、截圖，甚至監控 Console 輸出和網路流量。

"Cursor's Browser tool gives the AI agent direct control over a web browser... navigate, click, type, scroll, take screenshots, and monitor console output and network traffic..."

目前使用 @browser 指令讓 Agent 自動測試在外部託管的產品版本上效果較穩定，本地版本的自動化測試仍有限制。

另一個值得關注的新功能是 Design Side Bar（設計側邊欄），它讓你可以選取頁面上的任何 HTML 元素，即時修改 padding、layout 等屬性，非常適合做 Accessibility（無障礙性）檢查和快速的 UI 微調實驗。

📚 Browser 功能架構

開啟方式：三個點選單 → Open Browser，或快捷鍵 ⇧⌘B

能力範圍：導航、點擊、輸入、滾動、截圖、監控 Console 和網路請求

DevTools：右鍵即可開啟，與一般瀏覽器完全相同

限制：@browser 自動測試目前對外部託管網站較穩定

🛠️ 實務應用
Design Side Bar 對 PM 來說是個隱藏寶物——你可以在不動程式碼的情況下快速測試「如果按鈕再大一點？」「如果間距縮小？」這類問題。把截圖帶到設計評審會議，比空口描述有效率得多。
Part IV — Plugin Marketplace 與產品分析 Pipeline

Cursor 本週剛推出的 Plugin Marketplace 被作者評為「對非工程師最有用的更新」。Plugin 將 MCP 整合、Skills（技能）、Agents（代理）和 Hooks（鉤子）打包成一鍵安裝的套件。

"Plugins bundle together MCP integrations, Skills, Agents and Hooks into a single, one-click install."

文章示範了一個完整的產品分析 Pipeline：串接 Amplitude（行為數據）+ Linear（工單管理）+ Notion（文件記錄），一個指令就能監控關鍵指標、偵測異常、自動建立調查工單並更新 Notion 文件。

關鍵設計包含：一份 product-context.md 定義「正常」的指標範圍，一份 intel-workflow.md 作為 Master Prompt（主提示），最後存成斜線指令（Slash Command）以便重複執行。

💻 Pipeline 建構步驟

Step 1：安裝 Amplitude、Linear、Notion 三個 Plugin 並完成認證

Step 2：建立 Working Directory（context/ findings/ prompts/ 三個子資料夾）

Step 3：撰寫 product-context.md（定義正常指標範圍）

Step 4：撰寫 intel-workflow.md（Agent 執行邏輯）

Step 5：在 Notion 建立 Findings 頁面

Step 6：用 /add-command 存成可重複使用的斜線指令

⚠️ 重點提示：Context File 是關鍵
Pipeline 的品質取決於 product-context.md 的精確度。如果 Agent 誤報異常，就收緊範圍；如果漏報，就補充更多上下文。這跟所有 AI 系統一樣——Garbage In, Garbage Out。第一次執行後必須校準。
Part V — Claude Code + Figma MCP 雙向整合

最後一個用例是目前討論度最高的整合：Claude Code → Figma 雙向推送。過去 Figma 的 MCP 是唯讀的——你只能拉取設計上下文到 Cursor，但無法推送回去。

"Until recently, Figma's MCP was read-only... But now a new MCP tool called generate_figma_design makes the connection bidirectional..."

新的 generate_figma_design MCP 工具改變了這一點。Claude Code 現在可以擷取正在運行的本地 Web UI，並將其作為完全可編輯的 Frame 推送到 Figma 畫布上——包含真正的圖層和元件，不是扁平的截圖。

整個流程在 Cursor 的 Terminal（終端機）中透過 Claude Code 執行，而不是透過 Cursor 內建的 Agent Chat。你需要：安裝 Figma MCP、啟動本地應用、準備好 Figma 檔案 URL，然後一句話描述你要擷取什麼和推送到哪裡。

💻 操作步驟

Step 1：在 Cursor Terminal 中輸入 claude 啟動 Claude Code

Step 2：要求安裝 Figma MCP 並輸入 Figma Access Token

Step 3：重啟 Cursor（重要！否則 MCP 連線不會生效）

Step 4：確認本地 App 正在運行（如 localhost:5173）

Step 5：在 Claude Code 中描述擷取目標和 Figma 檔案 URL

🛠️ 實務應用：CDMO 場景
這個 Figma 雙向整合對內部工具的 UI 原型設計特別有意義。例如，你正在規劃一個新的 Batch Record Review Dashboard（批次記錄審查儀表板），可以先用 Claude Code 快速生成一個 functional prototype（功能原型），然後推送到 Figma 讓設計師接手精修——從「想法」到「可討論的設計稿」的時間從數天壓縮到數小時。
⚠️ 注意事項
作者特別提到忘記重啟 Cursor 導致 MCP 連線失敗的問題。安裝任何新的 MCP 後，務必完全關閉並重新開啟 Cursor。
Key Takeaways
Cursor 已不只是 IDE
Plugin Marketplace 的推出標誌著 Cursor 從「工程師的程式碼編輯器」轉型為「產品團隊的 AI 工作站」。PM、設計師都能在此建構端到端的自動化流程。
四種模式，各有其位
Agent 模式用於建構和執行，Ask/Plan 模式用於理解和規劃。非工程師最常用 Ask 來理解架構，用 Agent 來建構原型和 Pipeline。
Context File 決定 Pipeline 品質
無論是 product-context.md 還是 intel-workflow.md，AI Agent 的表現完全取決於你提供的上下文品質。第一次執行後的校準是必要步驟。
Figma MCP 從唯讀變雙向
generate_figma_design 讓 Claude Code 可以把本地 UI 推送為 Figma 可編輯 Frame。這打通了「原型 → 設計」的最後一哩路。
Practice Questions
Q1 概念理解 Cursor 的 Agent 模式和 Ask 模式在實際使用中的核心差異是什麼？為什麼非工程師需要區分這兩者？

Agent 模式具有自主執行能力——它會讀取檔案、寫入程式碼、執行命令，直到任務完成。Ask 模式只回答問題，不修改任何檔案。

對非工程師來說，這個區分非常重要：當你只想「理解」某個東西時（架構圖、技術解釋），用 Ask 模式更安全，不會意外修改程式碼。當你要「建構」東西時（原型、Pipeline），才切到 Agent 模式。這就像「只問問題」和「動手改東西」的差別——在 GMP 環境中，我們對「唯讀查詢」和「寫入修改」的權限管控概念完全一致。

Q2 產業應用 如果你要在 CDMO 環境中建構類似的「產品分析 Pipeline」，用 Cursor Plugin 串接哪些系統最有價值？需要考慮哪些合規限制？

在 CDMO 場景中，最有價值的串接可能是：LIMS/MES（取代 Amplitude，作為製造數據源）→ Deviation Management System（取代 Linear，作為調查工單系統）→ 電子批次記錄或 SharePoint（取代 Notion，作為文件記錄）。

但合規限制是最大的障礙：製造數據通常受 21 CFR Part 11 和 GMP 管控，不能直接透過 AI Agent 存取和分析。最可行的起點是用 非 GMP 的營運指標（如設備稼動率、排程達成率、客戶回應時間）來建構類似的 Pipeline，這些數據的合規敏感度較低，但同樣能產生有價值的營運洞察。

Q3 批判思考 文章呈現了一個相當理想化的 Plugin Pipeline 流程。在實際部署中，你認為最可能遇到的三個挑戰是什麼？

三個最可能的現實挑戰：

1. 認證與權限管理：三個 Plugin 的 OAuth Token 過期、權限變更、或公司 IT 政策限制 MCP 連線。在企業環境中，讓三個外部服務同時保持穩定連線並不簡單。

2. 誤報與校準成本：文章輕描淡寫了「校準 context file」這一步，但實際上這可能需要數週的反覆調整。指標的「正常範圍」本身就會隨季節、產品週期變化，需要持續維護。

3. Agent 判斷的可靠性：讓 AI Agent 自動建立 Linear 工單意味著你信任它的異常判斷。一旦誤判產生大量無效工單，團隊會迅速失去信任並停止使用。需要一個人工審查的 checkpoint——這呼應了 Copilot Mode 優於 Agent Mode 的原則。

🎯 我的行動結論
✔ 核心洞察
Cursor 的 Plugin Marketplace 代表了一個重要轉折：AI IDE 正在從「寫程式碼的工具」變成「串接工作流程的平台」。這跟我一直關注的 MCP 生態發展方向一致——真正的價值不在單一工具，而在工具之間的連接。對 COO 定位來說，理解這類「orchestration layer（編排層）」的演化比學會寫程式碼重要得多。
📈 立即應用
本週安裝 Cursor Desktop，用公司某個內部工具或文件系統的 repo 作為 Working Directory，測試 Ask 模式的架構視覺化功能。目標不是建構 Pipeline，而是先驗證「Cursor 能在多大程度上幫助我理解現有系統」這個基本問題。
⚠ 需要補充
文章缺乏企業環境的考量——IT 安全政策、資料外洩風險、Plugin 認證管理都未提及。在 GMP/CDMO 環境中，任何涉及生產數據的 AI Pipeline 都需要經過 Validation（驗證）和 Change Control（變更管制）。文章的 Pipeline 範例適合非管制環境，直接套用到製藥場景需要大幅調整。
