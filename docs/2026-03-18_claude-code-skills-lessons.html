Content

Part I
Skills = Folder, Not File
關於 Skills 最常見的誤解是它們「只是 Markdown 檔案」。但 Skills 最有趣的地方在於它們不只是文字檔，而是可以包含腳本、資源、資料等的資料夾，讓 Agent 可以探索和操作。
"A common misconception we hear about skills is that they are 'just markdown files', but the most interesting part of skills is that they're not just text files. They're folders that can include scripts, assets, data, etc."
在 Claude Code 中，Skills 還有各種設定選項，包括註冊動態 Hooks。Anthropic 發現最有趣的 Skills 正是那些創造性地使用這些設定選項和資料夾結構的 Skills。
"In Claude Code, skills also have a wide variety of configuration options including registering dynamic hooks. We've found that some of the most interesting skills in Claude Code use these configuration options and folder structure creatively."
📚 核心概念
Skill = Folder：SKILL.md 是入口，但整個資料夾才是完整的 Skill。可以包含 references/、assets/、scripts/ 等子目錄，Agent 會在適當時機讀取。
💡 比喻說明
把 Skill 想像成一本操作手冊：SKILL.md 是目錄頁，告訴你手冊裡有什麼。但真正的價值在手冊裡的流程圖（scripts）、範例表格（assets）、和參考資料（references）。只讀目錄頁，你只知道手冊存在；打開裡面的內容，你才能真正執行。
🛠️ CDMO 應用
你的 cdmo-feasibility-assessment Skill 目前主要是 SKILL.md。可以考慮加入 references/annex1-checklist.md 和 templates/assessment-report.md，讓 Agent 在需要時按需載入。
Part II
9 Skill Categories Taxonomy
Anthropic 在盤點所有 Skills 後，發現它們歸納為幾個反覆出現的類別。最好的 Skills 清楚地屬於某一類；最令人困惑的 Skills 則橫跨好幾類。
"After cataloging all of our skills, we noticed they cluster into a few recurring categories. The best skills fit cleanly into one; the more confusing ones straddle several."
以下是完整的 9 大分類。這不是最終定義，但它是思考你的組織是否缺少某些類型 Skills 的好方法。
"This isn't a definitive list, but it is a good way to think about if you're missing any inside of your org."
#CategoryPurpose
1Library & API ReferenceExplain correct usage of libraries/SDKs
2Product VerificationTest & verify code output
3Data Fetching & AnalysisConnect to data/monitoring stacks
4Business ProcessAutomate repetitive workflows
5Code ScaffoldingGenerate framework boilerplate
6Code Quality & ReviewEnforce code quality standards
7CI/CD & DeploymentFetch, push, deploy code
8RunbooksSymptom → investigation → report
9Infrastructure OpsRoutine maintenance with guardrails
⚠️ 重點提示
Single Responsibility Principle：最好的 Skill 清楚屬於一個類別。如果你的 Skill 同時做 verification + deployment + scaffolding，它可能需要拆分。
⚖️ 你的 Skills 盤點
以你目前的 Skills 對照：
✅ artifact-skill = Code Scaffolding (#5)
✅ cdmo-feasibility = Business Process (#4)
✅ amaran-presentation = Code Scaffolding (#5)
❓ Missing: Verification, Runbook, Code Quality

特別注意 #2 Product Verification：Thariq 說值得花一週讓工程師專門建立驗證 Skills。
💡 CDMO 類比
這 9 類像 GMP 文件系統的分類：SOP、Batch Record、Deviation Report、CAPA、Validation Protocol... 每類有明確用途。把 Deviation 寫成 SOP 格式會讓人困惑，同理，把 Runbook 寫成 Scaffolding Skill 也會讓 Agent 困惑。
Part III
Deep Dive: Categories 1-3
1. Library & API Reference — 解釋如何正確使用函式庫、CLI 或 SDK。這類 Skills 通常包含參考程式碼片段資料夾和一份讓 Claude 避免犯錯的 gotchas 清單。
"Skills that explain how to correctly use a library, CLI, or SDKs... often included a folder of reference code snippets and a list of gotchas."
2. Product Verification — 描述如何測試或驗證程式碼是否正常運作，通常搭配 Playwright、tmux 等外部工具。驗證 Skills 對於確保 Claude 輸出正確極為有用。值得讓工程師花一整週專門打造驗證 Skills。
"Verification skills are extremely useful for ensuring Claude's output is correct. It can be worth having an engineer spend a week just making your verification skills excellent."
進階技巧包括：讓 Claude 錄製其輸出的影片讓你能看到它到底測試了什麼，或在每一步強制執行程式化的狀態斷言。
"Consider techniques like having Claude record a video of its output so you can see exactly what it tested, or enforcing programmatic assertions on state at each step."
3. Data Fetching & Analysis — 連接你的資料和監控系統。這類 Skills 可能包含帶有認證資訊的函式庫、特定的 Dashboard ID、以及常見工作流程的說明。
"Skills that connect to your data and monitoring stacks. These skills might include libraries to fetch your data with credentials, specific dashboard ids."
🛠️ 實務應用
Category 1 在 CDMO 的對應：
想像一個 trackwise-api Skill，說明如何正確使用 TrackWise API 建立 deviation record，包含 gotchas 如「investigation field 不能超過 4000 chars」。
⚠️ 重點提示
Verification = Agent 的 QC
在 GMP 語境中，Product Verification Skill 就像 Agent 的品質管制步驟。沒有它，Agent 就是「做完不檢查」— 跟沒做 in-process check 一樣危險。

「錄製影片」技巧特別適合 UI 相關的驗證，讓你能回放 Agent 實際看到和測試的畫面。
📚 核心概念
Data Skill 的關鍵：不只是告訴 Agent 怎麼查資料，而是告訴它哪些資料在哪裡（Dashboard IDs、Table names、canonical user_id 欄位）。Domain knowledge 比 API syntax 更有價值。
Part IV
Deep Dive: Categories 4-6
4. Business Process & Team Automation — 將重複性工作流程自動化為單一指令。這類 Skills 通常是相當簡單的指令，但可能有更複雜的對其他 Skills 或 MCP 的依賴。將之前的執行結果存成 log 檔可以幫助模型保持一致性並反思先前的執行結果。
"Skills that automate repetitive workflows into one command... saving previous results in log files can help the model stay consistent and reflect on previous executions."
5. Code Scaffolding & Templates — 為你的 codebase 中特定功能生成框架模板程式碼。你可以將這些 Skills 與可組合的腳本結合。當你的 scaffolding 有無法純粹用程式碼覆蓋的自然語言需求時，它們特別有用。
"Skills that generate framework boilerplate for a specific function in codebase... especially useful when your scaffolding has natural language requirements that can't be purely covered by code."
6. Code Quality & Review — 在你的組織中強制執行程式碼品質並協助 code review。這些可以包含確定性腳本或工具以獲得最大穩健性。你可能想要在 Hooks 中自動執行這些 Skills，或在 GitHub Action 中運行。
"Skills that enforce code quality inside of your org and help review code. These can include deterministic scripts or tools for maximum robustness."
💡 比喻說明
Category 4 = 你的 SOP 自動化
每天的 standup、每週的 recap、建立 ticket... 這些在 CDMO 就像例行性文件：日報、週報、deviation ticket。Skill 把「人類記住流程 → 手動執行」變成「Agent 讀流程 → 自動執行」。
🛠️ 實務應用
adversarial-review 是一個特別值得注意的例子：它啟動一個全新的 sub-agent 來批判性地審查程式碼，實施修正，反覆迭代直到發現降級為吹毛求疵。

這跟 GMP 的 Peer Review 概念幾乎一致 — 找一個「全新視角」來審查你的工作。
⚠️ Log as Memory
Business Process Skills 存 log 的建議很關鍵。Agent 沒有跨 session 記憶，但 log 檔提供了一種窮人版 memory。下次執行時，Agent 讀取 log 就知道上次做了什麼。
Part V
Deep Dive: Categories 7-9
7. CI/CD & Deployment — 幫助你 fetch、push 和 deploy 程式碼。範例包括 babysit-pr（監控 PR → 重試 flaky CI → 解決 merge conflict → 啟用 auto-merge）。
"Skills that help you fetch, push, and deploy code inside of your codebase."
8. Runbooks — 接收一個症狀（如 Slack 訊息、警報或錯誤簽名），走過多工具調查流程，產出結構化報告。
"Skills that take a symptom (such as a Slack thread, alert, or error signature), walk through a multi-tool investigation, and produce a structured report."
9. Infrastructure Operations — 執行例行維護和操作程序，其中一些涉及需要護欄的破壞性操作。這讓工程師更容易在關鍵操作中遵循最佳實踐。
"Skills that perform routine maintenance and operational procedures — some of which involve destructive actions that benefit from guardrails."
📚 核心概念
Runbook = Deviation Investigation Skill
Category 8 最接近你想建立的 /triage-deviation。流程完全對應：症狀（OOS result）→ 多系統調查（batch record、環控、training record）→ 結構化報告。
⚠️ 重點提示
Guardrails for Destructive Ops
Category 9 提到的護欄概念在 GMP 中至關重要。orphan-cleanup Skill 的「找到 → 通知 → 等待期 → 確認 → 清除」流程，就像 CAPA 的「識別 → 調查 → 核准 → 執行 → 驗證」。
🛠️ babysit-pr 啟示
babysit-pr 是典型的 Agent Mode Skill：監控 → 判斷 → 自動修正 → 完成。但在 GMP 環境中，這種「自動修正」需要 human-in-the-loop。你可以建立一個「babysit」但在修正前暫停等待確認的版本。
Part VI
Don't State the Obvious + Build Gotchas
Claude Code 已經對你的 codebase 了解很多，Claude 對 coding 也有很多預設觀點。如果你要發布一個主要關於知識的 Skill，試著聚焦在那些能把 Claude 推離常規思考方式的資訊上。
"If you're publishing a skill that is primarily about knowledge, try to focus on information that pushes Claude out of its normal way of thinking."
frontend-design Skill 是一個好例子 — 它由 Anthropic 工程師透過與客戶反覆迭代來改善 Claude 的設計品味而建立，避免典型的模式如 Inter 字體和紫色漸層。
"The frontend design skill was built by one of the engineers at Anthropic by iterating with customers on improving Claude's design taste, avoiding classic patterns like the Inter font and purple gradients."
任何 Skill 中信號最高的內容就是 Gotchas Section。這些區段應該從 Claude 使用你的 Skill 時碰到的常見失敗點逐步累積。理想情況下，你會隨著時間更新 Skill 來捕捉這些 gotchas。
"The highest-signal content in any skill is the Gotchas section. These sections should be built up from common failure points that Claude runs into when using your skill."
⚠️ 重點提示
Gotchas = Skill 的 CAPA Log
每次 Agent 犯錯，就在 Gotchas 加一條。這跟 CAPA 的邏輯一樣：記錄失敗 → 分析根因 → 預防再發。Gotchas Section 就是 Skill 的持續改善機制。
💡 比喻說明
「不要陳述顯而易見的事」就像寫 SOP 時不需要解釋「打開電腦」。Claude 已經知道基本的 coding。你的 Skill 應該專注在你的組織特有的知識 — 那些 Claude 不可能從公開資料學到的東西。
🛠️ 對你的啟示
你的 Master Prompt v1.5.0 其實已經有 Gotchas 的雛形（例如 layout_balance_check、bilingual completeness 規則）。可以考慮在 artifact-skill 中加一個專門的 ## Gotchas 區段，收集所有已知的失敗模式。
Part VII
Progressive Disclosure & File System
你應該把整個檔案系統視為一種 Context Engineering 和 Progressive Disclosure 的形式。在 SKILL.md 中告訴 Claude 你的 Skill 裡有哪些檔案，它會在適當的時機讀取。
"You should think of the entire file system as a form of context engineering and progressive disclosure. Tell Claude what files are in your skill, and it will read them at appropriate times."
最簡單的 Progressive Disclosure 形式是指向其他 Markdown 檔案讓 Claude 使用。例如，你可以把詳細的函數簽名和使用範例拆分到 references/api.md。
"The simplest form of progressive disclosure is to point to other markdown files for Claude to use."
另一個例子：如果你的最終產出是一個 Markdown 檔案，你可以在 assets/ 中放一個範本檔供複製使用。你可以有 references、scripts、examples 等資料夾來幫助 Claude 更有效地工作。
"You can have folders of references, scripts, examples, etc., which help Claude work more effectively."
📚 核心概念
File System = Context Architecture
這完美呼應你的 three-layer architecture 原則（routing → module → data）。SKILL.md 是 routing layer，references/ 和 scripts/ 是 module layer，實際資料和範本是 data layer。
🛠️ 理想 Skill 資料夾結構
my-skill/
  SKILL.md          # Entry + routing
  references/
    api.md          # Detailed API docs
    gotchas.md      # Known failure points
  assets/
    template.md     # Output template
  scripts/
    validate.sh     # Verification script
  config.json       # User-specific setup
⚠️ Context 成本
Progressive Disclosure 的關鍵好處：SKILL.md 載入時只消耗少量 token，詳細內容只在需要時載入。這正是你在 Claude Code vs Claude.ai Projects 架構差異中觀察到的：Claude Code 可以選擇性載入，Projects 是全量注入。
Part VIII
Flexibility, Setup, Memory & Hooks
Avoid Railroading：Claude 通常會試著遵守你的指令，而且因為 Skills 非常可重用，你會想避免在指令中過於具體。給 Claude 它需要的資訊，但給它適應情境的彈性。
"Give Claude the information it needs, but give it the flexibility to adapt to the situation."
Setup Pattern：有些 Skills 需要使用者提供的初始設定。好的模式是把設定資訊存在 Skill 目錄中的 config.json。如果 config 未設定，Agent 就會詢問使用者。
"A good pattern to do this is to store this setup information in a config.json file in the skill directory."
Memory via Files：有些 Skills 可以透過在內部存儲資料來包含一種記憶形式。你可以用簡單的追加式文字 log 檔或 JSON 檔，甚至複雜到 SQLite 資料庫。注意：存在 Skill 目錄的資料在升級時可能被刪除，應使用 ${CLAUDE_PLUGIN_DATA} 作為穩定儲存路徑。
"Some skills can include a form of memory by storing data within them... Data stored in the skill directory may be deleted when you upgrade the skill."
On-Demand Hooks：Skills 可以包含只在 Skill 被調用時啟用的 Hooks，持續到 session 結束。用於那些你不想一直執行、但有時極為有用的 Hooks。例如 /careful 阻擋 rm -rf、DROP TABLE、force-push 等危險操作。
"Skills can include hooks that are only activated when the skill is called, and last for the duration of the session."
Store Scripts：你能給 Claude 最強大的工具之一就是程式碼。給 Claude 腳本和函式庫，讓它把精力花在組合上 — 決定下一步做什麼，而不是重建模板程式碼。Claude 可以即時生成腳本來組合這些功能。
"One of the most powerful tools you can give Claude is code. Giving Claude scripts and libraries lets Claude spend its turns on composition."
💡 Railroading 陷阱
寫 Skill 就像寫 SOP：太模糊沒有用，太詳細則讓執行者無法應對例外。最好的 SOP 定義目標和邊界，但留給操作員判斷的空間。Skill 也一樣。
⚠️ /careful Hook
/careful 是 On-Demand Hook 的完美案例：你不會希望它每次都跑（太煩），但當你碰 production 時絕對需要它。

CDMO 類比：就像進入無菌區才穿無菌衣，不是隨時都穿。情境決定護欄等級。
📚 Description = Trigger
Description 是給模型看的，不是給人看的。Claude Code 啟動時會建立所有 Skill 的描述列表，Claude 掃描這個列表來決定「這個請求有沒有對應的 Skill？」所以 description 不是摘要，而是觸發條件的描述。
🛠️ Composition Pattern
「給 Claude 腳本讓它組合」是一個強大的模式。想像一個 data-science Skill 裡有 helper functions 來 fetch 資料，Claude 可以即時生成腳本來組合這些函數做更複雜的分析。這把 Agent 的角色從「寫程式碼」提升到「設計解決方案」。
Part IX
Distribution, Marketplace & Measurement
分享 Skills 有兩種方式：將 Skills check in 到你的 repo（在 .claude/skills 下），或製作 plugin 並建立 Plugin Marketplace 讓使用者上傳和安裝。
"There are two ways you might share skills with others: check your skills into your repo (under ./.claude/skills) or make a plugin and have a Claude Code Plugin marketplace."
對於在相對較少 repo 上工作的小型團隊，check in skills 效果很好。但每一個 check in 的 Skill 都會稍微增加模型的 context。隨著規模擴大，內部 Plugin Marketplace 讓你可以分發 Skills 並讓團隊自己決定安裝哪些。
"For smaller teams working across relatively few repos, checking your skills into repos works well. But every skill that is checked in also adds a little bit to the context of the model."
Anthropic 沒有一個集中式團隊來決定哪些 Skills 進入 Marketplace。他們嘗試有機地找出最有用的 Skills：先在 sandbox 資料夾中上傳，在 Slack 推薦。當一個 Skill 獲得足夠的使用量後，才提 PR 移入 Marketplace。
"We don't have a centralized team that decides; instead we try and find the most useful skills organically."
Composing Skills：你可能想讓 Skills 彼此依賴。這種依賴管理尚未內建到 Marketplace 中，但你可以直接在 Skill 中引用其他 Skill 的名稱，如果已安裝，模型就會調用它們。
"You can just reference other skills by name, and the model will invoke them if they are installed."
Measuring Skills：Anthropic 使用 PreToolUse hook 來記錄公司內部的 Skill 使用情況。這讓他們能找到熱門的 Skills 或觸發率低於預期的 Skills。
"We use a PreToolUse hook that lets us log skill usage within the company... we can find skills that are popular or are undertriggering compared to our expectations."
⚖️ Repo vs Marketplace
Repo Check-in：簡單、版本控制、團隊自動同步。但每個 Skill 增加 context 負擔。

Plugin Marketplace：按需安裝、不增加不需要的 context。但需要維護基礎設施。

你的情況：個人使用為主 → Repo check-in 即可。但如果團隊採用，需要考慮 Marketplace。
⚠️ Curation Warning
很容易建立糟糕或冗餘的 Skills。Thariq 特別警告：在發布前確保有某種策展機制。這呼應了你的「Removal Test」原則 — 如果移除一個 Skill 後沒有人注意到，它可能不值得維護。
🛠️ 使用量追蹤
PreToolUse hook 記錄 Skill 使用量是一個聰明的做法。它解決了一個常見問題：你不知道你的 Skill 是否真的被觸發了。undertrigger 可能意味著 description 寫得不好（Agent 不知道何時使用），或 Skill 解決的問題其實不常見。
💡 Organic Curation
Anthropic 的「sandbox → traction → marketplace」流程像是 GMP 的驗證階段：IQ（安裝測試）→ OQ（操作測試）→ PQ（效能測試）。不是每個想法都能走到 PQ，只有證明有效的才正式上線。

Key Takeaways

Skill = Folder, Not File
SKILL.md 是入口，但真正的價值在整個資料夾結構：scripts/、references/、assets/。把檔案系統當作 Context Engineering 和 Progressive Disclosure 的載體。
Gotchas = Highest Signal
任何 Skill 中最高價值的內容是 Gotchas Section。從 Agent 的實際失敗點逐步累積，就像 CAPA log 一樣持續改善。
9 Types, Single Responsibility
最好的 Skill 清楚屬於一個類別（Library Reference、Verification、Runbook 等）。橫跨多類的 Skill 會讓 Agent 困惑，就像混合 SOP 和 Batch Record 一樣。
Description = Trigger, Not Summary
Skill description 是給模型看的觸發條件，不是給人看的摘要。寫得好，Agent 自動觸發；寫得差，Skill 形同虛設。

Practice Questions

Q1 概念理解 Thariq 說「最好的 Skills 清楚屬於一個類別」。如果你要建立一個同時做 deviation investigation + root cause analysis + CAPA tracking 的 Skill，根據本文的原則，你會怎麼拆分？

根據 9 大分類和 Single Responsibility 原則，應該拆分為三個 Skills：

1. deviation-triage (Runbook, #8)：接收症狀（OOS 結果、環控偏差），走過多系統調查流程，產出結構化報告。

2. root-cause-analyzer (Data Fetching & Analysis, #3)：連接 TrackWise、環控系統、training records，提供資料整合和相關性分析。

3. capa-creator (Business Process, #4)：根據調查結果，自動生成 CAPA 草稿，套用組織的模板和必填欄位。

這三個 Skills 可以透過 Composing Skills 的方式互相引用，但各自保持單一職責。

Q2 產業應用 文章提到 Product Verification Skills 值得花一整週工程師時間來建立。在 CDMO aseptic fill/finish 的情境中，你會建立什麼樣的 Verification Skill？它會驗證什麼？

在 aseptic fill/finish 情境中，一個 batch-record-verifier Skill 可以：

驗證項目：批次記錄的計算正確性（如填充量、稀釋比例）、環控數據與規格的一致性（differential pressure、particle count）、操作步驟的完整性（所有 sign-off 欄位是否填寫）。

技術手段：搭配 Python 腳本讀取 PDF/Excel 格式的 batch record，用 programmatic assertions 檢查每個欄位。類似 Thariq 提到的「在每一步強制執行程式化的狀態斷言」。

重要限制：這是 Copilot Mode 應用 — 驗證結果必須由 QA 人員審查確認，不能直接 auto-approve。GMP 要求人類最終簽核。

Q3 批判思考 本文來自 Anthropic 工程團隊，介紹的是他們自己產品 Claude Code 的 Skills 功能。考慮到這個來源身分，這篇文章可能遺漏了什麼？有哪些 Skill 的限制或問題是作者不太會主動提到的？

可能遺漏的面向：

1. Token/Cost 影響：文章只簡短提到 check-in 的 Skill 會增加 context，但沒有量化。100 個 Skills 各有 SKILL.md + references，載入成本是多少？這對 API 費用的實際影響？

2. Skill 衝突和除錯：當多個 Skills 被同時觸發，指令互相矛盾時怎麼辦？文章沒有討論 Skill 之間的優先級解決機制。

3. 維護成本：文章鼓勵建立更多 Skills，但「數百個 Skills」的維護負擔是什麼？有多少 Skills 過時了但沒人清理？

4. 失敗案例：文章只分享成功經驗，沒有討論「我們試過但失敗的 Skill 類型」或「這種 Skill 看起來好但實際上沒用」。

5. 競品比較：作為 Anthropic 員工，自然不會比較 Cursor Rules、Windsurf 的類似機制，也不會討論 Skills 架構的結構性限制。

這些並不削弱文章的價值 — 它仍然是目前最詳細的內部 Skills 使用心得。但應用 2-3x Discount Principle，對於「數百個 Skills 都很有用」的說法保持適度懷疑。
