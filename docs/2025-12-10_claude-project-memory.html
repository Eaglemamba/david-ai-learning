引言：Hook 的力量

作者在 LinkedIn 上每週發布 5-7 篇文章，上個月獲得超過 25 萬次曝光，但追蹤者只有 2,400 人。這個不成比例的成功來自一個關鍵因素：Hook（開場吸引語）。

"The right hook does 80% of the work. It stops people mid-scroll, makes them care enough to read, and eventually comment and share. A bad hook? Your best writing dies in silence."

問題是：作者使用 AI 來寫 LinkedIn 貼文，但一開始 Claude 總是產出 generic hooks（通用且平淡的開場）——安全、不會出錯，但也不會讓人停下來。

即使 Claude 已經讀過作者表現最好的文章，理解了他的聲音，輸出還是「還可以」——但「還可以」不會在 LinkedIn 爆紅。

直到作者開始用不同的方式使用 Project Memory。

ChatGPT vs Claude：Memory 的本質差異

你可能知道 ChatGPT 的記憶功能——它會記住你是誰，隨時間學習你的模式。但 ChatGPT 的記憶是一個black box（黑箱）：

  • 你無法真正編輯它
  • 你可以調整 system prompt，但無法直接控制它保留和優先處理什麼

Claude Projects 的運作方式不同：

  • 你可以直接編輯記憶
  • 你可以告訴 Claude 確切要記住什麼
  • 這些指令會帶入每一次對話——不是模糊的背景資訊，而是decision-making criteria（決策準則）

大多數人做錯了什麼

大多數人從 System Prompt 開始——這是正確的。你記錄自己的聲音、偏好、受眾、語調、背景資訊。你想到的每一條指令。

"This feels productive. More detail should mean better outputs, right?"

然後輸出還是感覺不對。不是完全錯誤，只是不像你。

所以你加更多。更多例子。更多說明。Prompt 越來越長，試圖涵蓋每一種邊緣情況。

這有幫助。但還是有差距。輸出在技術上遵循了指令，但感覺不像自己。你不斷手動修正同樣的模式。

"The problem wasn't the system prompt. The problem was I didn't actually know what I wanted until I saw what I didn't want."

Friction-Driven Memory：摩擦驅動記憶

什麼是 Friction（摩擦）？

Friction 是 Claude 給你的東西與你實際需要的東西之間的差距。當你讀 Claude 的輸出時想著：「這還可以，但不對」的那種感覺。

大多數人修正它然後繼續前進：

  • Claude 寫了一個通用的 hook？手動重寫它
  • Claude 聽起來太學術？自己簡化它
  • 問題解決了。下一個任務。

但那個摩擦會回來。明天。下週。每次你做類似的事情時。

"Because Claude doesn't know it made a mistake. You fixed the output, but you didn't fix the root cause."

到了第三次修正同樣的問題時，你就知道這是一個值得解決的模式。而你做的修正會揭示你事先無法表達的偏好——你不知道你討厭那種 hook 結構，直到 Claude 使用了它。

四步驟系統

Step 1: 即時注意摩擦（別只是修正它）

你正在與 Claude 合作。輸出中有些東西困擾你。你的本能是快速修正然後繼續前進。

停下來。

那個摩擦的時刻是有價值的。如果你修正它然後繼續，你就失去了數據。

反而，暫停並問自己：「這裡具體哪裡錯了？」

  • 是語調嗎？太正式？太隨意？太學術？
  • 是結構嗎？順序錯了？太長？漏了什麼？
  • 是一個模式嗎？你討厭的短語結構？感覺被強迫的修辭手法？

Step 2: 將摩擦轉換成指令

摩擦是一種感覺。Project memory 需要語言。你的工作是將一個轉換成另一個。

作者使用這個簡單框架：

項目 內容
What I got Claude 的實際輸出
What I needed 正確的應該是什麼
The gap 這兩者之間的差異是什麼？
The instruction 我應該告訴 Claude 什麼，讓這不再發生？

Step 3: 加入 Project Memory

現在把那個指令變成永久的。在 Claude Projects 中，你可以直接在對話中加入記憶。在更新前，作者通常會說：

"I noticed the hook you wrote explained the concept before revealing the insight. I need hooks that lead with the surprising claim first. Can you help me write a memory instruction that captures this preference?"

好的 memory entries 原則：

  • 具體，不要模糊
Bad: "Write better hooks"
Good: "Lead with the insight first. Don't set up context before the revelation."
  • 需要時包含例子
Don't write: 'Here's a feature, many people don't use it, but there's a pattern...'
Do write: 'You don't know what your project needs until it fails.'
  • 命名你要避免的模式
Avoid the 'They won't be X, they'll be Y' rhetorical contrast pattern.

Step 4: 測試和精煉

記憶加入了。但它有效嗎？

下次遇到類似情況時，注意觀察。Claude 有遵循指令嗎？摩擦消失了嗎？

  • 如果是：很好。繼續並等待下一個摩擦點出現
  • 如果否：指令可能不夠具體。精煉它

快捷方法：Friction-Detection Prompt

有時你在對話中間，邊做邊調整。你調整這裡的一個句子，重寫那裡的一個 hook。要求一個不同的角度。

你在 flow 中。你只是修正東西而沒有想太多。

但 Claude 正在注意。

Claude 可以看到你修正的模式，即使你太專注而沒注意到。每次你調整什麼，Claude 都在追蹤改變了什麼以及為什麼。

核心提示詞：
"Looking at our conversation so far, are there any patterns in how I've corrected your outputs or asked you to adjust things? What preferences or rules am I demonstrating that aren't captured in your system prompt or project memory yet?"

為什麼這有效：

  • Claude 對對話有完美的記憶。你可能忘了你連續調整了三次語調，但 Claude 不會
  • 它揭示盲點。你沒有意識到的摩擦被命名和識別
  • 它為你做翻譯工作。Claude 根據觀察到的模式草擬記憶條目

替代提示詞

做了多次修正後：

"I've made several edits to your outputs in this conversation. Can you identify the pattern and suggest a memory instruction that would prevent this friction next time?"

全面分析：

"Based on how I've responded to your outputs today - what I kept, what I changed, what I asked you to redo - draft 2-3 specific memory instructions that would improve our collaboration."

工作中快速檢查：

"Pause. Have I been correcting the same type of thing repeatedly in the last few chats? If so, what memory instruction would fix it?"

實際案例

案例 1：LinkedIn Hook 問題

情況：作者需要能讓人停止滾動的 hook——大膽、具體、引起好奇心的陳述。但 Claude 一直寫解釋性的 hook。在揭示之前先設定。先給背景再給重點。

摩擦：Claude 給的：

"The AI race is heating up. Companies are making important decisions about which AI providers to use. But the patterns emerging might surprise you."

這是反的。太多鋪陳。沒有張力。沒有具體性。

作者手動重寫：

"Anthropic just ate 20% of OpenAI's enterprise market in 24 months. And it's not even close to done."

最終 Memory Entry：

Hook Framework (LinkedIn):
- Lead with the insight or surprising claim FIRST
- Don't explain or set up context before the revelation
- Structure: Insight → Context → Explanation
- Make readers think "Wait, what?" before "Oh, I see"
- Avoid: "Here's X. Most people don't Y. But there's Z..."
- Instead: Lead with Z immediately

案例 2：Tool-Name Creep 問題

情況：作者在寫 Substack Notes——需要對任何使用 AI 的人都通用和相關的簡短社交媒體貼文。策略是專注於轉變和原則，而不是工具功能。但 Claude 一直提到特定的工具名稱。

摩擦：Claude 輸出：

"Cursor just changed how I think about coding. Instead of fighting syntax, the AI handles boilerplate and I focus on logic."

作者編輯：

"Programming became collaborative. The AI handles syntax patterns, I focus on problem-solving logic."

最終 Memory Entry：

Substack Notes Strategy - Universal Insights:
NEVER mention specific AI tools by name (Cursor, Claude, NotebookLM, ChatGPT, etc.)
Focus on transformations, not tools:
- Instead of "Cursor made me think in systems" → "Thinking in systems changed everything"
- Instead of "Claude taught me to code" → "Programming became collaborative"

案例 3：銷售郵件太企業化問題

情況：作者在寫黑色星期五促銷郵件。他的聲音是樂觀但誠實的。直接但不推銷的。但 Claude 一直默認為典型的行銷誇大語言。

摩擦：Claude 輸出：

"🔥 LAST CHANCE! Don't miss out on this incredible opportunity! This deal won't last forever - grab it now before it's gone! 🚨"

作者重寫：

"This ends Monday, then we're back to full price. You're either subscribing now or waiting another year for this offer."

最終 Memory Entry（部分）：

MAINTAIN (Authentic conviction):
✅ Direct challenge: "You're either subscribing now or waiting another year"
✅ Honest urgency: "This ends Sunday, then we're back to full price"

AVOID (Typical marketing hype):
❌ Hype energy: "🔥 LAST CHANCE 🔥"
❌ Typical marketing urgency: "Don't miss out!"

進階：Knowledge Base Files

一旦你開始建立 project memory，你最終會達到一個門檻。當前的記憶介面限制條目只能寫幾句話。有些項目可以用一段簡潔的指令來表達；其他的則需要更詳細的框架。

這就是 Project Knowledge Base Files 變得必要的地方。

這樣想：
Project Memory = Claude 在每次對話中應用的主動指令
Knowledge Base Files = Claude 在需要時查閱的參考框架

建議做法：

  • Memory entries 應該是簡潔的指針："Do X, reference Y framework, avoid Z pattern."
  • Knowledge base files 應該是全面的文檔：詳細的例子、邊緣案例、前後對比、決策樹

實作方式：請 Claude 為你創建記憶，生成為 Markdown 文件，然後上傳到你的 project knowledge base。

結論：這實際上教會你什麼

"This process teaches you how your own taste actually works."

大多數人認為他們知道自己想要什麼。他們有模糊的偏好：「我喜歡對話式寫作」或「我想要吸引人的 hook」或「保持簡單」。

但這些不是可執行的。它們是沒有座標的方向。

Friction 迫使你變得具體。

當 Claude 寫出不符合你品味的東西時，你不能只說「那不對」然後繼續。你必須表達為什麼它不對。你必須命名困擾你的具體事物。你必須把感覺轉換成語言。

這很難。因為我們大多數人以前從未有意識地檢視過自己的偏好。我們只是在看到時知道自己喜歡什麼。

但建立 project memory 要求你逆向工程你自己的品味。發現你保留什麼與你改變什麼之間的模式。注意到你一直在遵循但從未意識到存在的隱形規則。

"You're not just teaching Claude. You're teaching yourself."

行動建議

  1. 這週，開啟一個 Claude Project（如果已有，選最常用的那個）
  2. 工作時注意摩擦。當你發現自己在編輯 Claude 的輸出時，暫停。注意你在改什麼以及為什麼
  3. 使用 friction-detection prompt。讓 Claude 告訴你它注意到什麼
  4. 從小處開始。這週一個 memory entry 就夠了

核心概念解析：Project Memory vs System Prompt

System Prompt（系統提示詞）：你在開始工作前的最佳猜測。像是在正式約會前，你想像對方喜歡什麼類型的餐廳。

Project Memory（專案記憶）：你透過實際工作學到的東西。像是約會幾次後，你知道她對海鮮過敏、喜歡安靜的環境。

System Prompt 是假設，Project Memory 是證據。

比喻說明：餐廳主廚的學習

想像你是一個新餐廳的常客。一開始，主廚只知道你的基本資料（System Prompt）：「這位客人喜歡義大利菜」。

但透過多次用餐（產生 Friction）：

  • 你把洋蔥挑掉 → 主廚記下「不吃洋蔥」
  • 你稱讚青醬 → 主廚記下「偏好青醬」
  • 你抱怨太鹹 → 主廚記下「口味清淡」

這些就是 Memory Entries。主廚不是一開始就知道這些，而是透過觀察你的反應（Friction）學習的。

重點提示：為什麼「還可以」不夠好

在 AI 內容創作中，「還可以」是最危險的結果。它不會讓你停下來思考，但也不會讓讀者停下來閱讀。

真正的價值創造發生在：

  • 你感覺到不對的那一刻
  • 你停下來分析為什麼不對
  • 你轉化成指令讓 AI 學習

核心概念解析：Friction（摩擦）

定義：AI 輸出與你實際需要之間的差距。

信號：當你想說「這還可以，但不對」的時候。

價值：摩擦是數據。它告訴你 AI 還不理解什麼。

摩擦不是 bug，摩擦是系統本身。

比喻說明：健身教練的微調

好的健身教練不會一開始就給你完美的訓練計劃。他會：

  1. 給你一個基礎計劃（System Prompt）
  2. 觀察你做動作時哪裡卡住（Friction）
  3. 記錄下來並調整（Memory Entry）
  4. 下次訓練時直接用調整後的版本

「你深蹲時膝蓋會內扣」這種觀察，比「我要完美的深蹲」的指令有用 100 倍。

框架：Friction → Instruction 轉換公式

What I got Claude 實際給你的
What I needed 你真正想要的
The gap 兩者差異是什麼
The instruction 防止再發生的指令

重點提示：好的 Memory Entry 三原則

  1. 具體而非模糊：「先給洞見」而非「寫好的 hook」
  2. 包含例子：展示做與不做的對比
  3. 命名要避免的模式：給壞模式一個名字讓 AI 識別

比喻說明：偵探 Claude

使用 Friction-Detection Prompt 就像請一個偵探分析你的行為模式。

你可能沒注意到自己連續三次把 hook 中的解釋移到後面，但 Claude 這個「偵探」看得一清二楚。

它會說：「我注意到你每次都把驚人的主張移到最前面，刪除所有鋪墊語言...」

讓 AI 告訴你，你自己的品味是什麼。

練習思考 1

情境：你請 Claude 寫一篇關於時間管理的文章，它給了你一個充滿「10 個技巧」、「提升效率」等陳詞濫調的版本。你覺得不對，但說不出為什麼。

問題：用 Friction → Instruction 框架，你會怎麼分析這個問題並寫出 Memory Entry？

練習思考 2

情境：你發現 Claude 寫的電子郵件總是以「I hope this email finds you well」開頭，而你每次都要刪掉它。

問題：這個 Friction 揭示了你什麼偏好？你會寫什麼樣的 Memory Entry？

練習思考 3

情境：你是一個 B2B SaaS 公司的內容行銷人員。你想用這篇文章的方法來訓練 Claude 寫符合你品牌聲音的內容。

問題：你會設計什麼樣的 Friction-Detection Prompt 來發現你的品牌聲音特徵？

重點提示：真正的學習發生在哪裡

這個過程最意外的收穫不是讓 Claude 變得更好，而是讓你更了解自己。

Friction 迫使你逆向工程自己的品味——發現你一直在遵循但從未意識到的隱形規則。

你不只是在教 Claude，你也在教自己。

兩種工作流程

Option A（快速）：

工作 → Detection Prompt → 加入建議

Option B（詳細）：

工作 → 注意摩擦 → 轉換指令 → 測試精煉

實務應用：你的第一週計劃

  1. Day 1-2：正常使用 Claude，但開始注意「不對勁」的感覺
  2. Day 3-4：當你手動修改輸出時，問自己「為什麼要改？」
  3. Day 5：使用 Friction-Detection Prompt，讓 Claude 告訴你它觀察到的模式
  4. Day 6-7：選一個最明顯的模式，寫成 Memory Entry

目標：這週只加一條 Memory Entry，但要是真正從摩擦中學到的。

文章核心洞見總結

  1. 你不知道你要什麼，直到你看到你不要什麼
  2. 摩擦是數據，不是問題
  3. 把感覺轉化成語言是最難也最有價值的工作
  4. 這個過程教會你了解自己的品味

"Friction isn't a bug in the system. It's the system."
