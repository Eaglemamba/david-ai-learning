Part I — 雙資料夾架構
Two Folders, Not One
在深入之前，有一件事值得先知道：實際上有兩個 .claude 目錄，不是一個。第一個位於專案內部，第二個位於你的 home 目錄。
Before diving in, one thing worth knowing upfront: there are actually two .claude directories, not one. The first lives inside your project and the second lives in your home directory.
專案層級資料夾存放團隊配置。你把它提交到 git。團隊中每個人都使用相同的規則、相同的自訂指令、相同的權限策略。
The project-level folder holds team configuration. You commit it to git. Everyone on the team gets the same rules, the same custom commands, the same permission policies.
全域 ~/.claude/ 資料夾存放你的個人偏好和機器本地狀態，例如 session 歷史記錄和自動記憶。
The global ~/.claude/ folder holds your personal preferences and machine-local state like session history and auto-memory.
💡 比喻說明
想像一家製藥廠的 SOP 系統：專案層級就像廠區的標準作業程序（每個人都必須遵守），全域層級就像你個人的工作偏好設定（你習慣的工具排列、快捷鍵）。SOP 歸公司管，偏好歸個人管。
⚖️ 工具比較
層級路徑Git 提交用途
Project.claude/Yes團隊共用規則
Global~/.claude/No個人偏好
Local overrideCLAUDE.local.mdNo (gitignored)個人覆寫
⚠️ 重點提示
CLAUDE.local.md 和 settings.local.json 會自動被 gitignore，確保個人覆寫不會被提交到版本庫。這是個很聰明的設計 — 團隊標準和個人偏好可以共存。
Part II — CLAUDE.md 核心指令檔
CLAUDE.md: Claude's Instruction Manual
這是整個系統中最重要的檔案。當你啟動 Claude Code session 時，它讀取的第一個東西就是 CLAUDE.md。它直接將內容載入 system prompt，並在整個對話過程中記住它。
This is the most important file in the entire system. When you start a Claude Code session, the first thing it reads is CLAUDE.md. It loads it straight into the system prompt and keeps it in mind for the entire conversation.
簡單來說：你寫在 CLAUDE.md 裡的東西，Claude 都會遵守。如果你告訴它在實作之前永遠先寫測試，它就會這樣做。
Simply put: whatever you write in CLAUDE.md, Claude will follow. If you tell Claude to always write tests before implementation, it will.
專案根目錄的 CLAUDE.md 是最常見的設定。但你也可以在 ~/.claude/CLAUDE.md 放一個用於全域偏好，甚至在子目錄中放一個用於資料夾特定規則。Claude 會讀取所有這些檔案並合併它們。
A CLAUDE.md at your project root is the most common setup. But you can also have one in ~/.claude/CLAUDE.md for global preferences that apply across all projects, and even one inside subdirectories for folder-specific rules. Claude reads all of them and combines them.
該寫什麼：建置/測試/lint 指令、關鍵架構決策、不明顯的陷阱、import 慣例、命名模式、錯誤處理風格、主要模組的檔案和資料夾結構。
Write: Build, test, and lint commands; key architectural decisions; non-obvious gotchas; import conventions, naming patterns, error handling styles; file and folder structure for the main modules.
不該寫什麼：任何已經屬於 linter 或 formatter 設定的東西、可以直接連結的完整文件、解釋理論的長篇段落。
Don't write: Anything that belongs in a linter or formatter config; full documentation you can already link to; long paragraphs explaining theory.
保持 CLAUDE.md 在 200 行以內。超過這個長度的檔案會開始消耗太多 context，Claude 的指令遵從性實際上會下降。
Keep CLAUDE.md under 200 lines. Files longer than that start eating too much context, and Claude's instruction adherence actually drops.
💻 程式碼/API — 有效的 CLAUDE.md 範例

只有約 20 行。給了 Claude 在這個程式碼庫中高效工作所需的一切。

⚠️ 重點提示 — 200 行法則
這個 200 行限制是 context window 管理的核心原則。CLAUDE.md 的內容會被注入到 system prompt 中，每個 token 都在佔用你的 context window 空間。

這跟我們在 Master Prompt 設計中的經驗一致 — context engineering 的精髓是精準而非大量。與其給 Claude 一份 500 行的百科全書，不如給它 20 行的精確指令。
🛠️ 實務應用 — CDMO 情境
如果你的團隊用 Claude Code 開發 GMP 相關的自動化腳本，CLAUDE.md 應該包含：
- 資料完整性要求（ALCOA+ 原則）
- 日誌記錄規範（不得使用 console.log）
- 驗證規則（所有輸入必須 schema 驗證）
- 禁止事項（不得直接修改生產資料庫） 
# Project: Acme API

## Commands
npm run dev          # Start dev server
npm run test         # Run tests (Jest)
npm run lint         # ESLint + Prettier check

## Architecture
- Express REST API, Node 20
- PostgreSQL via Prisma ORM
- All handlers live in src/handlers/

## Conventions
- Use zod for request validation
- Return shape: { data, error }
- Never expose stack traces
- Use logger module, not console.log

## Watch out for
- Tests use real local DB
- Strict TypeScript: no unused imports
Part III — rules/ 模組化指令
The rules/ Folder: Modular Instructions That Scale
CLAUDE.md 對單一專案很好用。但一旦你的團隊成長，你最終會有一份 300 行的 CLAUDE.md，沒人維護，每個人都忽略。
CLAUDE.md works great for a single project. But once your team grows, you end up with a 300-line CLAUDE.md that nobody maintains and everyone ignores.
.claude/rules/ 資料夾解決了這個問題。裡面的每個 markdown 檔案都會自動和 CLAUDE.md 一起載入。你可以按關注點拆分指令，而不是塞在一個巨大的檔案裡。
The rules/ folder solves that. Every markdown file inside .claude/rules/ gets loaded alongside your CLAUDE.md automatically. Instead of one giant file, you split instructions by concern.
真正的威力來自路徑範圍限定的規則。在規則檔案中加入 YAML frontmatter 區塊，它就只會在 Claude 處理匹配的檔案時啟動。
The real power comes from path-scoped rules. Add a YAML frontmatter block to a rule file and it only activates when Claude is working with matching files.
沒有 paths 欄位的規則會無條件載入，每個 session 都會生效。
Rules without a paths field load unconditionally, every session.
💻 rules/ 資料夾結構範例
.claude/rules/
├── code-style.md
├── testing.md
├── api-conventions.md
└── security.md

每個檔案保持專注且容易更新。負責 API 慣例的團隊成員編輯 api-conventions.md，不會互相影響。

💻 路徑範圍限定規則

Claude 編輯 React 元件時不會載入這個規則檔案。只有在 src/api/ 或 src/handlers/ 裡工作時才會載入。

💡 比喻說明
就像製藥廠裡不同區域有不同 SOP — 無菌區的 SOP 跟倉儲區的完全不同。path-scoped rules 讓你的 Claude Code 也能「看到自己在哪個區域」，只套用該區域的規則。 
---
paths:
  - "src/api/**/*.ts"
  - "src/handlers/**/*.ts"
---
# API Design Rules

- All handlers return { data, error }
- Use zod for request body validation
- Never expose internal error details
Part IV — commands/ 自訂斜線指令
The commands/ Folder: Custom Slash Commands
Claude Code 有內建的斜線指令如 /help 和 /compact。commands/ 資料夾讓你可以新增自己的指令。
Out of the box, Claude Code has built-in slash commands like /help and /compact. The commands/ folder lets you add your own.
你放進 .claude/commands/ 的每個 markdown 檔案都會變成一個斜線指令。名為 review.md 的檔案會建立 /project:review。檔案名就是指令名。
Every markdown file you drop into .claude/commands/ becomes a slash command. A file named review.md creates /project:review. The filename is the command name.
! 反引號語法會執行 shell 指令並嵌入輸出。這讓這些指令真正有用，而不只是儲存的文字。
The ! backtick syntax runs shell commands and embeds the output. That's what makes these commands genuinely useful instead of just saved text.
使用 $ARGUMENTS 來傳入指令名之後的文字。例如執行 /project:fix-issue 234 會把 issue 234 的內容直接送入 prompt。
Use $ARGUMENTS to pass text after the command name. Running /project:fix-issue 234 feeds issue 234's content directly into the prompt.
專案 vs 個人指令：專案指令在 .claude/commands/ 中提交給團隊共用。個人指令放在 ~/.claude/commands/，顯示為 /user:command-name。
Project commands in .claude/commands/ are committed and shared with your team. For commands you want everywhere, put them in ~/.claude/commands/. Those show up as /user:command-name.

🛠️ 實務應用
Claude Code 的 commands/ 機制跟我們在 claude.ai Projects 裡用的 Master Prompt 有異曲同工之妙 — 都是把重複的指令結構化保存。差異在於 commands 還能嵌入 shell 指令的即時輸出，這是 Projects 做不到的。

潛在的 CDMO 用途：建立一個 /project:deviation-check 指令，自動拉出最近的程式碼變更並交叉檢查 SOP 合規性。 
---
description: Review current branch diff
---
## Changes to Review

!`git diff --name-only main...HEAD`

## Detailed Diff

!`git diff main...HEAD`

Review the above changes for:
1. Code quality issues
2. Security vulnerabilities
3. Missing test coverage
4. Performance concerns
---
description: Investigate and fix a GitHub issue
argument-hint: [issue-number]
---
Look at issue #$ARGUMENTS in this repo.

!`gh issue view $ARGUMENTS`

Understand the bug, trace it to root cause,
fix it, and write a test.
Part V — skills/ 自動觸發工作流
Skills: Auto-Invoked Workflows
Skills 和 Commands 表面上看起來相似，但觸發方式根本不同。Commands 等待你輸入斜線指令。Skills 會監聽對話並在時機正確時自動執行。
Skills look similar on the surface, but the trigger is fundamentally different. Commands wait for you. Skills watch the conversation and act when the moment is right.
每個 skill 位於自己的子目錄中，帶有一個 SKILL.md 檔案。SKILL.md 使用 YAML frontmatter 來描述何時使用它。
Each skill lives in its own subdirectory with a SKILL.md file. The SKILL.md uses YAML frontmatter to describe when to use it.
Commands 和 Skills 的關鍵區別：skills 可以將支援檔案打包在一起。@DETAILED_GUIDE.md 引用會拉入一份位於 SKILL.md 旁邊的詳細文件。Commands 是單一檔案，Skills 是套件包。
The key difference from commands: skills can bundle supporting files alongside them. The @DETAILED_GUIDE.md reference pulls in a detailed document that lives right next to SKILL.md. Commands are single files. Skills are packages.
Part VI — agents/ 子代理人
Agents: Specialized Subagent Personas
當任務複雜到需要一個專門的專家時，你可以在 .claude/agents/ 中定義子代理人角色。每個 agent 是一個 markdown 檔案，擁有自己的 system prompt、工具存取權限和模型偏好。
When a task is complex enough to benefit from a dedicated specialist, you can define a subagent persona in .claude/agents/. Each agent is a markdown file with its own system prompt, tool access, and model preference.
當 Claude 需要完成程式碼審查時，它會在自己獨立的 context window 中啟動這個 agent。Agent 完成工作、壓縮結果，然後回報。你的主要 session 不會被數千個中間探索的 token 塞滿。
When Claude needs a code review done, it spawns this agent in its own isolated context window. The agent does its work, compresses the findings, and reports back. Your main session doesn't get cluttered.
model 欄位讓你可以使用更便宜、更快的模型來處理專注任務。Haiku 能很好地處理大多數唯讀探索。把 Sonnet 和 Opus 留給真正需要它們的工作。
The model field lets you use a cheaper, faster model for focused tasks. Haiku handles most read-only exploration well. Save Sonnet and Opus for the work that actually needs them.

Part VII — settings.json 權限控制
settings.json: Permissions and Project Config
settings.json 控制 Claude 可以做和不可以做什麼。它是你定義哪些工具 Claude 可以執行、哪些檔案可以讀取、以及執行某些指令前是否需要詢問的地方。
The settings.json file controls what Claude is and isn't allowed to do. It's where you define which tools Claude can run, which files it can read, and whether it needs to ask before running certain commands.
allow 清單包含無需確認即可執行的指令。對大多數專案，一個好的 allow 清單涵蓋：你的 run 指令、唯讀 git 指令、以及檔案操作。
The allow list contains commands that run without Claude asking for confirmation. For most projects, a good allow list covers your run commands, read-only git commands, and file operations.
deny 清單包含完全被封鎖的指令，無論如何都不行。一個合理的 deny 清單會封鎖：破壞性 shell 指令如 rm -rf、直接網路指令如 curl、以及敏感檔案如 .env。
The deny list contains commands that are blocked entirely, no matter what. A sensible deny list blocks destructive shell commands, direct network commands, and sensitive files.
如果某個操作不在任何一個清單中，Claude 會在繼續之前詢問。這個中間地帶是刻意設計的。它給你一個安全網，而不需要預先考慮每一個可能的指令。
If something isn't in either list, Claude asks before proceeding. That middle ground is intentional. It gives you a safety net without having to anticipate every possible command upfront.
Part VIII — 全域 ~/.claude/ 與漸進式導入
Global ~/.claude/ and Getting Started
~/.claude/CLAUDE.md 會載入每一個 Claude Code session，跨所有專案。適合放你的個人編碼原則、偏好風格，或任何你希望 Claude 不論在哪個 repo 都記住的東西。
~/.claude/CLAUDE.md loads into every Claude Code session, across all your projects. Good place for your personal coding principles, preferred style, or anything you want Claude to remember regardless of which repo you're in.
~/.claude/projects/ 儲存每個專案的 session 記錄和自動記憶。Claude Code 在工作時會自動儲存筆記給自己：它發現的指令、觀察到的模式、架構洞察。這些會跨 session 保留。
~/.claude/projects/ stores session transcripts and auto-memory per project. Claude Code automatically saves notes to itself as it works: commands it discovers, patterns it observes, architecture insights. These persist across sessions.

🛠️ 漸進式導入策略
Step 1: 執行 /init，自動產生 CLAUDE.md。精簡到必要內容。
Step 2: 新增 settings.json，設定 allow/deny 規則。
Step 3: 為最常用工作流建立 1-2 個 commands。
Step 4: CLAUDE.md 太擁擠時，拆分到 rules/。
Step 5: 加入 ~/.claude/CLAUDE.md 個人偏好。

這就是 95% 專案需要的一切。Skills 和 Agents 是當你有需要打包的複雜工作流時才用。
⚠️ 重點提示 — 三層權限模型
Allow = 自動執行（白名單）
Deny = 完全封鎖（黑名單）
Neither = 詢問後執行（灰名單）

這跟 GMP 環境的電子記錄權限管理很像：有些操作免簽核（查看記錄）、有些永遠禁止（繞過驗證）、有些需要主管核准（修改批次記錄）。
📚 核心概念 — Auto-Memory
~/.claude/projects/ 裡的自動記憶是 Claude Code 的「長期記憶」。它不需要你手動設定 — Claude 會在工作過程中自動學習你的程式碼庫模式。你可以用 /memory 指令瀏覽和編輯這些記憶。 
{
  "$schema": "https://json.schemastore.org/
    claude-code-settings.json",
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Read",
      "Write",
      "Edit"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(curl *)",
      "Read(./.env)",
      "Read(./.env.*)"
    ]
  }
}

Key Takeaways

📝
CLAUDE.md 是最高槓桿檔案
它是 system prompt 的直接注入點。保持在 200 行以內，寫具體指令而非理論解釋。先把這個搞對，其他一切都是優化。
🔀
Commands / Skills / Agents 三層擴充
Commands 是手動觸發的斜線指令；Skills 是自動偵測並執行的工作流套件；Agents 是在獨立 context window 中工作的專家子代理。三者適用於不同的複雜度層級。
🔐
三層權限模型
settings.json 的 allow/deny/neither 模型提供精細的安全控制。白名單自動執行、黑名單完全封鎖、灰名單詢問確認。最小權限原則是核心設計。
🏗️
Project vs Global 分層設計
團隊標準放 project-level（git 提交），個人偏好放 global 或 .local 檔案（gitignored）。這個設計讓團隊一致性和個人靈活性可以共存。

Practice Questions

Q1 概念理解 CLAUDE.md、rules/ 資料夾中的規則檔、和 CLAUDE.local.md 三者之間的載入優先順序和可見範圍有何不同？

CLAUDE.md（專案根目錄）：每個 session 都載入，提交到 git，團隊共用。這是基礎層。

rules/ 中的檔案：也自動載入，但可以用 YAML frontmatter 的 paths 欄位限定只在特定路徑工作時才生效。提交到 git。

CLAUDE.local.md：自動 gitignored，僅限個人使用。它的內容會和 CLAUDE.md 合併，但不會影響團隊其他成員。

Claude 會讀取所有這些檔案並合併它們。如果有衝突，更具體的規則（如路徑限定的 rules）通常會優先。

Q2 產業應用 在一個需要 GMP 合規的 CDMO 軟體專案中，你會如何設計 settings.json 的 deny 清單？

GMP 環境的 deny 清單應該比一般專案更嚴格：

必須封鎖：任何直接修改生產資料庫的指令（如 Bash(psql *)）、存取驗證金鑰和密碼的操作（Read(./.env*)、Read(./secrets/*)）、破壞性操作（Bash(rm *)、Bash(drop *)）、繞過測試的部署指令。

建議封鎖：直接網路存取（Bash(curl *)、Bash(wget *)），因為 GMP 環境中不應該有未經審核的外部資料流入。

這體現了 21 CFR Part 11 的權限控制原則 — 系統應該限制操作到授權範圍內，並記錄所有操作。

Q3 批判思考 作者說「Skills watch the conversation and act when the moment is right」。從 Copilot vs Agent Mode 的角度，這種自動觸發在 GMP 環境中有什麼潛在風險？

Skills 的自動觸發機制本質上是 Agent Mode — AI 自主判斷何時介入。在一般軟體開發中這很有效率，但在 GMP 環境中，這違反了「copilot mode」的基本原則。

風險一：Skills 可能在不適當的時機觸發，對受監管的程式碼做出未經人類審核的變更。

風險二：觸發條件是基於 description 的語意匹配，而非精確規則。「大致正確」在 GMP 環境中是不夠的。

緩解策略：在 GMP 專案中，考慮只使用 Commands（手動觸發）而非 Skills（自動觸發），確保每個操作都有明確的人類決策點。如果使用 Skills，應限制其 allowed-tools 為唯讀操作（Read, Grep, Glob）。
