Part I — 選擇你的產品導師 (Choose Your Product Mentors)
一位新創公司創辦人透露，她使用客製化的 AI 版 Steve Jobs 來協助品牌策略決策。她用產品發表會逐字稿和策略演講訓練這個 GPT，讓它在她卡關時批判她的想法，並協助精煉到「10 分滿分」的概念。
A startup founder recently revealed that they use a custom built AI version of Steve Jobs to help guide their decision making... trained it on product launch transcripts and strategy talks.
作者提出一個更進階的構想：何不建立一整個世界級產品思考者團隊，跨越策略、設計、決策、AI 等 19 個專業維度，在需要時隨時諮詢？文章將這些維度歸類為 7 大領域：Strategy & Vision、Discovery & Research、Execution & Delivery、Design & Craft、People & Communication、Data & Measurement、AI Product Sense。
What if you could build an entire team of world class product thinkers and refer to them to help you to assess your designs, make decisions or craft your strategy?
💡 比喻說明
把這個系統想像成你的「私人董事會」 — 企業 CEO 都有外部董事提供不同觀點的挑戰與建議。這套系統就是用 AI 模擬這個概念，只是你的董事成員是根據公開資料重建的「思維模型」，而非真人。關鍵差異：真正的董事會有利益關係和政治動態，AI 版只有思維框架。
⚠️ 重點提示
文章的導師選擇框架包含 19 個技能維度，但真正的價值不在人選本身，而在「角色文件」(Persona Document) 的設計品質。一個精心設計的 5-6 頁角色文件，涵蓋 10 個維度（Core Identity, Core Principles, Questioning Style, Communication Style, Red Flags, Mentoring Process, Example Interactions, Key Phrases, Success Metrics, Limitations），才是這套系統運作的核心。
Part II — 在 Claude Code 中建構導師系統 (Build the Mentor System)
系統建構的第一步是建立清楚的目錄結構。作者設計了四個主要資料夾：mentors/（按專長分類的角色檔）、context/（公司和產品背景資訊）、sessions/（對話紀錄）、decisions/（重要決策存檔）。mentors 資料夾再細分為 strategic-vision、discovery、execution、design、people、data、specialized 等子目錄。
The mentors folder is categorized according to each mentor's specialist area, context is where you'll provide Claude some additional information about your product, sessions is where you'll store each decision making session.
接著，將事先用 Claude Web 建立好的角色檔（如 jobs-vision.md）儲存到對應目錄。最後建立 context/company-info.md 和 context/product-roadmap.md，提供公司使命、產品描述、目標客群、團隊規模、當前挑戰等背景資訊，讓導師的建議能基於你的實際情境。
If you want to give Claude and your mentors some additional context about your company, your product roadmap, strategy or anything else, you can add this into your context folders.

📚 核心概念：Context Grounding
這套系統的設計模式是 Persona + Context = Grounded Advice。角色檔定義「誰在說話」，context 檔定義「說給誰聽」。沒有 context，導師的建議只是通用哲學；有了你的公司資訊和 roadmap，建議才會具體到你的產品和市場。這與 RAG 的概念類似：用外部文件為 LLM 的回應提供基礎。
Part III — 打造互動前端介面 (Turn Mentors into a Chat App)
為了脫離命令列的限制，作者將導師系統封裝成一個本地端 Web 應用程式。技術架構是 Node.js + Express 後端、SSE (Server-Sent Events) 串流、Vanilla JS 前端，並使用 Anthropic Claude API 進行對話。前端包含深色側邊欄（導師卡片 + 歷史 session）和淺色聊天區域。
If you wanted to take things a step further, you could turn this into a web application that you can access with a front end to make it easier to switch between mentors.
關鍵設計：System Prompt 由導師 .md 檔 + 公司 context + 產品 roadmap 三者組合而成。這意味著每次對話都會載入完整的角色設定和你的業務背景，確保回應既符合導師風格又切合你的實際需求。Session 以帶 YAML frontmatter 的 Markdown 格式儲存在 sessions/ 目錄。
System prompt: mentor .md file + company context + product roadmap. Sessions saved as markdown with YAML frontmatter in ../sessions/

💡 比喻說明：System Prompt 三明治
想像 System Prompt 就像一個三層三明治：最上層是「導師人格」（他怎麼想），中間層是「你的公司」（他在幫誰），底層是「你的 roadmap」（他在幫什麼決策）。缺少任何一層，對話都會流於空洞。這個模式其實就是 Context Engineering 的實踐 — 不是寫更好的 prompt，而是提供更完整的 context。
Part IV — Agents、Skills 與 MCP 整合 (Extending with Real-World Data)
作者提出三種擴充機制。Skills（自訂斜線指令）：/mentor <name> 快速啟動導師 session，/decide <question> 讓所有導師同時回答同一問題，/prep <topic> 根據 roadmap 和歷史決策做會前準備。
Skills let you create custom commands that make interacting with your mentors faster and more intuitive.
Agents（平行與背景執行）：最強大的用法是「Panel Discussion」模式 — 同時啟動三個 Agent，每個讀取不同的導師檔，獨立回答同一個策略問題，最後合成比較分析。這模擬了真正董事會中不同觀點的碰撞。
One of the most powerful approaches would be creating a panel discussion where you'd spawn three agents in parallel, one per mentor, all answering the same strategic question.
MCP（Model Context Protocol）整合是讓系統「接地」的關鍵。連接 Linear/Jira 可以拉入真實的 feature request 和 bug report；連接 Analytics 可以讓導師參考實際使用數據。文章列出了 Linear、Figma、Notion、Jira、Google Analytics 五個 MCP 整合文件連結。
MCP integrations would let you connect your mentor system to real-world data and tools, transforming it from a theoretical exercise into something grounded in your actual product work.
⚖️ 三種擴充機制比較
機制 用途 觸發方式
Skills 快速指令、可複用流程 /mentor /decide
Agents 平行思考、背景處理 Claude Code 自動協調
MCP 真實數據接入 第三方 API 整合
🛠️ 實務應用：CDMO 場景延伸
這套架構可以直接轉化為「法規導師系統」：建立 PIC/S GMP Inspector、FDA Reviewer、EMA Assessor 等角色檔，搭配你的 SOP 和 deviation history 作為 context。用 /decide Skill 讓三個法規觀點同時評估一個 deviation 的分類決策。MCP 連接 LIMS/MES 後，導師的建議就能參考實際批次數據。

注意：法規情境中，AI 輸出永遠是 copilot 模式（100% 人工審核），而非 agent 模式。
⚠️ 重點提示：MCP 可用連接
文章列出的 5 個 MCP 整合文件：Linear (linear.app/docs/mcp)、Figma (developers.figma.com)、Notion (developers.notion.com/docs/mcp)、Jira (support.atlassian.com)、Google Analytics (developers.google.com/analytics/devguides/MCP)。你目前在 Claude.ai 已連接 Asana 和 Notion 的 MCP，可直接嘗試整合。
Key Takeaways
Persona = Markdown 檔
整套系統的核心是精心設計的 Markdown 角色文件（10 個維度、5-6 頁），而非 Claude Code 本身。工具只是載體，角色檔品質決定輸出品質。
Context Engineering 實踐
System Prompt = Persona + Company Context + Roadmap 的三層結構，體現了「context 比 prompt 更重要」的原則。小模型 + 完整 context 勝過大模型 + 空白 context。
Skills / Agents / MCP 三件套
Claude Code 的擴充能力分三層：Skills 做可複用指令、Agents 做平行思考、MCP 做數據接入。Panel Discussion 模式是最實用的 Agent 應用模式之一。
Non-Engineer 也能用 Claude Code
這篇文章的目標讀者是非工程師的 PM/產品人。核心操作只有：建目錄、存 .md 檔、簡單 prompt。前端可以靠 vibe coding 完成。
Practice Questions
Q1 概念理解 文章中的 Persona Document 包含 10 個維度。如果你只能保留其中 3 個維度來建立一個有效的 AI 導師，你會選哪 3 個？為什麼？

建議答案：Core Principles、Questioning Style、Limitations。

Core Principles 定義了導師「相信什麼」，這是所有建議的價值觀基礎。Questioning Style 決定了導師「如何挑戰你」，這是區分 AI 導師和普通 ChatGPT 對話的關鍵差異 — 好的導師不是給答案，而是問出你沒想到的問題。Limitations 讓 AI 知道何時說「我不知道」，避免過度自信的幻覺。其他維度如 Communication Style 和 Key Phrases 只是語氣裝飾，Example Interactions 可以被好的 few-shot prompt 替代。

Q2 產業應用 如果你要為 CDMO 的 Aseptic Fill-Finish 產線建立一套「法規導師系統」，你會選擇哪 3 位「導師」角色？Context 資料夾應該放什麼文件？

建議的 3 位導師：

1. PIC/S GMP Inspector — 專注製程驗證、環境監控和數據完整性，用 PIC/S Annex 1 和 PE 009 的邏輯思考。2. FDA CDER Reviewer — 專注 21 CFR Part 211/Part 11 合規、Deviation 嚴重性分類和 CAPA 有效性評估。3. QA Release Director — 模擬一位有 20 年經驗的品保放行主管，專注批次放行決策的風險平衡。

Context 資料夾建議內容：context/site-info.md（廠區資訊、設備清單、SA25 Vanrx isolator 規格）、context/product-matrix.md（產品清單、劑型、灌裝量）、context/regulatory-history.md（歷次稽核發現、主要 deviation 趨勢）、context/sop-index.md（關鍵 SOP 的摘要索引）。

Q3 批判思考 這套「AI 導師系統」最大的風險是什麼？在什麼情況下，它可能造成比沒有它更差的決策品質？

最大風險：錨定效應 (Anchoring Bias) 和虛假權威感。

當系統以「Steve Jobs 會這樣想」的方式包裝建議時，使用者很容易被名人光環錨定，降低自己的批判思考。但實際上，這只是基於公開資料的 LLM 角色扮演，不是真正的 Jobs 思維。更危險的場景是：當使用者面對模糊決策時，AI 導師給出看似確信的回答（因為 LLM 本質上傾向產生流暢、自信的文本），使用者可能因此跳過本應進行的深度分析和利害關係人討論。

在法規環境中風險更高 — 如果一個「FDA Reviewer 導師」告訴你 deviation 不需要升級，但你的 QA 判斷認為應該升級，選擇「聽 AI 的」可能導致合規風險。這就是為什麼法規情境中必須堅持 copilot 模式。

🎯 我的行動結論
✔ 核心洞察
這篇文章的技術含量不高，但它展示了一個重要的模式：Claude Code 的 Markdown 角色檔 + 目錄結構 + MCP 可以快速建構「情境化的 AI 顧問系統」。對我來說，真正的啟發不是「AI 導師」這個概念，而是 Persona + Context Grounding 的架構設計可以直接遷移到 CDMO 的法規決策支援場景。這與我一直在思考的「copilot vs agent 模式選擇」直接相關。
📈 立即應用
本週可以做的：在現有的 Claude Project 中建立一個「PIC/S Inspector」角色檔（使用文章的 10 維度框架），搭配我們的 site info 和 deviation trend 作為 context，測試它在最近一個 deviation 分類決策上的表現。不需要 Claude Code，直接用 Claude Project 的 Knowledge 功能就能實驗這個模式。
⚠ 需要補充
文章完全沒有討論 prompt 品質的驗證方法 — 你怎麼知道你的「Steve Jobs 角色檔」是否真的模擬了有價值的思維模式，而不只是表面的語氣模仿？在 CDMO 場景中，這變成：你怎麼驗證「FDA Reviewer 導師」的建議是否符合最新法規要求？這需要一套 evaluation framework，文章沒有觸及。下次閱讀 Anthropic 的 prompt evaluation 文件時，應該把這個 gap 補上。
