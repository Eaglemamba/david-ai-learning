Part I: OpenClaw 的三天踩坑紀實

Original Content (Bilingual)

作者嘗試了當時爆紅的 OpenClaw（前身為 Clawdbot/Moltbot），這是由奧地利開發者 Peter Steinberger 在 2025 年 11 月發布的開源個人 AI 代理框架。它讓使用者透過 WhatsApp、Telegram 等聊天平台操控 AI 執行真實任務：管理郵件、執行腳本、控制瀏覽器、排程自動化。
OpenClaw is an open-source, self-hosted AI agent that runs locally and executes actions on the user's behalf, interfacing through popular messaging applications.
Day 1 - 搭環境：Docker、各種依賴，裝到一半報錯，查了半小時原來是版本問題。
"Docker, various dependencies, half-way through installation it errored out, spent half an hour debugging only to find it was a version issue."
Day 2 - 配 API 和權限：一個平台一套邏輯，手動對接，改完這個壞那個。
"Each platform has its own logic, manual integration, fix one thing and break another."
Day 3 - 升級即崩潰：剛以為跑起來了，升級了一下，服務直接癱掉。Token 消耗也超出預期好幾倍。
"Just when I thought it was working, I upgraded and the service completely crashed. Token consumption also exceeded my expectations by several times."
最後看到 Cisco 出了報告說有第三方插件在做資料竊取，直接刪了。
"Saw Cisco's report about third-party plugins doing data exfiltration, and deleted it outright."

Analysis & Supplementary Context

⚠️ 重點提示：OpenClaw 安全事件時間線

作者提到的「Cisco 報告」遠比他描述的嚴重。根據 Cisco AI Threat & Security Research 團隊的完整評估：

1. 資料外洩實測：Cisco 測試了一個名為「What Would Elon Do?」的第三方 skill，發現它會靜默執行 curl 指令將使用者資料發送到外部伺服器，且完全不通知使用者。

2. 漏洞規模：安全研究員發現超過 42,000 個公開暴露的 OpenClaw 實例，洩漏 API 金鑰、對話記錄和帳號憑證。

3. 供應鏈攻擊：ClawHub（OpenClaw 的 skill 市場）中約 12% 的 skills 被發現是惡意的（341/2,857），代號「ClawHavoc」事件。

💡 比喻說明：Simon Willison 的「致命三角」

安全專家 Simon Willison（提出 prompt injection 一詞的人）定義了 AI Agent 的「致命三角」(Lethal Trifecta)：

1. 存取私人資料（郵件、文件）+ 2. 接觸不受信任的內容（第三方 skills）+ 3. 對外通訊能力（發送訊息、HTTP 請求）

OpenClaw 三者皆具，且預設信任 localhost，所有連線看起來都像本機流量。這就像一個 GMP 製造環境裡，品質管制 (QC) 直接授權給外包商自行放行，沒有獨立覆核機制。

📚 核心概念：Security by Default vs Security by Option

OpenClaw 官方文件自己承認：「There is no 'perfectly secure' setup.」安全是選配而非預設。這在受監管產業（如製藥）是根本性的架構問題——GMP 的核心原則是 built-in quality，不是 tested-in quality。同理，安全應該是 built-in security，不是 bolted-on security。

Part II: Claude Code Channels 架構解析

Original Content (Bilingual)

Claude Code 出了新功能叫 Channels。原理：Telegram 或 Discord 發消息 → Claude Code 在電腦上執行 → 結果發回手機。
"Claude Code released a new feature called Channels. Principle: send message via Telegram or Discord → Claude Code executes on your computer → results sent back to your phone."
不用自己搭伺服器。不用折騰第三方平台。Claude Code 裝好就能用。
"No need to set up your own server. No need to fuss with third-party platforms. Install Claude Code and you're ready."
安全邊界比較克制：只操作當前 project 目錄，不接管整台機器。配對以後還有發送者白名單，只有你的帳號能推訊息進來，其他人發了直接靜默丟棄。
"Security boundaries are more restrained: only operates within current project directory, doesn't take over the whole machine. After pairing, there's a sender allowlist — only your account can push messages, others are silently discarded."
第一次真正用是在咖啡館。想起項目裡有個腳本沒跑，拿出手機在 Telegram 發了一條：「幫我跑一下 run.py」。等了一會兒，有返回了。日誌、輸出、完成狀態，全在手機上。
"First real use was in a cafe. Remembered a script hadn't been run, pulled out my phone and sent via Telegram: 'run run.py for me.' Waited a bit. Got a response. Logs, output, completion status, all on my phone."

Technical Architecture Deep Dive

📚 核心概念：Channel = MCP Server + Event Push

Channel 本質上是一個 MCP (Model Context Protocol) 伺服器，以事件推送模式運作。與傳統的 request-response 不同，Channel 會在你的 Claude Code session 中「注入」外部事件。

技術流程：
1. 你安裝 Channel plugin（如 Telegram）→ plugin 作為 MCP server 運行
2. 啟動 Claude Code 時加上 --channels flag → 啟動 plugin
3. MCP server 連接到 Telegram（polling 模式）或 Discord（WebSocket 模式）
4. 收到訊息 → 包裝為 <channel> event → 推入 Claude Code session
5. Claude 用本機環境處理 → 透過 MCP server 提供的 reply tool 回覆

⚖️ 工具比較：Channels vs Remote Control vs OpenClaw

存取範圍 當前 project dir 當前 project dir 整台機器 + 100+ 服務
認證模型 Pairing + Allowlist claude.ai 帳號 預設信任 localhost

🛠️ 實務應用：何時用哪個？

Channels：你已經用 Claude Code，想從手機非同步觸發任務、接收結果通知。偏好可客製化的系統。

Remote Control：你想要 claude.ai 的完整介面（rich formatting、file preview），不想額外建 bot。設定最簡單。

OpenClaw：你需要跨 100+ 服務的深度整合（WhatsApp、iMessage、日曆、瀏覽器），願意自己承擔安全風險。

Part III: Telegram & Discord 設定實戰

Telegram 接入（5 步驟）

Step 1 - 建機器人：Telegram 找 BotFather，發 /newbot，使用者名稱以 bot 結尾，拿 token，複製好。
"Find BotFather in Telegram, send /newbot, username ending in bot, get token, copy it."
Step 2 - Claude Code 裝插件：
/plugin install telegram@claude-plugins-official
/reload-plugins
# 輸入 /telegram: 能自動補全 configure 就成功了
Step 3 - 填 Token：
/telegram:configure 你的token
Step 4 - 重啟，帶 channels 參數：
claude --channels plugin:telegram@claude-plugins-official --dangerously-skip-permissions
Step 5 - 配對 + 鎖白名單：給機器人隨便發條訊息，它會回一個配對碼。然後依序執行（順序不能反）：
/telegram:access pair <配對碼>
/telegram:access policy allowlist
⚠️ 重點提示：踩坑備忘

坑一：Bun 環境變數未載入。安裝 Bun 後必須執行 source ~/.zshrc，否則 Telegram MCP 服務起不來。報錯訊息不會直接告訴你原因。

坑二：多終端搶佔。Claude Code 預設使用最後一個終端。專門留一個乾淨的終端跑 channels，不要和其他任務混用。

坑三：Permission prompt 會卡死遠端 session。不加 --dangerously-skip-permissions，Claude Code 遇到需要確認的操作會暫停等待本機點擊。你不在電腦旁就一直卡著。

🛠️ 實務應用：安全取捨建議

--dangerously-skip-permissions 這個 flag 名稱本身就是警告。只在你信任的機器、你信任的 project 目錄中使用。MacStories 的實測確認，即使跳過權限，Channels 的操作範圍仍限於當前 project directory，不會自動存取整台機器的檔案系統。

替代方案：搭配 tmux 或 screen 讓 session 在背景持續運行，方便隨時從手機觸發任務。

Discord 接入（7 步驟）

Discord 步驟多幾個，邏輯一樣：
Step 1-2：Discord Developer Portal → New Application → Bot 頁面 → Reset Token → 複製 token。
Step 3：Bot 設定裡找 Privileged Gateway Intents，打開 Message Content Intent——這個不開，機器人看不到訊息內容。
"In Bot settings, find Privileged Gateway Intents and enable Message Content Intent — without this, the bot can't see message content."
Step 4：OAuth2 → URL Generator，Scope 選 bot，權限勾上：View Channels, Send Messages, Send Messages in Threads, Read Message History, Attach Files, Add Reactions。用生成的連結把機器人加進伺服器。
Step 5-7：安裝、設定、啟動、配對、鎖白名單。
/plugin install discord@claude-plugins-official
/discord:configure 你的token
claude --channels plugin:discord@claude-plugins-official --dangerously-skip-permissions

# 私訊機器人，拿配對碼，然後：
/discord:access pair <配對碼>
/discord:access policy allowlist
📚 核心概念：Telegram vs Discord 技術差異

Telegram：使用 polling 模式（定期輪詢新訊息），設定較簡單，適合個人 1:1 場景。照片會被壓縮，需要原檔要以「文件」方式發送。不支援語音訊息。

Discord：使用 WebSocket 即時連線，設定多了 Intent 和 OAuth 權限步驟，但支援 guild channel（團隊協作場景）。

兩者共通限制：session 必須保持開啟狀態，訊息才能進來。關掉 Claude Code，channel 就斷了。

Critical Analysis

⚠️ 重點提示：研究預覽的限制

Channels 目前仍是 Research Preview，有幾個實際限制需要注意：

1. Plugin 白名單：只能使用 Anthropic 官方維護的 claude-plugins-official 中的 plugin。自建 channel 需要用 --dangerously-load-development-channels flag。

2. 無持久化：session 關閉就斷線。要 always-on 需要搭配 tmux、screen 或 VPS 背景 process。

3. 無語音：Telegram plugin 不支援語音訊息。照片有壓縮。

4. 認證限制：必須用 claude.ai 帳號，Console 和 API key 認證不支援。

💡 比喻說明：Bounded Autonomy 與 GMP 思維

Claude Code Channels 的安全設計體現了「有界自主」(Bounded Autonomy)原則：

想像一個藥廠的生產線。OpenClaw 像是給外包商一把萬能鑰匙，能開所有門、存取所有系統。Claude Code Channels 則像是給核准人員一張限定區域的門禁卡——只能進入指定的生產房間（project directory），必須經過身份驗證（pairing + allowlist），而且所有操作都在既定框架內執行。

這正是Copilot Mode 在 Agent 時代的實踐：讓 AI 在明確的邊界內自主工作，同時保持人類的監督權。

💰 成本分析：隱性成本比較

OpenClaw：免費開源，但隱性成本巨大——設定時間（作者花了 3 天）、API token 消耗超預期、安全風險的潛在損失、維護和升級的持續投入。

Claude Code Channels：需要 Claude Code 訂閱（Max plan 或 API），但設定時間約 15-30 分鐘，安全模型內建，plugin 由 Anthropic 官方維護。Trade-off 是 vendor lock-in 和功能受限於 Anthropic 的 roadmap。

Key Takeaways

1. 安全邊界是架構決策，不是附加功能

OpenClaw 的「Security by Option」vs Claude Code 的「Security by Design」不只是工具差異，而是根本的設計哲學差異。Cisco 實測證明，開放式 skill 生態加上預設信任 localhost，等同於主動邀請供應鏈攻擊。

2. MCP 正在成為 AI Agent 的通用連接層

Channels 建立在 MCP 之上，這代表 Anthropic 正在用同一套協議統一 tool use、integration、和 remote messaging。理解 MCP 的事件推送模型，對未來的 AI 工具生態至關重要。

3. 「手機遙控 AI 工作」是真實需求

OpenClaw 180,000+ GitHub stars 證明這個需求的強度。Channels 提供了一個更安全但功能較受限的替代方案。市場會持續在「功能廣度」和「安全深度」之間拉鋸。

4. Research Preview = 搶先卡位的機會

Channels 目前功能有限（只有 Telegram/Discord、需要 session 持續開啟），但架構是可擴展的。提前熟悉 Channel plugin 的開發模式，等 GA 時就能快速建立自訂整合。

Practice Questions

Q1 概念理解 為什麼 OpenClaw 的「預設信任 localhost」設計會導致安全問題？這和 Claude Code Channels 的 sender allowlist 有什麼根本差異？

OpenClaw 部署時通常使用 nginx 或 Caddy 作為反向代理，這代表所有外部連線都以 127.0.0.1 的身份進入，被視為「可信任的本機流量」。攻擊者只需要找到暴露的端口就能直接存取系統。

Claude Code Channels 的 sender allowlist 則採用身份驗證層的方式：先透過 pairing 確認使用者 ID，再將 policy 切換為 allowlist 模式。未經配對的發送者訊息會被靜默丟棄，連錯誤回應都不會返回（避免資訊洩漏）。

根本差異在於：OpenClaw 信任「來源位置」（IP），Channels 信任「來源身份」（User ID）。在網路安全領域，位置可以偽造，身份更難偽造。

Q2 產業應用 在 GMP 環境中，如果要導入類似 Claude Code Channels 的遠端 AI 工具，你會設計什麼樣的 SOP 來確保合規性？

GMP 環境導入遠端 AI 工具的 SOP 應至少包含：

1. 風險評估 (Risk Assessment)：依 ICH Q9 品質風險管理原則，評估 AI 代理存取的資料範圍、操作權限、和潛在的偏離 (deviation) 風險。確認是 Copilot Mode（人類覆核）還是 Agent Mode（自主執行）。

2. 存取控制：類似 Channels 的 allowlist 機制，但需要加上 21 CFR Part 11 要求的電子簽章和審計追蹤。每個遠端指令應記錄 who, what, when, why。

3. 變更管控：任何 plugin 更新等同於電腦化系統變更 (Computerized System Change Control)，需要經過 IQ/OQ/PQ 驗證流程。

4. 偏離處理：預定義 AI 執行失敗或異常行為的偏離處理流程，包含通報層級、根因分析 (RCA)、和 CAPA 要求。

Q3 批判思考 作者刪掉 OpenClaw 的決策是否太草率？有沒有 OpenClaw 比 Channels 更適合的場景？

作者的決策對他的情境來說是合理的——他不是安全專家，也沒有需要深度跨平台整合的需求。但從更廣的視角看：

OpenClaw 更適合的場景：需要連接 iMessage、WhatsApp 等 Channels 不支援的平台；需要 AI 代理存取瀏覽器、日曆等系統層功能；有安全團隊能夠正確配置和監控；需要跨多個 LLM 供應商運行（OpenClaw 支援 Claude、GPT、DeepSeek）。

批判思考角度：OpenClaw 的安全問題很多是「部署問題」而非「架構問題」——正確配置的 OpenClaw 實例（網路隔離、skill 白名單、定期安全掃描）可以大幅降低風險。問題在於大多數使用者不會這樣做，就像大多數 Docker 容器也不會被正確配置一樣。

這也是「2-3x 折扣原則」的應用場景：OpenClaw 社群聲稱的便利性，實際體驗要打 2-3 倍的折扣，尤其是安全和穩定性方面。

🎯 我的行動結論
✔ 核心洞察
Claude Code Channels 驗證了一個重要架構模式：將 AI Agent 的遠端介面建立在 MCP 事件推送之上，而不是開放 API 端口。這個模式和我在 CDMO 環境中理解的安全原則一致——存取權限應該基於「身份 + 白名單」，而非「網路位置」。對我的 Claude Code headless pipeline 來說，Channels 可能是未來替代 cron + claude -p 的更優雅方案。
📈 立即應用
這週在 MacBook Air 上嘗試設定 Telegram Channel：(1) 確認 Claude Code 版本 >= v2.1.80；(2) 安裝 Bun runtime；(3) 走完 Telegram 5 步驟設定；(4) 測試從手機觸發一個簡單的文件處理任務。成功後評估是否能用來遠端觸發教育文件生成 pipeline。
