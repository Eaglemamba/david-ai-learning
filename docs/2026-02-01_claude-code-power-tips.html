這些技巧直接適用於任何需要大量程式碼開發或自動化工作的情境，特別是需要同時處理多個任務的專案管理者。

Part I: 並行工作 (Parallel Work) — 最大的生產力提升

同時開啟 3-5 個 Git Worktrees，每個運行獨立的 Claude 工作階段。這是團隊公認的最大生產力提升技巧。

"Spin up 3–5 git worktrees at once, each running its own Claude session in parallel. It's the single biggest productivity unlock."

有些人會為工作樹命名並設定 Shell Aliases（za, zb, zc），一鍵切換。也有人專門保留一個「分析」工作樹，只用來讀 logs 和跑 BigQuery。

"Some people also name their worktrees and set up shell aliases (za, zb, zc) so they can hop between them in one keystroke. Others have a dedicated 'analysis' worktree that's only for reading logs and running BigQuery."

比喻說明
想像你有 5 個助理同時工作，而不是讓一個助理依序處理 5 件事。這不是「多工」，而是真正的「並行」。
實務應用
CDMO 情境：可以同時讓一個 Claude 處理 SOP 文件更新、另一個分析偏差調查數據、第三個起草訓練材料。不需要等待任何一個完成再開始下一個。
Shell Alias 範例
# ~/.zshrc 或 ~/.bashrc
alias za='cd ~/projects/main-worktree-a && claude'
alias zb='cd ~/projects/main-worktree-b && claude'
alias zc='cd ~/projects/main-worktree-c && claude'
Part II: Plan Mode — 規劃優先，一次到位

每個複雜任務都先進入 Plan Mode。把精力投入在計畫上，讓 Claude 可以一次完成實作（1-shot）。

"Start every complex task in plan mode. Pour your energy into the plan so Claude can 1-shot the implementation."

有人讓一個 Claude 寫計畫，再開第二個 Claude 以 Staff Engineer 角度審查。另一個技巧：當事情出錯時，立刻切回 Plan Mode 重新規劃，不要硬推。

"One person has one Claude write the plan, then they spin up a second Claude to review it as a staff engineer... the moment something goes sideways, they switch back to plan mode and re-plan. Don't keep pushing."

驗證步驟也明確告訴 Claude 進入 Plan Mode，不只是 build 時使用。

"They also explicitly tell Claude to enter plan mode for verification steps, not just for the build."

核心概念
Plan Mode 的本質：將「思考」與「執行」分離。人類負責高層決策和品質把關，AI 負責執行細節。這符合 GMP 環境的 copilot model。
重點提示
當 Claude 開始「打轉」或產出品質下降時，這是切換回 Plan Mode 的信號。不要試圖用更多指令修正，而是重新規劃。
Prompt 範例
Enter plan mode. Before implementing:
1. Outline your approach
2. List potential risks
3. Define success criteria
4. Wait for my approval
Part III: CLAUDE.md — 讓 AI 自己寫規則

投資在你的 CLAUDE.md。每次修正錯誤後，結尾加上：「更新你的 CLAUDE.md，這樣你就不會再犯同樣的錯。」Claude 非常擅長為自己撰寫規則。

"Invest in your CLAUDE.md. After every correction, end with: 'Update your CLAUDE.md so you don't make that mistake again.' Claude is eerily good at writing rules for itself."

無情地編輯你的 CLAUDE.md。持續迭代直到 Claude 的錯誤率可測量地下降。

"Ruthlessly edit your CLAUDE.md over time. Keep iterating until Claude's mistake rate measurably drops."

有工程師讓 Claude 為每個任務/專案維護一個 notes 目錄，每次 PR 後更新，然後讓 CLAUDE.md 指向它。

"One engineer tells Claude to maintain a notes directory for every task/project, updated after every PR. They then point CLAUDE.md at it."

比喻說明
CLAUDE.md 就像 GMP 環境中的 SOP。不是一次寫完就不動，而是從每次偏差中學習、持續改版。差別在於：這個 SOP 是 AI 自己根據經驗撰寫的。
核心概念
這是「Context Engineering」的實踐。你不是在訓練模型，而是在塑造 Claude 讀取的情境。好的 CLAUDE.md 等於高品質的持久記憶。
Prompt 範例
Good work fixing that. Now update 
your CLAUDE.md with a rule to 
prevent this mistake in the future.

Be specific about:
- What went wrong
- How to detect it
- How to avoid it
Part IV: Skills 與 Slash Commands — 模組化你的工作流

建立自己的 Skills 並提交到 Git，在所有專案中重複使用。團隊建議：如果你一天做超過一次，就把它變成 Skill 或 Command。

"Create your own skills and commit them to git. Reuse across every project... If you do something more than once a day, turn it into a skill or command."

具體範例：建立 /techdebt 指令，在每次工作結束時執行，找出並消除重複程式碼。建立整合 7 天 Slack、GDrive、Asana、GitHub 的 context dump 指令。

"Build a /techdebt slash command and run it at the end of every session to find and kill duplicated code. Set up a slash command that syncs 7 days of Slack, GDrive, Asana, and GitHub into one context dump."

實務應用
CDMO 情境：建立 /deviation-report 指令，自動從 QMS 抓取偏差資料、分析根因模式、產生初步報告。或 /batch-review 整合批次記錄資料。
核心概念
Skills 是「可重用的 AI 能力模組」。與其每次從頭描述任務，不如封裝成可複用的指令。這與你現有的 Master Prompt 系統思路一致。
重點提示
Skills 提交到 Git 的好處：版本控制、團隊共享、跨專案使用。這是將個人效率轉化為組織能力的關鍵。
Part V: Bug Fixing — 自動化除錯流程

Claude 自己能修大部分 bugs。啟用 Slack MCP，貼上 bug thread 然後說「fix」。零上下文切換。

"Claude fixes most bugs by itself. Enable the Slack MCP, then paste a Slack bug thread into Claude and just say 'fix.' Zero context switching required."

或者直接說：「去修復失敗的 CI 測試。」不要微觀管理怎麼做。讓 Claude 看 Docker logs 來 troubleshoot 分散式系統，它的能力出人意料。

"Or, just say 'Go fix the failing CI tests.' Don't micromanage how. Point Claude at docker logs to troubleshoot distributed systems -- it's surprisingly capable at this."

比喻說明
不是「告訴醫生你要開什麼藥」，而是「描述症狀，讓醫生診斷」。給 Claude 問題的情境，讓它找出解決方案。
實務應用
整合 MCP 可以讓 Claude 直接存取 Slack、Gmail、Asana 等工具。這意味著可以建立端到端的問題解決流程：從發現問題到解決，最小化人工介入。
Part VI: 進階 Prompting 技巧

挑戰 Claude。說「嚴格審查這些變更，在我通過你的測試之前不要發 PR」。讓 Claude 成為你的 reviewer。

"Challenge Claude. Say 'Grill me on these changes and don't make a PR until I pass your test.' Make Claude be your reviewer."

或說「證明給我看這能用」，讓 Claude 比較 main 和 feature branch 之間的行為差異。

"Or, say 'Prove to me this works' and have Claude diff behavior between main and your feature branch."

一個平庸的修復後，說：「知道你現在知道的一切，重來一次，實作優雅的解決方案。」

"After a mediocre fix, say: 'Knowing everything you know now, scrap this and implement the elegant solution.'"

核心概念
這些 prompts 的共同點：提高 Claude 的標準。不是接受第一個輸出，而是推動它產出更好的版本。這需要你自己有足夠判斷力。
重點提示
「寫詳細的 specs，減少模糊性」—— 你越具體，輸出越好。這呼應了「Context is King」的原則。
Part VII: 環境設定、Subagents、資料分析與學習

環境設定：團隊愛用 Ghostty 終端機。用 /statusline 自訂狀態列顯示 context 使用量和 git branch。用語音輸入——說話速度是打字的 3 倍，prompts 會更詳細。

"Use voice dictation. You speak 3x faster than you type, and your prompts get way more detailed as a result. (hit fn x2 on macOS)"

Subagents：在任何請求後加上「use subagents」讓 Claude 投入更多運算。將個別任務分派給 subagents 以保持主 agent 的 context window 乾淨。

"Append 'use subagents' to any request where you want Claude to throw more compute at the problem. Offload individual tasks to subagents to keep your main agent's context window clean and focused."

資料分析：請 Claude Code 使用 bq CLI 即時拉取和分析指標。團隊有 BigQuery skill 在 codebase 中，每個人都用它直接在 Claude Code 跑分析查詢。Boris 本人已經超過 6 個月沒有寫過一行 SQL。

"Ask Claude Code to use the 'bq' CLI to pull and analyze metrics on the fly... Personally, I haven't written a line of SQL in 6+ months."

學習：在 /config 啟用「Explanatory」或「Learning」輸出風格，讓 Claude 解釋變更背後的「為什麼」。讓 Claude 生成視覺化 HTML 簡報來解釋不熟悉的程式碼。建立間隔重複學習 skill：你說明理解，Claude 問 follow-up 填補空白，儲存結果。

"Enable the 'Explanatory' or 'Learning' output style in /config to have Claude explain the *why* behind its changes. Build a spaced-repetition learning skill."

Practice Questions

Q1 概念理解 Boris 說「Pour your energy into the plan」，這背後的原理是什麼？為什麼規劃階段的投入能讓 Claude「1-shot」完成實作？

這涉及 LLM 的工作原理：Claude 一次生成一個 token，基於之前的 context。當你提供詳細、結構化的計畫時，你實際上是在「預設」Claude 的生成路徑。

好的計畫減少了 Claude 需要「推測」的空間，降低了歧義和錯誤方向的可能性。這就像給 GPS 精確的目的地地址，而非模糊的「去那個附近的地方」。

此外，Plan Mode 強迫你自己先想清楚，這本身就減少了後續修正的需求。

Q2 產業應用 在你的 CDMO 工作環境中，如何應用「讓 Claude 自己寫規則」的 CLAUDE.md 策略？考慮 GMP 合規性和文件控制的需求。

CDMO 環境的關鍵是：任何 AI 輔助產出都需要人工審核（copilot model）。CLAUDE.md 可以包含：

1. 產業術語規範：確保 Claude 使用正確的 GMP 術語（如「偏差」而非「錯誤」，「CAPA」而非「改進措施」）

2. 輸出格式規範：符合 21 CFR Part 11 的文件結構要求

3. 禁止事項：如「不要建議未經驗證的流程變更」、「不要假設設備規格」

但要注意：這不是讓 Claude 自動更新生產 SOP，而是在輔助工作（草稿、分析、學習）中建立一致性。最終文件仍需走正式變更控制流程。

Q3 批判思考 Boris 提到「Don't micromanage how」讓 Claude 自己解決問題。這個建議有什麼潛在風險？在什麼情況下你應該選擇更明確的指導？

「不要微觀管理」的前提是：(1) 你有能力審核輸出、(2) 錯誤成本可接受、(3) Claude 有足夠 context 做出好決策。

應該更明確指導的情況：

1. 合規敏感任務：涉及法規要求時，你需要確保 Claude 遵循特定路徑

2. 高成本錯誤：如果「try and fail」的成本很高（生產環境、外部客戶），先規劃再執行

3. 你無法審核的領域：如果輸出超出你的專業判斷能力，更明確的約束反而安全

這篇文章的情境是軟體開發，錯誤可以快速發現並回滾。不同領域需要調整策略。
