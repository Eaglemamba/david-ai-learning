Part I
本地實驗：同一段對話為何有時 30 秒有時 0.2 秒？
作者用 Ollama 在 Mac (Apple Silicon, 16GB) 上跑 Gemma 4 (8B 參數, 9.6GB)，餵入一篇 670 token 的文章再連續追問 5 個問題。結果發現：Turn 2 到 Turn 3，prompt 處理時間從 31 秒驟降到 0.25 秒——100 倍加速。
Turn 2 to Turn 3, prompt processing dropped from 31s to 0.25s — 100x speedup.
換成小模型 Qwen3.5 (0.8B, ~1GB) 做同樣測試，全程 200ms 上下波瀾不驚。沒有戲劇性的加速。
Qwen3.5 stayed around 200ms throughout — no dramatic speedup.
💡 比喻說明
想像你是 QC 實驗室的分析師。第一次做 HPLC 分析要準備標準品、校正儀器（30 分鐘）。如果接下來的樣品用同樣方法，儀器已校正好，直接上樣就行（0.5 分鐘）。這就是「快取命中」的概念——重複準備工作被省略了。模型越大（儀器越複雜），校正時間越長，快取省下的時間也越多。
⚠️ 重點提示

加速只發生在「消化輸入」階段，與「吐出回答」的生成速度無關。生成始終穩定在 13-20 tok/s。

Gemma 4 (4.5B active)Qwen3.5 (0.8B)

Part II
KV Cache 原理：注意力機制中的 QKV
Transformer 的核心公式：Attention(Q, K, V) = softmax(Q · K^T / sqrt(d)) · V。三個角色各司其職：Q (Query) 是當前新 token 的「我要找什麼」，每次不同，不能快取；K (Key) 是歷史 token 的「索引」，算完就固定；V (Value) 是歷史 token 的「具體內容」，算完也固定。
Q (Query) — "What am I looking for?" → different each time, can't cache. K (Key) — "What do I have?" → fixed once computed. V (Value) — "What's the content?" → fixed once computed.
KV Cache 就是把歷史 token 的 Key 和 Value 存起來。新 token 只需計算自己的 Q，然後查已有的 KV。這之所以可行，因為所有主流大模型都是 Decoder-only 架構——單向注意力，每個 token 只看前面的 token，前面 token 的 KV 算完就固定了。
This works because all mainstream LLMs are Decoder-only — unidirectional attention with causal masking.
📚 核心概念

因果遮罩 (Causal Mask)：T3 的 KV 永遠不變，新增 T4 不影響 T1/T2/T3。如果是雙向注意力 (BERT)，加一個新 token 會改變所有 token 的表示，快取全廢。這也是為什麼 BERT 做不了生成式 AI。

💡 CDMO 類比

把 KV Cache 想成 GMP 文件的版本控制。一份 Batch Record 前面 50 頁是固定的 SOP 內容（K/V），每批只改最後幾頁的實際數據（Q）。你不需要每批都重新審核前 50 頁——它們已經被驗證過了。Decoder-only 架構保證「前面的頁面不會因為後面新增內容而需要修改」，這正是快取能成立的前提。

⚠️ 快取是無損的

KV 從快取載入和現場計算的結果完全一致。生成結果（output token）不進 prompt 快取——因為每次生成內容不同（temperature > 0），存了也無法複用。但在下一輪對話中，上輪的生成結果被拼回 prompt，變成「輸入」的一部分，自然被快取覆蓋。

Part III
多輪對話的快取經濟學
很多人以為每輪對話都要「重新讀一遍」所有歷史，token 消耗是 N(N+1)/2 的二次增長。沒有快取確實如此。但有了快取，前綴匹配讓計費從二次增長變為近似線性增長。
Without cache: quadratic growth N(N+1)/2. With cache: approximately linear growth.
💰 成本分析

假設系統提示詞 20K tokens，每輪對話增加 ~1K tokens：

沒有快取：10 輪總計 ~255K tokens（全價）——二次增長，越來越貴。

有快取：10 輪總計 ~60K 等價 tokens——前綴以 1/10 價格讀取。省了 76%。

這就是為什麼「一個 session 持續對話」比「頻繁開新 session」省錢的根本原因。

🛠️ 實務應用

數字對比：

一個 session 持續 10 輪：1 次全價 + 9 次 1/10 = 1.9 份

每次開新 session：10 次全價 = 10 份

差了 5 倍。對 Pro/Max 訂閱使用者，這意味著同樣的方案能多做 3-5 倍的工作。

💡 CDMO 類比

就像 Campaign 生產：連續批次 (consecutive batches) 共用已驗證的設備狀態、環境監控基線和清潔驗證結果，每批只需確認增量變化。但如果每批都重新做 IQ/OQ/PQ，成本和時間都是倍數增長。「持續對話」= 連續批次，「開新 session」= 從頭做驗證。

Part IV
Claude Code 原始碼逆向：精密的快取工程
每次 API 呼叫，Claude Code 發送的不是一整塊 prompt，而是一個精心拼接的多層結構：
Each API call sends a carefully assembled multi-layer structure, not a monolithic prompt.
Block 1-2：計費歸因 + CLI 前綴（不快取）。Block 3：靜態指令（行為規則）→ global 快取，全球所有使用者共享。Block 4：動態內容（CLAUDE.md 等）→ org 層級快取。接著是 tools schema（session 內凍結），最後是 messages（最後一條放 cache_control 標記）。
Block 3: static instructions → global cache shared by ALL users. Block 4: dynamic content (CLAUDE.md) → org-level cache.
⚠️ 兩檔 TTL

預設 5 分鐘 — 所有使用者

延長 1 小時 — Pro/Max 訂閱使用者（未超額）、Anthropic 員工

原始碼 claude.ts:408-413 判斷條件：USER_TYPE 為 'ant' 或 isClaudeAISubscriber() 且未使用 overage。

💻 關鍵函式對照

getSystemPrompt() — 組裝系統提示詞

splitSysPromptPrefix() — 按 DYNAMIC_BOUNDARY 切分

buildSystemPromptBlocks() — 加 cache_control 標記

addCacheBreakpoints() — 在最後一條訊息上標記快取斷點

🛠️ 快取斷裂偵測

Claude Code 監控每次呼叫的 cache_read_input_tokens。如果比上次下降 >5% 且絕對值 >2000 tokens，判定為斷裂，並分析原因：系統提示詞變了？工具增減了？TTL 過期了？模型切了？

相比 Ollama 的「快取沒了你自己猜」，這是工程化的確定性方案。

Part V
快取像鏈條：斷在哪裡，後面全廢
快取是前綴匹配。理解這個就理解了一切：
Cache is prefix-matching. Understand this and you understand everything.
最貴：Block 3（全域靜態）失效 → 整個請求從頭算。中等：Block 4（CLAUDE.md）變了 → Block 3 還能複用。最省：只追加新訊息 → 前面全部複用。
Most expensive: Block 3 invalidation → full recalculation. Cheapest: append-only → everything reused.
⚠️ 切換模型 = 完全失效

Opus 和 Sonnet 的權重不同，KV 張量不能互用。切一次模型，50K tokens 的上下文要全價重算。在 TTL 內切回可能還能命中舊快取。下班前最後半小時，不要切模型。

📚 Sub-agent 無法共享快取

每個 sub-agent 是獨立 API 呼叫，和主執行緒有三個關鍵差異：

1. 工具集不同 — 主執行緒有全套工具，Explore agent 只有子集 → schema 不同 → 前綴不同

2. 訊息歷史獨立 — 和主執行緒無交集

3. 可能用不同模型 — sub-agent 可能用 Haiku/Sonnet，不同權重 = 零複用

每啟動一個 sub-agent 基本等於一次「迷你冷啟動」。所以 CLAUDE.md 裡寫「多用 agent 並行處理」要意識到每個 agent 都有獨立的快取開銷。

Part VI
保護你的快取：Claude Code 使用姿勢
🛠️ 保護快取（綠燈）

連續對話 — 前綴不變，增量快取，一個 session 持續對話

btw 共享 session — 可共享快取

CLAUDE.md 穩定 — 定期整理但不在工作到一半時整理

⚠️ 破壞快取（紅燈）

開新 session — 冷快取，~20K tokens 全價重算

改 CLAUDE.md — Block 4 起全部失效，配好就別動

加減 MCP 工具 — 工具 schema 變化 = 快取斷裂

切換模型 — 完全失效，按階段切，別頻繁切

/compact — 訊息歷史變了 = 斷裂，對話 >100K 時再用

發呆超過 TTL — 快取過期，1h 內說句話

💰 差異量化

系統提示詞 20K tokens，對話 10 輪：

一個 session 持續對話：1.9 份

每次開新 session：10 份

差了 5 倍。Pro/Max 訂閱使用者，同樣方案能多做 3-5 倍工作。

💡 CDMO 類比

GMP 環境中的等價概念：在無菌充填產線上，每次換品項 (changeover) 就像開新 session——要重新做環境監控基線、設備清潔驗證、無菌模擬。如果你能把同品項的訂單排成連續批次，中間只做簡易清潔 (minor changeover)，省下的驗證時間 = 省下的 token。「快取保護」= 「減少不必要的 changeover」。

Part VII
進階：Cache Keep-Alive 續命策略
Pro/Max 使用者的 TTL 是 1 小時。午飯吃 1.5 小時、開個冗長的會議，快取就過期了。但快取 TTL 在每次讀取時刷新——只要在過期前發一次匹配前綴的請求，快取就能無限續命。
Cache TTL refreshes on each read. Send a matching prefix request before expiry to keep the cache alive indefinitely.
🛠️ 方案設想

用 tmux 或 iTerm2 AppleScript，每 55 分鐘往 Claude Code 終端自動發一條輕量 prompt（如「我斷線了嗎？如果沒斷你只要簡單說 ok」）。這消耗極少 token，但保住了整個快取鏈。

注意：此方案為作者的設想，尚未驗證是否違反 Anthropic 的使用條款。自動化腳本向 API 發送請求仍會消耗少量額度。

Key Takeaways

KV Cache = 已計算的注意力快取

Decoder-only 架構的因果遮罩保證歷史 token 的 K/V 一旦算完就固定。模型越大，快取收益越大（Gemma4 達 148 倍加速）。

前綴匹配是快取的生命線

快取像鏈條，斷在哪裡後面全廢。Block 3 (全域) → Block 4 (CLAUDE.md) → tools → messages，任何一環改變都會導致後續失效。

持續對話 vs 開新 Session = 5 倍差距

10 輪對話中，持續 session 只需 1.9 份等價 token，開新 session 需要 10 份。理解這點就理解了「怎麼讓方案多撐 3-5 倍」。

Sub-agent 有獨立快取開銷

每個 sub-agent 工具集、訊息歷史、甚至模型都可能不同，幾乎無法複用主執行緒快取。合理使用 agent 要權衡快取代價。

Practice Questions
Q1 概念理解 為什麼 BERT 架構不能有效利用 KV Cache，而 GPT/Claude 可以？

BERT 使用雙向注意力，每個 token 同時看前後所有 token。加入新 token 後，所有已有 token 的 Key/Value 都會改變，快取全部作廢。GPT/Claude 使用 Decoder-only 的因果遮罩 (causal mask)，每個 token 只看前面的 token，因此前面 token 的 KV 算完就固定，新增 token 不影響它們。這是 KV Cache 能成立的結構性前提。

Q2 產業應用 在 CDMO 的 GMP 環境中，如果要用 Claude Code 建立一個偏差調查的自動化工具，你會如何設計 session 策略來最大化快取效益？

關鍵策略：按調查階段維持單一 session，不要每個小任務都開新對話。具體做法：

1. 在 session 開始時把 CLAUDE.md 配好（偏差調查的 SOP 模板、品質術語定義），之後不再修改 → 保護 Block 4 快取。

2. 固定 MCP 工具集（只啟用需要的工具：file read/write、grep），session 中途不加減 → 保護 tools schema 快取。

3. 避免在調查中途切換模型（全程用 Opus 或全程用 Sonnet）→ 避免 50K+ tokens 全價重算。

4. 如果調查需要暫停（等 QC 數據回來），設定自動 keep-alive 避免 TTL 過期。

5. 只在上下文真的超過 100K tokens 時才用 /compact，不要預防性地壓縮。

Q3 批判思考 作者聲稱快取可以讓方案「多撐 3-5 倍」。這個數字在什麼條件下成立，什麼時候會失效？

成立條件：系統提示詞佔比高（20K+）、對話輪次多（10+）、每輪增量小（~1K）、不觸發快取斷裂（不改 CLAUDE.md、不切模型、不加減工具）、TTL 內持續互動。這基本描述的是一個穩定的開發 session。

失效情境：(1) 短對話（只有 2-3 輪），快取還沒充分攤平寫入成本。(2) 頻繁修改 CLAUDE.md 或工具集，導致快取鏈反覆斷裂。(3) 對話內容本身很長（每輪 5K+ tokens），增量部分佔比高，快取比例下降。(4) 超過 TTL 未互動，快取過期重新全價計算。

套用 2-3x 折扣原則：作者的 3-5 倍是理想條件下的上限。實際使用中保守估計 2-3 倍較合理。但即使打折，「持續 session 比頻繁開新」仍是顯著省錢的策略。

🎯 我的行動結論
✔ 核心洞察
KV Cache 不是「自動發生的魔法」，而是一套需要使用者配合保護的前綴匹配機制。在 Claude Code 中，快取保護的核心原則就一句：別碰前綴，只在末尾追加。這和 GMP 中「驗證過的不要動，只在受控範圍內做變更」的思維完全一致。理解快取架構後，能把 Claude Code 的使用效率從「感覺很貴」提升到「合理控制」。
📈 立即應用
本週開始：(1) 在 Claude Code 工作時維持單一 session，不再每個小任務開新對話。(2) 把 CLAUDE.md 和 MCP 工具配置固定下來，session 期間不做修改。(3) 如果午休或開會超過 45 分鐘，回來前先發一條輕量訊息保活快取。(4) 記錄一週的 token 消耗變化，驗證快取保護策略的實際效果。
⚠️ 需要補充
文章基於原始碼逆向分析，但 Anthropic 可能隨時改變快取策略和 TTL 設定。作者的 keep-alive 自動化方案尚未驗證 ToS 合規性。此外，文章沒有討論 Claude.ai Web 介面的快取行為（是否與 Claude Code CLI 相同？），這對非開發者使用者也很重要。六個月後重讀時，應確認 TTL 參數和快取架構是否已更新。
