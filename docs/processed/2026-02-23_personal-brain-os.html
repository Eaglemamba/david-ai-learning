Executive Summary

Muratcan Koylan 厭倦了每次與 AI 對話都要重新介紹自己，於是打造了一套名為 Personal Brain OS 的檔案式個人作業系統。整個系統是一個 Git repository，包含 80+ 個 Markdown、YAML、JSONL 檔案，涵蓋他的語氣、品牌、目標、聯絡人、內容管線和失敗紀錄。核心架構原則是 Progressive Disclosure（漸進式揭露）：不一次餵給 AI 所有資料，而是透過三層路由（Routing → Module → Data）讓模型在每個任務只看到它需要的東西。這不是 Prompt Engineering，而是 Context Engineering ── 設計資訊架構，讓 AI 能做出正確決策。

Part I & II — 核心問題與模組化架構
大多數人認為 AI 助手的瓶頸在於 Prompting（提示方式）。寫一個更好的 Prompt，得到更好的回答。這對單次互動來說是對的，但當你想讓 AI 長期以你的身份運作時就完全行不通了。
"Most people think the bottleneck with AI assistants is prompting. Write a better prompt, get a better answer. That's true for single interactions. It falls apart when you want an AI to operate as you across dozens of tasks over weeks and months."
語言模型有有限的 Context Window，而且不是所有 Token 位置都同等重要。把所有資訊塞進 System Prompt 不只浪費，還會主動降低表現 ── 每個 Token 都在搶奪模型的注意力。我們的大腦也類似：別人花 15 分鐘簡報，你記得第一件事和最後一件事，中間模糊。語言模型有同樣的 U 型注意力曲線。
"Language models have a finite context window, and not all of it is created equal... dumping everything you know into a system prompt isn't just wasteful, it actively degrades performance. Every token you add competes for the model's attention."
解決方案：將個人 OS 拆成 11 個獨立模組。寫文章時，AI 載入語氣指南和品牌檔案；準備會議時，載入聯絡人資料庫和互動紀錄。模型永遠不會在內容任務中看到網路資料，也不會在會議任務中看到內容模板。
"Instead of writing one massive system prompt, I split Personal OS into 11 isolated modules. When I ask the AI to write a blog post, it loads my voice guide and brand files. When I ask it to prepare for a meeting, it loads my contact database and interaction history."
💡 比喻說明
想像你走進一間辦公室，秘書不會一次把所有文件都倒在你桌上。她會問：「今天要做什麼？」寫企劃書 → 拿品牌手冊；見客戶 → 拿客戶資料。Progressive Disclosure 就是讓 AI 有一個聰明的秘書。
📚 核心概念：Progressive Disclosure

三層載入架構：

Level 1 — Routing File（路由檔）
SKILL.md 永遠載入，告訴 AI「這是內容任務，載入品牌模組」或「這是網路任務，載入聯絡人」。

Level 2 — Module Instructions（模組指令）
如 CONTENT.md、OPERATIONS.md、NETWORK.md，每個 40-100 行，包含檔案清單、工作流程序列和行為規則。

Level 3 — Data Files（資料檔）
JSONL log、YAML config、研究文件。只在任務需要時載入。Agent 逐行讀取 JSONL，而非解析整個檔案。

關鍵數字：任何資訊最多兩跳（two hops）即可到達。

⚠️ Agent Instruction Hierarchy

Repository Level：CLAUDE.md — AI 首先讀取的 onboarding 文件，提供完整專案地圖。

Brain Level：AGENT.md — 7 條核心規則 + 決策表，將常見請求映射到精確動作序列。

Module Level：每個目錄的指令檔，包含領域特定的行為約束。

分層好處：可更新單一模組規則而不影響其他模組。

Part III — 檔案系統即記憶
最反直覺的決定之一：不用資料庫。不用向量儲存。不用任何檢索系統。只用磁碟上的檔案，用 Git 做版本控制。
"One of the most counterintuitive decisions I made: no database. No vector store. No retrieval system except Cursor or Claude Code's features. Just files on disk, versioned with Git."
每個檔案格式都有其選擇原因：JSONL 用於日誌（append-only、串流友好、每行獨立有效 JSON）、YAML 用於配置（層次清晰、支援註釋）、Markdown 用於敘事（LLM 原生讀取、到處可渲染、Git diff 乾淨）。
"JSONL for logs because it's append-only by design, stream-friendly... YAML for configuration because it handles hierarchical data cleanly, supports comments... Markdown for narrative because LLMs read it natively."
JSONL 的 append-only 特性防止了一類 bug：Agent 意外覆寫歷史資料。作者曾因 Agent 重寫 posts.jsonl 而丟失三個月的貼文互動數據。JSONL 下，Agent 只能新增行，刪除改用 "status": "archived" 標記，保留完整歷史。
"I lost three months of post engagement data early on because an agent rewrote posts.jsonl instead of appending to it. JSONL's append-only pattern isn't just a convention -- it's a safety mechanism."
💡 比喻說明
JSONL 像實驗室的 Logbook（實驗紀錄簿）：你只能往後寫，不能撕掉前面的頁面。即使寫錯了，也是畫線標註「作廢」而非銷毀。這在 GMP 環境中完全是同樣的原則 ── 數據完整性的根基就是「不可覆寫」。
📚 核心概念：Format-Function Mapping
格式用途Agent 安全性
JSONL日誌/紀錄（11 個檔案）
posts, contacts, interactions, bookmarks, ideas, metrics, experiences, decisions, failures, engagement, meetings		Append-only，防覆寫
YAML配置（6 個檔案）
goals, values, learning, circles, rhythms, heuristics		支援註釋，人機都可讀
Markdown敘事/指南（50+ 個檔案）

每個 JSONL 檔案開頭都有 schema 行：{"_schema": "contact", "_version": "1.0", "_description": "..."}，讓 Agent 讀資料前就知道結構。

⚠️ Episodic Memory（情節記憶）

多數「第二大腦」系統儲存事實。這個系統也儲存判斷：

experiences.jsonl — 關鍵時刻 + 情緒權重 (1-10)

decisions.jsonl — 決策 + 推理 + 備選方案 + 結果追蹤

failures.jsonl — 失敗 + 根因 + 預防措施

事實告訴 Agent 發生了什麼；情節記憶告訴 Agent 什麼重要、下次怎麼做。

Part IV — Skill System：教 AI 你的工作方式
檔案儲存知識，Skills 編碼流程。作者按照 Anthropic Agent Skills 標準建立結構化指令，告訴 AI 如何執行特定任務，並內建品質閘門（quality gates）。
"Files store knowledge. Skills encode process. I built Agent Skills following the Anthropic Agent Skills standard -- structured instructions that tell the AI how to perform specific tasks with quality gates baked in."
兩種 Skill 解決兩種問題。Reference Skills（如語氣指南）設定 user-invocable: false，Agent 寫作時自動靜默載入。Task Skills（如 /write-blog）設定 disable-model-invocation: true，只有使用者輸入 slash command 才觸發。
"Reference skills set user-invocable: false in their YAML frontmatter. The agent reads the description field and injects them automatically... Task skills set disable-model-invocation: true. The agent can't trigger them on its own."
當他輸入 /write-blog context engineering for marketing teams 時，五件事自動發生：語氣指南載入、反模式載入、部落格模板載入（7 段結構 + 字數目標）、受眾 persona 檢查、既有主題研究檢查。一個 slash command 觸發完整的上下文組裝。
"When I type /write-blog context engineering for marketing teams, five things happen automatically: the voice guide loads, the anti-patterns load, the blog template loads, the persona folder is checked, and the research folder is checked."
💡 比喻說明
Reference Skills 就像你實驗室的 SOP（標準作業程序）── 不需要每次操作前翻出來念一遍，它已經內化在操作員的習慣裡。Task Skills 則像 Batch Record（批次紀錄）── 只有在執行特定製程時才打開，按步驟逐一完成。
📚 核心概念：Voice System

語氣不是用形容詞描述的。作者用結構化數據定義：

五維度量表 (1-10)：
Formal/Casual: 6 | Serious/Playful: 4
Technical/Simple: 7 | Reserved/Expressive: 6
Humble/Confident: 7

Anti-patterns 檔案：
50+ 禁用詞（三級分類）、禁用開頭、結構陷阱（強制三分法、copula 迴避、過度 hedging）、每段落最多一個 em-dash。

4-pass 編輯流程：
1. 結構編輯（hook 夠吸引？）
2. 語氣編輯（禁用詞掃描）
3. 證據編輯（引用有來源？）
4. 朗讀測試

Part V — 日常運作：內容管線、CRM、自動化鏈
內容管線分 7 階段：Idea → Research → Outline → Draft → Edit → Publish → Promote。點子記錄到 ideas.jsonl，用 5 維度評分（定位對齊、獨特洞察、受眾需求、時效性、投入產出比），滿分 25，達到 15 分以上才推進。
"Seven stages: Idea, Research, Outline, Draft, Edit, Publish, Promote. Ideas are captured to ideas.jsonl with a scoring system -- each idea rated 1-5 on alignment with positioning, unique insight, audience need, timeliness, and effort-versus-impact. Proceed if total score hits 15 or higher."
個人 CRM 把聯絡人分成四個圈層，各有不同維護頻率：Inner（每週）、Active（雙週）、Network（每月）、Dormant（每季重啟）。每筆聯絡人有 can_help_with 和 you_can_help_with 欄位，交叉比對可自動發現互惠介紹機會。
"Contacts organized into four circles with different maintenance cadences: inner (weekly), active (bi-weekly), network (monthly), dormant (quarterly reactivation). Each contact record has can_help_with and you_can_help_with fields that enable the introduction matching system."
五個腳本處理週期性工作流程。週日 Weekly Review 依序執行：metrics_snapshot.py（更新數字）→ stale_contacts.py（標記需維護關係）→ weekly_review.py（產生摘要，含完成 vs 計畫、趨勢、下週優先事項）。Review 不只是報告 ── 它是下週規劃的起點。
"The Sunday weekly review runs three scripts in sequence... The review isn't a report -- it's the starting point for next week's planning. The automation creates a feedback loop: goals drive content, content drives metrics, metrics drive reviews, reviews drive goals."
🛠️ 實務應用：內容計分系統

每個點子用 5 維度評分（各 1-5 分）：

1. 定位對齊 — 符合個人品牌嗎？
2. 獨特洞察 — 我有別人沒有的角度嗎？
3. 受眾需求 — 目標讀者真的想知道嗎？
4. 時效性 — 現在發布有優勢嗎？
5. 投入產出比 — 花的力氣值得嗎？

門檻：總分 ≥ 15/25 才推進到 Research 階段。

Part VI — 踩坑教訓與核心原則
Schema 過度工程化。 初版 JSONL schema 每筆有 15+ 欄位，大多空白。Agent 遇到稀疏資料會嘗試填充或評論缺失。精簡到 8-10 個必要欄位，可選欄位只在有資料時才加。更簡單的 schema = 更好的 Agent 行為。
"My initial JSONL schemas had 15+ fields per entry. Most were empty. Agents struggle with sparse data -- they try to fill in fields or comment on the absence. I cut schemas to 8-10 essential fields."
語氣指南太長。 V1 的 tone-of-voice.md 有 1,200 行。Agent 前幾段寫得好，第四段就開始漂移 ── 語氣指令掉進了 Lost-in-the-Middle（中間遺失） 區域。重構後將最具辨識度的模式（簽名短語、禁用詞、開頭模式）放在前 100 行。
"Version one of tone-of-voice.md was 1,200 lines. The agent would start strong, then drift by paragraph four as the voice instructions fell into the lost-in-middle zone."
模組邊界比你想的更重要。 原本 identity 和 brand 在同一模組。Agent 只需要禁用詞列表卻載入整份個人簡介。拆開後，語氣相關任務的 Token 使用量減少 40%。每個模組邊界都是一個載入決策。
"I initially had identity and brand in one module. The agent would load my entire bio when it only needed my banned words list. Splitting them into two modules cut token usage for voice-only tasks by 40%."
核心原則。 這是 Context Engineering（上下文工程），不是 Prompt Engineering。Prompt Engineering 問「我怎麼把問題問得更好？」Context Engineering 問「AI 做出正確決策需要什麼資訊，以及如何結構化這些資訊讓模型真正使用它？」
"Prompt engineering asks 'how do I phrase this question better?' Context engineering asks 'what information does this AI need to make the right decision, and how do I structure that information so the model actually uses it?'"
⚠️ 五大踩坑教訓

1. Schema 過度工程化
15+ 欄位 → 8-10 必要欄位。稀疏資料讓 Agent 行為異常。

2. 語氣指南太長
1,200 行 → 關鍵規則前置 100 行。對抗 Lost-in-Middle 效應。

3. 模組邊界不對
identity + brand 合一 → 拆開省 40% Token。每個邊界都是載入決策。

4. Append-only 不可妥協
Agent 重寫 JSONL → 丟失三個月資料。JSONL 只能新增，不能覆寫。

5. 關鍵指令要放最前面
Token 位置影響回憶機率。最重要的規則放文件頂部。

Key Takeaways
🏗️ Progressive Disclosure 是核心架構
不要把所有資料塞進 System Prompt。用三層架構（Routing → Module → Data）讓 AI 每次只看到任務需要的資訊。任何資訊最多兩跳可達。模組隔離不只省 Token，更防止指令衝突。
📁 檔案格式即安全策略
JSONL 的 append-only 不只是慣例，是安全機制 ── 防止 Agent 覆寫歷史資料。YAML 的註釋功能讓人機都能讀配置。Markdown 是 LLM 原生語言。每個格式選擇都是一個架構決策。
🎯 Context Engineering > Prompt Engineering
Prompt Engineering 優化單次提問；Context Engineering 設計跨任務的資訊架構。前者像寫一封好 email，後者像建一套好的檔案系統。一次性 vs 永久性的差別。
⚡ Skill = 流程編碼 + 品質閘門
Reference Skills 自動靜默載入保一致性；Task Skills 手動觸發保精確性。每個 Skill 內建品質檢查點，Agent 自我評估後才呈現草稿。一個 slash command 觸發完整上下文組裝。
Practice Questions
Q1 概念理解 為什麼作者說「把所有資訊塞進 System Prompt 會主動降低表現」？這跟 Token 數量限制是不同的問題嗎？

是不同的問題。Token 數量限制是「裝不下」的硬性問題；但即使裝得下，也會有注意力稀釋的問題。語言模型有 U 型注意力曲線 ── 對開頭和結尾的 Token 記憶較好，中間段落容易「遺失」（Lost-in-the-Middle 效應）。每多加一段無關資訊，都在搶奪模型對真正重要指令的注意力。所以 Progressive Disclosure 的核心不只是省 Token，更是讓每個載入的 Token 都是高信號（high-signal）的。

Q2 產業應用 如果你要在 CDMO 環境中應用 Personal Brain OS 的架構，哪些模組會是第一優先？Agent 的 append-only 原則與 GMP 的數據完整性要求有何呼應？

第一優先模組：Deviation Investigation 模組（儲存歷史偏差類型、根因分析、CAPA 紀錄）和 Tech Transfer 模組（儲存製程參數、設備對照、驗證狀態）。這兩個領域每次都需要大量歷史脈絡，且每個案例都有獨特判斷需求。

Append-only 與 GMP 的呼應：JSONL 的 append-only 原則直接對應 ALCOA+ 中的「Original」和「Attributable」原則 ── 原始紀錄不可覆寫，修改必須保留稽核軌跡。"status": "archived" 的做法等同於 GMP 中的「畫線註銷但保留可讀」。這個系統天然符合 21 CFR Part 11 的數據完整性精神。

Q3 批判思考 這套系統的最大限制是什麼？在哪些場景下它會比 RAG（Retrieval Augmented Generation）更差？

最大限制：規模天花板。 當資料量從 80 個檔案成長到 800 個或 8,000 個時，純檔案系統的搜尋效率會急劇下降。JSONL 的逐行讀取在大檔案時消耗大量 Token。作者的系統適合個人尺度（一個人的知識庫），不適合團隊或組織尺度。

RAG 勝出的場景：當你需要從大量文件中找到語義相似的片段時（例如從 1,000 份 SOP 中找到最相關的三份），向量檢索的效率遠勝逐行掃描。RAG 也更適合處理非結構化長文件（研究論文、法規文件）。

平衡觀點：這套系統的真正價值不在於取代 RAG，而在於示範資訊架構設計的重要性。即使你最終用 RAG，Progressive Disclosure 和模組隔離的原則同樣適用。
