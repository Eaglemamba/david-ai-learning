Executive Summary

GBrain 是 Y Combinator 總裁 Garry Tan 開源的個人知識管理系統，將散落在各處的 Markdown 文件轉化為可搜尋的「第二大腦」。核心技術是 Postgres + pgvector 混合搜尋（結合 keyword + vector + RRF fusion），加上 Compiled Truth + Timeline 雙層知識模型。AI Agent 透過 Brain-Agent Loop 自動偵測實體、enrichment、建立交叉引用，讓知識隨每次對話自動複利累積。

這不只是一個搜尋工具 -- 是一套完整的 AI Agent 知識骨幹架構，包含 entity detection、7 步 enrichment pipeline、meeting ingestion、cron-based 自動化，以及 MCP server 整合。基於真實部署（14,700+ 文件、40+ skills、20+ cron jobs）的實戰經驗。

Note: 此專案剛開源（2026-04-08），生態系仍在早期階段。Garry Tan 的使用情境（VC/CEO 社交網絡管理）可能需要調適才能應用到其他領域。需要 Supabase Pro ($25/月) + OpenAI API + Anthropic API，有一定運維門檻。

Part I: Compiled Truth + Timeline -- 知識的雙層結構

PART I

GBrain 的核心設計理念是每個知識頁面都有一條 水平分隔線 將內容分為兩個區域。這不是隨便的排版決定 -- 這是整個系統最重要的架構選擇。

Every page in the brain follows the compiled truth + timeline pattern, separated by a horizontal rule.

分隔線上方：Compiled Truth（編譯真相）。你目前對這件事的最佳理解。當新證據改變了整體圖景時，整段重寫。如果你只讀這個部分，就知道所有你需要知道的。

Above the line: Compiled truth. Your current best understanding. Gets rewritten when new evidence changes the picture.

分隔線下方：Timeline（時間軸）。只能追加、永不修改的證據記錄。反向時間順序排列。每條 compiled truth 的主張都應該可以追溯到一或多條 timeline entries。

Below the line: Timeline. Append-only evidence trail. Never edited, only added to.

💡 比喻說明
想像 GMP 文件的結構：SOP（標準作業程序）就像 Compiled Truth -- 它代表「目前的最佳做法」，當驗證數據改變時就更新。而 Batch Record / Deviation Log 就像 Timeline -- 每筆記錄只增不改，是不可竄改的證據鏈。GBrain 把這個製藥界行之有年的「truth + evidence」模式應用到個人知識管理上。

這就是 Karpathy 的 LLM wiki 模式的殺手級特性：綜合報告是預先計算好的。不像 RAG 每次查詢都要讓 LLM 從頭推導知識，你的大腦已經做過這個工作了。交叉引用已經建好。矛盾已經被標記。

This is the Karpathy wiki pattern's killer feature: the synthesis is pre-computed. Unlike RAG, where the LLM re-derives knowledge from scratch every query, your brain has already done the work.

📚 核心概念：為何優於 RAG？

傳統 RAG 的流程：Query → Retrieve chunks → LLM 即時綜合 → 回答。每次都是從零開始。

GBrain 的流程：Agent 持續維護 Compiled Truth → Query 直接命中預先編譯好的綜合結論。交叉引用和矛盾在寫入時就處理好了。

關鍵差異：RAG 是「讀取時計算」，GBrain 是「寫入時計算」。後者的搜尋品質隨時間穩定提升，前者每次都看當下 chunk 的運氣。

⚠️ 重點提示：Source Attribution

GBrain 有一條硬性規則：每個事實都需要引用來源。格式：[Source: {who}, {channel}, {date} {time} {tz}]

這不是建議，是強制要求。六個月後有人讀一個 brain page，必須能追溯每一個事實的出處。沒有 URL 的推文引用 = 壞掉的 citation。

來源權威層級：使用者直接陳述 > 主要來源（會議/Email）> Enrichment API > 網頁搜尋 > 社群媒體。衝突時標註矛盾，不要悄悄選邊。

Part II: 混合搜尋架構 -- Keyword + Vector + RRF

PART II

GBrain 的搜尋不是單純的 keyword 或 vector -- 它結合了兩者，再用 Reciprocal Rank Fusion (RRF) 融合排名。這是因為兩種搜尋各有盲區。

Keyword search alone misses conceptual matches. Vector search alone misses exact phrases when the embedding is diluted. RRF fusion gets both right.

Keyword 搜尋的盲區：搜「ignore conventional wisdom」不會找到一篇標題叫「The Bus Ticket Theory of Genius」的文章，即使它完全在講這件事。因為精確字詞不匹配。

"Ignore conventional wisdom" won't find "The Bus Ticket Theory of Genius" even though it's exactly about that.

Vector 搜尋的盲區：當 embedding 被周圍的文字稀釋時，精確的名字和短語反而搜不到。所以你仍然需要 keyword 搜尋來抓精確匹配。

Vector search alone misses exact phrases when the embedding is diluted by surrounding text.

Multi-query Expansion：搜尋前先用 Claude Haiku 將你的問題擴展為多個語義變體。「biggest risks」可能展開為「competitive moats」「board prep concerns」「strategic threats」。

Multi-query expansion catches phrasings you didn't think of.

💡 比喻說明
想像你在 CDMO 裡同時搜 Deviation 資料庫：用「exact error code」搜是 keyword search（精確但窄），用「類似的偏差模式」搜是 vector search（寬泛但可能模糊）。GBrain 的做法是：兩個都搜，然後用統計方法（RRF）把兩個排名融合成一個最佳排名。
💻 搜尋流水線

Step 1: Query → Multi-query expansion (Claude Haiku)

Step 2: 雙路搜尋 -- Vector (HNSW cosine) + Keyword (tsvector ts_rank)

Step 3: RRF Fusion -- score = sum(1/(60 + rank))

Step 4: 四層去重 -- (1) 每頁最佳 chunk (2) Cosine > 0.85 去重 (3) Type 多樣性 60% cap (4) 每頁 chunk 上限

Step 5: Stale alerts -- compiled truth 比最新 timeline 舊時提醒

Part III: Entity Detection + Enrichment Pipeline

PART III

GBrain 要求在每一條訊息上觸發 Entity Detection -- 沒有例外（除非是純操作性訊息如「OK」「do it」）。用一個輕量子代理（cheap model 如 Claude Sonnet）捕捉兩類訊號。

Spawn a lightweight sub-agent on EVERY inbound message. The sub-agent captures two things with equal priority.

第一優先：Original Thinking。使用者的原創想法、觀察、論點、框架。這是整個系統中最高價值的訊號。要用使用者的原話逐字捕捉 -- 因為語言本身就是洞察。不要清理、不要改述。

Original Thinking is the highest-value signal. Capture the user's EXACT phrasing. The language IS the insight.

第二優先：Entity Mentions。人名、公司、媒體引用。對每個偵測到的實體：(1) 檢查 brain page 是否存在 (2) 不存在且值得追蹤就創建 (3) 頁面太薄就啟動 enrichment (4) 頁面豐富就默默載入作為上下文。

People, companies, media references. Check brain, create if notable, enrich if thin, load if rich.

⚠️ Iron Law: Back-linking

每個 entity page 都必須回連到提到它的來源頁面。一個沒有 back-link 的 mention 就是一個壞掉的 brain。格式：在 entity 的 Timeline 或 See Also 追加 - YYYY-MM-DD | Referenced in [page title](path) -- context

🔧 7 步 Enrichment Pipeline

1. Identify: 從訊號中提取人名、公司名

2. Check brain: 搜尋是否已有頁面 → UPDATE 或 CREATE 路徑

3. Extract signal: 不只抓事實，抓紋理 -- 他們相信什麼？在建什麼？什麼驅動他們？

4. Data lookups: Brain 交叉引用 → Web search → X/Twitter → Crustdata (LinkedIn) → Captain (funding) → Circleback (meetings)

5. Save raw data: 每個 API response 存到 .raw/ sidecar

6. Write to brain: 新頁面用 schema template，舊頁面追加 timeline + 更新 compiled truth

7. Cross-reference: 更新所有相關 entity pages 的 backlinks

💰 Tier 分級 API 花費
Tier對象API Calls
Tier 1核心人物、業務夥伴10-15
Tier 2偶爾互動的知名人士3-5
Tier 3僅被提及的人1-2

按重要性分配 API 預算。不要為一個路過的 mention 燒 20 個 API call。

Part IV: MECE 目錄架構 + Knowledge Schema

PART IV

GBrain 的知識庫遵循 MECE 原則（Mutually Exclusive, Collectively Exhaustive）：每一塊知識通過決策樹，落在正好一個目錄裡。沒有重複頁面、沒有歸屬模糊。

Every piece of knowledge passes through a decision tree and lands in exactly one directory. No duplicated pages, no ambiguity.

這是最重要的結構決策。沒有它，知識庫會腐爛 -- 同一個事實存在三個地方、三個版本，沒人知道哪個是最新的。每個目錄都有一個 README.md 解答兩個問題：(1) 什麼放這裡 (2) 什麼不放這裡。

Without it, knowledge bases rot — the same fact lives in three places with three different versions.

重要細微差異：MECE 適用於目錄，不適用於現實。一個真實的人可能同時是政治創辦人、朋友、捐贈者。MECE 選擇主要歸屬（people/），頁面本身用 typed backlinks 展現所有面向。防止重複頁面，不是防止重複關係。

MECE applies to directories, not to reality. Cross-references preserve adjacency without breaking the one-page-per-entity rule.

💡 比喻說明
在 CDMO 裡，這就像 Document Management System (DMS) 的分類結構：一份 SOP 只能歸屬一個文件類別（MECE），但可以透過交叉引用連結到其他相關文件。如果同一份 SOP 存在兩個版本、兩個資料夾，那就是 data integrity violation。GBrain 用同樣的邏輯管理知識。
💻 目錄結構

brain/ 根目錄包含：

RESOLVER.md -- 主決策樹（Agent 建新頁前必讀）

schema.md -- 頁面慣例和模板

people/ -- 一人一頁（含 .raw/ API sidecar）

companies/ -- 一組織一頁

deals/ -- 財務交易與條款

meetings/ -- 會議記錄 + 轉錄稿

concepts/ -- 心智模型和框架

originals/ -- 使用者原創想法（最高價值）

ideas/ -- 尚未有人執行的原始可能性

inbox/ -- 暫存未分類（schema 需演化的訊號）

⚠️ Entity Dedup Protocol

建新頁前必須：

1. 搜尋既有頁面（精確 + 模糊）

2. 搜尋 aliases：grep -rl "NAME" brain/people/

3. 檢查 .raw/ 的 email / social handle

4. 匹配到 → UPDATE（加 alias）

5. 沒匹配 → CREATE

合併 protocol：選較完整的頁面為存活者，合併 timeline + aliases + cross-refs，刪除重複頁。

Part V: 運營架構 -- Cron、Meeting Ingestion、Dream Cycle

PART V

GBrain 的生產部署跑了 20+ 個 cron jobs，自動化處理 email 監控（每 30 分鐘）、社群媒體攝取（每小時）、會議處理（每日 3 次）、晨間簡報（每日）、大腦維護（每週）。

A production agent runs 20+ recurring jobs that interact with the brain.

Meeting Ingestion 是最豐富的訊號來源。每次會議的處理流程：(1) 拉完整轉錄稿（不要 AI 摘要 -- AI 摘要會虛構框架）(2) 建立會議頁面，用 Agent 自己的分析重構（針對使用者的優先順序）(3) 強制傳播到 entity pages -- 更新所有出席者、公司、交易的頁面 (4) 提取行動項目。

Always pull the complete transcript, not just the AI summary. AI summaries hallucinate framing. Propagate to entity pages is MANDATORY.

Dream Cycle（夢境循環）是最重要的 cron job。在安靜時段（如 23:00-08:00）自動執行：掃描今天所有對話的 entities、enrichment 薄弱頁面、修復壞掉的 citations、將重要訊號提升到 MEMORY.md。

The dream cycle is why the brain compounds. During quiet hours: entity sweep, enrich thin spots, fix broken citations, consolidate memory.

💡 比喻說明
Dream Cycle 就像 CDMO 的夜間 batch processing：白天的生產活動（對話、會議）產生大量數據，晚上的自動化流程負責清理、歸檔、交叉驗證。就像 QA 的「overnight stability testing」-- 系統在你睡覺時自我修復和強化。
🔧 技術整合方式

CLI 使用：安裝後用 gbrain import（冪等，SHA-256 比對跳過未更改文件）、gbrain query（混合搜尋）、gbrain sync --watch（polling 增量同步）。

MCP Server：gbrain serve 啟動 stdio MCP server，可直接整合 Claude Code / Cursor。提供 query/get/put/search/link 等工具。

Library：import { PostgresEngine } from 'gbrain' 直接程式化調用搜尋、寫入、graph traversal。

依賴：Supabase Pro ($25/月)、OpenAI API（embeddings）、Anthropic API（expansion + chunking）。沒有 OpenAI key 也能跑（只有 keyword search），沒有 Anthropic key 也能跑（沒有 multi-query expansion）。

📚 智慧時區處理

Agent 讀取日曆的航班/飯店資訊推斷你的當前位置。飛到東京後，白天 Pacific 時區的 cron job 如果落在東京凌晨 3 點，會被自動延遲並合併到下一個晨間簡報。

零設定更改。Agent 從日曆自己推算。零錯過訊號，零凌晨 3 點叮咚。

Key Takeaways

Compiled Truth > RAG
GBrain 的核心洞察：知識應該在「寫入時」就編譯成綜合結論，而不是每次查詢時讓 LLM 從碎片重新推導。pre-computed synthesis + immutable timeline = 可追溯、持續改善的知識品質。
Brain-Agent Loop 是複利引擎
每個訊號都要 READ before respond + WRITE after learn。六個月後的 compound effect 是：Agent 對你的世界的理解超過你自己的工作記憶，因為它從不遺忘、從不停止索引。
混合搜尋是必要架構
Keyword 抓精確名字，Vector 抓語義概念，RRF Fusion 統計融合兩者排名。加上 Multi-query Expansion 擴展搜尋意圖。任何單一搜尋模式都有盲區。
知識管理 30 年失敗的根因被 LLM 解決
知識管理失敗是因為維護成本落在人類身上。LLM Agent 不會無聊、不會忘記更新交叉引用、可以一次觸碰 50 個文件。維護成本趨近於零時，wiki 才能真正存活。

Practice Questions

Q1 概念理解 Compiled Truth + Timeline 模式和傳統 RAG 的根本差異是什麼？為什麼 Garry Tan 說「synthesis is pre-computed」？

RAG 是「讀取時計算」：每次查詢都要從 chunk 庫中檢索碎片，然後讓 LLM 即時綜合出答案。答案品質取決於當次檢索的 chunk 質量和 LLM 的推理能力。

GBrain 是「寫入時計算」：每當新資訊進入系統，Agent 就會更新 Compiled Truth -- 把新證據整合進已有的綜合結論中。查詢時直接命中已經編譯好的知識，交叉引用和矛盾在寫入時就處理完了。

實際差異：RAG 在第 100 次查詢時的品質和第 1 次差不多（甚至可能更差因為 chunk 庫變大了）。GBrain 的 Compiled Truth 品質隨每次寫入穩定提升，因為每次都是在既有綜合結論上迭代。

Q2 產業應用 如果要將 GBrain 的 Compiled Truth + Timeline 模式應用到 CDMO 的偏差管理（Deviation Management），你會如何設計？

Compiled Truth 對應：每個偏差類型（如 particulate excursion、environmental monitoring OOS）的「目前最佳理解」-- 已知根因模式、有效的 CAPA、相關 SOP。當新偏差帶來新證據時，更新這個綜合結論。

Timeline 對應：每次偏差發生的不可修改記錄 -- 日期、批號、調查結論、CAPA 編號。只增不改，完整的證據鏈。

Entity Detection 對應：偏差報告中自動偵測到的實體 -- 設備編號、操作員、房間號碼、原料批號 -- 每個都有自己的 brain page，自動交叉引用。這樣就能回答「Filter A3 過去 12 個月的所有相關偏差」這類查詢。

注意：在 GMP 環境中，這只能作為 copilot 模式（人類審核所有 Agent 寫入），不能是 autonomous agent。21 CFR Part 11 要求所有記錄變更都有人類簽核。

Q3 批判思考 GBrain 的架構有哪些潛在的失敗模式或限制？在什麼情境下它的價值主張會打折？

冷啟動問題：GBrain 的價值來自複利累積。前 100 頁的 ROI 很低 -- 搜尋不夠豐富、交叉引用太少。Garry Tan 有 14,700+ 文件是因為他經營 VC 多年的累積。新用戶需要耐心。

Compiled Truth 可能過時但看起來很新：如果 Agent 沒有被正確觸發更新，Compiled Truth 可能包含過時資訊但沒有 stale alert。特別是當外部世界變化（如某人離職）但沒有訊號流入系統時。

成本和維護負擔：Supabase Pro $25/月 + OpenAI embedding 費用 + Anthropic API 費用 + 20+ cron jobs 的維運。對個人用戶來說這不是零成本。Garry 的使用情境（VC 社交網絡管理）天然產生大量 entity，但一般知識工作者的 signal 密度可能不夠高。

Agent 寫入品質依賴 LLM 能力：Compiled Truth 的品質取決於 Agent 的綜合能力。如果 Agent 不擅長判斷什麼重要、什麼矛盾，那 Compiled Truth 可能比 RAG 更糟 -- 因為壞的綜合結論會被當作「truth」持續使用。
