Part I Cowork 是什麼？不只是 Chat 換皮

Cowork 不是一個換了外觀的聊天介面。它是一個內建在 Claude Desktop 應用程式中的自主桌面 Agent。當你打開 Cowork 分頁，你等於是給 Claude 一台在你機器上運行的沙箱 Linux VM 的存取權限。

Cowork is not a chat interface with a new skin. It's an autonomous desktop agent built into the Claude Desktop app. When you open the Cowork tab, you're giving Claude access to a sandboxed Linux VM running on your machine.

在那個沙箱裡，Claude 可以寫程式碼、執行腳本、建立檔案（Word 文件、簡報、試算表、PDF），還能連接 Gmail、GitHub、Slack 等服務。你描述需求，Cowork 會規劃工作、拆解成平行運作的 sub-agents，然後以可點擊的檔案交付結果。

Inside that sandbox, Claude can write code, execute scripts, create files, and connect to services like Gmail, GitHub, and Slack. You describe what you need. Cowork plans the work, breaks it into sub-agents that run in parallel, and delivers output as clickable files.

📚 核心概念：Cowork 的五大差異化能力

1. 規劃與追蹤 - 將複雜任務拆解為子任務，即時顯示進度
2. 平行 Sub-agents - 產生獨立 Claude 實例同時處理不同部分
3. 真正的檔案輸出 - .docx/.pptx/.xlsx/.pdf 直接送到你的資料夾
4. 沙箱但有存取權 - VM 內運行，但可讀寫你共享的資料夾
5. 工具連接 - Gmail、GitHub、Slack、Google Drive + 自訂 MCP

💡 比喻說明

如果 Chat 像是跟助理「對話」請他幫忙，Cowork 更像是把一個專案交給一整個小團隊 -- 你當 PM，Cowork 當 Tech Lead，sub-agents 是各自負責不同模組的工程師。你看到的是看板上的進度條，拿到的是完成品。

Part II Chat vs. Cowork：為什麼是不同物種

Huryn 做了一個清楚的比較：Chat 是對話，Cowork 是工作流。Cowork 增加了真正重要的能力：sub-agent 協調平行工作、任務分解、檔案直接送到你的資料夾而非 chat artifacts。

Chat is for conversations. Cowork is for workflows. Cowork adds what matters for getting real work done: sub-agent coordination, task decomposition, and files delivered directly to your folder.

Part III Skills 與 Plugins 生態系統

Skills 是可重複使用的指令手冊，教 Claude 如何處理特定的重複性任務。說「建立一個 Word 文件」，docx skill 就會載入。Skills 不會一次全部載入 -- Claude 只讀取每個 skill 的簡短描述（約 100 tokens）來決定哪些相關，然後才載入完整指令。這保持了 context window 的乾淨。

Skills are reusable instruction manuals that teach Claude how to approach specific, repeatable tasks. Skills don't all load at once. Claude reads only a short description of each skill (~100 tokens) to decide which ones are relevant, then loads full instructions only when needed.

Cowork 有專屬的 Plugins 面板（Chat 沒有）。你可以瀏覽、安裝、上傳和建立 plugins。每個 plugin 打包了 skills 加上 slash commands。Skill 格式在 Claude 生態系統和第三方工具（Cursor, Windsurf, Codex CLI）間通用。

Cowork has a dedicated Plugins panel that Chat doesn't. Each plugin bundles skills with slash commands. The format works across Claude ecosystem and third-party tools like Cursor, Windsurf, and Codex CLI.

⚠️ 重點提示：Plugin 隔離規則

Cowork 和 Code Tab 的 Plugin 面板是獨立的。在一個裝的 plugin 不會出現在另一個。但透過 Claude Desktop Settings 上傳的 Skills 會在 Chat、Cowork、Code Tab 三個地方共享。

你可以把 Code 的開發者 plugins 加到 Cowork，反之亦然 -- 同樣的 skill 格式，完全跨相容。

Part IV MCP：連接 Cowork 到你的世界

MCP（Model Context Protocol）是 Anthropic 的開放標準。每個 MCP server 暴露 Claude 可以呼叫的工具。理解三種 MCP 連接方式的差異非常重要：

MCP stands for Model Context Protocol — the open standard by Anthropic. Each MCP server exposes tools Claude can call. There are three ways to connect MCP servers to Claude.

Remote Connectors 和 Custom Connectors 在所有地方都能用（包括瀏覽器的 claude.ai）。Extensions 是 Anthropic 包裝的本地 MCP server，一鍵安裝。自訂 MCP server 則需要編輯 JSON 設定檔。

Remote and custom connectors work everywhere — including claude.ai in your browser. Extensions are how Anthropic packages local MCP servers for one-click install. Custom MCP servers are managed by editing a JSON config.

每個 connector 都可以設定個別工具的權限：Allow（自動執行）、Ask（執行前確認）、Block（永不執行）。你可以允許 Claude 搜尋你的 email 但阻止它發送 email。

For every connector, you can set individual tools to Allow, Ask, or Block. You could allow Claude to search your emails but block it from sending them.

⚖️ 三種 MCP 連接方式比較
類型設定方式適用範圍
RemoteConnectors UI所有平台含 claude.ai
ExtensionsSettings 一鍵安裝Desktop 三個分頁
CustomJSON 設定檔Desktop 三個分頁

⚠️ Windows 陷阱

透過 Microsoft Store (MSIX) 安裝的 Claude Desktop，「Edit Config」按鈕可能打開錯誤的檔案。App 讀取的是 MSIX 虛擬化路徑，而非 %APPDATA%\Claude\。如果 MCP server 靜默載入失敗，檢查 GitHub issue #26073。

Part V Desktop Commander：一分鐘讓 Claude Desktop 強 2 倍

Huryn 認為 Desktop Commander 是投資報酬率最高的動作之一，而且只需不到一分鐘。安裝後，Chat、Cowork 和 Code Tab 都能在你的筆電上做幾乎任何事情，包括安裝 MCP server 或存取任何檔案。

That's one of the highest-ROI moves. And it takes less than 1 minute. Chat, Cowork, and Code Tab can do virtually anything on your laptop including installing MCP servers or accessing any file.

🛠️ 安裝步驟（4 步）

1. 打開 Claude Desktop
2. 在聊天視窗點擊：+ > Connectors > Manage connectors
3. 點擊：Browse connectors > All > Desktop Commander
4. 選擇哪些工具不需要你的核准

提示：不使用時停用 Claude in Chrome 擴充功能，避免 Claude 預設使用網頁操作。

Part VI 跨 Session 記憶方案

Huryn 提供了兩個層次的記憶方案。基礎版只需兩步：啟用 Desktop Commander + 在 Global Instructions 加入記憶管理指令。進階版則使用結構化記憶系統：.claude/memory/ 下分為 general.md、domain/{topic}.md、tools/{tool}.md 等檔案。

Two simple steps for basic memory. Advanced: Split into multiple files so Claude loads only what's relevant — structured memory rooted at .claude/memory/.

核心規則是：當你學到值得記住的東西，立刻寫入正確的檔案。每條記錄包含：日期、什麼事、為什麼。Session 開始時讀取 memory.md，其他檔案只在相關時才載入。

When you learn something worth remembering, write it to the right file immediately. Entries: date, what, why — nothing more. Read memory.md at session start. Load other files only when relevant.

Key Takeaways

Cowork = 非開發者的 Agent 入口

同樣的模型、同樣的 skill 格式，但用視覺介面取代 CLI。對 PM、營運、行銷人員來說，Cowork 是 Claude 生態中最被低估的工具。

三層擴展：Skills > Plugins > MCP

Skills 是指令手冊，Plugins 打包 skills + slash commands，MCP 連接外部服務。三層疊加讓 Cowork 從文件產生器變成全方位工作平台。

權限控制是關鍵

每個 MCP connector 都能設定 Allow/Ask/Block。Desktop Commander 雖然強大，但需要謹慎設定哪些操作需要你的核准，特別是在有敏感資料的環境。

記憶問題有解但需手動

memory.md 方案簡單有效，但依賴 Desktop Commander 且需要手動維護。這不是原生功能，而是社群的 workaround。

Practice Questions

Q1 概念理解 Cowork 的 sub-agent 機制與傳統 Chat 的差異，本質上解決了什麼問題？

Sub-agents 解決的核心問題是 context window 的瓶頸。在 Chat 中，所有對話都在同一個 context 中，複雜任務容易超出上下文限制或導致品質下降。Cowork 的 sub-agents 各自獨立的 context，可以平行處理不同子任務，最後匯總結果。

這類似於軟體工程中的「分而治之」-- 把一個大問題拆成多個小問題，各自解決後再合併。對使用者來說，這意味著可以交付更複雜的工作而不需要手動拆解任務。

Q2 產業應用 在 CDMO 的 GMP 環境中，Cowork 的 Desktop Commander 功能帶來什麼合規風險？如何設定才能在 copilot mode 下安全使用？

合規風險：Desktop Commander 讓 Claude 能存取和修改本機檔案。在 GMP 環境中，這可能觸及 21 CFR Part 11 的電子記錄要求 -- 任何對品質文件的修改都需要 audit trail 和授權簽核。

Copilot mode 設定建議：

  • 所有與 GMP 文件相關的操作設定為 Ask（執行前確認），絕不用 Allow
  • 將 Cowork 的共享資料夾限制在非 GMP 的工作區域（如個人學習、行政文件）
  • GMP 文件的處理仍使用 validated 系統，Cowork 僅用於草稿產生和分析
  • Block 所有自動發送 email 的功能，避免未審核的通訊送出
Q3 批判思考 Huryn 對 Scheduled Tasks 評價為「不可靠」，但推薦了 memory.md 的手動方案。這反映了 AI Agent 工具目前的什麼結構性限制？

這反映了當前 AI Agent 在「可靠自主」vs.「人類可控」之間的張力。Scheduled Tasks 需要 Agent 在無人監督下可靠執行 -- 這目前技術上仍不穩定。而 memory.md 本質上是把「記憶」的責任推回給人類（或半手動的 Agent），這其實是 copilot mode 的延伸。

結構性限制包括：1) 缺乏可靠的後台持久化機制；2) Agent 無法自我驗證跨 session 的一致性；3) 錯誤累積問題 -- 每次 session 重新讀取記憶都可能引入偏差。這也呼應了「2-3x discount principle」-- 對 Agent 自主能力的宣傳需要打折評估，手動 workaround 往往比自動化方案更可靠。

🎯 我的行動結論
✔ 核心洞察
Cowork 填補了 Chat（太簡單）和 Code（太開發者導向）之間的空白。對我作為 Operations Director 來說，文件製作、email 處理、檔案整理這些日常工作流，Cowork 的 sub-agent + 真檔案輸出比 Chat 的 artifacts 更適合。但在 GMP 環境中，它的角色仍然是 copilot -- 產出需要人類審核才能進入正式流程。
📈 立即應用
本週在 Claude Desktop 中開啟 Cowork tab，安裝 Desktop Commander。先用於非 GMP 的行政工作測試（如整理學習筆記、產生會議摘要文件）。評估 knowledge-work-plugins 的 productivity 和 product-management plugins 是否適合我的 COO 發展路線。
⚠ 需要補充
文章缺少 Cowork 的實際效能限制（如 sub-agent 數量上限、VM 資源限制）。memory.md 方案的長期可維護性也未驗證。需要自己測試 Cowork 在處理中文文件（特別是含表格的 Word/Excel）時的品質，這對 CDMO 場景很關鍵。另外，Scheduled Tasks 的替代方案（n8n/Make.com）與 Cowork 的整合可能性值得進一步研究。
