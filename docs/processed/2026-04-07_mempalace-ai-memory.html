Executive Summary

MemPalace 是一個開源的 AI 記憶系統，核心理念是「儲存一切，再讓它可搜尋」，而非讓 AI 決定什麼值得記住。系統由三大支柱構成：(1) Palace 階層結構（Wings/Halls/Rooms）提供 +34% 的檢索提升；(2) AAAK 壓縮方言實現 30x 壓縮、零資訊遺失；(3) 本地優先架構，完全離線運行，無需 API 金鑰。在 LongMemEval 標準測試中，零 API 模式即達 96.6% R@5，加上 Haiku rerank 達 100%。

2-3x Discount 提醒：專案方自行公布的 benchmark，附有可重現 runner 但尚未經第三方驗證。專案僅 6 commits，成熟度有限。

PART I
核心問題：AI 對話記憶的消失

每一次你與 AI 的對話 — 每個決策、每次除錯、每場架構辯論 — 都在對話結束時消失。六個月的工作，歸零。每次都從頭來過。

Every conversation you have with an AI — every decision, every debugging session — disappears when the session ends.

其他記憶系統讓 AI 決定什麼值得記住，萃取出「使用者偏好 Postgres」然後丟掉為什麼的脈絡。MemPalace 採取不同策略：儲存一切，再讓它可搜尋。

MemPalace takes a different approach: store everything, then make it findable.

💡 比喻說明
想像你有一本 1000 頁筆記本。AI 萃取派= 請助理濃縮成 5 頁，但可能漏掉關鍵推理過程。MemPalace 派= 保留整本，但加上精確的目錄索引 — 找什麼都有，而且是原話。
⚠️ 重點提示
AI 記憶領域過度工程化了萃取步驟。原始逐字文本 + 好的 embedding 搜尋，比大家想的更強 — 因為它不遺失任何資訊。

PART II
Palace 階層架構

MemPalace 源自古希臘記憶宮殿法（Method of Loci）。對話被組織成 Wings（人/專案）、Halls（記憶類型）、Rooms（具體主題）。

Conversations organized into wings, halls, and rooms. The structure makes it searchable.

📚 Palace 層級定義

Wings（翼）— 一個人或專案。無限多個。

Halls（廳）— 五種記憶走廊：facts、events、discoveries、preferences、advice。

Rooms（室）— 具體主題：auth-migration、ci-pipeline。

Tunnels（隧道）— 相同 Room 跨 Wings 時自動連接。

Closets / Drawers — AAAK 壓縮版 / 原始逐字稿。

💡 比喻說明
100 層大樓找文件。全域搜尋 = 逐層掃描。Wing = 知道哪棟。Wing+Hall = 知道哪層。Wing+Room = 知道哪間辦公室。每一步都大幅降低「找錯」的機率。
⚠️ 結構帶來 34% 檢索提升

22,000+ 筆對話記憶測試：

搜尋範圍R@10提升
全域搜尋60.9%baseline
特定 wing73.1%+12%
wing + hall84.8%+24%
wing + room94.8%+34%

結構不是裝飾，是產品的核心。

🛠️ Tunnel 跨域連接

同一 Room（auth-migration）出現在三個 Wings 時，Tunnel 自動交叉引用：Kai 的除錯紀錄、團隊的決策、Priya 的審批 — 三個視角無縫串聯。

Tunnel 跨域搜尋範例 — 同一 Room, 三個 Wings
wing_kai       / hall_events / auth-migration
  → "Kai debugged the OAuth token refresh"

wing_driftwood / hall_facts  / auth-migration
  → "team decided to migrate auth to Clerk"

wing_priya     / hall_advice / auth-migration
  → "Priya approved Clerk over Auth0"
PART III
AAAK 壓縮方言：30x 壓縮

AAAK 是為 AI Agent 設計的無損速記方言。不給人讀 — 給 AI 快速讀取。30 倍壓縮、零資訊遺失。~120 tokens 載入數月脈絡。

AAAK — A lossless shorthand dialect designed for AI agents. 30x compression, zero information loss.

因為 AAAK 是結構化文本，任何能讀文本的模型都能用 — Claude、GPT、Gemini、Llama、Mistral。不需解碼器或微調。

Works with any model that reads text. No decoder, no fine-tuning required.

⚠️ 為什麼是「方言」而非「編碼」？
AAAK 刻意設計為英文的極度精簡版。任何 LLM 不需特殊訓練即可理解；人類也能猜出大意；不需解碼步驟。這是 Context Engineering 的極致展現 — 最少 tokens 傳達最大資訊密度。
🛠️ CDMO 應用想像
AAAK 概念可用於壓縮 Tech Transfer 製程參數，作為 AI Copilot 快速檢索層。GMP 環境下原始文件仍需完整保留，但 AAAK 格式可加速 AI 輔助查詢。

Practice Questions

Q1概念理解為什麼「原文全存」策略能勝過 LLM 萃取系統？

核心在於資訊完整性。LLM 萃取做了不可逆判斷 — 萃取錯誤，記憶永遠消失。MemPalace 保留原文，embedding 模型可升級、搜尋策略可改進，但原始資料從未損壞。如同 GMP「原始資料不可覆寫」原則。

含義：AI 記憶領域可能過度工程化了「萃取」。好的 embedding + 結構化索引是被低估的 baseline。

Q2產業應用用 Palace 架構組織 CDMO 偏差調查記錄，如何設計 Wings/Halls/Rooms？

Wings — 按產品線：wing_product_A、wing_client_pfizer。

Halls — hall_root_causes、hall_capas、hall_trends、hall_regulatory、hall_lessons。

Rooms — particulate-contamination、fill-volume-oos、environmental-monitoring。

當不同產品線都有 particulate-contamination Room 時，Tunnel 自動連接 — 這正是 PIC/S 和 FDA 重視的 trend analysis 能力。

Q3批判思考應用 2-3x Discount，96.6% 零 API claim 有哪些質疑點？

1. 自行公布 benchmark，尚無第三方驗證。

2. 排行榜中 Mem0 數據來自不同測試集（ConvoMem），並排比較可能誤導。

3. 6 commits、stars 數據不一致 — 早期專案紅旗。

4.「零 API」仍需本機 sentence-transformers，是「零雲端 API」而非「零依賴」。

結論：概念紮實，但數字 claim 需 2-3x 懷疑。當作「值得觀察的早期專案」。
