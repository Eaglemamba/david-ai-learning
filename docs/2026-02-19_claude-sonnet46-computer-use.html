Part I: Sonnet 4.6 的三大升級

Claude Sonnet 4.6 的核心價值主張很直接：以旗艦模型五分之一的價格，提供旗艦等級的性能。在早期測試中，開發者實際上有 59% 的時間偏好 Sonnet 4.6 而非最昂貴的 Opus 4.5 模型。這代表 Anthropic 的定價策略正在積極壓低「智慧」的取得門檻。

"Claude Sonnet 4.6 now performs at the level of Anthropic's most expensive model (called Opus) at one-fifth the cost. In early testing, developers actually preferred Sonnet 4.6 over the flagship Opus 4.5 model 59% of the time."

另一個重要升級是 1 百萬 Token 的上下文窗口（Context Window），相當於好幾本完整小說的長度。如果你曾遇過 Claude 在對話中途「忘記」你之前說的內容，這就是解方。更大的記憶意味著更連貫的長對話體驗。

"The new model can hold up to 1 million tokens (roughly the equivalent of several full-length novels) in a single conversation. If you've ever had Claude lose track of what you were discussing mid-conversation, this is the fix."

最後，免費方案現在預設使用 Sonnet 4.6。這不是閹割版 — 而是與付費用戶和開發者使用的完全相同的模型。「Premium AI」和「Accessible AI」之間的差距顯著縮小。

"Everyone on the free plan now gets Sonnet 4.6 as the default. This isn't a watered-down version. It's the same model that paying users and developers are building with."

📚 核心概念：性能民主化

Anthropic 的策略值得注意：不是讓最新模型成為高價獨享，而是讓前代旗艦的能力快速普及。Sonnet 4.6 能匹敵 Opus 但價格只有 1/5，免費版也直接搭載 — 這是一個「智慧通膨」（intelligence deflation）的清楚訊號。對企業決策者來說，意味著「等等再導入 AI」的機會成本正在急速上升。

💡 比喻說明

想像你去買車。去年的頂級房車現在以國民車的價格在賣，而且同時開放免費試駕。你可能不會立刻買，但你絕對會重新評估通勤方式。AI 模型的定價正在經歷同樣的事。

💰 成本視角

文章提到 Sonnet 4.6 是 Opus 的 1/5 價格，但沒有給具體的 API 定價數據。對我們這種需要計算 ROI 的使用者來說，這是一個資訊缺口。關鍵問題：在你的 use case 中，Sonnet 4.6 是否已經「夠好」到可以從 Opus 降級？59% 的偏好率暗示答案可能是肯定的。

Part II: Computer Use — 真正的頭條

自 2024 年 10 月以來，Anthropic 一直在開發 Computer Use（電腦使用）功能。簡單說：AI 可以看到你的螢幕，並像你一樣操作軟體 — 點擊滑鼠、打字、在應用程式之間切換。

"Since October 2024, Anthropic has been developing something called 'computer use.' It's exactly what it sounds like. The AI can see your screen and interact with software the way you do, by clicking a mouse, typing on a keyboard, and navigating between applications."

當這個功能首次推出時，Anthropic 自己都承認它是「實驗性的、有時笨拙且容易出錯」。那是誠實的。它很慢、會犯錯，不是你敢拿來處理真實工作的東西。

"When this feature first launched, Anthropic themselves called it 'experimental, at times cumbersome and error-prone.' That was honest. It was slow, it made mistakes, and it wasn't something you'd trust with real work."

但變化很快。在一個標準化測試（評估 AI 完成真實軟體任務的能力，使用 Chrome、試算表、程式編輯器等）中，Claude 的分數從 2024 年 10 月的 14.9% 提升到 2026 年 2 月 17 日的 72.5%。16 個月內接近五倍的進步。

"On a standard test that evaluates AI's ability to complete real software tasks... Claude's score went from 14.9% in October 2024 to 72.5% on February 17, 2026. Nearly a five-times improvement in sixteen months."

實際應用上，Anthropic 報告早期使用者在「導航複雜試算表和填寫多步驟網頁表單」等任務中看到了「接近人類水準的能力」。不是理論上的能力，而是跨多個瀏覽器分頁和應用程式的實際可靠執行。

"Anthropic reports that early users are seeing 'human-level capability' in tasks like navigating complex spreadsheets and filling out multi-step web forms. Not theoretical capability. Actual, reliable execution across multiple browser tabs and applications."

⚠️ 重點提示：五倍進步的意義

14.9% → 72.5% 這個數字本身很驚人，但更重要的是它意味著什麼：Computer Use 從「有趣的 demo」跨越到「可用於生產環境」的門檻。想想 2024 年 10 月時，每 7 個任務只能完成 1 個。現在，每 4 個任務能完成 3 個。這是質的改變，不只是量的提升。

💡 比喻說明

把 Computer Use 想成你剛雇用的新員工。16 個月前，他只能完成 15% 的交辦任務 — 基本上是個需要你全程盯著的實習生。現在他能完成 72% — 他已經是一個可以獨立處理大部分日常工作的正職員工。你不會把最敏感的客戶案件交給他，但日常行政作業？完全可以放手。

📚 核心概念：不需要 API 整合

文章的一句話點出了關鍵差異：「AI 不需要與你的軟體建立特殊連結。它用和人一樣的方式使用軟體。」 這是 Computer Use 和傳統自動化（API、RPA）的根本不同。傳統方式需要每個系統都有整合介面。Computer Use 則是從 UI 層面操作，意味著即使是沒有 API 的老舊系統也能被自動化。

Part III: 對組織的影響 — Legacy Systems 的解方？

每個組織都有無法輕易自動化的軟體。早於現有團隊的舊 HR 系統、需要三個不同工具的費用核准流程、在現代整合方案出現前就建好的 CRM。

"Every organization has software it can't easily automate. The old HR system that predates your current team. The clunky expense approval workflow that involves three different tools. The CRM that was built before modern integrations existed."

過去自動化這些系統只有兩個選項：建客製化整合（昂貴、緩慢），或接受手動作業。根據 2025 年的生產力研究，一般辦公室員工每週花約 1.5 小時在不同商業應用之間複製貼上數據。

"According to productivity research from 2025, a typical office worker spends roughly 1.5 hours every week just copying and pasting data between business applications."

Computer Use 提供了第三種選項。AI 用和你的團隊一樣的方式操作軟體。不需要客製化連接器、不需要開發者工時、不需要系統改造。

"Computer use offers a third option. The AI navigates the software the same way your team does. No custom connectors. No developer hours. No system overhaul."

但 PwC 在其 2026 年 2 月的 AI 預測中也指出：「Agent 大約可以執行人們目前工作的一半任務，但這需要一種新型態的治理。」

"As PwC put it in their February 2026 AI predictions: 'Agents can do roughly half of the tasks that people now do, but that requires a new kind of governance.'"

🛠️ 實務應用：CDMO 場景思考

在 CDMO 環境中，想想那些沒有 API 的場景：手動填寫批次記錄系統、在不同 QMS 之間複製數據、或是從舊版 LIMS 導出報告。Computer Use 理論上可以處理這些 UI 層面的操作。但關鍵問題是：在 GMP 環境中，AI 操作軟體的 Data Integrity 如何驗證？Audit Trail 怎麼追蹤？這是文章完全沒觸及的議題。

⚠️ 重點提示：Governance 的關鍵

PwC 提到的「新型態治理」正是 Computer Use 從實驗到生產的最大障礙。想想看：你願意讓一個 AI Agent 自行填寫你的費用報銷表單嗎？如果它填錯了金額？如果它點了不該點的按鈕？這不只是技術問題，更是責任歸屬、風險管理和合規的問題。對高度監管的產業（如製藥），這個 Governance 門檻會更高。

⚖️ 自動化方式比較

傳統 API 整合：高可靠、可預測、需要開發資源、僅限有 API 的系統

RPA（如 UiPath）：UI 層面操作、需要固定流程腳本、維護成本高、UI 改變即失效

AI Computer Use：UI 層面操作、有一定適應能力、不需固定腳本、但準確率仍有上限（72.5%）且缺乏企業級 Audit Trail

Key Takeaways

1

性能民主化加速

Sonnet 4.6 以旗艦 1/5 的價格達到 Opus 等級性能，免費版也直接搭載。AI 能力的取得門檻正在快速降低，「觀望」的機會成本越來越高。

2

Computer Use 跨越門檻

從 14.9% 到 72.5% 的進步意味著 Computer Use 從實驗品走向生產工具。16 個月的進化速度暗示 12 個月後可能達到 90%+ 的可靠度。

3

Legacy System 的新解方

AI 不再需要 API 整合就能操作軟體。這對擁有大量老舊系統的組織（包括 CDMO）來說，是一個全新的自動化路徑。

4

Governance 是真正瓶頸

PwC 的警告很中肯：技術能力 ≠ 部署準備度。Computer Use 的治理框架 — 包括 Audit Trail、責任歸屬、錯誤處理 — 目前仍是空白，尤其在受監管產業。

Practice Questions

Q1 概念理解 Computer Use 和傳統 RPA（Robotic Process Automation）的根本差異是什麼？為什麼這個差異很重要？

傳統 RPA 依賴預先定義的腳本和固定的 UI 元素定位（如按鈕位置、CSS 選擇器）。一旦目標軟體更新 UI，腳本就可能失效，需要人工維護。

AI Computer Use 則是透過「視覺理解」螢幕內容來操作軟體 — 它看到的是畫面，不是程式碼。這意味著即使 UI 改變了佈局，AI 仍可能辨認出「提交按鈕」在哪裡，因為它理解的是語意而非座標。

這很重要是因為：它大幅降低了自動化的維護成本，讓沒有 API、沒有 RPA 專家的組織也能嘗試自動化。但代價是準確率目前只有 72.5%，而 RPA 在正常運作時接近 100%。

Q2 產業應用 在你的 CDMO 環境中，哪些工作流程最適合作為 Computer Use 的第一個試點？哪些絕對不適合？

適合試點的場景（低風險、重複性高）：從 ERP 系統導出報表並整理成特定格式、在不同系統間比對物料批號資訊、從環境監測系統擷取數據填入報告模板。這些任務失敗的後果可控，且有明確的人工覆核點。

絕對不適合的場景（GMP 關鍵、資料完整性）：批次記錄的電子簽章操作、偏差調查報告的提交、任何涉及 21 CFR Part 11 合規的數據輸入。原因：Computer Use 目前沒有符合 ALCOA+ 原則的 Audit Trail 機制，且 72.5% 的準確率在 GMP 環境中是不可接受的風險。

Q3 批判思考 文章聲稱 Computer Use 進步了「近五倍」。作為批判讀者，這個宣稱有哪些需要追問的地方？

幾個值得追問的點：

基準測試的代表性：OSWorld/WebArena 這類 benchmark 測試的任務與你實際的業務任務差距有多大？「能在 Chrome 中填寫表單」不等於「能在你的 SAP 系統中正確操作」。

從 14.9% 起步的低基數效應：從 15% 到 75% 聽起來是五倍，但如果改用「錯誤率」角度看，是從 85% 降到 27.5% — 大約三倍改善。而且從 72.5% 到 95%（企業可用門檻）的最後一段可能比前五倍還難。

利益衝突：數據來源是 Anthropic 自己。文章作者也自稱是 Claude 粉絲。套用你的 2-3 倍折扣原則，對時間線和能力宣稱都應該保持審慎。
