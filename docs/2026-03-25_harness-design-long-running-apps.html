Executive Summary

★★★★☆ 4.5/5 ↑ 技術深度, 實務應用 ↓ 學習曲線

這篇 Anthropic 工程部落格詳述了如何突破單一代理人（Solo Agent）的效能天花板。核心洞察：借鑒 GAN（生成對抗網路）的對抗結構，將「生成」與「評估」分離成獨立代理人，解決了 LLM 自我評估時過度寬容的根本問題。文章從前端設計的主觀品質評分出發，發展出 Planner-Generator-Evaluator 三代理人架構，最終實現了多小時自主編碼產出完整全端應用。從 Opus 4.5 到 Opus 4.6 的模型升級過程中，作者也展示了「簡化 Harness」的原則：移除不再必要的腳手架，保留真正 load-bearing 的元件。

Bias Flag: 本文為 Anthropic 第一方工程內容，展示自家模型能力，需以 2-3x Discount 評估宣稱的效能提升。成本數據（$124-$200/次）為真實數據但屬最佳案例展示。

Part I: 為什麼天真實作會失敗

Harness 設計對長時間自主代理編碼的有效性有實質性影響。早期實驗使用 Initializer Agent 將產品規格分解為任務清單，再由 Coding Agent 逐一實作功能，並在 session 之間傳遞產出物以保持上下文。社群也發展出類似方法，如「Ralph Wiggum」方法使用 hooks 或 scripts 讓代理人持續迭代。
"We've previously shown that harness design has a substantial impact on the effectiveness of long running agentic coding."
但問題依然存在。對於更複雜的任務，代理人仍然會隨時間脫軌。分析後發現兩個常見的失敗模式。
"For more complex tasks, the agent still tends to go off the rails over time."
📚 核心概念：兩大失敗模式

1. Context Window 填滿導致失去連貫性
模型在長任務中隨上下文窗口填滿而失去連貫性。部分模型還會出現 Context Anxiety（上下文焦慮）：當模型認為接近上下文限制時，會提前開始收尾工作。

2. 自我評估偏差
當被要求評估自己產出的工作時，代理人傾向自信地稱讚自己的作品——即使對人類觀察者來說品質明顯平庸。這在設計等主觀任務中尤其嚴重。

⚠️ 重點提示：Context Reset vs Compaction

Context Reset：完全清除上下文窗口，啟動全新代理人，搭配結構化 Handoff 傳遞前一代理人的狀態。提供乾淨的起點，但增加編排複雜度。

Compaction：在原位摘要早期對話內容，讓同一代理人繼續。保持連續性，但無法解決 Context Anxiety。

Sonnet 4.5 的 Context Anxiety 嚴重到 Compaction 不夠用，必須使用 Context Reset。

Part II: 前端設計 - 讓主觀品質可以評分

作者從前端設計開始實驗，因為這裡的自我評估問題最明顯。沒有任何干預的情況下，Claude 通常會傾向安全、可預測的佈局——技術上可用但視覺上平淡無奇。
"Absent any intervention, Claude normally gravitates toward safe, predictable layouts that are technically functional but visually unremarkable."
兩個關鍵洞察塑造了 Harness：第一，美感雖然無法完全簡化為分數，但可以用編碼設計原則與偏好的評分標準來改進。「這個設計漂亮嗎？」難以一致回答，但「這是否遵循了我們的好設計原則？」就有具體的評分依據。第二，將前端生成與前端評分分離，可以創建驅動 Generator 走向更強輸出的回饋迴路。
"While aesthetics can't be fully reduced to a score... they can be improved with grading criteria that encode design principles and preferences."
🛠️ 實務應用：四項評分標準
維度評分重點權重
Design Quality色彩、排版、佈局是否形成連貫整體而非零件集合高
Originality是否有自訂決策，還是模板預設和 AI 生成模式（如紫色漸層配白色卡片）高
Craft排版層級、間距一致性、色彩和諧、對比度低
Functionality使用者能否理解介面功能、找到主要操作低

Design Quality 和 Originality 權重較高，因為 Claude 在 Craft 和 Functionality 上已經表現良好，但設計和原創性往往「bland at best（最好也只是平淡）」。

💡 比喻說明：GAN 結構的啟示

想像你是一個畫家（Generator），有一個嚴格的美術評論家（Evaluator）持續看你的作品。如果你自己評論自己的畫，你會說「不錯啦」。但獨立的評論家會指出：「這個構圖太安全了，色彩選擇毫無個性。」——然後你才會真正去冒險嘗試。

這就是為什麼讓一個獨立的 Evaluator 變得嚴格，比讓 Generator 自我批評要容易得多。

Part III: 前端迭代迴路的運作

回饋迴路建構在 Claude Agent SDK 上。Generator 先根據使用者 Prompt 建立 HTML/CSS/JS 前端。然後 Evaluator 透過 Playwright MCP 直接與即時頁面互動——自行導覽頁面、截圖、仔細研究實作——然後對每個標準評分並撰寫詳細批評。回饋再流回 Generator 作為下一次迭代的輸入。
"I gave the evaluator the Playwright MCP, which let it interact with the live page directly before scoring each criterion and writing a detailed critique."
每次生成跑 5-15 次迭代，完整運行可長達四小時。Generator 在每次評估後做出策略決定：如果分數趨勢良好就精進當前方向，如果方法不行就完全轉向不同的美學方向。
"I ran 5 to 15 iterations per generation, with each iteration typically pushing the generator in a more distinctive direction."
一個引人注目的案例：提示模型建立荷蘭藝術博物館網站。到第九次迭代，產出了乾淨的深色主題首頁。然後在第十次循環，完全推翻方法，重新想像為空間體驗：CSS 透視渲染的 3D 房間、棋盤格地板、藝術品自由掛在牆上、用門道導覽取代滾動或點擊。
"...it scrapped the approach entirely and reimagined the site as a spatial experience: a 3D room with a checkered floor rendered in CSS perspective."
⚠️ 重點提示：迭代模式的觀察

分數通常隨迭代提升，但並非線性。作者經常偏好中間迭代版本而非最後一版。實作複雜度也隨輪次增加——Generator 在回饋下嘗試更有企圖的方案。

標準的措辭直接影響輸出。包含「最好的設計是博物館等級」這類語句會將設計推向特定的視覺收斂方向，表明與標準相關的提示語直接塑造了輸出的特質。

即使第一次迭代就比完全無 Prompt 的基線更好。標準和相關語言本身就引導模型遠離泛用預設，在任何 Evaluator 回饋之前。

📚 核心概念：技術堆疊

Claude Agent SDK：編排多代理人的基礎框架
Playwright MCP：讓 Evaluator 能「看見」和操作實際頁面
Few-shot 校準：用帶有詳細分數分解的範例來校準 Evaluator 判斷
策略分岔：每次迭代後決定 Refine（精進）或 Pivot（轉向）

Part IV: 擴展到全端開發 - 三代理人架構

Generator-Evaluator 迴路自然對應到軟體開發生命週期，其中 Code Review 和 QA 扮演與設計 Evaluator 相同的結構角色。在先前的長時間 Harness 中，已用 Initializer Agent、逐功能編碼、和 session 間的 Context Reset 解決了連貫性問題。
"The generator-evaluator loop maps naturally onto the software development lifecycle, where code review and QA serve the same structural role."
Opus 4.5 很大程度上自行消除了 Context Anxiety 行為，所以可以完全移除 Context Reset。代理人作為單一連續 session 運行，由 Claude Agent SDK 的自動 Compaction 處理上下文增長。
"Opus 4.5 largely removed that behavior on its own, so I was able to drop context resets from this harness entirely."
📚 核心概念：三代理人角色定義

Planner（規劃者）
接受 1-4 句簡短 Prompt，展開為完整 Product Spec。被指示保持有企圖的範圍，聚焦產品情境和高階技術設計，而非細節實作。避免過早規定粒度細節——如果 Planner 的細節有錯，錯誤會連鎖到下游實作。

Generator（生成者）
按 Sprint 逐一實作功能。技術堆疊：React + Vite + FastAPI + SQLite/PostgreSQL。每個 Sprint 結束自評後交給 QA。擁有 Git 版本控制。

Evaluator（評估者）
用 Playwright MCP 以使用者角度點擊測試運行中的應用，檢查 UI 功能、API 端點、資料庫狀態。對每個 Sprint 按標準評分——Product Depth、Functionality、Visual Design、Code Quality。每個標準有硬門檻，任一未達標即 Sprint 失敗。

Part V: Sprint Contract 與代理人間溝通

每個 Sprint 開始前，Generator 和 Evaluator 會協商 Sprint Contract——在寫任何程式碼之前，先同意這塊工作「完成」長什麼樣。Product Spec 刻意保持高階，Sprint Contract 填補了使用者故事與可測試實作之間的鴻溝。
"The generator proposed what it would build and how success would be verified, and the evaluator reviewed that proposal."
溝通透過檔案處理：一個代理人寫檔案，另一個讀取並在該檔案或新檔案中回應。這讓工作忠於 Spec，同時不過早指定實作細節。
"Communication was handled via files: one agent would write a file, another agent would read it and respond."
💡 比喻說明：Sprint Contract = 施工前的圖審

這就像 CDMO 在批次生產前的 Batch Record Review。你不會讓操作員自己決定「完成」是什麼意思——QA 和生產會先同意驗收標準。Sprint Contract 就是代理人版本的「先定義 Acceptance Criteria，再開始施工」。

Sprint 3 光是 Level Editor 就有 27 項驗收標準——粒度細到個別工具的行為預期。

Part VI: 實驗結果 - Solo vs 完整 Harness

使用「建立一個 2D 復古遊戲製作器」的 Prompt 對比測試。Solo 模式 20 分鐘 $9 完成，完整 Harness 6 小時 $200。成本超過 20 倍，但品質差異立即可見。
"The harness was over 20x more expensive, but the difference in output quality was immediately apparent."
Solo 版本：佈局浪費空間、工作流程僵硬、最關鍵的是遊戲本身是壞的——Entity 出現在畫面上但不回應輸入，Entity 定義和遊戲運行時之間的連線斷裂。
"My entities appeared on screen but nothing responded to input. The wiring between entity definitions and the game runtime was broken."
Harness 版本：Planner 將一句 Prompt 展開為跨 10 個 Sprint 的 16 功能 Spec。包含 Sprite 動畫系統、行為模板、音效、AI 輔助 Sprite 生成器和關卡設計師、遊戲匯出。核心功能（遊戲可以玩）確實運作。
"The spec called for a sprite animation system, behavior templates, sound effects and music, an AI-assisted sprite generator..."

⚠️ 重點提示：Evaluator 校準的困難

原生狀態的 Claude 是糟糕的 QA Agent。早期測試中，Evaluator 會辨識出合理的問題，然後說服自己這些不是大問題就通過了。也傾向表面測試而非探索邊界案例。

校準方法：閱讀 Evaluator 的日誌，找出判斷與作者不一致的地方，更新 QA 的 Prompt 來修正。需要好幾輪迭代才讓 Evaluator 的評分合理。即使校準後，小佈局問題、不直覺的互動、和深層功能的未發現 Bug 仍然存在。

Part VII: 簡化 Harness - 隨模型升級調整架構

第一組結果令人鼓舞，但也笨重、緩慢、昂貴。下一步是找到簡化 Harness 而不降低效能的方法。核心原則：Harness 中的每個元件都編碼了一個關於模型「做不到什麼」的假設，這些假設值得壓力測試——因為它們可能不正確，也因為隨著模型改進它們會快速過時。
"Every component in a harness encodes an assumption about what the model can't do on its own, and those assumptions are worth stress testing."
Opus 4.6 發布後提供了進一步簡化的動機。官方描述：「規劃更謹慎、代理任務持續更久、在更大的程式庫中更可靠運作、有更好的程式碼審查和除錯能力來抓自己的錯誤」——這些全是 Harness 原本設計來補充的能力。
"[Opus 4.6] plans more carefully, sustains agentic tasks for longer, can operate more reliably in larger codebases..."
作者首先完全移除了 Sprint 結構。Opus 4.6 可以不需要這種分解就原生處理工作。Evaluator 改為建構結束後的單次評估，而非逐 Sprint 評分。
"I started by removing the sprint construct entirely... the model could natively handle the job without this sort of decomposition."
📚 核心概念：Tool Decay 原則

這段完美呼應了我們的 Tool Decay 原則。Harness 元件就像 Prompt 中的規則——隨著模型能力提升，先前必要的約束變成限制。

實際案例：

Sprint 結構在 Opus 4.5 時是必要的（模型無法長時間保持連貫），但在 Opus 4.6 時變成不必要的開銷。Evaluator 的價值也隨模型能力移動：當任務在模型原生能力範圍內時，Evaluator 是不必要的開銷；但當任務在能力邊界上時，Evaluator 仍然提供真正的提升。

🛠️ 實務應用：Harness 簡化方法論

方法一（失敗）：大幅削減 Harness 並嘗試創新想法。結果：無法複製原始效能，也難以判斷哪些元件真正承載負荷。

方法二（成功）：逐一移除元件，觀察每個移除對最終結果的影響。系統性消去法。

這與 Anthropic 的「Building Effective Agents」原則一致：找到最簡單的可行方案，只在需要時增加複雜度。

Part VIII: 更新後的 Harness 結果 - DAW 案例

用更新的 Harness 測試：「在瀏覽器中使用 Web Audio API 建立一個完整功能的 DAW（數位音訊工作站）」。運行約 4 小時，成本 $124.70。大部分時間花在 Builder 上——連續運行超過兩小時而不需要先前 Opus 4.5 所需的 Sprint 分解。
"Most of the time went to the builder, which ran coherently for over two hours without the sprint decomposition."
QA Agent 仍然抓到了真正的缺口。第一輪回饋指出：應用看起來很強但多個核心 DAW 功能僅為顯示用，沒有互動深度——Clip 無法在時間軸上拖動、沒有樂器 UI 面板、沒有視覺化效果編輯器。第二輪仍發現：音訊錄製為 stub、Clip 調整大小未實作、效果視覺化僅為數值滑桿而非圖形化。
"The QA agent still caught real gaps... several core DAW features are display-only without interactive depth."
最終應用有所有核心部件：可運作的 Arrangement View、Mixer、Transport。透過 Prompt 就能組成一段簡短歌曲：Agent 設定節拍和調性、鋪旋律、建鼓軌、調混音器音量、加殘響。
"The core primitives for song composition were present, and the agent could drive them autonomously."

⚠️ 重點提示：Evaluator 的動態價值

Evaluator 不是固定的 yes/no 決策。其價值取決於任務是否超出當前模型可靠獨立完成的範圍。

在 4.5 上，邊界很近——大部分建構都在 Generator 能獨立做好的邊緣，Evaluator 捕捉了有意義的問題。在 4.6 上，邊界往外移——先前需要 Evaluator 的任務現在 Generator 自己就能處理。但對於仍在能力邊緣的部分，Evaluator 繼續提供真正的提升。

Key Takeaways

🎯 Generator-Evaluator 分離是關鍵槓桿

讓生成者自我批評幾乎不可能——LLM 天然傾向寬容自己的產出。但校準一個獨立的 Evaluator 使其嚴格，比讓 Generator 自我批判要容易得多。這是 GAN 結構的核心洞察。

🛠 Harness 元件應隨模型升級重新審視

每個元件都是對模型弱點的假設。Opus 4.5 需要 Sprint 分解和 Context Reset；Opus 4.6 不需要。逐一移除元件測試影響，比一次大幅重寫更有效。

📈 主觀品質可以被操作化

「這個設計好不好？」無法一致回答，但「這是否符合我們的設計原則？」可以。四維度評分標準加上 few-shot 校準，讓主觀判斷變得可靠且可重複。

💰 成本是真實的但可能值得

完整 Harness 比 Solo 貴 10-20x（$9 vs $124-200），但差異是「核心功能壞掉」vs「完整可用的應用」。對需要可靠輸出的場景，Evaluator 的邊際成本遠低於人工 debug。

Practice Questions

Q1 概念理解 為什麼「分離 Generator 和 Evaluator」比「讓 Generator 自我評估」更有效？請從 LLM 的行為特性解釋。

LLM 在評估自己產出的內容時有天然的寬容偏差——傾向找到理由說服自己作品是好的，即使品質明顯平庸。這不僅是提示問題，而是模型在生成與評估共享同一上下文時的結構性傾向。分離後，Evaluator 沒有「自我辯護」的動機，可以被獨立校準為嚴格的評審。校準獨立 Evaluator 比改變 Generator 的自我認知要容易得多——你只需調整 Evaluator 的 Prompt 和 few-shot 範例，而不需要改變 Generator 的整個生成行為。

Q2 產業應用 在 CDMO 環境中，這篇文章的「Sprint Contract」概念如何對應到你已有的品質系統？可以應用在哪些流程？

Sprint Contract 直接對應到 CDMO 的批次記錄預審和 Acceptance Criteria 定義。在 GMP 環境中，每次生產前 QA 和製造會同意：什麼算「合格」、要測哪些參數、CQA/CPP 的限值是什麼。這就是 Sprint Contract 的實體版本。

可應用場景：(1) 用 AI Agent 生成偏差調查報告初稿前，先定義驗收標準（覆蓋率、根因分析深度、CAPA 建議品質）(2) 技術轉移文件的分階段審查，每階段先同意什麼算「完成」(3) 變更控制的影響評估自動化，Generator 產出評估後由 Evaluator（可以是另一個 AI 或人類 QA）驗證完整性。但關鍵差異：在 GMP 中，Evaluator 必須是人類或人類監督的系統——純 AI Evaluator 在 GMP 關鍵操作中不被接受（Copilot Mode 原則）。

Q3 批判思考 文章宣稱 Evaluator 「caught real gaps」——但 Evaluator 本身也是 LLM。你認為這個架構的品質天花板在哪裡？什麼時候人類必須介入？

品質天花板至少有三個來源：(1) Evaluator 的品味上限——它只能按照被校準的標準評分，無法發現標準之外的問題。作者自己提到 Evaluator 校準需要「好幾輪迭代」閱讀日誌找分歧，這個校準本身依賴人類的判斷力。(2) 測試深度的限制——Evaluator 透過 Playwright 操作，但深層嵌套功能和邊界案例仍然遺漏。DAW 案例中「音訊錄製為 stub」到第二輪才被發現，暗示 Evaluator 的測試覆蓋率有限。(3) 產品直覺的缺失——作者指出兩個版本都沒有引導使用者「先建 Sprite 再建關卡」的 onboarding 流程，這是產品設計直覺的問題，不是程式碼 Bug。目前的 Evaluator 標準沒有覆蓋 UX 流程設計。

人類必須介入的時機：(1) 定義和校準評分標準 (2) 需要產品直覺而非技術正確性的判斷 (3) 安全關鍵或合規關鍵的決策 (4) 當 Evaluator 的回饋開始重複或停滯時。

🎯 我的行動結論
✔ 核心洞察
Generator-Evaluator 分離不只是一個 AI 架構模式——它是對 LLM 自我評估偏差的結構性解方。這個洞察對我在 CDMO 中設計 AI 輔助流程有直接意義：任何讓 AI 產出內容的場景，都應該預設分離「產出」和「品質檢查」角色，而不是讓同一個模型做兩件事。「Sprint Contract」= 先定義 Acceptance Criteria 再動工，這在 GMP 環境中已是標準做法，現在可以延伸到 AI-assisted 流程。
📈 立即應用
在下一個教育文件生成流程中，測試「Generator + 獨立 Evaluator 檢查 meta tag 完整性和 Bottom Line 品質」的兩步架構。具體做法：讓 Claude Code 的 headless mode 生成文件（Generator），再用第二次呼叫檢查生成結果是否符合 Master Prompt 的標準（Evaluator）。如果第二步發現缺失就自動補正。這是最小可行的 Generator-Evaluator 實驗。
⚠ 需要補充
