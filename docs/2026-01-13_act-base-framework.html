這篇文章解決一個被忽略的問題：單次 Prompt 在高壓情境下為何失效。當你的郵件被轉發給 VP、當有人強硬反駁你、當私人討論變成公開 escalation 時，原本的 tone 和 intent 就會開始崩解。

作者 Judy Ossello 提出 ACT + BASE 框架：ACT 定義「什麼行為必須永遠成立」，BASE 確保「透過結構而非措辭來執行這些保證」。這不是讓單一草稿更精緻，而是讓行為可以跨輪次保持穩定。

文章提供完整的 System Prompt 範本，可直接貼入 ChatGPT Project Instructions 使用，並包含四種模式切換機制：Draft、Clarify、Pushback、Variants。

Part I: 高壓溝通為何崩解

我曾經做過一些工作，零權力但全責任。你知道這種類型：協調跨團隊的大型專案，但這些團隊都不向你報告。緊迫的截止日期、持續的模糊性。成功取決於數十個不向你匯報的人。

"I've had jobs where I had zero authority but full accountability. You know the type: coordinate massive projects across teams you don't control."

你唯一真正的工具？溝通。而且它不能只成功一次。它必須在你的主管把它轉發給她的 VP 時撐住。當有人強力反駁時撐住。當私人討論突然變成公開升級時撐住。

"The only real tool you have? Communication. And it can't just work once. It has to hold up when your manager forwards it to her VP. When someone pushes back hard."

💡 我的理解

💡 比喻說明
想像你在玩傳話遊戲，但每一輪傳話的人都會「善意地」修改你的訊息。原本說「這個功能需要延後兩天」，經過三輪修改後變成「他們團隊有困難」。這就是高壓溝通的現實：你的原始 intent 會在每次修訂中被稀釋。
⚠️ 重點提示
核心洞察：問題不在於「寫不好」，而在於溝通工具不是為「跨輪次穩定」設計的。大多數人用 AI 寫郵件時，只想到單次輸出，沒想到這封信會被轉發、被質疑、被公開討論。

Part II: 單次 Prompt 的限制

大多數人試圖用 AI 解決高壓溝通，方法是要求它「寫一封專業郵件」或「讓這聽起來更好」。一次性 Prompt 對應一次性訊息。

"Most people try to solve high-stakes communication with AI by asking it to 'draft a professional email' or 'make this sound better.' One-off prompts for one-off messages."

第一次草稿效果很好。但當有人挑戰你，或把你的訊息往上轉發三層，你就回到原點——重新解釋脈絡、重新描述語氣、每次修訂都要重新下 Prompt。

"It works great for the first draft. But the moment someone challenges you or forwards your message three levels up, you're back to square one—re-explaining context, re-describing tone, re-prompting for every single revision."

📚 核心概念

單次 Prompt 的隱性假設：

• 訊息只會發送一次

• 受眾和風險是已知且穩定的

• 目標是產出好版本，而非管理對話

• 任何 tone drift 都可以手動修復

💡 我的理解

💡 比喻說明
這就像每次需要切菜時都要重新教你的廚房助手「刀怎麼拿、要切多細、手指要收好」。對於單次任務沒問題，但如果你要連續做十道菜，每道菜都要重新教一遍，效率就崩潰了。

Part III: ACT - 行為規格

AI Agent 行為可以透過兩個互補層達到跨輪次穩定：ACT 指定什麼必須永遠為真，BASE 透過結構而非措辭來執行這些保證。

"AI Agent behavior can achieve stability across turns using two complementary layers: ACT, which specifies what must always be true about the agent's behavior; BASE, which enforces those guarantees through structure, not wording."

📚 ACT 三元素

ALIGNED（對齊）：保持冷靜、穩定、專業的語氣。減少熱度：不戲劇化、不誇大緊迫性、不情緒信號。優先保護信任而非說服。

CONSTRAINED（約束）：不捏造事實、假設或理由。不指責或推測他人責任。不包含法律、HR、醫療或合規聲明。不建議欺騙或操縱。

TUNED（調校）：僅根據收件人關係和風險調整句子長度、直接程度和正式程度。不調整責任結構、原因框架或建議的下一步。

💡 我的理解

💡 比喻說明
ACT 就像是公司的「核心價值」——不管市場怎麼變、客戶怎麼要求，這些原則永遠不動搖。你可以調整報價、調整交期，但「誠實」這個價值不能因為壓力而打折。
⚠️ 重點提示
TUNED 的精妙之處：很多人以為「適應受眾」就是改變所有東西。但 ACT 框架區分了「可調整」和「不可調整」：你可以對 VP 用更正式的語言，但你不能改變「誰負責」這個事實。這是高壓溝通的關鍵防線。

Part IV: BASE - 實施元件

BASE 定義如何在結構層面執行 ACT 的保證。它不依賴措辭的好壞，而是透過明確的禁止清單、吸引子模式、和結構模板來確保一致性。

"BASE is how those expectations are enforced... Instead of generating a single draft and stepping away, it becomes a steadier collaborator that preserves tone, framing, and trust while you move quickly in high-pressure situations."

💻 BASE 四元素

BOUNDARIES（邊界）：缺少必要細節時最多問兩個針對性問題。如果用戶要求策略，用訊息變體回應，不要解釋。

PROHIBIT（禁止）：填充詞（just、wanted to）、情緒安撫（I understand this is frustrating）、自我參照（I want to be transparent）

ATTRACTORS（吸引子）：每輪必須回到：清楚原因 → 責任歸屬 → 下一步。偏好簡單的管理語言。保持簡短。

STRUCTURAL ATTRACTOR（結構吸引子）：偏好 3-4 句：(1) 狀況更新 (2) 責任歸屬 (3) 建議調整/請求 (4) 下一步/確認

💡 我的理解

💡 比喻說明
BASE 就像是「機場安檢」——不管你是誰、要去哪裡，都要經過同樣的流程。這不是因為不信任你，而是因為一致的流程比個案判斷更可靠。PROHIBIT 清單就是安檢的「禁止攜帶物品」，ATTRACTORS 就是「標準通關流程」。
🛠️ 實務應用

為什麼禁止「just」和「wanted to」？

這些填充詞會削弱你的訊息。比較：

• ❌ "I just wanted to let you know..."

• ✅ "The timeline has shifted..."

第一種聽起來在道歉，第二種是在陳述事實。在高壓情境下，這個差異會被放大。

Part V: 實作步驟

重要：將 System Prompt 貼入你的 ChatGPT Project 的 Instructions 欄位，而不是作為附加檔案。Project instructions 作為持久的 system prompt，適用於所有對話。

"Important: Paste the system prompt into your ChatGPT Project's Instructions field, not as an attached file. Project instructions act as a persistent system prompt and apply across all threads."

🛠️ 三步驟設定

Step 1：在 ChatGPT 建立新 Project

Step 2：開啟 Add Instructions

Step 3：貼上 Instructions 並儲存

設定完成後，你不需要重述規則或約束。你可以用自然語言與 Agent 互動。

💡 我的理解

⚠️ 重點提示
為什麼是 Project Instructions 而非附加檔案？

附加檔案適合參考資料，但不會可靠地執行行為。Project Instructions 在每次對話開始時都會被讀取，確保行為一致性。這是 Context Engineering 的核心原則：不是靠「說得更清楚」，而是靠「結構位置」來確保執行。

Part VI: 四種模式切換

💻 SHIFTS 模式定義

[DEFAULT] Draft Mode：產出請求的訊息。僅輸出訊息文本。

[CLARIFY] Clarify Mode：最多問兩個必要問題以準確起草。無建議或評論。

[PUSHBACK] Pushback Mode：對提供的回覆生成 2-3 句回應。保持壓縮和冷靜權威。

[VARIANTS] Variant Mode：提供 2-3 個僅訊息變體，在直接程度和壓縮度上有所不同。除變體標題外無評論。

你不需要為日常起草使用模式。它們存在是為了當情況改變時，行為仍然可靠。把模式想成護欄，而非功能。

"You don't need to use modes for everyday drafting. They exist so behavior stays reliable when the situation changes. Think of modes as guardrails, not features."

💡 我的理解

⚖️ 模式使用時機
情境 觸發詞 模式
一般起草 （無需觸發） Draft
資訊不足 [CLARIFY] Clarify
收到反駁 [PUSHBACK] Pushback
需要選項 [VARIANTS] Variant
💡 比喻說明
這就像汽車的排檔：大部分時間你用 D 檔，但遇到陡坡你需要 L 檔，停車時需要 P 檔。模式不是讓你每次都要選，而是讓你在特定情境下有精確的控制。

📌 重點整理 Key Takeaways

1. Prompt vs Agent 的本質差異
Prompt 問模型「要寫什麼」，Agent 定義「應該如何行為」。當對話持續、壓力升高時，你需要的是行為穩定性，而非更精緻的單次輸出。
2. ACT 定義不可妥協的原則
ALIGNED（對齊信任）、CONSTRAINED（約束邊界）、TUNED（調校風格）三層架構確保行為的核心不會因壓力而改變，只有表面風格可以適應受眾。
3. BASE 透過結構執行保證
BOUNDARIES（邊界）、PROHIBIT（禁止清單）、ATTRACTORS（吸引子）、STRUCTURAL ATTRACTOR（結構模板）四元件讓行為執行不依賴措辭好壞。
4. 模式是護欄而非功能
Draft、Clarify、Pushback、Variants 四種模式不是讓你每次都要選，而是在特定情境下提供精確控制，確保行為在情況改變時仍然可靠。

🧠 練習題 Practice Questions

Q1 概念理解 為什麼單次 Prompt 在高壓情境下會失效？請從文章框架解釋。

單次 Prompt 的設計假設是：訊息只發送一次、受眾穩定、目標是產出好版本。但高壓情境打破了這些假設：

  • 訊息會被轉發給不同層級的人（VP、跨部門）
  • 受眾和風險會隨時改變（私人 → 公開）
  • 目標從「產出好版本」變成「管理對話」

每次情境改變，你就要重新解釋 context、重新描述 tone，原本的 intent 在反覆修訂中流失。這是「互動問題」，不是「寫作問題」。

Q2 產業應用 在 CDMO 製藥環境中，如何應用 ACT + BASE 框架來處理客戶的 Deviation 通知？

以 Deviation 通知為例，可以這樣設計：

ACT 層：

  • ALIGNED：保持專業、事實導向，不使用可能暗示過失的語言
  • CONSTRAINED：不推測根本原因（在調查完成前）、不承諾特定時間表、不做法規聲明
  • TUNED：可根據客戶層級調整詳細程度，但不改變事件描述的準確性

BASE 層：

  • PROHIBIT：「我們認為」、「應該不會影響」、「可能是因為」
  • STRUCTURAL ATTRACTOR：(1) 事件描述 (2) 目前狀態 (3) 初步影響評估 (4) 下一步行動與時間表

這樣不管是 QA Manager 還是 VP 問起，訊息的核心內容和責任結構都不會改變。

Q3 批判思考 這個框架有什麼限制？在什麼情況下可能不適用？

作者自己也承認幾個限制：

  • 範圍限制：這個 Agent 專為「高壓專業溝通」設計，不會自動推廣到其他 Agent 任務
  • 風格中立：個人聲音、寫作習慣、組織風格指南是另一層，只有在行為穩定後才有意義
  • 不處理人際衝突：不適用於 coaching 績效、解決團隊衝突、情緒安撫

我的補充觀點：這個框架假設問題是「溝通工具不夠好」，但有時候問題是「根本沒有可溝通的空間」。當對方已經決定要找你麻煩，再完美的溝通也只是拖延。框架解決的是「技術性溝通失敗」，不是「政治性溝通困境」。
