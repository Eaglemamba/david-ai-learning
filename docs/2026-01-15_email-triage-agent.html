Part I：為什麼需要 Proactive Agent？

從 Reactive 到 Proactive 的關鍵轉變

作者在前一篇文章中展示了如何建立一個連接 Email、Calendar 和 Notion 的 on-demand assistant——一個當你詢問時才回應的助手。

"In the previous post, I showed you how to build a personal AI agent that connects your email, calendar, and Notion tasks—an on-demand assistant that responds when you ask."

但那個 Agent 有個問題：它是被動的 (Reactive)。你仍然必須檢查收件匣，告訴 Agent 該做什麼。你仍然是瓶頸。

"But that agent had a problem. It was reactive. You still had to check your inbox and tell the agent what to do. You were still the bottleneck."

作者真正需要的是：一個監控收件匣並自動處理到達郵件的 Agent。

"Here's what I actually needed: an agent that watches my inbox and processes emails automatically as they arrive."

核心概念

Reactive Agent（被動式）：等待使用者指令才行動。你問「幫我整理信箱」，它才開始工作。

Proactive Agent（主動式）：持續監控事件，自動觸發行動。不需要你介入，它就完成分類、標籤、回覆草稿。

比喻說明

想像你有一位秘書。Reactive 秘書坐在辦公室等你走過去說「幫我處理這堆信」。Proactive 秘書會自己去收發室、分類信件、把需要你簽名的放在桌上、不重要的直接歸檔，你只需要看「今日待辦摘要」。

重點提示

Proactive Agent 的價值不在於「做更多」，而在於移除你作為瓶頸的角色。你從「處理者」變成「審核者」。

Part II：Email Triage Agent 的設計原則

不是自動分類到資料夾，而是智慧標籤 + 有限度的自動回覆

作者明確指出這個 Agent 不是什麼：不是把所有東西分到資料夾（重要郵件會被埋沒）、不是未經審核就自動回覆（你會發蠢話）。

"Not one that sorts everything into folders (that's how important emails get buried). Not one that auto-replies without approval (that's how you send something stupid)."

這個 Agent 的核心功能：(1) 當未讀郵件到達時自動執行、(2) 分析每封郵件（寄件者、主旨、內容）、(3) 根據分類規則標籤郵件、(4) 發送處理過的郵件摘要。

"But an agent that: Runs automatically when unread emails hit my inbox; Analyzes each email (sender, subject, content); Applies my triage rules to categorize and label them; Sends me a summary email with everything processed."

作者早上收到的摘要郵件包含：3 封 URGENT（需要回應的會議邀請）、2 封 NEWSLETTER IDEAS（值得探索的讀者問題）、1 封 FOLLOW-UP REQUIRED、4 封歸檔。

"3 emails labeled URGENT (meeting invites that need responses); 2 emails labeled NEWSLETTER IDEAS (reader questions worth exploring); 1 email labeled FOLLOW-UP REQUIRED (partnership inquiry); 4 emails archived."

核心概念

智慧標籤 vs 自動資料夾：標籤保留郵件在原位，只是加上分類標記；資料夾則會移動郵件。標籤的好處是你仍然可以看到完整收件匣，但有視覺提示告訴你優先順序。

重點提示

Draft Reply（草稿回覆）的設計哲學：Agent 只「準備」回覆，不「發送」回覆。這是 AI 自動化的安全閥——你保留最終審核權，避免 AI 代你發出不當內容。

這正是我之前學到的 Copilot Model：100% 人工審核，適合高風險場景。

實務應用

在製藥 CDMO 環境中，這種「草稿不自動發送」的設計尤其重要。任何對外溝通（客戶、法規單位）都需要人工審核，避免資訊洩漏或承諾不當。

Part III：System Prompt 的六層標籤邏輯

從 URGENT 到 READ LATER 的決策樹設計

作者的 System Prompt 定義了六個標籤類別，並有明確的優先順序決策樹：

"For each email, you must decide on one primary label and apply it via the Gmail Add Label to Email tool."

URGENT：只用於「今天真正時間緊迫」的情況。關鍵詞包含 "today", "by end of day", "ASAP", "deadline today"。

"Use only if genuinely time-critical for TODAY... Email contains time-sensitive phrases: 'today', 'by end of day', 'EOD', 'ASAP'..."

FOLLOW-UP REQUIRED：需要回應但不急於今天。包含直接問題、會議安排、文件請求等。

"Use when: The email needs a response or action, but not strictly today... Direct questions that need answers, Meeting/scheduling requests..."

NEWSLETTER IDEAS：對於內容創作者特別設計，任何可以啟發 Newsletter 內容的郵件。

"Use when the email contains content that could inspire newsletter material... Interesting AI tools, case studies, or implementation examples..."

PROMOTIONAL / CHECK LATER / READ LATER：分別處理行銷郵件、陌生人的 Cold Outreach、以及資訊性但非緊急的內容。

"PROMOTIONAL: Marketing/sales emails, product announcements... CHECK LATER: Cold outreach from unknown senders... READ LATER: Informative but not action-oriented..."

核心概念

Decision Priority Order（決策優先順序）：當多個標籤都可能適用時，System Prompt 定義了明確的判斷順序：

1. 寄件者是否未知且明顯是 Cold Outreach → CHECK LATER

2. 是否需要今天做決定或行動 → URGENT

3. 是否需要回應但不急 → FOLLOW-UP REQUIRED

4. 是否是行銷/銷售郵件 → PROMOTIONAL

5. 內容是否能啟發 Newsletter → NEWSLETTER IDEAS

6. 預設 → READ LATER

比喻說明

這就像急診室的 Triage（檢傷分類）系統：不是「誰先來誰先看」，而是根據嚴重程度分配資源。紅色（立即處理）、橙色（緊急但可等）、黃色（需要關注）、綠色（可延後）。你的收件匣也需要這種系統。

重點提示

每封郵件只套用一個標籤：這是刻意的設計。多重標籤會造成認知負擔，你最後還是要決定「這封信到底屬於哪類」。單一標籤強制 Agent 做出判斷。

注意：你必須先在 Gmail 中建立這些標籤，否則 Agent 無法套用。

Part IV：Context 個人化設定

讓 Agent 理解你的工作情境、重要聯絡人、個人偏好

作者強調可以在 Context 區塊加入更多關於你自己的資訊。例如，你可能想讓特定網域的郵件獲得不同處理，因為他們是同事或主管。

"Before we continue, you can use the Context feature to add more information about yourself. For example, you might want to treat emails from specific domains differently because they're from colleagues or managers."

作者的個人範例：他建立 Newsletter「The AI Maker」，所以告訴 Agent 任何「有人按讚我的文章」的通知都自動歸檔，不要混在重要郵件中。

"For instance: I receive Substack notifications whenever someone likes my post. I've told the AI agent that I'm building my newsletter, The AI Maker, and that any email indicating someone liked my post should be moved to 'Archive'..."

實務應用

Context 設定範例（製藥 CDMO 情境）：

• 來自 @client.com 網域的郵件：預設 FOLLOW-UP REQUIRED，若主旨含「production」或「批次問題」則 URGENT

• 來自內部 QA 部門：若含「deviation」或「OOS」則 URGENT

• 法規更新通知：標記為 READ LATER 但加星號

• 設備供應商的行銷郵件：PROMOTIONAL

重點提示

Context 是 Agent 個人化的關鍵：System Prompt 定義「通用規則」，Context 定義「你的規則」。兩者結合才能讓 Agent 真正理解你的工作情境。

這也是為什麼同一個 Agent 架構，不同人用會有不同效果——Context 的深度決定了準確度。

Part V：Make.com Scenario 建構步驟

從 Gmail Watch 到 HTML Summary 的完整流程

Step 1：監控郵件 - 使用 Gmail: Watch emails 模組，監控 Inbox 資料夾、只處理未讀郵件、抓取後標記為已讀、設定每次處理上限（建議從 10 封開始測試）。

"Open a new scenario in Make, then add a starting module called Gmail: Watch emails... I want to simply monitor all emails in the 'Inbox' folder, process only unread messages, mark them as read once fetched, and set a limit of 50."

Step 2：Bundle 郵件 - 使用 Array Aggregator 將所有郵件打包成單一批次，才能一次傳給 AI Agent。否則 Agent 會對每封郵件分別處理並重複發送摘要。

"Next, we want these emails bundled into a single batch for processing so we can pass them all to the AI agent later. To do this, we need an Array Aggregator."

需要打包的欄位：Message ID、Thread ID（重要！用於生成 Draft Reply）、Subject、From (name)、From (email)、Snippet（Gmail 自動生成的摘要）。

"In Aggregated fields, select which parts of the emails you want to pass to the AI agent. Here's the checklist: Message ID, Thread ID (important for generating a draft reply), Subject, From (name), From (email), Snippet..."

重要建議：使用 Snippet 而非 Full text body。Full text body 可能太大，導致 Agent 無法處理所有郵件。

"My suggestion: Use Snippet instead of Full text body to analyze your emails because Full text body can be too large to process, and your AI agent may respond that it can't handle all emails."

核心概念

Array Aggregator 的作用：Make.com 預設是「每筆資料觸發一次後續流程」。當你收到 10 封郵件，沒有 Aggregator 時，AI Agent 會被呼叫 10 次，產生 10 封摘要信。

加入 Aggregator 後，10 封郵件被「打包」成一個陣列，AI Agent 只被呼叫 1 次，產生 1 封完整摘要。

成本分析

Snippet vs Full Body 的 Token 考量：

• Snippet：約 50-100 tokens/封

• Full Body：可能 500-2000+ tokens/封

假設每天處理 50 封郵件：

• Snippet：~5,000 tokens/天

• Full Body：~50,000 tokens/天

OpenAI GPT-4o 定價約 $5/1M input tokens，差異雖不大，但累積起來 Snippet 更經濟且更穩定。

重點提示

Scenario Settings 調整：Gmail 模組每次只能處理 10 封郵件。若你設定處理 50 封，需要在 Scenario settings 將 cycle 從 1 調整為 5，讓流程執行 5 輪。

程式碼/API

AI Agent 的 Additional System Instructions（傳入郵件陣列）：

You are an email triage agent to help me manage my emails and provide HTML summaries.

You will receive an array of email objects. Each email object contains:
- `id`: Gmail message ID
- `subject`: Email subject line
- `from_name`: Sender's name
- `from_email`: Sender's email address
- `snippet`: Snippet summary of the email content
- `thread_id`: Gmail thread ID

Process ALL emails in the array before generating the summary.

EXECUTION INSTRUCTIONS:
1. Iterate through the array of emails
2. For each email: Analyze → Determine label → Apply label via tool → Create draft if URGENT/FOLLOW-UP
3. After processing ALL emails: Compile summary → Generate HTML → OUTPUT as final text response

傳入的 Message（使用 Make.com 變數）：

Here's the array of emails you need to process:

{{4.array}}

注意：{{4.array}} 是 Make.com 的變數語法，代表前一個 Array Aggregator 模組的輸出。

Key Takeaways

這篇文章的四個核心要點

Proactive > Reactive
真正的生產力提升來自移除自己作為瓶頸。Proactive Agent 讓你從「處理者」變成「審核者」，每天只需看一封摘要信。
Draft, Not Auto-Send
Agent 只生成草稿，不自動發送。這是 AI 自動化的安全閥——保留人類審核權，避免發出不當內容。Copilot Model 的典型應用。
決策樹勝過模糊規則
System Prompt 中的六層優先順序（URGENT → CHECK LATER）讓 Agent 有明確的判斷依據，減少邊界情況的誤判。
Context 是個人化關鍵
通用 System Prompt + 個人 Context = 真正理解你工作情境的 Agent。同一架構，不同人用效果截然不同。
練習題
Q1 概念理解 為什麼作者選擇「智慧標籤」而非「自動分類到資料夾」？這兩種方法在使用者體驗上有什麼差異？

答案：自動分類到資料夾會讓郵件「消失」在視野中，重要郵件可能被埋沒在某個資料夾裡而錯過。智慧標籤則保留郵件在收件匣，只是加上視覺標記，讓你仍能看到完整收件匣的同時，快速識別優先順序。

使用者體驗差異：

  • 資料夾：需要主動切換檢視不同資料夾，認知負擔較高，容易遺漏
  • 標籤：單一視窗即可看到所有郵件的優先級，掃描效率更高
Q2 產業應用 如果你要在製藥 CDMO 環境中部署這個 Email Triage Agent，會如何修改六層標籤邏輯？請設計至少 3 個產業特定的標籤規則。

參考答案：

  • REGULATORY URGENT：來自 FDA、TFDA、EMA 的郵件，或主旨含「483」「Warning Letter」「inspection」等關鍵詞，優先於一般 URGENT
  • DEVIATION ALERT：含「OOS」「deviation」「CAPA」「batch failure」的內部通知，需要即時關注
  • CLIENT ESCALATION：來自客戶且主旨含「urgent」「critical」「production hold」的郵件，直接升級為最高優先
  • TECH TRANSFER：關於新產品或技術轉移的郵件，標記為需要追蹤但非緊急

重點是根據產業的「什麼不能等」來定義 URGENT，而非通用的時間敏感詞。

Q3 批判思考 這個 Email Triage Agent 的設計中，有什麼潛在風險或限制是作者沒有提到的？如何緩解這些風險？

潛在風險與限制：

  • 誤判的連鎖效應：如果 Agent 把真正緊急的郵件標為 READ LATER，你可能幾天都不會看到。緩解：定期抽查 READ LATER 標籤，或設定「未回覆超過 X 天」的二次提醒
  • Snippet 資訊不足：有些郵件的重要性藏在附件或郵件中段，Snippet 只抓開頭。緩解：對特定寄件者（如主管、重要客戶）使用 Full Body
  • Draft Reply 品質：AI 生成的草稿可能語氣不對或資訊錯誤，若審核不仔細會發出問題回覆。緩解：草稿只處理「確認收到、會回覆」等低風險回應
  • 隱私與合規：郵件內容傳給 OpenAI API，可能違反企業資料政策。緩解：使用私有部署或確認 API 不訓練資料
