Part I：什麼是 A2UI？核心概念解析
原文摘要
A2UI 讓 AI 代理人能夠在對話過程中判斷何時需要產生可點擊的互動介面，而非繼續純文字對話。 "A text only agent would involve a series of back and forth conversations while a product built with A2UI would understand when and how to generate a UI on the fly to make the UX easier for the user to achieve their goal."
深度解析

想像你和一位超級聰明的助理對話。傳統 AI 助理就像只會用 Line 文字聊天的朋友——你問「幫我訂明天 7 點的餐廳」，它會一個問題一個問題問你：「幾個人？」「什麼餐廳類型？」「有預算嗎？」

A2UI 則讓這位助理變成會即時畫出表單的人：它聽完你的需求後，直接遞給你一張已經填好部分資訊的預約單，你只需要確認或調整即可。這是從「對話式」到「介面式」的典範轉移。

💡 比喻說明

把 A2UI 想成「智慧餐廳服務生」的進化：

傳統服務生（純文字 Agent）：逐一詢問你要點什麼，你用嘴巴回答每一題

A2UI 服務生：聽到「我想要清淡一點的海鮮」後，直接遞給你一個客製化的菜單平板，上面只顯示符合條件的餐點，你用手指點選即可

關鍵差異：Agent 判斷什麼時候「說話」不如「給你工具」更有效率

UX 對比
純文字 Agent：
使用者：「訂明天 5 點 4 人的餐廳」
AI：「好的，請問是哪一天？」
使用者：「明天」
AI：「什麼時間？」
使用者：「我說了 5 點」
"Your app's AI: 'Got it. And for how many people?'"
為什麼這很重要

這個對比揭示了對話式 AI 的根本限制：它把使用者當成「資料輸入源」，而非「決策者」。

在 CDMO 場景中想像：當客戶詢問「我們的 QS-21 可以做 5 萬支的批次嗎」，與其來回確認規格，不如直接顯示一個產能計算器，讓客戶自己調整參數看結果。這是 A2UI 對製藥產業的潛在應用。

📚 核心概念：宣告式 UI（Declarative UI）

A2UI 的技術核心是宣告式 UI設計：Agent 不產生實際的 HTML 或 JavaScript 程式碼，而是產生一份「藍圖」（JSON 結構），描述需要什麼元件，而非如何渲染。

這就像建築師畫設計圖給包商，而不是自己去砌磚——關注點分離（Separation of Concerns）讓系統更安全、更靈活。

Part II：六步驟工作流程深度解析
步驟 1-2：請求與處理
使用者發出自然語言請求，Agent（內部或第三方）接收並處理。Agent 判斷已有哪些資訊、還缺什麼、以及什麼樣的介面最適合這次互動。 "The agent processes the user's intent, determines what information it already has, what it still needs, and what kind of interface would best serve this interaction."
關鍵洞察

這裡有兩個重點：

1. Multi-Agent 架構：你的 App 可以接收外部 Agent 的 UI 指令。想像 OpenTable 的訂位 Agent 可以在你的 App 裡渲染它設計的介面。

2. 意圖萃取：Agent 從「訂明天 7 點 2 人的餐廳」中萃取出三個欄位的預設值（日期、時間、人數），這是 NLU（Natural Language Understanding）的實戰應用。

💻 程式碼/API：JSON 藍圖範例

Agent 產生的 JSON 結構如下：

{
  "components": [
    { "type": "text", "content": "Reservation Details" },
    { "type": "datePicker", "label": "Date", "default": "2026-01-21" },
    { "type": "timePicker", "label": "Time", "default": "19:00" },
    { "type": "numberInput", "label": "Party Size", "default": 2 },
    { "type": "button", "label": "Submit Reservation", "action": "submit" }
  ]
}

注意：這只是資料結構，不是程式碼。Client 端的 Component Library 負責把 datePicker 渲染成實際的日期選擇器。

步驟 3-4：安全性設計
JSON 傳輸到客戶端時只是純資料，沒有可執行程式碼。客戶端只渲染它「認識」的元件類型，惡意 Agent 無法注入任意程式碼。 "There's nothing to execute, no scripts to sandbox. The client receives a shopping list of components to render, and it only renders components it already trusts."
安全架構解析

這是白名單式安全模型的實踐：

傳統 iframe 嵌入第三方內容有 XSS（跨站腳本）風險，因為你讓別人的程式碼在你的網站上執行。A2UI 的做法是：別人只能「點菜」，但「廚房」（Component Library）是你自己的。

在 GMP 合規的角度，這很像「核准供應商清單」——只有通過驗證的元件才能進入系統。

步驟 5-6：元件庫與渲染
客戶端維護自己的元件庫：按鈕、輸入欄、日期選擇器、卡片、圖表等。JSON 中的 type 會映射到這些預先核准的實作。 "A date picker from the agent renders as your date picker, matching the rest of your product's design."
Design System 的價值

這裡凸顯了Design System（設計系統）在 AI 時代的新角色：它不再只是「讓設計一致」的工具，而是Agent 可以操作的「API」。

對產品團隊的啟示：你的 Component Library 的完整性和文件品質，決定了 Agent 能產生多豐富的 UI。這是新的技術債項目。

⚠️ 重點提示：Agent 的 UI 詞彙表

A2UI 的核心限制：Agent 只能使用你定義過的元件類型。

這意味著產品團隊需要思考「什麼樣的 UI 模式是 Agent 可能需要的？」並提前建置。常見的元件類型包括：

  • 輸入類：text input, date picker, time picker, dropdown, checkbox
  • 展示類：card, image, chart, table, map
  • 動作類：button, link, form submit
  • 進階類：file upload, video player, signature pad
Part III：Google 的實際應用案例
案例一：Opal 平台
Opal 是 Google 的 vibe-coding 工具，讓使用者建立 AI 驅動的 mini apps。它已整合進 Gemini 網頁應用，使用者可以創建稱為 Gems 的客製化應用。 "A2UI is foundational to our work. It gives us the flexibility to let the AI drive the user experience in novel ways, without being constrained by a fixed front-end." - Dimitri Glazkov, Principal Engineer
Opal 的意義

Opal 驗證了「AI 原生應用」的新典範：應用程式的 UI 不是預先設計好的，而是AI 根據使用者需求即時組裝的。

這對「低程式碼/無程式碼」平台是重大威脅——為什麼要拖拉元件建立 App，當你可以直接描述你要什麼？

🛠️ 實務應用：Google 內部採用

Google Staff Engineer James Wren 表示，A2UI 讓內部團隊能快速開發「以豐富 UI 為常態」的 Agent，而非「以 UI 為例外」。

關鍵引言："This has been incredibly powerful for internal teams, allowing them to rapidly develop agents where rich user interfaces are the norm, not the exception."

這暗示 Google 內部可能有數十甚至數百個基於 A2UI 的內部工具正在開發中。

案例二：Disco 瀏覽器
Disco 是一個實驗性瀏覽器，可以根據使用者的瀏覽頁籤內容，自動生成互動式 mini apps。例如，在研究旅行時，它可以創建一個客製化的行程規劃器。 "In this example, you can see how researching a trip could involve the creation of custom mini app that lets you narrow down your search and filter activities based on your own preferences."
Context-Aware UI 的願景

Disco 雖然不直接使用 A2UI 協議，但體現了相同的核心概念：UI 應該根據情境動態生成。

想像在 CDMO 情境中：當你在查看多個客戶的批次紀錄時，系統自動生成一個「跨客戶產能比較器」，而不需要你去開一個新的報表。

⚖️ 工具比較：A2UI vs 傳統整合方式
特性 A2UI iframe 嵌入 API + 自建 UI
視覺一致性 高（使用宿主元件庫） 低（第三方樣式） 高（需自行開發）
安全性 高（白名單元件） 中（需 sandbox） 高（完全控制）
開發成本 中（需維護元件庫） 低（直接嵌入） 高（完整開發）
Agent 整合 原生支援 不支援 需自行設計
Part IV：對產品團隊的策略意涵
對設計師的影響
A2UI 將 UX 思維從「規劃螢幕」轉變為「定義 Agent 可組合的 UI 詞彙」。設計師需要設計能在無法完全預測的組合中良好運作的彈性元件。 "For designers, this means thinking less about specific screen layouts and more about designing flexible components that will work well in combinations you can't fully predict."
新的設計技能

這是設計原子化（Atomic Design）的極致演繹：

傳統設計是「頁面導向」——先想好使用者流程，再設計每個頁面。A2UI 時代需要「元件導向」——設計高度通用且可組合的元件，然後讓 Agent 決定如何組裝。

這很像藥品配方研發：你準備好各種原料藥和賦形劑，配方師（Agent）根據需求組合出不同劑型。

對工程師的影響
工程團隊的工作重心從「建構特定畫面」轉移到「建構能詮釋 A2UI 訊息的渲染管線」，並確保元件能優雅處理邊緣情況。 "Engineers would need to ensure components handle edge cases gracefully (things like unexpected data lengths, missing values, error states)."
新的技術債

這裡暗示了一個關鍵的技術投資方向：

1. 元件健壯性：每個元件都要能處理 Agent 可能傳來的「意外」資料

2. 渲染管線：需要一個穩定的 JSON → UI 解析器

3. 錯誤處理：當 Agent 請求未知元件時，系統要能優雅降級

⚠️ 重點提示：「無頭代理人」商業模式

文章提到一個有趣的商業模式：Headless Agent（無頭代理人）

你可以把你的專業 Agent（例如旅行訂票、費用管理、CRM）暴露給其他平台，UI 渲染由合作夥伴的 App 處理。這像是 API 經濟的進化版——你提供智慧，夥伴提供介面。

潛在機會：擁有豐富 Component Library 的平台會成為更有價值的 Agent 宿主，這可能成為新的平台競爭維度。

🛠️ 實務應用：CDMO 情境思考

雖然 A2UI 目前主要是消費級應用的技術，但可以思考製藥產業的潛在應用：

  • 客戶詢價介面：客戶問「能不能做 500L 規模的 mRNA 疫苗？」，Agent 動態生成產能評估表單
  • Tech Transfer 追蹤：根據專案進度，Agent 自動組裝相關的文件清單和核准狀態儀表板
  • 偏差調查輔助：輸入偏差描述後，Agent 產生可能原因的檢核表和調查流程圖

注意：任何涉及 GMP 紀錄的應用都需要符合 21 CFR Part 11 的電子簽章和稽核軌跡要求。

📝 練習題 Practice Questions

Q1 概念理解 為什麼 A2UI 使用 JSON 藍圖而非直接讓 Agent 產生 HTML/JavaScript？這種設計選擇解決了什麼問題？

核心原因：安全性與關注點分離

如果 Agent 直接產生 HTML/JavaScript，會有兩大風險：

  • 安全風險：惡意或被駭的 Agent 可能注入 XSS 攻擊程式碼，竊取使用者資料
  • 一致性問題：不同 Agent 產生的 UI 風格會不一致，破壞使用者體驗

JSON 藍圖是宣告式（Declarative）設計：Agent 只「描述」要什麼（What），Client 決定「如何」渲染（How）。這實現了：

  • 白名單安全：Client 只渲染認識的元件類型，未知指令會被忽略
  • 視覺一致性：所有 UI 都用宿主 App 的元件庫渲染，符合品牌識別
  • 跨平台適應：同一份 JSON 可以在 Web、iOS、Android 上用各自的原生元件渲染
Q2 產業應用 在 CDMO 的 Feasibility Assessment 流程中，A2UI 可以如何應用？需要考慮哪些法規合規要點？

可能的應用場景：

當客戶詢問「我們有一個 monoclonal antibody 產品，需要 10,000L 規模的無菌充填，可以評估嗎？」時，A2UI Agent 可以動態生成：

  • 產品類型選擇器（mAb / vaccine / cell therapy）
  • 規模和批次大小的數值輸入欄
  • 設備需求檢核表（隔離器規格、充填速度）
  • 時程估算表單（Takt time 計算器）

法規合規考量：

  • 21 CFR Part 11：如果生成的 UI 涉及電子紀錄，需要有電子簽章、稽核軌跡、存取控制
  • ALCOA+ 原則：動態生成的表單資料必須是可歸屬（Attributable）、清晰（Legible）、同步（Contemporaneous）、原始（Original）、準確（Accurate）的
  • 驗證要求：元件庫本身需要經過 CSV（Computer System Validation）驗證
  • 風險評估：Agent 的「判斷」（何時顯示什麼 UI）是否影響 GxP 決策？如果是，可能需要更高的驗證層級
Q3 批判思考 Eric Schmidt 預測「UI 將會消失」。請批判這個觀點：在什麼場景下動態生成的 UI 會優於預先設計的 UI？又在什麼場景下傳統 UI 仍然更好？

動態 UI 較優的場景：

  • 高度個人化需求：每個使用者需要不同的欄位組合（如客製化報表）
  • 一次性任務：使用者只用一次的功能，不值得設計完整流程
  • 探索性互動：使用者不確定自己要什麼，需要 AI 引導
  • Multi-Agent 整合：整合多個外部服務，無法預先設計所有可能的 UI

傳統 UI 較優的場景：

  • 高頻率重複任務：使用者需要建立肌肉記憶，固定位置的按鈕更有效率
  • 關鍵操作：涉及金錢、安全、法規的操作，需要經過驗證的固定流程
  • 無障礙需求：螢幕閱讀器使用者依賴穩定的頁面結構
  • 離線環境：無法連接 AI 服務時，需要預先載入的 UI
  • 品牌體驗：高度精緻的視覺設計難以透過 JSON 描述完整呈現

結論：Schmidt 的預測可能過於極端。更可能的未來是混合模式——核心流程用傳統 UI，邊緣情境和個人化需求用動態 UI 補充。
