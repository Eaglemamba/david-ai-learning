Executive Summary

Boris Cherny 是 Claude Code 的創建者。這個最初只是為了學 API 而隨手寫的終端聊天工具，在 2025 年 5 月公開發布後僅 6 個月就達到 10 億美元年化收入，目前貢獻了 GitHub 上 4% 的公開程式碼提交。這篇訪談摘要揭示了三個核心產品理念：潛在需求（Latent Demand）— 不要讓人做新事情，幫他們把正在做的事做得更簡單；為未來的模型構建 — 不要為今天的模型做產品；以及 The Bitter Lesson — 永遠不要跟模型對賭。Boris 從 Opus 4.5 開始 100% 使用 Claude Code 寫程式碼、卸載了 IDE、每天提交約 20 個 PR，並預測「軟體工程師」這個頭銜將逐漸消失。

Part I

從「我在聽什麼音樂」到 AGI 時刻

Boris 說 Claude Code 的誕生完全是偶然。他加入 Anthropic Labs 團隊時的任務是探索某種編碼產品，但他先需要搞懂 API 怎麼用，就隨手寫了個終端聊天 App。選終端的原因很簡單：不用寫 UI，當時就他一個人。
"I chose the terminal because I didn't have to write a UI. It was just me at the time."
當 Anthropic 推出 Tool Use 功能時，Boris 給模型加了一個 bash 工具，直接從文件範例搬過來的。他先讓模型用 cat 讀了個文件，然後問了一句：「我在聽什麼音樂？」Claude 寫了一段 AppleScript，調用他 Mac 上的音樂播放器查了當前播放的歌。
"That was my first real AGI moment. I realized: this thing just wants to use tools, it just wants to interact with the world."
原型做出兩天後給團隊內部測試。第二天早上，同事 Robert 已經在用這個東西寫程式碼了。到了對外發布評審時，Dario Amodei 看到內部日活增長圖幾乎是一條垂直線，問 Boris 是不是在強制工程師使用。Boris 說沒有，只是發了個帖子就傳開了。
"The internal DAU chart was basically a vertical line."
💡 比喻說明
Claude Code 的誕生像是有人為了試新鍋子的手感，隨便炒了盤蛋炒飯，結果這盤蛋炒飯變成了餐廳的招牌菜。Boris 沒有設計產品路線圖，他只是在解決自己的問題。這呼應了 Paul Graham 的名言：做自己想用的東西。
⚠️ 重點提示
Claude Code 到今天還在終端裡運行。Boris 自己都覺得不可思議：「那本來應該是起點，不是終點。」 這反映了一個深層邏輯 — 當模型進步太快時，任何精美的 UI 在六個月後都可能過時。最少的 UI 假設 = 最大的靈活性。
🛠️ 實務應用
早期工程師主要用 Claude Code 自動化 Git 操作和容器管理。第一個真正的編碼用例是寫單元測試，因為風險相對較低。這是一個聰明的進入策略 — 從低風險場景建立信任，再逐步擴展到核心開發工作。
Part II

CLAUDE.md 與 Plan Mode 的極簡哲學

CLAUDE.md 的由來又是一個潛在需求的例子。用戶自發開始寫 markdown 檔案讓模型讀，Boris 觀察到這個行為後才把它產品化。Boris 的個人 CLAUDE.md 只有兩行：第一行是提交 PR 後自動合併；第二行是 PR 提交後發到團隊頻道讓人審批。
"People just started writing markdown files for the model to read. We observed that behavior and productized it."
Boris 的建議很直接：如果你的 CLAUDE.md 太長了，刪掉，從頭來過。模型能力隨著每次更新在變，你需要的是做最少的事情讓模型走上正軌。每個新模型需要的指令越來越少。
"Delete it and start over. You'll find each new model needs fewer and fewer instructions."
Plan Mode 的全部技術實現就是在 prompt 裡加一句「請不要寫程式碼」。Boris 周日晚上 10 點看到用戶反覆要求「先想方案，別寫 code」，花 30 分鐘寫完程式碼，當晚發布。
"Plan Mode has no secret. It's just adding 'please don't write code' to the prompt."
📚 核心概念：Scaffolding 是技術債
Boris 提到圍繞模型構建的「腳手架」(scaffolding) 可以把性能提升 10-20%，但下一個模型出來後這些優化就會被抹平。這也是他們一直留在命令列的原因 — 任何 UI 在六個月後都不會還有意義。這是一個反直覺的產品決策：最少的功能 = 最長的壽命。
🛠️ CLAUDE.md 最佳實踐
Boris 的做法：只放你反覆重複的東西，其他情況直接 prompt 就行。團隊共享 CLAUDE.md 約 2000 token。Boris 每週會多次在別人 PR 上直接 tag Claude，讓它把新規則加進去。

對比 Garry Tan 的 CLAUDE.md 已膨脹到好幾千 token。Boris：「刪掉後如果模型跑偏了再加一點回來。」
Part III

三大產品原則

原則一：潛在需求 (Latent Demand) — 人們只會做他們已經在做的事。你不能讓人做新事情，只能幫他們把正在做的事做得更簡單。Boris 說自己前幾次創業都不理解這個概念。
"People will only do what they're already doing. You can't make people do new things, you can only make what they're doing easier."
原則二：為未來的模型構建 — 不要為今天的模型做產品，要為六個月後的模型做。Claude Code 正式發布時只能寫 Boris 約 10% 的程式碼，但團隊賭對了模型進步方向。如果你只為當前模型最佳化，找到的 PMF 很快就會被別人用下一代模型超越。
"Don't build for today's model, build for the model six months from now."
原則三：The Bitter Lesson — 永遠不要跟模型對賭。他們辦公區牆上裱框了 Rich Sutton 2019 年寫的這篇文章。核心觀點是：利用通用計算的方法最終總是勝過依賴人類知識的手工設計方法。
"Never bet against the model."
💡 The Bitter Lesson 在 Claude Code 的實踐
Boris 說他們隨時面臨一個選擇：投入工程資源加一個 scaffolding 功能，獲得 10-20% 提升；或者等兩個月，下一個模型大概率自己就能做到了。他們總是這樣權衡取捨，並假設任何 scaffolding 都是技術債。

Claude Code 沒有任何部分是六個月前的。它一直在被重寫、重寫、再重寫。80% 的程式碼可能只有幾個月的歷史。
⚠️ TypeScript 的啟示
Boris 十多年前寫了一本 TypeScript 的書。TypeScript 團隊觀察到 JavaScript 程式設計師怎麼寫程式碼，然後圍繞這些行為去設計類型系統，而不是讓程式設計師改變寫法。Claude Code 也是同樣的邏輯：不是原則驅動，而是「我自己想用什麼工具」。
Part IV

新手 vs 老手：初心者心態的力量

Boris 認為跟上模型能力邊界的關鍵是初心者心態和謙遜。高級工程師過去被獎勵「有很強的觀點」，但很多這些觀點現在已經不再適用了。
"Senior engineers were rewarded for having strong opinions, but many of those opinions no longer apply."
他講了一個內存洩漏的故事：Boris 按老辦法來（抓內存快照、開發者工具、看程式碼），團隊另一個工程師 Chris 直接讓 Claude Code 處理。Claude 自己抓了快照、自己寫了分析工具、比 Boris 更快找到了洩漏。
"This is something I have to constantly relearn, because my brain is sometimes still stuck six months ago."
Boris 招人時會問一個簡單問題：「舉一個你犯錯的例子。」他想看候選人能不能承認錯誤、把責任歸到自己身上。他自己估計大約一半的想法是錯的。
"About half of my ideas are wrong. You just have to keep trying."
📚 兩種高效工程師畫像
Boris 說團隊中最高效的人呈兩極分布：

極端專才：如 Jarred Sumner（Bun 創建者）和 Bun 團隊，對 JavaScript 運行時和開發工具的理解比任何人都深。

超級通才：橫跨產品、基礎設施、設計、用戶研究、商業。Boris 喜歡「做奇怪事情的人」。

例子：團隊成員 Daisy 要加新功能，她沒有直接實現，而是先寫了一個讓 Claude Code 能夠測試任意工具的工具，然後讓 Claude 自己去實現了那個新功能。元程式設計思維。
Part V

Agent 拓撲與 Mama Claude

Boris 引出了「Agent 拓撲」(Agent Topologies) 這個概念。核心思想是「不相關的上下文窗口」：多個 Agent 各有獨立的上下文，不會被彼此的上下文「污染」。更多上下文就是更多推理時計算 (inference-time compute)，配合合適的協作結構就能做更大的事。
"Unrelated context windows — multiple agents each with their own independent context, not polluted by each other's context."
Anthropic 的 Plugins 功能完全由一群 Agent 在週末自動協作構建完成。一個工程師給 Claude 一個規格說明 (spec)，讓它用 Asana 的看板來管理。Claude 在 Asana 上創建了一堆任務，然後自動生成了一批子 Agent，這些 Agent 開始自動領取任務執行。
"The main Claude just gave instructions, and they figured it out themselves."
Boris 提到現在大多數 Agent 啟動時其實是由另一個 Claude 來提示的（而非人類），以子 Agent 的形式。在程式碼裡，子 Agent 就是遞迴的 Claude Code。他們內部叫主 Agent 「Mama Claude」。
"Most agents are now prompted by another Claude, not by a human. We call the main agent 'Mama Claude.'"
📚 核心概念：不相關的上下文窗口
傳統做法是把所有資訊塞進一個上下文窗口。Agent 拓撲的突破在於：讓多個 Agent 各自保持乾淨的上下文，只在需要時交換結果。

這就像一個高效的團隊會議：每個人帶著自己的專業知識來，只分享結論和關鍵發現，而不是讓每個人都讀完所有人的研究筆記。
🛠️ Boris 的多 Agent 實踐
Boris 說他一直這麼做：如果任務比較難，他會讓 3 個、5 個甚至 10 個子 Agent 並行研究，然後匯總結果。

更常見的模式：Boris 讓 Claude 做某件事，Claude 看程式碼庫時通過 git blame 發現某個工程師之前改過相關部分，就自動在 Slack 上找那個人問一個確認問題，拿到答案後繼續幹活。
Part VI

1000 倍工程師與「軟體工程師」頭銜的消亡

Anthropic 所有技術員工每天都在用 Claude Code。自推出以來，人均生產力累計增長了 150%。Boris 用自己在 Meta 的經歷做對比：當時 2% 的生產力提升就是幾百人工作一年的成果。150%？聞所未聞。
"At Meta, a 2% productivity improvement was hundreds of people working for a year. 150%? Unheard of."
Boris 個人從 Opus 4.5 開始就 100% 使用 Claude Code 寫程式碼，卸載了 IDE，每天提交約 20 個 PR，不再手動編輯任何一行程式碼。Anthropic 整體的 AI 編碼比例在 70-90% 之間。
"I uninstalled my IDE. I submit about 20 PRs a day. I don't manually edit a single line of code anymore."
Boris 認為程式設計將對所有人「基本解決」。「軟體工程師」這個頭銜可能會消失，變成「builder」或「產品經理」之類的稱呼。工程師不只寫程式碼，還要寫 spec、跟用戶交談。他們團隊已經是這樣了：PM 寫程式碼、設計師寫程式碼、EM 寫程式碼、財務人員也寫程式碼。
"The title 'software engineer' may disappear, becoming 'builder' or 'product manager.'"
外部數據方面：Mercury 統計顯示約 70% 的初創公司選擇 Claude 作為首選模型；SemiAnalysis 估計 4% 的公開程式碼提交來自 Claude Code。Claude Code 還被用於 NASA 火星探測器 Perseverance 的航線規劃。
"4% of public code commits come from Claude Code. It's been used for NASA's Perseverance rover flight planning."
⚠️ Co-Work：非技術人員的 Claude Code
Boris 發現有人用 Claude Code 監控番茄植物、從損壞硬碟恢復婚禮照片、做財務分析。在 Anthropic 內部，所有設計師、整個財務團隊、整個資料科學團隊都在用 Claude Code — 不是為了寫程式碼。

Co-Work 的底層就是 Claude Code，同一個 Agent。Felix 和團隊在大約 10 天內完成構建，100% 由 Claude Code 編寫。

Key Takeaways
👁
潛在需求 > 創造需求
CLAUDE.md、Plan Mode 都不是 Boris 發明的功能，而是用戶已經在做的事。產品開發的核心是觀察，不是想像。對於 CDMO 的 AI 導入，觀察團隊「已經在費力做的事」比構想全新工作流更有效。
🔮
為六個月後的模型構建
Claude Code 發布時只能寫 10% 的程式碼，團隊賭的是模型進步方向。任何 scaffolding 都是技術債。在高度監管環境中，這意味著不要過度投資在「繞過模型限制」的工程上。
⚠
初心者心態是競爭優勢
Boris 的內存洩漏故事：老方法不敵新方法。高級工程師的「強觀點」可能是包袱。承認一半想法是錯的，持續測試和迭代，比堅持過去的經驗更有價值。
🤖
Agent 拓撲是下一個前沿
多個 Agent 各有獨立上下文、自動協作、遞迴子 Agent（Mama Claude）。Plugins 由 Agent 週末自動構建。大多數 Agent 現在由另一個 Claude 提示，而非人類。這是規模化的新範式。
Practice Questions
Q1 概念理解 Boris 說「潛在需求」是他最重要的產品理念。請解釋這個概念，並說明 CLAUDE.md 和 Plan Mode 如何各自體現了這個理念。

潛在需求 (Latent Demand) 的核心是：人們只會做他們已經在做的事。你不能讓人做新事情，只能幫他們把正在做的事做得更簡單。

CLAUDE.md：用戶自發開始寫 markdown 檔案讓模型讀（已有行為），Boris 觀察到後才將其產品化（降低摩擦）。他沒有設計一個複雜的設定介面，而是把用戶已經在做的事變成正式功能。

Plan Mode：用戶反覆在 prompt 中寫「先想方案，別寫 code」（已有行為），Boris 只是把這句話變成一個按鈕（降低摩擦）。技術實現就是在 prompt 裡加一句「請不要寫程式碼」。

兩者的共同模式：觀察用戶的笨拙解決方案 → 將其產品化為一鍵操作。

Q2 產業應用 在製藥 CDMO 的 GMP 環境中，「為未來模型構建」和「不跟模型對賭」這兩個原則會如何影響你的 AI 導入策略？請考慮法規限制。

這兩個原則在 GMP 環境中的應用需要做調適：

「為未來模型構建」的調適：在製藥環境中，不應過度投資在「繞過模型限制」的 scaffolding 上。例如：如果目前模型無法可靠地自動審查批次紀錄，與其花大量工程資源建立複雜的規則引擎來補足，不如先以「copilot 模式」部署，人類審查 + AI 輔助標記，等模型能力提升後再擴展自動化範圍。

「不跟模型對賭」的 GMP 限制：在 21 CFR Part 11 和 ALCOA+ 框架下，不能像 Boris 一樣假設「模型會自動變好」。法規要求每次變更都需驗證。因此策略是：選擇模組化架構（易於升級模型而不重新驗證整個系統），並將 AI 輸出保持在「輔助建議」而非「自動決策」的範圍內。

底線：保持架構彈性，投資在「介面層」而非「補丁層」，讓未來的模型升級成為組態變更而非系統重建。

Q3 批判思考 Boris 預測「軟體工程師」頭銜將消失，取而代之的是「builder」角色。你認為這個預測在什麼條件下成立？什麼條件下不成立？

成立的條件：模型持續快速進步（The Bitter Lesson 繼續有效），程式碼生成品質達到「人類審查即可」的水準，開發工具生態完全整合 AI（IDE、CI/CD、測試框架），以及企業文化接受跨職能角色。Boris 自己的實踐（每天 20 PR、100% AI 編碼）已經展示了這個未來的雛形。

不成立的條件：高安全性/高合規環境（航太、醫療設備、金融交易系統）可能永遠需要能「從頭理解程式碼」的專業人員；模型進步遇到瓶頸（如 scaling laws 失效）；或是出現重大安全事件導致監管收緊，要求人類可追溯地理解每一行程式碼。

更可能的中間態：「軟體工程師」頭銜不會消失，但職責會根本改變 — 從「寫程式碼」變成「設計系統、驗證 AI 輸出、管理 Agent 拓撲」。就像會計師沒有因為 Excel 消失，但職責從「算數」變成了「分析和判斷」。

🎯 我的行動結論
✔ 核心洞察
Boris 的「潛在需求」理念直接呼應了我在 CDMO 推動 AI 導入時的經驗：與其設計全新的「AI 驅動工作流」，不如觀察團隊目前用 Excel 和紙本費力完成的任務，然後讓 AI 介入降低摩擦。這和我的「copilot vs agent」框架完全對齊 — 先觀察，再介入，從低風險場景建立信任。「scaffolding 是技術債」這個觀點也提醒我：不要過度投資在補足目前模型限制的工程上。
📈 立即應用
本週做一次「潛在需求審計」：觀察團隊在哪些任務上重複使用笨拙的解決方案（如手動複製貼上批次紀錄數據、用 email 追蹤偏差狀態）。列出 3 個最明顯的「已有行為但摩擦很高」的場景，這些就是 AI 導入的最佳切入點。同時，重新檢視我的 CLAUDE.md — 如果超過 2000 token，精簡到核心指令。
⚠ 需要補充
這篇訪談主要來自矽谷快速迭代的語境，在 GMP 環境中「不跟模型對賭」需要加上法規緩衝 — 每次模型升級都需要變更管理和驗證。Boris 的「80% 程式碼只有幾個月歷史」在受監管環境中是不可接受的。需要進一步研究如何在保持架構彈性的同時滿足 21 CFR Part 11 的追溯性要求。建議 3 個月後重讀，對照屆時模型能力的變化。
