Part I — 為什麼單一 CLAUDE.md 會讓 AI 變笨

AI 犯錯有三個根源：錯誤的資訊、遺漏的資訊、雜訊。階層式 CLAUDE.md 主要解決的是「雜訊」問題。

AI errors stem from three sources: wrong info, missing info, and noise. Hierarchical CLAUDE.md primarily addresses the "noise" problem.

你在寫前端的時候，後端的 API 格式規範對 Claude 來說就是雜訊。它佔了 Context Window 的空間，卻對當前任務毫無幫助，反而可能讓 Claude 分心。寫越多，Claude 反而越笨。

When writing frontend code, backend API format specs are noise to Claude. They occupy Context Window space without helping the current task, potentially distracting Claude. The more you write, the dumber Claude gets.

💡 比喻說明

想像你是外科醫師，正在做心臟手術。助手同時把骨科手術流程、眼科器材清單、牙科麻醉劑量全攤在手術台上。資訊量很大，但全是雜訊 — 佔空間、分散注意力。

CLAUDE.md 塞滿所有規則，就是這個狀況。Claude 的 Context Window 就是那張手術台，空間有限。

⚠ 重點提示

這個「雜訊」問題在 CDMO 情境中也常見 — 製程驗證時，你不會把所有 SOP 同時攤在 Clean Room 裡。只帶當前步驟需要的文件，這就是「階層式」的核心邏輯。

Part II — 階層式 CLAUDE.md 的實作結構

正確的做法是分層。Claude Code 會自動從當前工作目錄往上遞迴（recursive）載入所有 CLAUDE.md。在 frontend/ 底下工作時，只會載入根目錄 + frontend/CLAUDE.md，後端規則不會被載入。

Claude Code automatically loads all CLAUDE.md files recursively from the current directory upward. Working in frontend/ loads only root + frontend/CLAUDE.md; backend rules won't be loaded.

動態載入：有些東西太大了（教學文件、API 速查表），直接塞進 CLAUDE.md 會爆。更好的做法是把 CLAUDE.md 當成索引 — 只列出「去哪裡找」，Claude 需要的時候再動態載入。

For oversized content (tutorials, API cheat sheets), use CLAUDE.md as an index — list where to find things, let Claude load dynamically when needed.

💻 目錄結構
📚 分層原則

根目錄：每次對話都需要的東西（哲學、命名、Git）

子目錄：該領域專屬規則（前端 Component、後端 API）

判斷標準：某條規則只在特定目錄會用到，就不該放在根目錄浪費全域 Context

project/
├── CLAUDE.md              ← 全域規則
│   ・核心開發哲學
│   ・全域命名規範
│   ・Git commit 慣例
│   ・專案架構概述
├── frontend/
│   └── CLAUDE.md          ← 前端專屬
│       ・Component 寫法
│       ・狀態管理慣例
│       ・UI 框架注意事項
└── backend/
    └── CLAUDE.md          ← 後端專屬
        ・API 格式
        ・資料庫操作慣例
        ・第三方服務整合
Part III — /Lesson：單次 Session 的經驗提取

即便有了階層式架構，Claude 仍然會反覆修不好同一個 Bug、執著於錯誤的 API 用法。你花了力氣把它導正、最終完成任務了，但如果什麼都不做，下次遇到同樣問題它還是會再犯。

Even with hierarchical structure, Claude still repeatedly fails at the same bug or persists with wrong API usage. Without action, it'll make the same mistakes next time.

/Lesson 是 Custom Command，在任務完成後讓 Claude 回頭 review 對話。做兩件事：(1) 回顧對話，分析走錯的原因；(2) 提取經驗，寫入對應層級的 CLAUDE.md。

/Lesson is a Custom Command that makes Claude review the conversation after task completion: (1) analyze what went wrong, (2) extract lessons into the corresponding CLAUDE.md level.

關鍵是「寫入對應層級」— 通用教訓（如「收到明確指令就直接執行」）寫根目錄，特定框架的踩坑寫子目錄。

The key is "write to the corresponding level" — generic lessons go to root, framework-specific pitfalls go to subdirectories.

💡 比喻說明

/Lesson 就像每次看完診後寫的病歷記錄。剛治好一個難症，趁記憶新鮮把診斷過程、踩過的坑、最後有效的處方全寫下來。下次遇到類似症狀，翻病歷就好。

🛠 建立方式
⚖ /Lesson vs /project-insights

/Lesson：處理「單次 session」，精準但依賴人記得去做

/project-insights：處理「跨所有 session」的趨勢，全面但有延遲

兩者互補，不是二選一

# 在專案根目錄建立 Custom Command
mkdir -p .claude/commands/
touch .claude/commands/lesson.md

# lesson.md 裡定義：
# 1. 回顧當前對話的指示
# 2. 分析哪裡走錯的指示
# 3. 將結論寫入 CLAUDE.md 的指示
Part IV — Claude Code 本機存了什麼：對話紀錄與 Facet

Claude Code 默默在本機存了大量資料。對話紀錄（JSONL）存在 ~/.claude/projects/{encoded-path}/{session-id}.jsonl，你跟 Claude 講的每句話、回覆、工具使用、Token 消耗全部都在。

Claude Code silently stores extensive local data. Conversation logs (JSONL) at ~/.claude/projects/{encoded-path}/{session-id}.jsonl contain every message, reply, tool usage, and token consumption.

索引檔（sessions-index.json）幫約 70% 的 session 整理好了基本資料。Facet 則是 /insights 產生的後設分析，標註每個 session 的目標分類、結果、摩擦點、幫助程度。

The index file covers ~70% of sessions with basic metadata. Facets are meta-analyses generated by /insights, tagging each session's goal categories, outcomes, friction points, and helpfulness.

💻 本機資料路徑
📚 Facet 欄位一覽
欄位白話意思範例
goal_categories在做什麼bugfix, code_migration
outcome結果如何fully_achieved, not_achieved
friction_counts卡關摩擦wrong_approach, buggy_code
claude_helpfulness多有幫助very_helpful, essential
session_type工作模式iterative_refinement
⚠ 數據規模參考

作者的一個專案：44 天、260 個 session、整個資料夾超過 600MB，最大單一 session 達 54MB。22,000+ 則使用者訊息，21,000+ 次工具呼叫。

# 對話紀錄
~/.claude/projects/{path}/{session}.jsonl

# 索引
~/.claude/projects/{path}/sessions-index.json

# Facet 後設分析
~/.claude/usage-data/facets/{session-id}.json
Part V — /project-insights：專案級自動審計

Claude Code 官方的 /insights 很強 — 回頭翻 30 天對話紀錄，分析工具使用、卡關點、成功率。但有兩個問題：它分析所有專案混在一起，以及你不知道要改哪個專案的 CLAUDE.md。

Claude Code's official /insights is powerful — analyzing 30 days of history for tool usage, friction points, success rates. But it mixes all projects together, and you don't know which project's CLAUDE.md to fix.

作者自建的 /project-insights 解決這些問題：(1) 只看當前專案、(2) 重用 /insights 已分析的 Facet 不重複花 Token、(3) 直接產出 CLAUDE.md 改進建議。

The custom /project-insights fixes these: (1) scopes to current project only, (2) reuses existing Facet data, (3) outputs concrete CLAUDE.md improvement suggestions.

技術上是兩個檔案：SKILL.md（告訴 Claude 怎麼分析、報告格式）和 extract.py（Python 腳本，從幾百 MB 的 JSONL 濃縮成 ~100KB JSON）。Python 串流處理 260 個 session，3.6 秒搞定。

Technically just two files: SKILL.md (analysis instructions, report format) and extract.py (Python script condensing hundreds of MB of JSONL into ~100KB JSON). Streams through 260 sessions in 3.6 seconds.

💻 /project-insights 結構
📚 extract.py 做什麼

1. 接收專案路徑，找到對應 session 資料夾

2. 先讀索引拿基本 metadata（70% 免解析 JSONL）

3. 串流掃描每個 JSONL：訊息數、工具次數、Token、錯誤

4. 比對哪些 session 已有 Facet

5. 挑 10 個「最值得深入看」的未分析 session

6. 全部彙總成 JSON 輸出

⚠ 10 個 session 怎麼挑

最多訊息的 3 個（最複雜）、最多錯誤的 2 個（卡最久）、最新的 3 個（最近習慣）、最耗 Token 的 2 個（成本最高）

~/.claude/skills/project-insights/
├── SKILL.md         # 分析指令 + 報告格式
└── scripts/
    └── extract.py   # 資料濃縮腳本

# 安裝步驟
mkdir -p ~/.claude/skills/project-insights/scripts
chmod +x extract.py
# 在任何專案目錄打 /project-insights 即可
Part VI — 報告的七個分析區塊 + 實際改進案例

SKILL.md 定義了七個分析區塊：(1) 概覽統計、(2) 工作內容分析、(3) 工具使用模式、(4) 摩擦點分析（最有價值）、(5) Session 結果分佈、(6) 重複模式、(7) CLAUDE.md 改進建議。

SKILL.md defines seven analysis blocks: overview, work content, tool usage, friction analysis (most valuable), session outcomes, repeated patterns, and CLAUDE.md improvement suggestions.

⚖ 摩擦點排名（作者實際數據）
摩擦類型次數改善方向
buggy_code33加入第三方 API 速查表（cheat sheet）
wrong_approach24加強架構說明 + 「直接執行不要產計畫」
misunderstood_request9更明確的指令範本
user_rejected_action8加強操作範圍限制
excessive_changes3明確界定修改範圍
🛠 三個實際改了 CLAUDE.md 的例子

1. Bash > Read → 加「優先使用專用工具」規範
工具分析顯示 Bash(6,128) 多於 Read(5,560)，讀檔該用 Read 不是 cat。加了對照表後 Claude 明顯改善。

2. wrong_approach → 加「直接執行不要產計畫」
收到「fix all 15 findings」還在寫修正計畫。加規則：「收到明確指令時直接執行」。

3. buggy_code → 加第三方 API 速查表
shadcn-vue Checkbox 用舊版語法、Strapi lifecycle hook 行為不同。寫進 CLAUDE.md 後不再重犯。

⚠ 覆蓋率的坑

/insights 每次最多只處理 50 個 Facet（跨所有專案）。作者 254 個 session 跑一次只有 27 個 Facet（11% 覆蓋率）。連續跑多次後：11% → 26% → 45% → 56%（143 個 Facet），分析結果完全不同。花十分鐘多跑幾次是值得的。

📚 改進建議的四種類型

[新增規範] — wrong_approach 太多 → 加強架構說明

[修改規範] — 既有規範沒被遵守 → 改寫得更明確

[新增知識] — 同樣問題反覆出現 → 加入常見陷阱

[效率提升] — 重複任務太多 → 建議自動化

Key Takeaways

🏗 階層式 = 空間隔離

CLAUDE.md 按目錄分層，Claude Code 自動遞迴載入。只讓 AI 在對的時間看到對的規則，保護有限的 Context Window 不被雜訊稀釋。判斷標準：只在特定目錄有用的規則，就不該放根目錄。

🔄 /Lesson = 即時病歷

任務完成後的 Custom Command，讓 Claude 回顧對話、分析走錯的原因、提取經驗寫入對應層級的 CLAUDE.md。精準但依賴人記得做。通用教訓寫根目錄，專屬踩坑寫子目錄。

📊 /project-insights = 定期健檢

Python 腳本濃縮數百 MB 對話紀錄，Claude 分析產出七區塊報告。最有價值的是「摩擦點分析」和「CLAUDE.md 改進建議」。記得多跑幾次 /insights 提高 Facet 覆蓋率。

🎯 PDCA = 自我演化

三層疊起來形成閉環：載入規則 → 協作開發 → 審計數據 → 更新規則。CLAUDE.md 不再是靜態文件，而是會根據數據自我改善的系統。「憑感覺」變成「靠數據」。

Practice Questions

Q1 概念理解 為什麼作者說「寫越多，Claude 反而越笨」？這和 Context Window 有什麼關係？

Claude 的 Context Window 是有限資源。當 CLAUDE.md 塞滿所有規則（前端 + 後端 + 部署 + 命名...），與當前任務無關的規則會佔據 Context 空間，形成「雜訊」。LLM 的注意力機制會被這些無關資訊稀釋，導致對真正重要規則的遵循度下降。

這就像 GMP 文件管理的邏輯 — 你不會把所有 SOP 同時帶進 Clean Room。只帶當前製程步驟需要的文件，確保操作人員專注在對的指引上。階層式 CLAUDE.md 做的就是同樣的「資訊隔離」。

Q2 產業應用 如果你要在 CDMO 的 AI 輔助系統中實作類似的「階層式指令架構」，你會怎麼設計分層？

可以參考這個思路：

根目錄（全域）：GMP 核心原則、數據完整性要求（21 CFR Part 11）、變更管理流程、文件編號慣例 — 不論哪個部門都要遵守的規則。

manufacturing/：製程參數範圍、設備操作 SOP（如 SA25 Vanrx isolator）、環境監測標準、QS-21 佐劑處理流程。

quality/：偏差調查流程、CAPA 分類標準、穩定性試驗規範、放行條件。

regulatory/：PIC/S 指引對照、法規提交格式、各市場 CTD 差異。

核心邏輯一樣 — AI 在處理製程偏差時，不需要載入法規提交格式；在準備提交文件時，不需要載入設備操作細節。

Q3 批判思考 這套 PDCA 循環框架有什麼限制？在什麼情境下可能不適用？

限制 1 — 前提是你要用 Claude Code：整套系統建立在 Claude Code 的本機 JSONL 存儲和 /insights Facet 系統上。如果你用的是 claude.ai 網頁版、API、或其他 IDE 整合，這些本機數據不存在。

限制 2 — Facet 覆蓋率問題：作者自己也坦承，多跑幾次 /insights 才能達到 56% 覆蓋率。未分析的 44% session 可能藏有重要模式。

限制 3 — /Lesson 依賴人的紀律：作者承認「人難免會懶惰或遺忘」。如果團隊裡多人使用，紀律更難維持。

限制 4 — 對 GMP 環境的適用性：在高度管制環境中，AI 生成的「改進建議」不能直接寫入正式 SOP。需要加一個人工審核 + 變更管理的閘門，這在文章中沒有提到。這是 copilot mode（人審 100%）vs agent mode 的判斷點。

📈 立即應用
雖然我主要用 claude.ai 而非 Claude Code，但「階層式架構」的邏輯可以遷移 — 在 Claude Projects 中，我可以把 Master Prompt 拆成「核心教學方法（永遠載入）」和「特定文件類型模板（按需引用）」兩層。本週動手做：把 v1.5 Master Prompt 中「只在特定文件類型用到」的 CSS/HTML 模板段落抽出，改用索引引用方式。
