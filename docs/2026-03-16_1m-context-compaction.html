Executive Summary

Anthropic 於 2026 年 3 月 13 日宣布 Opus 4.6 和 Sonnet 4.6 的 1M token context window 正式 GA。本文作者 Nyk 從實戰角度提出警告：更大的 context window 不等於更好的 agent 表現。文章系統性地定義了三種 context rot 失敗模式（Context Saturation、Instruction Dilution、Reasoning Loop Collapse），並提出以 Compaction API + CLAUDE.md 記憶層級 + Adaptive Thinking 組成的結構化解決方案。內容包含可複製的 API 設定碼、Claude Code 操作紀律、以及完整的操作檢查清單。

批判性提醒：作者宣稱的「failure rate 從 40% 降至 3%」缺乏方法論說明與樣本量揭露，應以 2-3x discount 看待。部分技術細節（如 scoped rules 的 paths frontmatter）尚未在 Anthropic 官方文件中完整記載。

Part I: 三種 Context Rot 失敗模式

作者開宗明義：問題從來不是 window 大小，而是 context rot — 效能隨著對話填充而退化，不是因為模型「忘記」，而是因為訊號淹沒在雜訊中。
"The problem was never the window size. It was context rot — performance degrades as conversations fill, not because the model forgets, but because signal drowns in noise."
📚 Failure 1: Context Saturation（上下文飽和）
Agent 累積 tool call 結果、檔案內容、搜尋輸出和推理軌跡。到 100K tokens 時，window 中約 60% 是雜訊。模型對 window 中所有內容賦予大致相同的注意力權重 — 它無法區分 50 次 tool call 前讀取的檔案和 3 秒前打開的檔案。

關鍵洞察：一個 800K tokens 雜訊的 1M window，表現比 150K tokens 精選訊號的 200K window 更差。
⚠️ Failure 2: Instruction Dilution（指令稀釋）
你的 system prompt 說「使用 pnpm」，CLAUDE.md 說「commit 前跑測試」，README 說「使用 npm」。到 200K tokens 時，這些指令和 150K tokens 對話歷史競爭注意力。一開始佔 context 5% 的指令，到 200K tokens 時只佔 0.5%。

這不是幻覺問題，是注意力預算問題。Anthropic 內部測試發現指令遵從度會隨 context 填充而明顯下降。
💡 Failure 3: Reasoning Loop Collapse（推理迴圈崩潰）
Agent 遇到 bug → 生成修復 → 測試 → 新錯誤 → 再修復… 到第五次迭代時，每次嘗試及其錯誤軌跡都在 window 中。模型開始混合第 2 次的推理和第 4 次的程式碼，提出已經試過的修復方案。

這是最燒 token 且產出最低的失敗模式。Agent 不是缺乏智慧，而是 context 中包含五條矛盾的推理鏈。
⚖️ 三種失敗模式對照表
失敗模式 根因 修復
Context Saturation 舊 tool results 累積為雜訊 填滿前壓縮
Instruction Dilution 指令佔比隨歷史增長而下降 壓縮後重新注入指令
Reasoning Loop 矛盾推理鏈共存 壓縮保留當前狀態，丟棄舊嘗試
🛠️ CDMO 場景對應
Context Saturation → 偏差調查 Agent：如果 Agent 讀取了 30 份 batch records 和 10 份 SOP，早期讀取的檔案內容成為雜訊，干擾對當前偏差的分析。

Instruction Dilution → 合規檢查 Agent：GMP 規則在 system prompt 中，但經過 200K tokens 的文件處理後，模型可能遺漏 21 CFR Part 11 的電子簽章要求。

Reasoning Loop → Tech Transfer 除錯：Agent 在試調充填參數時，第五次嘗試的推理混入了第二次的失敗參數。
🔍 批判性分析
「三種失敗模式」的分類框架直觀有用，但作者沒有提供定量證據。「60% 雜訊」和「5% 降至 0.5%」是概念性估算，非實測數據。此外，Anthropic 的 MRCR v2 benchmark（76% at 1M tokens）顯示長 context 檢索能力已大幅提升，失敗模式的嚴重程度可能隨模型世代改善而降低。

Part II: Compaction Harness — API Level 設定

Anthropic 隨 Opus 4.6 推出了 Compaction API（beta），當 input tokens 超過閾值時自動觸發，摘要較舊的 context 並以壓縮狀態繼續。
"Anthropic shipped the Compaction API as a beta feature with Opus 4.6. It triggers automatically when input tokens exceed a threshold, summarizes older context, and continues with a compressed state."
📚 核心參數解析
trigger.value：壓縮觸發時機。預設 150,000 tokens。作者設為 80,000 — 提早壓縮能保留更多訊號。最低下限為 50,000 tokens。

instructions：自訂摘要提示詞，完全取代預設摘要邏輯。用它告訴壓縮器要保留什麼（當前任務狀態、檔案路徑、測試結果）和丟棄什麼（舊搜尋結果、放棄的方法）。

pause_after_compaction：生成摘要後暫停，允許你檢視或修改摘要再繼續。開發時有用，生產環境關閉。
作者建議用領域專屬的壓縮指令取代預設的通用壓縮器。自訂指令明確指定保留項目（任務目標、檔案路徑、最新測試輸出、branch 名稱）和丟棄項目（超過 5 次 tool call 的檔案內容、未被採用的搜尋結果、放棄的推理軌跡）。
"The default compactor is generic. For production agents, replace it with domain-specific instructions."
⚠️ 定價注意
原文宣稱「zero price premium」，但需注意：Opus 4.6 原始發布時，超過 200K tokens 的請求有 long-context premium（$10/$37.50 per million tokens）。2026/3/13 的 GA 公告移除了此溢價，但 Anthropic 官方頁面上仍可見舊定價資訊。使用前務必確認當前定價政策。
💻 Compaction API 基礎設定
Python — 啟用 Compaction（beta header 必須） 
import anthropic

client = anthropic.Anthropic()

response = client.beta.messages.create(
  betas=["compact-2026-01-12"],
  model="claude-opus-4-6",
  max_tokens=16000,
  thinking={"type": "adaptive"},
  messages=messages,
  context_management={
    "edits": [{
      "type": "compact_20260112",
      "trigger": {
        "type": "input_tokens",
        "value": 80000   # 作者建議比預設 150K 更早觸發
      },
    }]
  },
)
💻 自訂 Compaction Instructions
YAML-style 壓縮指令範本（放入 instructions 參數） 
preserve:
  - current task objective and acceptance criteria
  - file paths of all files being modified
  - most recent test output (pass/fail, errors)
  - active branch name and commit context
  - any numbered step sequence in progress

discard:
  - file contents from reads older than 5 tool calls
  - search results that were not acted on
  - reasoning traces from abandoned approaches
  - duplicate error messages from retry loops

format the summary as:
## Current State
[task + branch + files]
## Recent Results
[last test output + last tool result]
## Active Constraints
[CLAUDE.md rules still in effect]

Part III: Claude Code — /compact 操作紀律

在 Claude Code 中，compaction 會在 context 接近約 85% 容量時自動執行。但作者認為 85% 已經太晚 — 在壓縮觸發前效能就已經退化了。
"If you use Claude Code, compaction runs automatically when context approaches ~85% capacity. The problem is that 85% is already too late — performance has degraded before compaction fires."
🛠️ 操作紀律清單
50% 手動 /compact：在 context 用量到 50% 時主動執行（用 cost display 檢查）。

/clear 切換主題前：不要讓任務 A 的陳舊 context 污染任務 B。

大量資料讀取後 /compact：任何一次超過 20K tokens 的檔案讀取或搜尋操作後，立即壓縮。
CLAUDE.md 完全存活於 compaction 之後。每次 /compact 後，Claude 會從磁碟重新讀取 CLAUDE.md 並重新注入。這就是為什麼 CLAUDE.md 中的指令會持續，但僅在對話中給予的指令會消失。
"CLAUDE.md fully survives compaction. After /compact, Claude rereads your CLAUDE.md from disk and re-injects it fresh. This is why instructions in CLAUDE.md persist, but instructions given only in conversation disappear."
⚠️ 自訂 /compact 指令
Claude Code 支援在 /compact 後面附加自訂指令：
/compact Focus on code samples and API usage

你也可以在 CLAUDE.md 中設定預設的壓縮行為。這讓每次壓縮都按照你的領域需求保留關鍵資訊。
💡 比喻：Context 管理就像 GMP 批次紀錄
想像你在管理一個 aseptic fill 的批次紀錄。每個步驟都有詳細記錄，但到了 deviation investigation 時，你不會把整份 500 頁的 batch record 攤在桌上 — 你會摘要關鍵事件時間線，只保留與偏差相關的參數紀錄。

Compaction 就是 Agent 的「偏差調查摘要」：保留決策紀錄和當前狀態，丟棄已完成步驟的原始細節。
⚖️ 對話指令 vs CLAUDE.md 指令
特性 對話中指令 CLAUDE.md 指令
Compaction 存活 否 — 被摘要吞噬 是 — 從磁碟重新注入
注意力權重 隨 context 增長而稀釋 每次壓縮後以完整強度重現
適用場景 臨時性、單次任務指令 永久性規則和約束
🔍 批判性分析
「50% 就壓縮」是相當激進的建議。Anthropic 官方 1M GA 公告強調的價值點恰恰是「更少的壓縮、更完整的對話」。過於頻繁的壓縮本身就是訊號損失 — 每次壓縮都丟失細節。最佳 trigger 點應取決於任務類型：例行性 tool call 密集型工作可能適合早壓縮，但需要全局推理的分析任務可能需要保留更多原始 context。

Part IV: CLAUDE.md 記憶層級架構

作者提出核心論點：沒有結構化記憶的 1M token window 就是一個 1M token 垃圾收集器。CLAUDE.md 層級架構是維持訊噪比的關鍵機制。
"A 1M token window without structured memory is a 1M token garbage collector. The CLAUDE.md hierarchy is how you keep signal density high as context grows."
📚 三層記憶架構
Global（~/.claude/CLAUDE.md）：你的工具、風格偏好、絕對不可違反的規則。跨所有專案生效。

Project（./CLAUDE.md）：技術棧、測試指令、專案慣例。僅在該專案目錄下生效。

Scoped（./.claude/rules/*.md）：按需載入。只有當 Agent 觸碰到匹配檔案時才載入。例如 API 設計規則只在編輯 API 檔案時載入。
人們忽略的關鍵洞察：子目錄中的 CLAUDE.md 檔案是按需載入，不是啟動時全部載入。一個 500 行的專案根目錄 CLAUDE.md 浪費 context。應該拆分成 scoped rules，只在 Agent 觸碰匹配檔案時載入。
"CLAUDE.md files in subdirectories load on demand, not at startup. A 500-line CLAUDE.md at the project root wastes context. Split it into scoped rules that load only when the agent touches matching files."
⚠️ Auto Memory 管理
Claude 會自動在 ~/.claude/projects/<project>/memory/MEMORY.md 寫入筆記 — 失敗的 build 指令、除錯洞察、架構模式。

限制：MEMORY.md 的前 200 行在 session 開始時載入，超過 200 行的部分被截斷。10 個 session 後會累積約 30% 冗餘條目。

維護紀律：每 10 個 session 審查一次 → 刪除與 CLAUDE.md 重複的條目 → 將詳細筆記移到主題檔案（debugging.md, patterns.md）→ 保持 MEMORY.md 作為簡潔索引。
💻 記憶層級結構
~/.claude/CLAUDE.md          # Global: 工具、風格、不可違反規則
./CLAUDE.md                  # Project: 技術棧、測試指令
./.claude/rules/
  api-design.md              # Scoped: 只在觸碰 API 檔案時載入
  testing.md                 # Scoped: 只在觸碰測試檔案時載入

Part V: Adaptive Thinking 與操作檢查清單

Opus 4.6 的 Adaptive Thinking 讓模型自行決定何時及多深入地推理。這與 context 管理的互動方式很少有人注意到。
"Opus 4.6 ships adaptive thinking — the model decides when and how much to reason. This interacts with context management in a way few operators have noticed."
📚 四個 Effort 層級
Low：簡單查找、格式化、快速回答 — 最少 thinking tokens
Medium：例行操作的平衡點
High（預設）：大多數任務的推薦設定
Max：最深度推理，最多 thinking tokens

Context 預算策略：對例行 tool call（檔案讀取、搜尋）使用 medium effort，對決策（選擇哪個修復、是否繼續或升級）使用 high effort。低 effort = 短 thinking traces = 更慢的 context 累積 = 更少的 compaction 事件。
作者的核心邏輯：每次 compaction 事件都會損失一些訊號。透過 effort 管理將 compaction 頻率降低 40%，直接轉化為更好的任務完成率。
"Every compaction event loses some signal. Reducing compaction frequency by 40% through effort management translates directly to better task completion rates."
🛠️ 完整操作檢查清單
☐ Compaction trigger 設為 80K tokens（非預設 150K）
☐ 撰寫自訂 compaction instructions（保留任務狀態、丟棄雜訊）
☐ 將 CLAUDE.md 拆分為 scoped rules（使用 .claude/rules/ + paths）
☐ 保持 MEMORY.md 在 200 行以內（細節移至主題檔案）
☐ 例行操作用 medium effort，決策用 high effort
☐ Claude Code 中在 50% context 時手動 /compact
☐ 切換主題前 /clear — 不讓陳舊 context 污染新任務

  model="claude-opus-4-6",
  max_tokens=16000,

Key Takeaways

🧠 Context Rot > Window Size
更大的 window 不等於更好的效能。800K tokens 的雜訊比 150K tokens 的精選訊號更差。Context Engineering 的核心是維持訊噪比，而非追求更大的容量。
🗜️ 提早壓縮，用自訂指令
Compaction trigger 設為 80K（非預設 150K），用領域專屬指令取代通用摘要。明確指定保留項目和丟棄項目，壓縮後的摘要品質決定後續任務表現。
📋 CLAUDE.md 是持久記憶
對話中的指令在壓縮後消失，CLAUDE.md 從磁碟重新注入。永久性規則放 CLAUDE.md，臨時指令放對話。Scoped rules 按需載入，避免浪費 context。
⚡ Effort = Context Budget
Adaptive Thinking 的 effort 層級不只影響推理品質，也影響 context 累積速度。例行操作用 medium、決策用 high，減少壓縮頻率就是減少訊號損失。

Practice Questions

Q1 概念理解 為什麼作者認為 1M token window 加上 800K tokens 的累積內容，效能會比 200K window 加上 150K tokens 更差？這和模型的 attention 機制有什麼關係？

因為 Context Saturation：模型對 window 中所有內容賦予大致相同的注意力權重。當 800K tokens 中有 60%+ 是過時的 tool results、放棄的推理路徑和已過期的搜尋結果時，模型需要從龐大的雜訊中找到訊號。這比在 150K tokens 的精選內容中找到訊號更難。

從 attention 機制角度看：Transformer 的 self-attention 讓每個 token 和所有其他 token 互動。更多的雜訊 token 意味著更多不相關的 attention connection，稀釋了對真正重要內容的注意力。Anthropic 的 MRCR v2 benchmark 提升（76%）確實改善了這個問題，但 benchmark 場景（needle retrieval）和實際 agent 工作（在雜訊中做決策）有本質差異。

Q2 產業應用 假設你正在為 CDMO 的偏差調查系統設計一個 Claude Agent，需要讀取多份 batch records、SOP 和過去的 CAPA 報告。你會如何設計 compaction instructions 來避免三種失敗模式？

針對 Context Saturation：在 compaction instructions 的 discard 區段指定「已完成審查且結論已記錄的 batch record 原始內容」和「已確認不相關的 SOP 版本」。保留項目應包含「當前偏差的事件時間線」和「已識別的根因候選項」。

針對 Instruction Dilution：將 GMP 合規規則放在 CLAUDE.md（而非對話中），確保壓縮後規則會重新注入。特別是 ALCOA+ 數據完整性要求和 21 CFR Part 211 的關鍵條款。

針對 Reasoning Loop：在 discard 區段指定「已排除的根因假設及其驗證過程」，在 preserve 區段保留「當前最可能的根因假設和支持證據」。format 區段應要求摘要按照偏差調查報告的結構（事件描述 → 根因分析進度 → 待驗證項目）。

Q3 批判思考 作者主張「50% context 就手動 /compact」，但 Anthropic 的 1M GA 公告強調的賣點是「更少壓縮、更完整的對話」。這兩個立場矛盾嗎？什麼情境下各自正確？

不矛盾，但適用場景不同。

Anthropic 的「更少壓縮」論點針對的是需要全局推理的任務：法律文件審查、跨檔案程式碼依賴分析、長期研究。這些任務中，保留完整 context 的價值超過雜訊成本。

作者的「50% 就壓縮」論點針對的是tool call 密集的 agent 工作：大量檔案讀取、搜尋、測試迴圈。這些操作快速累積低價值的中間結果，壓縮的收益高於保留的價值。

真正的原則應該是：根據 context 中「活躍訊號」vs「已處理資料」的比例來決定壓縮時機，而非使用固定的百分比閾值。一個讀了 500K tokens 法律文件的對話可能不需要壓縮，一個做了 50 次 tool call 只用了 100K tokens 的 agent 可能立即需要壓縮。

🎯 我的行動結論
✔ 核心洞察
Context Engineering 的本質是訊噪比管理，不是容量管理。這和 GMP 環境的文件管理邏輯一致 — 在偏差調查中，500 頁 batch record 的價值不在於頁數多寡，在於你能多快找到異常參數。同理，1M tokens 的價值不在 window 大小，在於 Agent 能否在其中找到當前決策所需的訊號。CLAUDE.md 作為 IF-ELSE directory 的設計理念（Context Engineering 原則）在這篇文章中得到了具體的技術落地。
