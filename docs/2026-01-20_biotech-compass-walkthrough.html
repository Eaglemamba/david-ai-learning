Executive Summary

這篇文章是 Biotech Compass 工具的完整實作教學。Biotech Compass 是一個結合 ChatGPT 提示工程與 Google Colab 視覺化的分析框架，讓投資者能在 15 分鐘內快速掌握生技公司的藥物管線、研發模式與財務健康狀況。

作者以 #BioPick2026 競賽中的三家公司（Palisade Bio、Corvus Pharmaceuticals、SELLAS Life Sciences）為範例，示範從 SEC 文件擷取數據、透過分層式提示詞控制 LLM 幻覺、到最終產出視覺化圖表的完整流程。這套方法論的核心價值在於將非結構化的財報資料轉化為可比較的標準化圖表，同時嚴格限制 AI 推論範圍以確保資料可信度。

Part I：運行 Compass 所需工具
Tools needed to run the Compass

Compass 的設計盡量降低對 LLM 操作、程式環境或公司官方文件的專業需求。你只需要：免費的 ChatGPT 帳號（建議使用 Plus 但非必要）、瀏覽器、Google Colab（免費）、公司新聞稿與 SEC 申報文件（至少需要 10-Q 或 10-K），以及 Biotech Compass 的提示詞與程式碼。

"The compass has been designed to require minimal expertise or confidence with LLMs, coding environments or official company documents."

技術能力只需要文件檢索（透過網頁搜尋）以及將提示詞或程式碼複製貼上到 ChatGPT 或 Google Colab。

"Technical capacity only requires document retrieval (through web search) and copying and pasting prompts or code into ChatGPT or Google Colab respectively."

核心概念
這個工具的設計哲學是「低門檻、高價值」。作者刻意避免需要程式設計能力或專業金融終端機的解決方案，讓任何能操作瀏覽器的人都能使用。這種設計思維值得學習：真正有價值的 AI 工具不是炫技，而是讓更多人能解決實際問題。
實務應用
對 CDMO 情境的啟發：這種「文件輸入 → AI 萃取 → 標準化輸出」的模式，可以應用在客戶藥物資料的初步分析。例如：客戶提供的 CMC 文件 → 提示詞萃取關鍵參數 → 標準化的可行性評估初稿。
Part II：所需文件來源
Documents needed to run the Compass

作者收集了範例公司最近的季度新聞稿與 SEC 申報文件。Palisade Bio 無法找到季度新聞稿——這是常見問題；但新聞稿只是用來輔助 SEC 申報文件。由於所有上市公司都必須提交 SEC 文件，這些文件應該總是可以取得進行分析。

"I was unable to retrieve a quarterly press release for Palisade Bio. This is a common problem; however, the press release is only there to support the SEC filing. As all public companies submit SEC filings, these documents should always be available for analysis."

作者刻意連結到文件的來源頁面而非文件本身，讓新投資者能看到這類資料通常在公司網站的哪裡找到。所有 SEC 申報文件也可以直接透過 sec.gov 取得。

"I have linked to the source pages for these documents rather than the documents themselves, so new investors can see where such materials are typically found on company websites."

比喻說明
把 SEC 申報文件想像成「必修科目」，新聞稿是「選修加分」。10-Q（季報）和 10-K（年報）是法規強制要求的標準化格式，內容經過審計；新聞稿則是公司自願發布的「行銷版本」，可能美化或省略某些資訊。分析時以 SEC 文件為主，新聞稿為輔。
重點提示
SEC 文件類型速查：
  • 10-K：年度報告，最完整的公司概況
  • 10-Q：季度報告，更新近期發展
  • 8-K：重大事件即時揭露（併購、高管異動等）
Part III：分層式資料收集提示詞
Collecting Data with Layered Prompts

從這些官方文件擷取資料很直接。只需將文件載入 ChatGPT 並依序執行提示詞即可。提示詞的設計是為了消除 LLM 的幻覺與推論。每個提示詞都謹慎地擴展 LLM 的文件範圍，以收集 Biotech Compass 所需的資料。

"The prompts are designed to eliminate hallucination and inference by the LLM. Each prompt broadens the LLM's scope of documents carefully to collect the data needed for the Biotech Compass."

依序執行五個提示詞：(1) Overview Prompt 從 SEC 與新聞稿萃取；(2) Backfill QC Prompt 識別可填補空白的外部來源；(3) Backfill Data Acquisition Prompt 用核准來源填入缺失欄位；(4) Open Web Assist Prompt 作為最後手段使用公開網路來源；(5) CSV Output Prompt 格式化最終輸出。

"Crucially, the CSV Output prompt can be run after Prompt 1, Prompt 3, or Prompt 4, depending on how tightly the user wants to constrain the Compass chart to specific data sources."

核心概念
這套提示詞架構體現了「漸進式信任擴展」的設計原則：
  • 第一層：只信任官方申報文件（最高可信度）
  • 第二層：公司官網 + ClinicalTrials.gov（官方但非強制審計）
  • 第三層：公開網路來源（最低可信度，標註來源）
使用者可以選擇在任何一層停止，控制資料的「純度」。
CDMO 應用啟發
這種分層式資料收集架構可以直接應用到製藥 CMC 文件分析：
  • 第一層：客戶提供的正式文件（IND、IMPD）
  • 第二層：公開 API 資料庫、已發表文獻
  • 第三層：產業報告、專家訪談紀錄
每一層都標註資料來源，讓審核者清楚知道哪些是「硬事實」、哪些是「軟推論」。
Part IV：繪圖與分析洞察
Plotting Data and Analyzing the Compass

CSV 檔案可以直接上傳到 Google Colab 並使用 Biotech Compass 程式碼執行。唯一需要修改的程式碼是更新 CSV 檔名，讓腳本指向正確的輸入檔案。

"The only required code change is updating the CSV filename so the script points to the correct input file."

第一個明顯的觀察是 SELLAS 和 Corvus 重疊了。對我來說，這是功能而非缺陷。重疊的節點表示兩家公司佔據相似的利基市場。解決方法很直接：從 CSV 檔案中依次移除每家公司並重新執行程式碼，產生兩張可以並排比較的圖表。

"The first thing that stands out is that SELLAS and Corvus overlap. To me, this is a feature rather than a flaw. Overlapping nodes suggest that two companies occupy a similar niche."

比喻說明
把 Biotech Compass 想像成「公司 DNA 比對」。當兩家公司在圖表上重疊，就像兩個人的基因高度相似——他們可能是競爭對手、潛在併購目標，或者至少值得放在一起深入比較。重疊不是「顯示錯誤」，而是「發現訊號」。
重點提示
快速洞察三家公司：
  • Palisade Bio：高度依賴單一資產，策略聚焦或結構性風險？
  • Corvus：管線多元化，涵蓋生物製劑與小分子藥物
  • SELLAS：有一個未分類的臨床前資產，值得進一步調查
Part V：Compass 的設計哲學
The Purpose and Limitations

這樣看來，Compass 自然引出一系列後續問題，而非結論：Palisade 的單一資產聚焦是刻意策略還是結構性風險？Corvus 的資產在其更廣泛管線中有多大差異化？SELLAS 那個未分類的項目到底是什麼，為什麼它無法清楚歸入標準分類？

"Viewed this way, the Compass naturally raises a set of follow-on questions, rather than conclusions."

對我來說，這正是 Biotech Compass 的目的。從零背景知識開始，它讓我能在 15 分鐘內紮實掌握這些公司在做什麼，更重要的是，我接下來應該問什麼問題。

"Starting from no prior knowledge, it allows me to get a solid handle on what these companies do, and more importantly, what questions I should be asking next in just 15 minutes."

核心概念
作者明確定位 Compass 為「導航工具」而非「決策引擎」。它的價值在於：
  • 快速建立基礎認知框架
  • 識別需要深入調查的訊號
  • 產生結構化的後續問題清單
這種「提問機器」的定位避免了過度依賴 AI 輸出的風險。
使用者注意事項
作者特別提醒：重疊節點是預期行為（表示相似利基）、「Other」分類和空白欄位不是錯誤（表示來源文件的模糊或缺失）。這些設計選擇體現了嚴格的資料收集和防止幻覺推論的原則。

重點整理

01
分層式提示詞架構
透過五層漸進式提示詞控制 LLM 資料來源範圍，從最高可信度的 SEC 文件逐步擴展到公開網路，使用者可在任何層級停止以控制資料純度。
02
低門檻工作流程
只需 ChatGPT + 瀏覽器 + Google Colab，透過複製貼上即可完成從文件萃取到視覺化的全流程，15 分鐘內產出可比較的公司分析圖表。
03
導航而非決策
Compass 定位為「提問機器」——產出的是後續調查問題清單，而非投資結論。重疊、空白、未分類都是訊號，不是錯誤。
04
幻覺控制設計
每個提示詞都明確限制 LLM 只能處理指定文件、禁止推論、必須標註來源，將「不知道」視為有價值的輸出而非需要填補的空白。

練習題

Q1 概念理解 為什麼 Biotech Compass 的提示詞設計要「分層」而非一次性完成所有資料收集？

分層設計解決了 LLM 應用的核心矛盾：資料完整性 vs. 資料可信度。

  • 單層設計的問題：如果一次性允許 LLM 搜尋所有來源，它可能會用低可信度資料填補空白，而使用者無法區分哪些是官方數據、哪些是網路推論。
  • 分層設計的價值：使用者可以在每一層檢視「缺了什麼」，然後有意識地決定是否要用更低可信度的來源填補。這種「知情妥協」比「無聲混合」更符合投資分析的審慎原則。
  • 彈性輸出：CSV Output Prompt 可以在 Prompt 1、3 或 4 之後執行，讓不同風險偏好的使用者選擇自己的「資料純度」。
Q2 產業應用 如何將 Biotech Compass 的分層提示詞架構應用到 CDMO 的客戶藥物可行性評估？

可以設計類似的分層架構來處理客戶 CMC 文件：

  • 第一層（Overview）：只從客戶提供的正式文件（IND Module 3、IMPD Section S）萃取藥物特性、製程參數、穩定性數據。不推論、不假設。
  • 第二層（Backfill QC）：識別缺失欄位，列出可能的補充來源（公開專利、已發表文獻、API 資料庫），但不自動填入。
  • 第三層（Controlled Fill）：經過 SME 核准後，從指定來源填入缺失資料，並明確標註「來源：客戶文件」vs.「來源：公開資料庫」。
  • 輸出：可行性評估初稿，每個數據點都有來源標註，讓審核者快速識別哪些需要向客戶確認。

這種架構符合 GxP 的資料追溯要求，同時大幅縮短初步評估時間。

Q3 批判思考 Biotech Compass 將「未分類」和「空白」視為功能而非錯誤，這種設計哲學有什麼潛在風險？

這是一個權衡設計，有明確的優缺點：

優點（作者的立場）：

  • 避免 LLM 幻覺填補，保持資料純度
  • 空白本身是訊號，提示使用者需要進一步調查
  • 符合「導航工具」而非「決策引擎」的定位

潛在風險：

  • 使用者誤解：不熟悉工具的人可能認為空白 = 資料不存在，而非資料未被萃取
  • 比較失真：如果公司 A 的文件較詳細而公司 B 較簡略，圖表會讓 A 看起來「更豐富」，但這可能只是披露風格差異
  • 行動偏誤：空白可能被解讀為「沒有問題」而非「需要調查」，導致該追問的地方被忽略

緩解方式：在圖表旁加上「資料完整度指標」，讓使用者知道每家公司有多少欄位是空白的。

我的行動結論
核心洞察
Biotech Compass 的分層提示詞架構解決了我一直在思考的問題：如何在 CDMO 場景中使用 LLM 萃取客戶文件資料，同時保持 GxP 要求的資料追溯性。關鍵設計原則是「漸進式信任擴展」——從最高可信度來源開始，每擴展一層都需要使用者明確同意，而非讓 AI 自動判斷。
立即應用
本週嘗試將這個架構應用到一份客戶 RFP 的初步評估：設計三層提示詞（客戶文件 → 公開資料庫 → 網路搜尋），產出可行性評估初稿，每個數據點標註來源層級。這可以測試「分層萃取」在製藥文件場景的可行性。
需要補充
文章沒有提供實際的提示詞內容（承諾稍後釋出）。在套用到 CDMO 場景前，需要研究作者如何在提示詞中實現「禁止推論」和「強制來源標註」——這是控制幻覺的關鍵技術細節。標記此文在提示詞釋出後重新閱讀。
