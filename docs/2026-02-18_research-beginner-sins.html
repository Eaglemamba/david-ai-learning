九大惡：逐條深度解析
1

AI 生成 code 卻不理解內容 HIGH

能跑但無法解釋 tensor、gradient、設計原因
這是 AI 時代最根本的陷阱。用 ChatGPT 或 Copilot 生成程式碼很容易，但如果你無法解釋每一行為什麼這樣寫，你就是在建造一棟你看不懂藍圖的房子。當 bug 出現時（一定會出現），你完全無法 debug。這正是 Copilot Mode（副駕駛模式）的核心原則：AI 加速你已經理解的工作，而不是取代你的理解。
CDMO 連結

跟 GMP 環境一模一樣：你可以抄一份 SOP，但如果不理解背後的科學原理，遇到偏差 (Deviation) 時就無法做出正確的根因分析 (Root Cause Analysis)。「能跑」不等於「能解釋」，在法規環境裡就是「能做」不等於「能回答稽核員的問題」。

2

無法依 requirements.txt 建立環境 HIGH

conda / dependency 出錯就停止
環境管理是可重現性 (Reproducibility) 的基礎。如果你連 pip install -r requirements.txt 失敗都不知道怎麼解決，代表你不理解 Python 的套件相依性系統。常見問題：版本衝突、系統層套件缺失、conda 與 pip 混用。解法是學會讀錯誤訊息，而不是重裝整個環境。
3

不會基本 Git Workflow HIGH

只會 upload 檔案、沒有 commit / branch 概念
把 GitHub 當 Google Drive 用是最常見的初學者行為。Git 不只是「存檔」，它是一套變更追蹤系統 (Version Control System)。每個 commit 是一個有意義的里程碑，branch 讓你安全地實驗。沒有 Git 紀律，你就無法回答「上週二的版本跟現在差在哪裡」這種基本問題。
CDMO 連結

這就是文件版本控制 (Document Control) 的軟體版。GMP 要求每份文件都要有版本號、變更歷史、核准紀錄。Git commit history 本質上就是你的 code 的變更管制紀錄 (Change Control Log)。

4

不會使用 CLI 工具 MED

依賴 GUI、無法用 command line 執行實驗
GUI 的問題不是「不好」，而是不可自動化、不可重現。你無法把「點擊第三個按鈕然後選擇 Options」寫成腳本。CLI 讓你的每一個操作都是可記錄、可重複、可腳本化的。在 AI 時代尤其重要：Claude Code、Cursor 等工具的核心互動方式就是 terminal。
5

大量程式塞進單一 Jupyter Notebook MED

缺乏模組化與結構設計
Notebook 很適合探索性分析 (Exploratory Analysis)，但不適合當主要開發環境。一個 500 行的 notebook 就像一份沒有分章節的報告 — 你無法重複使用其中的任何一段。正確做法是把可重用的邏輯抽取成 .py 模組，notebook 只負責呼叫和展示結果。
6

把 Notebook 當 Production Code MED

版本控制困難、無法協作維護
#5 和 #6 是一體兩面。Notebook 的 JSON 格式讓 Git diff 幾乎無法閱讀，cell 執行順序的隱性依賴讓別人（包括三個月後的你）無法可靠地重現結果。Rule of thumb：如果這段 code 會被跑超過三次，它就應該是一個 .py 檔案。
7

訓練完只輸出一張 final.png HIGH

沒有 log、learning curve、過程資訊
一張最終結果圖就像只看病人的出院診斷而不看病歷。訓練過程 (Training Dynamics) 才是真正的知識：loss curve 的形狀告訴你模型是否收斂、是否 overfit、learning rate 是否適當。沒有過程記錄，你無法判斷結果是真的好，還是剛好在某個 seed 下看起來好。
CDMO 連結

完全對應到製程驗證 (Process Validation) 的邏輯。FDA 不只看最終產品的放行測試結果，他們要看整個製程的 In-Process Control 數據、趨勢圖、關鍵製程參數 (CPP) 的紀錄。只有 final.png 等於只有 COA 沒有 Batch Record。

8

不記錄實驗設定與 Metadata HIGH

沒有 seed / config，結果不可重現
可重現性 (Reproducibility) 是科學方法的基石。如果你無法從 config 檔案完整重建實驗，你的結果就不是科學，只是軼事。最低標準：random seed、hyperparameters、data split ratio、環境版本，全部存成 JSON 或 YAML，跟結果一起 commit。
9

認為 Code 能跑就等於完成研究 HIGH

不分析 dynamics、failure mode、原因
這是九大惡中最核心的一條，也是 #1 的鏡像。#1 說的是「AI 幫你寫但你不懂」，#9 說的是「你自己寫完但不追問為什麼」。研究的價值不在於 code 能跑，而在於你理解模型為什麼成功、在哪些條件下會失敗、failure mode 的 pattern 是什麼。「能跑」只是起點，「能解釋」才是終點。
CDMO 連結

這就是 Quality by Design (QbD) 的精神：不是「batch 過了放行測試就好」，而是你要理解製程的 Design Space，知道哪些參數組合會失敗，為什麼會失敗。這也是偏差調查 (Deviation Investigation) 的核心能力 — 不只修好問題，而是理解問題的根本原因。

重點整理

理解 > 執行

AI 時代最危險的陷阱是用速度取代理解。Copilot 加速已知，不取代未知。

過程 > 結果

Training log、config、learning curve 比 final.png 有價值一百倍。沒有過程就沒有科學。

可重現 = 可信任

Seed、config、environment、data split — 都記錄下來，讓三個月後的你能完整重建。

工具紀律是基本功

Git、CLI、環境管理不是進階技能，是研究者的基本素養，跟識字一樣基礎。

練習題
Q1 概念理解 第 1 惡和第 9 惡為什麼說是「鏡像關係」？它們的共同根源是什麼？

#1（AI 生成 code 不理解）和 #9（code 能跑不追問為什麼）共同的根源是「滿足於表面功能性而放棄深度理解」。#1 是在輸入端放棄理解（讓 AI 替你想），#9 是在輸出端放棄理解（不分析結果的含義）。兩者夾擊的結果是：你變成一個「程式碼搬運工」而非研究者。核心原則：code 是工具，理解才是產出。

Q2 產業應用 在 CDMO 環境中，哪三條「惡」最直接對應到 GMP 合規要求？請說明對應關係。

#3 Git Workflow = 文件版本控制 (Document Control)：GMP 要求所有文件有版本號、變更歷史、核准流程，Git commit history 本質上就是 code 的變更管制紀錄。

#7 只有 final.png = 只有 COA 沒有 Batch Record：法規要求完整的 In-Process Control 數據和趨勢紀錄，不只是最終放行結果。

#8 不記錄 Metadata = 違反 Data Integrity (ALCOA+)：21 CFR Part 11 和 PIC/S 指南要求數據的 Attributable、Legible、Contemporaneous、Original、Accurate。沒有 seed 和 config 就是違反 Reproducible 原則。

Q3 批判思考 這篇貼文缺少什麼？如果你要補充第十惡，會是什麼？

這篇貼文主要聚焦在「個人工程實踐」，缺少的是「協作與溝通」面向。建議的第十惡：「不寫文件、不寫 README、不寫 docstring」。很多研究者的 code 只有自己看得懂，沒有 README 說明如何跑、沒有 docstring 解釋函數用途。這讓知識無法傳遞，也讓 code review 和協作幾乎不可能。在 AI 時代，好的文件還能讓 AI 更有效地理解和協助你的 codebase。

我的行動結論
✔ 核心洞察
這九條惡的底層邏輯其實就是我一直在講的 Copilot vs Agent Mode 框架的延伸 — 不管工具多強大，不可替代的人類判斷點 (O-ring Bottleneck) 永遠是「理解力」而非「執行力」。AI 讓執行成本趨近於零，但理解的價值因此更高。這篇貼文用研究場景驗證了這個論點。
📈 立即應用
把 #7 和 #8 的原則帶入我自己的 AI 學習系統：為每個 Make.com workflow 和 Claude Project 建立一份「config log」— 記錄版本、變更原因、測試結果。這就是把研究工程紀律應用到 AI 自動化專案的具體行動。
⚠ 需要補充
原文是社群短貼文，每一條都只有一句話，深度不夠。更重要的是缺少「怎麼修」的具體路徑 — 知道問題容易，建立正確習慣難。需要另外找一篇關於「Research Engineering Best Practices」的系統性文章來補足方法論。
