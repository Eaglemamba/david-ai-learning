Executive Summary

Tw93 基於半年深度使用 Claude Code（雙帳號、每月 40 美元）的實戰經驗，提出了一套六層架構模型來理解 Claude Code 的運作本質：上下文層、控制層、工具層、執行層、驗證層與治理層。文章的核心論點是：Claude Code 不是 ChatBot，而是一個循環式的 Agent 系統，卡住的原因幾乎不是模型不夠聰明，而是上下文被污染、控制層缺失、或驗證閉環斷裂。

文章從概念邊界（MCP/Skills/Hooks/Subagents 的區別）出發，深入上下文成本結構（200K 中真正可用的僅約 160-180K）、Skills 的三種設計模式、Hooks 的確定性約束、Subagents 的隔離價值、Prompt Caching 的架構影響，到 CLAUDE.md 的契約式寫法。每個主題都附帶具體配置範例與反模式警告。

Part I: 核心運作循環與六層模型
Analysis / 分析
Claude Code 的本質不是「回答問題」，而是一個反覆循環的代理過程：收集上下文 → 採取行動 → 驗證結果 → 完成或回到收集。很多人把它當 ChatBot 用，結果上下文越來越亂、規則越寫越長卻越不遵守。
"Claude Code's core isn't 'answering' -- it's a repeating agent loop: Collect context → Take action → Verify results → [Done or back to collect]"
Tw93 的核心洞察：卡住的地方幾乎從來不是模型不夠聰明，更多時候是給了錯誤的上下文，或者寫出來了但根本沒法判斷對不對。只強化其中一層，系統就會失衡。
"What gets stuck is almost never that the model isn't smart enough -- more often it's wrong context, or output with no way to verify correctness."
💡 比喻說明
把 Claude Code 想成一個新進員工 + 公司治理系統的組合。員工再聰明，如果你給他錯誤的 SOP、沒有驗收標準、沒有權限控制，他一樣會做出問題。CLAUDE.md 是員工手冊、Hooks 是稽核制度、Skills 是作業指導書、Subagents 是外派團隊。少任何一層都會出漏洞。
Original Key Points
📚 六層架構模型
1. Context Layer - CLAUDE.md / Skills / Memory / Rules
2. Control Layer - Hooks / Permissions / Sandbox
3. Tool Layer - MCP Servers / Built-in Tools / Custom Tools
4. Execution Layer - Subagents / Plan Mode
5. Verification Layer - Tests / Lint / Screenshots
6. Governance Layer - Health Check / Iteration / Anti-patterns
⚠️ 重點提示
排查問題的思路：結果不穩定 → 查上下文載入順序；自動化失控 → 看控制層有沒有設計；長會話品質下降 → 中間產物污染了上下文，換新會話比反覆調 prompt 有用。
Part II: 概念邊界 -- MCP / Plugin / Tools / Skills / Hooks / Subagents
Analysis / 分析
這六個概念常被混用，但設計意圖完全不同。Tw93 的一句話總結：給 Claude 新動作能力用 Tool/MCP，給它一套工作方法用 Skill，需要隔離執行環境用 Subagent，要強制約束和審計用 Hook，跨專案分發用 Plugin。
"For new action capabilities use Tool/MCP, for a methodology use Skill, for isolated execution use Subagent, for mandatory constraints use Hook, for cross-project distribution use Plugin."
Comparison Table
MechanismPurposeInvocation
Tool/MCP新增動作能力（API call、DB query）Agent 自動選擇
Skill按需載入的工作流 / 領域知識描述符常駐 + 內容按需
Hook確定性約束（格式化、阻斷、審計）事件觸發、不進上下文
Subagent隔離執行（掃碼、並行研究）主線程派生
Plugin跨專案工具分發安裝後可用
Part III: 上下文工程 -- 最重要的系統約束
Analysis / 分析
很多人把上下文當「容量問題」，但真正的問題不是不夠長，而是太吵了。200K 上下文中，固定開銷就吃掉 15-20K（系統指令 + Skill 描述符 + MCP 工具定義 + LSP 狀態），半固定的 CLAUDE.md 和 Memory 再吃 5-10K，動態可用的只剩 160-180K。
"Context gets stuck not because it's not long enough, but because it's too noisy. Useful info gets drowned by irrelevant content."
最驚人的發現：一個典型 MCP Server 包含 20-30 個工具定義，每個約 200 tokens。接 5 個 Server，光這部分固定開銷就到了 25,000 tokens（12.5%）。在需要讀大量程式碼的場景，這 12.5% 真的很關鍵。
"A typical MCP Server (e.g. GitHub) has 20-30 tool definitions, ~200 tokens each. Connect 5 servers and that's 25,000 tokens (12.5%) in fixed overhead."
🛠️ 上下文分層策略
始終常駐 → CLAUDE.md：專案契約 / 建構命令 / 禁止事項
按路徑載入 → .claude/rules/：語言 / 目錄 / 檔案類型特定規則
按需載入 → Skills：工作流 / 領域知識
隔離載入 → Subagents：大量探索 / 並行研究
不進上下文 → Hooks：確定性腳本 / 審計 / 阻斷
Context Cost Breakdown
💻 200K 上下文實際分配
200K Total Context
├── Fixed Overhead (~15-20K)
│   ├── System Instructions: ~2K
│   ├── Skill Descriptors: ~1-5K
│   ├── MCP Tool Definitions: ~10-20K  ← Biggest hidden cost
│   └── LSP State: ~2-5K
│
├── Semi-fixed (~5-10K)
│   ├── CLAUDE.md: ~2-5K
│   └── Memory: ~1-2K
│
└── Dynamic Available (~160-180K)
    ├── Conversation History
    ├── File Contents
    └── Tool Call Results
⚠️ 壓縮機制的陷阱
預設壓縮演算法按「可重新讀取」判斷，早期的 Tool Output 和檔案內容會被優先刪掉，連帶把架構決策和約束理由也丟了。解法：在 CLAUDE.md 裡寫 Compact Instructions，明確指定壓縮時必須保留的優先順序。另一個更主動的方案：讓 Claude 寫一份 HANDOFF.md，記錄當前進度和決策，開新會話時直接讀這個檔案。
Part IV: Skills 設計 -- 按需載入的工作流，不是模板庫
Analysis / 分析
Skill 的官方定義是「按需載入的知識與工作流」。關鍵區別：描述符（descriptor）常駐上下文，完整內容按需載入。這意味著每個啟用的 Skill 都在偷你的上下文空間，描述符越長，固定開銷越大。
"Skill descriptors are always in context; full content loads on demand. Every enabled Skill steals your context space."
好的 Skill 描述應該讓模型知道「何時該用我」，而不是「我是幹什麼的」-- 這兩個差很多。Claude Code 團隊反覆強調 "progressive disclosure"（漸進式披露）：先索引和導航，再按需拉取細節。
"Description should tell the model 'when to use me', not 'what I do' -- these two are very different."
⚖️ Skill 描述符效率對比
低效（~45 tokens）：
description: This skill helps you review code changes in Rust projects. It checks for common issues like unsafe code, error handling... Use this when you want to ensure code quality before merging.

高效（~9 tokens）：
description: Use for PR reviews with focus on correctness.
Three Skill Types
📚 三種 Skill 類型
1. Checklist 型（品質門禁）
發佈前跑一遍，確保不漏項。例：release-check -- cargo build / clippy / version bump / CHANGELOG。

2. Workflow 型（標準化操作）
高風險操作的顯式調用 + 內建回滾步驟。例：config-migration -- backup → dry-run → apply → verify，含 rollback 指令。

3. Domain Expert 型（決策框架）
運行時出問題時按固定路徑收集證據，不要瞎猜。例：runtime-diagnosis -- 收集 doctor 輸出 + logs + plugin state，依 Decision Matrix 分診。
⚠️ auto-invoke 策略
高頻（>1 次/會話）→ 保持 auto-invoke，優化描述符
低頻（<1 次/會話）→ disable-auto-invoke，手動觸發
極低頻（<1 次/月）→ 移除 Skill，改為文件中的文檔
Part V: 工具設計 -- 讓 Claude 少選錯
Analysis / 分析
給 Agent 用的工具和給人用的 API 不是一回事。給人用追求功能齊全，給 Agent 用，重點是讓它更容易用對。名稱前綴按系統分層、對大回應支援 response_format: concise/detailed、錯誤回應要教模型如何修正。
"Tools for agents and APIs for humans are different. For agents, the key is making it easy to use correctly, not feature-complete."
💡 AskUserQuestion 演進案例
Claude Code 團隊嘗試了三種方式讓 Agent 在中途停下來問使用者：
V1：給 Bash 工具加 question 參數 → Claude 大多直接忽略
V2：要求輸出特定 markdown 格式 → 格式脆弱，經常忘記
V3：做成獨立的 AskUserQuestion 工具 → 調用即暫停，效果最好

教訓：如果你就是要 Claude 停下來做某件事，給它一個專門的工具，比加 flag 或約定輸出格式穩定得多。
Tool Design Principles
🛠️ 工具設計原則
Good Tool: 名稱語義清晰、單一動作、schema 完整、錯誤可修正
Bad Tool: 名稱模糊、參數隱含假設、錯誤碼不透明、回應過大

具體建議：
• 名稱前綴按系統分層：github_pr_*, jira_issue_*
• 大回應支援 response_format: concise
• 能合併成高層任務工具時，不暴露底層碎片
• 避免 list_all_* 讓模型自行篩選
⚠️ 什麼時候不該再加 Tool
• 本地 shell 可靠完成的事
• 只需要靜態知識、不需要外部互動
• 需求更適合 Skill 的工作流約束
• 還沒驗證過 schema 能被模型穩定使用
Part VI: Hooks -- 強制插入你自己的邏輯
Analysis / 分析
Hooks 容易被當成「自動執行的腳本」，但更準確的理解是：把不能交給 Claude 臨場發揮的事情，收回到確定性的流程裡。格式化要不要跑、保護檔案能不能改、任務完成後要不要通知 -- 這些事不要指望 Claude 每次都自己記得。
"Hooks are about pulling things that can't be left to Claude's discretion back into deterministic flows."
Hook 的執行結果不進上下文，這和 Tool/Skill 最大的區別。它是旁路的確定性邏輯，不佔上下文空間，不影響模型的推理路徑。在 100 次編輯的會話中，每次節省 30-60 秒，累積節省 1-2 小時。
"Hook output stays outside the context window -- the biggest difference from Tools/Skills."
🛠️ 三層疊加設計
CLAUDE.md：聲明「提交前必須通過測試和 lint」
Skill：告訴 Claude 在什麼順序下運行測試、如何看失敗、如何修復
Hook：對關鍵路徑執行硬性校驗，必要時阻斷

三樣少任何一層都會有漏洞。只寫 CLAUDE.md，Claude 經常當沒看見；只靠 Hooks，細節判斷又做不了。
Hook Points & Examples
📚 支援的 Hook 點
PreToolUse -- 工具執行前（可阻斷）
PostToolUse -- 工具執行後（自動校驗）
Notification -- 任務完成通知
SessionStart -- 會話啟動時注入動態上下文
💻 PostToolUse Hook 範例
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit",
      "pattern": "*.rs",
      "hooks": [{
        "type": "command",
        "command": "cargo check 2>&1 | head -30",
        "statusMessage": "Running cargo check..."
      }]
    }],
    "Notification": [{
      "type": "command",
      "command": "osascript -e '...notification...'"
    }]
  }
}
Note: 限制輸出長度（| head -30），避免 Hook 輸出反而污染上下文。
Part VII: Subagents -- 隔離執行，不是平行處理
Analysis / 分析
Subagent 是從主對話派出去的獨立 Claude 實例，有自己的上下文窗口。核心價值不是「並行」，而是隔離：掃程式碼、跑測試、做審查這類會產生大量輸出的事，交給 Subagent，主線程只拿摘要，不被中間過程污染。
"Subagent's core value isn't 'parallelism' but isolation -- main thread gets summary only, not polluted by intermediate output."
⚠️ Subagent 反模式
• 子代理權限和主線程一樣寬 → 隔離沒有意義
• 輸出格式不固定 → 主線程拿到沒法用
• 子任務之間強依賴、頻繁共享中間狀態 → 不適合用 Subagent
Configuration Guidelines
🛠️ 配置要點
tools / disallowedTools：限定能用什麼工具，別給和主線程一樣寬的權限
model：探索任務用 Haiku/Sonnet，重要審查用 Opus
maxTurns：防止跑飛
isolation: worktree：需要動檔案時隔離檔案系統

內建類型：Explore（唯讀、省成本）、Plan（規劃調研）、General-purpose（通用）
Part VIII: Prompt Caching -- Claude Code 架構的核心
Analysis / 分析
這是很多教程都沒展開講的部分，但它深刻影響 Claude Code 的成本結構和設計取捨。工程界有句話「Cache Rules Everything Around Me」，Claude Code 的整個架構都是圍繞 Prompt Caching 構建的。高快取命中率不只降低成本，也幫助獲得更寬鬆的速率限制。
"Claude Code's entire architecture is built around prompt caching. High cache hit rates don't just lower cost -- they also help create more relaxed rate limits."
Prompt Caching 是按前綴匹配工作的：從請求開頭到每個 cache_control 斷點之前的內容都會被快取。所以順序很重要：System Prompt（靜態）→ Tool Definitions（靜態）→ Chat History（動態）→ 當前輸入。
"Prompt caching works on prefix matching -- from the beginning of the request to each cache_control breakpoint."
⚠️ 破壞快取的常見陷阱
• 在靜態 System Prompt 中放入帶時間戳的內容
• 非確定性地打亂工具定義順序
• 會話中途增刪工具
• 會話中途切換模型 -- 快取是模型唯一的，已和 Opus 對話 100K tokens 後切到 Haiku，要為 Haiku 重建整個快取，反而更貴
💡 比喻說明
把 Prompt Cache 想像成一本「前半部固定、後半部手寫」的筆記本。前半部（System + Tools）印好了不能改，所有人都用同一版。一旦你把前半部撕掉重印（改了 System Prompt 或換了模型），之前的快取就全部作廢，要重新花錢印。所以動態資訊（時間戳、Branch 名稱）要放在「手寫」的後半部。
Caching Architecture Details
💻 Prompt Layout for Caching
// Claude Code Prompt Order (cache-optimized)
1. System Prompt     → Static, locked
2. Tool Definitions  → Static, locked
3. Chat History      → Dynamic, appended
4. Current Input     → Last position

// Dynamic info goes in user message,
// NOT in system prompt:
<system-reminder>
  Current time: 2026-03-13T10:00:00Z
  Branch: feature/new-ui
</system-reminder>
📚 Compaction 實際流程
觸發條件：上下文快滿時
執行方式：Claude Code 開一個 fork 調用，把完整對話歷史喂給模型 + "Summarize this conversation"
成本：命中快取所以只需 1/10 價格
結果：幾十輪對話被替換成 ~20K tokens 摘要，System + Tools 保留，掛上之前的檔案引用

defer_loading 機制：數十個 MCP 工具不全量包含，發送輕量級 stub（只有工具名 + defer_loading: true），模型通過 ToolSearch 工具「發現」它們，完整 schema 只在選擇後才載入。快取前綴保持穩定。
Part IX: 驗證閉環 -- 沒有 Verifier 就沒有工程上的 Agent
Analysis / 分析
Tw93 的核心判斷標準：如果一個任務你都說不清楚「Claude 怎麼才算做對了」，那它大概率也不適合直接交給 Claude 自動完成。驗證標準本身都沒有，Claude 再聰明也跑不出正確答案。
"If you can't articulate 'how does Claude know it's done right', it probably shouldn't be given to Claude for autonomous completion."
🛠️ CDMO 連結：驗證 = 放行標準
這和 GMP 的放行判定完全一致：沒有預先定義的驗收標準（acceptance criteria），就不能說產品合格。在 Claude Code 裡，make test + make lint 就是你的 IPC/FPC（In-Process Check / Final Product Check）。寫任務 Prompt 或 Skill 時就把驗收標準定義清楚，就像 batch record 上的 CQA 欄位一樣。
Verification Hierarchy
📚 三層驗證層級
Layer 1 (Minimum): 命令退出碼、lint、typecheck、unit test
Layer 2 (Integration): 整合測試、截圖對比、contract test、smoke test
Layer 3 (Production): 生產日誌驗證、監控指標、人工審查清單
💻 驗證定義範例
## Verification (in CLAUDE.md)

For backend changes:
- Run `make test` and `make lint`
- For API changes, update contract tests

For UI changes:
- Capture before/after screenshots

Definition of done:
- All tests pass
- Lint passes
- No TODO left behind
Part X: 高頻命令的工程意義
Analysis / 分析
這些命令的共同目的只有一件事：主動管理上下文，別等系統自己處理。Tw93 把命令分為三類：上下文管理、能力與治理、會話連續性與並行。
"These commands all do one thing: actively manage context -- don't wait for the system to handle it."
⚠️ 不常見但很好用的命令
/simplify -- 對剛改完的程式碼做三維檢查（複用、品質、效率），代替手動 review
/rewind -- 回到某個 checkpoint 重新總結，不是撤銷，適合 Claude 沿錯誤路徑探索太久時
/btw -- 不打斷主任務的快速旁路問答
/insight -- 分析當前會話，提煉值得沉澱到 CLAUDE.md 的內容
⎋⎋ ESC -- 回到上一條輸入重新編輯，比重開會話省事
Ctrl+B -- 長時間 bash 命令移到背景執行
Command Categories
📚 上下文管理類
/context -- 查看 token 佔用結構，排查 MCP 佔比
/clear -- 清空會話，同一問題被糾偏兩次以上就重來
/compact -- 壓縮但保留重點，配合 Compact Instructions
/memory -- 確認哪些 CLAUDE.md 真的被載入了
🛠️ 會話連續性類
claude --continue -- 恢復最近會話
claude --resume -- 選擇器恢復歷史會話
claude --continue --fork -- 分叉，同起點不同方案
claude --worktree -- 隔離 git worktree
claude -p "prompt" -- 非交互模式，接 CI/腳本
claude -p --output-format json -- 結構化輸出
Part XI: 如何寫一個好的 CLAUDE.md
Analysis / 分析
CLAUDE.md 更像是你和 Claude 之間的協作契約，不是團隊文檔，也不是知識庫。裡面只放那些每次會話都得成立的事。Tw93 的建議很直接：一開始甚至可以什麼都不寫，等你發現自己老是在重複同一件事，再補進去。
"CLAUDE.md is a collaboration contract -- not a team doc or knowledge base. Only put in things that must hold true every single session."
Anthropic 官方自己的 CLAUDE.md 大約只有 2.5K tokens。每次都要知道的放 CLAUDE.md，只對部分檔案生效的放 rules，只在某類任務中需要的放 Skills。最喜歡的技巧：每次糾正 Claude 的錯誤後，讓它自己更新 CLAUDE.md。
"Anthropic's own CLAUDE.md is only ~2.5K tokens. After correcting Claude, tell it: 'Update your CLAUDE.md so you don't make that mistake again.'"
CLAUDE.md Template
💻 高品質模板結構
# Project Contract

## Build And Test
- Install: `pnpm install`
- Dev / Test / Typecheck / Lint

## Architecture Boundaries
- HTTP handlers in src/http/handlers/
- Domain logic in src/domain/
- Do NOT put persistence in handlers

## Safety Rails
NEVER: modify .env, lockfiles, CI secrets
ALWAYS: show diff before committing

## Verification
Backend: make test + make lint
UI: before/after screenshots

## Compact Instructions
Preserve: architecture decisions,
modified files, verification status,
open TODOs, rollback notes
⚠️ 不該放的內容
• 大段背景介紹
• 完整 API 文件
• 空泛原則如「寫高品質程式碼」
• Claude 讀 repo 即可推斷的資訊
• 大量背景資料和低頻任務知識（放 Skills）
Part XII: 實戰經驗 -- Kaku 專案中的踩坑
Analysis / 分析
Tw93 用 Claude Code 開發了開源 terminal 專案 Kaku（Rust + Lua），混合語言加上自定義配置系統暴露出典型的 agent 協作問題。核心發現：環境透明比你想像中重要。Claude Code 調用的都是真實的 shell、git、package manager，只要有一層不透明，它就只能開始猜。
"Environment transparency matters more than you think. Once Claude starts guessing about the environment, reliability drops fast."
解法：在 CLI 中加入 doctor 命令，把環境狀態、依賴和配置收成結構化健康報告。同時確保 CLI 有 init、config、reset 等語義清楚的子命令，先把狀態收斂住，再暴露編輯入口，順序一反過來就很容易亂。
"Converge state first, then expose edit entry points. Reverse the order and things get messy fast."

Part XIII: 常見反模式與配置健康檢查
Analysis / 分析
Tw93 列出的反模式清單，本質上就是六層架構中每一層可能出錯的地方。從上下文層的「CLAUDE.md 寫太長自己先污染自己」到執行層的「Subagent 權限太寬」到治理層的「跳過驗證就提交」。
⚠️ 核心反模式
Context: CLAUDE.md 500+ 行、MCP 全開不斷開閒置、不用 /compact 等自動壓縮
Skills: 描述太短觸發太廣（help with backend）、正文太長、一個 Skill 覆蓋五件事
Tools: 暴露太多底層碎片、錯誤碼不透明、沒驗證過 schema 穩定性
Hooks: 放入需要語義判斷的複雜邏輯（應放 Skill）
Subagent: 權限和主線程一樣寬、輸出格式不固定
Verify: 跳過驗證就提交、沒有 Definition of Done
Health Check Tool
🛠️ claude-health 健康檢查
Tw93 基於六層架構框架整理成開源 Skill：

npx skills add tw93/claude-health

在任意會話跑 /health，自動識別專案複雜度，對 CLAUDE.md、rules、skills、hooks、allowedTools 和實際行為模式各跑一遍檢查，輸出優先級報告：
🔴 需要立刻修
🟠 結構性問題
🟢 可以慢慢做
📚 三階段成長模型
Stage 1: 當 ChatBot 用（問答式）
Stage 2: 當工具用（功能驅動）
Stage 3: 當系統治理（約束下自主運行）

從「這個功能怎麼用」變成「怎麼讓 agent 在約束下自己跑起來」-- 兩件事感覺差很多。

Key Takeaways

Context is the Bottleneck
200K 上下文中真正可用的僅 160-180K。MCP 工具定義是最大隱形成本（5 個 Server 佔 12.5%）。上下文治理比 Prompt 技巧更重要。
Layer Your Controls
CLAUDE.md 宣告規則、Skills 定義工作流、Hooks 強制執行。三層少任何一層都會出漏洞，這和 GMP 的 SOP + WI + 稽核三層設計一模一樣。
Design for Cache
Prompt Caching 是按前綴匹配，靜態內容（System Prompt、Tool Definitions）放前面，動態內容放後面。中途切模型 = 重建整個 Cache，通常更貴。
Verify or It Didn't Happen
「Claude 說完成了」沒有意義。如果你說不清楚「什麼叫做完」，就不適合直接交給 Claude 自主完成。驗證標準本身都沒有，再聰明也跑不出正確答案。

Practice Questions

Q1 概念理解 Tw93 說「卡住的地方幾乎從來不是模型不夠聰明」。在他的六層架構中，當 Claude Code 的輸出品質在長會話中逐漸下降時，最可能是哪一層出了問題？該如何排查和修復？

最可能是Context Layer（上下文層）出了問題。長會話中，中間產物（Tool Output、讀取的檔案內容）不斷累積，佔據了原本給「有用資訊」的空間。當系統自動壓縮（Compaction）啟動時，預設演算法會按「可重新讀取」判斷，把早期的架構決策和約束理由一起丟掉。

排查步驟：(1) 用 /context 檢查 token 消耗結構，看 MCP 固定開銷和歷史對話佔比；(2) 檢查 CLAUDE.md 是否有 Compact Instructions，確保壓縮時保留架構決策；(3) 考慮用 /clear 開新會話，或讓 Claude 先寫一份 HANDOFF.md 再交接。

修復：在 CLAUDE.md 加入 Compact Instructions，依優先順序列出壓縮時必須保留的資訊（架構決策 > 修改檔案 > 驗證狀態 > TODO > Tool 輸出）。對於任務切換用 /clear，同一任務進入新階段用 /compact。

Q2 產業應用 在 CDMO 的 GMP 環境中，你正在考慮用 Claude Code 來自動化 batch record review 流程。根據 Tw93 的六層架構和 Copilot/Agent 邊界，你會如何設計 CLAUDE.md、Skills、Hooks 這三層的分工？哪些環節絕對不能讓 Claude 自主完成？

CLAUDE.md（協作契約）：聲明核心規則 -- batch record review 必須在 human review 完成前不得標記為 "Pass"；所有 CQA（Critical Quality Attribute）欄位必須逐一驗證；NEVER 自動修改原始數據；ALWAYS 輸出完整的偏差摘要。

Skills（工作流）：建立 batch-review Skill，定義步驟：(1) 讀取 batch record PDF，(2) 對照 CQA 規格逐欄檢查，(3) 標記異常值，(4) 生成偏差摘要報告，(5) 停在「等待人工確認」。每個步驟有明確的輸入/輸出格式和停止條件。

Hooks（強制約束）：PreToolUse Hook 阻斷任何嘗試修改原始 batch record 的操作；PostToolUse Hook 在生成報告後自動執行格式檢查（是否包含所有必要欄位）。

絕對不能自主完成的環節：最終放行判定（release decision）、OOS/OOT 的根本原因判斷（root cause determination）、任何涉及數據完整性的修改（ALCOA+ 原則）。這些是 O-Ring Bottleneck -- AI 在周圍加速，但這些節點的人類判斷不可替代。

Q3 批判思考 Tw93 的文章幾乎所有案例都來自軟體開發場景（Rust、TypeScript、CI/CD）。他的六層架構模型在非程式碼的知識工作（如文件撰寫、策略分析、專案管理）中是否同樣適用？有哪些層會失效或需要重新定義？

仍然適用的層：Context Layer（上下文治理在任何 LLM 工作流中都是核心約束）、Control Layer 的部分功能（如 Compact Instructions、Plan Mode）、Governance Layer（反模式識別、健康檢查思維）。

需要重新定義的層：

Verification Layer 是最大挑戰。程式碼有 make test 和 lint 提供客觀判定，但文件撰寫和策略分析的「正確性」高度主觀。你需要把 Verifier 從「自動化測試」轉換為「結構化 checklist」-- 例如「報告是否包含所有必要章節」「數據來源是否引用」「結論是否與分析一致」。

Tool Layer 在非程式碼場景中大幅簡化（可能只需要檔案讀寫和搜尋），但 Skill Layer 的重要性反而上升 -- 你需要更精確的 domain-specific 工作流指引來替代程式碼世界中 compiler 和 linter 提供的即時回饋。

Hooks Layer 在非 CLI 場景幾乎失效。目前 Hooks 依賴檔案匹配和 shell 命令，對於 Claude.ai Projects 或 Make.com 自動化，等價物可能是 Make.com 的 Router + Error Handler 或 Claude.ai 的 System Prompt 硬約束。這是文章的明確盲點。
