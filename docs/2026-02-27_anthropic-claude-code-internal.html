這份文件來自 Anthropic 內部，記錄了 10 個不同部門如何在日常工作中使用 Claude Code。從資料基礎建設、產品開發、安全工程到法務部門，涵蓋了開發者與非技術人員兩種截然不同的使用模式。

Part I
Data Infrastructure 資料基礎建設團隊
Developer Team
資料基礎建設團隊負責整合公司所有業務資料。他們用 Claude Code 做三件核心的事：自動化例行資料工程任務、排除複雜基礎設施問題、以及為非技術成員建立可執行的工作流程。
The Data Infrastructure team organizes all business data for teams across the company, using Claude Code for automating routine data engineering tasks.
最令人印象深刻的案例是 Kubernetes 截圖除錯：當叢集無法排程新的 Pod 時，團隊將儀表板截圖餵給 Claude Code，它一步步引導他們在 Google Cloud UI 中找到 IP 位址耗盡的警告，並提供建立新 IP pool 的精確指令。這完全繞過了對網路專家的依賴。
They fed screenshots of dashboards into Claude Code, which guided them through Google Cloud's UI menu by menu until they found a warning indicating pod IP address exhaustion.
另一個重要模式是純文字工作流程 (Plain Text Workflows)：財務團隊成員只需用純文字描述資料工作流程，載入 Claude Code 就能自動執行整個流程。零程式碼經驗的員工也能執行複雜的資料查詢和 Excel 輸出。
Employees with no coding experience could describe steps like "query this dashboard, get information, run these queries, produce Excel output," and Claude Code would execute the entire workflow.
⚠️ 重點提示

Claude.md 是效能倍增器

團隊強調：你的工作流程文件寫得越詳細，Claude Code 的表現就越好。這與 Context Engineering（上下文工程）的核心邏輯完全一致 — 指令品質決定輸出品質。

🛠️ 實務應用

團隊最佳實踐：

  • 用 MCP servers 取代 CLI 處理敏感資料（更好的安全控制和日誌記錄）
  • 舉辦團隊使用分享會（knowledge sharing sessions），讓成員展示各自的工作流程
  • 每次工作結束時要求 Claude Code 總結並建議改進，形成持續改善循環
  • 開啟多個 Claude Code 實例進行平行任務管理，切換回來時不會失去上下文
💡 比喻說明

把 Claude.md 想成你雇用新助理時給的「工作手冊」。手冊寫得越清楚，助理第一天就能上手的工作越多。每次工作結束更新手冊，就像讓助理越來越了解你的偏好。

Part II
Security Engineering 安全工程團隊
Developer Team
安全工程團隊專注於軟體開發生命週期的安全性、供應鏈安全和開發環境安全。他們的使用模式揭示了一個關鍵洞察：Claude Code 不只是寫程式碼的工具，更是「跨領域貢獻的加速器」。
The Security Engineering team focuses on securing the software development lifecycle, supply chain security, and development environment security.
在事件處理中，他們將 Stack Traces 和文件餵給 Claude Code，讓它追蹤程式碼中的控制流程。原本需要 10-15 分鐘的手動程式碼掃描現在約 5 分鐘就能完成。
This significantly reduces time-to-resolution for production issues, allowing them to understand problems that would normally take 10-15 minutes in about 5 minutes.
Terraform 審查是另一個亮點：將基礎設施變更計畫複製到 Claude Code，直接詢問「這會做什麼？我會後悔嗎？」這大幅縮短了安全審查的迴圈時間，消除了開發者等待安全團隊批准時的瓶頸。
They copy Terraform plans into Claude Code to ask "what's this going to do? Am I going to regret this?" creating tighter feedback loops.
最重要的工作流程轉變：從原本的「設計文件 → 粗糙程式碼 → 重構 → 放棄測試」模式，轉為測試驅動開發 (TDD)，先讓 Claude Code 寫偽代碼，引導它做 TDD，定期檢查方向。
Instead of their previous "design doc → janky code → refactor → give up on tests" pattern, they now ask Claude Code for pseudocode and guide it through test-driven development.
📚 核心概念

Custom Slash Commands 的威力

安全工程團隊使用了整個 Monorepo 中 50% 的自訂 Slash Command。這些自訂指令簡化了特定工作流程，加速了重複性任務。這說明：投資建立「個人化指令庫」的回報率極高。

🛠️ 實務應用

團隊最佳實踐：

  • 「讓 Claude 先說」— 不要問針對性問題取得程式碼片段，而是告訴它「邊做邊提交」讓它自主工作
  • 用 Claude Code 合成文件和建立 Runbook，取代手動搜尋知識庫
  • 提供寫作範本和格式偏好，直接產出可用於 Slack、Google Docs 的文件
  • 新專案能在數天而非數週內開始有意義的貢獻
Part III
Product Development 產品開發團隊
Developer Team
Claude Code 團隊用自己的產品來開發 Claude Code — 這是最純正的 Dogfooding（吃自己的狗糧）。他們的使用模式最清楚地展示了任務分類直覺 (Task Classification Intuition) 的重要性。
The Claude Code team uses their own product to build updates to Claude Code.
非同步模式：使用 Auto-accept Mode (Shift+Tab)，讓 Claude 寫程式碼、跑測試、持續迭代。給它不熟悉的抽象問題，讓它自主工作，然後審查約 80% 完成的方案再接手最後潤飾。關鍵：從乾淨的 git 狀態開始，定期提交 checkpoint。
Engineers use Claude Code for rapid prototyping by enabling "auto-accept mode" (shift+tab), starting from a clean git state and committing checkpoints regularly.
同步模式：對於觸及應用程式商業邏輯的核心功能，團隊與 Claude Code 同步工作，給出詳細的 Prompt 和具體的實作指示，即時監控過程以確保程式碼品質和架構正確性。
For more critical features touching the application's business logic, the team works synchronously with Claude Code, giving detailed prompts with specific implementation instructions.
成功案例：實作 Vim 按鍵綁定功能，大約 70% 的最終實作來自 Claude 的自主工作，只需幾次迭代就完成了。
Roughly 70% of the final implementation came from Claude's autonomous work, requiring only a few iterations to complete.
⚠️ 重點提示

任務分類法則

這是整份文件最重要的框架之一：

  • 非同步 (Auto-accept)：外圍功能、原型設計、不熟悉的抽象問題
  • 同步 (監督模式)：核心商業邏輯、關鍵修復、高風險變更

這跟 CDMO 的品質管理邏輯完全一致：常規操作可以部分自動化，關鍵品質判斷必須人工監督。

🛠️ 實務應用

建立自我驗證循環 (Self-Sufficient Loops)：

設定 Claude 自動執行 build、test、lint 來驗證自己的工作。這讓 Claude 能更長時間自主工作並抓住自己的錯誤。特別有效的做法是要求 Claude 先寫測試再寫程式碼。

Part IV
RL Engineering 強化學習工程團隊
Developer Team
RL 工程團隊的使用方式最為務實和坦誠。他們明確指出：Claude Code 首次嘗試只有約三分之一的成功率，需要額外引導或手動介入。但即便如此，整體仍然節省了大量時間。
They acknowledge it only works on first attempt about one-third of the time, requiring either additional guidance or manual intervention.
他們採用「嘗試與回滾 (Try and Rollback)」方法論：頻繁提交 checkpoint，讓 Claude 嘗試自主實作，不行就回滾重來。這種方法比試圖修正 Claude 的錯誤更有效率。
They now use a "try and rollback" methodology, frequently committing checkpoints so they can test Claude's autonomous implementation attempts and revert if needed.
一個實用技巧：在 Claude.md 中加入防止重複犯錯的指令，例如「用 pytest 不要用 run，不要不必要地 cd — 直接用正確的路徑。」這顯著改善了一致性。
Add instructions to your Claude.md file to prevent Claude from making repeated tool-calling mistakes.
💡 比喻說明

「老虎機策略」(Slot Machine Strategy)

Data Science 團隊用了一個精準的比喻：把 Claude Code 當老虎機 — 存好狀態，讓它跑 30 分鐘，要麼接受結果，要麼從頭來。重新開始的成功率通常比試圖修正 Claude 的錯誤更高。

⚠️ 重點提示

「先一次搞定，不行再協作」策略：

給 Claude 一個快速 prompt 讓它嘗試完整實作。如果成功（約三分之一），你省下了大量時間。如果不行，再切換到更協作的引導模式。這是一個很好的期望值管理框架。

Part V
Inference 推論團隊
Developer Team
推論團隊的案例最能體現 Claude Code 作為「技能差距橋樑」(Skill Gap Bridge) 的價值。沒有機器學習背景的團隊成員，靠 Claude Code 將研究時間減少了 80% — 原本需要一小時的 Google 搜尋和文件閱讀，現在只需 10-20 分鐘。
What would require an hour of Google searching and reading documentation now takes 10-20 minutes, reducing research time by 80%.
跨語言程式碼翻譯也是一大亮點：測試功能時需要不同程式語言（如 Rust），團隊只需解釋要測試什麼，Claude 就用所需語言寫出邏輯，消除了「只為測試而學新語言」的障礙。
They explain what they want to test and Claude writes the logic in the required language (like Rust), eliminating the need to learn new languages just for testing purposes.
📚 核心概念

信任建立路徑 (Trust Building Path)

推論團隊建議的採用策略很實用：

  • 先測試知識庫功能（問各種問題，看是否比 Google 快且準確）
  • 從程式碼生成開始（給具體指令，驗證正確性）
  • 建立信任後再擴展到更複雜的任務

這是一個漸進式的信任建立框架，適合任何組織的 AI 工具導入。

Part VI
Data Science & Visualization 資料科學與視覺化團隊
Developer Team
這個團隊的案例說明了一個重要轉變：從一次性 Jupyter Notebook 轉向持久化 React Dashboard。以前用完即丟的分析筆記本，現在變成可重複使用的工具。這看似是技術選擇，實則是思維轉變 — 從「分析一次」到「建立可重用資產」。
Instead of building one-off Jupyter notebooks that get discarded, the team now has Claude build permanent React dashboards that can be reused across future model evaluations.
團隊成員坦言「對 JavaScript 和 TypeScript 了解很少」，卻用 Claude Code 從零建立了整個 React 應用，包括一個 5,000 行的 TypeScript 應用。關鍵在於視覺化應用相對「低上下文」— 不需要理解整個 monorepo。
Despite knowing "very little JavaScript and TypeScript," the team uses Claude Code to build entire React applications, like a 5,000-line TypeScript app.
實際效益：日常重構任務達到 2-4 倍時間節省。但團隊也誠實地提到一個重要策略：當 Claude 的方案不好時，重新開始比試圖修正的成功率更高。
Starting over often has a higher success rate than trying to fix Claude's mistakes.
💡 比喻說明

從「用完即丟的便利貼」到「系統化的操作手冊」

Jupyter Notebook 像工廠裡的便利貼 — 記了一次分析就丟掉。React Dashboard 像標準操作程序 (SOP) — 每次新的模型評估都能重複使用。Claude Code 讓「建立 SOP」的門檻從「需要前端工程師」降低到「描述你要什麼」。

🛠️ 實務應用

「打斷求簡」策略：

監督 Claude 工作時，不要猶豫直接打斷它問「你為什麼要這樣做？試試更簡單的方法。」模型預設傾向更複雜的解決方案，但回應簡化要求時表現良好。

Part VII
API Knowledge API 知識團隊
Developer Team
API 知識團隊把 Claude Code 定位為每個任務的「第一站」(First Stop) — 任何 bug 修復、功能開發或分析，第一步都是問 Claude Code 應該檢查哪些檔案。這取代了傳統的「手動瀏覽程式碼庫 → 建立上下文 → 開始工作」的耗時流程。
The team uses Claude Code as their "first stop" for any task, asking it to identify which files to examine for bug fixes, feature development, or analysis.
一個微妙但重要的轉變：團隊成員現在有信心處理不熟悉領域的 bug，而不是找其他人幫忙。這不只是技術提升，更是心理上的賦能 — 團隊報告感到更快樂、更有生產力。
The team now has the confidence to tackle bugs in unfamiliar parts of the codebase instead of asking others for help. Team reports feeling happier and more productive.
📚 核心概念

迭代夥伴 vs. 一次性解決方案

團隊的關鍵建議：將 Claude Code 視為迭代夥伴 (Iterative Partner) 而非一次性解決方案。從最少資訊開始，讓 Claude 引導你走過流程，而非預先載入大量說明。這與 Prompt Engineering 的最佳實踐一致：漸進式提供上下文。

Part VIII
Growth Marketing 成長行銷團隊
Non-Technical Team
成長行銷團隊是一人團隊，但用 Claude Code 實現了「一人多角」的操作能力。最具代表性的案例：建立了一個 Agentic Workflow，處理包含數百個現有廣告及績效指標的 CSV 檔案，識別表現不佳的廣告進行迭代，並生成符合嚴格字數限制的新變體。
As a non-technical team of one, they use Claude Code to automate repetitive marketing tasks and create agentic workflows that would traditionally require significant engineering resources.
效率提升的數字非常驚人：廣告文案創作從 2 小時縮短到 15 分鐘，創意產出增加 10 倍。他們還建立了 Figma 外掛，能在半秒內批次生成 100 個廣告變體。
Ad copy creation reduced from 2 hours to 15 minutes. The team can now test vastly more ad variations with automated generation.
更深層的價值：他們建立了 Meta Ads MCP Server，直接在 Claude Desktop App 中查詢廣告績效，以及一個記錄假設和實驗的記憶系統，讓每次新的廣告迭代都能參考之前的測試結果。
They created an MCP server integrated with Meta Ads API and implemented a rudimentary memory system that logs hypotheses and experiments across ad iterations.
⚠️ 重點提示

非技術人員的 Claude Code 三步法

  • Step 1：先在 Claude.ai 充分構思和規劃整個工作流程
  • Step 2：讓 Claude.ai 將一切總結成 step-by-step 的實作 prompt
  • Step 3：進入 Claude Code 逐步執行，而非一次性要求完成

這個「先想後做」的模式是非技術人員成功採用 Claude Code 的關鍵。

🛠️ 實務應用

找到「有 API 的重複任務」：

尋找涉及重複操作且工具有 API 的工作流程（如廣告平台、設計工具、分析平台）。這是自動化的最佳候選場景。將複雜工作流程拆分成專門的子代理人（如標題 Agent vs. 描述 Agent），讓除錯更容易、輸出品質更高。

Part IX
Product Design 產品設計團隊
Non-Technical Team
產品設計團隊的案例最具革命性。設計師現在直接用 Claude Code 實作前端修改（字體、顏色、間距），工程師注意到他們正在做「通常不會看到設計師做的大型狀態管理變更」。Figma 和 Claude Code 佔據了他們 80% 的工作時間。
Engineers noted they're making "large state management changes that you typically wouldn't see a designer making." Figma and Claude Code open 80% of the time.
團隊描述了兩種截然不同的用戶體驗：開發者得到「增強型工作流程」（更快的執行），而非技術用戶得到的是「天啊，我變成開發者了」的工作流程（全新的能力）。
Developers get "augmented workflow" (faster execution), while non-technical users get "holy crap, I'm a developer workflow" (entirely new capabilities).
效率數字：複雜專案從一週協調時間縮短到兩個 30 分鐘的通話。例如，全面移除「research preview」訊息的工作，包含搜尋所有實例、審查周圍文案、與法務即時協調、實作更新。
Complex projects like GA launch messaging that would take a week of coordination now completed in two 30-minute calls.
🛠️ 實務應用

非技術人員的設定建議：

  • 請工程師協助初始設定（Repository 設定和權限）— 技術入門門檻高，但設定好後日常使用很順暢
  • 建立自訂 Memory 檔案，告訴 Claude「你是一個沒有編碼經驗的設計師，需要詳細說明和較小的漸進式變更」
  • 善用 Command+V 貼上截圖，Claude Code 擅長讀取設計並生成功能性程式碼
⚖️ 工具比較

GitHub Actions 整合模式：

設計師可以直接提交 Issue 描述需要的變更，Claude 透過 GitHub Actions 自動提出程式碼解決方案，不需要打開 Claude Code。這是「系統模式」的最佳範例 — AI 在背景自動運作，人類只需要描述需求。

Part X
Legal 法務團隊
Non-Technical Team
法務團隊的案例最出人意料也最有啟發性。一位團隊成員在一小時內建立了一個輔助溝通的無障礙 App，為有說話困難的家人設計了一個預測文字應用，使用原生語音轉文字功能，填補了語言治療師推薦的現有工具的空白。
In just one hour, they created a predictive text app using native speech-to-text that suggests responses and speaks them using voice banks.
在工作層面，法務團隊建立了「電話樹」系統協助員工找到合適的律師，以及 G Suite 應用自動化週報和追蹤法務審查狀態。這些都是不需要傳統開發資源就能建立的自訂工具。
They created prototype "phone tree" systems and G Suite applications that automate weekly team updates and track legal review status.
作為產品律師，他們也立即識別出 MCP 深度整合的安全隱患，指出保守的安全姿態將隨著 AI 工具存取更多敏感系統而產生障礙。這種「創新與風險管理的平衡」觀點，對受監管行業特別有價值。
They immediately identify security implications of deep MCP integrations, noting how conservative security postures will create barriers as AI tools access more sensitive systems.
💡 比喻說明

「不完美原型」的力量

法務團隊的最大心得：克服分享「玩具」或未完成作品的恐懼。就像在 CDMO 中，一個快速的概念驗證實驗（即使不完美）往往比完美的計畫書更能推動決策。分享原型能啟發跨部門的人看到他們未曾想過的可能性。

⚠️ 重點提示

法務團隊的合規視角：

Key Takeaways
1. 任務分類決定使用模式
Claude Code 的效能取決於你是否正確分類任務：外圍功能用非同步自主模式，核心邏輯用同步監督模式。這不是技術選擇，而是風險管理決策 — 與 GMP 的關鍵品質屬性 (CQA) 判斷邏輯完全一致。
2. Claude.md 是最被低估的槓桿
從資料基建到 RL 工程，每個團隊都強調 Claude.md 文件和自訂 Slash Commands 的重要性。這本質上就是 Context Engineering — 你投入多少上下文品質，就得到多少輸出品質。安全工程團隊獨佔 50% 的自訂指令使用量。
3. 非技術人員的「先想後做」模式
行銷、設計、法務三個非技術團隊共享同一模式：先在 Claude.ai 充分構思和規劃，再進入 Claude Code 逐步實作。設計團隊 80% 時間在 Figma + Claude Code 之間切換。法務團隊一小時就建立了無障礙 App。
4. 重新開始 > 修正錯誤
多個團隊不約而同地發現：當 Claude Code 的輸出不理想時，存好狀態、從頭來過的成功率高於試圖修正它的錯誤。首次嘗試成功率約三分之一，但加上 checkpoint + rollback 策略，整體仍然大幅節省時間。
Practice Questions
Q1 概念理解 根據文章中的案例，「同步模式 (Synchronous)」和「非同步模式 (Async)」的選擇標準是什麼？這兩種模式分別適合什麼類型的任務？

同步模式適用於觸及核心商業邏輯、關鍵修復、高風險變更的任務。產品開發團隊明確指出，對於涉及應用程式商業邏輯的核心功能，他們會同步工作、給出詳細 Prompt、即時監控程式碼品質和架構。

非同步模式 (Auto-accept)適用於外圍功能、原型設計、不熟悉領域的探索性工作。關鍵前提是從乾淨 git 狀態開始、定期提交 checkpoint，這樣 Claude 走偏時可以快速回滾。

底層邏輯是風險與可逆性：如果結果可以輕易撤銷（回滾 git），就適合非同步；如果錯誤代價高（觸及核心邏輯），就必須同步監督。

Q2 產業應用 假設你要在 CDMO 環境中導入 Claude Code 來自動化 Campaign Report 和 CoA 生成，基於本文中的最佳實踐，你會如何設計 Claude.md 文件和任務分類？

根據文件中多個團隊的經驗，Claude.md 設計需要包含：

1. 工作流程文件：詳細描述 Campaign Report 的標準格式、欄位定義、資料來源（仿資料基建團隊的方法）。

2. 防錯指令：加入具體的「不要做什麼」規則（仿 RL 工程團隊的 Claude.md 防重複犯錯指令），例如「不要修改驗證過的批次號碼欄位」、「所有數值保留小數點後兩位」。

3. 任務分類：CoA 中的數據填入屬於「低風險、可驗證」— 適合非同步模式。但最終的品質放行判斷屬於「核心品質判斷」— 必須人工同步審查（Copilot Mode）。這與產品開發團隊的「外圍 vs. 核心」分類法完全一致。

4. 合規護欄：仿安全工程團隊的 MCP 建議，使用受控的資料存取通道而非直接 CLI 存取，確保 21 CFR Part 11 的審計軌跡完整性。

Q3 批判思考 這份文件來自 Anthropic 內部，所有案例都使用 Anthropic 自己的產品。這對文件的可信度和可推廣性有什麼影響？哪些發現最可能、哪些最不可能適用於外部組織？

潛在偏差：Anthropic 員工比一般用戶更了解 Claude 的能力邊界，也更有動機展示正面結果。他們的技術環境（monorepo 結構、模型開發基礎設施）遠比典型企業複雜。因此，效率數字（如 80% 時間節省）需要打折看待。

最可推廣的發現：Claude.md 的價值、任務分類框架、checkpoint + rollback 策略、非技術人員的「先想後做」模式。這些都是流程和方法論層面的洞察，不依賴特定技術環境。

最不可推廣的發現：5,000 行 TypeScript app 的開發、Monorepo 內的跨團隊 codebase 導航、特定的 GitHub Actions 整合。這些依賴於 Anthropic 的特定技術堆疊和 Claude Code 對自家程式碼庫的高度熟悉度。

遺漏的部分：文件完全沒有討論失敗案例的細節、成本數據、學習曲線所需時間、以及安全性和資料外洩風險。這些是外部組織評估採用時最需要的資訊。
