Executive Summary

Aniket Panjwani 整理了九種使用 Claude Code 爬取網路資料的方法，涵蓋從最簡單的「直接叫 Claude Code 爬」到使用 Agent Browser 處理認證登入的進階場景。文章的核心洞察是：資料爬取的難度取決於目標網站的結構化程度和認證機制，而 Claude Code 作為 agentic coding 工具，可以透過搭配不同的外部工具來應對各種場景。這是一份偏「工具清單 + 使用場景」的實戰型文章，技術深度不高但覆蓋面廣，適合作為快速參考手冊。

Content 內容
Way 1
Just Ask Claude Code to Scrape 直接叫 Claude Code 爬
對於大量的網站，你可以直接告訴 Claude Code 去爬取該網站，告訴它你要什麼資料，然後請它輸出到 CSV 或 SQLite 檔案。
For a large set of sites, you can just tell Claude Code to scrape the site, tell it what you want scraped, and ask it to write it out to a CSV or a SQLite file.
Claude Code 會自行探索網站，可能會寫一個 Python 腳本，執行腳本，甚至可能寫一些單元測試，然後把資料寫到你電腦上的某個位置。
It will poke around the site for you, probably write a Python script, run the script, maybe even write some unit tests, and then just write out the data somewhere on your computer.
🛠️ 實務應用

這是最「零配置」的方法。適用於結構清楚的公開網站（如政府資料、學術機構頁面、公開產品列表）。Claude Code 會自己決定用 requests + BeautifulSoup 還是其他方式。

⚠️ 重點提示

適用場景：靜態 HTML 頁面、公開資料集、簡單表格

限制：動態載入內容（SPA）、需要認證的頁面、有反爬蟲機制的網站

輸出格式：CSV, SQLite, JSON

💡 比喻說明

想像你叫一個很聰明的助理去圖書館幫你影印特定章節 — 如果那本書就在開架區（公開網站），助理自己就能搞定。但如果書鎖在特藏室（需要認證），你得先拿到鑰匙給他。

Way 2
Ask Claude Code to Find Endpoints 尋找 API 端點
很多有趣的資料不是以靜態頁面呈現，而是透過某個 API 呼叫動態載入。有時候 Claude Code 會自行逆向工程找到那個 API 呼叫，但有時候你需要推它一把，明確告訴它去找。
A lot of interesting data is not rendered as a static page, but is being loaded dynamically via some API call. Sometimes Claude Code will reverse engineer that API call itself, but sometimes you have to nudge it along.
唯一的差別是：在這個方法中你要求它尋找 endpoint。只要給出那個詞、那個小小的提示，有時候就能得到比直接叫它爬取更好的結果。
The only difference is that in this method you ask it to look for endpoints. Just giving that word, that little nudge, will sometimes get you better results.
📚 核心概念：Prompt Nudge

Prompt Nudge 提示推力：Claude Code 本身有能力做很多事，但特定關鍵字能引導它走更有效率的路徑。「endpoint」這個詞就像告訴它：別爬 HTML，去找背後的 JSON API。這和 Prompt Engineering 的 specificity principle 完全一致。

⚠️ 重點提示

適用場景：SPA (Single Page App)、動態載入的資料表、需要 AJAX 呼叫的網站

關鍵詞提示：在 prompt 中使用 "find the API endpoint"、"look for the data endpoint"、"reverse engineer the API"

優勢：直接拿到結構化 JSON，省去 HTML 解析

🛠️ CDMO 應用

如果你需要爬取藥品定價資料庫、臨床試驗資料（如 ClinicalTrials.gov 的 API），或供應商報價平台，這種「找 endpoint」的方法往往比直接爬 HTML 更穩定、更快。

Way 3
ScrapeCreators 社群媒體專用 API
大多數社群媒體網站上有很多有用的資料是可以爬取的，但它們把 endpoint 設計得特別難逆向工程。它們有自己的反機器人規則，而且每週都會更換 selector 的運作方式。
A lot of useful data on most social media sites is scrapeable, but they make their endpoints particularly difficult to reverse engineer. They have their own anti-bot rules, and they change the way selectors work each week.
作者推薦使用 Scrape Creators（scrapecreators.com），它提供幾乎每個社群媒體的 API 端點。作者建議建立一個 Skill 來封裝這些端點，作為一次性工具讓 agentic coding 工具永遠可以存取。
I would recommend creating a skill for the Scrape Creators endpoints as a one-time utility that then your agentic coding tool will always have access to.
📚 Skill 設計模式

重要的架構概念：將外部 API 封裝為 Claude Code Skill。一次性設定好 API key、端點格式、常用參數，之後每次爬取只需告訴 Claude Code「用 ScrapeCreators 爬取 X」。

這和 Master Prompt 系統的設計思維一致 — 前期投資建立可複用模板，後期邊際成本趨近於零。

💰 成本分析

ScrapeCreators：付費 API 服務

ROI 考量：如果每週都需要爬社群媒體，付費服務的穩定性遠勝自行維護 scraper。偶爾用的話可能不值得。

Way 4
Apify Actor 爬蟲市集
Apify 是一個爬蟲市集。對於很多難以爬取的網站，已經有人做好了可租用的爬蟲，在 Apify 上叫做「actors」。
Apify is a marketplace of scrapers. For a lot of difficult-to-scrape websites, people have made rentable scrapers that are available on Apify (called "actors").
作者喜歡用的一個是 Google Maps 爬蟲，對社會科學家做直接分析或建立代理指標很有用，也適合商業人士做競爭分析或尋找當地潛在客戶。
One scraper I like to use is the Google Maps scraper, which can be quite useful for social scientists or for business people to do competition analysis or to find local leads.
唯一的問題是你必須付費。有些按使用量計費，有些需要按月租用。
The only problem is you have to pay for these. Some you pay by usage. Some you have to rent by month.
⚖️ 工具比較

Apify vs ScrapeCreators：ScrapeCreators 專注社群媒體；Apify 是通用市集，覆蓋 Google Maps、電商、新聞等更廣泛場景。

Apify vs 自建爬蟲：自建零成本但維護成本高（網站改版就壞）；Apify actor 由社群維護，穩定性較高但需付費。

🛠️ CDMO 應用

Google Maps 爬蟲可用於市場調查 — 搜尋特定區域的 CDMO 設施、診所分布、藥局密度等，作為 Feasibility Assessment 的輔助資料來源。

Way 5
Firecrawl → Markdown → Structured Extraction
很多你想要的資料並不是高度結構化的。例如作者在做 EconNow 專案時，需要爬取大量經濟學求職市場候選人頁面，每個頁面都有自己的 HTML 結構。
A lot of data that you're going to want to get is not going to be highly structured. Each of these pages had their own HTML structure.
常見的技術是將網頁轉為 Markdown，然後讓 LLM 解析 Markdown 並建立結構化輸出。Firecrawl 是一個付費服務，可以輕鬆將網頁轉為 Markdown。
A common technique is to turn the web page into Markdown and then have an LLM parse the Markdown and create some kind of structured output.
如果你正確設定了 structured outputs，你就能讓 LLM 從不同的非結構化網站中解析出特定欄位。
If you set up your structured outputs correctly, you'll be able to get the LLM to parse out particular kinds of fields from these different varying unstructured websites.
📚 HTML → Markdown → JSON Pipeline

三階段資料管線：

1. HTML → Markdown：去除樣式和結構雜訊，保留純內容

2. Markdown → LLM：用 LLM 理解語義，不依賴固定 selector

3. LLM → Structured JSON：透過 structured outputs 確保輸出格式一致

關鍵優勢：不需要為每個網站寫不同的 scraper。

💰 成本分析

Firecrawl：付費服務，edge case 處理好

OpenAI Structured Outputs：API 呼叫費用，規模化時需計算 token 成本

作者建議：預算有限就用免費替代方案

Way 6
DIY HTML → Markdown → Structured Extraction
你可能會問：為什麼要付費用 Firecrawl？自己轉成 Markdown 不就好了？答案是可以的。有免費工具：turndown（mixmark-io）和 markitdown（Microsoft）。
Can't you just turn it into Markdown yourself? And the answer is yeah, you can. There are tools that allow you to do that yourself too.
作者使用 Firecrawl 的原因是它在邊緣情境的處理更好。預算小的話就用免費套件。
The reason I use Firecrawl is because I find it handles certain edge cases better. If you're on a small budget, just use one of these packages instead.
另一個重點：對於小規模的資料，你甚至不需要呼叫外部 API。可以讓 Claude Code 自己做結構化提取。但數千份文件的話就不實際了。
You don't even need to send the Markdown to an external API call for small-scale stuff. You could just have Claude Code itself do the structured extraction.

面向FirecrawlDIY

⚠️ 規模決策點

~100 頁：Claude Code 直接做結構化提取

1000+ 頁：Firecrawl + OpenAI API batch

Way 7
yt-dlp — YouTube 資料爬取
yt-dlp 是一個工具，可以爬取任何 YouTube 影片、其 metadata 和字幕。
yt-dlp is a tool which lets you scrape any YouTube video, its metadata, and the subtitles of the videos.
作者基本上不再看影片了。他下載字幕，然後讓 Claude Code 為他建立個人化摘要，將影片套用到他實際關心的脈絡中。
I basically never watch videos anymore. I'll just download the subtitles and then have Claude Code create a personalized summary for me.
YouTube 影片中有大量有用的資料，作者認為這個工具特別被低估。
There is a ton of useful data in YouTube videos, and I really think that this tool in particular is highly under-exploited.
🛠️ 學習系統整合

這和我們的 AI 學習系統高度相關。目前 source 以文章為主，但 YouTube 頻道（AI Maker Labs, Ethan Mollick 演講等）也有大量知識。

潛在 workflow：yt-dlp 下載字幕 → Claude Code 轉為結構化筆記 → 匯入 Master Prompt 系統生成教育文件。

💡 比喻說明

把 yt-dlp 想像成「影片→文字」的翻譯機。就像我們把文章餵進 Master Prompt 系統一樣，yt-dlp + Claude Code 可以把影片餵進同樣的知識管線。影片不再是「需要花時間看」的內容，而是可以被程式化處理的資料來源。

Way 8
Reddit JSON Endpoint
Reddit 有一個 JSON 端點，可以用來找到幾乎任何內容。你只需在 Reddit URL 的末尾加上 .json，你的 agentic coding 工具就能以 JSON 文件存取 Reddit 的所有資料。
Reddit has a JSON endpoint. You just add on ".json" to the end of a Reddit URL, and then your agentic coding tool has access to everything on that part of Reddit as a JSON document.
作者設定了一些 Skill，用來追蹤各種他關心的 subreddit 上的討論動態。那些 Skill 就是 Claude Code 去打 Reddit 的 JSON 端點。
I have some skills set up to keep a pulse on what people are talking about on a large variety of subreddits. Those skills are just Claude Code hitting the JSON Reddit endpoints.
💻 技術細節

範例端點：

https://old.reddit.com/r/claudecode.json

直接回傳 JSON 結構，包含 post 標題、內容、投票數、留言等。不需要 API key，不需要 OAuth — 最低門檻的結構化資料來源。

🛠️ Skill 設計靈感

可建立一個 Skill，定期爬取指定 subreddit（如 r/claudecode, r/pharmaceutical 等），摘要熱門話題，追蹤特定關鍵字趨勢。這是 Claude Code 作為「情報收集 agent」的實戰範例。

Way 9
Agent Browser + Credentials 認證瀏覽
有很多網站受到某種認證保護。要通過認證，有兩種方式。
There's a lot of sites that are protected behind some kind of authentication. You have two possible approaches.
第一，你可以進行認證交換，取得 cookie，然後 Claude Code 使用該 cookie 來認證並查看需要登入的內容。
First, you can do that authentication exchange. Then you will sometimes get a cookie, and then Claude Code can use that cookie to authenticate.
另一個選擇是使用 Vercel 的 Agent Browser（agent-browser.dev）— 一個為 agent 優化的瀏覽器自動化 CLI。
The other option is to use Agent Browser by Vercel. It's a browser automation CLI created by Vercel which is optimized for use by agents.
你可以將登入憑證存放在 Claude Code 可存取的地方（安全金鑰交換，或直接放在 .env 檔案中），然後建立一個 Skill 讓 Claude Code 用 Agent Browser 登入、爬取資料。
You could store your login credentials somewhere that Claude Code has access to, either in some online vault or just yolo it and store it in your environment variables or in some .env file.
⚠️ 安全性注意

憑證管理：作者提到可以「yolo」直接存在 .env — 個人使用或許可接受，企業環境中絕對不行。

GMP 環境警示：在製藥/CDMO 環境中，任何涉及認證的自動化都需要考慮 21 CFR Part 11 的電子記錄和電子簽名要求。即使是「above the GMP line」的研究工具，憑證管理也應遵循基本安全原則。

📚 Agent Browser 定位

Agent Browser 是 Vercel 推出的工具，讓 AI agent 能像人類一樣瀏覽網頁。它解決了傳統 headless browser（如 Puppeteer）在 agent 整合上的痛點 — 更簡單的 CLI 介面、更好的錯誤處理、針對 agentic workflow 優化。

Key Takeaways 重點整理
1. 複雜度決定方法
從靜態 HTML（Way 1）到認證登入（Way 9），選擇方法的核心考量是目標網站的複雜度和結構化程度。永遠從最簡單的方法開始嘗試。
2. Prompt Nudge 的威力
Way 2 的核心洞察：只要在 prompt 中加入 "endpoint" 這個詞，就能引導 Claude Code 走更有效的路徑。關鍵字的選擇就是 prompt engineering 的實戰。
3. Skill 封裝是力量倍增器
將外部工具（ScrapeCreators, Reddit JSON, yt-dlp）封裝為 Claude Code Skill，是一次性投資、長期回報的架構決策。和 Master Prompt 系統的設計哲學一致。
4. LLM 取代 Selector
HTML → Markdown → LLM 的管線（Ways 5-6）代表爬蟲技術的典範轉移：從依賴脆弱的 CSS selector 到使用 LLM 的語義理解。降低維護成本但增加 token 成本。
Practice Questions 練習題
Q1 概念理解 Way 1（直接爬取）和 Way 2（尋找 endpoint）的結果有什麼本質差異？在什麼情況下 Way 2 會比 Way 1 好？

本質差異：Way 1 爬取的是 rendered HTML（瀏覽器看到的頁面），需要解析 DOM 結構；Way 2 找到的是底層 API 回傳的 JSON 資料，已經是結構化的。

Way 2 更好的情況：當網站是 SPA（Single Page Application）時，HTML 只是一個空殼，實際資料都是透過 API 動態載入。直接爬 HTML 只會拿到一個空的 div，而找到 endpoint 則能直接拿到完整的結構化資料。此外，API 端點通常比 HTML 結構更穩定，不會因為前端 redesign 而失效。

Q2 產業應用 在 CDMO 的日常工作中，哪三種爬取方法最有可能派上用場？請考慮合規性限制。

最可能有用的三種：

1. Way 2（Find Endpoints）：適合爬取 ClinicalTrials.gov、FDA database 等有公開 API 的政府資料庫。資料公開，無合規顧慮。

2. Way 7（yt-dlp）：適合爬取產業研討會、KOL 演講的字幕，轉為知識筆記。不涉及機密資料，適合「above the GMP line」的學習和市場情報收集。

3. Way 5/6（Markdown + LLM）：適合爬取競爭對手的公開網站進行 competitive intelligence 分析。注意：即使資料公開，仍需確認不違反網站的 Terms of Service。

合規性提醒：爬取到的資料如果用於決策文件（如 Feasibility Assessment），必須作為「參考資料」而非「官方來源」，並標注資料來源和取得方式。

Q3 批判思考 作者對 Agent Browser + credentials 的態度是「yolo it」。從資安和 AI agent 治理的角度，這種態度的風險是什麼？

風險分析：

1. 憑證洩漏：.env 檔案如果不小心 commit 到 Git repo，憑證就公開了。這是最常見也最危險的情境。

2. Agent 權限邊界不清：給 AI agent 完整的登入憑證，等於讓它可以做「你能做的所有事情」。如果 agent 行為不符預期，造成的損害可能超出爬取範圍。

3. 稽核追蹤缺失：Agent 用你的帳號登入後的操作，在 audit log 中記錄為「你」的操作。你需要為此負責。

建議做法：使用專門的「bot 帳號」、限制 API scope、使用 vault（如 1Password CLI）而非 .env、確保每次 agent 操作都有本地 log。

🎯 我的行動結論
✔ 核心洞察
資料爬取不是「一種方法用到底」— 它是一個工具選擇矩陣，根據目標網站的複雜度（靜態/動態/認證）和資料結構化程度來決定最佳方法。最重要的設計模式是「將工具封裝為 Skill」，這和我目前 Master Prompt 系統的設計哲學完全一致：前期投資 → 長期複用。
📈 立即應用
本週可做：(1) 測試 yt-dlp 下載一支 AI 影片的字幕，用 Claude Code 生成摘要，驗證是否可以整合進學習系統的 source pipeline；(2) 試打 Reddit JSON endpoint（r/claudecode.json），評估是否值得建立一個定期監測 Skill。
⚠ 需要補充
文章缺少：(1) 反爬蟲機制的深度討論（rate limiting, CAPTCHA, IP blocking）；(2) 法律和 Terms of Service 的考量 — 能爬不代表該爬；(3) 資料品質驗證 — LLM 提取的結構化資料如何確保準確性。建議 3 個月後重讀，屆時 Agent Browser 和 Claude Code 可能都有重大更新。
