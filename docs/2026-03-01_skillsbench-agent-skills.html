SkillsBench: Agent Skills 到底有多有效？

首個系統性基準測試 - 86 任務、11 領域、7,308 軌跡的實證研究
Source: Xiangyi Li et al. (BenchFlow / Amazon / 多校聯合) | arXiv:2602.12670 | Feb 2026 Preprint
★★★★★ 4.5/5 ↑ Technical Depth, Practical Value ↓ Timeliness (Preprint)

Executive Summary

這是第一個將 Agent Skills 作為第一級評估物件的基準測試。研究團隊在 11 個領域的 84 個任務上，測試了 7 種模型-代理組合，產生 7,308 條執行軌跡。核心發現：人工策劃的 Skills（Curated Skills）平均提升通過率 +16.2 個百分點，但效果因領域差異巨大（Healthcare +51.9pp vs Software Engineering +4.5pp）。最震撼的發現：模型自行生成的 Skills 幾乎無效 -- AI 無法可靠地撰寫它自己受益的程序性知識。此外，2-3 個聚焦 Skills 優於大量綜合文件，小模型搭配好的 Skills 可以擊敗無 Skills 的大模型。

Part I: Agent Skills 是什麼？

Agent Skills 是結構化的程序性知識包，在推論時增強 LLM Agent 的能力。它不需要微調模型 -- 而是在執行階段提供「標準作業程序（SOP）、領域慣例、任務特定啟發式方法」等指引。
"Agent Skills are structured packages of procedural knowledge that augment LLM agents at inference time."
一個 Skill 包含：SKILL.md（自然語言指令）加上可選的資源（腳本、模板、範例）。這個定義明確排除了 system prompts、few-shot examples、RAG 檢索和工具文件 -- 因為它們缺乏結構性和程序性。
"A Skill is a structured package comprising instructions, code templates, resources, and verification logic."
論文用了一個精妙的類比：基礎模型是 CPU，Agent Harness（如 Claude Code、Gemini CLI）是作業系統，Skills 則是應用程式。這個三層架構解釋了為何 Skills 可以跨模型、跨平台移植。
"Foundation models provide base capabilities (analogous to CPUs), agent harnesses orchestrate context and tools (operating systems), and Skills extend competence to specialized domains (applications)."
💡 比喻說明

想像你雇了一個非常聰明的新員工（LLM）。他什麼都能聊、邏輯很好，但不知道你們公司的 SOP。

Vanilla（無 Skills） = 把任務丟給他，讓他自己摸索

Curated Skills = 給他一份資深同事寫的操作手冊

Self-Generated Skills = 讓他自己先寫操作手冊，再照著做

研究證實：給手冊有效（+16.2pp），讓他自己寫手冊則幾乎沒幫助。

📚 Skill 組成結構
  • SKILL.md -- 自然語言指令（工作流程、SOP、領域慣例）
  • Resources/ -- 可執行腳本、程式碼模板、參考文件、範例
  • 基於檔案系統 -- 易於編輯、版本控制、分享、跨平台使用

Part II: 基準測試設計

SkillsBench 建構在 Harbor 框架之上，每個任務都是獨立的 Docker 容器，包含環境、Skills、確定性驗證器和 Oracle 解答。最關鍵的方法論創新：配對評估（Paired Evaluation） -- 每個任務都在「無 Skills」和「有 Skills」兩個條件下執行，直接測量 Skills 效果的 delta。
"Unlike Terminal-Bench, which evaluates raw model capability, SkillsBench evaluates every task under both vanilla and Skills-augmented conditions, enabling direct measurement of Skills efficacy."
嚴格的品質控制體系：任務指令由人類撰寫（經 GPTZero 驗證），驗證器使用程式化斷言（非 LLM-as-judge），CI 管線進行洩漏審計確保 Skills 提供指引而非直接答案。322 份候選提交中最終篩選出 86 個任務。
"Deterministic verifiers with programmatic assertions - no LLM-as-judge variance."
任務按難度分層：Core（17 題，<60 分鐘）、Extended（42 題，1-4 小時）、Extreme（26 題，>4 小時）。涵蓋 11 個領域，從軟體工程到醫療保健、製造業、金融等。
"86 tasks spanning 11 domains, stratified by difficulty based on estimated human completion time."
⚠️ 重點提示：為什麼是 Deterministic Verifier？

許多 AI 評測使用「LLM-as-judge」（讓另一個 AI 打分），但這引入了主觀性和可變性。SkillsBench 使用 pytest 程式化斷言：答案對就是對、錯就是錯，0 或 1。

這對 CDMO 產業來說特別有共鳴 -- 就像我們的品質檢驗不能用「看起來差不多」來判定，必須有明確的規格標準。

⚖️ Agent Harness 比較

測試了三個主要 Agent Harness：

  • Claude Code (Anthropic) -- 原生 Skills 整合，支援 4 個模型（Opus 4.5/4.6, Sonnet 4.5, Haiku 4.5）
  • Gemini CLI (Google) -- Flash 3 / Pro 3
  • Codex CLI (OpenAI) -- GPT-5.2

Claude Code + Opus 4.5 展現最高絕對增益 (+23.3pp)，反映其對 Skills 規範的原生優化。

Part III: 核心發現 -- 七大結論

Finding 1：Skills 提供顯著但可變的效益。Curated Skills 在所有 7 個模型配置中都提升了表現，平均 +16.2pp，絕對增益從 +13.6pp 到 +23.3pp 不等。
"Curated Skills raise average pass rate by 16.2 percentage points, but effects vary widely."
Finding 2：Gemini CLI + Flash 3 達到最高整體表現（48.7% with Skills），但 Claude Code + Opus 4.5 有最高增益 (+23.3pp)。
Finding 3：Self-generated Skills 幾乎無效。這是最具顛覆性的發現。模型被要求在解題前先生成自己的 Skills，但平均效果接近零甚至為負。軌跡分析揭示兩個失敗模式：(1) 模型知道需要特定 Skills 但生成的程序不夠精確；(2) 對高領域知識任務，模型根本沒意識到需要專業 Skills。
"Self-generated Skills provide no benefit on average, showing that models cannot reliably author the procedural knowledge they benefit from consuming."
Finding 4：Skills 效益因領域差異極大。Healthcare (+51.9pp) 和 Manufacturing (+41.9pp) 受益巨大，而 Software Engineering (+4.5pp) 和 Mathematics (+6.0pp) 效益極小。解釋：模型在訓練數據充足的領域已經很強，外部程序性指引的邊際效益低。
Finding 5：2-3 個 Skills 是最佳數量。超過 4 個 Skills 出現邊際遞減。聚焦、精簡的程序性指引每次都勝過詳盡的文件庫。
"Focused Skills with 2-3 modules outperform comprehensive documentation."
Finding 6：中等長度 Skills 優於綜合性文件。Compact Skills delta +18.9pp，而 comprehensive documentation 僅 +5.7pp -- 差距近 4 倍。Agent 難以從冗長內容中提取相關資訊。
Finding 7：小模型+Skills 可以超越無 Skills 的大模型。Claude Haiku 4.5 with Skills (27.7%) 顯著超越 Opus 4.5 without Skills (22.0%)。這對成本敏感的部署方案有重大意義。

💡 CDMO 視角解讀

Healthcare +51.9pp 和 Manufacturing +41.9pp 這兩個數字對我們特別有意義。這些領域的共同特點：高度程序化、嚴格合規要求、專業知識密集 -- 正是 CDMO 的核心特徵。

這暗示：為我們的偏差調查、批次記錄管理、Tech Transfer 流程撰寫精準的 Skills，可能帶來遠超平均水平的效益。

Part IV: Skills 設計原則與實務啟示

Skills 的分層消融實驗（L0-L3）揭示了最大的效能躍升來自 L2 到 L3 -- 也就是加入可執行程式碼範例和參考資源的那一步。僅有 SKILL.md 文字指引是不夠的；程序性內容（procedural content）才是驅動改善的關鍵，不僅僅是「額外的上下文」。
"The biggest performance jump comes from adding executable code examples and reference resources."
Hacker News 社群對這篇論文提出了重要的補充觀點：論文測試的「Self-generated Skills」是讓模型從零開始撰寫 Skills，但實務中更常見的做法是讓模型從解題過程中歸納經驗再寫成 Skills。這兩種情境的效果可能截然不同，論文並未涵蓋後者。
"Having the LLM write down a skill representing the lessons from the struggle you just had to get something done is quite different from what they're referring to." -- Hacker News commenter
Skills 不是萬靈丹。16/84 個任務出現負面效果，最嚴重的 taxonomy-tree-merge 下降了 39.3pp。這意味著：盲目套用 Skills 是有害的 -- 必須針對特定領域和任務進行部署策略評估。
🛠️ 實務應用：Skills 設計清單

Do:

  • 聚焦 2-3 個 Skills，每個針對一類任務
  • 在 SKILL.md 中包含可執行的程式碼範例
  • 保持精簡 -- compact > comprehensive
  • 針對模型訓練數據不足的領域優先部署
  • 建立 A/B 測試機制衡量 Skills 實際效果

Don't:

  • 把所有文件堆成一個巨大的 Skills 包
  • 依賴模型自行生成 Skills
  • 對已知模型表現良好的領域強加 Skills
  • 忽略 Skills 可能造成負面效果的風險
💰 成本分析啟示

Haiku 4.5 + Skills (27.7%) vs Opus 4.5 without Skills (22.0%)

如果 Haiku 的 API 成本是 Opus 的 1/10 -- 那麼好的 Skills 投資帶來的 ROI 是驚人的。策略：在受控環境中先用小模型+Skills 測試，確認可行後再決定是否需要升級模型。

Key Takeaways

📊
Curated Skills 有效但非萬能
平均 +16.2pp 的提升很顯著，但領域差異從 +4.5pp 到 +51.9pp，且 16/84 個任務出現負效果。Skills 部署需要領域特定策略，不是「裝了就好」。
🚨
AI 無法自行撰寫有效 Skills
Self-generated Skills 效果約等於零。這是對「完全自主 AI Agent」願景的現實校正 -- 程序性知識必須由人類專家策劃，至少在目前階段如此。
✂️
Less is More -- 2-3 個 Skills 最佳
Compact Skills (+18.9pp) 比 comprehensive documentation (+5.7pp) 強近 4 倍。聚焦的 SKILL.md 加上可執行範例，優於堆砌大量文件。Agent 跟人一樣：太多資訊反而處理不好。
💱
Skills 改變模型規模經濟學
Haiku + Skills 超越 Opus without Skills。這意味著投資在 Skills 策劃可能比升級到更貴的模型更划算，對成本敏感的企業部署尤其重要。

Practice Questions

Q1 概念理解 為什麼 Self-generated Skills 無效，但 Curated Skills 有效？這告訴我們關於 LLM 能力的什麼？

LLM 擅長消費程序性知識（按照指令執行），但不擅長生產程序性知識（從零撰寫有效程序）。這反映了「知道怎麼做」和「知道該做什麼」的根本差異。模型在提示下能執行複雜步驟，但缺乏識別和編碼領域專業知識的後設認知能力。

兩個失敗模式：(1) 知道需要但不精確 -- 如「用 pandas 處理數據」但沒有具體 API pattern；(2) 根本不知道需要 -- 對高領域知識任務缺乏意識。這與 Copilot vs Agent 模式的區分完全吻合。

Q2 產業應用 如果你要為 CDMO 的偏差調查流程設計 Agent Skills，基於本論文的發現，你會怎麼設計？

基於 Finding 4（Healthcare/Manufacturing 受益最大）和 Finding 5（2-3 Skills 最佳），我的設計策略：

Skill 1：偏差分類 SOP -- SKILL.md 包含偏差分類決策樹（Critical/Major/Minor）、根據 PIC/S 和 21 CFR Part 211 的判定標準，加上 2-3 個歷史案例作為範例。

Skill 2：根因分析模板 -- 魚骨圖結構化提示、5 Why 分析框架、CAPA 連結邏輯。包含可執行的模板（Finding 6：程式碼範例帶來最大躍升）。

不要做：不要把整個 SOP 庫塞進去（comprehensive documentation 只有 +5.7pp），不要讓 AI 自己生成流程（Self-generated ~0pp）。

Q3 批判思考 Hacker News 社群指出論文的 Self-generated Skills 測試方法與實務差異。你如何評估這個批評？

這個批評非常合理且重要。論文測試的是「從零生成 Skills」（cold start），但實務中更常見的模式是「從經驗歸納 Skills」（reflection-based）-- 讓 AI 在執行任務後，把學到的經驗整理成 Skills 供未來使用。

這兩者的差異類似於：讓新人「憑空寫 SOP」 vs 讓資深員工「把他的經驗整理成 SOP」。後者的成功率顯然更高。

但這並不完全推翻論文結論 -- 即使是 reflection-based Skills，品質仍可能不如人類專家策劃的版本。更可能的結論是：AI-assisted + Human-reviewed Skills 才是最佳實務，而非純粹的 AI 或純粹的人類。

🎯 我的行動結論
✔ 核心洞察
這篇論文用 7,308 條軌跡的硬數據證實了我一直在實踐的「Copilot 模式」哲學 -- AI 需要人類撰寫的結構化指引才能真正發揮價值。Self-generated Skills 的失敗直接支持了「wetware before software」原則：程序性知識的策劃是不可替代的人類判斷工作。Healthcare (+51.9pp) 和 Manufacturing (+41.9pp) 的數據更確認了 CDMO 是 Skills 高回報領域。
📈 立即應用
本週行動：(1) 審視我目前的 Claude Project Knowledge，用「2-3 個聚焦 Skills」原則重構，確保每個 Skill 包含可執行範例而非只有文字描述。(2) 為 CDMO Feasibility Assessment Skill 加入具體的決策樹和計算模板（L2→L3 提升）。(3) 將此論文的數據納入我的「AI 在製藥業 ROI」簡報素材 -- 特別是「小模型+Skills 超越大模型」的成本論點。
⚠ 需要補充
論文未涵蓋 reflection-based Skills 生成（HN 社群的有效批評）。也未測試 Skills 的長期維護成本 -- 在快速迭代的 AI 工具環境中，Skills 多久需要更新？此外，11 個領域中沒有直接對應「藥品製造 GMP 合規」的任務，Healthcare 類別的具體任務內容值得進一步查看。3 個月後重讀時，應同時查看是否有 follow-up 研究填補了 reflection-based Skills 的空白。
