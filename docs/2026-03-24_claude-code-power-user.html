Executive Summary

Matt Van Horn 分享了他使用 Claude Code 的完整工作流系統：以 plan.md 檔案作為一切工作的核心起點，結合語音輸入、4-6 路並行工作階段、自動化設定、以及 /last30days 研究工具，將傳統「80% 寫程式 + 20% 規劃」的比例徹底翻轉。他的方法不僅適用於程式開發，也延伸到策略文件、會議記錄轉化、開源貢獻、甚至迪士尼旅遊規劃。核心主張：No IDE. Just plan.md files and voice.

批判性觀察：本文為個人經驗分享，使用者為高度熟練的 power user（263 commits/30 days），部分工作流（如 dangerously skip permissions）有安全風險。文中提到的 Compound Engineering Plugin 來自 Every Inc（與作者有合作關係），需注意利益關係。多工並行的效率宣稱缺乏量化對照資料。

Content Sections

Part I: Plan-First 哲學 — 一切從 /ce:plan 開始

Van Horn 最核心的觀點：當你有任何想法的瞬間，輸入 /ce:plan。不是「讓我想想」，不是「讓我開始寫程式」。產品點子、GitHub bug、終端機錯誤截圖 — 全部先進入規劃階段。
"The single most important thing I've learned: the moment I have an idea, it's /ce:plan. Not 'let me think about this.' Not 'let me start coding.' /ce:plan. Every time."
執行 /ce:plan 時，底層會同時啟動多個研究 Agent：一個分析你的程式碼庫（讀取檔案、找出慣例模式），另一個搜尋 docs/solutions/ 資料夾裡過去的 bug 修復紀錄。如果主題需要，還會有更多 Agent 同步研究外部最佳實踐和框架文件。
"One analyzes your codebase - reads your files, finds patterns, checks your conventions. Another searches your docs/solutions/ for learnings from past bugs."
最終產出一份結構化的 plan.md：問題描述、建議方法、需要修改的檔案、含核取方塊的驗收標準、以及從你自己程式碼中提取的設計模式。不是泛用建議，而是植根於你的程式碼庫、你的慣例、你的歷史。
"Then it consolidates and writes a structured plan.md: what's wrong, what approach to take, which files to touch, acceptance criteria with checkboxes, patterns to follow from your own code."
/ce:work 接手執行計畫：分解任務、逐一實作、執行測試、勾選標準。Context 中斷時？開啟新 session，指向 plan.md，從斷點繼續。plan.md 就是能存活於一切中斷的檢查點。
"Context gets lost? Start a new session, point it at the plan, pick up where you left off. The plan is the checkpoint that survives everything."
📚 核心概念：80/20 翻轉

傳統開發是 80% 寫程式 + 20% 規劃。Van Horn 的方法完全翻轉：80% 時間在規劃（用 Opus 思考），20% 讓 subagent 蜂群式執行。

這與 Anthropic 官方「Building Effective Agents」文章的理念一致：AI Agent 的價值不在取代人類編碼，而在將人類判斷力集中在規劃層，把執行層委託給 AI。

⚠️ 重點提示：Compound Engineering Plugin

/ce:plan 和 /ce:work 來自 Compound Engineering Plugin（Every Inc 開發）。安裝指令：

/plugin marketplace add EveryInc/compound-engineering-plugin

Van Horn 是此 Plugin 的 #3 貢獻者（21 commits），與 Every Inc 有合作關係。評估其推薦時需考慮此利益關係。

💡 CDMO 類比

plan.md 的角色類似 批次生產紀錄（Batch Record）：在任何製造步驟開始前，先確認配方、設備、驗收標準全部到位。就像 GMP 要求的「先審核再執行」，Van Horn 要求的是「先 plan 再 build」。

context 中斷後用 plan.md 恢復，等同於製程異常後的偏差調查 + 復工程序。

Part II: 語音輸入 — 用說的比打的快

Van Horn 過去討厭語音備忘錄 — Apple 內建語音輸入讓他想摔手機。但語音對 LLM 輸入完全不同：轉錄不需要完美，因為 Claude Code 理解上下文。你可以含糊、斷句、重新開始。語音終於能用了，因為接收端夠聰明，能補上空缺。
"Voice finally works because the listener is smart enough to fill in the gaps."
他推薦兩個工具：Monologue（@usemonologue，也是 Every Inc 產品）將語音直接輸入到聚焦的應用程式中；WhisperFlow 也是不錯的選擇。他甚至買了鵝頸麥克風放在辦公室。
"Monologue pipes speech into whatever app is focused. You talk, it types into Claude Code."
Van Horn 表示他正在自動駕駛的 Tesla 中、送小孩上學途中口述這篇文章。語音不只是程式開發的輸入方式，而是整個內容創作流程的核心。
"I'm dictating this right now from Full Self-Driving in my Tesla, dropping off my kids."

💡 比喻說明：LLM 容錯的語音

傳統語音轉文字像是跟一個嚴格的抄寫員說話 — 每個字都要清楚，否則出錯。LLM 驅動的語音輸入像是跟一個聰明的助理說話 — 他們理解你要表達什麼，即使你說得含糊。

這也是 context engineering 的體現：LLM 利用周圍的程式碼上下文來修正語音轉錄的錯誤。

Part III: 4-6 路並行工作 — 上下文切換的極致

Van Horn 的日常工作方式：4 到 6 個 Ghostty 視窗，每個執行獨立的 Claude Code session。一個在寫 plan，一個在根據另一份 plan 執行 build，一個在跑 /last30days 研究，一個在修他測試時發現的 bug。
"Four to six Ghostty windows, each running a separate Claude Code session. One is writing a plan. One is building from a different plan."
當 /ce:plan 在第一個視窗啟動研究 Agent 時，他切到第二個視窗執行已完成的 plan。同時第三個視窗貼上新的 bug。等他循環回第一個視窗時，plan 已經完成等待在 Zed 中。
"While /ce:plan spins up research agents in one window, I switch to another window and /ce:work a plan that's already written."
這也是為什麼bypass permissions 是不可妥協的。如果每個 session 每個動作都要問「允許嗎？」，你無法在視窗間切換。它們都需要自主運行。檢查、反應、繼續。Git 隨時可以回滾。
"If every session asks 'Allow?' on every action, you can't context-switch. They all need to run autonomously."
代價：他的 MacBook 大約一小時就沒電。六個 Claude session 並行執行。他剛訂了新的 MacBook Pro。
"This is also why my MacBook dies in about an hour."
⚠️ 重點提示：並行工作的前提

多路並行需要三個前提才能有效運作：

1. bypass permissions — 每個 session 必須能自主執行，否則人工瓶頸抵消並行效益

2. 完成音效 — 需要知道哪個 session 結束了（下一段設定）

3. Git 安全網 — 任何錯誤都可以 git revert，這是 bypass permissions 的安全保障

💡 CDMO 類比：製程並行

類似 CDMO 廠區內同時進行多條產線的並行製造：一條線在做配液，另一條在充填，第三條在凍乾。每條線有獨立的 Batch Record（= plan.md），操作員在產線間巡迴監控。

差異：CDMO 並行有嚴格的交叉汙染防護和 SOP；Van Horn 的並行防護是 Git。在 GMP 環境中，bypass permissions 模式絕對不適用。

Part IV: 三項關鍵設定 — 解放 Claude Code 的枷鎖

Claude Code 預設模式對每次編輯、每個指令都要求權限確認。Van Horn 認為需要三項設定改變才能真正發揮效率。
"Claude Code's default mode asks permission for every edit, every command. You need three config changes."
💻 設定 1：Dangerously Skip Permissions

在 ~/.claude/settings.json 中加入：

{
  "permissions": {
    "allow": [
      "WebSearch", "WebFetch", "Bash", "Read", "Write",
      "Edit", "Glob", "Grep", "Task", "TodoWrite"
    ],
    "deny": [],
    "defaultMode": "bypassPermissions"
  },
  "skipDangerousModePermissionPrompt": true
}

skipDangerousModePermissionPrompt: true 是關鍵。沒有它，Claude 每次新 session 都會要求確認。也可以用 Shift+Tab 臨時切換。

💻 設定 2：完成時播放音效

在同一個 settings.json 中加入 hooks：

{
  "hooks": {
    "Stop": [{
      "hooks": [{
        "type": "command",
        "command": "afplay /System/Library/Sounds/Blow.aiff"
      }]
    }]
  }
}

用途：離開螢幕，聽到音效再回來。多路並行時，用來判斷哪個 session 完成了。

💻 設定 3：Zed Autosave（500ms）

在 Zed 設定（Cmd+,）中：

{
  "autosave": {
    "after_delay": {
      "milliseconds": 500
    }
  }
}

效果：Zed 每 500 毫秒自動儲存。Claude Code 監視檔案系統。雙方即時同步 — 像 Google Docs 協作，但另一個協作者是 AI。Ghostty 放一半螢幕，Zed 放另一半，看著同一個檔案。

⚠️ 安全性警告

「Dangerously skip permissions」如其名 — 有危險性。啟用後 Claude Code 可以不經確認執行任何 Bash 指令、讀寫任何檔案。Van Horn 依賴 Git 作為安全網，但對於以下情境需要特別謹慎：

- 含有敏感資料的目錄（API keys、.env 檔案）
- 系統層級指令（rm -rf、chmod 等）
- 多人共用的開發環境
- 任何 GMP/法規相關的工作環境 — 絕對不要使用此設定

Part V: 研究先行 — /last30days 的威力

在執行 /ce:plan 之前，Van Horn 通常先跑 /last30days 做研究。例如他在考慮 Vercel 的 agent-browser 和 Playwright 之間的選擇時，跑了一次研究指令，幾分鐘內取得：78 篇 Reddit 討論、76 則 X 貼文、22 個 YouTube 影片、15 篇 HN 文章。
"I ran /last30days Vercel agent browser vs Playwright. In a few minutes: 78 Reddit threads, 76 X posts, 22 YouTube videos, 15 HN stories."
研究結果顯示 agent-browser 使用的 context tokens 少了 82-93%，而 Playwright 光是工具定義就吃掉 13,700 tokens。他把整個研究結果輸入 /ce:plan，產出的計畫植根於社群當下的真實認知，而非六個月前的訓練資料。
"The plan came out grounded in what the community actually knows right now, not six-month-old training data."
/last30days 是開源專案（GitHub 4.5K stars），能同步搜尋 Reddit、X、YouTube、TikTok、Instagram、HN、Polymarket 及一般網頁。Van Horn 將此用於一切決策：選函式庫前、開發功能前、寫文章前。
"/last30days is open source (4.5K stars). It searches Reddit, X, YouTube, TikTok, Instagram, HN, Polymarket, and the web in parallel."
📚 核心概念：Research-Plan-Build 三段式

Van Horn 的完整循環：Research → Plan → Build。

/last30days 解決的是 LLM 訓練資料時效性問題 — 將「社群即時認知」注入 plan 的上下文中。這本質上是一種手動 RAG（Retrieval Augmented Generation）：先檢索最新資訊，再餵入 LLM 做規劃。

🛠️ 實務應用：搜尋來源涵蓋

同步搜尋的平台：Reddit、X (Twitter)、YouTube、TikTok、Instagram、Hacker News、Polymarket、一般網頁搜尋

適用場景：技術選型、函式庫評估、趨勢分析、競品研究、寫作前的素材收集

GitHub：github.com/mvanhorn/last30days-skill

Part VI: 會議轉 Plan — Granola + Claude Code

Van Horn 與一位候選人共進午餐，討論了一個尚未開發的新產品構想。一個半小時的對話，夾雜著壽司、餐廳、小孩話題。他用 Granola（會議記錄工具）錄音。午餐後，把完整逐字稿貼進 Claude Code：/ce:plan turn this into a product proposal。
"I pasted the full transcript - ninety minutes mixed with tangents about sushi - into Claude Code: /ce:plan turn this into a product proposal."
關鍵在於：Claude Code 已經知道公司產品程式碼在 GitHub 的位置，也能存取他的公司策略資料夾 — 每一份先前寫過的策略 plan.md。所以處理逐字稿時，它不只是從午餐對話中提取點子，而是同時交叉比對實際程式碼庫和每個過去的策略決策。
"It was cross-referencing against our actual codebase and every strategic decision we've made before."
結果：一次就產出了完整的產品提案 — 目標、使用者故事、技術方法、里程碑。自動忽略了關於餐廳的部分。當晚寄給候選人。那位候選人現在全職加入團隊。Granola 現在支援 MCP，可以在 Claude Code 中直接使用。
"One-shotted an incredible proposal. Goals, user stories, technical approach, milestones. Ignored the parts about restaurants."
💡 比喻說明：累積上下文的複利效應

Van Horn 的策略資料夾就像一個不斷增值的「決策記憶庫」。每一份新的 plan.md 都建立在所有過去 plan 的基礎上。這是知識複利：第 10 份策略文件的品質，遠高於第 1 份，因為 AI 有更多歷史脈絡可以參考。

這個概念連結到你的 context engineering 認知：context window 的價值不在大小，而在累積的相關資訊密度。

🛠️ 實務應用：會議工具整合

Granola — 會議錄音 + 自動逐字稿，現支援 MCP 直接整合 Claude Code

工作流：開會（Granola 錄音）→ 逐字稿自動進入 Claude Code → /ce:plan 產出結構化提案

關鍵成功因素：Claude Code 必須能存取公司程式碼 + 策略文件庫，才能交叉比對產出高品質提案

Part VII: 超越程式碼 — 策略文件、遠端存取、Token 管理

Van Horn 撰寫公司策略文件時，Claude Code 和 Markdown 檔案並排開啟。用 Monologue 口述：「給我三個市場策略方案，列出每個的優缺點。」三個選項立刻出現在 Zed 中。「第二個最接近但第一個的用詞更好，合併它們。」即時更新。策略文件、產品規格、競爭分析、這篇文章本身 — 全用相同的工作流。
"Strategy docs, product specs, competitive analysis, this article. Same workflow. Talk, plan, iterate."
Mac Mini 遠端工作站：透過 Telegram 整合，從手機傳訊息給 Mac Mini 上的 Claude Code。吃飯時想到一個 bug？在 Telegram 輸入 /ce:plan fix the timeout issue，回到電腦時 plan 已經等在 Zed 中。搭飛機時用 tmux 連線到 Mac Mini — WiFi 中斷 20 分鐘也不影響，session 在遠端持續運行。
"tmux into your Mac Mini first and the session runs on that machine. Your laptop is just a window."
Token 用量管理：4-6 路並行的 Opus session 會快速消耗 $200/月的 Claude Max 額度。解法：同時訂閱 $200/月的 Codex plan，用 Compound Engineering 的 /ce:work --codex 將執行層工作委託給 Codex credits。Claude 負責規劃，Codex 負責大量實作。兩個方案互補。
"Claude for planning, Codex for heavy implementation."

⚠️ 重點提示：tmux 飛行模式

問題：Claude Code 在飛機 WiFi 下體驗差 — 連線中斷時 session 直接死亡，甚至不會通知你。

解法：先 tmux 連到遠端 Mac Mini，所有 Claude Code session 在那台機器上運行。你的筆電只是顯示窗口。WiFi 中斷 → 重連 → session 完整保存。

這也適用於任何不穩定的網路環境。

Part VIII: 實戰案例 — 迪士尼世界旅遊規劃

為了展示完整工作流用於非程式碼場景，Van Horn 分享了一個迪士尼樂園旅遊規劃的實例。在足球場邊看小孩比賽時，用筆電演示了完整流程：
Step 1：/last30days Disney World — 兩分鐘內取得 66 篇 Reddit 討論（11,804 upvotes）、34 則 X 貼文、8 個 YouTube 影片。價格衝擊是主要討論主題，一篇 $8,500 旅遊報告收到 183 則留言。
Step 2-3：口述了完整需求（含錯字 — Claude Code 不在意）：四個園區、指定遊樂設施、Genie Plus 策略、預算考量。Claude 研究 Agent 啟動，交叉比對 /last30days 資料，產出結構化 plan.md：園區順序、精確的 Lightning Lane 預訂策略、三個鬧鐘提醒。
Step 4-6：在 Zed 中檢視 plan，請 Claude 為另一位家長也做一份 305 行的三日行程。接著直接部署到 Vercel 網站。最後把 .md 檔丟進 Telegram，讓 OpenClaw 設定日曆事件 + cron 備份提醒。語音到研究到計畫到網站到自動提醒，全部在足球場邊完成。
"Voice to research to plan to website to automated reminders. At a soccer field."
📚 核心概念：工作流的通用性

迪士尼案例的重要性不在旅遊規劃本身，而在於證明同一套 Research-Plan-Build 工作流適用於任何領域。Van Horn 用完全相同的方法論處理：

- 程式碼開發
- 策略文件撰寫
- 開源專案貢獻
- 文章寫作
- 旅遊規劃

框架的通用性是其最大價值。

🛠️ 實務應用：端到端自動化

迪士尼案例展示了完整的工具鏈串接：

/last30days → 即時資訊搜集
/ce:plan → 結構化計畫
Zed → 檢視與微調
Vercel → 部署為網頁
Telegram + OpenClaw → 自動提醒設定

每一步的輸出直接成為下一步的輸入，零手動轉換。

Key Takeaways

Plan-First 是系統，不是習慣
plan.md 不只是「先想清楚」的好建議 — 它是跨 session 的狀態保存機制、AI Agent 的任務規格書、以及 Git 回滾時的參考基準。70 份 plan 檔案 + 263 commits 在 30 天內，量化了這套方法的產出規模。
語音 + LLM 容錯 = 新的輸入典範
語音輸入之所以能用，是因為 LLM 的上下文理解能力彌補了語音轉錄的錯誤。這不是語音辨識技術的進步，而是「接收端智慧」的質變。Van Horn 的整篇文章、策略文件、甚至程式碼都是口述完成的。
並行工作需要系統級設定
4-6 路並行不是打開更多終端機就能實現的。它需要 bypass permissions（自主執行）、音效通知（狀態感知）、autosave（即時同步）三者配合。缺少任何一項，並行效率就會崩潰。
工作流框架的通用性
Research → Plan → Build 循環適用於程式碼、策略文件、開源貢獻、文章寫作、旅遊規劃。框架的價值在於可遷移性 — 一旦建立，每個新領域的適應成本趨近於零。

Practice Questions

Q1 概念理解 Van Horn 說 plan.md 是「能存活於一切中斷的檢查點」。這個設計解決了 AI coding 工作流中的什麼根本問題？

LLM 的 context window 有限且不持久 — 每次新 session 都從零開始。plan.md 將規劃結果「外部化」為檔案，脫離了 context window 的限制。這解決了三個問題：(1) context 長度不夠時可以分段執行，(2) session 中斷時不需重做分析，(3) 多個 Agent/session 可以共享同一份任務定義。本質上，plan.md 是人類與 AI 之間的「共享記憶體」，類似分散式系統中的持久化狀態（persistent state）。

Q2 產業應用 如果你要將 Van Horn 的 Research-Plan-Build 循環應用到 CDMO 的偏差調查（Deviation Investigation）流程中，每個階段對應什麼？有什麼需要調適的？

Research（/last30days 對應）：搜集相關偏差歷史紀錄、CAPA 資料庫、SOP 版本變更記錄、類似產品的 FDA Warning Letter 案例。

Plan（/ce:plan 對應）：產出結構化的偏差調查計畫 — 根本原因假設、需要檢視的文件清單、訪談對象、時間線、驗收標準。

Build（/ce:work 對應）：執行調查、撰寫報告、提出 CAPA 建議。

關鍵調適：GMP 環境中 AI 必須是 Copilot Mode — 每一步都需要 SME（Subject Matter Expert）審核簽名。bypass permissions 完全不適用。此外，偏差調查的 plan.md 等同於受控文件，需要版本管理和審計追蹤（Audit Trail），不能只靠 Git。

Q3 批判思考 Van Horn 報告了 70 份 plan + 263 commits 在 30 天內的產出。這些數字能證明 Plan-First 方法論的有效性嗎？缺少什麼資訊？

不能完全證明，因為缺少對照：

(1) 基線比較：他在採用 Plan-First 之前的 30 天產出是多少？沒有 before/after 比較，就無法歸因於方法論本身。

(2) 品質指標：commits 數量不等於程式碼品質。需要知道 bug rate、code review 通過率、revert 比例。

(3) 單人樣本：這是 N=1 的個案研究，Van Horn 是高度熟練的 power user，結果可能無法推廣到一般開發者。

(4) 利益關係：他是 Compound Engineering Plugin 的核心貢獻者，推廣此工作流直接受益。

儘管如此，plan.md 作為「持久化上下文」的設計理念仍然有獨立的工程價值，值得在受控條件下測試。

🎯 我的行動結論
✔ 核心洞察
plan.md 作為「跨 session 持久化上下文」的概念，與我正在建構的 Claude Code 自動化 pipeline 高度相關。我的 successor prompt / handoff 文件本質上就是一種 plan.md — 差異在於 Van Horn 的 plan.md 是每個任務一份，而我的 handoff 是整個 pipeline 一份。可以考慮將 pipeline 分解為多個獨立的 plan.md，每份對應一個可獨立執行的步驟，讓 Claude Code headless mode 能更靈活地並行處理。
