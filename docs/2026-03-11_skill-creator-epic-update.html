Part I: Skill-creator 是什麼？為何重要？

Skill-creator 是 Anthropic 官方推出的 Skills 生成器，堪稱整個 Skills 生態的基石。你可以用自然語言描述需求，它就能幫你生成一個完整的 Skill。
Skill-creator is Anthropic's official skill generator - the foundation of the entire Skills ecosystem. You describe your needs in natural language, and it creates a complete Skill for you.
作者指出，Claude 之所以能力這麼強，有一半的原因要歸功於 Skills。每個 Skill 就像一個技能包，掛載越多，Agent 能做的事情就越多。
The author argues that half of Claude's capability comes from Skills. Each Skill is like a skill pack - the more you attach, the more the Agent can do.
Skills 是整個 Agent 未來大繁榮生態的基石。而 Skill-creator 作為生成 Skills 的「母 Skill」，它的更新等於整個生態的基礎設施升級。
Skills are the cornerstone of the Agent ecosystem's future prosperity. As the "mother Skill" that generates all others, Skill-creator updates represent infrastructure-level upgrades.

Part II: 四大新能力總覽

這次更新一口氣加入四個全新能力：
This update introduces four brand-new capabilities at once:
1. 評估系統（Evaluation System） - 跑完直接告訴你這個 Skill 到底行不行。解決了過去 Skill 建完就是黑盒的問題。
Evaluation System - run it and immediately know if the Skill works. Solves the previous black-box problem.
2. 基準測試（Benchmarking） - 將通過率、耗時、Token 用量全都量化。有 Skill vs 無 Skill 的對比數據一目了然。
Benchmarking - quantifies pass rate, execution time, and token consumption. Clear comparison between with-Skill vs without-Skill.
3. 多代理並行測試（Multi-agent Parallel Testing） - 每個測試在乾淨環境獨立運行，支持 A/B 盲評，結果不互相污染。
Multi-agent Parallel Testing - each test runs independently in a clean environment, supports A/B blind evaluation, results don't cross-contaminate.
4. 描述調優（Description Optimization） - 自動幫你修改 Skill 描述，讓該觸發的觸發，不該觸發的別亂觸發。
Description Optimization - automatically refines Skill descriptions so the right triggers fire and wrong ones don't.
核心問題：之前缺少評估機制。用工業化體系的話來說，沒有品質檢測就沒有品質保證。這次更新把整個評估體系全都補上了。
The core problem: the lack of an evaluation mechanism. In industrial terms, without quality testing there's no quality assurance. This update fills the entire evaluation gap.
💡 比喻說明
想像你開了一家工廠（Agent），雇了一堆員工（Skills）。以前的問題是：員工入職後沒有績效考核，你不知道誰表現好、誰在摸魚。Skill-creator 的更新就像裝上了完整的 KPI 體系 - 觸發率就是出勤率，基準測試就是工作品質評估，多代理並行就是避免同事互相抄答案。
⚠️ 重點提示
更新方式極簡：直接對你的 Agent 說：「https://github.com/anthropics/skills/tree/main/skills/skill-creator 這個 Skills 更新了，幫我更新到最新版」。Claude Code、OpenClaw、OpenCode 等都適用。
📚 核心概念
Skills 的本質定位：Skills 是 Agent 生態的基石。Agent 本身不一定多厲害，真正的差異化來自於它掛載了多少高品質的 Skills。這個觀點與 Anthropic 官方「Building Effective Agents」文中的 tool-augmented approach 一致。
🛠️ 實務應用
案例背景：作者先前用 GitHub 上的 yt-dlp 做了一個影片下載 Skill，後來想擴展功能 - 不只下載，還要自動生成中英雙語文字講稿。於是用 Skill-creator 新建了一個講稿生成 Skill，整個過程約 3-5 分鐘。

Part III: 描述調優完整流程

當你有多個相似 Skill 時，最怕的就是觸發打架 - 該觸發的不觸發，不該觸發的亂觸發。作者的案例：yt-dlp（下載影片）和講稿生成（轉文字）兩個 Skill 的觸發條件都是「給一個影片連結」，極易衝突。
When you have multiple similar Skills, the biggest fear is trigger conflicts - the right one doesn't fire while the wrong one does. The author's case: both yt-dlp (download video) and transcript generation (convert to text) trigger on "video link input."
Step 1：自動生成測試查詢 - Skill-creator 會讀取你的 Skill 描述，然後自動生成兩組查詢：應觸發的 10 條 + 不應觸發的 10 條。設計巧妙之處在於故意把邊界情況（edge cases）都擺進去，逼模型在模糊地帶做判斷。
Step 1: Auto-generate test queries - Skill-creator reads your Skill description and generates two sets: 10 should-trigger queries + 10 should-not-trigger queries. Cleverly includes edge cases to force the model to make decisions in gray areas.
Step 2：互動式確認介面 - 生成一個網頁，所有查詢排列整齊，每條右邊有開關，標著是否應該觸發。你可以逐條審閱，覺得哪條判斷不對，直接關掉。
Step 2: Interactive confirmation UI - generates a webpage with all queries neatly listed, each with a toggle switch for trigger/no-trigger. Review one by one and flip any you disagree with.
Step 3：導出評估集 + 自動迭代 - 確認完畢後點擊「導出評估集」，優化循環在後台啟動，最多跑 5 輪迭代。
Step 3: Export evaluation set + auto-iteration - after confirmation, click "export evaluation set" and the optimization loop runs in the background, up to 5 iterations.
Step 4：結果視覺化 - 跑完後會看到一個巨型表格：每一列是查詢樣本，每一行是迭代版本。綠色勾表示觸發成功，紅色叉表示未觸發。藍色列是測試集，其餘是訓練集。
Step 4: Results visualization - a large table shows queries as columns, iteration versions as rows. Green checks = trigger success, red X = no trigger. Blue columns = test set, rest = training set.
防過擬合設計：樣本分成 60% 訓練集和 40% 測試集，在訓練集上迭代優化，最終用測試集表現來選最佳版本。跑完後，最優描述自動寫回 SKILL.md，全程不用手動操作。
Anti-overfitting design: samples split 60% training / 40% test. Iterates on training set, selects best version based on test set performance. The optimal description is automatically written back to SKILL.md.
💡 比喻說明
描述調優就像訓練機場安檢系統。你給它一堆「應該攔下」和「應該放行」的樣本，讓它學會區分。60/40 訓練測試分割就是留一部分「新行李」做最終驗收，確保系統不是靠死記硬背，而是真的學會了判斷規則。
⚠️ 重點提示
Anthropic 官方實測數據：在 6 個文檔類 Skill 上測試，5 個的觸發率都有提升。僅僅是用新版 Skill-creator 優化描述，不改任何 Skill 邏輯本身。
📚 核心概念
訓練集/測試集分割（Train/Test Split）：機器學習的基本方法論。避免模型只是「背答案」（過擬合），而是要在從未見過的資料上也能表現良好。Skill-creator 用 60/40 的比例，在 Skill 描述優化場景下是合理的選擇。
🛠️ 實務應用
描述調優耗時：整個過程（5 輪迭代）大約需要 10-20 分鐘。期間 Skill-creator 會定期匯報進度。考慮到這是一次性投資，ROI 非常高 - 一次調優可以長期受益。

Part IV: 多代理並行評估

觸發對了不等於 OK - Skill 能穩定觸發之後，還要評估它在實際任務上的表現。作者用講稿生成 Skill 做了完整演示。
Correct triggering doesn't equal OK - after stable triggering, you still need to evaluate actual task performance. The author demonstrates with the transcript Skill.
Skill-creator 會先完整讀取你的 Skill 文件，搞清核心流程。然後問你想測哪個方面（可選全面評估）。它會根據 Skill 功能自動設計三類測試場景，同時設計量化驗收標準。
Skill-creator reads your Skill file completely, understands the core flow, then asks what aspect to test (full evaluation available). It auto-designs three test scenario categories with quantitative acceptance criteria.
核心突破：4 個獨立子代理同時並行運行。以前的評估是按順序跑，前一個任務的上下文會污染下一個的結果。你以為是 Skill 的功勞，其實是對話歷史幫了忙。現在每個代理都在完全乾淨的環境裡獨立運行，有自己的 Token 計數和時間指標。
Core breakthrough: 4 independent sub-agents run in parallel. Previous evaluations ran sequentially, where prior context contaminated results. Now each agent runs independently in a clean environment with its own token count and time metrics.

Part V: 基準測試量化數據

測試完成後，瀏覽器會彈出評估查看頁面，有兩個標籤頁：
After testing, the browser shows an evaluation page with two tabs:
輸出標籤頁：直接查看每個測試用例的輸出。下方有回饋框，可以標注問題點。這些回饋會被存起來，下次改進 Skill 時直接使用。
Output tab: view each test case output directly. Feedback box below for annotating issues. Feedback is saved for use in the next Skill improvement cycle.
基準測試標籤頁：有 Skill vs 無 Skill 的量化對比。作者案例的數據 - 有 Skill 通過率 100%，無 Skill 基線 9%，差值 91.5%。費用上，有 Skill 每次約 4000 Token，無 Skill 約 1750 Token，差距 2250 Token。
Benchmark tab: quantitative comparison with vs without Skill. Author's data - with Skill pass rate 100%, without Skill baseline 9%, delta 91.5%. Cost: ~4000 tokens with Skill vs ~1750 without, 2250 token difference.
閉環設計：Anthropic 舉了 PDF Skill 的例子 - 在處理表格時，Claude 需要把文字放在特定座標上，但因為缺乏明確引導經常放歪。這個問題在評估過程中被發現，修復定位邏輯後問題解決。評估結果存在本地，下次改進時自動帶入，形成「測試 → 發現 → 修復 → 再測試」的完整循環。
Closed-loop design: Anthropic's PDF Skill example - when handling tables, Claude needed to place text at specific coordinates but frequently misaligned. The issue was caught during evaluation, fixed in positioning logic. Results stored locally, automatically loaded for next improvement cycle.
📚 核心概念
上下文污染（Context Contamination）：當測試按順序在同一對話中執行時，前面任務的對話歷史會影響後面任務的表現。模型可能「記住」了前面的答案模式，讓後面的結果看起來更好，但這不是 Skill 本身的功勞。獨立環境並行測試是解決此問題的標準做法。
⚖️ 工具比較
有 Skill vs 無 Skill（講稿生成案例）：
通過率：100% vs 9%（差值 91.5%）
Token 消耗：~4000 vs ~1750（多 2250）
結論：Token 成本增加約 2.3 倍，但通過率從接近零提升到 100%。投入產出比極高。
⚠️ 重點提示
回饋的累積效應：每次評估時標注的問題點會被持久化存儲。下次用 Skill-creator 改進時，這些回饋會自動帶入上下文。這意味著 Skill 的品質是累積進步的，不是每次從零開始。
🛠️ 實務應用
PDF Skill 案例啟示：有些 Bug 在日常使用中不容易發現（例如座標偏移），但在系統性評估中會被暴露。這跟製造業的 SPC（統計過程控制）原理相同 - 個案不一定看得出問題，但統計數據會說話。

Part VI: 兩種 Skill 類型的不同評估策略

在評估之前，你得先搞清楚你的 Skill 屬於哪一種。本質上，Skills 分為兩種類型：
Before evaluating, you need to know which type your Skill is. Fundamentally, Skills fall into two categories:
第一種：能力提升型（Capability Enhancement） - 教 Claude 做它本來不擅長的事。例如官方的前端設計 Skill、文檔創建 Skill，裡面寫了大量技巧，是光靠 Prompt 拿不到的效果。大多數人自己做的 Skill 都屬於這類。
Type 1: Capability Enhancement - teaches Claude to do what it's not naturally good at. Examples: official frontend design Skill, document creation Skill, containing techniques that prompting alone can't achieve.
第二種：編碼偏好型（Coding Preference / Workflow） - 告訴 Claude 按你的規矩來。Claude 本身每一步都能做，但你的 Skill 把這些步驟按你的流程串起來。例如：會議紀要整理 Skill（按公司固定格式轉錄音為行動項文件）、週報生成 Skill（從各平台拉數據按指定格式排列）。
Type 2: Coding Preference/Workflow - tells Claude to follow your rules. Claude can do each step, but your Skill chains them in your specific process. Examples: meeting notes Skill (company format), weekly report Skill (pull data, format).

Part VII: 不同類型的評估重點

能力提升型的評估重點：測的是模型更新後這個 Skill 還有沒有存在的必要。用 A/B 測試對比，有 Skill 和沒 Skill 各跑一次。如果結果差不多，這個 Skill 就可以退休了。
Capability Enhancement evaluation focus: test whether the Skill is still necessary after model updates. A/B test with and without Skill. If results are similar, the Skill can be retired.
編碼偏好型的評估重點：測的是另一件事 - 有沒有老老實實按你的流程走？有沒有漏步驟？有沒有自作主張改了順序？有沒有忘了你特別說過的某個要求？
Coding Preference evaluation focus: did it faithfully follow your process? Any skipped steps? Any unauthorized reordering? Any forgotten specific requirements?
作者的最終建議：所有的 Skills 都值得重新優化和評估一遍。更新到最新版 Skill-creator 後，對你所有的 Skills 進行一次全面的觸發率優化和品質評估，是目前 ROI 最高的操作。
The author's final advice: all Skills are worth re-optimizing and re-evaluating. After updating to the latest Skill-creator, a comprehensive trigger optimization and quality evaluation of all your Skills is the highest ROI activity right now.

你的 Skills 分類清單：根據你目前的 Skills，快速判斷類型：
- artifact-skill (教育文件生成) → 編碼偏好型（按 Master Prompt 格式走）
- amaran-presentation → 編碼偏好型（品牌識別規範）

這個分類直接決定你的評估策略。

Key Takeaways

1. 評估機制是 Skills 生態的轉折點
過去 Skill 建完就是黑盒，現在有了完整的「建立 → 調優 → 測試 → 量化 → 迭代」閉環。這是從「手工作坊」到「工業化生產」的跨越。
2. 描述調優 ROI 最高
10-20 分鐘的一次性投入，就能提升觸發準確率。Anthropic 官方測試 6 個 Skill 中 5 個有提升，且完全自動化。
3. 上下文污染是隱形殺手
多代理並行測試解決了順序評估的根本缺陷 - 確保每個測試結果反映的是 Skill 本身的能力，而非對話歷史的加成。
4. 區分 Skill 類型決定評估方向
能力提升型測「是否還有必要」，編碼偏好型測「是否按流程走」。搞錯方向會浪費評估資源，得出錯誤結論。

Practice Questions

Q1 概念理解 為什麼「多代理並行測試」比「順序測試」更可靠？請用「上下文污染」的概念解釋。

順序測試在同一對話中依次執行多個測試用例，前面任務的對話歷史（包含正確答案的模式、格式範例等）會自動成為後續任務的上下文。這意味著後面的測試可能「借用」了前面的經驗，表現出比 Skill 實際能力更好的結果。

多代理並行測試讓每個測試在完全獨立的環境中運行，各自有獨立的 Token 計數和時間指標。測試結果只反映 Skill 本身的品質，而非累積的對話上下文。這就像藥物臨床試驗中的「雙盲設計」- 確保結果的純淨性。

Q2 產業應用 在你的 CDMO 環境中，artifact-skill（教育文件生成）屬於哪種類型？如果用新版 Skill-creator 評估它，應該關注什麼指標？

artifact-skill 主要是「編碼偏好型」- 它的核心價值不是教 Claude 新能力，而是確保 Claude 按照 Master Prompt v1.5.0 的格式規範走（雙語格式、Tabbed 結構、Rating Bar、Dashboard Meta Tags 等）。

評估時應關注的指標包括：(1) 格式合規性 - 是否完整包含所有必要區塊（meta tags、rating bar、bottom line 等）？(2) 雙語對齊準確性 - 中文翻譯是否對應正確的英文原文？(3) 佈局一致性 - 是否遵循 v1.4.3 的 1100px 置中佈局？(4) 分類標籤準確性 - doc-tags 是否符合 standard_categories？這些都是可量化的檢查點，非常適合自動化評估。

Q3 批判思考 文章聲稱「所有 Skills 都值得重新評估」。你認為這個建議有什麼盲點或需要補充的地方？

幾個需要打折扣的地方：(1) 時間成本 - 每個 Skill 的描述調優需要 10-20 分鐘，完整評估可能更久。如果你有 20+ 個 Skills，全部跑一遍需要數小時。應該用 Pareto 原則，先評估使用頻率最高的 Top 5。(2) Token 成本 - 文章沒提到評估過程本身的 Token 消耗。4 個並行代理 + 5 輪迭代的 Token 用量不會少，需要納入 ROI 計算。(3) 評估標準的主觀性 - 觸發率可以量化，但「輸出品質」的判斷仍然有主觀成分。自動化評估只能檢查格式合規，無法完全替代人工品質判斷。(4) 模型版本依賴 - 在某個 Claude 版本上優化的描述，在模型升級後可能需要重新調優。這個維護成本文章沒有提及。
