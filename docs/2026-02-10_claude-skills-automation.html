Part I
什麼是 Claude Skills？為什麼它不同？
一般的 Claude 能寫 email、摘要文件。但 Skills 教會 Claude 專門的工作流程（specialized workflows），當你提到相關需求時會自動觸發。
"Regular Claude writes emails and summarizes documents. Skills teach Claude specialized workflows that trigger automatically when you mention them or ask for what they do."
Skills 能建立的東西遠超文字回覆：Word 文件、PDF、PowerPoint 簡報、含有運作公式的 Excel 試算表、互動式儀表板、組織圖、流程圖。
"Word documents, PDFs, PowerPoint presentations, Excel spreadsheets with working formulas, interactive dashboards, org charts, workflow diagrams."
根據統計，約 27% 的 Claude 輔助工作是「原本不會做」的任務——儀表板和分析工具如果手動做根本不划算。重度使用者回報 50-100% 的生產力提升。
"Approximately 27% of Claude-assisted work consists of tasks that wouldn't have been done otherwise... Power users report 50-100% productivity gains."
💡 比喻說明
想像 Claude 是一位新進員工。沒有 Skill 時，他什麼都能做一點但不精通。Skill 就像是你給他的專業 SOP 手冊——一旦有了提案製作 SOP，他就能每次都產出格式一致、品質穩定的提案，不用你每次重新教。
💰 ROI 快速計算
前提：Claude Professional $17/月
假設：你的時間價值 $150/小時，Skills 每月省 10 小時
結果：$1,500 釋放產能 vs $17 月費 = 88x ROI

作者的實際案例：4 份提案 x 4.5 小時節省 = 18 小時/月
⚠️ 重點提示
需要 Claude Professional 方案（$17/月）才能使用 Skills 功能。這是進入門檻，但只要每月省下 2 小時就能回本。
Part II
Projects vs Skills：搞清楚差異
Projects 是上下文（context）——Claude 知道什麼。你在 Project 裡上傳文件、設定指令、建立適用於該工作區的知識。例如：一個「行銷」Project 裡放品牌指南和過去的活動文件。
"Claude Projects are workspaces where you upload documents for Claude to reference, add custom instructions... Example: A 'Marketing' Project with brand guidelines and past campaign docs."
Skills 是能力（capabilities）——Claude 能做什麼。Skills 教 Claude 如何建立特定產出，而且跨所有 Projects 和對話都能運作。
"Claude Skills are specialized workflows that teach Claude how to create specific outputs... Work across ALL your Projects and conversations."
你可以在 Projects 裡面使用 Skills。當作者在「客戶工作」Project 裡使用提案產生器 Skill 時，它結合了 Project 的上下文（客戶歷史、品牌指南）和 Skill 的工作流程（如何格式化提案）。
"You can use Skills inside Projects. When I'm in my 'Client Work' Project and use my proposal-generator skill, it combines the context from my Project with the skill's workflow."

💡 比喻說明
把 Projects 想成員工的辦公桌（上面放著這個專案需要的所有資料），Skills 想成員工的專業證照（不管在哪個部門都能用的技能）。一個有 GMP 證照的員工，不管調到 QA 部門還是生產線，都能發揮那項能力。
Part III
如何建立 Skill（完整步驟）
作者強調要嚴格按照步驟操作。雖然 Claude 可以嘗試幫你建立 Skill 的可下載檔案，但「大約 50% 的時候會失敗」。以下是手動建立的可靠方法：
"I'm going to walk you through this step by step. Don't skip ahead... I know Claude can attempt to create them for you in a downloadable file, but it fails to do it right around 50% of the time."
🛠️ Step 1：建立資料夾
在電腦的檔案瀏覽器中建立新資料夾。命名規則：小寫 + 短橫線。
正確：proposal-generator
錯誤：Proposal Generator 或 proposal_generator
🛠️ Step 2：建立 SKILL.md 檔案
用任何文字編輯器（Windows 的記事本、Mac 的 TextEdit、VS Code）建立空白檔案。檔名必須是 SKILL.md（全大寫）。將工作流程指令貼入後存檔到資料夾內。
🛠️ Step 3：壓縮成 ZIP
Mac：右鍵資料夾 → 壓縮。Windows：右鍵 → 傳送到 → 壓縮的資料夾。
🛠️ Step 4：上傳至 Claude
Claude.ai → 左下角個人頭像 → Settings → Capabilities → Skills → "+ Upload skill" → 選擇 ZIP 檔 → 等待成功訊息。之後在任何對話中直接使用即可。
⚠️ 進階技巧：assets 資料夾
在 Skill 資料夾內建立 assets 子資料夾，放入 Logo、PowerPoint 模板、品牌指南等。Skill 會自動偵測並套用這些資源，讓產出自動符合你的品牌風格。
Part IV
5 個改變工作效率的 Skill 實戰案例
作者在經營 jsalinas.org 的教練與 AI 建構實務中測試了數十個 Skills，以下 5 個每週固定節省最多時間：
"I've tested dozens of Skills running jsalinas.org. These five consistently save me the most time every single week."
Skill #1：語音筆記 → 客戶提案
作者用 WisprFlow 在晨間散步時錄製專案細節，貼到 Claude 配合提案 Skill，5 分鐘內產出格式化提案。上月成果：4 份提案，省下 18 小時。
"I record project details during my morning walk with WisprFlow, paste those notes into Claude with my proposal skill, and review a formatted proposal 5 minutes later. Last month: 4 proposals. Time saved: 18 hours."
Skill #2：研究資料 → 董事會級簡報
22 分鐘內建出 12 頁非營利組織董事會簡報。之前至少要兩天，還需要額外排版修改。
"Built a 12-slide nonprofit board deck in 22 minutes last Tuesday. Before Skills? Two days minimum."
Skill #3：CSV → 互動式儀表板
每月匯出客戶資料、上傳 CSV、觸發儀表板 Skill，就能得到含篩選和圖表的互動式 HTML 儀表板。不需要 Power BI 或 Tableau。
"I export client data monthly, upload the CSV, mention my dashboard skill, and get an interactive view... no Power BI or Tableau needed."
Skill #4：自動更新的財務模型
47 個項目、6 個分頁。更新一個假設（如教練費率），283 個儲存格自動重新計算。以前花 3 小時建的模型第一次編輯就壞，現在 20 分鐘搞定且永不出錯。
"Forty-seven line items across six tabs. When I update one assumption, like my coaching rate, 283 cells recalculate automatically... Now takes 20 minutes and never breaks."
Skill #5：流程視覺化
每週使用。客戶問如何驗證 AI 輸出？描述 NotebookLM 工作流程給 Skill，2 分鐘後產出視覺化圖表，直接貼到 email 裡。
"Instead of three paragraphs, I described my NotebookLM workflow to this skill. Two minutes later, had a visual diagram. I pasted that diagram into an email, done!"

📚 核心概念：Skill 的運作邏輯
每個 Skill 的 SKILL.md 檔案本質上就是一個結構化的 System Prompt，定義了：(1) 要問使用者哪些問題、(2) 資料如何處理、(3) 輸出格式和品質標準。它也會檢查 Claude 的 Memory 和 assets 資料夾來個人化產出。
Part V
該先建哪個 Skill？+ 常見問題
作者的建議很簡單：問自己「每週哪份文件最耗時間？」然後從那個開始。只挑一個，不要一次做五個。
"Ask yourself: 'What document drains the most time from my week?' Pick one. Just one. Not all five this week."
SKILL.md 檔案可以編輯修改，配合你的具體需求客製化。Skills 在 Projects 內也能運作，且可以與團隊共享（傳 ZIP 或 .md 檔即可）。
"Download them, customize the instructions for your specific needs, then upload... Send them the .zip file or the SKILL.md file. They follow the same upload steps."

Key Takeaways
Skills ≠ Projects
Projects 是知識上下文（Claude 知道什麼），Skills 是工作流程能力（Claude 能做什麼）。兩者可以結合使用，在 Project 內觸發 Skill 會同時套用雙方優勢。
4 步建立，零程式碼
資料夾（小寫短橫線命名）→ SKILL.md 檔案 → ZIP 壓縮 → 上傳至 Claude Settings。整個過程 3 分鐘，之後跨所有對話可用。
ROI 極高的投資

Practice Questions
Q1 概念理解 如果一位同事說「我已經在 Project 裡設定了所有指令，為什麼還需要 Skill？」，你會如何解釋兩者的關鍵差異？

Project 的指令只在該 Project 內有效，而且主要定義的是「Claude 知道什麼」（上傳的文件、背景知識、行為準則）。Skill 定義的是「Claude 能做什麼」——一套可重複使用的專業工作流程，而且跨所有 Projects 和對話都能運作。

打個比方：Project 像是你桌上的參考資料，Skill 像是你的專業證照。你在 QA 部門的桌上放不同的文件，但你的 GMP 稽核能力不管坐在哪張桌子都能用。最強的組合是在 Project 內使用 Skill——同時有上下文和能力。

Q2 產業應用 在 CDMO 環境中，你會如何應用 Claude Skills 的概念？考慮 GMP 合規和文件管控的限制，哪些 Skill 類型最適合？哪些需要額外的人工審查機制？

在 CDMO 環境中，最適合的 Skill 類型依風險等級排列：

低風險 / 可直接採用：workflow-visualizer（視覺化 SOP 流程、製程轉移步驟）、dashboard-builder（產能利用率、批次狀態追蹤——但資料需去識別化後再上傳）。

中風險 / 需人工覆核：presentation-builder（管理層報告、客戶可行性評估簡報）、financial-model-builder（產能規劃、成本估算模型）。這些的「copilot mode」審查是必要的。

高風險 / 僅供草稿參考：proposal-generator 用於 tech transfer 提案或客戶回覆時，產出必須經過 QA 和法規團隊審查，且不能將 GMP 敏感資料直接上傳到 Claude。21 CFR Part 11 的電子記錄要求意味著 Claude 的產出不能直接成為正式 GMP 文件，但可以大幅加速草稿階段。

Q3 批判思考 作者聲稱「50-100% 生產力提升」和「40+ 小時/月節省」。這些數字可信嗎？有哪些隱藏成本或條件文章沒提到？

這些數字可能對作者的特定情境成立，但需要加幾個 asterisk：

隱藏成本：(1) 建立和迭代 SKILL.md 的時間——作者說 3 分鐘/個是上傳時間，但撰寫一個好的工作流程指令可能需要數小時的試錯；(2) 品質審查時間——AI 產出仍需人工檢查，特別是面對客戶的提案和財務模型；(3) 學習曲線——從「能用」到「用得好」之間有一段調校期。

條件限制：(1) 作者的工作高度文件導向（教練業務），如果你的工作以實體操作或即時決策為主，節省幅度會小得多；(2) 27% 的「原本不會做的任務」可能代表 Parkinson's Law——有了工具就做更多不一定需要的分析；(3) 50-100% 的生產力提升引用的是整體 Claude 使用數據，不是只有 Skills。

結論：數字大方向可信，但實際效益因工作性質而異，且需要前期投資來建立和調校 Skills。

🎯 我的行動結論
✔ 核心洞察
Claude Skills 本質上就是「可重複使用的結構化 Prompt + 輸出規範」，這和我在 Master Prompt 系統中做的事情是同一個邏輯。差別在於 Skills 被整合進 Claude 的產品層，有更好的檔案輸出能力（.docx, .pptx, .xlsx）。對我來說，最大的啟發不是文中的 5 個 Skill，而是「Skill = 把我已經在做的 Prompt Engineering 標準化成可分享的資產」這個概念。
📈 立即應用
這週可以做的事：(1) 把我目前的 CDMO Feasibility Assessment 流程試著包裝成一個 Skill，看看能否讓初步評估的草稿階段從 2 小時縮短到 30 分鐘；(2) 測試 workflow-visualizer 的概念，用來快速產出 tech transfer 流程圖給客戶展示。這兩個都屬於「copilot mode」使用，人工審查仍是最後把關。
⚠ 需要補充
文章完全沒有討論：(1) SKILL.md 的撰寫最佳實踐——「如何寫出好的 Skill」才是真正的核心能力，但文章只提供了下載連結沒有解釋設計邏輯；(2) 版本管理——Skill 更新後如何迭代？(3) 安全性考量——在受管制產業中上傳資料到 Claude 的資料治理問題。需要另外研究 Anthropic 的 Skills 技術文件和資料處理政策。
