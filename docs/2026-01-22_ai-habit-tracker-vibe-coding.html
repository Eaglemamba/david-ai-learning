Part I：什麼是 Vibe Coding？

作者過去 6 個月用 Claude 成功建造了 10 多個工作流程，而這個 AI 習慣追蹤 App 是他認為對日常生活影響最大的一個。最關鍵的是：整個 App 只花了不到 30 分鐘就建造完成。

"Over the past 6 months, I've vibe-coded 10+ successful workflows using Claude. Of all my builds, this is the single most impactful workflow I've built for my daily life... The best part is, it only took <30 minutes to build."

這個工作流程的核心是一個「遊戲化」習慣追蹤 App，透過將日常任務轉換成遊戲任務和經驗值系統，來幫助使用者達成目標。

"An AI habit tracking app that 'gamifies' your life to help you crush your goals."

核心概念
Vibe Coding 是一種用自然語言指令讓 AI 生成完整程式碼的開發方式。不需要寫程式，只需要清楚描述你想要什麼，AI 就會生成可運作的應用程式。這種方式讓非工程師也能快速打造客製化工具。
比喻說明
想像你是製藥廠的 Operations Director，以前要做一個追蹤批次生產進度的 Dashboard，需要跟 IT 部門來回溝通好幾週。現在你可以直接用自然語言告訴 Claude：「我需要一個追蹤 Batch Record 完成率的介面，要顯示每條產線的進度...」，30 分鐘後就有一個可用的原型。

Part II：App 功能架構

這個 App 包含多個專屬 Dashboard，每個都針對特定領域設計：

1. Homepage Command Center：每日概覽的指揮中心

2. Sprint：遊戲化任務追蹤的「動量」Dashboard

3. Schedule：整合會議的行事曆總覽

4. Trading Terminal：每日交易/市場研究專用終端

5. Kinetic Performance：健身/體能任務追蹤

6. Biogenic Fueling：營養/健康任務追蹤

7. History：每週/月進度追蹤

8. Settings：偏好設定與各種模式

實務應用

CDMO 應用情境：你可以把這個架構改造成生產管理 Dashboard：

  • Command Center → 當日生產排程總覽
  • Sprint → Batch Release Checklist（每項有完成 XP）
  • Schedule → 跨部門會議與 Tech Transfer 時程
  • Terminal → Deviation Dashboard
  • Kinetic → 設備維護追蹤
  • History → 月度 KPI 趨勢
重點提示
注意這個 App 的設計邏輯：不是一個萬用工具，而是針對使用者的具體生活領域（交易、健身、營養）各自設計專屬介面。這種「模組化領域」的思維，比「一個通用介面」更能提高實際使用率。

Part III：如何建造這個系統

工具設定：作者推薦使用 Claude Opus 4.5 搭配 Max Plan。如果沒有 Max Plan，會很快遇到使用量限制，只能分批處理（每幾小時回來繼續）。

"Claude Opus 4.5 with Max Plan is best. If you're on anything less than a Max plan, you'll hit usage limits pretty fast."

省錢技巧：可以用 Google AI Studio（免費）先寫程式碼，再貼到 Claude 做優化，這樣可以繞過 Claude 的使用量限制。

"Pro tip: Use Google AI Studio. You can use Google AI Studio to code (free), then simply paste it into Claude, which will help bypass Claude usage limits."

規劃階段：在開始建造前，先用 ChatGPT 規劃功能。給它你的目標和背景，讓它幫你腦力激盪應該包含哪些功能。

"Before instructing Claude to build, I highly recommend prompting ChatGPT with an overview of the project to generate helpful feature ideas."

程式碼/API

Claude Project 設定建議：

1. 在 Claude Web 建立新 Project，命名為「AI Habit Tracker Workflow」

2. 加入基本文字 Context：「I'm building an AI habit tracker system...」

3. 這樣 Claude 會在整個對話中保持上下文記憶

比喻說明
這個工作流程像是「雙廚房策略」：用 ChatGPT 做菜單規劃（規劃功能），用 Google AI Studio 做食材備料（初步程式碼），最後用 Claude 做精緻擺盤（最終優化）。每個工具做它最擅長的事，整體效率最高。

Part IV：Prompt Engineering 結構

作者分享了他的 Mega Prompt 結構：[Role] + [Goal/Task] + [Features to Build] + [Constraints]

"This prompting structure worked perfectly: [Role] + [Goal/Task] + [Features to Build] + [Constraints]"

[Role] 角色：「Act as a world-class senior frontend engineer and elite UI/UX designer specializing in high-performance productivity tools and gamified interfaces.」

[Goal/Task] 目標：建造一個極簡「Cyber-Executive」風格的每日計劃器，專為 Power User 設計，包含 Sprint 執行、XP 遊戲化系統、高優先會議排程。

[Features] 功能：包含 Dashboard、Sprint、Schedule、Trading、Gym、Nutrition、History、Settings 八大模組。

[Constraints] 限制條件：深色背景 #0a0a0c、霓虹青綠 #00c8e0、電子紫 #9d5cff、成功綠 #22c55e。字體用 JetBrains Mono 和 Inter。

"Visual Aesthetic: Strictly 'Cyber-Minimalist' dark mode. Background: #0a0a0c. Accents: Neon Cyan (#00c8e0), Electric Purple (#9d5cff), and Success Green (#22c55e)."

重點提示

Modular Prompting 策略：避免一次要求 10 個功能。大型 LLM 在專注於單一「元件」時表現最好。作者稱之為「Micro-Sprinting」：

1. 先修好 Meetings Tab

2. 再優化 History Charts

3. 最後加上音效

這樣可以保持程式碼乾淨，Debug 也容易 10 倍。

實務應用

擴展 App 的正確姿勢：

"I want to [ADD FEATURE].
Functional Requirement: [...]
Data Structure: [...]
Visual Direction: [...]
Existing Context: The app uses 
constants.tsx and types.ts..."

這個結構讓 Claude 知道現有架構，避免重寫整個 App。

Part V：進階技巧

偷懶技巧：如果不想自己寫 Prompt，可以把文章中的所有圖片複製貼上，叫 Claude 逆向工程建造 App。

"If you're lazy, you can copy/paste all the images from this article and ask Claude to reverse-engineer and build the app."

偷設計：看到喜歡的 UI，就截圖給 Claude，用 Prompt：「Keep the logic of my App.tsx exactly the same, but update the styling of my cards to match the glassmorphism and padding seen in this image.」

"Steal Aesthetics: If you see a UI you like, take a screenshot and share it with Claude."

實際使用建議：把 Claude 下載到手機並放在容易存取的位置。每天早上花時間整理 To-do List，然後送給 App 作為 Prompt 來載入任務。

"Download Claude to your mobile device and place it in an easily accessible location. This will help you actually use the workflow & increase productivity."

重點提示

記憶體問題對策：App 可能會遇到記憶體問題（例如不載入每週/月歷史）。解決方法是主動送 Prompt：

「Ensure memory loads between desktop and mobile, make sure my data is saved every day, etc.」

這種「預防性 Prompting」可以避免後續問題。

比喻說明
「偷設計」這招就像製藥業的 Tech Transfer：你不是從零開發製程，而是看到別人成功的產品，把它的配方（視覺設計）拿過來，再根據自己的設備（邏輯需求）做調整。合法、高效、聰明。

重點整理

Vibe Coding 的核心價值
用自然語言在 30 分鐘內建造功能完整的 App。不需要程式背景，只需要清楚描述需求和限制條件。
Mega Prompt 結構
[Role] + [Goal] + [Features] + [Constraints] 是經過驗證的最佳結構。限制條件（顏色、字體、技術架構）越具體，結果越好。
Modular 微調策略
不要一次要求 10 個功能。專注單一元件、逐步疊代，程式碼更乾淨、Debug 更容易。
Multi-Tool 協作
ChatGPT 規劃功能、Google AI Studio 寫初稿、Claude 做最終優化。善用免費工具繞過使用量限制。

練習題

Q1 概念理解 為什麼作者建議用 Modular Prompting 而非一次要求所有功能？

答案：大型 LLM 在專注於單一「元件」時表現最好（類似人類的認知負荷限制）。一次要求太多功能會導致：

  • 程式碼結構混亂，不同功能之間的邊界模糊
  • 出現 Bug 時難以定位問題來源
  • AI 可能會在某些功能上偷工減料來完成任務

分階段建造（Micro-Sprinting）讓每個元件都能得到充分注意力，也方便後續維護和擴展。

Q2 產業應用 如果要用這個方法為 CDMO 建造一個 Batch Release Dashboard，Mega Prompt 的 [Constraints] 部分應該包含什麼？

答案：考慮到製藥法規要求（GMP、21 CFR Part 11），[Constraints] 應該包含：

  • 資料記錄：所有操作必須有 timestamp 和操作者 ID（ALCOA+ 原則）
  • 不可刪除：只能標記完成/撤銷，不能真正刪除記錄（Audit Trail）
  • 視覺設計：高對比度、清晰狀態標示（Pass/Fail/Pending）、可列印格式
  • 技術架構：資料必須持久化到 localStorage 或後端，不能只存在記憶體
  • 權限層級：不同角色看到不同資訊（Operator vs QA vs QP）
Q3 批判思考 這篇文章沒有提到的潛在風險或限制是什麼？

答案：文章呈現了最佳情境，但實際使用可能遇到：

  • 資料安全：localStorage 的資料沒有加密，敏感資訊（交易策略、健康數據）有外洩風險
  • 長期維護：AI 生成的程式碼可能難以理解和修改，技術債累積
  • 跨裝置同步：作者提到「記憶體問題」，但沒有真正解決跨裝置的資料同步
  • 依賴風險：如果 Claude 服務中斷或收費方案改變，整個工作流程可能失效
  • 過度客製化：太多專屬功能可能讓 App 變得複雜難用，違背「習慣追蹤」的初衷

這些不代表方法不好，而是在實際導入時需要考慮的因素。
