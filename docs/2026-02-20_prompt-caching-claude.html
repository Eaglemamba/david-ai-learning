Executive Summary

這篇來自 Claude Code 工程團隊的文章，揭示了一個讓長時間運行的 AI Agent 在成本和延遲上可行的關鍵技術：Prompt Caching（提示快取）。核心原理是 prefix matching（前綴匹配），API 會快取從請求開頭到每個 cache_control 斷點的所有內容。文章分享了 Claude Code 團隊在大規模優化中學到的反直覺教訓，包括排列順序策略、避免中途變更工具或模型、以及如何安全地進行 Context 壓縮（Compaction）。這不是理論文章，而是生產環境中用血淚換來的工程實踐。

Part I: Prompt Caching 核心原理
工程界有句名言「Cache Rules Everything Around Me」（快取主宰一切），對 AI Agent 來說同樣適用。像 Claude Code 這樣的長時間運行的 Agent 產品，之所以在成本和延遲上可行，靠的就是 Prompt Caching：讓我們重複利用前一輪對話的運算結果，大幅降低延遲和成本。
"Cache Rules Everything Around Me" — Long running agentic products like Claude Code are made feasible by prompt caching which allows us to reuse computation from previous roundtrips and significantly decrease latency and cost.
Prompt Caching 的運作原理是前綴匹配（prefix matching）：API 會將從請求開頭到每個 cache_control 斷點的所有內容進行快取。這代表你放置內容的順序極其重要，你需要讓盡可能多的請求共享相同的前綴。
Prompt caching works by prefix matching — the API caches everything from the start of the request up to each cache_control breakpoint. This means the order you put things in matters enormously.
💡 比喻說明

想像你在圖書館排書架。Prompt Caching 就像把所有人都會查的百科全書放在最前面，個人借閱的小說放在最後面。前面的書一旦就位就不動（被快取），後面的書可以隨時更換（動態內容）。如果你突然把百科全書的位置換掉，所有人都得重新找書（cache miss）。

📚 核心概念：排列順序

Claude Code 的最佳排列為：

  1. Static System Prompt & Tools（全域快取）
  2. Claude.MD（專案內快取）
  3. Session Context（工作階段快取）
  4. Conversation Messages（動態，最後）

原則：靜態內容在前，動態內容在後。

⚠️ 重點提示：脆弱性警告

這個排列順序非常脆弱！Claude Code 團隊曾因以下原因意外打破排列：在靜態 System Prompt 中放入時間戳記、非確定性地打亂 Tool 定義順序、動態更新 Tool 參數等。

Part II: 用 Messages 傳遞更新，不要改 Prompt
有時候你放在 Prompt 中的資訊會過時，例如時間變了或使用者修改了檔案。直覺上你會想更新 Prompt，但這會導致 cache miss，對使用者來說可能非常昂貴。
There may be times when the information you put in your prompt becomes out of date. It may be tempting to update the prompt, but that would result in a cache miss and could end up being quite expensive for the user.
考慮改用在下一輪的 Messages 中傳入更新資訊。在 Claude Code 中，我們會在下一個 user message 或 tool result 中加入 <system-reminder> 標籤來傳遞更新資訊（例如：現在是星期三），這樣就能保住快取。
Consider if you can pass in this information via messages in the next turn instead. In Claude Code, we add a <system-reminder> tag in the next user message or tool result with the updated information for the model.
🛠️ 實務應用

錯誤做法：每次時間更新就修改 System Prompt

正確做法：保持 System Prompt 不變，在 User Message 中加入：

<system-reminder>
  Current time: Wednesday 2:30 PM
  Updated file: config.yaml
</system-reminder>

這個模式的精髓：寧可讓訊息稍微冗餘，也不要打破快取前綴。

Part III: 不要在對話中途切換模型
Prompt 快取是依模型區分的，這會讓 Prompt Caching 的成本計算變得非常反直覺。如果你已經和 Opus 對話了 100k tokens，想問一個簡單問題，切換到 Haiku 實際上比讓 Opus 回答更貴，因為你需要為 Haiku 從頭建立整個 Prompt 快取。
Prompt caches are unique to models. If you're 100k tokens into a conversation with Opus and want to ask a fairly easy question, it would actually be more expensive to switch to Haiku than to have Opus answer, because we would need to rebuild the prompt cache for Haiku.
如果你需要切換模型，最佳方式是使用子代理（subagents），讓 Opus 準備一個「交接訊息」給另一個模型。Claude Code 的 Explore Agents 就是這樣使用 Haiku 的。
If you need to switch models, the best way to do it is with subagents, where Opus would prepare a "handoff" message to another model on the task that it needs done.
💰 成本分析：反直覺的數學

假設你有 100k tokens 的對話歷史：

選項 A：繼續用 Opus
快取命中 → 只需支付少量 cache read 費用 + 輸出 tokens

選項 B：切換到 Haiku
Haiku 需要重新建立 cache → 100k tokens 全部重新計費（cache miss）

即使 Haiku 每 token 成本更低，100k tokens 的 cache rebuild 成本遠超繼續用 Opus 的 cache read 成本。

💡 比喻說明

這就像你在 A 餐廳已經點了滿桌的菜還沒吃完，只因為想喝杯便宜的咖啡就換到 B 餐廳 — 結果得在 B 餐廳重新點所有菜。那杯咖啡再便宜，也不值得你重點整桌菜。

Part IV: 絕對不要在對話中途增減 Tools
在對話中途更改 Tool 集合是最常見的打破 Prompt Caching 的方式之一。直覺上你應該只給模型它當下需要的工具，但因為 Tools 是快取前綴的一部分，增加或移除任何一個 Tool 都會使整個對話的快取失效。
Changing the tool set in the middle of a conversation is one of the most common ways people break prompt caching. Because tools are part of the cached prefix, adding or removing a tool invalidates the cache for the entire conversation.
Plan Mode 的設計案例：直覺做法是進入計劃模式時，只保留唯讀工具。但這會打破快取。Claude Code 的做法是始終保留所有 Tools，用 EnterPlanMode 和 ExitPlanMode 作為 Tools 本身。當使用者切換到計劃模式時，Agent 透過系統訊息得知它在計劃模式中的指令。Tool 定義永遠不變。
Instead, we keep all tools in the request at all times and use EnterPlanMode and ExitPlanMode as tools themselves. The tool definitions never change.
額外好處：因為 EnterPlanMode 是模型可以自己呼叫的 Tool，它可以在遇到困難問題時自主進入計劃模式，完全不需要打破快取。
This has a bonus benefit: because EnterPlanMode is a tool the model can call itself, it can autonomously enter plan mode when it detects a hard problem, without any cache break.
📚 核心概念：設計原則

不要改變 Tool 集合 → 用 Tool 本身來表達狀態轉換

這是一個深刻的設計範式轉移：與其動態管理可用工具，不如讓工具本身成為狀態機的一部分。模式切換不是「移除工具」，而是「呼叫代表切換的工具」。

🛠️ Tool Search: Defer 而非 Remove

Claude Code 可能載入數十個 MCP Tools。全部包含太昂貴，但移除會破壞快取。

解決方案：defer_loading。不移除 Tools，而是發送輕量的 stub（只有 Tool 名稱 + defer_loading: true），模型可以透過 ToolSearch Tool 在需要時「發現」它們。完整的 Tool schema 只在模型選擇時載入。

這確保快取前綴穩定：相同的 stubs 始終以相同順序存在。

Part V: Context 壓縮（Compaction）的 Cache-Safe 策略
Compaction 是當你用完 Context Window 時的處理方式：摘要目前對話，然後以該摘要開始新的工作階段。但 Compaction 在 Prompt Caching 方面有許多反直覺的邊界情況。
Compaction is what happens when you run out of the context window. We summarize the conversation so far and continue a new session with that summary.
具體來說，當我們執行壓縮時，需要將整個對話發送給模型來產生摘要。如果這是一個使用不同 System Prompt、沒有 Tools 的獨立 API 呼叫（最直覺的實作方式），主對話的快取前綴完全不匹配。你需要為所有輸入 tokens 支付全額費用，大幅增加使用者的成本。
If this is a separate API call with a different system prompt and no tools, the cached prefix from the main conversation doesn't match at all. You pay full price for all those input tokens.
解決方案 — Cache-Safe Forking：執行壓縮時，使用與父對話完全相同的 System Prompt、User Context、System Context 和 Tool 定義。先放入父對話的歷史訊息，然後將壓縮提示作為新的 User Message 附加在最後。從 API 角度看，這個請求和父對話的最後一個請求幾乎一模一樣，所以能重用快取前綴。
When we run compaction, we use the exact same system prompt, user context, system context, and tool definitions as the parent conversation. We prepend the parent's conversation messages, then append the compaction prompt as a new user message at the end.
💡 比喻說明

想像你正在寫一本很長的會議紀錄。紀錄本快寫滿了，你需要換新本繼續。

錯誤做法：拿一本空白新本，重新抄寫所有內容來做摘要（全額重新輸入）。

正確做法：在原本的紀錄本最後一頁寫「請總結」，利用你已經翻閱過的所有頁面（已快取的前綴），只需多寫一行指令。

⚠️ Compaction Buffer

這個方法需要你預留一個「壓縮緩衝區（compaction buffer）」，確保 Context Window 中有足夠空間來放入壓縮訊息和摘要輸出 tokens。

好消息是 Anthropic 已將 Compaction 直接內建到 API 中，開發者可以直接使用這些經過優化的模式。

📚 核心原則：Fork 操作共享父前綴

任何需要分支運算的操作（壓縮、摘要、技能執行），都應使用與父對話相同的 cache-safe 參數，以確保命中父對話的快取前綴。這是一個可以推廣到所有 Agent 系統的通用設計原則。

Key Takeaways
🎯 前綴匹配是一切的基礎
Prompt Caching 是前綴匹配。前綴中任何位置的任何變更都會使其後的所有內容失效。整個系統設計都要圍繞這個限制來做。排列順序正確了，大部分的快取就自然到位。
💬 用 Messages 取代 System Prompt 變更
任何你想在 System Prompt 中更新的資訊（時間、模式切換、檔案更新），都改用對話中的 Messages 傳遞。保護快取前綴的穩定性，永遠是優先考量。
🚫 不要在對話中途改 Tools 或 Models
用 Tool 本身來表達狀態轉換（如 EnterPlanMode），而非改變 Tool 集合。用 defer_loading stubs 取代移除 Tools。切換模型用 subagent 交接，不直接替換。
📊 像監控 Uptime 一樣監控 Cache Hit Rate
Claude Code 團隊對 cache miss rate 設定警報，視為 SEV 等級事件。幾個百分點的 cache miss 率就能劇烈影響成本和延遲。Fork 操作（compaction）需共享父對話的前綴。

Practice Questions

Q1 概念理解 為什麼在 100k token 對話中切換到更便宜的模型反而更貴？請用 Prompt Caching 原理解釋。

因為 Prompt Cache 是依模型區分的（model-specific）。當你從 Opus 切換到 Haiku 時，Haiku 沒有任何已建立的快取前綴，需要重新處理整個 100k tokens 的輸入（cache miss）。即使 Haiku 的每 token 成本更低，重建 100k tokens 快取的費用遠高於讓 Opus 利用現有快取（cache read）來回答的費用。

正確做法是使用 subagent 模式：讓 Opus 準備一個精簡的交接訊息，發給 Haiku 處理特定子任務，而不是把整個對話 context 搬過去。

Q2 產業應用 在你的 CDMO 環境中，如果要建立一個 AI Agent 來協助批次記錄審查（Batch Record Review），如何應用 prompt caching 原則來設計系統？

可以這樣設計 prompt 排列順序：

1. Static System Prompt（全域快取）：GMP 規範要求、ALCOA+ 原則、審查 SOP 標準、常見偏差類型定義。這些法規知識幾乎不變，放在最前面。

2. Product-Specific Context（專案快取）：特定產品的製程參數規格、批次大小、關鍵品質屬性（CQA）。同產品的所有批次共享此快取。

3. Session Context（工作階段快取）：當前批次的基本資訊、設備狀態。

4. Messages（動態）：具體的審查問題、偏差描述、逐頁的批次記錄資料。

如果製程參數在審查過程中需要更新（如設備校驗結果），使用 <system-reminder> 模式在 message 中更新，不要修改 System Prompt。

Q3 批判思考 文章中 "defer_loading" 的 Tool stub 策略可能有什麼潛在的缺點或風險？

幾個值得考慮的面向：

可發現性（Discoverability）降低：模型只看到 Tool 名稱而沒有完整 schema，可能無法判斷何時該使用某個 Tool，導致遺漏適合的工具。模型需要先「猜測」再搜尋，增加了中間步驟。

延遲交換成本：雖然省了快取成本，但當模型需要使用 deferred tool 時，需要額外一次 ToolSearch 呼叫來載入完整 schema，這會增加該次互動的延遲。這是典型的「空間換時間 vs 時間換空間」取捨。

工具順序穩定性：如果 stub 的排列順序不穩定（例如新增 MCP Tools 後位置變動），仍然會打破快取。需要有機制確保 stub 列表的排序是確定性的。

🎯 我的行動結論
✔ 核心洞察
Prompt Caching 不只是「API 使用的小技巧」，而是決定 Agent 系統是否在經濟上可行的架構級決策。「用 Tool 本身表達狀態轉換」這個設計模式，本質上是在說：為了快取穩定性，你需要重新思考整個系統的抽象層次。這和 GMP 環境中「不要為了方便而改變驗證過的流程」的思維完全一致 — 穩定性本身就是一種價值。
📈 立即應用
本週在使用 Claude Code 或 Claude API 時，有意識地檢查：我的 System Prompt 中有沒有不必要的動態內容？我的 Tools 定義是否保持穩定？可以用這篇文章的框架來 audit 目前的 Make.com 自動化流程中的 Anthropic API 呼叫，確認是否有優化空間。同時，在建立 AI Agent 概念驗證時，從第一天就將 prompt 排列順序設計為 cache-friendly。
⚠ 需要補充
文章沒有提供具體的成本數字（cache hit vs miss 的價差倍數）和延遲基準數據，需要查閱 Anthropic 官方定價頁面來量化實際效益。此外，文章主要從 Claude Code（CLI Agent）角度出發，對於 Web-based 或企業級 Agent 的 caching 策略（如多用戶共享快取）著墨較少。建議在 Anthropic 的 API 文件中深入閱讀 auto-caching 和 API-level compaction 的技術規格。
